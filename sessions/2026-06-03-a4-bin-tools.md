## What the sweep surfaced

### Headline 1: psql secret-scrub cluster

`~/.psql_history` records `CREATE USER ... PASSWORD '...'` and
`ALTER ROLE ... PASSWORD ...` **verbatim**. The only filter is
`HISTCONTROL=ignorespace`/`ignoredups` — no password-pattern filter
(unlike many shells). The history file is `0600` but `open(O_CREAT)`
does NOT use `O_NOFOLLOW`, so a `PSQL_HISTORY` env through a writable
dir is a symlink-redirect vector.

`-L logfile` captures every query including embedded PASSWORD
literals; `fopen(logfile, "a")` honors umask (NOT enforced 0600).

Password buffers freed without `explicit_bzero` in:
- `startup.c:249-302` — `main()` connection password
- `command.c:2604-2607` — `\password` `pw1`/`pw2`
- `settings.h:103` — `pset.db` indirectly holds libpq's cached creds

### Headline 2: pg_basebackup backup-stream trust model

The mirror image of A3's pg_dump "trust the source" finding. pg_dump
trusts the *archive file on disk* (third-party tampering); pg_basebackup
trusts the *server it streams from* (compromise/buggy window).

- `pg_basebackup.c:1145-1150` — server-controlled `spclocation` becomes
  output dir, only `-T olddir=newdir` user mapping filters it
- `streamutil.c:367-394` — server-reported `data_directory_mode` sets
  the local umask; server says `0777` → world-readable backup;
  validation is just `sscanf` success
- `pg_basebackup.c:2858` — `--waldir` not validated as absolute before
  `symlink()`
- `pg_createsubscriber.c:1460,214,47-48` — publisher conninfo (may
  include password) written to `pg_createsubscriber.conf` in subscriber
  data dir; renamed to `.disabled` but NOT shredded — intentionally
  retained "for debugging"

### Headline 3: initdb `--pwfile` stale-TODO

The most egregious finding: upstream **explicitly acknowledged the gap
in a code comment** and said *"Ideally this should insist that the
file not be world-readable. However... we'll skip the paranoia for
now"* — that "for now" has shipped for years. A world-readable pwfile
is silently accepted.

`superuser_password` (`initdb.c:1732`) is a file-scope `static char *`
set once and never freed/zeroed. Same shape as the A2 libpq finding
(60+ files don't use `explicit_bzero`).

### Cross-corpus pattern — four secret-scrub findings now form a series

| Source | What leaks | Where |
|---|---|---|
| A2 libpq | `PGconn`-held passwords, scram keys, pgpass, oauth tokens | full connection lifetime; `explicit_bzero` in 2 of 60+ files |
| A4 psql | `\password` pw1/pw2; `main()` password arg; PSQL_HISTORY; `-L` logfile | command.c:2604; startup.c:249; input.c:148; common.c:1158 |
| A4 streamutil | static-global `password` for replication tools | streamutil.c |
| A4 initdb | `superuser_password` + escape_quotes copy until exit | initdb.c:1732 |

**Single proposed mitigation across all four:** introduce a `SecretBuf`
type in `src/common` with `secret_alloc` / `secret_free` (latter calls
`explicit_bzero` then `free`). Sweep #5 (`src/common/`) is the hosting
site — closes the four findings in one coordinated series.

### What's working well

- **`processSQLNamePattern`** (in `fe_utils/string_utils.c`) is the
  single chokepoint for psql's `\d*` pattern handling. The B2 agent
  audited every `\d*` callsite through `validateSQLNamePattern`; no
  pattern-injection vulnerability exists because patterns become
  string-literal LIKE values, never identifier interpolation. **Same
  helper is reused by pg_dump.c** — cross-binary trust boundary
  worth elevating to `knowledge/idioms/sql-name-pattern.md`.

- **`:'var'` and `:"var"` quoted-form interpolation IS injection-safe**
  (implemented in psqlscan.l, not in variables.c). Bare `:var` is
  unsafe by design — but the quoted forms close the `\gset` server→
  client SQL-injection concern as long as users use them.

- **The `\restrict <key>` framing** in command.c (the pg_dump→psql
  hardening from A3) is exactly scoped: it prevents server-injected
  **backslash** commands during dump replay; it does NOT prevent
  malicious SQL the server emits and pg_dump faithfully forwards.
  Defining the scope this clearly is a corpus-side contribution.
