## What the sweep surfaced

### Headline 1: The SecretBuf hosting site

The single most important finding of this batch. The fifth installment
of the secret-scrub cluster (A2 libpq + A4 psql + A4 streamutil + A4
initdb + now A5 common) shows that **`src/common/` is where the helpers
all four subsystems call live**. A unified `SecretBuf` type in
`src/include/common/secretbuf.h` + `src/common/secretbuf.c` would:

- Close 10+ A5 sites directly (md5_common, scram-common, hmac, saslprep,
  sprompt, fe_memutils, logging, pg_get_line, sha1, cryptohash_openssl).
- Migrate the 4 prior cross-corpus call sites in one patch each.
- Add no new dependencies (`src/common` depends on nothing else).
- Match the existing `_create`/`_update`/`_final`/`_free` cryptohash idiom.

**API sketch:**
```c
typedef struct SecretBuf SecretBuf;
SecretBuf *secret_alloc(size_t len);
void *secret_data(SecretBuf *buf);
void secret_free(SecretBuf *buf); /* explicit_bzero + free */
```

The crypto helpers already do `explicit_bzero(ctx, sizeof(*ctx))` on
`_free` uniformly across in-tree + OpenSSL variants. The discipline
exists for context state; what's missing is the caller-owned wrapper
for the secret bytes that *leave* the context (digests, derived keys,
the cleartext input).

### Headline 2: Backup-trust model echoes A3 pg_dump

`blkreftable.c` + `parse_manifest.c` together form the mirror image of
A3's pg_dump archive-trust finding:

- **`blkreftable.c`** authenticates the BRT file via magic + trailing
  **CRC-32C only**. An attacker who rewrites the body trivially
  recomputes the CRC. The `limit_block` field is fully attacker-
  controlled: a hostile BRT with `limit_block = max` and an empty
  modified-block bitmap tells `pg_combinebackup` "zero blocks modified"
  → **silent stale data** in the combined backup.
- **`parse_manifest.c`** uses SHA-256 over the manifest, which is
  **integrity not authenticity**: attacker who rewrites manifest +
  per-file CRCs together evades all checks. Default per-file checksum
  is CRC-32C (collision-trivial), explicitly disclaimed as crypto-grade
  in the `checksum_helper.h` comment.

The defense in both cases is the same shape A3 identified for pg_dump:
file-format checksums protect against accidental corruption, not against
an attacker who controls the chain.

### Headline 3: Decompression bombs + parser DoS

- **`pg_lzcompress.c`** has no input/output ratio bound on decompression —
  classic bomb. Mirror of A4's walmethods finding.
- **`jsonapi.c`** depth cap is 6400 in the incremental parser ONLY;
  recursive-descent `pg_parse_json` relies on `check_stack_depth()`
  which is a **no-op in libpq frontend**. Hostile JSON consumed via the
  recursive path is bounded only by OS stack. Stale TODO from many
  releases admits "clients need some way to put a bound on stack
  growth".
- **`stringinfo.c`** length is `int` + `MaxAllocSize`-bound (1 GiB);
  attacker who can grow a single StringInfo (JSON parse, query result)
  hits this ceiling silently.
- **`unicode_norm.c`** has no input-length cap and quadratic worst-case
  canonical reordering.
- **`wchar.c`** `pg_utf8_verifystr` backtracks at end-of-input on
  partial trailing sequence.

### Headline 4: GUC-boundary shell-injection

`percentrepl.c::replace_percent_placeholders` does NO shell escaping;
its output goes straight to `system(3)` via `archive_command`/
`restore_command`. Safety relies on an undocumented "placeholder
values are not attacker-controlled" invariant. `archive.c` similarly
does not shell-quote `%p`/`%f`/`%r` — upstream documentation tells
admins to wrap `"%p"` in their GUC value.

This is the **GUC-level analog** of the psql `\!` / `\copy PROGRAM`
findings from A4 and the pg_dumpall `system()` chain from A3 — same
trust boundary, different binary.

### Headline 5: controldata_utils torn-write window

`pg_control` is the cluster's single source of truth. Update is a
single 8 KiB `write()` with no shadow file, no rename-into-place, no
two-copy protocol. The CRC-retry loop in the reader only handles a
racing writer, not a torn-write-on-disk surviving reboot. Mitigated
in practice by 4 KiB sector atomicity on modern HW.

### Notable correctness findings

- **`pg_abs_s64(PG_INT64_MIN)`** is the only safe path for negating
  `INT64_MIN`. Every direct `abs()`/`labs()`/`(uint64)-x` on
  user-controlled signed integers across the backend is a UB landmine.
  Grep audit of bare `abs(`/`-x` on signed user input is the
  highest-leverage Phase D follow-up identified.
- **DSM control-handle generation** uses `pg_prng_uint32` (seeded once
  from `pg_strong_random`, then every forked backend advances the same
  stream). Local attacker on same host has narrowed search space for
  pre-creating `/dev/shm/PostgreSQL.<handle>` paths. Mitigated by
  collision retry. Should be `pg_strong_random`.
- **OpenSSL-vs-fallback dispatch** is build-system pick-one-of-two
  (no `#ifdef USE_OPENSSL` in headers); asymmetric error semantics
  (backend `ereport(ERROR)` vs frontend `NULL`-return) for the same
  public API.

### What's working well (record)

- **`cryptohash_*`/`hmac_*` already scrub context state** — every `_free`
  does `explicit_bzero(ctx, sizeof(*ctx))` across in-tree and OpenSSL
  variants uniformly. The discipline EXISTS for opaque-handle state.
- **`pg_strong_random` is available everywhere** (falls back OpenSSL →
  Windows CryptoAPI → `/dev/urandom`). Where weak `pg_prng` is used
  for security purposes (SCRAM nonces, MD5 salts, UUIDv4, mock auth),
  it correctly uses `pg_strong_random`. The only borderline case is
  DSM handle generation.
- **`processSQLNamePattern`** (cross-binary chokepoint from A4) continues
  to apply: `src/common/` doesn't have its own pattern-injection surface.
- **Unicode generators confirmed** at `src/common/unicode/generate-*.pl`
  with `make update-unicode` target.
