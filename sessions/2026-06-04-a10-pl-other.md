## Headlines (the cross-PL trust-gate comparison)

The full headlines live at the top of each subsystem register; the
core finding is the **cross-PL trust-gate ranking**, repeated here.

### Trust-gate ranking: pltcl ≥ plperl > plpgsql > plpython

| PL | Mechanism | Strength | Why |
|---|---|---|---|
| **pltcl** | `Tcl_CreateSlave(safe=1)` at C-level command-dispatch | **Strongest** | Dangerous commands (`exec`, `socket`, `open`, `file delete`, `load`) NOT PRESENT in safe interp's command table at all. Maintained as Tcl-core feature. |
| **plperl** | `PL_op_mask = plperl_opmask` generated from Perl's Opcode.pm at build time | Medium | Opcode-level gate; primitive exists in interpreter, just can't compile a ref through mask. Drift-prone as Perl adds new ops. **NOT Safe.pm despite docs.** |
| **plpgsql** | Nothing — language has no I/O constructs | N/A (by elimination) | Can't escape what doesn't exist. Trusted because the language is purely SQL-via-SPI. |
| **plpython** | `superuser = true` at `CREATE EXTENSION plpython3u` | Coarsest | Python's `import`/`__builtins__`/getattr/ctypes make Safe.pm-equivalent impossible. Once an admin grants USAGE, every grantee has full filesystem + network access. |

### Other A10 headlines

1. **`PLy_cursor_plan` is missing the `is_PLyPlanObject` check** that
   `PLy_spi_execute` has (`plpy_cursorobject.c:88-89, :185` vs
   `plpy_spi.c:165`). Text-fallback `"O|O"` parse accepts any
   PyObject as a "plan". **Most concrete Phase D candidate from
   A10** — worth a focused reachability trace.

2. **One Python interpreter per backend, never finalized**
   (`plpy_main.c:70`). `sys.modules` mutations, monkey-patches, and
   ctypes loads persist for backend lifetime. **Dramatically
   amplified by transaction-pooling connection multiplexers** — a
   function on connection A monkey-patches `os.open`; connection B
   (pooled to same backend) inherits the patch.

3. **GIL held continuously across SPI calls.** No
   `Py_BEGIN_ALLOW_THREADS` anywhere in plpython. Python threads
   inside a function cannot productively run during SPI; backend-
   wide latency hit if a plpython function spawns threads.

4. **`plpy.execute(text)` injection surface = plpgsql injection ∪
   every PG type's `_in()` function.** Text path identical to
   plpgsql `exec_stmt_dynexecute` (no parameterization). PLUS the
   entire scalar-return path means every PG type's `_in()` is
   reachable from a Python return value via `PLyObject_ToScalar`
   (`plpy_typeio.c:1077-1096`). Broader attack surface than plpgsql.

5. **`plpy.subtransaction()` swallows Python `try/except` and
   COMMITS subxact** (`plpy_subxactobject.c:178-186`). **Opposite of
   plpgsql `EXCEPTION`** which forces rollback. Real cross-PL
   footgun for users porting code.

6. **plperl one Perl interpreter per `(trusted?user_id:0)`, never
   evicted.** Long-lived connection-pooled backends accumulate one
   Perl interpreter per distinct UID. Combined with `plperl.on_init`
   running once in postmaster (when preloaded) being inherited by
   every fork: postmaster-side untrusted Perl code can persist
   across the entire instance.

7. **`pltcl.start_proc` / `pltclu.start_proc` PGC_SUSET with same-
   prolang + non-SECURITY-DEFINER checks** (`pltcl.c:642-658`).
   Trust-gate keystone, design is sound. Worth documenting because
   the protective design is non-obvious.

8. **pltcl's dual-posture is the cleanest in the tree** — single
   .c file via `bool pltrusted` parameter threaded through every
   function. More maintainable than plperl's parallel handler paths.
