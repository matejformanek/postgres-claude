## What the sweep surfaced

### Headline: pg_restore's archive-format trust model is "trust the source"

Found in B2. There is essentially **no defense against a maliciously-
crafted archive file being replayed by pg_restore as superuser**:

- `_printTocEntry` writes `te->defn` verbatim into the SQL stream that
  pg_restore replays — at full privilege if the user is superuser.
- `processSearchPathEntry` stashes the verbatim defn for emission — an
  attacker-controlled `SET search_path` in the archive runs at restore.
- Format backends are length-prefix-only without semantic validation.
- The `\restrict <key>` / `\unrestrict <key>` framing in `RestoreArchive`
  explicitly addresses **malicious server response to dump queries**
  (corrupted result rows blocking psql meta-commands at restore time),
  **NOT malicious archive content**.

**Concrete Phase D fuzzing targets:**
1. **Custom format** (`pg_backup_custom.c`) — `_PrintTocData`'s
   linear-scan lets a hostile archive populate `dataPos`/`dataState`
   of arbitrary other TEs at chosen offsets via the
   `othertctx->dataPos = thisBlkPos` side-effect, gated only by
   `WriteInt(dumpId)` sanity check. `_CustomReadFunc` does
   `pg_malloc(blkLen)` of an attacker-chosen int-widened-to-size_t.
   `_skipData` non-seekable + seekable variants both have malloc /
   fseeko vulnerabilities.
2. **Directory format** (`pg_backup_directory.c`) — `setFilePath(buf,
   "<dir>/<filename>")` blindly concatenates the in-archive filename
   from the TOC. Restore-side reads follow symlinks (no `O_NOFOLLOW`).
   Path-traversal hazard via attacker-controlled `te->filename` and
   `lofname`.

**Hardening pitch (the Phase D coordinated patch):**
- Reject archives whose `dump_version` doesn't match pg_restore's own
  (currently warned but allowed).
- Reject `..` and absolute paths in directory-format filenames.
- Cap individual data-block sizes in custom and tar.
- Validate `blkType` and compression-algo bytes in custom format.
- Add `O_NOFOLLOW` to directory-format read paths.
- Add a "trusted" / "untrusted" archive mode flag.

### Other high-priority Phase D leads

- **pg_dumpall + `expand_dbname=true` with server-supplied `datname`**
  (`connectdb.c:154`): if a hostile cluster has a database named
  `host=evil port=1234 dbname=foo`, libpq's connection-string expansion
  might redirect the next per-DB connection. Highest-priority Phase D
  lead for pg_dumpall; needs verification against libpq's actual
  `conninfo_uri_parse_options` rules.

- **`restrict_nonsystem_relation_kind` central security boundary**
  (`pg_dump.c`): the GUC prevents an attacker-controlled view or
  foreign table from running code as the dumping role. It's
  temporarily relaxed to `"view"` (omitting `foreign-table`) in 3
  places during foreign-table COPY — owner-controlled FDW can be a
  code-exec primitive against the dumper. Mis-revert = privilege
  escalation against subsequent dump work in the same connection.

- **`savedPassword` parallel-clone proliferation** — `pg_strdup`'d per
  worker; N copies in process memory during parallel restore; no
  scrub-on-teardown. Same pattern as libpq's PGconn credential lifetime
  finding from A2.

- **`pg_init_privs` baseline collection invariant**: missing it =
  overgranted permissions silently in restored objects.

- **pg_dumpall globals dumped WITHOUT transaction snapshot** —
  race window between role/membership/GUC reads.

- **pg_dumpall→pg_dump uses `system()` with shell-quoted args** —
  depends on `appendShellString` correctness.

### What's working well

- **pg_dump's SQL-assembly discipline is solid.** `fmtId()` +
  `appendStringLiteralAH/Conn` + `quote_all_identifiers=true`
  belt-and-suspenders + forced `standard_conforming_strings=on`. The
  16 callsites in `dumputils.c` were audited correct by the B4 agent.
  The residual risk is the caller contract (pg_dump.c must pre-quote
  args to ACL builders), not the helpers themselves.

- **Section-boundary anchors** with compile-time `StaticAssertDecl`
  guard the pre-data → data → post-data emission ordering across all
  48 object kinds.

- **Parallel-restore worker safety** is good though it rests on subtle
  invariants — leader builds read-side data structures before workers
  exist; workers mutate only clone state. `CloneArchive` flat memcpy +
  selective NULL-out (connection, currUser, currSchema, lo_buf) is the
  contract. `clone->ropt->txn_size = 0` override is a hard-to-spot
  but correct invariant.

- **Compression vtable + library version detection** is clean across
  all 4 backends (none/gzip/lz4/zstd). The only structural concern is
  the decompression-bomb DoS — no input/output size ratio bound on
  inflate loops — but this is a lower-priority concern than the
  archive trust model.
