## The five-stage pipeline (in one diagram)

```
patch reference (CF# / PR# / file)
  │
  │  /pg-review <ref>
  ▼
Stage 0 — mechanical preamble (5-10 min)
  • fetch patch (WebFetch for CF; gh pr diff for PR; file path direct)
  • git am on dev/cf<N>-review (or pr<N>-review)
  • ninja install
  • meson test regress/regress
  • meson test --suite isolation
  • targeted suites (--subsystem hint)
  │
  │  hand off
  ▼
Stage 1 — project discovery (5 min, main agent)
  • map touched files → subsystems (knowledge/subsystems/*.md)
  • load relevant per-file docs (knowledge/files/*.md)
  • extract claims from patch's commit message
  • build the dispatch block (passed to every critic)
  • spot-check 3-5 cites for corpus drift
  │
  │  fan out (single message, 4 parallel Agent calls)
  ▼
Stage 2 — 4 critic sub-agents IN PARALLEL (~10-20 min wall time)
  ┌─────────────────────────┐  ┌─────────────────────────┐
  │ A: Architecture +       │  │ B: Breaking-change scan │
  │    INV-* invariants     │  │    (on-disk, WAL,       │
  │    (vs subsystem docs)  │  │    catalog, ABI)        │
  └─────────────────────────┘  └─────────────────────────┘
  ┌─────────────────────────┐  ┌─────────────────────────┐
  │ C: Test coverage        │  │ D: Style + commit msg   │
  │    (regress vs iso vs   │  │    (pgindent, coding-   │
  │    TAP, corner cases)   │  │    style, msg style)    │
  └─────────────────────────┘  └─────────────────────────┘
  │
  │  consolidate
  ▼
Stage 3 — orchestrator consolidates (10 min)
  • dedupe across critics
  • resolve conflicts
  • severity sort: blocking / warning / nit / open question
  • verdict: Ready for Committer / Waiting on Author / etc.
  │
  ▼
Stage 4 — synthesize PG-house-style review email
  • imperative, plain text, no HTML, ~76 col wrap
  • Blocking issues numbered, then Warnings, then Nits, then Q&A
  • Testing-performed block (exact commands + counts)
  • Appendices: per-critic raw findings + stage-0 mechanical log
  • Output: sessions/<date>-cf<N>-review.md
```

## Design decisions

### Why a skill + a command rather than just a command

The slash command's job is mechanical and bash-shaped (download patch,
apply, build, test). The skill's job is judgement (load corpus,
orchestrate critics, synthesize). Splitting them mirrors the
brainstorm/plan/implement pattern: the command is the trigger, the
skill is the procedure.

This also means the skill is usable WITHOUT the command — if the patch
is already applied + built in `dev/`, invoke the skill directly with
`--skip-build`. The command is the convenience layer.

### Why 4 critics, not 2 or 8

`plan-review-comprehensive` uses 4. The validation run's seven
review-checklist phases collapse cleanly onto four non-overlapping
slices:

- **Architecture + INV-* invariants** absorbs Phase 6 (Architecture).
- **Breaking-change** absorbs Phase 6's ABI/WAL/catalog/on-disk bullets
  + Phase 2's "interacts with other features" check.
- **Test coverage** absorbs Phase 3 (Feature test) + Phase 2's
  "includes tests" check.
- **Style + commit message** absorbs Phase 5 (Coding) + Phase 7
  (Review review) + Phase 1's "filename + format" check.

Phase 4 (Performance) is left to ad-hoc — most patches don't have
performance claims; the synthesizer adds a perf-review-asked question
to the email if the patch claims any speedup.

### Why parallel fan-out (not sequential)

Each critic loads non-overlapping slices of the corpus. Sequentially
that's 4× the wall time. In parallel it's max(t1, t2, t3, t4) + small
orchestration overhead. The corpus is read-only; there's no
shared-state contention.

### Why `pg-patch-review` distinct from `review-checklist`

`review-checklist` is the seven-phase scaffold — it's the syllabus
each critic applies INSIDE its assigned slice. `pg-patch-review` is
the orchestration layer above it (project discovery + parallel
critic fan-out + synthesis).

The two compose: a critic loads `review-checklist` and walks the
phase relevant to its slice. The skill's hand-off table makes this
explicit.

### Why a separate slash command for the mechanical phase

The user has to APPROVE several heavy actions during stage 0 —
fetching from a URL, `git am` (which can fail mid-stream), running
`meson test` (5-10 min). Putting these in a slash command means the
user sees them under the `/pg-review` umbrella, not buried inside a
"skill is now doing this" black box. Same rationale as `/pg-brainstorm`
fronting `pg-feature-brainstorm`.

### What this commit explicitly does NOT do

- **No first real run.** The pipeline is wired but not exercised on a
  fresh CF entry. Queued as Next §2 in STATE.md. Calibration target:
  reproduce the v0 CF #6402 result in less wall time.
- **No `patch-submission` integration.** The skill notes that
  pre-submission self-review should use `patch-submission` (which
  invokes `review-checklist` directly today), but the integration to
  also invoke the 4-critic fan-out from `patch-submission` is queued
  for later.
- **No corpus changes.** The skill explicitly does NOT write to
  `knowledge/` (per R10 of the rules file). If a review surfaces a
  corpus gap, fix it in a SEPARATE `hf(corpus):` commit.
- **No changes inside `dev/`.** This is all meta-repo wiring.

## What I learned wiring this up

- The synthesizer's tone IS the `commit-message-style` skill's tone
  (imperative, ~76 col wrap, plain text). Treating the review email
  as "a commit message, but longer, addressed to the author" gives a
  consistent voice across pg-claude's upstream-bound outputs (commits,
  patches, review emails).
- The "claims the patch makes" extraction in stage 1 is the single
  most decisive item. The CF #6402 v0 review hinged on validating
  one claim ("no behavior change") against the actual diff; baking
  that as a stage-1 step makes it standard.
- The corpus-drift spot-check at the end of stage 1 doubles as a
  health check for the knowledge corpus. If reviews start consistently
  bailing here, it's a signal to schedule a `pg-corpus-maintainer`
  pass before the next CF window.
- The orchestrator's job in stage 3 is mostly de-duplication. Two
  critics WILL flag the same thing from different angles; merging is
  the principal value-add.
