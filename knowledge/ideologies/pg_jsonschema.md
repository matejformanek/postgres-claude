## Domain & purpose

pg_jsonschema "adds support for JSON schema validation on `json` and `jsonb`
data types" (`README.md:17`) `[from-README]`. It is "a (very) thin wrapper around
the [jsonschema] rust crate" (`README.md:127`) `[from-README]`: the four base SQL
functions — `json_matches_schema`, `jsonb_matches_schema`, `jsonschema_is_valid`,
`jsonschema_validation_errors` (`README.md:21-27`) — each forward a `serde_json`
`Value` to the crate and return a `bool` or a `text[]` of error strings
(`src/lib.rs:9-40`) `[verified-by-code]`. The intended use is a `CHECK`
constraint that pins a `json`/`jsonb` column to a schema (`README.md:80-104`)
`[from-README]`.

The reason to document it: the base functions are *purely stateless* — they
recompile the validator from scratch on every row. To kill that recompilation
cost the extension grew a **second API tier**: a `jsonschema` base type plus a
two-level (per-callsite + backend-local LRU) compiled-validator cache
(`src/compiled/`). That cache is the anthropologically interesting part. Core
Postgres has no caching layer for `jsonb` operator/function evaluation; here an
extension hand-rolls one against `fn_extra` and a `thread_local`, and in doing so
makes a set of memory-context and lifetime bets that core's own per-call
functions never make.

## How it hooks into PG

A bare pgrx module: `pg_module_magic!()` (`src/lib.rs:7`) and `#[pg_extern]`
functions, no `_PG_init`, no GUC, no hooks `[verified-by-code]`. The `.control`
file marks it `relocatable = false` and `superuser = true` with
`default_version = '@CARGO_VERSION@'` substituted by pgrx at build time
(`pg_jsonschema.control:2-5`, `README.md:183`) `[verified-by-code]`.

- **Base validation functions**, all `#[pg_extern(immutable, strict,
  parallel_safe)]`: `json_matches_schema`/`jsonb_matches_schema` →
  `jsonschema::is_valid` (`src/lib.rs:9-17`); `jsonschema_is_valid` →
  `jsonschema::meta::validate` (`src/lib.rs:19-28`); `jsonschema_validation_errors`
  → `validator_for(...).iter_errors(...)` collected into `Vec<String>`
  (`src/lib.rs:30-40`) `[verified-by-code]`.
- **A `jsonschema` base type** declared by pgrx's `#[derive(PostgresType, …)]` on
  a one-field `struct JsonSchema { value: String }` with custom `#[inoutfuncs]`
  (`src/compiled/mod.rs:17-30, 42-52`) `[verified-by-code]`. Its on-disk
  representation is the *canonical* JSON string of the schema.
- **Two casts**, `json → jsonschema` and `jsonb → jsonschema`, created via
  `pgrx::extension_sql!` over `jsonschema_from_json`/`jsonschema_from_jsonb`
  (`src/lib.rs:42-62`) `[verified-by-code]`.
- **Compiled-schema functions** taking the `JsonSchema` type plus a raw
  `fcinfo: pg_sys::FunctionCallInfo`, e.g. `jsonb_matches_compiled_schema`
  (`src/lib.rs:64-108`) `[verified-by-code]`.

The fmgr glue (`PG_FUNCTION_INFO_V1`, the `Datum` marshalling, the `Pg_magic_func`
ABI) is generated by pgrx's `#[pg_extern]` proc-macro — pg_jsonschema never writes
a `Datum foo(PG_FUNCTION_ARGS)` entry point by hand. See
`[[knowledge/ideologies/pgrx]]` for how that codegen reproduces the C extension
ABI. Cross-ref `[[knowledge/idioms/fmgr]]`,
`.claude/skills/fmgr-and-spi/SKILL.md`,
`.claude/skills/extension-development/SKILL.md`.

## Where it diverges from core idioms

### 1. A hand-rolled, two-level validator cache where core has none

Core Postgres evaluates a `jsonb` predicate function fresh on every tuple; there
is no per-schema memoization for jsonb functions. pg_jsonschema's compiled tier
adds one, explicitly motivated: "a 3.3 MB FHIR schema hashed across 20k rows
takes ~13 s via LRU alone vs ~1.8 s with the L1 callsite cache, which reduces
each hot-path lookup to a pointer dereference" (`src/compiled/callsite.rs:42-45`)
`[from-comment]`.

- **L1** is a per-callsite slot stashed in `fcinfo->flinfo->fn_extra`, lifetime
  bounded by `fn_mcxt` (`src/compiled/callsite.rs:32-34, 94-118`)
  `[verified-by-code]`. This is exactly the `fn_extra` cross-call scratch idiom
  core uses for things like SRF state and cached `FmgrInfo`s — but here it caches
  a compiled `Arc<jsonschema::Validator>`.
- **L2** is a backend-local LRU keyed on the canonical schema string, capacity
  128, hit only on an L1 miss (`src/compiled/cache.rs:9-30`,
  `src/compiled/callsite.rs:36-37`) `[verified-by-code]`.

The base functions (`src/lib.rs:9-40`) bypass both tiers and recompile every
call; the README's own benchmark records the gap (~195 ms vs ~110 ms on a
trivial schema, "the gain grows with schema complexity", `README.md:259-264`)
`[from-README]`.

### 2. The LRU lives in a `thread_local`, justified by the fork model

The L2 cache is a `thread_local! { static CACHE: RefCell<Cache> }`
(`src/compiled/cache.rs:13-15`) `[verified-by-code]`, with the reasoning written
into the file: "PostgreSQL backends are single-threaded OS processes, so a
`thread_local` `RefCell` is sufficient — no mutex needed"
(`src/compiled/cache.rs:1-4`) `[from-comment]`. This is a correct reading of the
per-connection fork model (one backend = one process = one thread), but it is a
divergence in mechanism: core caches in palloc'd `MemoryContext`-owned structures
(`CacheMemoryContext`, syscache, the plan cache), not in Rust process-static
heap. The LRU's entries are never reset by a `MemoryContextReset`; they live for
the life of the backend (or until LRU eviction at capacity 128). Cross-ref
`[[knowledge/idioms/memory-contexts]]`, `.claude/skills/memory-contexts/SKILL.md`,
and the `[[knowledge/ideologies/pgrx]]` note on the fork model.

### 3. The L1 entry is freed by a `MemoryContextCallback`, not by pgrx

The most internals-heavy code in the extension: the L1 `FnExtraCache` is
`palloc`'d in `fn_mcxt` (`src/compiled/callsite.rs:94-99`) and registers a
`MemoryContextCallback` via `MemoryContextRegisterResetCallback` so that
`drop_fn_extra_cache` runs `std::ptr::drop_in_place` when `fn_mcxt` is reset
(`src/compiled/callsite.rs:21-29, 107-118`) `[verified-by-code]`. This is the
correct — and subtle — way to make a Rust `Drop` type (holding an `Arc` and a
`String`) coexist with PG's region allocator: palloc gives raw memory, the reset
callback gives the destructor a hook. The callback nulls `fn_extra` *before*
dropping so a re-entrant drop "sees a clean slate" (`src/compiled/callsite.rs:22-27`)
`[from-comment]`. Core's own `fn_extra` users are typically plain-C structs with
no destructor, so they never need this dance; a Rust extension that caches an
owning type *must*. Cross-ref `[[knowledge/idioms/memory-contexts]]`,
`[[knowledge/idioms/fmgr]]`.

### 4. Stability-aware cache reuse via `get_fn_expr_arg_stable`

L1 distinguishes a *stable* schema argument (a constant/immutable expression)
from a varying one by calling `pg_sys::get_fn_expr_arg_stable(flinfo, 0)`
(`src/compiled/callsite.rs:69`) `[verified-by-code]`. If the schema arg is stable
the slot is reused **unconditionally** — no string compare — because it cannot
change between rows; otherwise reuse requires
`cached.schema == schema.value` (`src/compiled/callsite.rs:61-66`)
`[verified-by-code]`. The "schema changed at the same callsite" refresh path
(`src/compiled/callsite.rs:71-92`) is what the `test_callsite_cache_refresh_on_schema_change`
test exercises (`src/lib.rs:283-300`) `[verified-by-code]`. This is a genuinely
planner-aware optimization most hand-written C functions skip — it reaches into
`fmgr`'s expression-context introspection rather than treating each call as
opaque.

### 5. The `jsonschema` type's on-disk form is the *canonical* schema string

`JsonSchema::compile` canonicalizes the value with
`jsonschema::canonical::json::to_string` before storing
(`src/compiled/mod.rs:34-39`) `[verified-by-code]`, and the type's `output`
writes that canonical string verbatim (`src/compiled/mod.rs:49-52`). So
`'{"b":1,"a":2}'::jsonschema::text` comes back `{"a":2,"b":1}`
(`src/lib.rs:202-207`, test) `[verified-by-code]`. Two consequences that lean on
core type-system machinery:

- The derive stack `#[derive(PostgresType, PostgresEq, PostgresHash, PartialEq,
  Eq, Hash, …)]` (`src/compiled/mod.rs:17-26`) means two schemas that canonicalize
  identically are **SQL-equal** and hash-equal — `test_jsonschema_equality`
  asserts `'{"type":"string","maxLength":5}'::jsonschema =
  '{"maxLength":5,"type":"string"}'::jsonschema` (`src/lib.rs:364-373`)
  `[verified-by-code]`. Canonicalization is what maximizes cache hits
  (`src/compiled/mod.rs:11-15`) `[from-comment]`.
- Unlike core `jsonb` (which stores a parsed binary tree), `jsonschema` stores a
  *string* and recompiles on read; the validator is the cached side-effect, not
  the stored datum. Cross-ref `[[knowledge/idioms/catalog-conventions]]`,
  `.claude/skills/catalog-conventions/SKILL.md`.

### 6. Errors are crate strings surfaced through pgrx, not native `ereport`

The extension never calls `ereport`/`errcode` directly. Three distinct
error behaviors, all divergent from core's SQLSTATE discipline:

- **Return `bool`/`text[]`, don't raise**: `*_matches_schema` returns `false` on a
  non-matching instance; `jsonschema_validation_errors` returns the crate's
  error strings as a `Vec<String>` (`src/lib.rs:30-40`) `[verified-by-code]`. The
  `CHECK`-constraint pattern relies on this — the *constraint* raises, the
  function just answers.
- **`notice!` on an invalid schema**: `jsonschema_is_valid` emits
  `notice!("Invalid JSON schema at path: {}", …)` and returns `false`
  (`src/lib.rs:19-28`) `[verified-by-code]` — a `NOTICE`, not an error.
- **`pgrx::error!` (which longjmps) on a malformed cast input**: the `jsonschema`
  type's `input` does `serde_json::from_slice(...).unwrap_or_else(|err|
  pgrx::error!("invalid JSON: {err}"))` (`src/compiled/mod.rs:43-47`), and
  `compile` raises on a canonicalization or compilation failure
  (`src/compiled/mod.rs:35-37, 54-58`) `[verified-by-code]`. So casting a bad
  literal to `jsonschema` *does* raise, with the raw crate message and no
  SQLSTATE chosen by the author — pgrx maps `error!` onto an `ereport(ERROR)`
  with a generic code. Cross-ref `[[knowledge/idioms/error-handling]]`,
  `.claude/skills/error-handling/SKILL.md`.

## Notable design decisions (cited)

- **All functions are `immutable, strict, parallel_safe`** (`src/lib.rs:9, 14,
  19, 30, 42, 47, 64, 74, 84, 97`) `[verified-by-code]`. `IMMUTABLE` is the
  right label *only because* validation is a pure function of (schema, instance):
  the cache is a transparent memo, so the same inputs always give the same
  answer regardless of cache state. The compiled functions are `IMMUTABLE`
  despite mutating `fn_extra` and the LRU — defensible, since those are caches,
  but it is a claim the author is making, not one the runtime checks. `STRICT`
  means a NULL schema or instance short-circuits to NULL before the body runs.
- **`parallel_safe` with a process-local cache**: each parallel worker is its own
  backend/process, so each has its own `thread_local` LRU and its own `fn_extra`;
  there is no shared mutable state to make parallel evaluation unsafe
  (`src/compiled/cache.rs:1-15`) `[inferred]`. The cost is cache duplication
  across workers, not incorrectness.
- **Capacity-128 LRU, no time-based eviction** (`src/compiled/cache.rs:11`)
  `[verified-by-code]` — a fixed bound chosen without configurability (no GUC).
- **`superuser = true`, `relocatable = false`** in the control file
  (`pg_jsonschema.control:4-5`) `[verified-by-code]` — install is superuser-only;
  the extension's objects are not schema-relocatable.
- **The base and compiled tiers share one canonical-string cache key**
  (`get_or_compile` in `src/compiled/callsite.rs:7-9` and `get_or_insert` in
  `src/compiled/cache.rs:18-30`), so a schema compiled via the type's `input`
  populates the same L2 LRU a later compiled-function call reads
  `[verified-by-code]`.

## Anthropology takeaway

pg_jsonschema is two extensions stacked. The bottom one is the textbook pgrx
case: four `IMMUTABLE STRICT PARALLEL SAFE` functions that forward jsonb to a
Rust crate and return `bool`/`text[]`, raising nothing (the `CHECK` constraint
does the raising). The top one — the `src/compiled/` tier — is the sharp
divergence: an extension growing its *own* memoization layer for a per-tuple
function, something core deliberately does not provide for jsonb. That layer is
where the interesting bets live. (a) **Lifetime/longevity**: the L2 validators
sit in a process-static `thread_local` LRU that no `MemoryContextReset` ever
touches — they leak by design until LRU eviction or backend exit, justified by
the fork model. (b) **Rust-`Drop`-meets-palloc**: the L1 entry is the cleanest
corpus example of using `MemoryContextRegisterResetCallback` to give a region
allocator a destructor hook, a problem core's plain-C `fn_extra` users never
have. (c) **The `IMMUTABLE` label is load-bearing and unchecked**: the compiled
functions mutate `fn_extra` and the LRU on every call yet claim immutability;
the claim holds only because the author *asserts* the cache is a transparent
memo — a good cautionary entry for a `knowledge/issues` note on
"IMMUTABLE-but-stateful" functions, the same family of concern that
`[[knowledge/ideologies/uuidv47]]` raises from the opposite direction (an I/O
function that is labelled-or-assumed immutable but actually depends on session
state). pg_jsonschema gets it *right* where uuidv47 stresses it, which makes the
pair a useful contrast.
