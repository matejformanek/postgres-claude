# pg_variables — a session's own typed, optionally-transactional key/value store, living in backend-local hash tables that core Postgres deliberately never gives you

> Produced by `pg-extension-anthropologist`. Repo: `postgrespro/pg_variables` @ `master`.
> All `file:line` cites point into that repo (raw.githubusercontent.com), **not** `source/`.
> Fetched 2026-07-16.

## Domain & purpose

pg_variables hands a single backend session a namespaced set of **typed, named,
in-memory variables** — scalars (int/text/numeric/timestamp/date/jsonb/any/array)
*and* whole collections of `record`s keyed by a primary key — that outlive the
statement and, by default, outlive `BEGIN`/`COMMIT`/`ROLLBACK` too. Variables are
grouped into *packages* (`pgv_set('vars', 'int1', 101)` → package `vars`, variable
`int1`), and an empty package auto-deletes. [from-README] (`README.md:9-43`)

The point is state that core Postgres has no first-class home for. GUCs are
text-only, flat, and not collections; temporary tables are real relations —
catalog rows, MVCC, a `pg_class` entry, VACUUM — far too heavy to use as a scratch
variable, and MVCC-bound rather than plainly mutable. pg_variables sits in that
gap: mutable session-scoped storage that is *not* a relation, *not* in any catalog,
*not* MVCC-versioned, and *not* shared across backends. [inferred]

The control file names the domain exactly — `comment = 'session variables with
various types'`, `default_version = '1.2'`, `relocatable = true`
(`pg_variables.control:2-5`). [verified-by-code]

## How it hooks into PG

It is almost entirely a set of **SQL-callable C functions** — `PG_MODULE_MAGIC`
(`pg_variables.c:29`) plus a block of `PG_FUNCTION_INFO_V1` declarations
(`pg_variables.c:32-47`): record ops `variable_insert` / `variable_update` /
`variable_delete` / `variable_select` (+ `_by_value` / `_by_values`), and package
ops `variable_exists` / `package_exists` / `remove_variable` / `remove_package` /
`remove_packages` / `get_packages_and_variables` / `get_packages_stats`.
[verified-by-code]

The scalar setters/getters are generated by two macros rather than hand-written per
type: `VARIABLE_SET_TEMPLATE(type, typid)` (`pg_variables.c:694`) and
`VARIABLE_GET_TEMPLATE(...)` (`pg_variables.c:651`). The **deprecated** typed
entry points (`variable_set_int`/`variable_get_text`/…) pin the OID at compile time
— `INT4OID`, `TEXTOID`, `NUMERICOID`, `TIMESTAMPOID`, `TIMESTAMPTZOID`, `DATEOID`,
`JSONBOID` (`pg_variables.c:720-726`, `681-687`). The **current** polymorphic API
(`pgv_set`/`pgv_get` over `anynonarray`/`anyarray`) resolves the real type at call
time via `get_fn_expr_argtype(fcinfo->flinfo, 2)` (`pg_variables.c:690-691`,
`729-730`). [verified-by-code]

`_PG_init` (`pg_variables.c:3080`) does only three things: registers one
user-settable GUC `pg_variables.convert_unknownoid` via `DefineCustomBoolVariable`
at `PGC_USERSET` (`pg_variables.c:3082-3091`); registers the transaction and
subtransaction callbacks (`RegisterXactCallback`/`RegisterSubXactCallback`,
`pg_variables.c:3096-3098`); and chains the executor-end hook
(`prev_ExecutorEnd = ExecutorEnd_hook; ExecutorEnd_hook = variable_ExecutorEnd`,
`pg_variables.c:3101-3102`). There is **no** `shared_preload_libraries` requirement,
no bgworker, no shmem request — the module is loaded on demand and its whole world
is process-local. `_PG_fini` (PG < 15) unregisters the callbacks and restores the
hook (`pg_variables.c:3109-3115`). [verified-by-code]

**Record variables** ride on the type cache and tuple infrastructure. `pgv_insert`
(`variable_insert`, `pg_variables.c:733`) reads the row's type via
`HeapTupleHeaderGetTypeId`/`GetTypMod`, resolves the descriptor with
`lookup_rowtype_tupdesc` (`pg_variables.c:813-817`), then on first insert calls
`init_record(record, tupdesc, variable)` (`pg_variables.c:832`) and on subsequent
inserts `check_attributes(...)` to enforce a stable shape (`pg_variables.c:843`),
finally `insert_record` (`pg_variables.c:847`). A `RecordVar` holds its own hash
table `rhash`, the `TupleDesc`, a dedicated `MemoryContext hctx`, and cached
`hash_proc`/`cmp_proc` FmgrInfos for the primary-key column (`pg_variables.h:36-46`);
lookups build a `HashRecordKey` carrying pointers to those procs
(`pg_variables.h:144-158`, used at `pg_variables.c:1144-1145`). The record-layer
functions themselves (`init_record`, `insert_record`, `update_record`,
`delete_record`, `check_attributes`, `coerce_unknown_first_record`) are declared
`extern` in the header (`pg_variables.h:186-195`) but are **not** defined in
`pg_variables.c`; they live in a companion translation unit outside this manifest.
[inferred]

`pgv_select` is a set-returning function: `variable_select` (`pg_variables.c:1018`)
runs `hash_seq_init`/`hash_seq_search` over `record->rhash` across `SRF` per-call
invocations (`pg_variables.c:1052`, `1091`). Because a caller may abandon the scan
mid-stream (cursor / `LIMIT`), the extension tracks every live `hash_seq_search` so
it can `hash_seq_term` them later — see the divergence section. [verified-by-code]

## Where it diverges from core idioms

**All mutable state lives in the extension's own backend-local hash tables under a
dedicated long-lived MemoryContext — nothing touches shared memory, the catalog, or
MVCC.** The root is a single static `HTAB *packagesHash` and `MemoryContext
ModuleContext` (`pg_variables.c:128-129`). `ensurePackagesHashExists` creates
`ModuleContext` as a child of **`CacheMemoryContext`** — a backend-lifetime context —
and builds `packagesHash` inside it, keyed by `NAMEDATALEN` with `sizeof(Package)`
entries (`pg_variables.c:1802-1816`). Each package then owns *two* variable hash
tables and *two* sub-contexts: `varHashRegular`/`hctxRegular` and
`varHashTransact`/`hctxTransact` (`pg_variables.h:110-121`), built by `makePackHTAB`
as children of `ModuleContext` (`pg_variables.c:1830-1847`). Scalar values are
`datumCopy`'d into the owning package context so they persist independently of the
caller's short-lived memory (`pg_variables.c:613-614`). Because `packagesHash` is a
plain process-local `static` rooted in `CacheMemoryContext`, **every backend has its
own private universe of packages** — there is no cross-backend visibility and no
locking, unlike a temp table (visible to its creating backend but a genuine relation)
or a GUC (per-session but text and flat). [verified-by-code]

**Non-transactional is the default, and non-transactional variables survive
`ROLLBACK`.** `pgv_set(..., is_transactional => false)` is the default
(`README.md:9-25`). A regular variable is stored in `varHashRegular` and is simply
never enrolled in the transaction machinery, so `BEGIN … ROLLBACK` leaves it intact
— the README demonstrates `int2` set inside an aborted transaction persisting
(`README.md:15-25`). This is the deliberate inversion of every core storage idiom:
the storage engine's whole job is to make writes disappear on rollback; here the
*default* is that they do not. [verified-by-code/from-README]

**Transactional visibility is re-implemented by hand, outside the storage engine,
via a stack of value snapshots plus xact/subxact callbacks.** Opt in with
`is_transactional => true` and the value goes into `varHashTransact`. Each
transactional object (package or variable) is a `TransObject` carrying a `dlist_head
states` history of `TransState` snapshots (`pg_variables.h:65-96`); `GetActualState`
is just the head of that list (`pg_variables.h:198-205`). On the first mutation
within a transaction level, `createSavepoint` pushes a fresh `VarState` (allocated in
`hctxTransact`) and `copyValue` clones the previous value forward
(`pg_variables.c:2299-2322`, `copyValue` at `pg_variables.c:2143-2179`, cloning
scalars with `datumCopy` and records by re-`init_record` + copy). A per-subxact-level
**changesStack** (`pg_variables.c:141`) records which objects changed at each depth;
`pushChangesStack` creates a new stack node with its own `AllocSetContext` when a
subtransaction starts (`pg_variables.c:2563-2599`). Commit/rollback then walk that
list: `pgvTransCallback` maps `XACT_EVENT_PRE_COMMIT → RELEASE_SAVEPOINT` and
`XACT_EVENT_ABORT → ROLLBACK_TO_SAVEPOINT` (`pg_variables.c:2982-3005`), and
`pgvSubTransCallback` maps `SUBXACT_EVENT_START_SUB → pushChangesStack`,
`COMMIT_SUB → RELEASE`, `ABORT_SUB → ROLLBACK` (`pg_variables.c:2946-2969`).
`rollbackSavepoint` pops a state (or removes the object entirely if it was created
inside the rolled-back level — matching the README's "variable will not exist"
case), `releaseSavepoint` collapses the child level's state into its parent
(`pg_variables.c:2336-2410`, `2416-…`). This is transaction-scoped versioning
rebuilt on linked lists and memory contexts — a private MVCC for a handful of
variables, entirely above the AM layer. [verified-by-code]

**Memory lifetime & cleanup on backend exit.** Deleting is context-deletion, not
per-datum frees: `freeValue` drops a whole record by `MemoryContextDelete(record.hctx)`
or `pfree`s a by-ref scalar (`pg_variables.c:2181-2193`); `removePackageInternal`
deletes `hctxRegular` outright (`pg_variables.c:1435-1440`). When the last package
disappears, `processChanges` tears down the whole thing —
`MemoryContextDelete(ModuleContext); packagesHash = NULL; ModuleContext = NULL`
(`pg_variables.c:2735-2743`). There is **no** `on_proc_exit`/`before_shmem_exit`
registration anywhere; final backend-exit cleanup is implicit — `ModuleContext`
hangs under `CacheMemoryContext` and is reclaimed by normal process/context teardown
when the backend dies (matching the per-connection fork model, where each forked
backend simply takes its private heap to the grave). [inferred]

**`pgv_free()` respects rollback for transactional vars but not regular ones.** The
comment on `remove_packages` says "Memory context will be released after committing"
(`pg_variables.c:1477-1509`); `removePackageInternal` frees `hctxRegular` *immediately*
but only *marks* transactional state invalid, so a `ROLLBACK` after `pgv_free()`
resurrects transactional variables while regular ones are gone for good — exactly the
asymmetry the README calls out (`README.md:368-392`). [verified-by-code/from-README]

## Notable design decisions

- **Rooted in `CacheMemoryContext`, not `TopMemoryContext`.** `ModuleContext =
  AllocSetContextCreate(CacheMemoryContext, PGV_MCXT_MAIN, …)` — package/variable
  storage is backend-lifetime but survives statement and transaction resets.
  (`pg_variables.c:1802-1804`) [verified-by-code]
- **Two hash tables + two contexts per package** cleanly separate the
  rollback-exempt regular variables from the snapshot-tracked transactional ones, so
  the transaction callbacks only ever traverse `varHashTransact`.
  (`pg_variables.h:110-121`; `pg_variables.c:1830-1847`) [verified-by-code]
- **A "reverse check" makes transactionality immutable per name.** Creating a
  variable searches the *opposite* table first and errors with `variable "%s"
  already created as %sTRANSACTIONAL` if found — you cannot flip a variable's
  transactional flag after creation. (`pg_variables.c:2062-2071`) [verified-by-code]
- **Type identity is enforced on every access**, not just creation:
  `getVariableInternal` compares `variable->typid` to the requested OID and raises
  `variable "%s" requires "%s" value` via `regtypeout` (`pg_variables.c:2007-2017`);
  record/scalar mismatch raises `"%s" isn't a %s variable` (`pg_variables.c:2019-2023`).
  [verified-by-code]
- **Names are `NAMEDATALEN`-bounded keys.** `getKeyFromName` rejects names ≥
  `NAMEDATALEN-1` with `name "%.*s" is too long`, mirroring core identifier limits.
  (`pg_variables.c:1780-1791`) [verified-by-code]
- **Live `hash_seq_search` scans are tracked and force-terminated** to avoid leaking
  scan slots when an SRF consumer stops early. `variables_stats`/`packages_stats`
  lists live in `TopTransactionContext` (`pg_variables.c:144-160`); `freeStatsLists`
  `hash_seq_term`s every entry at query end and at pre-commit/abort
  (`pg_variables.c:3044-3074`, called from `variable_ExecutorEnd` at
  `pg_variables.c:3038` and `pgvTransCallback` at `pg_variables.c:3007-3008`).
  [verified-by-code]
- **`pgv_stats()` reports real allocated bytes** by walking the package's
  `MemoryContext` tree via the context `stats` method (`getMemoryTotalSpace`,
  `pg_variables.c:1643-1671`; summed per package at `pg_variables.c:1754-1760`) —
  memory accounting the storage engine gives relations for free but variables would
  otherwise lack. [verified-by-code]
- **`LastPackage`/`LastVariable` are a one-slot lookup cache**, invalidated by
  `resetVariablesCache` on any structural change (`pg_variables.c:132-135`,
  `1469-1475`). [verified-by-code]
- **Declared incompatible with connection pooling / autonomous transactions.**
  Under the Postgres Pro EE build, `compatibility_check` errors out
  `pg_variables extension is incompatible with connection pooling` when the backend
  is not dedicated — because session-local state is meaningless if the session is
  shared (`pg_variables.c:2749-2773`). [verified-by-code]

## Links into corpus

- [[orafce]] — the closest sibling in spirit: an Oracle-compatibility layer whose
  `DBMS_*` packages also keep per-session package state and even a shared-memory
  message bus. pg_variables is the same "give a session first-class named state"
  urge, minus the Oracle surface and minus any shmem — strictly backend-local.
- [[pgtt]] — session-scoped Global Temporary Tables: the *relational* answer to the
  same "I want per-session scratch state" need. Reading the two side by side sharpens
  pg_variables' bet: pgtt pays the full relation/MVCC/catalog cost to get SQL-native
  tables; pg_variables refuses that cost and lives in a private hash table instead.

Core concepts referenced in prose without wiki links: GUCs (`PGC_USERSET` text
variables), `MemoryContext` lifetimes (`CacheMemoryContext`), temp tables as real
MVCC relations, and the transaction/subtransaction callback API
(`RegisterXactCallback`/`RegisterSubXactCallback`). This meta-repo's own
session-variables (`@x` sesvars) feature is a third point of contrast: sesvars wires
typed session variables *into the grammar and PARAM machinery* of core, whereas
pg_variables bolts equivalent state on from outside as plain SQL functions over a
private hash table — same goal, opposite integration depth.
