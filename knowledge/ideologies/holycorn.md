# holycorn ‚Äî a Foreign Data Wrapper whose per-scan row source is a user-written mruby class executed inside the backend

- **Repo**: `franckverrot/holycorn`, branch `master`, LGPLv3, author Franck Verrot (`@franckverrot`) [from-README] (README.md:304-315).
- **Fetched**: `README.md` (314 lines), `holycorn.c` (576 lines, the sole C file, at repo ROOT not `src/`), `holycorn.control` (5 lines), `Makefile` (13 lines), `holycorn--1.0.sql` (14 lines), `Rakefile` (31 lines). `src/` returns 404; layout is flat.

## Domain & purpose

holycorn is an FDW *framework*, not a wrapper for one fixed remote. Where `postgres_fdw`/`mongo_fdw` hard-code a specific backend and marshal a known wire protocol, holycorn delegates row production to a **user-supplied Ruby class**: you write a class that answers `.new(env)` and whose instance answers `.each`, and holycorn embeds an mruby VM in the scan loop to call it (README.md:213-214, holycorn.c:289,321). The class is named either by `wrapper_class` (a class compiled into the linked mruby build) or `wrapper_path` (a `.rb` file loaded at scan start) (README.md:279-283, holycorn.c:256-262). It is the **Ruby cousin of multicorn** (the Python FDW framework ‚Äî see [[knowledge/ideologies/multicorn]]): same "scriptable FDW" idea, but embedding mruby rather than CPython, and choosing mruby explicitly for its sandboxing over MRI/CRuby (README.md:7-8). Ships one built-in wrapper, `HolycornRedis` via the `mruby-redis` gem (README.md:19-22).

## How it hooks into PG

- `holycorn_handler` returns a `FdwRoutine` (holycorn.c:73-93) filling: `GetForeignRelSize`=`rbGetForeignRelSize`, `GetForeignPaths`=`rbGetForeignPaths`, `GetForeignPlan`=`rbGetForeignPlan`, `BeginForeignScan`=`rbBeginForeignScan`, `IterateForeignScan`=`rbIterateForeignScan`, `EndForeignScan`=`rbEndForeignScan`, `ExplainForeignScan`=`rbExplainForeignScan`, `ReScanForeignScan`=`fileReScanForeignScan`, and (guarded `PG_VERSION_NUM >= 90500`) `ImportForeignSchema`=`rbImportForeignSchema` (holycorn.c:76-90). This is the standard scan-only `FdwRoutine` shape ‚Äî see [[knowledge/idioms/fdw-routine-callbacks]].
- `.control`: `comment = 'Ruby foreign-data wrapper provider'`, `default_version = '1.0'`, `relocatable = true` (holycorn.control:2-5). Install SQL declares `holycorn_handler()`/`holycorn_validator(text[],oid)` and the `FOREIGN DATA WRAPPER holycorn` (holycorn--1.0.sql:3-14).
- **Option naming**: the foreign table's `OPTIONS` carry `wrapper_path` XOR `wrapper_class`; all *other* options are pushed down verbatim into the Ruby constructor as a hash (holycorn.c:136-172, README.md:284-286). `rbGetOptions` strips the two reserved keys and returns the rest as `other_options` (holycorn.c:153-171).
- **Where mruby is invoked**: the VM is opened and the class instantiated in `BeginForeignScan` (holycorn.c:252-289); each `IterateForeignScan` call invokes `iterator.each` once to fetch the next row (holycorn.c:321). See [[knowledge/idioms/fdw-iterate-scan]].

## Where it diverges from core idioms

THE LOAD-BEARING STORY: **an mruby interpreter is embedded in the backend process and driven inside the FDW scan loop ‚Äî arbitrary user Ruby executes in-backend, one VM per scan, with essentially no firewall between Ruby's failure modes and PG's.**

1. **VM lifecycle ‚Äî per-scan, never cached.** `mrb_open()` runs in `rbBeginForeignScan` (holycorn.c:252) and `mrb_close()` in `rbEndForeignScan` (holycorn.c:459). One fresh interpreter is created and torn down for every scan node; nothing is cached across scans, rescans, or backends (`fileReScanForeignScan` is an empty stub, holycorn.c:454-455, so a re-scan does NOT reset the Ruby iterator ‚Äî a correctness gap [inferred]). `ImportForeignSchema` opens its own separate short-lived VM (holycorn.c:506,459-equivalent path).
2. **Ruby‚ÜíDatum marshalling boundary.** `rbIterateForeignScan` expects `each` to return an array (one element per column) (holycorn.c:321-327). It then `switch`es on `mrb_type(column)` (holycorn.c:343-446): `MRB_TT_FIXNUM`‚Üí`Int64GetDatum` (:357-361), `MRB_TT_TRUE/FALSE`‚Üí`BoolGetDatum` (:347-356), `MRB_TT_FLOAT`‚Üí`Float8GetDatum(mrb_flo_to_fixnum(...))` (:368-372 ‚Äî note it truncates the float through `mrb_flo_to_fixnum` before boxing, an apparent bug [inferred]), `MRB_TT_STRING/SYMBOL/OBJECT/CLASS/MODULE/PROC/ARRAY/HASH/RANGE/FIBER`‚Üía C-string Datum via the `StringDatumFromChars` macro (:309-312), and `MRB_TT_DATA`‚Üí`handle_column` (:433-434). `handle_column` special-cases Ruby `Time`‚Üí`timestamptz` by converting to unix epoch usecs minus `POSTGRES_TO_UNIX_EPOCH_USECS` (holycorn.c:485-491). **Critically, the marshaller never consults the foreign table's declared column type-OIDs** ‚Äî it maps whatever Ruby type it sees onto a fixed PG representation, and `StringDatumFromChars` stores a raw `CStringGetDatum` (with a bizarre `"s%s\0"` literal-`s` prefix, holycorn.c:311) straight into `slot->tts_values[i]`, not a `text` varlena (holycorn.c:309-312) [inferred: type-unsafe / loose].
3. **Error / exception boundary ‚Äî a firewall only at construction, none during iteration.** After `mrb_funcall(..., "new", ...)` holycorn explicitly checks `mrb_exception_p` and turns an mruby exception into `elog(ERROR, ...)` with the Ruby `inspect` string (holycorn.c:289-299). But the hot path ‚Äî `mrb_funcall(..., "each", ...)` in `rbIterateForeignScan` (holycorn.c:321) ‚Äî has **no** exception check: a raised mruby exception is not caught, not converted to `ereport`, and there is no `PG_TRY`/`PG_CATCH` anywhere in the file. This is markedly weaker than the C-exception *firewalls* documented for [[knowledge/ideologies/pgrouting]] (C++ `try/catch`‚Üí`ereport`) and [[knowledge/ideologies/multicorn]] (Python error ‚Üí PG error translation): holycorn firewalls instantiation but leaves the row loop unguarded [verified-by-code, holycorn.c:304-452].
4. **Memory ownership ‚Äî two heaps, no unification.** Ruby objects live on mruby's own GC heap (owned by the `mrb_state`, freed en masse at `mrb_close`, holycorn.c:459), entirely outside PG's `palloc`/MemoryContext world (see [[knowledge/idioms/memory-contexts]]). On the PG side every `IterateForeignScan` `palloc`s fresh `tts_isnull`/`tts_values` arrays *and* a per-string output buffer (holycorn.c:331-332,310) into the surrounding executor context ‚Äî re-allocated on every single row with no `pfree`, so long scans accumulate garbage until the query context is reset [inferred: leak].
5. **Trust / security surface.** Arbitrary user Ruby (`wrapper_path` points at any readable `.rb`, `fopen`+`mrb_load_file`, holycorn.c:256-259) runs *in the backend process* with the backend's OS privileges. mruby is chosen precisely because it sandboxes more than MRI (README.md:7-8), but a `wrapper_path` still lets any table-definer execute code in-process [inferred from README.md:7-8 + holycorn.c:256-259].
6. **Static libmruby linkage.** `SHLIB_LINK` links `vendor/mruby/build/i686-pc-linux-gnu/lib/libmruby.a` (a static archive) plus the static `libhiredis.a` from the bundled `mruby-redis` gem (Makefile:6). mruby is vendored and built out-of-tree first (`rake` ‚Üí `vendor_mruby`+`build_mruby`, Rakefile:5-19) before `make` builds the `.so` [verified-by-code].

## Notable design decisions

- **Static, vendored VM.** mruby + gems are `git clone --depth=1`'d into `vendor/` and compiled to `.a` archives, then statically linked ‚Äî the whole interpreter is baked into `holycorn.so` (Makefile:6, Rakefile:5-19).
- **Option parsing / script resolution.** `wrapper_path` XOR `wrapper_class` is enforced in three places ‚Äî the validator (holycorn.c:129-131), `rbGetOptions` (holycorn.c:167-169), and the import path (holycorn.c:548-550). `wrapper_class` must be a single word already compiled into the mruby build (README.md:222-223); `wrapper_path` is `fopen`+`mrb_load_file` at scan begin (holycorn.c:256-259). All non-reserved options plus PG build constants (`PG_VERSION`, `PG_VERSION_NUM`, `PACKAGE_STRING`, ‚Ä¶) are handed to `.new` as a hash (holycorn.c:264-289, README.md:228-236).
- **Read-only, no writes.** No `ExecForeignInsert`/`Update`/`Delete` in the routine (holycorn.c:76-90); README confirms "doesn't support INSERTs yet" (README.md:104).
- **No qual pushdown.** `GetForeignRelSize` only reads options (holycorn.c:174-179); `GetForeignPlan` runs `extract_actual_clauses` and attaches quals for *local* recheck but pushes nothing to Ruby (holycorn.c:222-239). Costs are a fixed, deliberately-expensive fudge (`cpu_tuple_cost * 100`, holycorn.c:471-474). `ExplainForeignScan` is empty (holycorn.c:242-243). The Ruby class always yields its full row set; PG filters afterward [verified-by-code].
- **`IMPORT FOREIGN SCHEMA`** delegates DDL generation to the Ruby class's `import_schema` method, requiring the magic remote schema name `holycorn_schema` (holycorn.c:501-575, README.md:95-99).
- **Pre-PG12 FDW/slot API [inferred/from-comment drift].** The code directly writes `slot->tts_isempty`, `slot->tts_nvalid`, and reallocates `tts_isnull`/`tts_values` (holycorn.c:328-332) ‚Äî the *concrete* `TupleTableSlot` layout removed by the PG12 abstract-slot refactor. `list_delete_cell(options, lc, prev)` uses the 3-arg signature dropped in PG13 (holycorn.c:158,161). Version guards top out at `PG_VERSION_NUM >= 90600` (holycorn.c:196) and README targets "PostgreSQL 9.4+" (README.md:28). It will not compile unmodified against modern PG [inferred].
