## Domain & purpose

pgvectorscale implements **StreamingDiskANN**, "a novel index type … inspired by
the DiskANN algorithm, based on research from Microsoft" (`README.md`)
`[from-README]`, registered as an index access method named `diskann`
(`CREATE INDEX … USING diskann (embedding vector_cosine_ops)`, `README.md`)
`[from-README]`. It does **not** replace pgvector; it *requires* it
(`requires = 'vector'`, `pgvectorscale/vectorscale.control:6`) `[verified-by-code]`
and indexes pgvector's `vector` column type, complementing pgvector's own HNSW/IVFFlat
AMs for large-scale ANN. The reason to document it: it is the corpus's cleanest example
of a **complete, durable index access method implemented in Rust via pgrx** — with its
own on-disk graph page format, its own buffer/WAL plumbing wrapped in Rust RAII, and a
"streaming" (insert-supporting, resumable) graph search — every layer of which is a
deliberate re-expression of a core C subsystem in another language and memory model.

## How it hooks into PG

A pgrx extension. `pgrx::pg_module_magic!()` (`src/lib.rs:3`) stands in for
`PG_MODULE_MAGIC`; submodules are `pub mod access_method; mod util;` (`src/lib.rs:5-6`)
`[verified-by-code]`. `_PG_init` is a thin `extern "C-unwind"` shim that initializes
three submodules — distance, options, GUC (`src/lib.rs:14-19`) `[verified-by-code]`:

```
access_method::distance::init();
access_method::options::init();
access_method::guc::init();
```

- **AM registration** is done with pgrx `#[pg_extern]` + `extension_sql`: a
  `diskann_amhandler(internal) RETURNS index_am_handler` function
  (`src/access_method/mod.rs:35-38`) and an idempotent
  `CREATE ACCESS METHOD diskann TYPE INDEX HANDLER diskann_amhandler`
  (`mod.rs:42-46`) `[verified-by-code]`.
- **`IndexAmRoutine`** is filled in the handler: `ambuild = Some(build::ambuild)`,
  `aminsert = Some(build::aminsert)`, `ambeginscan/amgettuple/amendscan = Some(scan::…)`
  (`mod.rs:67-77`) `[verified-by-code]`. Capability flags include
  `amcanorderbyop = true`, `amcanmulticol = true`, `amoptionalkey = true`,
  `amcanparallel = false` (`mod.rs:57-64`) `[verified-by-code]` — i.e. it is an
  ORDER BY <op> (nearest-neighbour) AM, not a WHERE-qual AM.
- **GUCs** (query- and build-time) are defined in `src/access_method/guc.rs`:
  `diskann.query_search_list_size` (default 100), `diskann.query_rescore` (default 50),
  and parallel-build knobs `diskann.parallel_flush_interval`,
  `diskann.parallel_initial_start_nodes_count`,
  `diskann.min_vectors_for_parallel_build`, `diskann.force_parallel_workers`
  (`guc.rs:8-69`) `[verified-by-code]`.
- **Opclasses / operators** are registered via `extension_sql!` (`mod.rs:139`),
  binding `vector_cosine_ops` / `vector_l2_ops` / `vector_ip_ops` (`<=>`, `<->`, `<#>`)
  `[from-README]`.

Cross-ref `[[knowledge/architecture/access-methods]]`,
`.claude/skills/access-method-apis/SKILL.md`.

## Where it diverges from core idioms

### 1. The index AM is Rust-via-pgrx, not C

Core index AMs (nbtree, GIN, GiST, BRIN, hash) are C, registering an
`IndexAmRoutine*` from an `amhandler` returning `internal`. pgvectorscale does the same
*shape* but the callbacks are Rust `extern "C-unwind"` functions
(`build::ambuild`, `build::aminsert`, `scan::ambeginscan`/`amgettuple`/`amendscan`,
wired in `mod.rs:67-77`) `[verified-by-code]`, and the SQL glue is generated by pgrx
macros (`#[pg_extern]`, `extension_sql`) rather than hand-written `.sql`. The
contrast partner is `[[knowledge/ideologies/zombodb]]` / `[[knowledge/ideologies/paradedb]]`,
which also implement index AMs that offload the actual indexing — but pgvectorscale
keeps the graph **inside PG pages** (next point), so it is a more faithful, if
foreign-language, re-implementation of the core AM contract.

### 2. Its own on-disk page format: a "Tape" of graph nodes, not nbtree/GIN pages

There is no B-tree. The DiskANN graph is laid out by a bespoke `Tape` abstraction —
"a memory abstraction over Postgres pages" (`src/access_method/build.rs:54`)
`[from-comment]` — that appends variable-size nodes across relation pages, starting a
new `WritablePage` when the current page lacks aligned free space
(`src/util/tape.rs:53-73`) `[verified-by-code]`. Each page carries a custom special
area `TsvPageOpaqueData` with a magic `TSV_PAGE_ID` (0xAE24) and a `PageType` enum
(MetaV1 … Meta, plus node/neighbor page types) in the opaque region
(`src/util/page.rs:29-37, 66-72`); pages are set up with `PageInit` reserving room for
that opaque struct (`page.rs:99`) `[verified-by-code]`. The `Storage` trait abstracts
node persistence — `create_node()` writes a vector node to a `Tape` and returns an
`ItemPointer`; `set_neighbors_on_disk()` / `get_neighbors_with_distances_from_disk()`
persist and read the adjacency lists (`src/access_method/storage.rs:50-58, 100-112`)
`[verified-by-code]`. So node identity is a `(block, offset)` `ItemPointer` into the
index relation — PG's own TID-shaped addressing — but the *content* of a page is a
DiskANN adjacency graph, nothing the core AMs would recognise.

### 3. It reuses pgvector's `vector` by reinterpreting its varlena layout, not by linking

It requires the `vector` extension and indexes `vector` columns, but it does **not**
call into pgvector's C symbols. Instead it declares a `#[repr(C)]`
`PgVectorInternal { vl_len_: i32, dim: i16, unused, x: __IncompleteArrayField<c_float> }`
that mirrors pgvector's varlena+dim+flex-array struct, detoasts the incoming `Datum`,
and casts the pointer to its own struct (`src/access_method/pg_vector.rs:7-14, 119-145`)
`[verified-by-code]`. Vectors arrive through the AM callback args as
`LabeledVector::from_datums(values, isnull, &meta_page)` (`build.rs:457, 498`) and
`PgVector::from_pg_parts(values, isnull, 0, …)` (`build.rs:595`) `[verified-by-code]`.
This is a hard ABI coupling: pgvectorscale's correctness depends on pgvector's on-disk
`Vector` struct not changing — a dependency the core AMs never take on another
extension's type.

### 4. Memory model: Rust heap + `Vec` for the build graph, wrapped in a PG MemoryContext

Core AMs build inside `palloc`'d memory under a per-build `MemoryContext`. pgvectorscale
straddles both: the `BuildState` holds a `PgMemoryContexts` ("diskann build context",
`build.rs:70, 81`) that it `set_as_current()` / `reset()`s per heap tuple
(`build.rs:110-111`) `[verified-by-code]` — but the graph neighbour cache itself is a
Rust `BuilderNeighborCache` sized to a fraction of `maintenance_work_mem`
(`BUILDER_NEIGHBOR_CACHE_SIZE = 0.8`, `build.rs:538`) holding data in Rust `Vec`s
(`graph` constructed at `build.rs:281, 379`) `[verified-by-code]`. So the hot build
state lives on the **Rust heap**, outside PG's context accounting, with the PG context
used mainly to scope per-tuple detoast/temporaries. `maintenance_work_mem` is honoured
by convention (the cache fraction), not enforced by `MemoryContext` limits
`[inferred]`.

### 5. Durability: it WAL-logs via Generic WAL, with Rust RAII as the abort guard

Unlike many "index AM abuse" extensions that punt on crash safety, pgvectorscale
**does** WAL-log page edits — through PG's *Generic WAL* facility, not a custom rmgr.
`WritablePage` calls `GenericXLogStart(index)` then `GenericXLogRegisterBuffer(state, buffer, 0)`
on construction (`src/util/page.rs:87-91`), `MarkBufferDirty` + `GenericXLogFinish` on
commit (`page.rs:226-227`), and crucially the Rust `Drop` impl calls
`GenericXLogAbort()` if a `WritablePage` is dropped without committing
(`page.rs:229-237`) `[verified-by-code]`. That last point is a genuinely Rust-idiomatic
twist on a C invariant: the "every started xlog is finished or aborted" discipline is
enforced by the destructor rather than by careful `PG_TRY`/goto cleanup. Generic WAL
means no bespoke redo function is needed (page-delta replay is generic). Cross-ref
`.claude/skills/wal-and-xlog/SKILL.md`. Caveat: `CREATE INDEX` on an `UNLOGGED` table
is unsupported (`README.md`) `[from-README]`.

### 6. "Streaming" = real `aminsert` + a resumable graph search; concurrency leans on buffer pins

`aminsert` is implemented (not a stub) — single vectors are inserted into the live graph
(`build.rs:343-352`), while `ambuildempty` is an explicit `not yet implemented` panic
(`build.rs:540-542`) `[verified-by-code]`. Scan state is a `TSVResponseIterator` that
holds resumable DiskANN traversal progress (`lsr: ListSearchResult`, a `search_list_size`,
and a `resort_buffer: BinaryHeap<ResortData>` for re-ranking) across `amgettuple` calls
(`src/access_method/scan.rs:212-217`) `[verified-by-code]`; each `next()` advances
`graph.greedy_search_iterate()` and skips deleted tuples
(`offset == InvalidOffsetNumber`) (`scan.rs:233-248`) `[verified-by-code]`. For
concurrency, after returning a result the scan keeps an index page pinned —
`state.last_buffer = Some(PinnedBufferShare::read(…))` "per PostgreSQL locking
requirements" (`scan.rs:339-346`) `[verified-by-code, from-comment]` — i.e. it relies on
buffer pin/share semantics to keep a referenced node alive across the
amgettuple-returns-to-executor boundary, the same hazard nbtree solves with its own pin
discipline. Cross-ref `.claude/skills/locking/SKILL.md`.

## Notable design decisions (cited)

- **Two storage layouts.** `StorageType::Plain` (uncompressed) vs
  `StorageType::SbqCompression` (Statistical Binary Quantization), selected by the
  `Storage` trait's `page_type()` (`storage.rs:48, 125-126`) `[verified-by-code]`;
  `memory_optimized` (SBQ) is the default and inner-product is "not compatible with
  plain storage" (`README.md`) `[from-README]`.
- **Build-time index parameters** (not GUCs) parsed by `options::init` (`lib.rs:17`):
  `num_neighbors` (50), `search_list_size` (100), `max_alpha` (1.2),
  `num_dimensions`, `num_bits_per_dimension`, `storage_layout` (`README.md`)
  `[from-README]` — classic AM `reloptions` surface.
- **Parallel build is opt-in and gated** by vector count
  (`diskann.min_vectors_for_parallel_build` default 65536) and a flush interval, with a
  `force_parallel_workers` override (`guc.rs:30-69`) `[verified-by-code]`; yet
  `amcanparallel = false` (`mod.rs:64`) — parallelism is in the *build* path, not in
  parallel index *scans*.
- **`relocatable = false`, `superuser = true`, `requires = 'vector'`** in the control
  file (`vectorscale.control:4-6`) `[verified-by-code]` — a hard install-time dependency
  and elevated-privilege install.
- **pgrx version pin** `pgrx = "=0.16.1"`, with `rkyv` (zero-copy archived
  serialization, used for the on-page node `ArchivedType`), `simdeez` (SIMD distance),
  `rand`, and a local `pgvectorscale_derive` proc-macro (`pgvectorscale/Cargo.toml:21-25`)
  `[verified-by-code]`; `crate-type = ["cdylib", "rlib"]`, PG 14–18 via features
  (`Cargo.toml:5-16`) `[verified-by-code]`.

## Anthropology takeaway

pgvectorscale is the corpus's most *complete* foreign-language index AM: it does not
abuse the AM interface to escape PG (the ZomboDB/ParadeDB pattern), it re-implements a
real durable index — graph nodes in PG pages, a `Tape` allocator, Generic-WAL durability,
a live `aminsert`, and a resumable greedy search — entirely in Rust through pgrx. The
interesting divergences are at the seams. (a) **Memory**: the build graph lives in Rust
`Vec`s sized to a *fraction* of `maintenance_work_mem`, so the limit is honoured by
arithmetic, not by `MemoryContext` enforcement — a thing a leak-audit or OOM
investigation must know, because `pg_backend_memory_contexts` will not see the bulk of
the build footprint. (b) **Durability via Drop**: the "started Generic xlog must be
finished or aborted" invariant is delegated to a Rust destructor (`GenericXLogAbort` in
`Drop`), an elegant restatement of a C cleanup discipline that's worth flagging as a
pattern other pgrx AMs should copy. (c) **ABI coupling to another extension's on-disk
type** (pgvector's `Vector` struct, mirrored not linked) is a fragility the core AMs
never accept and a concrete "what breaks if pgvector changes its varlena layout"
question for any upgrade matrix. The honest gaps in this note: I deep-read `lib.rs`,
`mod.rs`, the control/Cargo files, and skimmed `build.rs`, `scan.rs`, `storage.rs`,
`page.rs`, `tape.rs`, `pg_vector.rs`, `guc.rs` via single-pass fetches — I did **not**
audit the SBQ quantization math, the `rkyv` archived node layout, or the parallel-build
worker coordination, so claims about those rest on signatures and the README.
