## 1. Domain & purpose

pg_onnx runs **ML inference inside PostgreSQL** using Microsoft's ONNX Runtime.
You import a trained `.onnx` model into a table as bytes, then call SQL functions
to run inference over JSONB inputs and get JSONB outputs `[from-README]`
(`README.md:9`, `README.md:188-215`). The headline use case is a `BEFORE INSERT`
trigger that fills a `prediction` column from model output at write time
(`README.md:217-265`).

The stated design problem is the crown-jewel divergence. PostgreSQL forks **one
backend process per connection**; if every backend loaded ONNX Runtime and a
model onto the GPU, "the system and GPU would run out of memory" `[from-README]`
(`README.md:10-12`). pg_onnx therefore does **not** run inference in the calling
backend. It runs `onnxruntime-server` (a vendored submodule) as a **single
dynamic background worker** that owns and recycles **one ONNX Runtime session
per model file**, and backends reach it as thin clients over an IPC bridge
(`README.md:10-36` mermaid diagram).

## 2. How it hooks into PG

- **`_PG_init` + dynamic bgworker registration.** On first load `_PG_init`
  registers a single dynamic background worker via
  `RegisterDynamicBackgroundWorker` — *not* the static
  `RegisterBackgroundWorker` at `shared_preload_libraries` time
  (`pg_onnx/pg_background_worker.cpp:72`). The worker sets
  `BGWORKER_SHMEM_ACCESS | BGWORKER_BACKEND_DATABASE_CONNECTION`,
  `bgw_start_time = BgWorkerStart_RecoveryFinished`,
  `bgw_restart_time = BGW_NEVER_RESTART`, and `bgw_function_name = "worker_main"`
  (`pg_onnx/pg_background_worker.cpp:62-70`) `[verified-by-code]`. `_PG_init`
  guards against re-entry from within a worker and against double registration
  (`MyBackendType == B_BG_WORKER … || state->handle != NULL`,
  `pg_onnx/pg_background_worker.cpp:47-55`). After registering it blocks on
  `WaitForBackgroundWorkerStartup` and then busy-polls shared `state->port` up
  to 1000×10 ms until the worker's TCP listener is up
  (`pg_onnx/pg_background_worker.cpp:80-102`) `[verified-by-code]`.

- **SQL-callable C functions.** Six `PG_FUNCTION_INFO_V1` entry points
  (`pg_onnx/pg_onnx.cpp:7-12`): `pg_onnx_inspect_model_bin` plus five
  `pg_onnx_internal_*` session functions. The SQL install script wires them and
  adds pure-SQL/plpgsql wrappers `pg_onnx_import_model`, `pg_onnx_drop_model`,
  `pg_onnx_list_model` (`pg_onnx/pg_onnx--1.2.1.sql:22-145`). Notably
  `pg_onnx_execute_session` is declared `LANGUAGE C IMMUTABLE STRICT`
  (`pg_onnx/pg_onnx--1.2.1.sql:139-145`) — marking a network-round-trip
  inference call IMMUTABLE is an aggressive choice that lets it be used in
  index/trigger contexts but is arguably a volatility mismarking `[inferred]`.

- **Model storage as a large object + catalog table.** `CREATE EXTENSION`
  creates schema `ext_pg_onnx` with a `model` table keyed `(name, version)`
  that stores options/inputs/outputs as JSONB and the model bytes indirectly
  via an `lo_oid` OID column (`pg_onnx/pg_onnx--1.2.1.sql:1-14`).
  `pg_onnx_import_model` runs `lo_from_bytea(0, $3)` to spill the BYTEA into a
  large object (`pg_onnx/pg_onnx--1.2.1.sql:52`), so multi-hundred-MB models
  aren't held whole in a datum.

- **CMake build, not PGXS/meson.** The build is driven by CMake
  (`CMakeLists.txt:1-13`, `pg_onnx/CMakeLists.txt`), using a custom
  `add_postgresql_extension(... VERSION 1.2.1 SCRIPTS pg_onnx--1.2.1.sql ...)`
  macro (`pg_onnx/CMakeLists.txt:49-59`). This is itself a divergence: the
  overwhelming majority of extensions use a PGXS `Makefile` or (recently) meson.
  The `.control` file is **generated by the CMake macro** — there is no static
  `pg_onnx.control` in the repo `[verified-by-code]` (absent from `SOURCE_FILES`
  and macro emits it). The submodule `onnxruntime-server` is added first
  (`CMakeLists.txt:6-9`) with OpenSSL/test/install disabled, and pg_onnx links
  `onnxruntime_server_static` (`pg_onnx/CMakeLists.txt:62`). C++17 throughout
  (`CMakeLists.txt:4`), which is a further divergence from core PG's C.

## 3. Where it diverges from core idioms

**THE divergence: out-of-process inference over a loopback-TCP bridge.** The
usual extension model runs everything in the calling backend's address space
(the `.so` executes in-process). Sibling in-DB-ML extensions do exactly that:
`[[pg-strom]]` runs GPU CustomScan nodes in the backend, and `[[morphingdb]]`
links libtorch and runs inference in-backend. pg_onnx deliberately does the
opposite — the backend is a **thin client**:

- The bgworker side spins an `onnxruntime-server` **TCP server** on a
  thread-run `boost::asio::io_context`, binding an ephemeral port and publishing
  it into shared memory (`state->port = server.port()`)
  (`pg_onnx/bridge/bgworker_side/bgworker.cpp:14-24`) `[verified-by-code]`. The
  worker's main loop is the textbook `WaitLatch(WL_LATCH_SET | WL_TIMEOUT |
  WL_POSTMASTER_DEATH, 100L, PG_WAIT_EXTENSION)` idiom, treating
  `WL_POSTMASTER_DEATH` and SIGTERM as terminate signals
  (`pg_onnx/bridge/bgworker_side/bgworker.cpp:31-56`).
- The process (backend) side opens a fresh loopback socket to
  `127.0.0.1:state->port` **per call**, writes a binary `protocol_header`
  (`type`, `json_length`, `post_length`, `length`, all network-byte-order via
  `htons`/`HTONLL`) followed by the JSON request, then reads the framed response
  back and `json::parse`s past the header
  (`pg_onnx/bridge/process_side/api_client.cpp:9-72`) `[verified-by-code]`. So
  the IPC bridge is **not shared memory and not a UNIX socket** — it is a
  loopback-TCP client/server protocol reusing onnxruntime-server's own wire
  format. Shared memory is used only for the tiny handshake struct
  (handle/pid/port/config), not for data transfer.

- **The `process_side/` ↔ `bgworker_side/` split** is the bridge boundary.
  `bridge/process_side/task/task_*.cpp` marshal a request and call
  `api_request()`; `bridge/bgworker_side/bgworker.cpp` hosts the server that the
  vendored session manager answers. Task types (`Orts::task::type::
  EXECUTE_SESSION`, `CREATE_SESSION`, …) come from the submodule, not from PG
  (`pg_onnx/bridge/process_side/task/task_execute_session.cpp:15`,
  `task_create_session.cpp:145`).

- **How model bytes cross the bridge.** For a local worker, `create_session`
  streams the large object out of the catalog via a `lo_post_data` wrapper (a
  `post_data` subclass driving `be_lo_open`/`be_loread`/`be_lo_lseek64` through
  `DirectFunctionCall*`), spills it to a backend temp file with
  `OpenTemporaryFile`/`FileWrite`, and passes the **temp file path** to the
  worker in the request `option.path` rather than shipping bytes over the socket
  (`pg_onnx/bridge/process_side/task/task_create_session.cpp:13-47`,
  `110-152`) `[verified-by-code]`. Because worker and backend share a
  filesystem, the model transfers by path, not by copy. (Remote-standalone mode,
  still stubbed, would instead POST the bytes as the request body — the
  `post`/`content_length` machinery in `api_client.cpp:39-48` exists for that.)
  The bytes themselves are read from the catalog by an SPI query
  `SELECT lo_oid, option FROM ext_pg_onnx.model WHERE name=$1 AND version=$2`
  (`task_create_session.cpp:54-108`).

- **Session lifecycle: recycled per model, not per backend.** The worker keeps
  one `Orts::onnx::session_manager` (`bgworker.cpp:19`); sessions are keyed by
  `(model, version)` and shared across all backends. `task_execute_session`
  implements **lazy create-on-miss**: if EXECUTE returns
  `{"error":"session not found"}` it calls `create_session` and retries once
  (`pg_onnx/bridge/process_side/task/task_execute_session.cpp:15-25`)
  `[verified-by-code]`. This is the memory-safety payoff of the whole design:
  one GPU/ONNX session per model file for the *entire cluster*, independent of
  how many backends are connected.

- **C++ exception → `ereport` safety across the FFI/bridge boundary.** Every
  SQL-callable entry wraps its body in `try { … } catch (std::exception &e) {
  elog(ERROR, "%s: %s", __FUNCTION__, e.what()); }`
  (`pg_onnx/pg_onnx.cpp:29-58`, `65-69`, `86-90`, `107-111`, `128-132`,
  `152-156`) `[verified-by-code]`. This is the mandatory boundary discipline
  when a C++ extension calls into PG's `longjmp`-based error system: a C++
  exception must be caught and converted to `ereport(ERROR)` before it can
  escape into PG C frames, and conversely PG errors `longjmp` past C++
  destructors. The `#undef Abs` before including onnxruntime headers
  (`pg_onnx/pg_onnx.hpp:41-43`) is a related C-vs-C++ symbol-collision fix.

- **Shared state via `ShmemInitStruct`, allocated lazily.** `extension_state()`
  attaches (or creates) a `packed` `extension_state_t` under the fixed name
  `"pg_onnx_state"` (`pg_onnx/extension_state.cpp:10-27`,
  `pg_onnx/pg_onnx.hpp:52-66`). Note it calls `ShmemInitStruct` with
  `RequestAddinShmemSpace` **commented out** (`extension_state.cpp:16`) and never
  registers a `shmem_request_hook` — it relies on slack in the main shmem
  segment, a fragility the core idiom explicitly warns against `[inferred]`.

## 4. Notable design decisions (file:line)

- Dynamic (runtime) bgworker, never-restart, recovery-finished start:
  `pg_onnx/pg_background_worker.cpp:62-72`.
- Backend blocks polling `state->port` for worker readiness after
  `WaitForBackgroundWorkerStartup`: `pg_onnx/pg_background_worker.cpp:80-102`.
- Worker main = asio TCP server thread + WaitLatch supervisory loop:
  `pg_onnx/bridge/bgworker_side/bgworker.cpp:14-56`.
- Per-call loopback socket + hand-rolled binary framing header:
  `pg_onnx/bridge/process_side/api_client.cpp:15-35`, `50-72`.
- Model streamed via large object → temp file → pass-by-path:
  `pg_onnx/bridge/process_side/task/task_create_session.cpp:110-145`.
- Lazy session create-on-miss with single retry:
  `pg_onnx/bridge/process_side/task/task_execute_session.cpp:15-25`.
- SPI read of the model catalog row inside the task:
  `pg_onnx/bridge/process_side/task/task_create_session.cpp:54-108`.
- `_PG_fini` terminates + waits for the worker on unload:
  `pg_onnx/pg_background_worker.cpp:105-114`.
- GUC `pg_onnx.num_threads` (PGC_SUSET) sizes the ONNX thread pool; remote-host
  GUCs are stubbed/commented: `pg_onnx/pg_background_worker.cpp:11-38`.
