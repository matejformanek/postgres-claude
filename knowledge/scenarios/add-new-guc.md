# Scenario — Add a new GUC

## Scope — what's in / out

**In scope:**
- A new built-in GUC: one record added to
  `src/backend/utils/misc/guc_parameters.dat`, which `gen_guc_tables.pl`
  expands into the matching `ConfigureNamesBool/Int/Real/String/Enum`
  entry in the generated `guc_tables.c` at build time. The backing C
  variable, default, bounds, optional check/assign/show hooks, and a
  `postgresql.conf.sample` line all land in this scenario.
  `[verified-by-code]` (`src/backend/utils/misc/guc_parameters.dat:14-34`,
  `src/backend/utils/misc/gen_guc_tables.pl:1-30`).
- A custom GUC from an extension or test module: a
  `DefineCustomBoolVariable` / `IntVariable` / `RealVariable` /
  `StringVariable` / `EnumVariable` call inside `_PG_init` plus a
  `MarkGUCPrefixReserved("<prefix>")` so unknown
  `<prefix>.*` settings produce errors after the library is loaded.
  `[verified-by-code]` (`src/backend/utils/misc/guc.c:5049`,
  `src/backend/utils/misc/guc.c:5186`, `src/include/utils/guc.h:358-406`).
- Documentation under `doc/src/sgml/config.sgml` (or per-extension SGML
  for a contrib), plus a regression test that asserts `pg_settings`
  visibility and any non-trivial check/assign hook behavior.

**Out of scope:**
- A new GUC whose sole purpose is a planner cost constant — that change
  belongs in `add-new-cost-model-knob.md`, which also touches
  `optimizer/cost.h`, `cost_*.c`, and shape-specific tests.
- A new GUC that exists only to toggle a hook on/off — usually combined
  with `add-new-hook.md`; both checklists union under the planner.
- A new `pg_stat_*` view's "what to collect" knob — those tend to ride
  inside `add-new-pg-stat-view.md` and use the same `.dat` plumbing.
- Renaming, deprecating, or removing an existing GUC — has its own
  back-compat dance (`GUC_CUSTOM_PLACEHOLDER`, release-note text, dump
  filter in `dumputils.c`); not a generic scenario yet.

## Pre-flight

- **Companion skills:** load `gucs-config` — covers the
  `DefineCustom*Variable` family, picking `GucContext`
  (`PGC_POSTMASTER` / `PGC_SIGHUP` / `PGC_SUSET` / `PGC_USERSET` /
  `PGC_BACKEND` / `PGC_INTERNAL`), the check/assign/show contract, the
  `GUC_UNIT_*` / `GUC_LIST_*` / `GUC_REPORT` / `GUC_EXPLAIN` flag set,
  and string-GUC `guc_malloc` allocation rules.
  `[verified-by-code]` (`.claude/skills/gucs-config/SKILL.md:1-20`).
- **Canonical commit:** `a8f45dee917` — *"Add wal_sender_shutdown_timeout
  GUC to limit shutdown wait for replication"* (Fujii Masao, 2026). A
  clean built-in GUC patch: 10 lines in `guc_parameters.dat`, one line
  in `postgresql.conf.sample`, ~40 lines in `config.sgml`, the global
  variable + use site in `walsender.c`, and a new TAP test under
  `src/test/subscription/t/`. Read it before drafting your own.
  `[verified-by-code]` (`git -C source show a8f45dee917`).
- **Common pitfalls (one-line each):**
  - Editing the generated `guc_tables.c` directly — it has an "autogenerated"
    banner and is overwritten by `gen_guc_tables.pl` on every build.
    `[verified-by-code]` (`src/backend/utils/misc/gen_guc_tables.pl:1-30`,
    `src/backend/utils/.gitignore`).
  - Missing the `postgresql.conf.sample` line for a non-`GUC_NOT_IN_SAMPLE`
    GUC — release scripts and `check_guc` lint flag this.
    `[from-comment]` (`src/backend/utils/misc/guc_parameters.dat:25-27`).
  - Forgetting `MarkGUCPrefixReserved` in an extension — `<prefix>.*`
    typos silently parse as placeholder GUCs instead of erroring.
    `[verified-by-code]` (`src/backend/utils/misc/guc.c:5186`).
  - Wrong `GucContext`: a setting that needs a postmaster restart marked
    `PGC_SIGHUP` (or worse `PGC_USERSET`) gives users a working SET that
    quietly does nothing. `[from-comment]` (`src/include/utils/guc.h:71-95`).
  - String GUC `assign_hook` that frees the incoming `newval` — the
    string buffer is `guc_malloc`'d and managed by guc.c; freeing it
    crashes. `[from-comment]` (`src/include/utils/guc.h` near
    `GucStringAssignHook`).

## Pitfalls

- **Generated `guc_tables.c` looks editable but isn't.** It's listed in
  `src/backend/utils/.gitignore` and rebuilt from `guc_parameters.dat`
  on every compile via the `gen_guc_tables.pl` rule. Reviewers reject
  hand-edits there. `[verified-by-code]`
  (`src/backend/utils/misc/gen_guc_tables.pl:1-30`,
  `src/backend/utils/.gitignore`).
- **Alphabetical order in `guc_parameters.dat` matters.** The file is
  human-edited and the comment block at the top (line ~24) says
  "in alphabetical order"; placing your record elsewhere passes the
  generator but loses the next merge conflict. `[from-comment]`
  (`src/backend/utils/misc/guc_parameters.dat:24`).
- **Group choice drives both `pg_settings.category` and the SGML
  section.** `enum config_group` in `guc_tables.h` enumerates the
  legal values; mismatching SGML section vs. group makes the docs and
  `pg_settings` disagree. `[from-comment]`
  (`src/include/utils/guc_tables.h:50-53`).
- **`GUC_UNIT_*` flags interact with `min`/`max`.** Bounds are in the
  unit you declared; `GUC_UNIT_MS` + `min => '0'` + `max => 'INT_MAX'`
  caps at ~24 days. For "millisecond GUC up to a few minutes" prefer
  `GUC_UNIT_MS` with a hand-picked `max`. `[verified-by-code]`
  (`src/include/utils/guc.h:232-244`).
- **String GUCs and parallel workers.** `GUC_LIST_INPUT` strings get
  re-parsed in workers; if your check_hook does expensive validation,
  it runs per worker. Cache the parsed result in `assign_hook` using
  the `extra` field, not in a side global. `[from-comment]`
  (`src/include/utils/guc.h` near `GucStringCheckHook`).
- **Custom GUCs disappear if the library is not preloaded.** A user
  who runs `SET myext.foo = 1` *before* `LOAD 'myext'` (or before
  postmaster loads it via `shared_preload_libraries`) gets a placeholder
  GUC that holds the string but isn't type-checked. `MarkGUCPrefixReserved`
  closes this hole — without it, typos like `myext.foo` are silently
  accepted. `[verified-by-code]` (`src/backend/utils/misc/guc.c:5186`).
- **`PGC_POSTMASTER` GUCs need a restart, not SIGHUP.** Picking
  `PGC_SIGHUP` for a knob that's read once at startup (e.g. shared-memory
  sizing) gives users a working SET that doesn't take effect — a classic
  silent-misconfiguration bug. `[from-comment]`
  (`src/include/utils/guc.h:71-95`).
- **Synchronization traps (must change together):**
  - `guc_parameters.dat` ↔ `postgresql.conf.sample` (unless
    `GUC_NOT_IN_SAMPLE`). Edit one, edit the other.
  - `guc_parameters.dat`'s `group` ↔ `config.sgml` section heading. The
    GUC's section in the SGML docs MUST match the `group` so
    `pg_settings.category` and the docs agree.
  - Custom GUC (`DefineCustom*Variable`) ↔ `MarkGUCPrefixReserved`. Always
    pair them in `_PG_init`.
  - `flags => 'GUC_LIST_QUOTE'` ↔ `variable_is_guc_list_quote()` in
    `src/bin/pg_dump/dumputils.c`. Without this, `pg_dump` round-trip
    breaks for that GUC.
