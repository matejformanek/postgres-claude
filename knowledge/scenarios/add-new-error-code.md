# Scenario — Add a new SQLSTATE / error code

## Scope — what's in / out

**In scope:**
- One new line in `src/backend/utils/errcodes.txt`: pick a 5-char SQLSTATE
  (existing 2-char class + new 3-char subclass), `E`/`W`/`S` severity,
  `ERRCODE_<NAME>` macro, and the lowercase `spec_name` used as the
  plpgsql `EXCEPTION WHEN` condition.
- The downstream files (`errcodes.h`, `plerrcodes.h`, `pltclerrcodes.h`,
  `spiexceptions.h`, `errcodes-table.sgml`) **regenerate automatically**
  from `errcodes.txt` — never hand-edit them. `[verified-by-code]`
  (`src/backend/utils/errcodes.txt:10-22`, `src/backend/utils/Makefile:55-56`,
  `src/pl/plpgsql/src/meson.build:18-22`).
- At least one site in the backend that actually raises the new code via
  `ereport(..., errcode(ERRCODE_<NAME>), ...)`. An errcode with zero
  raise sites is dead weight; add the call where the condition is
  detected.
- A regression test that triggers the error and matches against the new
  SQLSTATE (and ideally a plpgsql `EXCEPTION WHEN <spec_name>` test).

**Out of scope:**
- Reusing an existing SQLSTATE — no errcodes.txt change needed, just an
  `ereport()` with the existing macro. Not a scenario.
- Reshuffling the existing taxonomy / renaming codes — historically rare
  and breaks user code that pattern-matches on `SQLSTATE`. Discuss on
  -hackers first.
- ecpg client-side codes (class `Y*`) — same file, but different
  audience; handle as a one-off.

## Pre-flight

- **Companion skills:** load `error-handling` — it covers the `ereport()`
  / `errcode()` / `errmsg()` discipline and where in elog.c the
  `errcode_for_*` helpers translate errno → SQLSTATE.
  `[verified-by-code]` (`.claude/skills/error-handling/SKILL.md`).
- **Canonical commit:** `2d8ef5e24fd` — *"Add new error code for 'file
  name too long'"* (Michael Paquier, 2024). A textbook two-file patch:
  one line in `errcodes.txt` plus a four-line case in
  `errcode_for_file_access()`. Read it before starting — it's the
  reference example for the minimal shape of this change-class.
  `[verified-by-code]` (`git -C source show 2d8ef5e24fd`).
- **Common pitfalls (one-line each):**
  - Picking a subclass that the SQL standard could later assign — if the
    class begins with `0-4` or `A-H`, the subclass MUST begin with `5-9`,
    `I-Z`, or `P` (PostgreSQL convention). `[from-comment]`
    (`src/backend/utils/errcodes.txt:61-74`).
  - Hand-editing `errcodes.h` / `plerrcodes.h` / `errcodes-table.sgml` —
    they carry an "autogenerated, do not edit" banner and will be
    clobbered on the next build. `[verified-by-code]`
    (`src/backend/utils/generate-errcodes.pl:29`,
    `src/pl/plpgsql/src/generate-plerrcodes.pl:10`).
  - Forgetting the `spec_name` (4th column) — without it, the plpgsql
    condition name and the SGML appendix entry are silently skipped.
    `[from-comment]` (`src/backend/utils/errcodes.txt:33-35`).
  - Adding the line outside the right `Section:` header — generator
    treats `Section:` as a class boundary for SGML, so misplaced rows
    appear in the wrong table. `[verified-by-code]`
    (`src/backend/utils/errcodes.txt:39-44`,
    `doc/src/sgml/generate-errcodes-table.pl`).

## Pitfalls

- **Subclass collisions with the standard.** Classes `0-4` and `A-H`
  are owned by the SQL standard; subclasses defined by the standard
  also begin with `0-4`/`A-H`. PG-defined subclasses in those classes
  MUST start with `P` (or `5-9`/`I-Z`) to be future-proof. Violating
  this risks future-version conflict. `[from-comment]`
  (`src/backend/utils/errcodes.txt:64-74`).
- **Each class must have a `000` generic.** When carving a brand-new
  class, add the `<class>000` generic first; downstream tooling and
  the appendix lean on this. `[from-comment]`
  (`src/backend/utils/errcodes.txt:57-59`).
- **`E/W/S` severity field is not free-form.** Only `E` (error), `W`
  (warning), `S` (success) are recognized by the generators; anything
  else silently breaks the parse. `[verified-by-code]`
  (`src/backend/utils/generate-errcodes.pl`).
- **Forgetting `\set VERBOSITY verbose` in regression tests.** Default
  `psql` verbosity does not print the SQLSTATE — your test will pass
  even if the code didn't actually change to the new SQLSTATE. Use
  `verbose` when you want the assertion to ride on SQLSTATE rather
  than `errmsg` text. `[from-docs]` (psql `\errverbose` /
  `VERBOSITY`).
- **Hidden out-of-tree consumers.** `errcodes.txt` is installed under
  `$(datadir)/errcodes.txt` for extensions and external tooling to
  parse. `[verified-by-code]` (`src/backend/utils/Makefile:89-90`,
  `src/backend/utils/meson.build:3`). Renaming a code (vs. adding one)
  is a breaking change for anyone consuming this file.
- **Synchronization traps (must change together):**
  - `errcodes.txt` ↔ all six generated files. Build regenerates them
    automatically — do not commit hand-edits. If you see a generated
    file in `git diff` outside of a clean rebuild, you edited the
    wrong layer.
  - `errcodes.txt` (`spec_name` column present) ↔ plpgsql/Python/Tcl
    condition tests. If a downstream PL test references the
    `spec_name` you must populate the 4th column or the condition is
    not registered.

## Verification (exact test invocations)

```bash
# Confirms the generated headers carry the new macro and the
# downstream PL/SQL condition tests still pass.
meson compile -C dev/build-debug

# Backend regression — add or extend a .sql/.out pair under
# src/test/regress that triggers the new ereport().
meson test -C dev/build-debug --suite regress

# plpgsql EXCEPTION WHEN coverage — relevant if the new code is a
# user-facing condition that a stored procedure should be able to
# catch.
meson test -C dev/build-debug --suite plpgsql

# PL/Python + PL/Tcl smoke (only if those PLs are configured in the
# build) — `spiexceptions.h` / `pltclerrcodes.h` are regenerated, and
# `--with-python` / `--with-tcl` builds will surface stale references.
meson test -C dev/build-debug --suite plpython
meson test -C dev/build-debug --suite pltcl

# Docs build picks up the regenerated errcodes-table.sgml.
meson compile -C dev/build-debug alldocs
```

If you added a brand-new SQL file for the regression test, name it
explicitly (e.g. `src/test/regress/sql/<class>_errors.sql`) and add it
to the schedule in `src/test/regress/parallel_schedule` (or
`serial_schedule`).
