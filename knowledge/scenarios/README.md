# Scenarios — task-shaped playbooks ("the web of context")

The rest of `knowledge/` is **reference-shaped** (one doc per file, per
subsystem, per idiom). This layer is **task-shaped**: one playbook per
common change-class, answering "I want to do X — what's the FULL sweep
of files I have to touch, in what order, with what pitfalls?"

Created because the corpus was excellent at "what is bufmgr.c" and
terrible at "to add a new built-in data type, here are the 12-14 files
you must edit and the order to edit them in." Reference layer + skill
layer + idiom layer + scenarios layer = the web of context. A scenario
is the lookup table the planner consults *first*, before doing fresh
greps.

## When to write a new scenario

Whenever the user is about to (or has just) done one of these:

- A change-class that touches **≥3 files in ≥2 directories** and is
  likely to recur (someone will do it again — adding a new SQL keyword,
  new data type, new GUC, new index AM, new WAL record).
- A change where forgetting one file leaves the tree in a half-baked
  state that compiles but misbehaves at runtime (the classic
  `psqlscan.l` / `pgc.l` sync trap for keyword changes).
- A change that has a well-known *pattern* in PG history — multiple
  upstream commits doing the same shape.

If a change-class is one-shot or hyper-specific to a single subsystem,
it doesn't need a scenario; the subsystem doc + per-file docs cover
it.

## The scenario template

Every scenario uses the same skeleton. See `_template.md` for the
copy-paste version. The required structure:

```yaml
---
scenario: <kebab-case-slug-matching-filename>
when_to_use: <one-sentence trigger>
companion_skills: [skill1, skill2, ...]   # all skills that participate
related_scenarios: [other-scenario-1, ...] # cross-link siblings
canonical_commit: <upstream SHA of a representative historical patch>
last_verified_commit: <SHA the file checklist was last validated against>
---

# Scenario — <Imperative title>

## Scope — what's in / out
## Pre-flight — companion skills, canonical commit, common pitfalls
## File checklist (the FULL sweep)
| # | File | Why | Per-file doc | Companion skill |
## Phases — suggested split for pg-feature-plan
## Pitfalls — from knowledge/issues/* + prior shadow runs
## Verification — exact test invocations
## Cross-refs
```

## The linking rules

A scenario is a **hub** in the web. It links *out* to:

- Per-file docs (`knowledge/files/.../*.md`) for every row in the
  checklist.
- Companion skills (`.claude/skills/<name>/SKILL.md`) for the relevant
  procedural knowledge.
- Idioms (`knowledge/idioms/*.md`) for cross-cutting patterns
  (memory-contexts, ereport, fmgr) the playbook depends on.
- Subsystem docs (`knowledge/subsystems/*.md`) for the high-level
  context.
- Issue register (`knowledge/issues/<subsystem>.md`) for known traps in
  the affected area.

Per-file docs link *back* to scenarios via an `## Appears in scenarios`
section (auto-generated by Layer C1 of the web-of-context plan). That
backlink makes the web bidirectional — opening any file in the corpus
shows which scenarios it participates in.

`related_scenarios:` in the frontmatter is the scenario↔scenario edge.
Every entry there MUST point at a real scenario file in this directory
(enforced by Layer C5 refresh discipline).

## The refresh discipline

Scenarios go stale. The mitigation is the `last_verified_commit:`
field plus an "anchor-drift" warning surfaced by `pg-feature-plan`.

- **`canonical_commit`**: the upstream SHA of a representative
  historical patch that did this change-class. Pin once at creation;
  never auto-bump (it's the historical reference, not the current
  state).
- **`last_verified_commit`**: the upstream SHA at which the file
  checklist was last validated. Bump every time a refresh routine
  re-verifies the cites.

`pg-feature-plan` Step 0 reads `last_verified_commit`. If the plan's
anchor SHA ≠ the scenario's `last_verified_commit`, the planner emits
a "scenario stale" warning and runs a fresh grep pass to validate the
checklist before treating it as authoritative.

A scenario refresh = re-verify every file in the checklist still
exists, the rationale still holds, and no NEW required files have
crept into the change-class. The cloud routine
`.claude/cloud/pg-corpus-maintainer.md` owns this on a schedule;
manual refresh is also fine.

## The hard-integration contract with pg-feature-plan

(See `.claude/skills/pg-feature-plan/SKILL.md` Step 0 + M3 gate.)

When a planner invocation matches a scenario:

1. The scenario's file checklist becomes the **starting authoritative
   §3 table** of the plan. Every file in the checklist MUST land in
   the plan's §3 table.
2. The planner can ADD sites it discovers via grep. It can NEVER drop
   sites the scenario named. To drop a site, the user must approve and
   the scenario itself must be edited.
3. If more than one scenario matches (composite feature), the planner
   takes the union of their checklists.
4. If no scenario matches, the planner flags this — the scenarios
   layer has a gap to fill.

This is what makes scenarios "load-bearing" rather than "advisory".

## How a scenario is picked

The decision tree in `_index.md` is the human-facing routing surface.
For programmatic / first-touch routing:

1. `pg-feature-brainstorm` (Phase 1 of the planner) matches the
   brainstormed approach against `_index.md` and names the candidate
   scenario(s) in the brainstorm doc.
2. `pg-feature-plan` (Phase 2) re-checks the match in Step 0; if
   exactly one matches, it pins; if multiple, it unions; if none, it
   escalates.
3. `/pg-scenario <slug>` (skips brainstorm) lets a user who already
   knows the change-class pull the scenario directly and emit a
   starter plan from its checklist.

## What lives outside this directory

- **Per-file docs** — reference, not playbook. The "what is this file"
  layer.
- **Subsystem docs** — architecture, not workflow. The "how does this
  subsystem hang together" layer.
- **Idioms** — patterns, not file checklists. Cross-cutting techniques
  shared across many scenarios.
- **Issues** — surfaced concerns, not plans. Scenarios link out to
  these for known traps in the affected area.
- **Planning slugs** (`planning/<slug>/`) — forward-looking design docs
  for a specific feature in flight. A scenario is the *generic
  playbook* for a change-class; a planning slug is a *specific
  instance* of executing that playbook.

## Coverage

The full set of scenarios + each one's freshness lives in
`progress/scenarios-coverage.md`. The decision tree + one-liner index
lives in `_index.md` in this directory.
