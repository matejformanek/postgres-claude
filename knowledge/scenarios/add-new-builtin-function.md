# Scenario — Add a new built-in SQL function

## Scope — what's in / out

**In scope:**
- One new C function with a `PG_FUNCTION_INFO_V1` declaration plus a
  matching `pg_proc.dat` row.
- The cheapest catalog-touching change-class: a single OID, a single
  catversion bump, a single regression test file (or addition to an
  existing one).
- Volatility, strictness, leakproofness, parallel-safety — anything
  expressible by `pro*` columns.
- Optional `func/func-XXX.sgml` documentation entry.

**Out of scope:**
- New operator (commutator / negator / restrict / join estimators) →
  see `add-new-operator`.
- Function that becomes an aggregate's sfunc / finalfunc → see
  `add-new-aggregate-function`.
- Function tied to a new data type (I/O, send/recv, type modifiers) →
  see `add-new-data-type`.
- Window functions and procedures (`prokind = 'w'` / `'p'`): same
  catalog shape but exec semantics differ; not covered here.

## Pre-flight

- **Companion skills:** load `catalog-conventions` (BKI / OID / catversion
  rules) and `fmgr-and-spi` (FMGR v1 calling convention, `PG_GETARG_*` /
  `PG_RETURN_*` macros).
- **Canonical commit:** `df6949ccf7a` — *Add tid_block() and tid_offset()
  accessor functions* (Andres Freund, 2026-04-05). Two C functions, one
  `pg_proc.dat` block, one catversion bump, regression tests in
  `tid.sql` / `tid.out`, doc in `func/func-tid.sgml`, and an
  `opr_sanity.out` delta for the leakproof list. Tiny but complete.
- **Common pitfalls (one-line each):**
  - Forgot to bump `CATALOG_VERSION_NO` — local cluster works, anyone
    pulling your patch can't `initdb` against the same data directory
    (see `knowledge/idioms/catalog-conventions.md` §3).
  - OID collision with a concurrent patch — caught by
    `src/include/catalog/duplicate_oids` (run before submit).
  - Mismatched `prosrc` ↔ C symbol — links cleanly, fails at first call
    with "could not find function in shared library".
  - `proleakproof => 't'` without satisfying leakproof rules (no
    error-on-data, no side-channel) — flagged by `opr_sanity` regression
    or by a committer (see `knowledge/issues/utils-adt.md` for prior
    cases).

## Phases — suggested split for `pg-feature-plan`

The tree must build at the end of each phase.

1. **Phase 1 — C implementation, no catalog wiring.** Files: [2].
   Write the `PG_FUNCTION_INFO_V1(name)` + `Datum name(PG_FUNCTION_ARGS)`
   in the chosen `.c`. Use the standard FMGR macros (`PG_GETARG_*`,
   `PG_RETURN_*`, `PG_ARGISNULL` if not strict). Phase-end check: tree
   compiles (`meson compile -C dev/build-debug`). Function is unreachable
   from SQL yet — that's fine.

2. **Phase 2 — Catalog wiring + catversion.** Files: [1, 3, 8, 9].
   Run `./unused_oids` in `src/include/catalog/`; pick a starting OID
   in 8000-9999. Add the `pg_proc.dat` row near related entries. Bump
   `CATALOG_VERSION_NO`. Run `./duplicate_oids` — must be empty.
   Phase-end check: rebuild + `initdb` against a fresh data dir +
   `psql -c "SELECT myfunc(…)"` returns the expected result.

3. **Phase 3 — Tests + docs.** Files: [4, 5, 6, 7, 11].
   Add SQL exercises to the existing test group; regenerate expected
   output if needed. If you set `proleakproof`, accept the
   `opr_sanity.out` delta. Document in `func/func-<group>.sgml`.
   Phase-end check:
   `meson test -C dev/build-debug --suite regress --suite docs`
   passes; new test name shows up in the regress output.

(If your function genuinely needs a new `.c` file, fold its creation +
meson/Makefile wiring into Phase 1.)

## Pitfalls

- **Forgotten catversion bump** — symptom: `FATAL: database files are
  incompatible with server` after pulling your patch onto an existing
  cluster. Mitigation: bump in Phase 2, same commit as the `.dat`
  edit. See `knowledge/idioms/catalog-conventions.md` §3
  [from-comment](source/src/include/catalog/catversion.h:26-29).
- **`prosrc` typo** — the entry compiles, `initdb` succeeds, the first
  SQL call fails with `could not find function "foo" in file "…"`. Grep
  `src/include/utils/fmgrprotos.h` after build to confirm the prototype
  for your C symbol appears.
- **`proisstrict => 't'` (the default) but C function dereferences a
  NULL arg** — never triggers a crash because fmgr short-circuits, but
  if you actually need NULL-handling logic, set `proisstrict => 'f'` and
  use `PG_ARGISNULL(n)` inside the function [verified-by-code](source/src/include/catalog/pg_proc.h:60-130).
- **`provolatile` wrong** — defaulting to `v` (volatile) is safe but
  blocks constant-folding and parallel plans. If the function is a
  pure transformation of its args, mark `i` (immutable); if it depends
  on transaction-visible state, `s` (stable). Wrong direction = silently
  wrong results in some plans.
- **`proleakproof => 't'` for a function that can ereport on data** —
  RLS leak. The `opr_sanity` regression prints the full leakproof list;
  CI flags additions that look suspicious, and committers will too. If
  unsure, don't mark it [from-comment](source/src/test/regress/expected/opr_sanity.out:525-530).
- **OID overlap with another in-flight patch** — `duplicate_oids` will
  catch it locally; CI catches it on submission. Re-run `unused_oids`
  and shift. Committer's `renumber_oids.pl` collapses the range at
  merge time, so don't fight for a specific number.

- **Synchronization traps** (sibling files that must change together):
  - `pg_proc.dat` row ↔ `catversion.h` bump (same commit, every time).
  - `.dat` `prosrc` symbol ↔ C function name (must match exactly,
    case-sensitive).
  - New `.c` file ↔ `src/backend/utils/adt/meson.build` +
    `src/backend/utils/adt/Makefile` (both build systems).
  - Brand-new regress test file ↔ `parallel_schedule` entry.

## Verification (exact test invocations)

```bash
# Build (forces Gen_fmgrtab.pl + genbki.pl regen)
meson compile -C dev/build-debug

# Re-initdb against a clean data dir — required after catversion bump
rm -rf dev/data-debug && dev/install-debug/bin/initdb -D dev/data-debug

# Smoke-test the function
dev/install-debug/bin/postgres -D dev/data-debug -k /tmp &
psql -h /tmp -c "SELECT myfunc(…);"

# Regression — scope this scenario expects to exercise
meson test -C dev/build-debug --suite regress

# Specifically: opr_sanity (catches catalog inconsistencies) and the
# test group you added rows to
meson test -C dev/build-debug --suite regress --test regress  # full

# Docs (SGML validity)
meson test -C dev/build-debug --suite docs

# OID hygiene (run from src/include/catalog/, not via meson)
cd source/src/include/catalog && ./duplicate_oids   # expect empty
```

If you created a brand-new regression test file, name it explicitly
here: e.g. `meson test -C dev/build-debug --suite regress
--test <new_test_name>`. Otherwise the existing group's test name
covers it.
