## 1. crake — `XversionUpgrade-REL_16_STABLE-HEAD` — ICU soname gap (recurring)

- animal: **crake** — Fedora 44, gcc 16.1.1, `-Dicu=enabled`, `TestICU`
  module pinned `REL_20`.
- branch/stage: `master` / `XversionUpgrade-REL_16_STABLE-HEAD`
  (cross-version pg_upgrade from a saved REL_16 install to HEAD).
- run: 2026-06-10 04:57:02 (also 03:37, 02:02 same day — repeating every run).
- log: `show_log.pl?nm=crake&dt=2026-06-10 04:57:02`.
- signature:
  `…/saves.crake/REL_16_STABLE/bin/postgres: error while loading shared
  libraries: libicuuc.so.76: cannot open shared object file: No such file or
  directory`.
- rootcause (high confidence): **same as 2026-06-09**. crake keeps a *saved*
  REL_16_STABLE binary install to drive the cross-version upgrade test. That
  binary was linked against `libicuuc.so.76`; the host's ICU was since upgraded
  and only ships a newer soname, so the old `postgres` can't start, the upgrade
  step can't dump from the old cluster, and the stage fails. This is a
  **buildfarm-host environment break local to crake**, not a PostgreSQL bug —
  the saved REL_16 binary needs rebuilding (or the old libicu kept around).
  No action for the corpus; track only to avoid mis-triaging it as a HEAD
  regression. Persisting ≥2 days now.

## 2. dikkop — `ECPG-Check` — SIGSEGV in ecpg thread test (recurring)

- animal: **dikkop** — FreeBSD 14.4-RELEASE, clang 19.1.7.
- branch/stage: `master` / `ECPG-Check`.
- run: 2026-06-10 04:14:28 (continuation of 2026-06-09 18:19:27).
- log: `show_log.pl?nm=dikkop&dt=2026-06-10 04:14:28`.
- signature:
  `# (test process was terminated by signal 11: Segmentation fault) /
  # 1 of 67 tests failed.` in `src/interfaces/ecpg/test`, with the tail of the
  run reaching the `thread/*` group (`ecpg/thread-thread ERROR: table
  "test_thread" does not exist` just before the abort) — consistent with the
  segfault landing in the ECPG threaded client tests.
- rootcause (medium confidence): **same family as 2026-06-09** — an ECPG
  threaded-test SIGSEGV specific to FreeBSD + clang on dikkop. The other ERRORs
  in the server log (duplicate-key, "more than one row", relation-does-not-exist
  in `preproc-whenever`/`sql-sqljson`) are the **expected** negative-test
  output of the ecpg suite, not the failure; the actual failure is the signal-11
  crash of a test client process. Likely a thread-local / allocator interaction
  in the generated ecpg client under FreeBSD libc, or a flaky races in the
  thread test on this platform. Worth watching whether it correlates with any
  recent `src/interfaces/ecpg/` change; nothing in this window's 4 commits
  touches ecpg, so it predates them.
