## Root-cause notes

### `goldfish` — serial `timedout` across master/REL_18/REL_17/REL_16

**Likely a flake / environment issue, not a code regression.** When a single
animal times out on *every* branch (including long-frozen stable branches that
saw no relevant commits) within the same window, the common cause is the build
host itself — an overloaded/slow macOS runner, a stuck TAP test waiting on a
timeout, or a hung `pg_ctl start`/`IPC::Run` child — not anything in master.
`timedout` is the buildfarm's stage when the whole run exceeds its wall-clock
budget, so it doesn't even pin a specific test. **Action:** treat as the
`goldfish` known-flake fingerprint; only escalate if a *second, faster* animal
starts timing out on master in the same area. No corpus impact.

### `fairywren` — REL_18_STABLE `pg_upgradeCheck`

`fairywren` is a Windows (MSYS2/MinGW) animal; `pg_upgradeCheck` runs
`src/bin/pg_upgrade`'s cross-version regression under `vcregress`/meson. On a
single stable branch (not master, not the other branches) this is most often an
**environment / Windows-path-or-locale flake** in the pg_upgrade test harness
(temp-dir cleanup, `\r\n` dump diffs, or a transient file lock from AV/indexer)
rather than a server bug — REL_18_STABLE is frozen and `pg_upgrade` itself saw
no commits in this window. **Action:** needs-eyes-but-low-priority; confirm by
checking whether the next `fairywren` REL_18 run is green (transient) or whether
the same `pg_upgradeCheck` diff repeats (then it's a real stable-branch issue).
Log was too large to inline-fetch (>10 MB); triage from the stage + animal
profile. No corpus impact.
