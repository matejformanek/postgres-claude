### 1. taipan / bushmaster / urutu / canebrake ‚Äî Check (`generated_virtual`) ‚Äî master ‚Äî **REAL BUG, STILL OPEN** ‚≠ê‚≠ê‚≠ê

- Animals: `taipan` (Debian Sid gcc 13), `bushmaster` (Debian Sid clang 17),
  `urutu` (Debian Sid clang 16), `canebrake` (Debian Sid gcc 13).
- Stage: `Check`, regress test **#159 `generated_virtual`**, on essentially every
  master HEAD in the window (`e5c4058`, `64797ad`, `850b921`, `29fb598`,
  `bdae2c2`, `ff8bec8`, `7ca548f`, `1f2297b`, `f04781d` rows all show it). The
  commit in each row is just whatever was HEAD at run time ‚Äî **none of them
  cause it.**
- **Rootcause (high confidence ‚Äî verified from the regression.diffs in the
  taipan 2026-06-18 15:39 log):** unchanged from yesterday. A regular NOT NULL
  column placed *after* a VIRTUAL generated column deforms to the wrong value:

  ```
  CREATE TABLE gtest21c (a int NOT NULL,
                         b int GENERATED ALWAYS AS (a * 2) VIRTUAL NOT NULL,
                         c int NOT NULL);
  INSERT INTO gtest21c (a, c) VALUES (10, 42);
  SELECT a, b, c FROM gtest21c;
  --  expected:  10 | 20 | 42
  --  got:       10 | 20 |  0     (taipan/bushmaster/urutu/canebrake)
  ```

  Reading `0` for `c` instead of `42` is the signature of tuple deformation
  miscomputing the physical offset of a column that follows a zero-storage
  VIRTUAL generated column. It reproduces only on Debian-Sid (newest gcc 13 /
  clang 16-17) and not on older toolchains, which is consistent with a
  compiler-optimization-exposed defect (uninitialized/strict-aliasing read)
  rather than an animal-local environment problem.
- **What changed since 2026-06-17:** nothing ‚Äî the test (`1f7dfe8c`) is still in
  the tree, the bug is still unfixed, and the four animals have stayed red for
  24h+. The earlier theory that this is a real correctness bug (not a flaky test)
  is now strongly reinforced by its persistence. **This warrants a fix or a test
  revert upstream.** Full analysis: `knowledge/buildfarm-lessons/2026-06-17.md`.
