## Selected failures

### 1. urutu / taipan / bushmaster / canebrake — Check (`generated_virtual`) — master — **REAL BUG** ⭐⭐⭐

- Animals: `urutu` (Debian Sid, clang 16), `taipan` (Debian Sid, gcc 13),
  `bushmaster` (Debian Sid, clang 17), `canebrake` (Debian Sid, gcc 13).
- Stage: `Check`, failing regress test **#159 `generated_virtual`**. The commit
  shown in the buildfarm row varies (`3048e81`, `d2cea63`, `da82fbb`) because it
  is whatever was HEAD at run time — **the failure is not caused by those
  commits.** It first appears right after `1f7dfe8c` ("Add tuple deformation test
  for virtual generated columns", David Rowley, 2026-06-17T04:57Z) landed, and
  recurs on every subsequent run on these animals.
- **Rootcause (high confidence — verified from the regression diff):** the new
  test reads back a table mixing a VIRTUAL generated column with trailing regular
  NOT NULL columns, and the trailing real column deforms to the **wrong value**:

  ```
  CREATE TABLE gtest21c (a int NOT NULL,
                         b int GENERATED ALWAYS AS (a * 2) VIRTUAL NOT NULL,
                         c int NOT NULL);
  INSERT INTO gtest21c (a, c) VALUES (10, 42);
  SELECT a, b, c FROM gtest21c;
  --  expected:  10 | 20 | 42
  --  got:       10 | 20 |  0     (on urutu/taipan/bushmaster/canebrake)
  ```

  Column `c` returns `0` instead of `42`. This is a **tuple-deformation bug for
  virtual generated columns**: when a VIRTUAL column sits between real
  attributes, the deform path appears to mis-track the attribute offset/null
  bitmap for the column after the virtual one, so the trailing real column reads
  as zero. It is nondeterministic across the fleet (some animals green, some red
  on the same SHA), which points at an uninitialized/garbage-dependent path
  rather than a deterministic logic error — consistent with a deformation/slot
  bug surfacing only under certain memory contents or build flags.
- **Why it matters:** virtual generated columns are a young feature (PG 18+).
  This is a correctness defect returning **silently wrong query results**, not a
  crash, which is the worse failure class. The committer's own machine was green,
  so it slipped past pre-commit testing — exactly the buildfarm's job to catch.
- **Lesson / watch:** expect a fast upstream follow-up (likely in the executor's
  generated-column projection or `slot_getsomeattrs`/`heap_deform_tuple` virtual
  -column handling). If a backend fix lands touching
  `src/backend/executor/execExpr*.c` or `access/common/heaptuple.c`, add a corpus
  note. Re-check on the next run: the test should go green fleet-wide once fixed.
  [verified-by-code: buildfarm regression.diffs on urutu @2026-06-17 07:43,
  taipan @15:02, bushmaster @18:59]
