## 1. shelduck · master · **Build** [`d15a6bc` then `11ed011`] — CONFIRMED root cause: gcc 4.8.5 lacks `_Generic`

- **Animal:** SUSE Linux Enterprise Server 12 SP5, **gcc 4.8.5**, **s390x (z15)**,
  owner markwkm. Failures: 2026-07-14 12:21 [`d15a6bc`], 2026-07-15 13:25 [`11ed011`].
- **History:** OK at `1ae87df` (07-13 09:26), then Build FAILS starting exactly at
  `d15a6bc` and stays red. Clean bisect boundary.
- **Root cause (confirmed from `show_log.pl`):** commit `d15a6bc2e16d` ("Replace
  `__builtin_types_compatible_p` with `_Generic`") makes `StaticAssertVariableIsOfTypeMacro()`
  — and thus `unconstify()`/`unvolatize()` — expand to C11 `_Generic`. **gcc 4.8.5
  does not implement `_Generic`** (GCC gained it in 4.9). The log shows:
  `src/include/c.h:1093: warning: implicit declaration of function '_Generic'`
  followed by hard errors `c.h:1388: error: expected expression before 'const'`
  and `c.h:1127: error: expression in static assertion is not an integer`. First
  casualty is `file_utils.c:712`:
  `void *zerobuf_addr = unconstify(PGIOAlignedBlock *, &zbuffer)->data;`
- **Assessment:** genuine portability regression for pre-4.9 GCC, *not* a flake.
  The commit's own message claims `_Generic` "works on all compilers" — true for
  C11-conformant ones, but shelduck's toolchain predates C11 `_Generic`. Since PG
  master already requires C11, the practical outcome is that shelduck's gcc 4.8.5
  can no longer build master; expect either an animal toolchain bump by the owner
  or an upstream min-compiler note. Worth watching whether a hackers thread /
  follow-up commit addresses the 4.8.x case. (Note the subsequent `11ed011`
  failure is the *same* Build break, not a new one — 11ed011 is a trivial
  error-message fix that can't itself break the build.)
