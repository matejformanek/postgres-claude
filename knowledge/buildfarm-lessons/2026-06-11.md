  09:55 and 10:36 snapshots — persistent, not a one-off.
- **Evidence** (`upgrade.fairywren/HEAD/dumpdiff-REL_16_STABLE`):
  ```
  @@ -596484,7 +596484,7 @@
  -4.6 .. 7.0
  +4.6 .. ~7.0
  ```
- **Rootcause (high confidence):** this is the on-disk output change from
  today's commit `0e1f1ed157e9` "seg: Fix seg_out() to preserve the upper
  boundary's certainty indicator". The cross-version test loads `contrib/seg`
  data into a REL_16 cluster (whose buggy `seg_out` dropped the `~`, dumping
  `4.6 .. 7.0`), pg_upgrades to HEAD, and re-dumps; HEAD's fixed `seg_out`
  emits `4.6 .. ~7.0`, so the origin-vs-converted diff is non-empty and the
  stage fails. The stored datum is unchanged — only its textual rendering is —
  so this is a *correct* fix surfacing as a dump mismatch, exactly the category
  the XversionUpgrade comparison is designed to catch.
- **Action:** none for `master` or the corpus. This is a test-harness concern:
  the upgrade-adjust machinery (`src/test/perl/PostgreSQL/Test/AdjustUpgrade.pm`)
  needs to normalize the seg `~upper` rendering for REL_16→HEAD, or the seg
  cross-version fixture needs adjusting. Expect the same dumpdiff on every animal
  that (a) runs XversionUpgrade and (b) gets far enough to reach the dump
  comparison, until that filtering lands. Tracked against
  `[[knowledge/files/contrib/seg/seg.c.md]]` and the
