## §1 — unicorn `pg_amcheckCheck` on master `[f0a4f28]`

`f0a4f28` is a **docs-only** commit (pg_dump/dumpall/upgrade SGML wording about
external statistics — see `knowledge/upstream-deltas/2026-06-23.md`). A
docs-only change cannot regress `pg_amcheckCheck`, so this is an intermittent
Windows/MSVC failure that simply landed on whatever revision was HEAD at
checkout time. `unicorn` has a documented history of sporadic
`pg_amcheckCheck` failures (covered in `knowledge/buildfarm-lessons/2026-06-06.md`
carry-over, last cluster 06-01/06-02/06-03). The captured stage log tail shows
only `xid_wraparound` SKIP noise (test not enabled in `PG_TEST_EXTRA`), not a
deterministic amcheck assertion. **Rootcause guess:** Windows-specific timing /
TAP-node teardown flake; not actionable unless it recurs on multiple animals
against the same commit. `[from-log]` `[inferred]`

## §2 — goldfish chronic `timedout` (all branches)

`goldfish` (macOS 15.7.5, Homebrew gcc 15.2.0_1) reports stage `timedout` on
*every* run sampled — master (`80bb0eb`, `9a60f29`), REL_18_STABLE (`ac6a58a`,
`e025267`), and back through 06-19/06-20/06-21 (`dc51167`, `0319040`,
`f25a07b`, `9bcb8a6`). When an animal times out on every revision regardless of
branch or commit content, the signal is host-level (build/test wall-clock
budget exceeded on a slow or contended macOS runner), **not** a PG code defect.
**Rootcause guess:** animal capacity / `wait_timeout` mis-config on the goldfish
host. Not actionable for PG hackers; worth an owner ping if it never clears.
`[from-buildfarm-listing]` `[inferred]`
