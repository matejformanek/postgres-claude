## Notable failures (root-cause guesses)

### 1. taipan — `Check` stage — REAL value-corruption diff `[8d22f52]` *(most interesting)*
- Animal: `taipan`, Debian Sid, gcc 13, branch `master`, build `2026-06-18 23:48`.
- Stage `Check`: `not ok 159 + generated_virtual`, `1 of 245 tests failed`.
- The `regression.diffs` hunk (expected vs results, `generated_virtual.out`
  around line 731):
  ```
   CREATE TABLE gtest21c (a int NOT NULL,
                          b int GENERATED ALWAYS AS (a * 2) VIRTUAL NOT NULL,
                          c int NOT NULL);
  -  a  | b  | c
  -  10 | 20 | 42
  +  a  | b  | c
  +  10 | 20 | 0
  ```
  Column `c` (a plain `int NOT NULL`, expected `42`) came back `0`.
- **Root-cause guess:** a real but transient bug in **virtual generated-column**
  evaluation in which the gencol computation aliased/zeroed an *adjacent*
  non-generated column's value. The same `Check` failure clusters across several
  Debian-Sid animals (taipan gcc-13, bushmaster clang-17, canebrake gcc-13,
  urutu clang-16) at a band of consecutive master commits
  (`ff8bec8`→`1f2297b`→`7ca548f`→`f04781d`→`8d22f52`) on 06-18, then stops — the
  signature of a regression that lived for a few hours and was fixed. The
  changeset for this build includes `f04781df5da "Silence 'may be used
  uninitialized' compiler warning"`, which is suggestive of an
  uninitialized-read in the same area. **Worth a corpus note** under virtual
  generated columns if it recurs; likely already fixed upstream. *(Stale: 06-18,
  outside 24h — flagged for the human refresh pass to confirm against current
  master.)*

### 2. goldfish — `timedout` stage `[0319040]` — environmental (slow animal)
- Animal: `goldfish`, macOS 15.7.5, Homebrew GCC 15.2.0, `2026-06-20 14:15`
  (newest failure on the board).
- Detail log: `'meson_test_timeout' => undef`, **`timed out after 14400 secs`**
  (the buildfarm's own 4-hour wall clock, not a test assertion).
- **Root-cause guess:** the animal is simply too slow / overloaded to finish a
  full meson `test` run inside its 4 h limit; goldfish shows the *same* `timedout`
  stage at `f25a07b` (06-19 20:15) and `dc51167` (06-19 10:15) too — a standing
  per-animal capacity problem, **not a code regression**. Owner should raise
  `wait_timeout`/`meson_test_timeout` or thin the test set. Classify as flake.
