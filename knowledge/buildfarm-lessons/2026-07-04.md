## Failures (8)

### 1. koel · master · Configure `[a8c2547]` · 2026-07-03 22:04 — **environmental (animal misconfig)**

Fedora 44, gcc 16.1.1. Configure aborts at linker detection:

```
ERROR: Unable to detect linker for compiler `gcc -Wl,--version -fuse-ld=lld`
collect2: fatal error: cannot find 'ld'
```

**Root cause (confirmed from log):** the animal's `CFLAGS` request the LLVM
linker (`-fuse-ld=lld`) but `lld` isn't installed on the box, so meson's
linker-detection probe fails. Pure host configuration problem — **not a
PostgreSQL bug.** Two consecutive Configure failures (21:49 + 22:04) confirm a
persistent env state, not a transient. Lesson: bleeding-edge-toolchain animals
(Fedora 44 / gcc 16) that pin a non-default linker must actually have it
installed; a Configure-stage failure with `cannot find 'ld'` is always the
animal, never the tree.

### 7. crake · REL_15/16/17/18_STABLE · abi-compliance-check · 2026-07-04 ~16:00-17:09 — **expected tooling noise (back-branch ABI diff)**

Fedora 44, gcc 16.1.1. The `abi-compliance-check` module flagged on **four**
back-branches (15, 16, 17, 18) in the same run. When an ABI check trips
simultaneously across every stable branch on one animal, it is the check's
*baseline* that moved (toolchain/abi-dumper update on Fedora 44 / gcc 16), not
four independent real ABI breaks. **Best guess: tooling/baseline drift, not a
per-branch regression.** A genuine ABI break shows on one branch where a specific
back-patch changed a struct, not uniformly across all four. Lesson: a synchronized
abi-compliance-check failure across all live back-branches on a single animal is
a baseline-refresh signal — verify the abi-dumper/abi-compliance-checker version
before treating any single branch as broken.
