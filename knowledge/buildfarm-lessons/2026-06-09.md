## 1. `crake` — XversionUpgrade-REL_16_STABLE-HEAD: stale saved binary vs. bumped host ICU (NOT a code regression)

- **Animal:** crake — Fedora 44, gcc 16.1.1.
- **Branch:** HEAD. **Stage:** `XversionUpgrade-REL_16_STABLE-HEAD`, failing on
  every commit it built in-window: `[93a3e68]` 18:42, `[cd7b204]` 18:17,
  `[9d8cdcb]` 11:12, `[bd4bd30]` 2026-06-08 23:50 — plus a long tail of earlier
  commits on the same page.
- **Rootcause `[from-log]`:** the saved REL_16 binary can't even print its
  version. From `upgrade.crake/HEAD/REL_16_STABLE-ctl.log`:
  > `/home/andrew/bf/root/saves.crake/REL_16_STABLE/bin/postgres: error while
  > loading shared libraries: libicuuc.so.76: cannot open shared object file: No
  > such file or directory`
  > `no data was returned by command ""…/REL_16_STABLE/bin/postgres" -V"`
  > `program "postgres" is needed by pg_ctl but was not found …`
  The host's ICU was upgraded (soname moved past `libicuuc.so.76`, i.e. ICU 76 →
  77+ on Fedora 44), so the **previously-saved** REL_16_STABLE `postgres`,
  linked against the old ICU, no longer starts. The cross-version `pg_upgrade`
  step needs that old-cluster binary to run, so every HEAD snapshot fails at the
  same spot regardless of the commit under test.
- **Why this supersedes the 2026-06-08 guess:** yesterday's buildfarm note
  attributed crake's xversion failures to a vague "Fedora 44 + gcc 16 / perl
  mismatch." The actual log pins it to a concrete **ICU soname bump invalidating
  the saved old-version binary** — an animal-maintenance issue (the saved tree
  needs rebuilding against current ICU), not a Postgres bug.
- **Action:** none for the corpus or for `master`. Self-resolves when crake's
  owner rebuilds/re-saves the REL_16 binary against current ICU. Do **not**
  associate with the catversion bump (`9d8cdcb`); the failure predates it and is
  an ICU link error, not a catalog mismatch.

## 2. `dikkop` — ECPG-Check: SIGSEGV in `thread/alloc` (FreeBSD/clang, isolated)

- **Animal:** dikkop — FreeBSD 14.4-RELEASE, clang 19.1.7.
- **Branch:** HEAD. **Stage:** `ECPG-Check` at `[cd7b204]` 2026-06-09 18:19:27.
- **Rootcause `[from-log]`:** a single ecpg regression test crashed:
  > `not ok 65 - thread/alloc`
  > `# (test process was terminated by signal 11: Segmentation fault)`
  All neighbours (`thread/prep`, `thread/descriptor`, `thread/thread_implicit`)
  passed; only `thread/alloc` segfaulted, and the run completed `1..67`.
- **Why it's not a `cd7b204` regression:** commit `cd7b204` only touches the
  `pgrepack` decoding plugin / REPACK — nothing in `src/interfaces/ecpg`. A
  segfault confined to one ecpg *thread* test on one FreeBSD/clang animal is the
  signature of a threading/timing flake (or a latent ecpg thread-allocation bug)
  rather than anything the in-window commits introduced.
- **Note / watch:** `dikkop ECPG-Check` also showed up on 2026-06-04 (last
  recorded as a one-off). Two hits in ~5 days on the same animal+stage nudges
  this from "pure flake" toward "intermittent ecpg thread-safety issue on FreeBSD
  clang worth watching." If `thread/alloc` segfaults on a *second* animal, escalate
  to a real bug. For now: no corpus action.
