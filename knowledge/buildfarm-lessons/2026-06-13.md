## 1. fairywren ‚Äî `sslCheck` ‚Äî `ssl/001_ssltests`: revoked UTF-8 client cert + server-side CRL dir (NEW-ish, code-adjacent)

- animal: **fairywren** (Windows / Msys2 Server 2019, gcc 14.1.0, `-Dssl=openssl`,
  OpenSSL-Win64).
- branch/stage: `master` / `sslCheck`; run 2026-06-13 09:30:47 `[3e3d787]`.
- log: `show_log.pl?nm=fairywren&dt=2026-06-13 09:30:47` (~11 MB).
- signature: 326 of 327 meson tests pass; `327/327 postgresql:ssl /
  ssl/001_ssltests ERROR exit status 1`, with
  `# Failed test 'certificate authorization fails with revoked UTF-8 client
  cert with server-side CRL directory: matches'`.
- rootcause guess `[from-buildfarm-log]`: this is the CRL-*directory* (hashed
  `c_rehash`-style dir, as opposed to a single CRL file) path of the SSL TAP
  suite, exercised with a client certificate whose DN contains UTF-8. On
  Windows/Msys with OpenSSL-Win64 the failure is almost certainly in how the
  revoked cert is matched against the on-disk CRL directory ‚Äî either the
  OpenSSL-Win64 build hashes/looks up the CRL dir differently, or the UTF-8 DN
  is encoded differently on this platform so the "should be revoked" cert is not
  recognized as revoked (the test asserts the connection *fails* with a match
  message and instead got a different outcome). Most likely **platform/OpenSSL-
  build specific**, not a core-code regression ‚Äî only this one Windows animal is
  red on this subtest, and the other 6 ssl sub-suites (`002_scram`,
  `003_sslinfo`, `004_sni`, etc.) passed on the same run. Worth a watch: if a
  non-Windows animal goes red on the same subtest, re-classify as a real CRL-dir
  regression. Relevant core code lives under `src/backend/libpq/be-secure-openssl.c`
  (CRL dir load) ‚Äî no corpus per-file doc edit warranted on one platform-only hit.

## 3. crake ‚Äî `perl-check` ‚Äî DBD::Pg `undefined symbol: PQclosePrepared` (CONTINUING, host env)

- animal: **crake** (Fedora 44, gcc 16.1.1) ‚Äî Andrew Dunstan's animal.
- runs: many in-window, e.g. 2026-06-12 14:42:48 `[87df6e6]`, plus earlier rows.
- log: `show_log.pl?nm=crake&dt=2026-06-12 14:42:48`.
- signature:
  ```
  Can't load .../auto/DBD/Pg/Pg.so for module DBD::Pg:
    undefined symbol: PQclosePrepared, version RHPG_18
  Compilation failed in require at ./contrib/intarray/bench/bench.pl line 10.
  ```
- rootcause guess `[from-buildfarm-log]`: pure **host-environment skew**. The
  system-installed `DBD::Pg` Perl module on crake is linked against a packaged
  libpq that **lacks `PQclosePrepared`** (added in PG 17; symbol version
  `RHPG_18` is the distro's libpq build). `perl-check` (perlcritic) walks every
  `.pl`, and `contrib/intarray/bench/bench.pl` does `use DBI;`/`DBD::Pg` at
  `BEGIN`, so the load failure aborts the syntax check of that file. Not a PG
  code issue ‚Äî crake needs its system `perl-DBD-Pg` rebuilt against a libpq that
  exports `PQclosePrepared`. Recurring; exclude from "real" failures.
