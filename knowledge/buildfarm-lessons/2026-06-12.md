## 1. massasauga / parula / leafhopper — `TestModules` — new `test_cplusplusext` needs libstdc++ (NEW)

- animals: **massasauga** (Amazon Linux 2, gcc 15.x nightly), **parula** (Amazon
  Linux 2, gcc experimental), **leafhopper** (Amazon Linux 2023, gcc
  experimental) — three AL animals failing **identically**.
- branch/stage: `master` / `TestModules` (build of `src/test/modules`).
- runs: 2026-06-12 13:11:19 (massasauga), 12:35:06 (parula), 12:33:03
  (leafhopper) — all at `[5459223]`.
- logs: `show_log.pl?nm=massasauga&dt=2026-06-12 13:11:19`,
  `…nm=parula&dt=2026-06-12 12:35:06`.
- signature (verbatim):
  ```
  make -C test_cplusplusext all
  g++ -Wall -Wpointer-arith -Werror=vla … test_cplusplusext.cpp
  ../../../../src/include/c.h:91:10: fatal error: type_traits: No such file or directory
  make[1]: *** [test_cplusplusext.o] Error 1
  make: *** [all-test_cplusplusext-recurse] Error 2
  ```
- rootcause (high confidence): a **newly added in-tree test module**
  `src/test/modules/test_cplusplusext` compiles a `.cpp` translation unit that
  includes PostgreSQL's `c.h`. `c.h` (around line 91) now `#include <type_traits>`
  under C++ — part of the recent work to let `c.h`/`postgres.h` be included from
  C++ extension code. These three Amazon Linux animals have a working **g++
  driver** but **no libstdc++ development headers installed** (`type_traits` is a
  C++ stdlib header), so the new module cannot build and the whole `TestModules`
  stage fails. This is a **buildfarm-host package gap** (install
  `libstdc++-devel` / `gcc-c++` stdlib headers on the AL animals), *not* a
  PostgreSQL bug — but it is worth a corpus note because it is the first in-tree
  module to hard-require a C++ toolchain, so any animal building `src/test/modules`
  now needs a complete C++ stdlib, not just a `g++` binary. Track until the
  animal owners install the headers or the module gains a configure guard.

## 2. massasauga — `Check` — xml regression diff in the xpath-change window (self-resolved)

- animal: **massasauga** (Amazon Linux 2, gcc 15.x nightly).
- branch/stage: `master` / `Check` (core `pg_regress`), run 2026-06-12 02:48:23,
  HEAD `[a8ee70b]`.
- log: `show_log.pl?nm=massasauga&dt=2026-06-12 02:48:23` (16 MB — the failing
  `regression.diffs` dominates).
- signature: `not ok 218   + xml` (the rest of the suite passes; one stray
  `FATAL: terminating logical replication worker due to administrator command`
  in the server log is normal shutdown noise, not the cause).
- rootcause (high confidence): a **transient expected-output lag**, not a code
  bug. `8bf257ae` (01:25Z, "Fix handling of namespace nodes in xpath()") changed
  xpath output and updated the primary `xml.out` / `xml_1.out`, but the
  libxml-variant **`xml_2.out`** (which massasauga's libxml build selects) was
  only updated ~2 h later in `0b74df6` (03:37Z, "Update expected regression test
  output for xml_2.out"). massasauga happened to build at `a8ee70b` (02:44Z) —
  inside that window — so its `xml` test diffed against a stale `xml_2.out` and
  went red. Any animal that builds at or after `0b74df6` is green again. Classic
  "behaviour change lands, one expected-output variant catches up in a follow-up
  commit, the buildfarm flags the gap." No action — included to close the loop
  with the deltas writeup (commits #4 and the `0b74df6` follow-up).

- runs: many this window — `[5459223]` 17:12 & 12:57, `[0b74df6]` 04:05,
  `[a8ee70b]` 03:16, `[0e47bb5]` 02:10, `[389bd4c]` 00:56, `[44196fd]`
  (06-11 23:37) … essentially **every run**.
- log: `show_log.pl?nm=crake&dt=2026-06-12 12:57:03`.
- signature:
  `…/saves.crake/REL_16_STABLE/bin/postgres: error while loading shared
  libraries: libicuuc.so.76: cannot open shared object file: No such file or
  directory`.
- rootcause (high confidence): **identical to the 2026-06-09/06-10 entries** —
  crake keeps a *saved* REL_13/REL_16 binary to drive the cross-version upgrade;
  that binary is linked against `libicuuc.so.76`, the host ICU has since bumped
  soname, so the old `postgres` won't start and the upgrade dump step fails. A
  **host-environment break local to crake** (saved binary needs rebuilding or the
  old libicu kept around), not a HEAD regression. Persisting ≥4 days now; the
  fact that it fires on every tip SHA is itself the tell that it's
  commit-independent.

## 4. thorntail — `module-test_checksumsCheck` — UBSan stall (recurring class)

- animal: **thorntail** — Debian sid, gcc 11.2.0, configured
  `CC='ccache gcc -fsanitize=undefined -fsanitize-undefined-trap-on-error'`,
  `--enable-cassert --enable-injection-points`.
- branch/stage: `master` / `module-test_checksumsCheck`, run 2026-06-12 06:31:05,
  HEAD `[0b74df6]`.
- log: `show_log.pl?nm=thorntail&dt=2026-06-12 06:31:05`.
- signature: the `pg_checksums-check` sub-stage passes (00:00:08) but
  `module-test_checksums-check` runs **04:21:21** before the stage is recorded as
  a failure — an abnormally long wall time for that module.
- rootcause (medium confidence): a **UBSan-trap or stall under
  `-fsanitize-undefined-trap-on-error`**, not a content diff. With trap-on-error,
  a UBSan violation `SIGILL`s the backend with no diagnostic; combined with the
  4 h+ stage time this reads as a backend dying or hanging mid-`test_checksums`
  rather than an output mismatch. The log doesn't surface a clean assertion
  string, so this stays **medium** confidence — flag for a closer look if it
  recurs on a non-UBSan animal (which would upgrade it from "sanitizer-only
  environment artifact" to a real bug in the checksums test module).
