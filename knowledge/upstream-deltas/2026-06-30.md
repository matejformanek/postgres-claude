## Deep explainers

### `8d85cb889a39` ‚Äî bufmgr: Fix race in LockBufferForCleanup() [[knowledge/files/src/backend/storage/buffer/bufmgr.c.md]]

A missed-wakeup race introduced by 5310fac6e0f + c75ebc657ffc, which began
allowing a shared pin to be released **while `BM_LOCKED` is set**. The old
sequence was: check refcount > 1 ‚Üí release header lock ‚Üí set
`BM_PIN_COUNT_WAITER` ‚Üí sleep. If the last conflicting unpin happened in the
window after the refcount check but before `BM_PIN_COUNT_WAITER` was published,
`UnpinBuffer()` saw no waiter flag, sent no wakeup, and
`LockBufferForCleanup()` slept forever.

Fix: publish the waiter flag **while still holding the buffer header lock**,
using `pg_atomic_fetch_or_u64(&bufHdr->state, BM_PIN_COUNT_WAITER)` (an atomic
OR that preserves concurrent refcount decrements), then **recheck the refcount
under `BM_LOCKED`**. If `BUF_STATE_GET_REFCOUNT == 1`, the cleanup condition is
already met ‚Üí clear the waiter flag and `goto cleanup_lock_acquired` without
sleeping; otherwise sleep as before. The shared exit path (recovery-conflict
log + ps-display cleanup) was hoisted into a `cleanup_lock_acquired:` label
(diff lines `bufmgr.c:6715`‚Äì`6822`). Reported by buildfarm animal **skink**
(valgrind) as intermittent `048_vacuum_horizon_floor.pl` timeouts.
Backpatch-through: 19. Relevant to the buffer-pin/content-lock layer in the
`locking` skill and any INV-* about `BM_PIN_COUNT_WAITER` publication ordering.

### `8e684ce11dda` ‚Äî Fix unlogged sequence corruption after standby promotion [[knowledge/files/src/backend/access/transam/xlogutils.c.md]]

Symptom: after promoting a standby, `nextval()` on an unlogged sequence created
on the primary could TRAP `pd_special >= SizeOfPageHeaderData` (assert builds)
or `ERROR: bad magic number in sequence`. Root cause: `seq_redo()`
(`sequence_xlog.c:63`) updated the **init-fork** page in shared buffers but
never flushed it to disk. `ResetUnloggedRelations()` rebuilds the main fork at
promotion by copying the init fork **directly from disk, bypassing shared
buffers** ‚Äî so a stale init fork on disk overwrote the WAL-replayed page.

Fix introduces a shared helper
`XLogFlushBufferForRedoIfInit(record, block_id, buffer)` in
`xlogutils.c:321`+ (decl `include/access/xlogutils.h:90`): it reads the block's
fork tag and `FlushOneBuffer()`s iff `forknum == INIT_FORKNUM`. `seq_redo()`
now calls it after `MarkBufferDirty()`. The same helper **replaces the existing
inline `XLogRecGetBlockTag` + `if (forknum==INIT_FORKNUM) FlushOneBuffer()`
boilerplate** in `hash_xlog.c` (`hash_xlog_init_meta_page`,
`hash_xlog_init_bitmap_page`) ‚Äî a net simplification. New TAP test
`recovery/t/054_unlogged_sequence_promotion.pl`. Backpatch-through: 15.
Touches WAL-redo init-fork invariants (`wal-and-xlog` skill); the
`hash_xlog.c` + `sequence_xlog.c` per-file docs both need the
`XLogFlushBufferForRedoIfInit` re-cite.

### `1de468099d27` ‚Äî Disallow SRFs within window OVER clauses (Tom Lane) [[knowledge/files/src/backend/parser/parse_func.c.md]]

`check_srf_call_placement()` (`parse_func.c:2700`) previously treated
`EXPR_KIND_WINDOW_PARTITION` / `EXPR_KIND_WINDOW_ORDER` as "effectively
GROUP BY/ORDER BY" and set `p_hasTargetSRFs = true`. Those two cases are now
deleted and fall through to the frame cases, which reject SRFs ‚Äî so
`SELECT ‚Ä¶ OVER (PARTITION BY generate_series(...))` (and ORDER BY / frame
bound) now errors. Rationale: a window function must not change the row count,
which an SRF in its definition would violate. Users should use a LATERAL FROM
clause instead. Backpatch-through: 19 (squeezed in before the v19 cut). Fixes
bugs #17502 and #19535; new negative regress rows in `tsrf.sql`.

### `2ef57e636fc9` ‚Äî Clean up CPU-identification macros (Tom Lane) [[knowledge/files/src/include/c.h.md]]

Replaces the scattered, inconsistent CPU-guard spellings (`__i386`,
`__ppc__`, `_M_X64`, `_M_AMD64`, `_M_ARM64`, `USE_SSE2`, ‚Ä¶) with gcc's
canonical trailing-underscore forms (`__x86_64__`, `__aarch64__`,
`__powerpc__`, ‚Ä¶), and adds a normalization stanza to `c.h:106`+ that
*defines* the gcc spelling when the compiler only provides its own (notably
MSVC, which defines none of gcc's). Touched: `atomics.h`, `arch-x86.h`,
`pg_bitutils.h`, `pg_cpu.h`, `instr_time.h`, `s_lock.h`, `pg_cpu_x86.c`,
`d2s.c`. **The stanza ends in `#else #error "cannot identify target
architecture"`**, which Tom flagged as possibly-too-aggressive ‚Äî and it
immediately fired on the buildfarm (clang/Ubuntu **greenfly**, Build stage).
See the buildfarm-lessons doc. Corpus impact spans the whole port/atomics +
s_lock CPU-detection cluster (8 tracked headers/files).
