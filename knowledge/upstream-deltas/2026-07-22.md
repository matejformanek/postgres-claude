## Deep explainers

### 1 · `8767a10cb8c5` — walsummarizer stuck on invalid WAL tail (Robert Haas, backpatch-17)

`SummarizeWAL()` documents `maximum_lsn` as "the switch point when reading a
historic timeline, or the most-recently-measured end of WAL when reading the
current timeline." The `WalSummarizerMain` loop always passed `latest_lsn`
(measured end-of-WAL) regardless of timeline — an oversight. The fix
computes `maximum_lsn = XLogRecPtrIsValid(switch_lsn) ? switch_lsn :
latest_lsn` and passes that instead of `latest_lsn`:

- diff: `walsummarizer.c` `WalSummarizerMain()`, new local at `:352` and the
  `SummarizeWAL(...)` call at `:414-417`.

Why it matters: `SummarizeWAL` checks every record both starts and ends
before `switch_lsn`, so with an all-valid archive it still stops correctly.
But if a **corrupted file** exists in the archive *and* the current
timeline's end has advanced past the switch point, the wrong `maximum_lsn`
lets it try to read past the switch point, hit an invalid record, error out,
and **retry forever** — a stuck WAL summarizer, which stalls incremental
backups. Concrete trigger: a promoted primary writes a `.partial` file, an
operator renames it to drop the suffix, and it gets archived with an invalid
tail. `[[knowledge/files/src/backend/postmaster/walsummarizer.c.md]]` —
the doc's "Lifecycle"/summarize-loop section should note the
historic-timeline `maximum_lsn` = `switch_lsn` invariant. `[verified-by-code]`

### 2 · `ccfd4b683867` — `lookup_type_cache()` OOM-safety (Michael Paquier / Matthias van de Meent)

Previously `lookup_type_cache()` pushed `type_id` onto `in_progress_list`
*before* the `TypeCacheHash` `HASH_FIND`. But `TypeCacheHash` uses
`type_cache_syshash()`, which can trigger first-time init of the **TYPEOID
catcache**, and that init can OOM. On the resulting error-recovery path,
`finalize_in_progress_typentries()` sees an in-progress type OID and can
attempt catcache init *again* outside a transaction context → assertion
failure.

Fix: move the two `in_progress_list` writes to *after* the `hash_search`
(`typcache.c:459` region → now `:471-472`), with a comment pointing at
`ConditionalCatalogCacheInitializeCache()`. Reproduced only via injected
probabilistic allocation failures; **no backpatch**.
`[[knowledge/files/src/backend/utils/cache/typcache.c.md]]` — add the
"in-progress list is populated after the initial lookup, to keep OOM error
recovery re-entrancy-safe" note. `[verified-by-code]`

### 3 · `e65c331b8fbf` — split pgstat pending-data setup (Michael Paquier / Sami Imseih)

New exported routine `pgstat_prep_pending_from_entry_ref(PgStat_EntryRef *)`
(`pgstat.c:1361+`, prototype `pgstat_internal.h:683`) that attaches pending
storage to an *already-obtained* entry ref: it derives `kind` from
`entry_ref->shared_entry->key.kind`, asserts a `flush_pending_cb`, and
lazily allocates `entry_ref->pending` from `pgStatPendingContext`, pushing
onto `pgStatPending`. `pgstat_prep_pending_entry()` is refactored to call
`pgstat_get_entry_ref()` then the new helper (`pgstat.c:1311-1319`).

Purpose: a stats kind that must **check a capacity bound** (e.g. a GUC
limiting max entries) before deciding whether to create can now do
`pgstat_get_entry_ref()` → inspect → `pgstat_prep_pending_from_entry_ref()`
as two steps, saving a redundant shmem-hash lookup. The only ordering change
is that `pgStatPendingContext` is now created *after* the entry-ref lookup;
noted as immaterial. Existing callers and out-of-core custom stats kinds are
unaffected. `[[knowledge/files/src/backend/utils/activity/pgstat.c.md]]` —
document the new two-step prep API alongside `pgstat_prep_pending_entry()`.
`[verified-by-code]`

### 4 · `fc6425fbe7a1` — `generate_partition_qual()` OOM-safety (Michael Paquier / Matthias van de Meent)

Same failure shape as #2, in relcache land. The old code assigned
`rel->rd_partcheckcxt` and `rel->rd_partcheck` *while* building them, so an
allocation failure inside `copyObject(result)` (running in the new context
parented to `CacheMemoryContext`) left the relcache with a half-filled
`rd_partcheck` and leaked the context. The rewrite (`partcache.c:411`
region) builds into a *transient* context parented to
`CurrentMemoryContext`, copies, then `MemoryContextSetParent(partctx,
CacheMemoryContext)` and assigns `rd_partcheckcxt`/`rd_partcheck` only at the
very end — so either everything is linked or nothing is, and a mid-build OOM
cleans up with the transaction context. **No backpatch.**
`[[knowledge/files/src/backend/utils/cache/partcache.c.md]]` — add the
"allocate-transient-then-reparent" ordering invariant to the
`generate_partition_qual` notes. `[verified-by-code]`
