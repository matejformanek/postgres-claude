## Deep dives

### 1. `a0354e29c41a` — Revert "Enable fast default for domains with non-volatile constraints" (Andrew Dunstan)

A feature that landed earlier in this cycle is being reverted. The original
let `ALTER TABLE … ADD COLUMN <domaintype>` skip a full table rewrite even when
the domain carried non-volatile CHECK/NOT NULL constraints, by evaluating the
default under **soft-error handling** (`ErrorSaveContext`) and falling back to a
rewrite only if the constraint check tripped.

What the revert removes:

- **The whole soft-error exec-expr surface.** `ExecInitExprWithContext()` and
  `ExecPrepareExprWithContext()` are deleted from `execExpr.c`; the
  `state->escontext` field assignment goes away; the prototypes are dropped from
  `include/executor/executor.h:332,380` (now back to plain `ExecInitExpr` /
  `ExecPrepareExpr`). `[from-diff]` This is the load-bearing change for the
  corpus: any plan that referenced `ExecInitExprWithContext` is now stale.
- **`tablecmds.c::ATExecAddColumn`** reverts to the simpler rule: a domain *with
  any constraints* is treated as effectively volatile for fast-default purposes
  (`!has_domain_constraints` gates the fast path again), restoring the historical
  behavior that a constraint-violating default errors only when the table has
  rows. The `has_volatile` plumbing through `DomainHasConstraints(…, &has_volatile)`
  is gone; the call is now `DomainHasConstraints(atttypid, NULL)`.
- The `fast_default` regress test loses the entire `domain5…domain9` block.

Why it matters: the soft-error-during-DDL idea was a clever way to avoid
rewrites, but evaluating a domain CHECK with `errsave` and interpreting
`escontext.error_occurred` as "fall back to rewrite" was apparently judged
unsound enough to back out rather than patch. Corpus impact: refresh
[[knowledge/files/src/backend/executor/execExpr.c.md]] (the `*WithContext` API no
longer exists), [[knowledge/files/src/include/executor/executor.h.md]], and
[[knowledge/files/src/backend/commands/tablecmds.c.md]] (ADD COLUMN fast-default
rule). See `source/src/backend/commands/tablecmds.c` ATExecAddColumn and
`source/src/backend/executor/execExpr.c` ExecInitExpr/ExecPrepareExpr.

### 2. `fb23cc7e81db` — Fix syslogger NULL-pointer-dereference in EXEC_BACKEND (Álvaro Herrera)

A crash on EXEC_BACKEND platforms (Windows, plus `--enable-exec-backend`
builds). The error-output path decided "am I the syslogger, so write straight to
the file?" via `MyBackendType == B_LOGGER`. On EXEC_BACKEND the syslogger child
sets `MyBackendType = B_LOGGER` **before** it re-opens (`syslogger_fdopen`) the
inherited `syslogFile`/`csvlogFile`/`jsonlogFile` descriptors, so any log message
emitted in that window calls `write_syslogger_file()` on a NULL `FILE *`. In the
non-EXEC_BACKEND fork model the files are already open at fork time, so the bug
didn't reproduce there. `[from-diff]`

The fix introduces a new global `bool syslogger_setup_done`
(`source/src/backend/postmaster/syslogger.c`, declared in
`source/src/include/postmaster/syslogger.h:88`):

- Set `true` immediately in `SysLoggerMain` under `#ifndef EXEC_BACKEND` (files
  already open), and only *after* the `syslogger_fdopen` block under EXEC_BACKEND.
- Every direct-write guard that was `MyBackendType == B_LOGGER` becomes
  `syslogger_setup_done`: `csvlog.c:256`, `jsonlog.c:295`, and three sites in
  `elog.c::send_message_to_server_log` (`elog.c:3834,3848,3888`).

The pattern is a clean "capability ready" flag superseding an "identity" check —
identity (B_LOGGER) was true too early; the new flag tracks the actual
precondition (fds restored). Corpus impact: refresh
[[knowledge/files/src/backend/utils/error/elog.c.md]],
[[knowledge/files/src/backend/utils/error/csvlog.c.md]],
[[knowledge/files/src/backend/utils/error/jsonlog.c.md]] (the B_LOGGER direct-write
test changed), and add the new global to any syslogger note.

### 3. `7d13b03a2e67` — Fix FOR PORTION OF for inheritance children (Peter Eisentraut)

`FOR PORTION OF` is the SQL:2011 temporal `UPDATE/DELETE … FOR PORTION OF
<rangecol> (lo, hi)` that rewrites the touched row into "leftover" slices outside
the affected portion. It was broken for **traditional table inheritance**
children (as opposed to declarative partitions). `[from-diff]`

Two coupled fixes in `nodeModifyTable.c::ExecForPortionOfLeftovers`:

- **Routing vs. direct insert.** The old code keyed off
  `resultRelInfo->ri_RootResultRelInfo` being set — true for *both* partitions
  and inheritance children — and always inserted leftovers through the root with
  tuple routing. For inheritance children that loses child-specific columns. The
  fix computes `partitionRouting` explicitly:
  `rootRelInfo->ri_RelationDesc->rd_rel->relkind == RELKIND_PARTITIONED_TABLE`,
  and only does the child→root map + root redirect when truly partitioned;
  inheritance children insert leftovers directly into the child to preserve their
  columns.
- **Eager per-leaf state init.** The lazy "first time we hit this partition,
  clone a `ForPortionOfState` from the root with our own slot" block is pulled out
  of the hot path into a new `static ExecInitForPortionOf(mtstate, estate,
  resultRelInfo)`. The range attno is now read from `fpoState->fp_rangeAttno`
  (resolved once at init) instead of recomputing `forPortionOf->rangeVar->varattno`
  + child-to-root map remap on every leftover.

`+353/-51` across 4 files including a new `ForPortionOfState` init helper and
`regress/…/for_portion_of.{sql,out}` coverage for the inheritance case. Corpus
impact: refresh [[knowledge/files/src/backend/executor/nodeModifyTable.c.md]] and
[[knowledge/files/src/include/nodes/execnodes.h.md]] (`ForPortionOfState` /
`ri_forPortionOf` lifecycle). Note for [[knowledge/subsystems/executor.md]]: the
"partition vs inheritance child" distinction is now an explicit `relkind` check,
not a `ri_RootResultRelInfo`-presence inference.

### 4. `06e94eccfd91` — Fix missed checks for hashability of container-type equality (Tom Lane)

A correctness bug in the planner's hashability decision for **range and
multirange** equality. `array_eq` and `record_eq` already had special-case
handling: they're marked `oprcanhash`, but whether a *specific* instance is
actually hashable depends on the element/field types, so the code consulted the
typcache (`F_HASH_ARRAY` / `F_HASH_RECORD`). `range_eq` (OID 3882) and
`multirange_eq` (OID 2860) are *also* marked `oprcanhash` but were **not** in
those special cases — so the planner could choose a hash plan for a range whose
subtype isn't hashable, leading to a runtime failure. `[from-diff]`

The fix:

- Gives the two operators OID symbols (`RANGE_EQ_OP`, `MULTIRANGE_EQ_OP` in
  `pg_operator.dat` — compile-time only, no catversion bump).
- Adds a new `get_op_hash_functions_ext(opno, inputtype, …)` in `lsyscache.c`
  that, for the four container ops (array/record/range/multirange), verifies the
  typcache `hash_proc` matches the expected `F_HASH_*` before delegating to
  `get_op_hash_functions`. The doc comments on the old `get_op_hash_functions` /
  `get_compatible_hash_operators` now warn they can succeed for container ops
  whose contained type isn't hashable.
- Extends `op_hashjoinable` (lsyscache.c) to the two range cases.
- Updates the three callers that needed the type-aware check:
  `subselect.c::hash_ok_operator` (adds RANGE_EQ_OP/MULTIRANGE_EQ_OP),
  `clauses.c::convert_saop_to_hashed_saop_walker` (now passes
  `exprType(leftarg)`), and `selfuncs.c::eqjoinsel` (passes the left input type).
- Tidies `typcache.c::lookup_type_cache`: the multirange hashability check is
  folded into the same `else if` chain as array/record/range (it had been a
  separate stanza) for both the standard and extended hash-proc paths.

Why it matters: this is the planner trusting `oprcanhash` too far for
parameterized container types — the same class of bug array_eq/record_eq were
already guarded against, now closed for ranges. Corpus impact: refresh
[[knowledge/files/src/backend/utils/cache/lsyscache.c.md]] (new
`get_op_hash_functions_ext`), [[knowledge/files/src/backend/utils/cache/typcache.c.md]],
[[knowledge/files/src/backend/optimizer/plan/subselect.c.md]],
[[knowledge/files/src/backend/optimizer/util/clauses.c.md]],
[[knowledge/files/src/backend/utils/adt/selfuncs.c.md]],
[[knowledge/files/src/include/utils/lsyscache.h.md]]. Relevant to
[[knowledge/subsystems/optimizer.md]] hashability/oprcanhash discussion.

### 5. `75dcc63dac0e` — Fix publisher retain_dead_tuples check when also changing origin (Amit Kapila)

A one-character logic fix in logical-replication `ALTER SUBSCRIPTION`. When a
single command both flips `retain_dead_tuples` and sets `origin = ANY`, the code
recomputed `check_pub_rdt = retain_dead_tuples && origin==ANY` with `=`, which
**clobbered** a `check_pub_rdt` value already set earlier in the same command
(when `retain_dead_tuples` itself was the changed option). The fix uses `|=` so
the publisher retain-dead-tuples check survives both branches.
(`source/src/backend/commands/subscriptioncmds.c::AlterSubscription`, ~line 1752.)
`[from-diff]` Corpus impact: refresh
[[knowledge/files/src/backend/commands/subscriptioncmds.c.md]]; relevant to the
`retain_dead_tuples` / conflict-detection material in
[[knowledge/subsystems/replication.md]].
