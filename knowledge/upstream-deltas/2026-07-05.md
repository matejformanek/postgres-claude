## Deep explainers

### ★ `a8c2547eaac7` — disallow renaming a rule to `_RETURN` [rewrite]

Bugfix (#19543, reported by Adam Pickering), **backpatch-through 14**. PG has
an ancient convention: an `ON SELECT` rule *must* be named `_RETURN`, and every
other kind of rule must *not* be. Most call sites enforced it, but `ALTER RULE
… RENAME TO` did not — you could rename an ordinary (INSERT/UPDATE/DELETE) rule
to `_RETURN`. That corrupts dump/restore: the regenerated `CREATE RULE` command
then trips the `_RETURN`-must-be-ON-SELECT check and the restore fails.

The fix adds a symmetric guard in `RenameRewriteRule()`
(`source/src/backend/rewrite/rewriteDefine.c:855-863` post-patch): after the
existing "renaming an ON SELECT rule is not allowed" check, if the *new* name
equals `ViewSelectRuleName` (`"_RETURN"`) it now `ereport(ERROR,
ERRCODE_INVALID_OBJECT_DEFINITION, "non-view rule for \"%s\" must not be named
\"%s\"")`. So the two halves of the invariant are now both enforced at rename
time.

While there, Tom removed dead pre-7.3 backward-compat code in
`DefineQueryRewrite()` (`rewriteDefine.c:402` region): the old branch accepted a
`_RETviewname`-style name from ancient `pg_dump` output and silently rewrote it
to `_RETURN` (with the `NAMEDATALEN - 4 - 4` truncation dance). Support for
restoring pre-7.3 dumps was dropped long ago (cites `e58a59975` and nearby), so
this hunk collapses to a plain "must be named `_RETURN`" `ereport`. Net: the
`else`-branch view-rule path is simpler and the `strncmp(rulename, "_RET", 4)` /
`RelationGetRelationName` truncation logic is gone.

Test surface: `src/test/regress/{sql,expected}/rules` gain the rename-to-
`_RETURN` rejection case. Corpus:
`[[knowledge/files/src/backend/rewrite/rewriteDefine.c.md]]` needs a re-anchor
row (its recorded "Last verified commit" is `ef6a95c7c64` from 2026-06-01 —
badly stale, both the `RenameRewriteRule` guard and the `DefineQueryRewrite`
simplification are new). Cross-cuts `parser-and-rewrite.md` subsystem doc
(rule-name invariant). Relevant skill: `error-handling` (the
`ERRCODE_INVALID_OBJECT_DEFINITION` choice mirrors the sibling check).

### ★ `0e4f0827f639` — clean up properties orphaned by dropping a label [commands, property-graph]

Bugfix in the SQL/PGQ property-graph DDL (authors Ashutosh Bapat + zengman;
committer Peter Eisentraut). `AlterPropGraph()` in
`source/src/backend/commands/propgraphcmds.c` deletes `pg_propgraph_property`
rows that become orphaned when you drop an element or drop properties of an
element — but it missed the case where the properties are orphaned by dropping
a **label**. A label can carry properties; dropping the label left those
`pg_propgraph_property` catalog rows dangling.

The fix is a one-token predicate widening at
`propgraphcmds.c:1677` (post-patch): the "remove any orphaned
`pg_propgraph_property` entries" block guard grows a `|| stmt->drop_label`
term —

```c
if (stmt->drop_properties || stmt->drop_vertex_tables ||
    stmt->drop_edge_tables || stmt->drop_label)
```

— so the existing `foreach_oid(propoid, get_graph_property_ids(pgrelid))`
orphan-sweep now also runs on the drop-label path. The sweep body (which checks
each property against its still-live label/element associations before
`deleteDependencyRecordsFor`/catalog delete) was already correct; it just wasn't
being reached.

Test surface: `src/test/regress/{sql,expected}/create_property_graph` gain a
case that drops a label and asserts only the *orphaned* property (`zz` in the
test) disappears while properties still reachable via another label survive.
Corpus: `[[knowledge/files/src/backend/commands/propgraphcmds.c.md]]` re-anchor
(recorded anchor `e18b0cb7344`, now stale). This is the second property-graph
orphan-cleanup fix in the recent window — cf. `96418a6da9d0` ("Fix handling of

### ★ `9f03dab4574b` — simplify `dxsyn_lexize()` [contrib/dict_xsyn, tsearch]

Cleanup (author Ayush Tiwari, reviewed by Tom Lane; from thread #19525). The
extended-synonym text-search dictionary's lexize function built a temporary
NUL-terminated copy of its input before lowercasing:

```c
/* before */
char *temp = pnstrdup(in, length);
word.key = str_tolower(temp, length, DEFAULT_COLLATION_OID);
pfree(temp);
/* after */
word.key = str_tolower(in, length, DEFAULT_COLLATION_OID);
```

The copy existed only because pre-v18 this code called `lowerstr()`, which
required a NUL-terminated string. Since the switch to `str_tolower()` (which
takes an explicit `length` and tolerates non-NUL-terminated input) the
`pnstrdup`/`pfree` round-trip is pure waste. Removing it drops an allocation per
lexize call on the dict_xsyn hot path. No behavior change; no test change.

Corpus: `[[knowledge/files/contrib/dict_xsyn/dict_xsyn.c.md]]` has a
`[verified-by-code]` cite at `dict_xsyn.c:77-143` — the line numbers around the
lexize function shift, so re-anchor. Also `[[knowledge/subsystems/contrib-dict_xsyn.md]]`
if it cites the temp-buffer detail. Good worked example for the
`memory-contexts` idiom: `str_tolower`'s length-aware contract removing a
defensive copy is exactly the "don't `pstrdup` when the callee already takes a
length" pattern.
