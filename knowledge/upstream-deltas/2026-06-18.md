## Deep dives

### `e5c40584` — replication: avoid errors during ALTER SUBSCRIPTION

`GetSubscription(Oid, missing_ok, aclcheck)` gained a new shape:
`GetSubscription(Oid, missing_ok, conninfo_needed, conninfo_aclcheck)`
(`src/include/catalog/pg_subscription.h`, `src/backend/catalog/pg_subscription.c`
near `GetSubscription`). The struct is now `palloc0_object(Subscription)` so the
conninfo field is reliably NULL when not requested, with
`Assert(conninfo_needed || !conninfo_aclcheck)` guarding the flag combination.
The conninfo construction (which calls `GetForeignServer()` +
`ForeignServerConnectionString()` and can `ereport(ERROR)` when the subscription
owner has lost `ACL_USAGE` on the foreign server) is now skipped entirely for
ALTER paths that only touch local catalog state. `replication/logical/worker.c`
is updated to pass `conninfo_needed=true` where it actually connects. Net effect:
`ALTER SUBSCRIPTION ... (some local option)` no longer fails just because the
foreign-server grant was revoked out from under it.
[[corpus]] `knowledge/files/src/backend/catalog/pg_subscription.c.md`,
`knowledge/files/src/backend/commands/subscriptioncmds.c.md`,
`knowledge/files/src/backend/replication/logical/worker.c.md`,
`knowledge/files/src/include/catalog/pg_subscription.h.md`. The
`GetSubscription` signature change is an **internal-API break** any out-of-tree
caller must follow. `[verified-by-code]` (diff post-image).

### `702e9dfd` — replication: DROP SUBSCRIPTION with slot_name = NONE

Sibling fix to `e5c40584` on the DROP path. `DropSubscription()`
(`src/backend/commands/subscriptioncmds.c`) previously built the foreign-server
connection string eagerly via the inline ACL-check + `ForeignServerConnectionString()`
block, which `ereport`s when the owner lost `ACL_USAGE`. That block is extracted
into a new static `construct_subserver_conninfo(subserver, subowner, **err)` and
is only invoked when a replication slot actually has to be dropped on the
publisher. The `subconninfo` text is now read up front via `SysCacheGetAttr`
(nullable) instead of `SysCacheGetAttrNotNull`. Result: `DROP SUBSCRIPTION` with
`slot_name = NONE` (the "I'll clean the slot myself" path) succeeds even when the
foreign-server connection can no longer be constructed.
[[corpus]] `knowledge/files/src/backend/commands/subscriptioncmds.c.md`.
`[verified-by-code]`.

### `64797ad9` — ALTER DOMAIN VALIDATE CONSTRAINT locking

`validateDomainCheckConstraint()` (`src/backend/commands/typecmds.c`) loses its
`LOCKMODE lockmode` parameter; it now always passes `ShareLock` to
`get_rels_with_domain()`. Previously `AlterDomainValidateConstraint()` passed the
weaker `ShareUpdateExclusiveLock` on the theory that a not-yet-valid constraint
is still enforced against concurrent DML — but the new isolation test
`src/test/isolation/specs/alter-domain-validate.spec` shows the hole: a
concurrent `INSERT` of a domain value that violates the constraint can commit
*after* VALIDATE has scanned, leaving a validated constraint with a violating
row. With `ShareLock`, VALIDATE waits for in-flight DML to finish (the test
permutation shows `s3_validate` blocking on `s2_insert` and then correctly
erroring: "column ... contains values that violate the new constraint",
`convalidated = f`). This is a **lock-strength correctness fix**; see
`knowledge/subsystems/locking.md` for the heavyweight-lock taxonomy.
[[corpus]] `knowledge/files/src/backend/commands/typecmds.c.md`.
`[verified-by-code]`.

### `850b9218` — pgstat: PANIC with track_functions under concurrent drop

`pgstat_drop_entry(kind, dboid, objid)` becomes
`pgstat_drop_entry(kind, dboid, objid, bool missing_ok)`
(`src/backend/utils/activity/pgstat_shmem.c`, declared in the pgstat-internal
header). The hard `elog(ERROR, "trying to drop stats entry already dropped...")`
inside `pgstat_drop_entry_internal()` is downgraded to `Assert(!shent->dropped)`;
the live race-handling moves up into `pgstat_drop_entry()`, which now checks
`shent->dropped` and, when `missing_ok`, releases the dshash lock and returns
`true` instead of erroring. The five call sites are updated:
`pgstat_init_function_usage()` (`pgstat_function.c`) and the two
`AtEOXact_PgStat_DroppedStats()` calls (`pgstat_xact.c`) pass `missing_ok=true`
— these are exactly the `track_functions` / end-of-xact paths that can race a
concurrent drop of the same pgstats entry; the backend-shutdown
(`pgstat_shutdown_hook`) and replslot (`pgstat_replslot.c`) calls keep the strict
`false`. Fixes a reproducible PANIC where two backends drop the same function
stats entry concurrently. Touches the shared dshash under the
`PgStatShared_HashEntry` lock — see `knowledge/subsystems/locking.md`.
[[corpus]] `knowledge/files/src/backend/utils/activity/pgstat.c.md`,
`pgstat_shmem.c.md`, `pgstat_function.c.md`, `pgstat_replslot.c.md`,
`pgstat_xact.c.md`. The `pgstat_drop_entry` signature change is an
**internal-API break**. `[verified-by-code]`.

### `bdae2c20` — slotsync: avoid stale slot access after drop

In `drop_local_obsolete_slots()` (`src/backend/replication/logical/slotsync.c`),
the post-drop `ereport(LOG, "dropped replication slot ...")` and the matching
`UnlockSharedObject(DatabaseRelationId, ...)` were reading
`local_slot->data.database` and `local_slot->data.name` *after*
`ReplicationSlotDropAcquired()` had already freed/recycled that shmem slot — a
use-after-free on the slot's `ReplicationSlotPersistentData`. The fix snapshots
`Oid slot_database = local_slot->data.database` before taking the database lock
and `NameData slot_name = local_slot->data.name` before the acquire/drop, and
the LOG + unlock use the locals. This is the standdard "copy out shmem fields
before you release the object that owns them" idiom.
[[corpus]] `knowledge/files/src/backend/replication/logical/slotsync.c.md`.
`[verified-by-code]`.

### `29fb598b` — procarray: GetSnapshotData() OOM cleanup

In `GetSnapshotData()` (`src/backend/storage/ipc/procarray.c`, the lazy-alloc
block around the `snapshot->subxip` malloc), if the `subxip` array malloc fails,
the code previously `ereport(ERROR)`'d while leaving `snapshot->xip` (allocated
just above) dangling and non-NULL. A subsequent snapshot reuse on the same
`SnapshotData` would then see a partially-initialized snapshot. The fix
`free(snapshot->xip); snapshot->xip = NULL;` before the `ereport`, so a retry
starts clean. Narrow robustness fix on the hottest shmem read path in the
backend — see `knowledge/subsystems/locking.md` and snapshot lifecycle notes.
[[corpus]] `knowledge/files/src/backend/storage/ipc/procarray.c.md`.
`[verified-by-code]`.

### `ff8bec8c` — tablecmds: TOAST table for MERGE/SPLIT PARTITION

`createPartitionTable()` (`src/backend/commands/tablecmds.c`, right after the
`CommandCounterIncrement()` that follows partition creation) now calls
`NewRelationCreateToastTable(newRelId, (Datum) 0)`. MERGE/SPLIT PARTITION moves
rows from existing partitions into the freshly-created partition; if a moved row
carries an out-of-line (externalized) varlena value and the new partition has no
TOAST table, there is nowhere to store it. The regress additions
(`partition_merge.sql` / `partition_split.sql`) force the bug with
`ALTER TABLE ... ALTER COLUMN a SET STORAGE EXTERNAL` + a 10 000-char value, then
assert `reltoastrelid <> 0` on the new partition(s). This is a **data-integrity
fix** (prior behavior could fail or mis-store wide values on merged/split
partitions). [[corpus]] `knowledge/files/src/backend/commands/tablecmds.c.md`.
`[verified-by-code]`.

### `1f2297b5` — autovacuum: division-by-zero in MXID score

`relation_needs_vacanalyze()` (`src/backend/postmaster/autovacuum.c`) computes
`scores->mxid = (double) mxid_age / multixact_freeze_max_age`. The effective
`multixact_freeze_max_age` can be driven to **0** when multixact-member space
usage is high (the failsafe lowers the effective age aggressively), giving a
division by zero. The fix divides by `Max(1, multixact_freeze_max_age)` and the
companion `doc/src/sgml/maintenance.sgml` change documents that the relminmxid
score also scales up as the effective age is lowered by member-space pressure.
This is in the recently-added autovacuum **scoring** code (a 2026-06 area the
corpus and the user-question-harvester have both been tracking).
[[corpus]] `knowledge/files/src/backend/postmaster/autovacuum.c.md`.
`[verified-by-code]`.

### `c0f17b04` — hstore_plperl: interruptible reference unwinding

`plperl_to_hstore()` (`contrib/hstore_plperl/hstore_plperl.c`) unwinds Perl
references with `while (SvROK(in)) in = SvRV(in);`. A circular reference makes
that an infinite, uninterruptible loop. The fix adds `#include "miscadmin.h"`
and a `CHECK_FOR_INTERRUPTS()` inside the loop — cheap, and lets a user cancel.
This is the same hardening pattern Tom Lane applied to `jsonb_plperl` /
`jsonb_plpython` the day before (`da82fbb8`, see 2026-06-17 deltas).
[[corpus]] `knowledge/files/contrib/hstore_plperl/hstore_plperl.c.md`.
`[verified-by-code]`.
