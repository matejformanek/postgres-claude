## Deep explainers (corpus-touching + notable)

### The btree_gist sweep (7 commits, Tom Lane + Ayush Tiwari)

All seven share one `Discussion:` thread (王跃林's bug report,
`postgr.es/m/AH*AvQCYKhQGVvPWi1GiU4oY...`) and form a single coordinated pass
over `contrib/btree_gist`'s varlena/numeric consistency machinery. Ordered
newest→oldest they read as *fix the crash → tighten the bound logic → delete
the dead code → document what remains → normalize the signatures*.

#### `eef644e57c38` — Fix NotEqual on internal pages (the crash fix, backpatch 14)

The load-bearing bug. `gbt_var_consistent()` / `gbt_num_consistent()` handled
the `<>` (`BtreeGistNotEqualStrategyNumber`) strategy **without** the
leaf-vs-internal split that every other strategy uses. It called the datatype
`f_eq` method on *internal* keys — but internal (non-leaf) GiST keys can have a
different physical representation than leaf keys, so this is an OOB read,
potentially a crash, and generally wrong results.

The fix, in `btree_utils_var.c` `gbt_var_consistent()` (`[[knowledge/files/contrib/btree_gist/btree_utils_var.c.md]]`):

```c
case BtreeGistNotEqualStrategyNumber:
    if (is_leaf)
        retval = !(tinfo->f_eq(query, key->lower, collation, flinfo));
    else
        /* descend unless query == both bounds AND bounds aren't truncated */
        retval = tinfo->trnc || !(f_eq(query,lower) && f_eq(query,upper));
```

On a **leaf** page an entry has `lower == upper`, so `<>` is just the inverse of
`=` against `lower` — one comparison, not two. On an **internal** page we may
prune only when the query equals both bounds *and the bounds are not truncated*
(`tinfo->trnc`); a truncated bound hides the true range below, so we must
descend. `gbt_num_consistent()` gets the same shape (`btree_utils_num.c:297`)
for readability even though numeric keys don't truncate. Reported-by 王跃林,
Author Ayush Tiwari, Reviewed-by Tom Lane. **Backpatch-through: 14.**

#### `fea9c1884b20` — Tighten truncated-bound handling

Truncating an internal node's **upper** bound can make it compare *less* than
values actually present on the page, so there's a deliberate hack to inflate it;
the **lower** bound has no equivalent issue. The old code applied fuzzy
comparison to *both* bounds ("fuzzy thinking"). This removes the useless
lower-bound check in `gbt_var_node_pf_match`, and introduces `lower_is_below_query`
/ `upper_is_above_query` macros so the strategy cases test directly instead of
routing through `gbt_var_node_pf_match`. All tests were flipped to
`indexkey OP query` so the code matches the strategy names. `gbt_num_consistent`
was reshaped to match (no functional change there).

#### `b82d69abf64f` — Delete dead encoding-aware truncation logic

`gbt_var_node_cp_len()` guarded against splitting a multibyte character when
choosing a common-prefix length — but that path has been dead since `ef770cbb6`
removed text truncation. Deleting it also removes the last user of
`gbtree_vinfo.eml`, letting the `gbtree_vinfo` structs become `const` (a small
step toward a thread-safe backend). Tom notes the irony that `ef770cbb6` both
added *and* obsoleted this infrastructure.

#### `a9fa6c69e3f5` — Wrong comparator in gbt_bit_ssup_cmp (one-liner, user-visible)

`gbt_bit_ssup_cmp` (the sort-support comparator added for bit/varbit in v18,
`e4309f73f`) called `byteacmp` where leaf entries need `bitcmp`. No crash, but
indexes were built in the wrong sort order and are therefore *less efficient
than before the sort-support code existed*. **Operational note carried in the
commit: users may want to REINDEX btree_gist indexes on `bit`/`varbit` columns.**

#### `4b808ed77cd9` (signature sync) + `d6284bbd152c` (docs)

Cosmetic: pass the strategy number by value (not `const StrategyNumber *`) and
`const`-qualify the `key` input, syncing the two `*_consistent` signatures
across all 21 btree_gist type modules; and a large comment-only pass explaining
the internal-vs-leaf key representation and truncation invariants (Tom flags
that he suspects *more* latent bugs there, left for later). Not backpatched.

### `c40819ebf954` — Empty other-session temp tables (Daniil Davydov, backpatch 16)

`ce146621` made reads of another session's temp-table pages raise an error, but
an **empty** target slipped through: an `INSERT` goes through `hio.c` →
relation-extension, which `31966b151e6` (PG 16) split into
`ExtendBufferedRelLocal()` — a path that never had the `RELATION_IS_OTHER_TEMP`
check. Result: the extend "succeeds", leaving an invalid local buffer that later
fails to flush with *"no such file or directory"*. The fix adds the check at the
canonical spot, `ExtendBufferedRelCommon()` in
`bufmgr.c:2769` (`[[knowledge/files/src/backend/storage/buffer/bufmgr.c.md]]`):

```c
if (bmr.relpersistence == RELPERSISTENCE_TEMP)
{
    if (bmr.rel && RELATION_IS_OTHER_TEMP(bmr.rel))
        ereport(ERROR,
            (errcode(ERRCODE_FEATURE_NOT_SUPPORTED),
             errmsg("cannot access temporary tables of other sessions")));
    first_block = ExtendBufferedRelLocal(...);
}
```

The `RELATION_IS_OTHER_TEMP` doc comment in `rel.h:668` is updated to add
`ExtendBufferedRelCommon()` to the list of buffer-access entry points that
enforce the rule. Test `013_temp_obj_multisession.pl` gains an empty-table
INSERT case (backpatched to 16 alongside the fix). **Backpatch-through: 16.**

> **Corpus impact:** `knowledge/subsystems/storage-buffer.md` should note that
> `ExtendBufferedRelCommon` is now an other-temp gatekeeper. `rel.h` has **no**
> per-file doc (coverage gap — flagged in the refresh report).

### `b22b619056e8` — libpq trace of variable-length key (Anthonin Bonnefoy, backpatch 18)

`a460251f0a` grew `BackendKeyData` from a fixed 4-byte key to variable length
(up to 256 bytes) for the longer cancel keys. `fe-trace.c` still decoded it as a
single `Int32`, so `PQtrace` emitted a "mismatched message length" warning; the
same bug hit the `CancelRequest` (no-type-byte) path. Fix swaps the second
`pqTraceOutputInt32` for `pqTraceOutputNchar(length - *cursor + 1, ...)` in both
`pqTraceOutput_BackendKeyData` and `pqTraceOutputNoTypeByteMessage`
(`[[knowledge/files/src/interfaces/libpq/fe-trace.c.md]]`). **Backpatch-through: 18.**

### REPACK CONCURRENTLY — two fixes (Antonin Houska, backpatch 19)

`REPACK [CONCURRENTLY]` is new in the v19 line; both fixes are backpatched to 19.

- **`3be823486f2c`** — concurrent-change replay of a table with **stored
  generated columns** needs the transient table's `pg_attrdef` tuples present
  (currently *every* concurrently updated/inserted tuple re-stores the generated
  value). The fix adds `copy_attribute_defaults(old_heap_oid, new_heap_oid)` to
  clone them; ships a regression test that reproduces the bug.
- **`5ee9d7c299f2`** — `initialize_change_context()` was passing a bare
  `InitResultRelInfo` with no range table. It now cons's a real `RangeTblEntry`
  + `RTEPermissionInfo`, sets `updatedCols` to *all* non-dropped columns, and
  calls `ExecInitRangeTable`. Only affects btree indexes (which optimize the
  unchanged-column case); "all updated" errs to the safe side vs. the prior
  under-claiming. Both in `repack.c`
  (`[[knowledge/files/src/backend/commands/repack.c.md]]`).

### SQL/PGQ property-graph fixes (Peter Eisentraut area, 3 commits)

The property-graph (SQL/PGQ) feature keeps getting hardened:

- **`efd7d8d7d495`** — `GRAPH_TABLE(... COLUMNS('lit' AS c, NULL AS n))` left
  the output columns as `unknown`; `transformRangeGraphTable` now calls
  `resolveTargetListUnknowns` (gated on `p_resolve_unknowns`) so they resolve to
  `text`, matching normal SELECT behavior. `parse_clause.c:1008`
  (`[[knowledge/files/src/backend/parser/parse_clause.c.md]]`).
- **`96418a6da9d0`** — `DROP PROPERTY` by name checked only that the property
  existed in the graph catalog, not that it was on the *given label*, so it
  could pass `InvalidOid` to `performDeletion`. Now verifies the
  label↔property association first. `propgraphcmds.c`
  (`[[knowledge/files/src/backend/commands/propgraphcmds.c.md]]`).
- **`7afa11feca6c`** — enforce the SQL/PGQ rule that every graph element keeps
  ≥1 label: error if the drop would remove the last one. Safe against concurrent
  drops because property-graph modification holds `ShareRowExclusiveLock`.
