## Deep explainers (fresh 9)

### `c787c952bc88` — bufmgr: Fix order of operations in `UnlockBufHdrExt` · Andres Freund

A correctness fix to the CAS-based buffer-header unlock helper introduced this
cycle (`c75ebc657ffc`, PG 19). `UnlockBufHdrExt()` atomically sets bits, clears
bits, and adjusts refcount in one compare-and-swap. The buggy version **added
`set_bits` before subtracting `unset_bits`**, so a flag that was simultaneously
in both masks ended up *cleared*. The only known live consequence: `BM_IO_ERROR`
could fail to stay set on a buffer that both cleared and re-set it, silently
losing the I/O-error marker. The fix reorders to *unset first, then set*, so a
flag present in both masks ends up **set** — documented in the reworked header
comment. One-file change to the inline helper.
- `source/src/include/storage/buf_internals.h:465` (`UnlockBufHdr` comment),
  `:484` (`UnlockBufHdrExt` CAS body) `[verified-by-code]`
- Backpatch-through 19 (where `c75ebc657ffc` landed). No decent race-free
  regression test was addable; deferred to the 20 cycle per the message.
- Corpus: `[[knowledge/subsystems/storage-buffer]]`,
  `[[knowledge/idioms/hint-bits-setbufferdirty]]`, per-file
  `knowledge/files/src/include/storage/buf_internals.h.md`. The `BM_IO_ERROR`
  state-bit semantics and the CAS-unlock helper are exactly the buffer-manager
  invariants that doc tracks. `[inferred]` impact.

### `c71d43025d7a` — Don't lock tables in `get_tables_to_repack()` · ChangAo Chen (via Álvaro Herrera)

Cleanup + hardening of the new `REPACK` command (PG 19). When repacking a whole
database, the table list was built taking a transaction-scoped lock on each
relation — but that lock was released immediately when the list-building
transaction ended, making it useless (and only one of the three list-building
paths even took it). The commit removes that lock acquisition, mirroring how
`VACUUM` builds its list in `get_all_vacuum_rels()`, and instead **hardens the
later real-lock path** against concurrent drops:
- `repack_is_permitted_for_relation()` now copes with the relation vanishing
  mid-check (`source/src/backend/commands/repack.c:2162`, `:2178`, `:2223`).
- `ExecRepack()` switches `try_table_open()` → `try_relation_open()` and checks
  relkind separately, so a relation that morphed (table/matview → something
  else) is skipped rather than erroring (`repack.c:375`, `:453`).
- One `relation_close()` → `table_close()` to match the `table_open()` on the
  catalog (`repack.c:2249`). `[verified-by-code]`
- Backpatch-through 19. Corpus: `[[knowledge/subsystems/commands]]`,
  `[[knowledge/scenarios/add-a-utility-command]]`, `[[knowledge/idioms/locking]]`
  (the "lock, release-at-xact-end is useless" pattern), per-file
  `knowledge/files/src/backend/commands/repack.c.md`.

### `97a18c221b8a` — Fix data-checksum processing for temp relations and dropped databases · Fujii Masao

Two independent bugs in the online data-checksum enable/disable machinery (the
`datachecksum_state.c` launcher/worker in PG 19):
1. **Temp-relation filter.** `BuildRelationList()` included temp relations
   *without storage* (e.g. temp views), so enabling checksums could block
   forever waiting on a session that owned only a temp view. Now filtered to
   temp relations **with storage**, matching the non-temp path
   (`source/src/backend/postmaster/datachecksum_state.c:1530`, `:1548`).
2. **Shared-catalog double-mark.** The launcher assigns the first worker to
   process shared catalogs. If that worker's DB was dropped after selection but
   before processing, the shared catalogs were still marked processed, so later
   workers skipped them — checksum-enable could "succeed" with shared catalogs
   never checksummed. Fixed by marking shared catalogs processed **only after a
   worker completes successfully**, with a new `DATACHECKSUMSWORKER_DROPDB`
   result branch in `ProcessAllDatabases()` (`datachecksum_state.c:1352`).
   `[verified-by-code]`
- Backpatch-through 19. Corpus: `[[knowledge/subsystems/process-model]]`,
  `[[knowledge/skills/process-lifecycle]]`, per-file
  `knowledge/files/src/backend/postmaster/datachecksum_state.c.md` (note: this
  file is a repeat-offender — the 06-25/06-30 anchor-refresh clusters already

### `e9810253f095` — Fix data-checksum progress-counter initialization · Fujii Masao

Sibling to the above; fixes the `pg_stat_progress_data_checksums` view. The
progress framework's `pgstat_progress_start_command()` zeroes all counters, but
data-checksum progress uses **-1 as the "display-as-NULL" sentinel** and never
reset the counters to -1 — so counters that should read NULL showed 0 (workers
reporting zero database counters; the disabling launcher reporting zero
relation/block counters). Also `blocks_done` was not reset when a worker moved
to a new relation fork, letting it exceed `blocks_total` or show stale values on
empty forks. Fix: initialize the progress counters to -1 at report-start for
both launcher and worker, and reset `blocks_done` together with `blocks_total`
per fork.

  (view), `doc/src/sgml/monitoring.sgml:8095` `[verified-by-code]`

### `dfce19c2300a` — Forbid `FOR PORTION OF` on views with `INSTEAD OF` triggers · Aleksander Alekseev / Paul A. Jungwirth

Crash fix for the temporal-tables `FOR PORTION OF` feature (`8e72d914c528`).
Combining `FOR PORTION OF` with an updatable view that has an `INSTEAD OF`
trigger previously **crashed**. The fix adds an explicit guard in the
**rewriter** (not the parser — the commit's tests deliberately demonstrate the
check belongs in `RewriteQuery`, where the view/trigger shape is known):
```c
if (parsetree->forPortionOf &&
    rt_entry_relation->rd_rel->relkind == RELKIND_VIEW &&
    view_has_instead_trigger(rt_entry_relation, event, NIL))
    ereport(ERROR, (errcode(ERRCODE_FEATURE_NOT_SUPPORTED),
        errmsg("views with INSTEAD OF triggers do not support FOR PORTION OF")));
```
- `source/src/backend/rewrite/rewriteHandler.c:4171` (`RewriteQuery`)
  `[verified-by-code]`; regress coverage in `updatable_views.sql`. No backpatch
  trailer (master-only; `8e72d914` is a PG 19-cycle feature).

### `54cd6fc83176` — postgres_fdw: remove SPI from `postgresImportForeignStatistics` · Corey Huinker / Etsuro Fujita

The headline fresh commit. The new remote-statistics import path in
`postgres_fdw` (from `28972b6fc`) imported remote stats by calling the SQL
functions `pg_restore_relation_stats` / `pg_restore_attribute_stats` **via SPI
in read-write mode**. Those SQL functions take *schema-name + relation-name*
(two text args) rather than an OID, so a concurrent `ALTER SCHEMA ... RENAME`
between name-resolution and the SPI call produced `ERROR: schema "foo" does not
exist`. Two-part fix:
1. **New OID-taking C interface functions** callable directly from FDWs, added
   to the backend stats API (`src/include/statistics/statistics.h`,
   `src/backend/statistics/relation_stats.c`, `attribute_stats.c`). The guts of
   `relation_statistics_update` / `attribute_statistics_update` are factored
   into new OID-argument functions so both the SQL-callable workhorses and the
   new FDW interface share one code path.
2. **`postgres_fdw` switches** `postgresImportForeignStatistics()` /
   `fetch_remote_statistics()` off SPI onto the new single-OID interface —
   deleting a ~130-line block of SPI-restore plumbing
   (`source/contrib/postgres_fdw/postgres_fdw.c:371` shows a 136-line deletion).
- `source/contrib/postgres_fdw/postgres_fdw.c:5542`
  (`postgresImportForeignStatistics`), `:5612` (`fetch_remote_statistics`)
  `[verified-by-code]`; new API documented in `doc/src/sgml/fdwhandler.sgml`.

### `999a8412bbab` — Improve `log_statement_max_length` truncation reporting · Jim Jones

Three-part improvement to the `log_statement_max_length` GUC (PG 19). (1) The
prepared-statement text shown in the `EXECUTE` **DETAIL** message
(`errdetail_execute()`) bypassed the limit and was logged in full; now the same
truncation applies there. (2) Truncated text gets a trailing **ellipsis** so a
limit of 0 logs just the ellipsis, making truncation visible. (3) Instead of
scanning the whole query to decide whether to truncate, use `strnlen()` with
lookahead and `pg_mbcliplen()` so truncation never splits a multibyte char.
