### Deep 1 — `f2081a78`: race in `ReplicationSlotRelease()` for ephemeral slots

`src/backend/replication/slot.c`, `ReplicationSlotRelease()` ~L788.
`[verified-by-code]` (from diff). [[knowledge/files/src/backend/replication/slot.c.md]]

The function drops an ephemeral (`RS_EPHEMERAL`) slot via
`ReplicationSlotDropAcquired()` in the early branch. The bug: after that drop the
code *fell through* and still touched the now-freed/recycled slot — reading
`slot->data.xmin`, `slot->effective_xmin`, `slot->data.persistency`, and writing
`MyReplicationSlot = NULL` — even though for the dropped-slot path those reads
race against another backend that may have already re-acquired the freed
`ReplicationSlot` array entry.

The fix wraps the entire "restrain xmin / set inactive-since / mark persistent
inactive" block in an **`else`** so it runs **only when the slot was not
dropped**:

```c
if (slot->data.persistency == RS_EPHEMERAL) {
    ...
    ReplicationSlotDropAcquired(is_logical);
}
else {
    /* effective_xmin cleanup, GetCurrentTimestamp(), persistency handling,
       MyReplicationSlot = NULL  — all moved inside the else */
}
```

Net diff is `+35 −33`, one file: a pure control-flow re-indent, no new logic.
The teachable invariant: **after `ReplicationSlotDropAcquired()` the slot pointer
is dead — never dereference `slot` again on that path.** Worth an INV note in the
slot.c corpus doc if it tracks acquire/release lifecycle.

### Deep 2 — `ffeda042`: eager aggregation must skip the inner side of SEMI/ANTI joins

`src/backend/optimizer/util/relnode.c`,
`eager_aggregation_possible_for_relation()` ~L2845, plus a new paragraph in
`src/backend/optimizer/README` (~L1591). `[verified-by-code]` (from diff).
[[knowledge/files/src/backend/optimizer/util/relnode.c.md]]

Eager aggregation (the PG19 "push a partial GROUP BY below a join" optimization)
must not push a partial aggregate onto a relation that sits on the **inner (RHS)
side of a semi/anti join**: such a join does not preserve its inner rows in the
output, so a partial aggregate computed there "would not survive the join and
could not be combined by the final aggregation." Pushing it down yields **wrong
results**.

The fix adds a scan of `root->join_info_list`; for each `SpecialJoinInfo` with
`jointype == JOIN_SEMI || JOIN_ANTI` it bails when the candidate rel is inside
the join's inner side but not its outer side:

```c
if (bms_overlap(rel->relids, sjinfo->min_righthand) &&
    !bms_is_subset(sjinfo->min_lefthand, rel->relids))
    return false;
```

Subtlety (called out in the README hunk): an **anti join reduced from an outer
join** null-extends its inner side, so that rel already carries `nulling_relids`
and is caught by the *existing* outer-join check above — the new check
specifically targets semi/anti joins that do **not** null-extend, i.e. those from
`EXISTS / IN / NOT EXISTS / NOT IN` sublinks. Regression coverage added in
`src/test/regress/sql/eager_aggregate.sql`: a `NOT EXISTS` case proves the
partial agg is *not* pushed to the inner (plan shows a plain `Hash Anti Join`),
while an `EXISTS` case proves it is still pushed to the **outer** side (plan
shows `Partial HashAggregate` under a `Hash Right Semi Join`).

### Deep 3 — `ee2fa291`: wrong unsafe-flag bit in `check_output_expressions()`

`src/backend/optimizer/path/allpaths.c`, `check_output_expressions()` L4338.
`[verified-by-code]` (from diff). [[knowledge/files/src/backend/optimizer/path/allpaths.c.md]]

A one-character correctness fix in subquery-pushdown safety analysis. When the
subquery has window functions, point 4 of the qual-pushdown check must skip a
target entry that is not present in all `PARTITION BY` lists — but the test
masked against the **wrong** unsafe bit:

```c
-  UNSAFE_NOTIN_DISTINCTON_CLAUSE) == 0 &&
+  UNSAFE_NOTIN_PARTITIONBY_CLAUSE) == 0 &&
```

It was reading the `DISTINCT ON` flag where it should read the `PARTITION BY`
flag. Effect: a wrong/incomplete safety decision for pushing quals into a
subquery that uses window functions (the wrong bit could let an unsafe qual
through or block a safe one). `+1 −1`, no test in the commit — a pure logic
typo-class fix in a hot planner path.

### Deep 4 — `273a3c77`: copy-paste bug in `hash_record_extended()`

`src/backend/utils/adt/rowtypes.c`, `hash_record_extended()` L2030.
`[verified-by-code]` (from diff). (No corpus per-file doc for `rowtypes.c`; not
in the optimizer/storage/access "interesting path" set, but it's a real hashing
correctness bug, so noted here.)

Setting up the per-element `FunctionCallInfo` for the element hash function, the
code set `args[0].isnull` **twice** instead of setting `args[1].isnull`:

```c
   locfcinfo->args[1].value = Int64GetDatum(seed);
-  locfcinfo->args[0].isnull = false;
+  locfcinfo->args[1].isnull = false;
```

So the seed argument's `isnull` was left uninitialized (whatever the stack/reuse
held). For the extended (seeded) record-hash path this is the classic
uninitialized-`isnull` hazard — could feed a garbage NULL flag to the element's
hash support function. `+1 −1`. Affects hashing of composite/record values, e.g.
`hash partitioning` and hash joins/aggregation on `record` columns.
