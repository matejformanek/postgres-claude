## Deep explainers

### #4 `d5751c33cc3e` — Recovery-target GUCs: validation out of the assign hooks (Fujii Masao)

The load-bearing commit of the window. `recovery_target`,
`recovery_target_lsn/name/time/xid` are `PGC_POSTMASTER` GUCs, but their assign
hooks used to do more than store a value: they mutated the global
`recoveryTarget` state and could `ereport(ERROR)` when they thought more than one
target was set. Two problems: (a) assign hooks must not throw — the GUC machinery
is not longjmp-safe at assign time (`error-handling`); (b) deriving cross-GUC
state during per-GUC assignment makes the result **order-dependent**. The commit
message's concrete failure: set one target, then set another `recovery_target_*`
to `''`, and the empty-string assign could clear `recoveryTarget` even though a
valid target remained.

The fix strips the assign hooks in `xlogrecovery.c` down to storing their own
value (7 hunks in the `check_recovery_target*` region,
`xlogrecovery.c:4774-5064`), removes the now-dead `check_recovery_target`
cross-checks, and derives the effective `recoveryTarget` **once** in
`validateRecoveryParameters()`, which rejects multi-target configs with `FATAL`
(right elevel for startup config). `xlogrecovery.h:139` and `guc_hooks.h:103-115`
lose the retired hook signatures; `guc_parameters.dat` + `guc_tables.c` drop the
corresponding assign-hook wiring. Empty string now uniformly means "not set."
`[[knowledge/files/src/backend/access/transam/xlogrecovery.c.md]]` —
the assign-hook notes need the "hooks only store; validation is centralized in
`validateRecoveryParameters`" correction, plus the order-independence invariant.
Also touches `[[knowledge/files/src/include/access/xlogrecovery.h.md]]` and
`[[knowledge/files/src/include/utils/guc_hooks.h.md]]`. This is the change skink
tripped over (buildfarm #1 below) and #1/#5 clean up after.

### #9 `20fe90bbb93d` — FOR PORTION OF leftover rows skipped INSERT RLS (Peter Eisentraut)

A real RLS-bypass bug. `UPDATE/DELETE ... FOR PORTION OF` preserves the parts of
a row outside the target range by re-inserting "leftover" rows; those inserts go
through `ExecInsert()`, which enforces `WCO_RLS_INSERT_CHECK`. But the rewriter
only added WITH CHECK options for the *statement's* command — `UPDATE` produced
only `WCO_RLS_UPDATE_CHECK` (so `ExecInsert` skipped its check) and `DELETE`
produced none at all. Net effect: a leftover row could be inserted that violates
the table's INSERT RLS policy. The fix, in `get_row_security_policies()` at
`rowsecurity.c:396`, detects `root->forPortionOf != NULL` on the result relation
for `CMD_UPDATE`/`CMD_DELETE`, fetches the relation's `CMD_INSERT` policies via
`get_policies_for_relation()`, and adds them as `WCO_RLS_INSERT_CHECK` options
(SELECT rights deliberately not required — leftovers never appear in RETURNING).
Regression coverage added for both allowed and rejected leftovers in
`for_portion_of.sql` + `rowsecurity.sql`.
`[[knowledge/files/src/backend/rewrite/rowsecurity.c.md]]` — add the FOR PORTION
OF branch to the WITH-CHECK-options assembly notes; this is a good adversarial
test case for the RLS doc. See also the `row-level-security` skill.

### #10 `45cf7b1e5bf9` — Concurrent ALTER SUBSCRIPTION ... REFRESH SEQUENCES race (Amit Kapila)

A lost-update race in the PG19 logical-rep sequence-synchronization feature.
`ALTER SUBSCRIPTION ... REFRESH SEQUENCES` resets each sequence's subrel state to
`SUBREL_STATE_INIT` so it re-syncs. But a running sequence-sync worker that had
already fetched a publisher value and was about to mark the sequence `READY`
could overwrite the just-written `INIT` with its stale value — silently dropping
the refresh. Fix in `AlterSubscription_refresh_seq()`
(`subscriptioncmds.c:1363+`): stop any running sequence-sync worker **before**
resetting states, and move the reset loop out of the publisher-connection
`PG_TRY` block (the connection is now only needed for the origin check). The
argument that this is race-free leans entirely on lock ordering:
`AlterSubscription()` already holds `AccessExclusiveLock` on the subscription
object, which blocks a worker's `UpdateSubscriptionRelState()` (takes
`AccessShareLock`) and any relaunched worker's `InitializeLogRepWorker()` (also
`AccessShareLock` before reading `pg_subscription_rel`). Adds
`#include "storage/lock.h"`. Backpatch-through 19.
`[[knowledge/files/src/backend/commands/subscriptioncmds.c.md]]` — document the
stop-worker-before-reset ordering and the AEL-vs-AShareLock argument (a textbook
`locking`-skill example of using an existing heavyweight lock to serialize).

### #3 `d049a31a4cd2` — Restore of partitions with exclusion constraints (Álvaro Herrera / Japin Li)

Follow-up to `8c852ba9a4` (exclusion constraints on partitioned tables), which
didn't verify `pg_restore` worked for them. `CompareIndexInfo()` used to bail out
entirely — returning "not equal" — whenever either index had exclusion ops,
which broke ATTACH PARTITION matching during restore. The fix at
`catalog/index.c:2663` replaces the blanket reject with a real comparison: require
both sides to agree on *whether* they're exclusion indexes, and when they are,
compare `ii_ExclusionOps[i]`, `ii_ExclusionProcs[i]`, and `ii_ExclusionStrats[i]`
across all key attributes. Regression test in `indexing.sql` attaches a partition
where the constraints deliberately mismatch (`a with =` vs `a with &&`) and
confirms it's now correctly rejected.
`[[knowledge/files/src/backend/catalog/index.c.md]]` — update the
`CompareIndexInfo` note: exclusion indexes are now comparable, not an automatic
non-match.

### #11 `0da71d90d623` — Skip get_relids_in_jointree() when the query has no PHVs (Richard Guo)

the audit for the `1df9e8d96` outer-join touch.

### #8 `16ebc196d892` — pgstat backend-cache OOM leaves an inconsistent hash table (Michael Paquier)

Robustness fix in the cumulative-stats backend-local ref cache. In
`pgstat_get_entry_ref_cached()`, a fresh cache entry's `PgStat_EntryRef` was
allocated with plain `MemoryContextAlloc()`, which throws on OOM — but by then
the hash-table slot (`cache_entry`) already existed, so the throw left
`pgStatEntryRefHash` pointing at an entry whose `entry_ref` was never filled,
crashing or FATAL-ing at session exit depending on pending stats. Fix at
`pgstat_shmem.c:434`: switch to
`MemoryContextAllocExtended(..., MCXT_ALLOC_NO_OOM)`, and on NULL delete the
half-built hash entry via `pgstat_entry_ref_hash_delete()` *before* raising the
`ERRCODE_OUT_OF_MEMORY` error, so the table stays consistent. Shared memory is
untouched; not backpatched (unlikely in practice).
`[[knowledge/files/src/backend/utils/activity/pgstat_shmem.c.md]]` — add the
allocate-then-cleanup-on-OOM pattern to the entry-ref-cache notes (a clean
`memory-contexts` + `pgstat-framework` example of NO_OOM + manual rollback).

### #13 `f87375bfba5c` — pg_plan_advice: underscores in occurrence numbers (Daniel Gustafsson / Chao Li)

Contrib bug in the PG19 `pg_plan_advice` extension. Its scanner
(`pgpa_scanner.l`) accepts `_` as a digit separator like the core lexer, but then
converted occurrence numbers with `strtoint()`, which doesn't grok underscores —
so advice like `SEQ_SCAN(x#1_0)` failed to parse. Fix: use
`pg_strtoint32_safe()` like the core scanner, with regression coverage in
`sql/syntax.sql`. Backpatch-through 19 (where pg_plan_advice was introduced).
`[[knowledge/files/contrib/pg_plan_advice/pgpa_scan.c.md]]` — the module doc
should note the scanner uses `pg_strtoint32_safe` for separator-aware integer
parsing (the changed file is the `.l` source behind `pgpa_scan.c`).
