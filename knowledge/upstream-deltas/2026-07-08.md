## Deep explainers (behavior-changing + corpus-touching)

### `8e0c252753ba` — RETURNING OLD stale under BEFORE UPDATE trigger + concurrent update (Dean Rasheed)

The bug: an `UPDATE ... RETURNING old.col` on a table with a **BEFORE UPDATE
row trigger**, at READ COMMITTED, when the target row was concurrently updated
by another session. The trigger machinery (in `trigger.c`) does the EPQ
lock-and-wait, follows the update chain, and rewrites `tupleid` to the newest
version — but `ExecUpdate()` never noticed the tuple had moved and computed the
`OLD` values from the *outdated* tuple.

Fix is in `src/backend/executor/nodeModifyTable.c:2765` (`ExecUpdate`): clear
`context->tmfd.traversed` before `ExecUpdatePrologue()`, then after the
prologue, if `context->tmfd.traversed` is set, re-fetch the current version
with `table_tuple_fetch_row_version(..., SnapshotAny, oldSlot)` so `oldSlot`
reflects the row the update actually applied to
(`nodeModifyTable.c:2768`-2789). Dean notes the cleaner long-term fix is to
move the EPQ out of `trigger.c` into `ExecUpdate` like `ExecMergeMatched()`
does, but that would change the trigger API — too invasive to back-patch.
Back-patched to **v18** (where RETURNING OLD/NEW landed). Bug #19536; regression
coverage added to `src/test/isolation/expected/eval-plan-qual-trigger.out`.
Corpus: [[knowledge/files/src/backend/executor/nodeModifyTable.c.md]].

### `906b1e4a19a5` — Propagate stadistinct through GROUP BY / DISTINCT subqueries & CTEs (Richard Guo)

A meaningful planner estimation fix. Before this,
`examine_simple_variable()` in `src/backend/utils/adt/selfuncs.c:6113` bailed
out whenever a subquery/CTE used GROUP BY or DISTINCT (it could only recover
uniqueness for the *single*-column case). For multi-column grouping,
selectivity fell back to `1/DEFAULT_NUM_DISTINCT` (= 1/200), producing wildly
wrong estimates on filters/joins over grouped columns — a documented driver of
poor TPC-DS plans (spurious nestloops where a hash join wins).

Now, for DISTINCT / GROUP BY *key* columns that are simple Vars, the code
recurses into the subquery and lifts the base table's `stadistinct` (the
distinct-value count survives grouping). Crucially it *strips* MCV /
histogram / correlation slots from the copied `pg_statistic` tuple (grouping
reshapes the frequency distribution, so those are no longer valid) — new helper
`adjust_statstuple_for_grouping()`. A negative (fractional) `stadistinct` is
converted to an absolute count so it isn't misread against the subquery's
output rowcount, and `stanullfrac` is adjusted (single key → at most one NULL
group survives → `1/(ndistinct+1)`; multiple keys → approximated as 0). Result:
callers like `var_eq_const()` get a real `1/ndistinct` estimate instead of
1/200. Corpus: [[knowledge/files/src/backend/utils/adt/selfuncs.c.md]],
[[knowledge/subsystems/planner-cost-model.md]].

### `4721430f199f` + `3b0991059f33` + `9dce6b5a42bb` — nullingrels cleanup, retire `NRM_SUBSET` (Richard Guo)

A 3-commit sequence that tightens outer-join nullingrels bookkeeping in
`src/backend/optimizer/plan/setrefs.c`. Background: since commit `66e9df9f6`,
`identify_current_nestloop_params()` guarantees that Vars/PHVs in a
`NestLoopParam` expression carry *exactly* the outer-join relids that can null
them. That makes the old "actual Var may have a subset" slack unnecessary.

- `4721430f199f` ("Tighten nullingrels checks for outer joins") touches
  `createplan.c`, `setrefs.c`, and `include/nodes/plannodes.h` (+ a
  `pg_overexplain` test refresh).
- `3b0991059f33` removes the now-redundant `nrm_match` parameter from
  `fix_upper_expr()`.
- `9dce6b5a42bb` flips the `NestLoopParam` `fix_upper_expr()` call from
  `NRM_SUBSET` to `NRM_EQUAL` (`setrefs.c:2445`, `set_join_references`) and
  **deletes the `NRM_SUBSET` enumerator** entirely (`setrefs.c:34`), along with
  every `bms_is_subset(...)` branch in `search_indexed_tlist_for_var`
  (`setrefs.c:2971`) and `search_indexed_tlist_for_phv` (`setrefs.c:3025`).
  `NullingRelsMatch` now has only `NRM_EQUAL` / `NRM_SUPERSET`.

Net: the planner's sanity `elog(ERROR, "wrong varnullingrels ...")` cross-checks
are now strictly exact-or-superset, closing a class of "we settled for a subset"
imprecision. Corpus: [[knowledge/files/src/backend/optimizer/plan/setrefs.c.md]],
[[knowledge/subsystems/executor-and-planner.md]].

### `bb7ded1eebed` — Introduce `bms_offset_members()` (David Rowley)

New Bitmapset primitive at `src/backend/nodes/bitmapset.c:411`
(`bms_offset_members(const Bitmapset *a, int offset)`): produces a new set with
every member shifted by `offset`. It replaces the hand-rolled
`bms_next_member()` → `bms_add_member()` loops scattered around the tree with a
single word-level bitshift, saving both CPU and an allocation per iteration.
Semantics: negative offsets that would push a member below 0 drop that member;
an offset that would exceed `INT_MAX` raises `elog(ERROR, "bitmapset overflow")`
(guarded via `pg_add_s32_overflow`). Declared in
`src/include/nodes/bitmapset.h`; converted call sites include
`optimizer/plan/setrefs.c`, `optimizer/prep/prepjointree.c`,
`rewrite/rewriteManip.c`, and `statistics/extended_stats.c`; new coverage in
`src/test/modules/test_bitmapset/`. Corpus:
[[knowledge/files/src/backend/nodes/bitmapset.c.md]],
[[knowledge/idioms/node-infrastructure.md]].

### `d34ce773d2d3` — ILIKE `_` wrongly matched a partial multibyte char in C locale (Jeff Davis)

Regression from `5a38104b36`. `Generic_Text_IC_like()` in
`src/backend/utils/adt/like.c:191` takes a fast path in the C locale that
lowercases each byte lazily instead of calling `lower()` on the whole strings.
That byte-at-a-time trick is only valid for **single-byte** encodings: in a
multibyte encoding, `_` (match-one-character) could match a single *byte* of an
incomplete sequence. Fix guards the fast path with
`&& pg_database_encoding_max_length() == 1`. New regress row in
`collate.utf8.sql`: `SELECT 'café' ILIKE 'caf_' COLLATE "C"` now correctly
returns `t` (the `_` matches the whole multibyte `é`). Back-patched to **v19**.
Corpus: [[knowledge/files/src/backend/utils/adt/like.c.md]],
[[knowledge/subsystems/collation-provider.md]].

### `96ab9a990eed` — EXPLAIN crash deparsing SQL/JSON aggregates (Richard Guo)

When an aggregate is evaluated *above* the node that computes it (window
functions, or expressions postponed above the final sort), `setrefs.c` rewrites
the `Aggref`/`WindowFunc` into a `Var` referencing the lower node's output. For
`JSON_ARRAYAGG` / `JSON_OBJECTAGG`, deparsing the enclosing
`JsonConstructorExpr` then hit `elog(ERROR, "invalid JsonConstructorExpr
underlying node type")` because `get_json_agg_constructor()` didn't expect a
`Var`. Fix at `src/backend/utils/adt/ruleutils.c:12391` adds an
`IsA(ctor->func, Var)` branch (new helper `get_json_agg_constructor_expr`) that
chases the Var back to the underlying Aggref/WindowFunc and prints the original
JSON-aggregate syntax with its RETURNING / ON NULL / WITH UNIQUE options intact
— mirroring how `get_agg_expr()` already looks through such Vars. EXPLAIN-only.
Back-patched to **v16**. This is exactly the R13 "ruleutils / parse-tree
formatting" tier — an `EXPLAIN VERBOSE`-only break invisible to happy-path
regress. Corpus: [[knowledge/files/src/backend/utils/adt/ruleutils.c.md]],
[[knowledge/idioms/node-infrastructure.md]].

### `842169b6c110` — Aux processes advertise ProcNumber via atomics (Heikki Linnakangas)

Refactor of how the walwriter / checkpointer / autovacuum-launcher publish their
identity in shared memory. Previously `ProcGlobal->walwriterProc` /
`checkpointerProc` were plain fields read through a `volatile PROC_HDR *`, and
the AV launcher used an ad-hoc `av_launcherpid` field. Now
`InitAuxiliaryProcess()` sets these, they are `pg_atomic_uint32`
(`src/include/storage/proc.h`), and readers use `pg_atomic_read_u32()` — e.g.
`XLogSetAsyncXactLSN()` at `src/backend/access/transam/xlog.c:2673`. The AV
launcher's `av_launcherpid` in `postmaster/autovacuum.c` is removed in favour of
the same ProcNumber pattern (extensible to other aux procs). Follows the
existing atomic treatment of `procArrayGroupFirst` / `clogGroupFirst`.
Reviewed by Andres Freund. Corpus:
[[knowledge/files/src/backend/access/transam/xlog.c.md]],
[[knowledge/files/src/include/storage/proc.h.md]],
[[knowledge/subsystems/process-lifecycle.md]].

### `a4639d64e219` — Whole-row deps for ALTER COLUMN SET EXPRESSION (Peter Eisentraut / jian he)

### `306d7680f084` — WAIT_FOR_WAL_* wait events reclassified Client → IPC (Alexander Korotkov)

`WAIT_FOR_WAL_FLUSH` / `WAIT_FOR_WAL_REPLAY` / `WAIT_FOR_WAL_WRITE` were in the
`WaitEventClient` class, but that class is specifically for waiting on a socket
becoming readable/writable. These events can also block on local fsync / local
replay (disk- or CPU-bound), so `WaitEventIPC` is the correct class. Pure
reclassification in `src/backend/utils/activity/wait_event_names.txt` (moved
from the Client block ~L89 to the IPC block ~L159; the generator rebuilds the
enums). Reported by Noah Misch; back-patched to **v19**. Monitoring impact:
these events now surface with `wait_event_type = 'IPC'` in `pg_stat_activity`.
Corpus: [[knowledge/subsystems/wal-and-xlog.md]],
[[knowledge/idioms/pgstat-framework.md]].
