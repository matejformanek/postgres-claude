## Deep dives

### `e0511883` ‚Äî libpq: PQdescribePrepared rejected >7498 parameters

**File:** `src/interfaces/libpq/fe-protocol3.c`, the `VALID_LONG_MESSAGE_TYPE`
macro ~L38‚Äì46 (@`e0511883`). Touches
[[knowledge/files/src/interfaces/libpq/fe-protocol3.c.md]].

libpq guards against runaway message sizes: most backend message types have a
small length cap, and only an allowlisted set of "long message" types
(`DataRow`, `CopyData`, `FunctionCallResponse`, notices, `RowDescription`, ‚Ä¶)
are permitted to exceed it. The `ParameterDescription` ('t') message ‚Äî sent in
reply to a `Describe` of a prepared statement ‚Äî was **missing from that
allowlist**. A prepared statement with more than 7498 parameters produces a
`ParameterDescription` longer than the cap, so `PQdescribePrepared()` failed with
a protocol error instead of returning the descriptor. The one-line fix adds
`(id) == PqMsg_ParameterDescription` to the macro. Client-side only; no protocol
or wire change, just the client's own sanity threshold. Back-patchable.

### `a75bd485` ‚Äî replication: proper quoting of identifiers/literals in replication commands

**Files:** `src/backend/replication/libpqwalreceiver/libpqwalreceiver.c`
(new `appendQuotedString` + `appendQuotedIdentifier`/`appendQuotedLiteral`
macros ~L525‚Äì550, used in `libpqrcv_startstreaming` ~L573‚Äì620);
`src/backend/commands/subscriptioncmds.c` (same helper ~L520‚Äì544, used in
`ReplicationSlotDropAtPubNode` ~L2544); also `repl_scanner.l` and the
`pg_basebackup` family. Touches
[[knowledge/files/src/backend/replication/libpqwalreceiver/libpqwalreceiver.c.md]],
[[knowledge/files/src/backend/commands/subscriptioncmds.c.md]],
[[knowledge/files/src/backend/replication/repl_scanner.l.md]].

Replication commands (`START_REPLICATION ... SLOT "x"`, `DROP_REPLICATION_SLOT`,
`CREATE_REPLICATION_SLOT`, the `streaming`/`origin`/`publication_names`
options) were being assembled with raw `appendStringInfo(&cmd, " SLOT \"%s\"",
name)` and `appendStringInfo(&cmd, ", origin '%s'", str)` ‚Äî i.e. **no escaping**
of embedded quote characters. A slot name, origin, or publication name
containing a `"` or `'` would produce a malformed (or injectable) replication
command. The patch introduces a small `appendQuotedString(buf, str, quote)`
helper ‚Äî doubling any occurrence of the quote char ‚Äî wrapped by
`appendQuotedIdentifier` (`"`) and `appendQuotedLiteral` (`'`) macros, and
routes all the option-string emission through them. A pointed `Caution:` comment
notes the helper is sufficient *for the replication grammar* but **not for
regular SQL** (it'd be wrong for a literal when `standard_conforming_strings` is
off). This is the same helper duplicated into both the walreceiver and the
subscription-command path rather than shared, mirroring the existing module
split. Correctness/robustness; no wire-format change.

### `e5f94c48` ‚Äî query-jumble model: comment corrections (Const, location, ignore)

**Files:** `src/include/nodes/nodes.h` (the `pg_node_attr` doc block ~L105‚Äì116),
`src/include/nodes/parsenodes.h` (Query header ~L109‚Äì146), and
`src/include/nodes/primnodes.h` (Const ~L319‚Äì352, plus Aggref/FuncExpr/OpExpr/
ScalarArrayOpExpr/WindowFunc comment blocks). Touches
[[knowledge/files/src/include/nodes/nodes.h.md]],
[[knowledge/files/src/include/nodes/parsenodes.h.md]],
[[knowledge/files/src/include/nodes/primnodes.h.md]].

Comment-only, but it sharpens a model worth keeping straight in the corpus.
Three clarifications: (1) `query_jumble_location` is now documented as **only**
for `ParseLoc` fields (which are otherwise not jumbled), not "any integer field
named location". (2) The Query-tree header now states explicitly that jumbling
**deliberately ignores some semantically-significant fields** (notably `Const`
values) as a *policy decision* to merge queries that differ only in constants ‚Äî
previously the comment implied only insignificant fields are ignored. (3) The
`Const` comment is rewritten: only `consttype` is jumbled (different const types
can yield very different plans, useful to distinguish in pg_stat_statements),
and the `location` field carries the `query_jumble_location` attr so each const
is tracked as a parameter location. Several redundant per-node "X is irrelevant
for query jumbling" comments are removed (Aggref, FuncExpr, OpExpr,
ScalarArrayOpExpr, WindowClause, RangeTblFunction, CallStmt) since the attr tags
on the fields are now the single source of truth. The `WindowFunc` comment block
was also relocated to directly precede its struct. No `gen_node_support.pl`
output change.

### `f6e4ec0a` ‚Äî stats import: reject oversized MCV lists before deserialize

**File:** `src/backend/statistics/extended_stats_funcs.c`, `import_mcv()` ~L851
(@`f6e4ec0a`). Touches
[[knowledge/files/src/backend/statistics/extended_stats_funcs.c.md]].

`pg_restore_extended_stats()` lets a privileged caller inject extended-statistics
MCV lists directly (the PG 18 statistics-import feature). `import_mcv()` read
`nitems = ARR_DIMS(mcv_arr)[0]` straight from the supplied array but never
checked it against `STATS_MCVLIST_MAX_ITEMS` (10000) ‚Äî the cap that
`statext_mcv_deserialize()` enforces downstream. An array of 10001+ items would
sail past `import_mcv` and fail (or misbehave) deep inside deserialize. The fix
adds an early bound check that, on violation, raises a `WARNING`
(`ERRCODE_INVALID_PARAMETER_VALUE`, "number of items (%d) exceeds maximum (%d)")
and `goto mcv_error` ‚Äî matching the soft-failure style of the rest of the import
path (the SQL function returns `f` rather than ERRORing, so a bad stats blob in a
dump doesn't abort the whole restore). Regress adds a 10001-item case asserting
the warning + `f` return. Robustness fix for the import surface; no on-disk
change.

### `3f328049` ‚Äî ltree: split GiST distance (float) from btree compare (int) to fix int32 overflow

**Files:** `contrib/ltree/ltree_op.c` (`ltree_compare()` ~L46 rewritten + new
`ltree_compare_distance()` ~L80), `contrib/ltree/ltree_gist.c` (`ltree_penalty`
~L264 now uses the float variant), `contrib/ltree/ltree.h` (prototype). Touches
[[knowledge/files/contrib/ltree/ltree_op.c.md]],
[[knowledge/files/contrib/ltree/ltree_gist.c.md]].

The old `ltree_compare()` did double duty: a btree ordering function **and** the
GiST penalty's distance metric. For the distance role it scaled differences by
`* 10 * (an + 1)` where `an` is the remaining level count ‚Äî and that arithmetic
is `int32`. A very deep ltree (the regress test uses ~15000 levels) overflows the
multiply, producing a wrong (possibly negative) "distance" and a corrupted GiST
penalty, which degrades index quality. The fix **cleaves the two roles apart**:
`ltree_compare()` becomes a pure, overflow-free btree comparator returning
`-1/0/+1`-style results from plain `memcmp` + length/level differences (no
scaling); a new `ltree_compare_distance()` returns a **`float`** distance with
the `* 10.0 * (an + 1)` weighting (negative when `a < b`, consistent with the
ordering), and `ltree_penalty()` switches its `cmpl`/`cmpr` locals from `int32`
to `float` and calls the new function. Net: btree comparisons get simpler and
correct; GiST distance moves to float so the scaled product no longer overflows.
Correctness fix in a maintained contrib module; no catalog/on-disk change (the
penalty is computed at insert time, not stored).

### `e2a8cabc` ‚Äî concurrent repack: assert all separately-stored toast attrs were consumed

**File:** `src/backend/commands/repack.c`, `restore_tuple()` ~L2806
(@`e2a8cabc`). Touches [[knowledge/files/src/backend/commands/repack.c.md]].

Part of the new PG 19 concurrent (`REPACK ... CONCURRENTLY`) machinery.
`restore_tuple()` reconstructs a tuple from a spill file, pulling
separately-stored ("external"-during-repack) attributes back in and decrementing
a `natt_ext` counter as it consumes each one. The pre-existing underflow guard
(`natt_ext < 0` ‚Üí error) is converted from an `ereport(ERRCODE_DATA_CORRUPTED)`
to a plain `elog(ERROR, ...)` ‚Äî signalling this is a "can't happen" internal
invariant, not a user-facing corruption report. The new line adds the
**symmetric** check: after the restore loop, `natt_ext != 0` (i.e. attributes
were stored separately but never consumed) is also an `elog(ERROR,
"unexpected number of attributes stored separately (%d remaining)")`. Tightens
an internal consistency check so a repack bug surfaces immediately rather than
silently producing a short tuple. Defensive; master-only feature.

### `f50c329f` ‚Äî logical decoding: free speculative-insertion change on the error path

**File:** `src/backend/replication/logical/reorderbuffer.c`,
`ReorderBufferProcessTXN()` ~L2753 and `ReorderBufferResetTXN()` ~L2166
(@`f50c329f`). Touches
[[knowledge/files/src/backend/replication/logical/reorderbuffer.c.md]].

When decoding an `INSERT ... ON CONFLICT`, the reorder buffer holds the
speculative-insertion change in a local `specinsert` until the matching confirm
arrives. If an **ERROR is raised mid-decode** (the TAP repro uses a publication
row filter `WHERE (a / 0) > 0` that divides by zero when evaluated), the
`specinsert` change was not freed, leaking it and ‚Äî with asserts on ‚Äî tripping
an assertion. The original code tried to free `specinsert` inside
`ReorderBufferResetTXN()`, but reset only runs on the *streaming-abort* branch,
not on the general error/cleanup branch, and passing `specinsert` by value there
couldn't null out the caller's pointer anyway. The fix **removes the free (and
the `specinsert` parameter) from `ReorderBufferResetTXN()`** and instead frees
`specinsert` in `ReorderBufferProcessTXN()`'s cleanup block, before the
`ReorderBufferTXN` itself is freed, so every exit path (success, concurrent
abort, arbitrary ERROR) releases it exactly once. New `100_bugs.pl` case decodes
through the divide-by-zero row filter and asserts the decoder reports the error
without crashing. Memory-correctness/assert fix; back-patch candidate (logical
decoding has shipped for many releases).
