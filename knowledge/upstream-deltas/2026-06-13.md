## Deep treatment

### `3e3d7875e9` — Andrew Dunstan — cross-version upgrade test fixup for `seg_out()`

`[from-diff]` Single hunk in
`src/test/perl/PostgreSQL/Test/AdjustUpgrade.pm` (after the pre-14
`putenv`/`wait_pid` drop block, around line 196):

```perl
# delete seg row that pre-14 was printed incorrectly but would now
# be printed correctly
if ($dbnames{contrib_regression_seg})
{
    _add_st($result, 'contrib_regression_seg',
        "delete from test_seg where s = '4.6 .. ~7.0'");
}
```

What this is for: the cross-version upgrade test (`pg_upgrade` from an old
binary's dump into the current tree) compares the *old* server's dump against
the *new* server's dump of the upgraded cluster. `AdjustUpgrade.pm` is the
shim that edits the old dump (or pre-deletes offending rows) so legitimate,
expected output differences don't fail the comparison. A prior `seg_out()`
fix changed how `seg` renders a value like `4.6 .. ~7.0` (the `~` is `seg`'s
"approximate" marker). Pre-14 servers printed this row *incorrectly*; the
current server prints it *correctly*. That asymmetry would make the dump diff
fail on any old→new run that includes the `contrib_regression_seg` database.
The fix sidesteps it by deleting that single row in the
`contrib_regression_seg` DB before the comparison, gated on the DB actually
being present (`$dbnames{contrib_regression_seg}`).

Corpus relevance: this is test-harness plumbing, but the *cause* is the
`seg_out()` output-format change. `contrib/seg` is documented at
`[[knowledge/files/contrib/seg/seg.c.md]]` and has an issue register at
`[[knowledge/issues/seg.md]]`. The `seg_out()` fix commit itself is **not in
this window** (it predates the anchor's window edge); this commit is only its
buildfarm/test-suite catch-up. No behavior change to `seg` itself here.
`AdjustUpgrade.pm` has no per-file corpus doc (it's in the deferred-mechanical
`src/test` bucket per `progress/STATE.md`).

### `67bd944c0e` — Fujii Masao — doc fix: which view shows slot xmin age

`[from-diff]` `doc/src/sgml/maintenance.sgml`, in the "preventing transaction
ID wraparound" remediation list (around line 702). The substantive change is
the third `<listitem>`: the guidance for finding old replication slots to drop
previously linked to **`pg_stat_replication`** but told the reader to look for
`age(xmin)` / `age(catalog_xmin)` — columns that **do not exist** on
`pg_stat_replication` (which describes active WAL senders). Those columns live
on **`pg_replication_slots`**. The patch repoints the link target from
`monitoring-pg-stat-replication-view` to `view-pg-replication-slots` and the
display text from `pg_stat_replication` to `pg_replication_slots`:

```sgml
-<link linkend="monitoring-pg-stat-replication-view">pg_stat_replication</link> to
+<link linkend="view-pg-replication-slots"><structname>pg_replication_slots</structname></link> to
 find slots where age(xmin) or age(catalog_xmin) is large.
```

The same hunk also wraps `pg_prepared_xacts` and `pg_stat_activity` (the two
preceding list items) in `<structname>…</structname>` for consistent markup.
Doc-only; no code, no catalog, no behavior change. Backpatch-eligible as a pure
documentation correctness fix (the wrong view name would actively mislead a DBA
chasing wraparound). Corpus: no per-file doc for `maintenance.sgml`; the
replication-slot concepts are covered conceptually in
`[[knowledge/subsystems/replication-*]]` / the `replication-overview` skill,
none of which assert the wrong view name, so no corpus correction needed.
