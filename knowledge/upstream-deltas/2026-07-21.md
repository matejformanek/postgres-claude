## Deep explainers

### #21 `45cf7b1e5` — Handle concurrent sequence refreshes  *(logical rep)*
`ALTER SUBSCRIPTION ... REFRESH SEQUENCES` could race a running sequence-sync
worker: if the worker had fetched a sequence value from the publisher but not
yet marked it `READY`, a concurrent refresh resetting the sequence to `INIT`
could be silently clobbered by the worker's stale value — losing the refresh.
Fix: `AlterSubscription_refresh_seq()` in
`src/backend/commands/subscriptioncmds.c` (~L1362–1438) now stops any running
`WORKERTYPE_SEQUENCESYNC` worker via `logicalrep_worker_stop()` **before**
resetting sequences to `INIT`; adds `#include "storage/lock.h"` and an assertion
that `AccessExclusiveLock` (already held by `AlterSubscription()`) is present,
which blocks worker state updates and re-launch until the refresh commits.
[[knowledge/files/src/backend/commands/subscriptioncmds.c.md]] ·
[[knowledge/subsystems/logical-replication]]

### #20 `20fe90bbb` — Fix RLS checks for FOR PORTION OF leftover rows  *(RLS + temporal)*
`UPDATE/DELETE ... FOR PORTION OF` re-inserts the "leftover" fragments of a row
that fall outside the targeted time range. Those implicit inserts bypassed
INSERT RLS policies, because the rewriter only added WCOs matching the
statement's own command type. Fix: `get_row_security_policies()` in
`src/backend/rewrite/rowsecurity.c` (~25 lines) now also fetches and applies
INSERT policies as `WCO_RLS_INSERT_CHECK` when `root->forPortionOf` is set on an
UPDATE/DELETE target. Ships regression coverage in `for_portion_of.sql` and
`rowsecurity.sql` (allowed + disallowed leftovers) and three doc updates
(`create_policy.sgml`, `delete.sgml`, `update.sgml`).
[[knowledge/files/src/backend/rewrite/rowsecurity.c.md]] ·
[[knowledge/skills/row-level-security]]

### #19 `16ebc196d` — pgstat_get_entry_ref_cached() OOM hardening  *(pgstat)*
When allocating a new backend-local cache entry failed, the per-backend hash
table (`pgStatEntryRefHash`) was left referencing a half-created entry — risking
a crash or FATAL at session exit. Fix in
`src/backend/utils/activity/pgstat_shmem.c`, `pgstat_get_entry_ref_cached()`
(~L432–453): allocate with `MemoryContextAllocExtended(MCXT_ALLOC_NO_OOM)`, and
on NULL do `pgstat_entry_ref_hash_delete()` to drop the dangling key before
`ereport(ERROR)`. Backend-local structures only (no shared memory); **not
backpatched** (low practical impact).
[[knowledge/files/src/backend/utils/activity/pgstat_shmem.c.md]] ·
[[knowledge/skills/pgstat-framework]]

### #8 `fc6425fbe` — generate_partition_qual() OOM cache handling  *(relcache)*
Sibling to #19, same author, same defensive shape. In
`src/backend/utils/cache/partcache.c`, `generate_partition_qual()` (~L411–439)
previously `copyObject()`'d the partition check-constraint list directly into
`CacheMemoryContext`; an allocation failure mid-copy left a leaked context and a
semi-filled `rd_partcheck`/`rd_partcheckcxt` on the relation. Fix: build the
copy in a child of `CurrentMemoryContext` (transient), and only on success
`MemoryContextSetParent()` it under `CacheMemoryContext` and publish the relation
pointers. No backpatch. This is the canonical "build-then-reparent" idiom for
populating relcache-lifetime state safely.
[[knowledge/files/src/backend/utils/cache/partcache.c.md]] ·
[[knowledge/skills/memory-contexts]]

### #15 `d5751c33c` — Avoid ERROR in recovery target GUC assign hooks  *(xlogrecovery + guc)*
Recovery-target assign hooks (`assign_recovery_target`,
`assign_recovery_target_name`, `assign_recovery_target_time`) did cross-parameter
validation and could `ERROR` — which violates the GUC contract (assign hooks must
not fail) and made behavior order-dependent across the several
`recovery_target*` settings. Fix in
`src/backend/access/transam/xlogrecovery.c`: delete those three assign hooks;
add `DetermineRecoveryTargetType()` (~L4774–4820) called from
`validateRecoveryParameters()` (~L1067) which reads final values via
`GetConfigOption()` and raises **FATAL** on conflict *after* all assignments
settle. `recoveryTargetName` changes `const char *`→`char *`
(`src/include/access/xlogrecovery.h` L142); assign-hook rows dropped from
`guc_parameters.dat` (~L2472–2507); static `recovery_target_name_string` removed
from `guc_tables.c` (L667); declarations pruned from `guc_hooks.h`. Empty-string
assignments now correctly mean "unset." Companion test fix is #12.

### #10 `d774576f6` — Logical rep workers ignore default_transaction_read_only  *(logical rep)*
Sequence-sync updates call `setval()`, which calls `PreventCommandIfReadOnly()`
and thus fails when the subscriber sets `default_transaction_read_only = on`
(apply/tablesync workers dodge this by writing heaps directly, but the new
sequence path uses `setval()`). Fix: `InitializeLogRepWorker()` in
`src/backend/replication/logical/worker.c` (~L5809–5818) forces
`default_transaction_read_only` `off` via
`SetConfigOption(..., PGC_SUSET, PGC_S_OVERRIDE)`, mirroring the existing
`session_replication_role`/`search_path` overrides. On v19 the override is
narrowed to sequencesync workers only. TAP coverage in `036_sequences.pl`.
Reported by Noah Misch; by vignesh C.
[[knowledge/files/src/backend/replication/logical/worker.c.md]] ·
[[knowledge/subsystems/logical-replication]]

### #14 `d049a31a4` — Fix restore of partitions with exclusion constraints  *(catalog/index)*
Follow-up to `8c852ba9a4` (exclusion constraints on partitioned tables). Attaching
a partition index that backs an exclusion constraint was rejected outright, so
`pg_restore` of such partitions failed. Fix: `CompareIndexInfo()` in
`src/backend/catalog/index.c` (~L2663–2680) now compares the exclusion machinery
(`ii_ExclusionOps`, `ii_ExclusionProcs`, `ii_ExclusionStrats`) between parent and
partition index and permits the attach when they match, rejecting only genuine
mismatches. Regression coverage in `indexing.sql`. By Japin Li; reported by
Keith Paskett.
