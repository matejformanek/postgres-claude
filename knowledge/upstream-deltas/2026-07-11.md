## Deep explainers

### `eed6c0d33e09` — Update FSM after updating VM on-access  (Melanie Plageman) ★ bugfix, back-patched to 19

**One-line:** on-access (opportunistic) heap pruning that newly marks a page
all-visible now also records the page's free space in the FSM, closing a bloat
window opened by `b46e1e54d078`.

**The bug.** Commit `b46e1e54d078` let `heap_page_prune_opt()` — the
opportunistic pruning that runs on ordinary SELECT/UPDATE buffer access — set
the visibility-map `ALL_VISIBLE` bit when a prune made the page all-visible.
That was a latency win (VACUUM no longer had to be the one to set the bit) but
it created a coherence gap with the free-space map: once a page is
all-visible, a subsequent VACUUM is entitled to **skip** it (that is the whole
point of the VM), and the skip path does not update the FSM for skipped pages.
So a page that pruning had just opened up (dead tuples reclaimed → real free
space) could be marked all-visible *and* skipped by VACUUM, leaving the FSM
advertising a stale, too-low free-space category forever. Net effect:
inserters never learn the page has room → effectively permanent bloat.

**The fix** (`src/backend/access/heap/pruneheap.c`):
- New include of `storage/freespace.h` (`pruneheap.c:30`).
- Inside the `PageIsFull || PageGetHeapFreeSpace(page) < minfree` branch of
  `heap_page_prune_opt()`, two locals gate the new behavior:
  `record_free_space` / `freespace` (`pruneheap.c:325`).
- After pruning, iff the prune *newly* set the page all-visible
  (`presult.newly_all_visible`, `pruneheap.c:391`) the code captures
  `PageGetHeapFreeSpace(page)` while still holding the buffer content lock
  (`pruneheap.c:393-394`).
- After releasing the lock, it calls
  `RecordPageWithFreeSpace(relation, BufferGetBlockNumber(buffer), freespace)`
  (`pruneheap.c:407-408`).

**Why the narrow gate matters.** The old comment explained that
`heap_page_prune_opt()` deliberately did *not* touch the FSM, so that free
space opened by unrelated UPDATE/INSERT churn stays reserved for HOT updates
to *this* page rather than being handed out to arbitrary inserters. That
reservation policy is preserved: the FSM is updated **only** in the
`newly_all_visible` case — i.e. only when the alternative is a stale FSM entry
that VACUUM will never correct because it will skip the now-all-visible page.
The new comment also notes `RecordPageWithFreeSpace()` only dirties the FSM
when the recorded free-space *category* actually changes, and that VACUUM's
`FreeSpaceMapVacuum()` still runs over skipped ranges — so this targeted
write is cheap and non-redundant.

`newly_all_visible` is itself set in the prune result up in the shared prune
core (`pruneheap.c:1385-1394`), so both VACUUM's lazy prune and the on-access
prune feed the same flag; only the on-access path needed the FSM follow-up
because VACUUM's own path already records free space.

**Corpus impact.** Directly touches the FSM ↔ VM ↔ pruning interaction that
three corpus docs describe:
`[[knowledge/files/src/backend/access/heap/pruneheap.c.md]]`,
`[[knowledge/files/src/backend/storage/freespace/freespace.c.md]]`,
`[[knowledge/files/src/backend/access/heap/visibilitymap.c.md]]`, plus the
`free-space-map` and `vacuum-autovacuum` skills. This is the concrete fix for
the "FSM/VM on-access coherence gap" that `pg-community-pulse` flagged in the
07-10→07-11 cycle (STATE last-activity line). Back-patched through 19, so it is
a live correctness change, not master-only polish.

### `1c4b1de88855` — Shorten pg_attribute_always_inline to pg_always_inline  (Tomas Vondra)  ▸ mechanical rename

**One-line:** the force-inline macro `pg_attribute_always_inline` is renamed to
`pg_always_inline` tree-wide; pure spelling change, no semantics.

Definition site is `src/include/c.h` (the three-arm `#if __GNUC__&&__OPTIMIZE__
/ _MSC_VER / else` block that expands to `__attribute__((always_inline))
inline`, `__forceinline`, or plain `inline`). 15 files touched; every other
hunk is a call-site rename in hot paths that force inlining —
`heapam.c`, `xlog.c`, `copyfromparse.c`, `copyto.c`, `execExprInterp.c`,
`execTuples.c`, `nodeHashjoin.c`, `nodeSeqscan.c`, `queryjumblefuncs.c`,
`bufmgr.c`, `json.c`, `catcache.c`, plus headers `execScan.h` and
`instr_time.h`. The shorter name aligns with the existing
`pg_noinline` / `pg_attribute_*` naming and reads better at call sites.
**Extension/patch note:** any out-of-tree code that spelled
`pg_attribute_always_inline` will need the rename on master (no compatibility
alias was left behind). No behavioral change; no corpus doc needs a content
edit, only the identifier if quoted verbatim.

### `e615da8cb21b` — Fix for loop variables  (Peter Eisentraut, rev. Tom Lane)  ▸ mechanical type hygiene

**One-line:** a large sweep (70 files) fixing `for`-loop induction variables
whose type did not match the loop's end condition — signed/unsigned and
narrow/wide mismatches that "could lead to wraparound or signed/unsigned
mismatches," fragile rather than currently-buggy.

Representative hunk: `parse_target.c:expandRecordVariable()` changes
`for (Index level = 0; level < netlevelsup; ...)` to `for (int level = 0; ...)`
so the counter matches `netlevelsup`'s `int`. The sweep spans access methods
(`gin/ginget.c`, `gin/ginscan.c`, `nbtree/nbtpage.c`), the executor
(`execCurrent.c`), parser (`parse_relation.c`, `parse_target.c`), replication
(`reorderbuffer.c`), JIT (`llvmjit.c`), several contrib modules
(`pageinspect`, `pg_trgm`, `pg_plan_advice`, `pg_logicalinspect`), and win32
port glue. No functional change intended; compiler will collapse most of these.
Corpus docs for the touched files stay accurate — only quoted loop headers
would drift.

### `5f14f82280db` — Fix for loop variables used with lengthof  (Peter Eisentraut, rev. Tom Lane)  ▸ mechanical type hygiene

**One-line:** companion to `e615da8`; `lengthof()` yields `size_t`, so loops
iterating to `lengthof(arr)` now use `size_t` counters instead of `int`,
silencing signed/unsigned warnings at higher `-W` levels.

33 files, e.g. `heapam.c:index_delete_sort()` gains
`for (size_t g = 0; g < lengthof(gaps); g++)`. Spans backend (`heapam.c`,
`parallel.c`, `bootstrap.c`, `objectaddress.c`, `typecmds.c`, `bgworker.c`,
`datachecksum_state.c`, `logical/conflict.c`, `amutils.c`, `jsonb.c`,
`jsonpath_exec.c`, MB conversion procs), `src/bin` clients (`initdb`,
`pg_dump`, `pgbench`, `psql`), and `contrib/dblink`. Same "optimized beyond
recognition, no generated-code change" caveat from the commit message applies.
