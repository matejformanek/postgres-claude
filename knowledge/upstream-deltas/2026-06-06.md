## Deep dives

### `89eafad297` — virtual generated columns break offset caching in tuple deforming

Virtual generated columns (`GENERATED ALWAYS AS (...) VIRTUAL`, the default
form added in PG 18) are **not materialized** in the on-disk/heap tuple — they
are computed on read and physically stored as NULL. The attribute-offset
caching machinery in `TupleDescFinalize()` did not know this, so it could treat
a virtual generated attribute as a fixed-width, always-present column and cache
a byte offset past it. A later `slot_deform_heap_tuple()` would then read at the
wrong offset (the virtual column occupies a NULL bit, not the cached number of
bytes), corrupting deformed values of columns that follow.

Two coordinated fixes:

- `src/backend/access/common/tupdesc.c` — `TupleDescFinalize()` (~L517-540).
  The loop now pulls the full `Form_pg_attribute attr` alongside the
  `CompactAttribute`, and a virtual generated attribute
  (`attr->attgenerated == ATTRIBUTE_GENERATED_VIRTUAL`) now (a) forces
  `firstNonGuaranteedAttr` to that index (~L527-531) and (b) stops the
  offset-caching walk via `break` (~L535-540), exactly as a varlena/`attlen<=0`
  column does. A new comment spells out the "stored as NULL → don't cache
  beyond" reasoning.
- `src/backend/executor/execTuples.c` — `slot_deform_heap_tuple()` (~L1074-1086).
  Defensive `Assert(first_null_attr(tup->t_bits, natts) >= firstNullAttr)` in
  the "all required columns guaranteed to exist" branch: if `TupleDescFinalize()`
  ever again mis-classifies a NULL-bearing column as guaranteed, this trips in
  assert builds rather than silently mis-deforming.

The shape of the fix mirrors the existing `attlen <= 0` handling — a virtual
generated column is, for deforming purposes, "as unpredictable as a varlena":
its physical width in the tuple is not a function of its type. Touches
`[[knowledge/files/src/backend/access/common/tupdesc.c.md]]` and
`[[knowledge/files/src/backend/executor/execTuples.c.md]]`.

### `193a4ded94` — pg_surgery off-by-one between OffsetNumber and a 0-based array

`contrib/pg_surgery`'s `heap_force_kill` / `heap_force_freeze` collect the
caller-supplied TIDs into a per-page boolean scratch array `include_this_tid[]`
sized `MaxHeapTuplesPerPage`, then sweep the page's line pointers marking the
flagged ones. The bug: `OffsetNumber` (the offset half of a `ItemPointer`) is
**1-based** (valid range `1 .. MaxHeapTuplesPerPage`, `FirstOffsetNumber == 1`),
but the code indexed the **0-based** array with the raw offset.

Two sites in `heap_force_common()` (`contrib/pg_surgery/heap_surgery.c`):

- Write side (~L228-232): the guard `Assert(offno < MaxHeapTuplesPerPage)` could
  fail spuriously for a legitimate max-offset tuple, and `include_this_tid[offno]`
  wrote one slot too high (off the end at `offno == MaxHeapTuplesPerPage`). Fixed
  to `Assert(offno <= MaxHeapTuplesPerPage)` + `include_this_tid[offno - 1]`.
- Read side (~L247-250): the page sweep tested `include_this_tid[curoff]` for
  `curoff` running over `OffsetNumber`s; fixed to `include_this_tid[curoff - 1]`
  so it reads the slot the write side set.

Net effect of the old code: a TID at the highest offset on a page wrote one slot
past the array (stack/heap overrun by one bool), and every flag lookup was
shifted by one offset — so `pg_surgery` could act on the wrong tuple or miss the
intended one. Classic 1-based-vs-0-based mismatch; the fix is purely the `-1`
rebasing plus loosening the bound assert. `contrib/pg_surgery` has no per-file
corpus doc, so no crossref — but a clean cautionary example for any code that
uses an `OffsetNumber` as an array index without subtracting `FirstOffsetNumber`.
