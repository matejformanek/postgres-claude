## Deep dives

### `a4f02cab4b97` — datachecksums worker invocation identity (Heikki Linnakangas)

Caps a four-commit series (`0edbf72f` → `c008b7ea` → `c48e7b2c` →
`a4f02cab`) hardening the online-checksums worker/launcher coordination in
`src/backend/postmaster/datachecksum_state.c`. `[from-diff]`

The structural problem: the launcher tracked only `worker_pid` in
`DataChecksumsStateStruct`, so a worker that was *supposed* to be aborted (e.g.
because `pg_disable_data_checksums()` flipped `launch_operation` mid-run) could
still finish and write `worker_result` over a *fresh* invocation's slot — a
classic stale-actor write. The fix introduces invocation identity:

- New shmem fields: `uint64 worker_invocation_counter` (monotonic), and a
  per-current-worker triple `worker_invocation` / `database_oid` /
  `worker_pid`. `[from-diff]`
- `ProcessDatabase()` takes `DataChecksumsWorkerLock` `LW_EXCLUSIVE`,
  bumps `++worker_invocation_counter`, stamps the current `worker_invocation`
  and `database_oid`, and passes the **invocation number** to the bgworker via
  `bgw_main_arg = UInt64GetDatum(invocation)` — previously it passed the dboid.
  `[from-diff]`
- `DataChecksumsWorkerMain()` now reads `worker_invocation =
  DatumGetUInt64(arg)`, then under `LW_SHARED` checks
  `DataChecksumState->worker_invocation != worker_invocation` and **returns
  early** if it's stale; it reads `database_oid` from shmem rather than the
  arg. Every subsequent `worker_result` write is guarded by the same
  invocation equality test. `[from-diff]`
- The abort macro `CHECK_FOR_ABORT_REQUEST()` is split into
  `CHECK_FOR_LAUNCHER_ABORT_REQUEST()` (asserts
  `B_DATACHECKSUMSWORKER_LAUNCHER`) and `CHECK_FOR_WORKER_ABORT_REQUEST()`
  (asserts `B_DATACHECKSUMSWORKER_WORKER`, and additionally treats an
  invocation mismatch as an abort). `[from-diff]`

Corpus tie-in: this is a textbook **stale-actor / generation-counter** pattern —
the same shape as backend-ID + generation pairs elsewhere. The
`bgw_main_arg`-carries-an-opaque-identity idiom is worth a `[[knowledge/idioms/]]`
note if not already present. Touches `[[knowledge/files/src/backend/postmaster/datachecksum_state.c.md]]`
(per-file doc exists; queue for audit). Lock primitive: `DataChecksumsWorkerLock`
LWLock, `LW_EXCLUSIVE` for all state mutation — consistent with
`[[knowledge/skills/locking]]` guidance (shmem state mutation under exclusive
LWLock, reads under shared).

### `a272a58b9424` + `a40fdf658862` — FOR PORTION OF check relocation (Peter Eisentraut)

Two related corrections to where FOR PORTION OF (the SQL:2011 temporal
UPDATE/DELETE feature) validates its preconditions. `[from-diff]`

**Volatile-bound check → planner** (`a272a58b`):
`transformForPortionOfClause()` in `parser/analyze.c` previously called
`contain_volatile_functions_after_planning()` on `result->targetRange` at
parse-analysis time. That helper is a parse-time approximation; the real test
belongs after expression preprocessing. The check now lives in
`subquery_planner()` (`optimizer/plan/planner.c`): after
`preprocess_expression(root, parse->forPortionOf->targetRange,
EXPRKIND_TARGET)`, it runs `contain_volatile_functions()` and raises
`ERRCODE_FEATURE_NOT_SUPPORTED` "FOR PORTION OF bounds cannot contain volatile
functions". Net effect: bounds that const-fold to non-volatile now pass, and the
volatility judgment is made on the fully-preprocessed expression. `[from-diff]`

**FDW rejection → executor** (`a40fdf65`): the "foreign tables don't support
FOR PORTION OF" error also moved out of `transformForPortionOfClause()` (which
only saw the *named* target relation) and into `CheckValidResultRel()`
(`executor/execMain.c`). The signature of `CheckValidResultRel()` and
`ExecCheckPermissions()` gains a `ModifyTable *mtnode` parameter (see
`include/executor/executor.h`) so the check can read `mtnode->forPortionOf`;
callers in `copyfrom.c`, `execPartition.c` (`ExecFindPartition`,
`ExecInitPartitionInfo`) and `nodeModifyTable.c` (`ExecInitModifyTable`) are
updated. The behavioral win, per the new `postgres_fdw` regress rows: a
**child partition that is a foreign table** now correctly errors when targeted,
even if the root is local — *but* a foreign child partition that gets **pruned**
no longer raises a spurious error. `[from-diff]`

Corpus tie-in: signature change to `CheckValidResultRel`/`ExecCheckPermissions`
ripples through several executor per-file docs —
`[[knowledge/files/src/backend/executor/execMain.c.md]]`,
`[[knowledge/files/src/backend/executor/execPartition.c.md]]`,
`[[knowledge/files/src/backend/executor/nodeModifyTable.c.md]]`,
`[[knowledge/files/src/include/executor/executor.h.md]]`,
`[[knowledge/files/src/backend/commands/copyfrom.c.md]]`. Any doc quoting the
old arity of these functions is now stale; queue all five for audit.

### `cdae794af31b` — MERGE/SPLIT PARTITION honors default_tablespace (Alexander Korotkov)

`createPartitionTable()` in `commands/tablecmds.c` — the helper that
materializes a new partition during `ALTER TABLE ... MERGE PARTITIONS` and
`... SPLIT PARTITION` — previously copied `parent_relform->reltablespace`
verbatim. If the partitioned root had no explicit tablespace (reltablespace =
0), the new partition was forced to the database default, **ignoring
`default_tablespace`** — inconsistent with `CREATE TABLE ... PARTITION OF`.
`[from-diff]`

The fix mirrors `DefineRelation`'s logic: `tablespaceId =
parent_relform->reltablespace`; if invalid, `tablespaceId =
GetDefaultTablespace(newPartName->relpersistence, false)`; then an ACL check
(`object_aclcheck(TableSpaceRelationId, ...)` unless using the DB default), and
a hard `ERRCODE_INVALID_PARAMETER_VALUE` "only shared relations can be placed
in pg_global tablespace" guard against `GLOBALTABLESPACE_OID`. The new regress
rows in `partition_merge.sql` / `partition_split.sql` exercise all three arms:
explicit-root-tablespace wins, `default_tablespace` is honored when the root
has none, and `pg_global` is rejected even when it sneaks in via
`default_tablespace`. `[from-diff]`

Corpus tie-in: `[[knowledge/files/src/backend/commands/tablecmds.c.md]]`
(per-file doc exists; queue). This is the "match DefineRelation's tablespace
selection" invariant — a recurring source of partition-DDL bugs.

### `4abf411e2328` — pg_stat_io counts autovacuum-launcher extends (Melanie Plageman)

One-line correctness fix in `pgstat_tracks_io_op()`
(`utils/activity/pgstat_io.c`). The exclusion list said
`B_AUTOVAC_LAUNCHER || B_BG_WRITER || B_CHECKPOINTER` never perform
`IOOP_EXTEND`. But the autovacuum **launcher** does extend relations (e.g. when
it has to create per-database stats infrastructure), so its EXTEND I/O was
being silently dropped from `pg_stat_io`. The fix narrows the exclusion to
`B_BG_WRITER || B_CHECKPOINTER`. `[from-diff]`

Corpus tie-in: `[[knowledge/files/src/backend/utils/activity/pgstat_io.c.md]]`
(per-file doc exists; queue — the doc likely lists the
"backends-that-never-extend" set, which just shrank by one).

### `dac36601fd77` — out-of-bounds read in autoprewarm worker (Tomas Vondra)

`autoprewarm_database_main()` in `contrib/pg_prewarm/autoprewarm.c`. The inner
loop prewarmed a run of blocks for one relation, then did `i = p.pos;` (the
read-stream callback's final position) followed by `blk = block_info[i];` at
the *bottom* of the loop. When the callback advanced `i` to exactly
`prewarm_stop_idx` (end of the dump), that trailing `block_info[i]` read one
past the valid array. The restructure moves `blk = block_info[i]` to the *top*
of the loop guarded only by `i < prewarm_stop_idx`, with the
tablespace/filenumber "different relation" break done explicitly after the load
and *before* any dereference. A comment now warns the callback may have advanced
`i` past the last valid block. `[from-diff]`

Corpus tie-in: `[[knowledge/files/contrib/pg_prewarm/autoprewarm.c.md]]`
(per-file doc exists; queue). Classic "advance-then-deref" off-by-one with a
read-stream callback owning the index — worth a note in any read-stream idiom doc.

### `6468f7a853c3` — uuid→bytea cast is leakproof (Masahiko Sawada)

`pg_proc.dat`: `bytea(uuid)` (proname `bytea`, prosrc `uuid_bytea`, OID 6510)
gains `proleakproof => 't'`; `catversion.h` bumps to `202606251`; `opr_sanity`
expected output adds `bytea(uuid)` to the leakproof list. `[from-diff]`

Rationale: `uuid_bytea` is a fixed-16-byte `memcpy` into a fresh bytea with no
error path and no data-dependent control flow, so it cannot leak information
about its argument via errors or timing — the bar for `proleakproof`. Marking
it lets the planner push a cast-bearing qual below a security-barrier view or
RLS policy. `[from-diff]`

Corpus tie-in: no per-file doc for `pg_proc.dat` (catalog `.dat` files are not
per-file-tracked). Relevant to any `[[knowledge/idioms/]]` note on
leakproofness criteria and to RLS/security-barrier qual-pushdown discussion.
catversion bump noted in the Anchor decision.

### `7f5e0b22e5ea` — ECPG compiler NULL deref (Tom Lane)

`ECPGmake_struct_type()` in `interfaces/ecpg/preproc/type.c` did
`ne->struct_sizeof = mm_strdup(struct_sizeof)` unconditionally;
`struct_sizeof` can legitimately be NULL for certain nested-struct host
variable declarations, and `mm_strdup(NULL)` crashed the preprocessor. Guarded:
`ne->struct_sizeof = struct_sizeof ? mm_strdup(struct_sizeof) : NULL;`
`[from-diff]`

Corpus tie-in: `[[knowledge/files/src/interfaces/ecpg/preproc/type.c.md]]`
(per-file doc exists; queue). Per R13's scope ladder an ECPG-only change would
trigger `--suite ecpg`; noted for the human refresh.

### `dbaa4dc3c8dd` — huge_page_size upper bound = MAX_KILOBYTES (Michael Paquier)

`guc_parameters.dat`: `huge_page_size`'s `max` changes from `INT_MAX` to
`MAX_KILOBYTES`. The GUC's unit is KB, so an `INT_MAX`-KB ceiling (~2 PB) was
meaningless and inconsistent with other size-in-KB GUCs which all clamp at
`MAX_KILOBYTES`. A `check_huge_page_size` hook already validates the value
against the OS; this just makes the declared bound honest. `[from-diff]`

Corpus tie-in: no per-file doc for `guc_parameters.dat`. Relevant to any
`[[knowledge/skills/gucs-config]]` note on `GUC_UNIT_KB` / `MAX_KILOBYTES`
ceilings.
