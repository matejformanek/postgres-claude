## Deep dive ‚Äî `9a60f295bcb1` ‚Äî strip removed rels from PHVs at join removal (Richard Guo)

### The bug

`remove_useless_joins()` drops a provably-redundant LEFT JOIN and then
`remove_rel_from_query()` rewrites the surviving data structures to forget the
removed relid (`relid`) and its outer-join relid (`ojrelid`). Historically it
fixed up the relid *sets* of `RestrictInfo`s and `EquivalenceClass`es but **not**
the `PlaceHolderVar`s embedded inside those clauses' expression trees ‚Äî the
comment on `remove_rel_from_eclass` openly said "we don't worry about cleaning
out any nullingrel bits in contained Vars and PHVs." That was fine as long as
nobody recomputed a clause's relids from scratch.

The failure case: a surviving clause containing a PHV gets translated for an
**appendrel child** (e.g. a partition of the kept relation) and has its relids
recomputed by `pull_varnos()`. The PHV's stale `phrels` still names the removed
rel, so `pull_varnos()` reports a reference to a nonexistent RT index ‚Äî an
assertion failure / planner corruption. The new regression cases exercise
exactly this: a `parted_b` self-left-join feeding a partitioned outer rel under
`group by ()` (`src/test/regress/sql/join.sql` @ `9a60f295`, the two new
`explain (costs off)` queries before the `rollback`).

### The fix ‚Äî a PHV-aware mutator

The commit adds `remove_rel_from_phvs()` + `remove_rel_from_phvs_mutator()`
(`analyzejoins.c` @ `9a60f295`, new static functions after
`remove_rel_from_eclass`). The mutator walks an expression tree; on each
`PlaceHolderVar` it:

1. asserts `phlevelsup == 0` (upper-level PHVs are long gone by join-removal
   time),
2. recurses into the contained expression via `expression_tree_mutator`,
3. computes `newphrels = bms_difference(phv->phrels, removable)` where
   `removable = {relid, ojrelid}`,
4. **only** if `newphrels` is non-empty, replaces `phrels` and strips the same
   bits from `phnullingrels`.

The empty-`phrels` guard is the subtle part: a PHV evaluated *only* at the
removed relation would otherwise become a PHV with empty `phrels`, which the
rest of the planner assumes never happens. Such a PHV belongs to an
EquivalenceMember the caller drops immediately afterward, so it's left
untouched rather than corrupted. [[corpus]]
`knowledge/files/src/backend/optimizer/util/placeholder.c.md` documents the
`phrels`/`ph_eval_at` invariants this leans on; [[corpus]]
`knowledge/data-structures/plannerinfo.md` covers `lastPHId`.

### Two call sites, gated on `lastPHId != 0`

Both new call sites short-circuit when the query has no PlaceHolderVars
(`root->glob->lastPHId != 0`), so the common no-PHV plan pays nothing:

- In `remove_rel_from_query()`, for each surviving `otherrel` (when
  `is_outer_join && rti != relid`), every `baserestrictinfo` clause is run
  through `remove_rel_from_phvs()` (`analyzejoins.c` @ `9a60f295`, new
  `foreach_node(RestrictInfo, ‚Ä¶)` block).
- In `remove_rel_from_eclass()` ‚Äî whose signature now takes `PlannerInfo *root`
  ‚Äî every non-`Var`/non-`Const` `em_expr` is run through the same mutator
  **before** the early-return-if-EC-doesn't-mention-the-rel test. This is the
  load-bearing reordering: a member's `em_relids` tracks `ph_eval_at`, not the
  PHV's `phrels`, so a PHV can still name the removed rel even when the EC's
  relid set does not. The old code's `bms_is_member` guard at the call site is
  removed (now done inside the callee). [verified-by-code: diff @ `9a60f295`]

## Deep dive ‚Äî `80bb0ebcc11f` ‚Äî pgcrypto: avoid recursive ResourceOwnerForget() (Tom Lane)

### What changed

pgcrypto registers each live OpenSSL cipher object with the current
ResourceOwner so it gets freed at (sub)transaction end. The release callback is
`ResOwnerReleaseOSSLCipher()` in `contrib/pgcrypto/openssl.c`. Before this
commit it called `free_openssl_cipher((OSSLCipher *) DatumGetPointer(res))`
directly; after, it first stashes the pointer, **NULLs `cipher->owner`**, and
only then calls `free_openssl_cipher(cipher)`:

```c
static void
ResOwnerReleaseOSSLCipher(Datum res)
{
    OSSLCipher *cipher = (OSSLCipher *) DatumGetPointer(res);

    cipher->owner = NULL;
    free_openssl_cipher(cipher);
}
```
[verified-by-code: diff @ `80bb0ebc`]

### Why

`free_openssl_cipher()` is the general teardown path and itself calls
`ResourceOwnerForget()` to deregister the cipher from its owner ‚Äî correct when
freeing is initiated explicitly (e.g. cipher reset). But when the *resource
owner itself* is releasing the cipher (this callback fires during
`ResourceOwnerRelease`), calling `ResourceOwnerForget()` re-enters the owner
that is already mid-release ‚Äî a recursive forget on an owner whose array is
being torn down. NULLing `cipher->owner` first makes `free_openssl_cipher()`'s
forget a no-op, breaking the recursion. This is the standard PG resowner idiom:
the release callback clears the back-pointer so the shared free routine skips the
`ResourceOwnerForget` step. [[corpus]]
`knowledge/issues/pgcrypto.md` tracks pgcrypto resource-management notes;
[[corpus]] the broader resowner-callback contract lives in the resource-owner
material referenced from `knowledge/idioms/`.

### Scope

contrib-only, no catalog or WAL impact, no ABI change. Affects any pgcrypto
cipher object freed via transaction-abort resowner release (the
`encrypt`/`decrypt` path) rather than explicit reset. No per-file corpus doc
exists for `contrib/pgcrypto/openssl.c`, so this commit does **not** by itself
hold the anchor.
