## Deep explainers (behavior-changing + corpus-touching)

### `d007800f02a7` — Fix COUNT's logic for window run condition support (Chengpeng Yan, David Rowley)

The **window run condition** optimization lets the executor stop a
`WindowAgg` early when a qual like `WHERE count(*) OVER (...) <= 3` can never
become true again, provided the window function is monotonic. `int8inc_support`
(the planner support function for `count`) computes that monotonicity claim in
`src/backend/utils/adt/int8.c:795` (`int8inc_support`).

The bug: the old code decided monotonicity purely from the presence of an
`ORDER BY` in the window clause, ignoring the frame's **EXCLUDE** options. An
`EXCLUDE` clause can retroactively remove rows that earlier peers had counted,
which breaks the monotonic-increase assumption and produces **wrong results**
(rows filtered out that should have been returned). The fix adds an explicit
`frameOptions & FRAMEOPTION_EXCLUSION` guard: bail to `MONOTONICFUNC_NONE`
unless the narrow provably-safe shape is present — `COUNT(*)` with
`EXCLUDE CURRENT ROW` and no `FILTER` (excluding the current row always drops
the running `COUNT(*)` by exactly 1, so the qual's collapse point is still
well-defined; `COUNT(anycol)` isn't safe because a NULL current row wouldn't
have been counted). The peers check is also tightened to require RANGE mode
(`orderClause == NIL && (frameOptions & FRAMEOPTION_RANGE)`). New regress rows
in `window.sql` exercise `RANGE BETWEEN CURRENT ROW AND CURRENT ROW`.

This is an **executor/planner-support** correctness fix — see the
[[knowledge/subsystems]] executor + planner-cost-model material and R13's
executor/planner scope tier (any `int8inc_support`-style planner support
function is optimizer-adjacent). `[verified-by-code]` from the fetched diff.

### `9c9ddf109bd8` — Fix gistkillitems for GiST root page (Kirill Reshke)

GiST index scans mark dead tuples with LP_DEAD hints via `gistkillitems()`,
which needs the block number of the page whose items are being killed.
Previously `so->curBlkno` was only assigned inside `gistgettuple`'s descend
path (`src/backend/access/gist/gistget.c:730`-ish), so when the whole index
was a **single root page**, `curBlkno` never got the root's block number and
`gistkillitems` silently did nothing — LP_DEAD hints were dropped on small GiST
indexes, forcing repeated heap visibility checks. The fix hoists the
assignment into `gistScanPage` (`gistget.c:409` region: `so->curBlkno =
pageItem->blkno;`) alongside the existing LSN save, covering the root-page
case. The isolation test `src/test/modules/index/specs/killtuples.spec` flips
its expectation from "killitems doesn't work" (`t`) to working (`f` = not
forced to recheck), documenting the previously-latent bug. **[corpus]**
`knowledge/files/src/backend/access/gist/gistget.c.md`. `[verified-by-code]`.

### `b34fd845e03a` — Remove unnecessary volatile qualifiers (Tom Lane)

A tree-wide cleanup removing `volatile` from shared-memory pointer locals that
were defensive relics from before PG had proper atomics + memory barriers.
Touched (all with existing per-file docs): `clog.c`
(`TransactionGroupUpdateXidStatus` drops `volatile PROC_HDR *procglobal`, uses
`ProcGlobal` directly with the `pg_atomic_*` ops), `index.c`
(`reindex_index`'s `volatile bool skipped_constraint`), `async.c`
(`asyncQueueAdvance`'s `volatile QueuePosition *`), `syncrep.c`
(`SyncRepInitConfig`/`SyncRepReleaseWaiters`/`SyncRepGetCandidateStandbys` drop
`volatile WalSndCtlData *`/`volatile WalSnd *`), `procsignal.c`, `shm_toc.c`,
`lock.c`, `proc.c`, plus two `test_shm_mq` files. This is a **[locking]**-idiom
touchpoint: the `volatile`-pointer-to-shmem pattern is exactly the anti-idiom
the locking corpus warns about (atomics/barriers are the correct mechanism, not
`volatile`). Anyone citing these files for "how PG uses volatile on shmem"
should re-anchor — the answer is now "it mostly doesn't". **[corpus]** (8 files).
`[verified-by-code]`.

### `343594a26d37` + `ffd080d94fe7` — libpq SSL/GSS pending-byte draining (Jacob Champion)

A two-commit pair fixing a subtle client-side hang class. `pqReadData` reads
from the socket, but with SSL or GSS the transport layer can hold
already-decrypted bytes in its own buffer that the kernel's socket-readable
signal won't reflect. A client that polls `PQsocket()` for readability could
then block forever on data that libpq already has buffered.

- `343594a26d37` splits `pqReadData` into `pqReadData_internal` + a new
  `pqDrainPending`, and after any successful read, drains the SSL/GSS transport
  buffer so the post-condition "either the socket is still OS-readable or all
  bytes are consumed" holds (`fe-misc.c:614` region). An `Assert` checks that a
  zero-byte read implies zero pending.
- `ffd080d94fe7` generalizes the pending check: `pgtls_read_pending` (SSL-only,
  `bool`) becomes `pqsecure_bytes_pending` (`ssize_t`, SSL **and** GSS), with
  new `pg_GSS_bytes_pending` in `fe-secure-gssapi.c` and a
  `pgtls_bytes_pending` in `fe-secure-openssl.c` that guards against OpenSSL
  read-ahead breaking `SSL_pending()`. `pqSocketCheck` (`fe-misc.c:1099`) loses
  its `#ifdef USE_SSL`-only branch.

Client-side `src/interfaces/libpq` — no per-file corpus doc, but relevant to
the [[wire-protocol]] skill (transport-buffer vs socket-readable semantics).
`[verified-by-code]`.

### `592de8bd21e1` — Fix pg_dump ACL minimization for PROPERTY GRAPH (Tom Lane)

Property graphs (PG20 `CREATE PROPERTY GRAPH`, relkind `RELKIND_PROPGRAPH`)
had no `acldefault` mapping, so pg_dump couldn't compute the "default ACL" it
diffs against to minimize emitted GRANT/REVOKE. Three sites: `acldefault_sql`
in `utils/adt/acl.c:956` gains `case 'g': objtype = OBJECT_PROPGRAPH;`; the
`pg_property_graph_privileges` view in `information_schema.sql:3328` switches
`acldefault('r', …)` → `acldefault('g', …)`; and `getTables` in `pg_dump.c:7231`
builds the per-relkind `acldefault(CASE … 'g' … END)` expression, with a
back-compat path for dumping from 19beta1 servers that lack `acldefault('g')`
(emit the standard `{owner=r/owner}` array by hand). **[corpus]**
`acl.c.md` + `pg_dump.c.md`. `[verified-by-code]`.

### `650bb73c137c` — Rename register_unlink_segment → register_unlink_tombstone (Matthias van de Meent)

`md.c`'s deferred-unlink machinery only ever schedules the **tombstone** — the
empty first segment (`MAIN_FORKNUM`, segno 0) left behind by `mdunlink()` to
reserve the relfilenode until the next checkpoint. The old name/signature took
`forknum`/`segno` args that were always the same constants. The rename drops
the dead parameters, hardcodes `INIT_MD_FILETAG(tag, rlocator.locator,
MAIN_FORKNUM, 0)`, adds a clarifying header comment, and asserts
`ftag->forknum == MAIN_FORKNUM && ftag->segno == 0` in `mdunlinkfiletag`
(`md.c:1958` region). Pure refactor, no behavior change. **[corpus]**
`md.c.md` — the doc's description of the unlink-tombstone flow should track the
new name. `[verified-by-code]`.

### `21ffc271d475` — unicode_case.c: defend against truncated UTF8 (Jeff Davis)

Case-folding/word-boundary code in `unicode_case.c` and
`pg_locale_builtin.c` (`initcap_wbnext`) called `utf8_to_unicode()` +
`unicode_utf8len()` on the current offset without first checking that a full
multibyte sequence was present, so a truncated trailing byte (e.g. `"abc\xCE"`)
could read past the buffer. The fix computes the byte length up front (a new
inlinable `utf8_mblen()` local + `pg_utf_mblen()` in the wb path), and if
`offset + ulen > len` treats it as invalid UTF8 and stops cleanly. New
`case_test.c` assertions cover truncated (`"abc\xCE"`) and invalid-byte
(`"abc\xF8xyz"`) inputs, both expecting `needed == 3`. **[corpus]**
`pg_locale_builtin.c.md` + `unicode_case.c.md`. Ties to the
[[collation-provider]] builtin-provider material. `[verified-by-code]`.

### `04fc2564fbba` — Don't cast pgoff_t to possibly 32-bit types for output (Nathan Bossart)

Backend half of a two-commit offset-formatting cleanup (client-side half is
`c22d2f7fd46c`). Several `errmsg`/`elog` sites truncated 64-bit file offsets by
casting `pgoff_t`/`off_t` to `(uint32)`/`(unsigned)`: `rewriteheap.c`
(truncate-file error), `fd.c` (short-write error at offset), `walsummary.c`,
`walsender.c` (timeline-history file — plus a new
`histfilelen > UINT32_MAX → elog(ERROR)` guard documenting the remaining
implicit narrowing), and `pg_upgrade/slru_io.c`. All switch to
`(long long)` + `%lld`. **[corpus]** (`rewriteheap.c.md`, `walsummary.c.md`,
`walsender.c.md`, `fd.c.md`). `[verified-by-code]`.

### `bb4142fb68cf` — postgres_fdw: Report ANALYZE to pgstats after importing statistics (Fujii Masao)

When a foreign table is created/altered with `restore_stats 'true'`,
`postgresImportForeignStatistics` copies the remote's pg_statistic rows locally.
Previously that path never told the cumulative-stats system an analyze had
happened, so `pg_stat_get_analyze_count`/live/dead-tuple counters stayed stale.
The fix records `starttime = GetCurrentTimestamp()` and calls
`pgstat_report_analyze(relation, remstats.livetuples, remstats.deadtuples,
(va_cols == NIL), starttime)` (`postgres_fdw.c:5665` region), adding
`livetuples`/`deadtuples` to the remote-stats struct and including `pgstat.h`.
New `postgres_fdw.out` rows check the analyze count = 1 and live-tuple count
after import. Ties to the [[pgstat-framework]] skill (the recurring
harvester-flagged subsystem). `[verified-by-code]`.
