## Deep explainers

### `a75bd485b5ea` ‚Äî Clean up quoting of variable strings within replication commands

**What changed.** Backend and frontend code that *builds* replication
commands (`START_REPLICATION`, `CREATE_REPLICATION_SLOT`,
`DROP_REPLICATION_SLOT`, `ALTER_REPLICATION_SLOT`,
`READ_REPLICATION_SLOT`, `TIMELINE_HISTORY`) was quoting interpolated
slot names / option values sloppily ‚Äî typically
`appendStringInfo(&cmd, " SLOT \"%s\"", slotname)` with no escaping at
all. Tom Lane replaced every such site with a tiny purpose-built
`appendQuotedString()` that just *doubles* the relevant quote character,
plus `appendQuotedIdentifier`/`appendQuotedLiteral` wrapper macros.

**Why the old code was wrong on three counts** (from the commit log):
1. **Injection surface.** Raw `"%s"` interpolation lets a slot name
   containing `"` break out of the quoted token ‚Äî an option-injection
   risk *if* slot names/params ever come from untrustworthy input.
   Reported as a security bug (Reported-by: Team Dhiutsa) but the
   security team downgraded it to a regular bug (slot names from
   hostile sources judged "very improbable"). Still backpatched to 14.
2. **Wrong escaping primitive.** Several sites used `PQescapeLiteral()`,
   which emits `E'‚Ä¶'` syntax when the string contains a backslash ‚Äî
   and `repl_scanner.l` doesn't recognize `E'‚Ä¶'`. So a backslash in an
   option value produced a *syntax error*, not just a security worry.
3. **`quote_identifier()` misuse.** It won't quote a valid lowercase
   identifier unless it matches a *SQL* keyword ‚Äî but the replication
   grammar has its *own* keyword set, so the wrong things got quoted.

**The prerequisite scanner fix.** The doubled-quote approach only works
if the replication scanner *un-doubles* on the read side. It didn't ‚Äî
so the patch first teaches `repl_scanner.l` to collapse `""` inside a
double-quoted identifier, mirroring the existing `<xq>` rule for single
quotes:
```
<xd>{xddouble}	{ addlitchar('"', yyscanner); }    [from-commit-diff: repl_scanner.l +200]
```
This is the load-bearing change: without it, every newly-doubled
identifier would have arrived at the parser with literal `""` in it.

**Sites touched (cite map, post-commit master lines)** `[from-commit-diff]`:
- `commands/subscriptioncmds.c:521` ‚Äî new `appendQuotedString()` +
  macros; `DROP_REPLICATION_SLOT ‚Ä¶ WAIT` rebuilt at `:2547`.
- `replication/libpqwalreceiver/libpqwalreceiver.c:525` ‚Äî same helper
  (duplicated, not shared ‚Äî the walreceiver is a separately-linked
  `.so`); `libpqrcv_startstreaming` (`SLOT`, `streaming`, `origin`,
  `publication_names`) at `:573`, `libpqrcv_create_slot` at `:918`,
  `libpqrcv_alter_slot` at `:1025`. Notably this **removes** the
  `PQescapeLiteral`/`PQescapeIdentifier` calls (and their OOM-`ereport`
  guards) from `stringlist_to_identifierstr()`, which loses its now-unused
  `PGconn *conn` parameter (`:1232`).
- `bin/pg_basebackup/{pg_recvlogical.c:251, receivelog.c:574,
  streamutil.c:504}` ‚Äî frontend side gains `AppendQuotedIdentifier`/

### `e0511883cae2` ‚Äî Fix PQdescribePrepared with more than 7498 params

**What changed.** A one-line frontend fix. libpq's protocol-3 reader
(`fe-protocol3.c`) has a `VALID_LONG_MESSAGE_TYPE(id)` macro listing the
backend message types allowed to exceed the 30000-byte sanity cap on
incoming messages. `ParameterDescription` (`PqMsg_ParameterDescription`,
the `t` message replying to a Describe-on-prepared-statement) was *not*
in the list:
```
 (id) == PqMsg_RowDescription || \
 (id) == PqMsg_ParameterDescription)    [from-commit-diff: fe-protocol3.c +45]
```
**Why 7498.** A `ParameterDescription` carries a 4-byte type OID per
parameter plus a small fixed header (1 byte type + 4 byte length + 2
byte param count = 7). `7498 √ó 4 + 7 = 29999`; one more param tips it
over 30000 and the reader rejects the (perfectly valid) message. So any
prepared statement with ‚â•7499 parameters made `PQdescribePrepared()`
fail. Adding the type to the long-message allowlist lifts the cap for
that message only.

**Corpus impact.** Touches
[[knowledge/files/src/interfaces/libpq/fe-protocol3.c.md]]. Good fodder
for the recurring **wire-protocol** gap flagged by
`pg-user-question-harvester` ‚Äî the 30000-byte `VALID_LONG_MESSAGE_TYPE`
threshold and *which* message types are exempt is exactly the kind of
protocol detail the corpus under-documents.

`Backpatch-through: 14 ¬∑ Author: Ning Sun`

### `ca6969629d85` ‚Äî Modernize pg_bsd_indent's error/warning reporting code

**What changed.** Pure tooling cleanup of the `pgindent` companion
`pg_bsd_indent`. The three near-identical `diag2/diag3/diag4(level, msg,
[a], [b])` functions (fixed 0/1/2 trailing `int` args) collapse into one
varargs `diag(int level, const char *msg, ...)` marked
`pg_attribute_printf(2, 3)`; `diag2/3/4` survive as backwards-compat
macros in `indent.h`:
```
void diag(int level, const char *msg, ...) pg_attribute_printf(2, 3);   [indent.h +42]
#define diag3(level, msg, a) diag(level, msg, a)                        [indent.h +54]
```
**Why.** Late-model clang's `-Wformat` wants the printf attribute on
these functions, and it's correct. Two call sites in `indent.c` passed
`size_t` (`nitems(...)`) through the old `int`-typed `diag3`, relying on
the implicit `size_t‚Üíint` coercion; with a real printf attribute their
`%d` formats had to become `%zu` (`indent.c:539`, `:812`). The
`io.c:553` bodies switch from `fprintf(‚Ä¶, a, b)` to `vfprintf(‚Ä¶, ap)`.

**Why it matters to us.** No functional change, but back-patched to 16
specifically to **suppress compiler warnings** on the buildfarm ‚Äî a
reminder that `src/tools/pg_bsd_indent` is warning-clean-on-new-clang
territory, and that `pg_attribute_printf` is the canonical PG spelling
of `__attribute__((format(printf,‚Ä¶)))`. Relevant to the `coding-style`
skill's pgindent section. No per-file corpus doc exists for
`pg_bsd_indent` (not core backend) ‚Äî left uncrossreferenced.
