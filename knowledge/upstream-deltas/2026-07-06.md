## Deep explainers

### тШЕ `42b7ff3aaefa` тАФ leading-`\` handling in nondeterministic LIKE [utils/adt, hot path]

Tom Lane, backpatch-through 18. A follow-on to `85b7efa1c`. The main loop of
`MatchText()` handled a leading `\` *before* the nondeterminism branch, forcing
a **literal byte match** for the escaped character. Two bugs followed: (1) an
ordinary character that should be subject to nondeterministic matching was
instead matched bytewise; (2) if the escaped character was multibyte, both
pointers advanced by exactly one byte, so the next iteration entered the
nondeterministic path *in the middle of a character* тАФ potentially making
`pg_strncoll()` misbehave. The fix moves the `\` stanza *below* the
nondeterminism stanza (the nondeterministic code already de-escapes correctly),
so `\` is only special-cased on the deterministic path. A `bool
nondeterministic = (locale && !locale->deterministic)` local is hoisted out of
the hot loop, replacing repeated `locale && !locale->deterministic` fetches.
New regress rows in `collate.icu.utf8.sql` cover `'AB' LIKE '\ab'`, `'\a%'`,
`'\a\%'` under `case_insensitive`.
Corpus: `[[knowledge/files/src/backend/utils/adt/like_match.c.md]]` needs a
re-anchor (loop structure reordered; new `nondeterministic` local).

### тШЕ `d6ffcae32a10` тАФ de-escaping literal `\\` in nondeterministic LIKE [utils/adt]

Nitin Motiani (bug #19474), backpatch-through 18. Sibling to the above, in the
*subpattern de-escape* loop of `MatchText()`. The old code
`if (*c == '\\') ; else *(b++) = *c;` **unconditionally skipped every
backslash**, so a literal `\\` (an escaped backslash) emitted *nothing*
instead of one `\`. Fix: `if (*c == '\\') c++; *(b++) = *c;` тАФ on a backslash,
advance to the escaped byte and emit *that* (the loop had already verified it
isn't the last byte). `'back\slash' LIKE 'back\slash%'` now matches, and
`'backslash' LIKE 'back\\slash%'` correctly returns false.
Corpus: same `like_match.c` re-anchor row.

### тШЕ `2d7808e6fc2c` тАФ restore exact-match index optimization across collations [like_support.c, planner support fn]

Jelte Fennema-Nio, backpatch-through 18. `85b7efa1c` accidentally broke the
`Pattern_Prefix_Exact` тЖТ equality-indexqual conversion when the index collation
differs from the expression collation. The old guard `if (indexcollation !=
expr_coll) return NIL;` rejected the optimization outright. The correct rule:
a **deterministic** expression collation agrees with every other deterministic
collation on equality (it is bytewise), and an index built under a
nondeterministic collation returns a *superset* of the bytewise-equal entries тАФ
which the (lossy-by-default) `=` indexqual then rechecks via the original
LIKE/regex operator. New macro `NONDETERMINISTIC(coll)` (= `OidIsValid(coll)
&& !get_collation_isdeterministic(coll)`) captures the "bytea has InvalidOid
collation тЗТ deterministic" case cleanly. Guard becomes `if (indexcollation !=
expr_coll && NONDETERMINISTIC(expr_coll)) return NIL;`. The headline
beneficiary named in the commit: **psql `\d tablename`** тАФ without this fix it
seqscans `pg_class` (whose catalogs use `"C"`, which the optimizer doesn't see
as identical to `"POSIX"`) instead of an index point-lookup.
Corpus: `[[knowledge/files/src/backend/utils/adt/like_support.c.md]]` re-anchor
(new `NONDETERMINISTIC` macro; `match_pattern_prefix` guard logic).

### тШЕ `44fb59fc605e` тАФ qual pushdown past grouping with mismatched equivalence [optimizer, correctness]

Richard Guo, backpatch-through 18. **The most consequential commit this run.**
Two long-standing planner optimizations move a qual across a grouping boundary:
`subquery_planner` transfers HAVING clauses to WHERE (pre-aggregation), and
`qual_is_pushdown_safe` pushes outer restrictions into a subquery past its
`DISTINCT` / `DISTINCT ON` / window `PARTITION BY` / set-op grouping. Both give
**wrong results** when the moved clause's equivalence relation disagrees with
the grouping's, because the clause then filters rows the grouping would have
merged. Two failure forms: (a) a type in multiple btree opfamilies whose
equalities disagree (`record_ops` vs `record_image_ops`); (b) a nondeterministic
collation, where comparing the column under a *different* collation тАФ or
wrapping it in a function/operator тАФ can distinguish values the collation deems
equal.

The fix adds a **shared walker** `expression_has_grouping_conflict()` in
`clauses.c` (new decl in `include/optimizer/clauses.h`), parameterized by a
`grouping_eqop_callback` that maps each `Var` to the grouping equality operator
for its column (or `InvalidOid` for non-grouping Vars). A grouping column is
safe to reference **only as a direct operand of a comparison under its own
collation/eqop**; for nondeterministic collations every other reference (a
different collation, or any function/operator over the column) is rejected. Two
call sites supply callbacks: HAVING recovers the `SortGroupClause->eqop` via the
GROUP `Var`'s `varattno` (which requires running *before* `flatten_group_exprs`
while `havingQual` still holds GROUP Vars тАФ this is why the patch is v18+, it
relies on the `RTE_GROUP` mechanism from `247dea89f`); subquery pushdown
recovers the eqop from `distinctClause`, a window's `partitionClause`, or any
grouping node in the `SetOperationStmt` tree (new
`subquery_column_grouping_eqop` / `setop_column_grouping_eqop` /
`setop_has_grouping` statics in `allpaths.c`). The walker fires only when there
*is* an equivalence boundary (gated by the existing
`UNSAFE_NOTIN_DISTINCTON_CLAUSE` / `UNSAFE_NOTIN_PARTITIONBY_CLAUSE` flags or a
recursive set-op grouping check). A documented residual gap: with a
deterministic collation, a function feeding a finer comparison (e.g.
`record_image_ops` over a rebuilt record, or `scale()` over numeric) is *not*
caught, because that would require knowing a type's equality is bitwise.
Corpus (all re-anchor): `[[knowledge/files/src/backend/optimizer/path/allpaths.c.md]]`,
`[[knowledge/files/src/backend/optimizer/plan/planner.c.md]]`,
`[[knowledge/files/src/backend/optimizer/util/clauses.c.md]]`,

### тШЕ `e8914ec22f8f` тАФ `satisfies_hash_partition` VARIADIC NULL crash [partitioning]

Robert Haas, backpatch-through 14. `satisfies_hash_partition(regclass, modulus,
remainder, VARIADIC ...)` crashed on `VARIADIC NULL::sometype[]` because both
the modulus/remainder-supplied path and the variadic-array path called
`PG_GETARG_ARRAYTYPE_P(3)` without a `PG_ARGISNULL(3)` guard, dereferencing a
NULL array. A companion fix (`f3b0897a1213`, PG 11) had handled *related* NULL
cases but overlooked this one. The fix adds `PG_ARGISNULL(3)` short-circuits
returning `false` in both branches (and relocates the `variadic_array`
declaration+fetch below the guard in the second branch). New regress row:
`SELECT satisfies_hash_partition('mchash'::regclass, 4, 0, VARIADIC
NULL::int[]);` тЖТ `f`.

### тШЕ `763ee7ea00b0` тАФ drop switch statements in `vector8_shift_{left,right}` [port/simd]

Nathan Bossart (reviewed John Naylor). Cleanup of `ec8719ccbf`: AArch64
`vshlq_n_u32`/`vshrq_n_u32` require an *integer-literal* shift count, which had
forced a `switch` over every caller's expected shift value. Both are replaced
with `vshlq_u32(v, vdupq_n_s32(i))` (and `-i` for the right shift, since a
negative NEON shift count means shift-right), which takes a runtime vector shift
count and compiles to the same machine code on modern compilers. Removes the
"if you add a caller, update the switch" maintenance hazard.
