## Deep explainers

### ★ `2e606d75c0bf` — min()/max() aggregate for `uuid` [catalog]

`uuid` already had a full comparison operator set and a btree opclass (it is
totally ordered), so `min()`/`max()` were the only common aggregates missing.
The commit adds two fmgr support functions in
`source/src/backend/utils/adt/uuid.c` — `uuid_larger()` and `uuid_smaller()`,
each a two-line wrapper over the existing `uuid_internal_cmp()`
(`PG_RETURN_UUID_P((cmp > 0) ? arg1 : arg2)`) — and registers `max(uuid)` /
`min(uuid)` in `pg_aggregate.dat` with `aggsortop => '>(uuid,uuid)'` (resp.
`<`) and `aggtranstype => 'uuid'`, mirroring the `bytea_larger`/`bytea` rows.
`CATALOG_VERSION_NO` bumps `202606302 → 202607011`. Because `uuid` values
compare lexicographically over all 128 bits, for **UUIDv7** (timestamp in the
high bits) `min`/`max` return oldest/newest — a documented, intended
consequence.
Corpus: `[[knowledge/files/src/backend/utils/adt/uuid.c.md]]` needs a
re-anchor row (new SRF-less builtins). `pg_aggregate.dat` / `pg_proc.dat`
have no per-file doc (untracked). This is a textbook `catalog-conventions` +
`fmgr-and-spi` change — see those skills for the aggsortop/aggtranstype
pattern.

### ★ `7d3448961da3` — btree_gist NaN handling for float4/float8 [contrib, GiST opclass]

Bugfix (#19501, #19524), backpatch-through 14. The `float4`/`float8`
`btree_gist` opclasses compared keys with raw C `==`/`<`/`>`. IEEE-754 makes
every NaN comparison false, so GiST disagreed with both the regular float
comparison operators and the core btree opclass (which uses
`float[4|8]_cmp_internal()`, treating all NaNs as equal and sorting NaN after
every non-NaN). The penalty and distance functions were also NaN-unsafe, and
penalty could misbehave on IEEE infinities; the distance bug was visible from
SQL. Fix: route comparisons through the same NaN-aware core functions, and
rewrite penalty/distance to the rule "NaNs are equal but maximally far from
non-NaNs". `penalty_num()`'s shared integral/float macro is split into two
`static inline` functions so the integral path isn't burdened with NaN/inf
logic. **No on-disk format change, but a `btree_gist` index containing NaN
entries should be REINDEXed.** Corpus: contrib/btree_gist has no per-file doc;
relevant to `access-method-apis` (GiST support-function semantics).

### ★ `181b6185c79e` — better auto-generated names for expression indexes [commands, parser]

Previously an index on an expression got the column name literally `"expr"`
(so `CREATE INDEX ON t ((f(x)))` → `t_expr_idx`), and multiple expression
indexes collided into `expr`, `expr1`, … The commit decouples index-column
naming from `FigureColname` and adds `ChooseIndexExpressionName()` +
`ChooseIndexExpressionName_walker()` in
`source/src/backend/commands/indexcmds.c` (new `CIEN_context { Relation rel;
StringInfo buf; }`). The walker digs through the expression, extracting Var
names, Const string reps, and function names, joined by underscores; operator
names are deliberately ignored (`a + b` and `a * b` both → `a_b`) and Const
reps are stripped to alphanumerics (`x + 1.0` → `x_10`) — both to avoid
needing to double-quote the generated name. `ChooseIndexColumnNames()` gains a
`Relation rel` parameter (signature change threaded through `DefineIndex`).
Regression `.out` churn is broad (`create_index`, `indexing`, `alter_table`,
`create_table*`, `inherit`, `rangetypes`, plus `contrib/seg`), e.g.
`pt12_expr_idx` → `pt12_mydouble_category_1_idx`. Landed at cycle start (PG 19
devel) to gather feedback; naming rules explicitly "may be tweaked". Corpus:
`[[knowledge/files/src/backend/commands/indexcmds.c.md]]`,
`[[knowledge/files/src/backend/parser/parse_target.c.md]]`,
`[[knowledge/files/src/backend/parser/parse_utilcmd.c.md]]`,
`[[knowledge/files/src/include/nodes/parsenodes.h.md]]` all need re-anchor
(new function, changed `ChooseIndexColumnNames` signature, new `parse_target.h`
export). Cross-cuts `parser-and-nodes` (expression-tree walker idiom).

### ★ `55f0a13e96be` — clear base-backup progress on failure [backup]

Bugfix, backpatch-through 15. `pg_stat_progress_basebackup` could show a stale
entry after a base backup that failed mid-stream, if the client kept the
replication connection open (normally invisible with `pg_basebackup`, which
disconnects on error). Root cause: progress cleanup ran only on success —
`basebackup_progress_done()` was called at the tail of
`perform_base_backup()` in `source/src/backend/backup/basebackup.c`. Fix moves
the cleanup into a new `bbsink_progress_cleanup()` callback on the progress
sink (`basebackup_progress.c`), so it runs on both success and failure paths
via the sink's `.cleanup` op. Corpus:
`[[knowledge/files/src/backend/backup/basebackup.c.md]]`,
`[[knowledge/files/src/backend/backup/basebackup_progress.c.md]]`,
`[[knowledge/files/src/include/backup/basebackup_sink.h.md]]` need re-anchor.

### ★ `f6fdc2a4a737` — MD5 deprecation warning on the `password` auth path [libpq]

Bugfix, backpatch-through 19. `bc60ee860` added a post-auth connection warning
for MD5-encrypted passwords, but only on the `md5` auth **method**. A role with
an MD5-encrypted password can also authenticate via the `password` method,
which was left without the warning. The commit adds two statics in
`source/src/backend/libpq/auth.c` — `md5_password_warning_enabled()` and
`queue_md5_password_warning()` — the latter switching to `TopMemoryContext`,
`pstrdup`-ing the warning+detail, and calling
`StoreConnectionWarning(..., md5_password_warning_enabled)`.
`CheckPasswordAuth()` now records `md5_password = (get_password_type(shadow_pass)
== PASSWORD_TYPE_MD5)` and, on `STATUS_OK`, queues the warning before
`set_authn_id()`; `CheckMD5Auth()` is refactored to call the same helper.
Gated by the `md5_password_warnings` GUC (default on). Corpus:
`[[knowledge/files/src/backend/libpq/auth.c.md]]`,
`[[knowledge/files/src/backend/libpq/crypt.c.md]]` re-anchor. Touches the
`memory-contexts` (TopMemoryContext for connection-lifetime strings) and
`gucs-config` idioms.

### ★ `e8f851d61727` — don't truncate off_t to 32-bit in error message [backup]

Bugfix. In `source/src/backend/backup/basebackup_server.c`, two short-write
`ereport`s formatted the file offset with `%u` and `(unsigned) mysink->filepos`,
truncating a 64-bit `off_t` to 32 bits. Base backups routinely exceed 4 GB, so
the offset in a disk-full error could be wrong. Changed to `%lld` +
`(long long) mysink->filepos`. Commit notes "more issues like this in the tree"
but separates this one as a real bug. Corpus:
`[[knowledge/files/src/backend/backup/basebackup_server.c.md]]` re-anchor.

### ★ `be69a5ff1fd9` — tighten UNION output row-count estimate [planner]

Planner estimation fix. A `UNION` (not `UNION ALL`) dedups, so its output has
no more rows than its input, but the planner set the set-op relation's row
count to the full appended input size — an inflated estimate that propagated
upward and caused bad plan choices (e.g. hash join + seqscan where an index
nestloop was cheaper). In `source/src/backend/optimizer/prep/prepunion.c`,
`generate_union_paths()` now accumulates a per-child distinct-group estimate
`dNumChildGroups`: for each non-dummy `UNION` child it either takes
`rel->rows` (non-subquery) or the group estimate returned by
`build_setop_child_paths()` (now passed `&childGroups` when `!op->all`). The
result rel's `rows` is initialized to the appended size (correct for
`UNION ALL`) then overwritten with the summed distinct-group count for `UNION`.
Rests on `distinct(A ∪ B) ≤ distinct(A) + distinct(B)` and each child's group
estimate ≤ its row count, so the new estimate only ever tightens the old one.
Corpus: `optimizer/prep/prepunion.c` per-file doc re-anchor. Directly relevant
to `executor-and-planner` (cost/cardinality estimation).
