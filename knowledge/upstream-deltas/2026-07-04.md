## Deep explainers

### 1. `a8c2547eaac7` — Disallow renaming a rule to "_RETURN" (Tom Lane)

**Files:** `src/backend/rewrite/rewriteDefine.c` (function `RenameRewriteRule`).
**Class:** correctness / catalog-integrity guard. **No catversion bump. Not backpatched** (per the commit; a HEAD-only tidy-up of an old asymmetry).

**The invariant.** PostgreSQL nails a hard equivalence between "the rule that
defines a view" and "the rule literally named `_RETURN`". A view's single
`ON SELECT` rule is *always* named `ViewSelectRuleName` = `"_RETURN"`
[verified-by-code: `rewriteDefine.c:857-865` @ `a8c2547e`, and the creation-side
comment at `rewriteDefine.c:448-452` "the rule must be named _RETURN … prevents
accidentally or maliciously replacing a view's query"]. `RenameRewriteRule`
already enforced **one** direction of that equivalence:

```c
/* We disallow renaming ON SELECT rules, because they should always be
 * named "_RETURN". */
if (ruleform->ev_type == CMD_SELECT + '0')
    ereport(ERROR, … errmsg("renaming an ON SELECT rule is not allowed"));
```

**The gap it closes.** Nothing stopped the *converse*: renaming some ordinary
`ON INSERT`/`ON UPDATE`/`ON DELETE` rule *to* `_RETURN`. That would plant a rule
named `_RETURN` on a relation whose event type is not `CMD_SELECT`, breaking the
"`_RETURN` ⇒ view-defining SELECT rule" assumption that other code relies on.
The new block is the mirror check [verified-by-code: `rewriteDefine.c:856-865`
@ `a8c2547e`]:

```c
/* Conversely, if it's not an ON SELECT rule then it must *not* be named
 * _RETURN. */
if (strcmp(newName, ViewSelectRuleName) == 0)
    ereport(ERROR,
            (errcode(ERRCODE_INVALID_OBJECT_DEFINITION),
             errmsg("non-view rule for \"%s\" must not be named \"%s\"",
                    RelationGetRelationName(targetrel), ViewSelectRuleName)));
```

Note the encoding idiom `ruleform->ev_type == CMD_SELECT + '0'`: `ev_type` in
`pg_rewrite` is stored as a single `char` digit, so the C enum `CMD_SELECT` is
compared after adding `'0'` [from-comment/verified-by-code:
`rewriteDefine.c:848-850` @ `a8c2547e`]. The guard fires *before* the
`CatalogTupleUpdate`, so no partial catalog state is possible.

**Corpus impact.** `[[knowledge/files/src/backend/rewrite/rewriteDefine.c]]` —
its `RenameRewriteRule` section should note the now-symmetric two-way `_RETURN`
guard. Ripples into `[[subsystems/parser-and-rewrite]]` (view/rule invariants).

### 2. `e0ff7fd9aa2e` — Make property graph object descriptions better translatable (Peter Eisentraut)

**Files:** `src/backend/catalog/objectaddress.c` (`getObjectDescription`, the
`Propgraph*RelationId` cases). **Class:** i18n / message quality. **No catversion
bump. Not backpatched** — SQL/PGQ property graphs are a PG-19-devel feature.

**The problem.** The property-graph object-description cases built their strings
by appending fragments — a translatable noun spliced onto a separately-built
relation description — which gives translators disconnected pieces and no way to
reorder for their language's grammar. The fix switches every case to a single
gettext-marked **full-sentence template** with `%s` placeholders, so the
complete phrase is one translation unit [verified-by-code:
`objectaddress.c:4101-4107, 4138-4140, 4167` @ `e0ff7fd9`]:

```c
if (pgeform->pgekind == PGEKIND_VERTEX)
    appendStringInfo(&buffer, _("vertex %s of %s"), NameStr(pgeform->pgealias), rel.data);
else if (pgeform->pgekind == PGEKIND_EDGE)
    appendStringInfo(&buffer, _("edge %s of %s"),   NameStr(pgeform->pgealias), rel.data);
else
    appendStringInfo(&buffer, "??? element %s of %s", …);   /* NOT translated: dev-only */
…
appendStringInfo(&buffer, _("label %s of %s"),
                 get_propgraph_label_name(pgelform->pgellabelid),
                 getObjectDescription(&oa, false));
```

Two idioms worth flagging for the corpus:
- The `"??? element …"` fallback is deliberately **left out of `_( )`** — a
  can't-happen developer diagnostic, not a user-facing string, so it stays
  untranslated [verified-by-code: `objectaddress.c:4106` @ `e0ff7fd9`]. This is
  the standard PG convention: only translate strings a user can actually see.
- Nested description composition — the element-label case recurses via
  `getObjectDescription(&oa, false)` on the parent element and embeds the result
  as the second `%s` [verified-by-code: `objectaddress.c:4138-4140` @
  `e0ff7fd9`], so the outer template owns word order across the whole "label X
  of vertex Y of relation Z" chain.

**Corpus impact.** `[[knowledge/files/src/backend/catalog/objectaddress.c]]` —
the `getObjectDescription` notes should record the property-graph cases and the
"full-sentence `_( )` template, untranslated dev-fallback" i18n idiom (a good
exemplar for `[[idioms/ereport]]`-adjacent message style, though this is
`getObjectDescription` not `ereport`).
