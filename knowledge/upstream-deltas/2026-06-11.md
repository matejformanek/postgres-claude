## Deep dives

### 1. `99db753c6569` ‚Äî Fix type confusion in AddRelsyncInvalidationMessage (Robert Haas)

`AddRelsyncInvalidationMessage()` queues a `SharedInvalRelSyncMsg` onto the
relcache message subgroup. The `SharedInvalidationMessage` union has separate
arms `rc` (`SharedInvalRelcacheMsg`) and `rs` (`SharedInvalRelSyncMsg`), and the
builder was writing through the **wrong** arm: `msg.rc.id/dbId/relId` and reading
duplicates via `msg->rc.*`, even though the message it constructs carries the
`SHAREDINVALRELSYNC_ID` tag `[from-diff]`. The fix swaps every access to the `rs`
arm (`msg.rs.id/dbId/relid`, `msg->rs.*`) at `utils/cache/inval.c:511‚Äì523`
`[from-diff]`.

Why it's harmless *today*: the commit message states the two structs are
byte-identical except the field is spelled `relId` vs `relid`, so the emitted
bytes are the same. It's a correctness/clarity fix that removes a trap for any
future change that makes the two layouts diverge (e.g. a new field on only one).
Touches `[[knowledge/files/src/backend/utils/cache/inval.c.md]]`, which documents
the `ProcessMessageSubGroup` / `InvalidationMsgsGroup` machinery ‚Äî the doc's
description of the relsync message path should note that the *builder* and the
*apply* side must agree on the union arm.

### 2. `7dd15325952f` ‚Äî IS JSON / JSON(): protect against expressions uncoercible to text (√Ålvaro Herrera)

`transformJsonParseArg()` handles the input expression to `JSON(...)` /
`IS JSON`. When the argument's type is `UNKNOWNOID` or in the string type
category, it implicit-casts to `text`. The bug: it took the result of
`coerce_to_target_type(..., TEXTOID, ...)` without checking for NULL
`[from-diff]`. `coerce_to_target_type()` returns NULL when no cast path exists ‚Äî
which never happens for built-in string-category types (all have a text cast),
but **can** for a user-defined type placed in `TYPCATEGORY_STRING` without a
text cast. The NULL then became `JsonIsPredicate.expr`, and the executor
dereferenced it ‚Üí crash.

The fix, at `parser/parse_expr.c:4202‚Äì4215` `[from-diff]`, captures
`exprLocation(expr)` before the coercion and, if the coercion returns NULL,
raises `ERRCODE_CANNOT_COERCE` `errmsg("cannot cast type %s to %s", ...)` with a
`parser_errposition`. A belt-and-suspenders `Assert(expr != NULL)` is also added
to `makeJsonIsPredicate()` at `nodes/makefuncs.c:991` `[from-diff]`. Regression
coverage lands in `sqljson.sql`. Introduced by `6ee30209a6f1`; backpatch to 16.
Touches `[[knowledge/files/src/backend/parser/parse_expr.c.md]]` and
`[[knowledge/files/src/backend/nodes/makefuncs.c.md]]`.

This is a textbook case of the **`error-handling` skill** rule that fmgr/coerce
helpers returning NULL-on-failure must be NULL-checked before the result flows
into a node that a later phase trusts non-NULL.

### 3. `79c65b9d97fe` ‚Äî Fix parsing of parenthesised OLD/NEW in RETURNING list (Dean Rasheed)

PG18's `RETURNING OLD/NEW` lets a RETURNING list reference both the pre- and
post-image of a row via the `varreturningtype` field on `Var`. The bug surfaced
for *parenthesised whole-row / field projections* like `(old).colname` and
`(old).*`: the code paths that expand a whole-row `Var` resolved the namespace
item with `GetNSItemByRangeTablePosn(pstate, varno, varlevelsup)`, which keys
only on rtindex + nesting depth and so could return the **wrong**
`ParseNamespaceItem` ‚Äî one whose `p_returning_type` didn't match the Var's
`varreturningtype` `[from-diff]`. Result: `(old).x` could resolve as if it were
`(new).x`.

The fix introduces `GetNSItemByVar(ParseState *, Var *)` at
`parser/parse_relation.c:540‚Äì567` `[from-diff]`, which walks `p_namespace` and
matches on **both** `p_rtindex == var->varno` **and**
`p_returning_type == var->varreturningtype` (declared at
`include/parser/parse_relation.h:35` `[from-diff]`). Three call sites switch to
it: `coerce_record_to_complex()` (`parse_coerce.c:1038`),
`ParseComplexProjection()` (`parse_func.c:2059`), and `ExpandRowReference()`
(`parse_target.c:1454`) `[from-diff]`. `GetNSItemByRangeTablePosn()` gains an
`NB:` comment steering Var-origin callers to the new function. Bug #19516;
backpatch to 18. Touches the five parser per-file docs listed in the refresh
report (`parse_coerce.c`, `parse_func.c`, `parse_relation.c`, `parse_target.c`,
plus the header) ‚Äî note the new public symbol `GetNSItemByVar` for
`[[knowledge/files/src/include/parser/parse_relation.h.md]]`.

This is squarely **`parser-and-nodes` skill** territory:
`ParseNamespaceItem` resolution keyed on the *full* identity of a `Var`
(including `varreturningtype`), not just its range-table position.

### 4. `0e1f1ed157e9` ‚Äî seg: preserve the upper boundary's certainty indicator (Heikki Linnakangas)

`contrib/seg` stores a floating-point range with optional `<`/`>`/`~`
"certainty" markers on each boundary. `seg_out()` reconstructs the textual form.
The upper-boundary branch decided whether to emit the certainty char with:

```c
if (seg->u_ext == '>' || seg->u_ext == '<' || seg->l_ext == '~')   /* before */
if (seg->u_ext == '>' || seg->u_ext == '<' || seg->u_ext == '~')   /* after  */
```

at `contrib/seg/seg.c:155` `[from-diff]`. The third disjunct tested `l_ext`
(the **lower** extension) ‚Äî a copy-paste slip ‚Äî so a value whose upper bound was
`~7.0` printed as plain `7.0` unless the *lower* bound also carried `~`. The fix
tests `u_ext`. Expected-output churn lands in `contrib/seg/expected/seg.out`
(+44).

**Operational consequence:** this changes the dumped text of already-stored seg
data. The buildfarm cross-version upgrade test (`XversionUpgrade-REL_16_‚Ä¶HEAD`)
dumps from an old cluster (buggy `seg_out` ‚Üí `4.6 .. 7.0`), upgrades, and
re-dumps on HEAD (fixed ‚Üí `4.6 .. ~7.0`); the diff fails the stage. `fairywren`
hit exactly this today ‚Äî see `knowledge/buildfarm-lessons/2026-06-11.md` ¬ß1.
Touches `[[knowledge/files/contrib/seg/seg.c.md]]`.

### 5. `eb4e7224a1c6` ‚Äî Fix race with timeline selection in logical decoding during promotion (Michael Paquier / Bertrand Drouvot)

A standby promoting to primary passes through a window where
`RecoveryInProgress()` still returns `true` (shared `SharedRecoveryState` has not
yet flipped to `RECOVERY_STATE_DONE`) but the old timeline's WAL segments have
already been removed or recycled. A logical decoder reading a page in that window
picks the *old* timeline as `currTLI` and fails with
`ERROR: requested WAL segment ... has already been removed` `[from-comment]`. Not
a correctness bug ‚Äî a retry succeeds ‚Äî but it dents availability and was a source
of sporadic buildfarm failures since logical decoding on standbys (v16).

The fix is in `logical_read_xlog_page()` at
`replication/walsender.c:1106‚Äì1132` `[from-diff]`. When
`am_cascading_walsender` (i.e. `RecoveryInProgress()` is still true), instead of
unconditionally taking `GetXLogReplayRecPtr(&currTLI)`, it first consults
`GetWALInsertionTimeLineIfSet()`:

```c
insertTLI = GetWALInsertionTimeLineIfSet();
if (insertTLI != 0)
    currTLI = insertTLI;
else
    GetXLogReplayRecPtr(&currTLI);
```

`InsertTimeLineID` is published *before* the old segments are removed and before
`SharedRecoveryState` switches, and promotion copies the old timeline's last
segment into the new timeline up to the switchpoint ‚Äî so reading on the
insertion timeline is safe and skips the removed segments. The TAP test uses a
new injection point `promotion-after-wal-segment-cleanup` placed in
`StartupXLOG()` at `access/transam/xlog.c:6574` `[from-diff]`, right after
`CleanupAfterArchiveRecovery()`, to deterministically park the startup process
in the bad window. Test backpatched to v17; fix to v16. Touches
`[[knowledge/files/src/backend/replication/walsender.c.md]]` and
`[[knowledge/files/src/backend/access/transam/xlog.c.md]]`.

Relevant skills: **`replication-overview`** (walsender / timeline selection) and
**`wal-and-xlog`** (timeline switchpoint semantics). The comment block added at
the call site is itself a good corpus citation for *why* the insertion timeline
is the safe choice during the promotion window.
