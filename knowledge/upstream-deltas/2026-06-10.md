## Deep dives

### `e18b0cb734` — `MarkBufferDirtyHint()` looked up a shared descriptor for local buffers

`MarkBufferDirtyHint(Buffer buffer, bool buffer_std)` is the hint-bit dirty
path — used by setters of `HEAP_XMIN_COMMITTED`/`HEAP_XMAX_*` infomask hint
bits, the freespace map, the visibility map, and btree page-flag hints. It
deliberately distinguishes **shared** buffers (which need the header spinlock
and, when checksums/`wal_log_hints` are on, a backup-block WAL stamp) from
**local** buffers (temp-table buffers, which need neither — see
`[[knowledge/files/src/backend/storage/buffer/bufmgr.c.md]]` §"local buffers
never need the header spinlock", corpus cite `bufmgr.c:7531`).

The bug: the very first statement computed
`bufHdr = GetBufferDescriptor(buffer - 1)` **unconditionally**, before either
the `BufferIsValid()` guard or the `BufferIsLocal()` early-return. For a local
buffer, `buffer` is **negative** (local buffers use negative `Buffer` ids), so
`buffer - 1` is a negative index into the *shared* `BufferDescriptors` array —
an out-of-bounds read of a bogus `BufferDesc`. The value happened to be
discarded on the local path (which returns after `MarkLocalBufferDirty()`), so
it was latent rather than a live crash, but it is still an invalid array access
and trips under sanitizer/assert builds.

The fix reorders three lines (post-commit `bufmgr.c:5831‑5845`): the
`GetBufferDescriptor()` call moves to **after** the `BufferIsValid()` check and
the `BufferIsLocal()` early-return, so the shared descriptor array is only
indexed once we know `buffer` is a valid *shared* buffer id. Net diff is +2/-2
(one line deleted at the top, re-added lower down).

Corpus impact: the hint-bit description in
`[[knowledge/files/src/backend/storage/buffer/bufmgr.c.md]]` cites
`MarkBufferDirtyHint` at `bufmgr.c:5830`; that anchor and the surrounding
"Dirtying" line table (`bufmgr.c:3156, 5830, …`) shift by the reorder and the
"local buffers early-return" invariant gains a concrete bug-of-record. Also
relevant to `[[knowledge/wiki-distilled/Hint_Bits.md]]` and
`[[knowledge/subsystems/storage-buffer.md]]`. Flagged in
`progress/refresh-2026-06-10.md`.

### `b70d5672d0` — pg_buffercache regained executor-side return-row type checking

`pg_buffercache_pages()` returns `SETOF record`, so callers supply a column
list in the `AS p(...)` clause. Historically the C function built an explicit
`TupleDesc` matching the values it actually produces and handed it to the
executor, which then ran `tupledesc_match()` to confirm the caller's declared
column **types** matched — catching e.g. a query that declares `isdirty` as
`text` when the function emits `bool`.

A prior refactor to `InitMaterializedSRF()` dropped that published descriptor:
`InitMaterializedSRF()` populates `rsinfo->setDesc` from the *caller-supplied*
`expectedDesc`, so the function was effectively rubber-stamping whatever types
the caller declared — the mismatch check silently became a no-op. This commit
restores the verification:

- New static `build_buffercache_pages_tupledesc(int natts)`
  (post-commit `pg_buffercache_pages.c:227‑256`) builds the canonical 8- or
  9-column descriptor (`bufferid int4`, `relfilenode oid`, …, `isdirty bool`,
  `usagecount int2`, and `pinning_backends int4` only when
  `natts == NUM_BUFFERCACHE_PAGES_ELEM`), then `BlessTupleDesc()`es it.
- After `InitMaterializedSRF()`, the function switches into
  `rsinfo->econtext->ecxt_per_query_memory`, builds the real descriptor, and
  **overrides** `rsinfo->setDesc` with it (`pg_buffercache_pages.c:112‑126`).
- A pointed comment warns **not** to `pfree()` the previous `setDesc`: for
  `RECORD` results it can alias `rsinfo->expectedDesc`, which the executor
  still references — freeing it would be a use-after-free.

The new regression line (`sql/pg_buffercache.sql`, `expected/…out`) declares
`isdirty text` and now correctly gets
`ERROR: function return row and query-specified return row do not match /
DETAIL: Returned type boolean at ordinal position 7, but query expects text.`

Corpus impact: `[[knowledge/files/contrib/pg_buffercache/pg_buffercache_pages.c.md]]`
("Public API" cites `pg_buffercache_pages` at `:85`; the per-query-memory
switch and the new helper change line anchors below that point) and
`[[knowledge/issues/pg_buffercache.md]]`. This is also a reusable idiom worth a
note: **a materialized-SRF returning `record` must re-publish its own
`setDesc` if it wants executor-side type checking** — `InitMaterializedSRF()`
alone copies the caller's `expectedDesc` and checks nothing. Flagged in
`progress/refresh-2026-06-10.md`.
