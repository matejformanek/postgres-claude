## Deep dives

### `993a7aa0` — FOR PORTION OF: cross-leftover pollution in BEFORE INSERT triggers

**File:** `src/backend/executor/nodeModifyTable.c`, `ExecForPortionOfLeftovers()`
~L1601–1618 (@`993a7aa0`). Touches [[knowledge/files/src/backend/executor/nodeModifyTable.c.md]].

When `UPDATE … FOR PORTION OF` splits a temporal row, the unchanged head/tail
ranges ("leftovers") are re-inserted. The loop initialised `leftoverSlot` once
(`didInit`) and reused it for both leftovers. If a **BEFORE ROW INSERT trigger
mutates the slot** (e.g. `NEW.name := NEW.name || '+insert'`), the mutation from
the *first* leftover insert leaked into the *second*, so the two leftovers came
out inconsistent.

The fix adds an `else` arm to the `didInit` branch that **re-stores the original
tuple into `leftoverSlot` before each subsequent leftover**, via
`execute_attr_map_slot()` when a partition attribute map exists, else
`ExecForceStoreHeapTuple()`:

```c
else
{
    /* Re-copy the original row ... BEFORE ROW INSERT triggers ... can
       modify the slot contents. */
    if (map != NULL)
        execute_attr_map_slot(map->attrMap, oldtupleSlot, leftoverSlot);
    else
        ExecForceStoreHeapTuple(oldtuple, leftoverSlot, false);
}
```

Regress: new case asserts both leftovers get `foo+insert+insert` (each insert
fires the trigger once on a fresh copy). The follow-up `bd57abbb` only
re-pgindents these lines — no behavior change. Correctness bug, master-only
(temporal UPDATE is a PG 18+ feature), no on-disk/WAL impact.

### `7ac030d5` — FOR PORTION OF now requires UPDATE permission on the range column

**File:** `src/backend/parser/analyze.c`, `transformForPortionOfClause()`
~L1549, L1599–1605 (@`7ac030d5`). Touches [[knowledge/files/src/backend/parser/analyze.c.md]].

Previously a `// you don't need UPDATE permission on it` comment deliberately
**omitted** the application-time range column from `updatedCols`. This commit
reverses that decision: the range column *is* rewritten by the leftover machinery,
so the planner/executor should require UPDATE privilege on it (and column-level
`BEFORE UPDATE OF <col>` triggers should fire). It now does:

```c
RTEPermissionInfo *target_perminfo = pstate->p_target_nsitem->p_perminfo;
...
target_perminfo->updatedCols =
    bms_add_member(target_perminfo->updatedCols,
                   range_attno - FirstLowInvalidHeapAttributeNumber);
```

Two visible consequences in the regress deltas: (1) `BEFORE UPDATE OF valid_at`
row/statement triggers now fire on a FOR PORTION OF update; (2) the new
`temporal_partitioned` test adds a `GENERATED … STORED` column to exercise
`updatedCols` **attribute-number remapping across partitions with mismatched
attnums** — the riskier path, since `updatedCols` is a bitmapset keyed by attno
offset and must be mapped per-partition. Privilege/semantics change, master-only.
Note this is also a **permission-tightening**: a role with UPDATE on non-range
columns only may now be rejected.

### `ce6bf3cd` — setweight(): user-facing error for invalid weight symbols

**File:** `src/backend/utils/adt/tsvector_op.c`, new static `parse_weight()`
~L210–243 (@`ce6bf3cd`). Touches [[knowledge/files/src/backend/utils/adt/tsvector_op.c.md]].

Three copy-pasted `switch (cw)` weight decoders (in `tsvector_setweight`,
`tsvector_setweight_by_filter`, `tsvector_filter`) are unified into one
`parse_weight(char)` returning 0–3. The behavioral improvement is the `default`
arm: the old code did `elog(ERROR, "unrecognized weight: %d", cw)` — an
*internal-error* elog leaking a raw byte value for what is really bad **user
input** (`setweight(tsv, 'x')`). Now it `ereport`s `ERRCODE_INVALID_PARAMETER_VALUE`
and renders the byte safely:

```c
if (cw >= ' ' && cw < 0x7f)
    ereport(ERROR, (errcode(ERRCODE_INVALID_PARAMETER_VALUE),
                    errmsg("unrecognized weight: \"%c\"", cw)));
else  /* \ooo octal, like charout() */
    ereport(ERROR, (errcode(ERRCODE_INVALID_PARAMETER_VALUE),
                    errmsg("unrecognized weight: \"\\%03o\"", (unsigned char) cw)));
```

`tsvector_filter` also gets a neat `mask |= 1 << parse_weight(...)` since
weights 0–3 map onto bits D/C/B/A. Classic *elog→ereport for user-reachable
errors* + *non-ASCII-safe message* idiom (see `error-handling` skill). No
catalog/ABI impact; SQLSTATE for these errors changes from XX000 to 22023.

### `e88bd273` — guard USE_AVX… macros off the x86_64 path in c.h

**File:** `src/include/c.h` ~L1340–1365 (@`e88bd273`). Header — no existing
per-file corpus doc.

In macOS "universal" builds the configure step may run on an x86_64 host and
define `USE_AVX2_WITH_RUNTIME_CHECK` / `USE_AVX512_CRC32C_WITH_RUNTIME_CHECK` /
`USE_AVX512_POPCNT_WITH_RUNTIME_CHECK`, but the actual compile for the aarch64
slice must not pull in AVX code. The fix restructures the arch `#if`: on the
non-x86_64 branch it `#undef`s the three AVX macros, then independently tests
`__aarch64__ && __ARM_NEON` for `USE_NEON`. Build-portability only; relevant to
anyone touching SIMD dispatch (`pg_popcount`, CRC32C). No runtime behavior change
on single-arch builds.

### `90354030` — libpq: report the actual servicefile after default-file fallback

**File:** `src/interfaces/libpq/fe-connect.c`, `parseServiceInfo()` ~L6050–6062
(@`90354030`). Touches [[knowledge/files/src/interfaces/libpq/fe-connect.c.md]].

When a `service=` lookup is satisfied by the **system** `pg_service.conf`
(fallback, no explicit `servicefile=` / `PGSERVICEFILE`), the resolved
`servicefile` connection option was left empty. Now, once `group_found`, it
back-fills the option via `conninfo_storeval(options, "servicefile", serviceFile,
…)`, returning status 3 on the only possible failure (OOM, message already set).
TAP `006_service.pl` adds a check that `\echo :SERVICEFILE` reflects the default
file. UX/introspection fix; back-patchable in spirit (libpq, no ABI change).
