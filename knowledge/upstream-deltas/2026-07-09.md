### `86ab7f4c721d` — JSON_TABLE PLAN clause  (Amit Langote) ★ feature

Implements the SQL-standard `PLAN` / `PLAN DEFAULT` clause of `JSON_TABLE`,
which lets a query author name how nested column paths are joined (`INNER` /
`OUTER`, `UNION` / `CROSS`) instead of relying on the implicit default.

Files: `parser/gram.y` (new productions `json_table_plan_*`,
`json_table_default_plan_*`), `parser/parse_jsontable.c`
(`makeJsonTablePathScan`, `makeJsonTableSiblingJoin`,
`validateJsonTableChildPlan`), `utils/adt/jsonpath_exec.c` (executor:
`JsonTablePlanState` gains `cross` / `outerJoin` control fields +
`JsonTableRescan`; sibling joins dispatch on `IsA(plan, JsonTableSiblingJoin)`),
`nodes/makefuncs.c` + `makefuncs.h` (`makeJsonTableDefaultPlan`),
`nodes/parsenodes.h` + `primnodes.h` (new `JsonTablePlanSpec`; sibling-join
node grows `bool cross`, path scan grows `bool outerJoin`), `ruleutils.c`
(deparse of the PLAN clause).

- The PLAN body is a tree of `simple` / `outer` / `inner` / `union` / `cross`
  productions built in the grammar and validated in `parse_jsontable.c`:
  `PLAN(...)` can only reference **named** paths; an unnamed path the plan must
  mention is caught later as a "path not covered by the plan" / name-mismatch
  error. `PLAN DEFAULT (...)` selects among `INNER|OUTER` and `UNION|CROSS`
  defaults via `makeJsonTableDefaultPlan`.
- Executor: nested-column joins now honour `planstate->cross` (advance the
  right child, then step the outer row) vs union/outer semantics inside
  `JsonTablePlanNextRow`; `JsonTableRescan` resets a sub-plan.
- Companion catversion bump is `21e958c303bc` (202607072 → **202607091**) —
  new parse-node shapes are stored, so the bump is required.

### `e01b23b84e43` — Add an `enable_groupagg` GUC  (David Rowley) ★ planner

New `PGC_USERSET` / `QUERY_TUNING_METHOD` / `GUC_EXPLAIN` boolean (boot_val
`true`) that disables **sort-based** grouping/aggregation the way
`enable_hashagg` already disables hashed aggregation. It fills a long-standing
gap: previously there was no switch to steer the planner *away* from GroupAgg.

- Defined in `utils/misc/guc_parameters.dat` (the newer `.dat`-driven GUC
  registration; not a hand-written `guc_tables.c` entry) + `optimizer/cost.h`
  extern + `costsize.c:151` global.
- `cost_agg()` reworked so each strategy is charged a `disabled_nodes`
  increment against the *right* switch: `AGG_PLAIN` (incl. the empty grouping
  set, `numGroupCols == 0`) is never disabled; `AGG_SORTED` with grouping
  columns → `enable_groupagg`; `AGG_HASHED` → `enable_hashagg`; `AGG_MIXED`
  hashes ≥1 rollup so it keys off `enable_hashagg` (its sorted rollups come
  back through `cost_agg` separately as `AGG_SORTED`). `cost_group()` adds
  `+ (enable_groupagg ? 0 : 1)`.
- Also disables the **sort-based dedup** counterparts, since they are the
  natural mirror of the hashed paths `enable_hashagg` governs:
  `create_unique_path()` and `create_setop_path()` (`SETOP_SORTED`) bump
  `disabled_nodes` when the switch is off (`pathnode.c`).

### `dff789e55540` — Fix advertising autovacuum launcher's ProcNumber  (Andres Freund)

The av-launcher publishes its `ProcNumber` into `ProcGlobal->avLauncherProc`,
but it was being set/cleared in `InitAuxiliaryProcess` / `AuxiliaryProcKill` —
and the av-launcher is **not** an aux process, it's a regular backend. So the
advertisement never actually happened.

- Fix moves the `pg_atomic_write_u32(&ProcGlobal->avLauncherProc, ...)` into
  `InitProcess` (set) and `ProcKill` (clear + assert), guarded by
  `MyBackendType == B_AUTOVAC_LAUNCHER` (`storage/lmgr/proc.c`).
- Comment on the `PROC_HDR.avLauncherProc` field updated: "avlauncher is not
  an aux process, but it is advertised the same way" (`include/storage/proc.h`).
- Drive-by in `postmaster/checkpointer.c`: drops the `volatile PROC_HDR
  *procglobal = ProcGlobal;` local and reads `&ProcGlobal->checkpointerProc`
  directly (same de-volatile cleanup class as `b34fd845` last week).

### `881033ae8b55` — Don't create SPLIT/MERGE partitions as internal relations  (Álvaro Herrera)

`ALTER TABLE ... SPLIT/MERGE PARTITION` was creating the new partition(s) with
the `is_internal` flag, which suppresses dependency records and DDL-visibility
that a *user-created* table should have. Fix flips several `is_internal`
arguments from `true` → `false` in `commands/tablecmds.c`:

- `createPartitionTable()` passes `is_internal = false`.
- `createTableConstraints()`: `StoreAttrDefault(..., false)`,
  `AddRelationNewConstraints(..., false, ...)` (defaults + not-null).
- `ATExecMergePartitions` / `ATExecSplitPartition`: `performDeletionCheck` of
  the source partition drops the `PERFORM_DELETION_INTERNAL` flag (→ `0`), so
  the drop is treated as a normal user drop.

### `8f7af125e037` — Remove WaitEventCustomCounterData  (Michael Paquier)

Simplifies the custom wait-event ID allocator. The old
`WaitEventCustomCounterData` struct paired a `nextId` counter with a `slock_t
mutex`; the counter is only ever bumped once per custom wait-event *definition*
(rare, startup-ish), so the spinlock was overkill.

- Struct + its `#include "storage/spin.h"` removed; the shared allocator is now
  a bare `static int *WaitEventCustomCounter` (`utils/activity/wait_event.c`).
- `ShmemRequestStruct` shrinks from `sizeof(WaitEventCustomCounterData)` to
  `sizeof(int)`; `WaitEventCustomNew` drops the `SpinLockAcquire/Release`
  around the `>= WAIT_EVENT_CUSTOM_HASH_SIZE` check and `nextId++`.
- `typedefs.list` entry removed.
