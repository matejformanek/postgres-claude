## Deep explainers

### 2 · `ed62d26cacac` — Fix VM clear WAL logging by registering VM blocks — **headline data-corruption fix**

The night's most consequential commit, and a genuine data-corruption bug in the
incremental-backup path. Heap WAL records that *clear* visibility-map bits
(emitted by `heap_insert`, `heap_multi_insert`, `heap_delete`, `heap_update`,
`heap_lock`) did **not register** the VM blocks they modified with the WAL
machinery. The WAL summarizer (`WAL_SUMMARIZER`, the engine behind PG 17+
incremental backup) only records *registered* blocks — so an incremental backup
taken across such operations **omitted the changed VM pages entirely**. On
restore, `pg_combinebackup` reconstructed a VM still carrying stale
`all-visible`/`all-frozen` bits. Two concrete failure modes:

- **Wrong query results**: an index-only scan trusting a stale `all-visible` bit
  skips the heap-fetch visibility recheck and returns rows it shouldn't.
- **Incorrect `relfrozenxid` advancement**: VACUUM's page-skipping logic trusts
  the stale `all-frozen` bit, skips a page that actually needs freezing, and
  advances `relfrozenxid` past unfrozen tuples — a wraparound-safety hazard.

Secondary effect: because the VM buffer was never registered, PG never emitted
**FPIs (full-page images)** of VM pages when clearing bits. A torn VM page won't
raise an error (VM is read `ZERO_ON_ERROR`); with checksums on it's detected and
zeroed, but without checksums a torn write silently corrupts.

The fix registers the VM buffer(s) alongside the heap block in each of these WAL
records. Touches `src/backend/access/heap/heapam.c` (all five record-emitting
paths — `heap_insert` ~L2055, `heap_multi_insert` ~L2421, `heap_delete` ~L3026,
`heap_update` ~L3879/L4100, `heap_lock` ~L5316), `heapam_xlog.c` (replay side
reads the VM blocks back), `visibilitymap.c`, `pruneheap.c`, and
`contrib/pg_surgery/heap_surgery.c`. Regression coverage in
`src/bin/pg_walsummary/t/002_blocks.pl` + new `pg_combinebackup/t/012_vm_consistency.pl`
(commit 1). `[[knowledge/subsystems/access-heap]]`, `[[knowledge/subsystems/wal-and-xlog]]`,
`[[knowledge/idioms/hint-bits-setbufferdirty]]`, backup/incremental corpus.

### 3 · `2340fb8f02b2` — VM clear takes a RelFileLocator, drops the "fake relcache"

Preparatory refactor for commit 2. `visibilitymap_clear()`-family previously
required a `Relation` object just to pin the VM buffer, so callers with only a
`RelFileLocator` (e.g. replay) built a *fake relcache entry*. This aligns the
clear path with `visibilitymap_set()` by taking a `RelFileLocator` directly; an
upcoming change reads VM buffers via `XLogReadBuffer*()` which needs no relation.
Also drops a spurious `const` on `visibilitymap_set()`'s `RelFileLocator` param.
`src/backend/access/heap/visibilitymap.c`, `src/include/access/visibilitymap.h`,
+4 callers. `[[knowledge/subsystems/access-heap]]`.

### 4 · `10911fde088a` — Named macros for heap WAL block-reference IDs

Also preparatory. WAL block-ref IDs were bare integers (0,1,2…) assigned by
convention — fine when a record registers one block or replays in registration
order. Commit 2 registers up to **two VM blocks in addition** to the heap
block(s), replayed *out of registration order*, so bare integers become
error-prone. Introduces `HEAP_INSERT_BLKREF_HEAP`, `HEAP_MULTI_INSERT_BLKREF_HEAP`,
`HEAP_MULTI_INSERT_BLKREF_VM`, `HEAP_UPDATE_BLKREF_HEAP_NEW/_OLD`,
`HEAP_DELETE_BLKREF_HEAP`, `HEAP_LOCK_BLKREF_HEAP` in `heapam_xlog.h`.
`[[knowledge/subsystems/wal-and-xlog]]`, `knowledge/files/src/include/access/heapam_xlog.h.md`.

### 5 · `a2fd8d65a0b7` — FSM vacuum of bulk-extended relation includes the last block

One-line off-by-one fix (`hio.c:401`). When bulk-extending a relation,
`RelationAddBlocks()` adds the not-immediately-used new blocks to the FSM then
calls `FreeSpaceMapVacuumRange()` to propagate free space up the tree. But that
function's **end block is exclusive**, and the code passed `last_block` (since
`00d1e02be24`). If that block happened to be the first covered by a *new* FSM
page, its free space was never propagated up and stayed invisible to FSM
searches until the next FSM vacuum. Fix: pass `last_block + 1`.
`[[knowledge/subsystems/free-space-map]]`, `knowledge/files/src/backend/access/heap/hio.c.md`.

### 8 · `ee654419d5fe` — blkreftable read-side bounds checks (hardening)

`src/common/blkreftable.c:657,687`. Code downstream assumes fork numbers and
chunk sizes read from a block-reference-table file are in range; the reader
didn't validate them. A corrupted file could index off the end of the fork-number
or chunk-entry arrays → potential crash. Adds `sentry.forknum < 0 ||
> MAX_FORKNUM` and per-chunk `chunk_size[i] > MAX_ENTRIES_PER_CHUNK` guards that
route to the reader's `error_callback`. Relevant to `pg_combinebackup` /
incremental-backup robustness (see also commit 2's summarizer angle).
`knowledge/files/src/common/blkreftable.c.md`.

### 10 · `f38afa4abb04` — Reject concurrent sequence refreshes (logical-repl race)

Real concurrency bug in sequence synchronization (the PG 19 sequence-replication
feature). `ALTER SUBSCRIPTION ... REFRESH SEQUENCES` could race a running
sequence-sync worker: if a second refresh reset the sync state while the worker
had already fetched values from the publisher but not yet applied them, the
worker could overwrite the subscriber with **stale values** and mark sync
complete. Fix rejects the command with an error (rerun after sync completes) when
a sync worker is already running. `src/backend/commands/subscriptioncmds.c`,
test in `src/test/subscription/t/036_sequences.pl`.
`[[knowledge/subsystems/logical-replication]]`,
`knowledge/files/src/backend/commands/subscriptioncmds.c.md`.

### 16 · `8c551aab156d` — equalPolicy() checks polpermissive (RLS relcache)

Two-line consistency fix in `relcache.c:980`. `equalPolicy()` (relcache's
policy-equivalence check) never compared `polpermissive`. Today `ALTER POLICY`

### 17 · `d68576bcfc72` — Strip removed-relation PHVs from join clauses (planner)

Follow-up to `9a60f295b`, which stripped stale `PlaceHolderVar`s left by
left-join removal from surviving rels' `baserestrictinfo` and EquivalenceClass
members but **overlooked join clauses**. A PHV embedded in a join clause can
retain the removed rel/join in its `phrels`, because `remove_rel_from_query()`
fixes the `RestrictInfo`'s own relid sets but not the PHVs inside the expression.
Normally harmless (later processing consults the relid sets), but a restriction
clause *derived from* such an OR join clause inherits the stale PHV, and when
translated for an appendrel child it can misbehave. Fix extends the stripping to
join clauses. `src/backend/optimizer/plan/analyzejoins.c` (~L459,L679,L865),
regression rows in `join.sql`/`join.out`.
`[[knowledge/subsystems/executor-and-planner]]`,
`knowledge/files/src/backend/optimizer/plan/analyzejoins.c.md`.

### 19 · `e53544593835` — postgres_fdw: don't push down non-relabeling ArrayCoerceExpr

Wrong-results/remote-error fix. `62c3b4cd9ddc` taught postgres_fdw to push down
`ArrayCoerceExpr`, but `foreign_expr_walker()` (`deparse.c:707`) recursed only
into the input array and **never examined `elemexpr`** — the per-element
conversion that gives the coercion its meaning. `deparseArrayCoerceExpr()` then
shipped a bare `arg::resulttype` cast (or nothing for an implicit-format
coercion), leaving the *remote* server to re-resolve the element conversion
against its own catalogs/session — producing wrong results or errors whenever the
element conversion wasn't a plain relabel. Fix: only push down when `elemexpr` is
a plain relabeling, matching how scalar coercions are treated.
`[[knowledge/subsystems/fdw-development]]`,
`knowledge/files/contrib/postgres_fdw/deparse.c.md`.

### 20–25 · JSON_TABLE `PLAN`-clause cleanup cluster (Korotkov) — follow-ups to `86ab7f4c721d`

Six commits polishing the JSON_TABLE `PLAN` clause landed 07-09:

- **21 `6d8fd5a8` (perf, real)** — `86ab7f4c721d` accidentally made a `NESTED
  PATH` re-run its jsonpath expression several times per parent row (badly slow
  as nested arrays grow, even without a `PLAN` clause). Two leftovers from the
  plan/join executor rework: `JsonTablePlanScanNextRow()` still reset+advanced
  the nested plan (now `JsonTablePlanNextRow()`'s job), and
  `JsonTableResetNestedPlan()` eagerly called `JsonTableResetRowPattern()` (which
  evaluates the path) on top of setting the reset flag. Net `-26` lines in
  `jsonpath_exec.c` (~L4698–4787).
- **23 `cd5fd08b` (correctness)** — `generateJsonTablePathName()` produced
  `json_table_path_N` names in the *same namespace* as user names without a
  collision check; a generated name colliding with a user path referenced by a
  `PLAN` clause silently dropped a sibling path + its columns from output. Now
  bumps the counter until unused. `parse_jsontable.c`.
- **24 `160ef275` (dump/restore)** — `get_json_table_plan()` deparse parenthesized
  a parent/child plan's child only when it was a sibling (UNION/CROSS) join, so
  `PLAN (p0 OUTER (p1 INNER p11))` round-tripped as the *non-equivalent*
  `PLAN (p0 OUTER p1 INNER p11)` — a dump/restore hazard. Now parenthesizes any
  non-bare-path child. `ruleutils.c`.
- **25 `af6fad87` (revert)** — reverts the part of `86ab7f4c721d` that made a
  table-level `ON ERROR` cascade as the per-column default (so top-level
  `ERROR ON ERROR` turned per-column errors hard). The SQL standard *does* mandate
  the cascade, but it should be a deliberate, separately-documented change; also
  reverts the paired `ruleutils.c` deparse that dropped an explicit `ERROR ON
  EMPTY` from dumped views. `parse_jsontable.c` + `ruleutils.c`.
- **20 `9ee5241d` (dead-code)** — replaces an unreachable "PATH name not found in
  nested columns list" error in `transformJsonTableNestedColumns()` with an
  `Assert` (validateJsonTableChildPlan() already guarantees the invariant).
- **22 `4a82912b` (docs)** — synopsis fix: `ON ERROR` after `PLAN`, `path_expression`
  not `json_path_specification`, `<literal>` markup, OUTER-before-INNER.

`[[knowledge/subsystems/jsonpath-and-jsonb]]`, `[[knowledge/subsystems/parser-and-nodes]]`,

### 26 · `d15a6bc2e16d` — Replace `__builtin_types_compatible_p` with `_Generic`
