## Deep dives

### 1. `93a3e6839bf8` — Fix race when logical decoding activation is concurrently interrupted (Masahiko Sawada)

This is a concurrency-correctness fix in the **PG19 on-demand logical decoding /
`effective_wal_level`** machinery (`logicalctl.c`). That feature lets a cluster
run at `wal_level = replica` and *promote* WAL logging to logical detail
(`xlog_logical_info = true`) only once a logical slot actually exists, tracked in
shared `LogicalDecodingCtl` under `LogicalDecodingControlLock`. Two flags matter:
`logical_decoding_enabled` (the durable state) and `xlog_logical_info` (the
"emit logical info to WAL" knob that every backend caches). `[from-diff]`

**The race.** `EnableLogicalDecoding()` runs under a
`PG_ENSURE_ERROR_CLEANUP` whose abort callback,
`abort_logical_decoding_activation()`, used to *directly* revert
`xlog_logical_info = false` and broadcast
`PROCSIGNAL_BARRIER_UPDATE_XLOG_LOGICAL_INFO`. If backend A's activation is
interrupted (e.g. statement cancel) at the same moment backend B legitimately
finishes activating a *different* logical slot, A's abort callback would tear
down the `xlog_logical_info` that B's valid slot still needs. The old callback
even asserted `!LogicalDecodingCtl->logical_decoding_enabled`, which a concurrent
B violates.

**The fix has three coupled parts:**

- **Abort defers to the checkpointer.** `abort_logical_decoding_activation()` now
  drops all the direct flag manipulation and asserts, and simply calls
  `RequestDisableLogicalDecoding()`. The checkpointer's disable path only resets
  `xlog_logical_info` when **no valid logical slots exist**
  (`CheckLogicalSlotExists()`), so B's slot is respected. `[from-diff]`
- **Wider lock scope in `EnableLogicalDecoding()`.** `LWLockAcquire(
  LogicalDecodingControlLock, LW_EXCLUSIVE)` moves to **before**
  `START_CRIT_SECTION()` (was inside it), and the matching release moves to
  **after** `END_CRIT_SECTION()`. The whole flag flip
  (`logical_decoding_enabled = true`, `pending_disable = false`, WAL-logging the
  enable) now happens entirely under the lock, and the `LOG` "logical decoding is
  enabled…" message is emitted *after* the lock release — the new comment notes
  that because activation holds a logical slot, a concurrent deactivation cannot
  interleave its log line ahead. `[from-diff]`
- **`DisableLogicalDecoding()` handles the half-activated state.** New local
  `was_enabled` captures `logical_decoding_enabled` up front. The early-out
  predicate changes from `!logical_decoding_enabled || (!in_recovery &&
  CheckLogicalSlotExists())` to `(!logical_decoding_enabled &&
  !xlog_logical_info) || …` — so a disable still runs when an interrupted
  activation left `xlog_logical_info = true` while `logical_decoding_enabled`
  stayed `false`. The WAL "disable" record and the `LOG` line are now gated on
  `was_enabled`, suppressing a spurious disable record/message when only the
  `xlog_logical_info` knob needed clearing. `[from-diff]`

The TAP test `051_effective_wal_level.pl` gains a concurrent-activation race
scenario using the `logical-decoding-activation` injection point: one backend
parks mid-activation, a second concurrently creates `test_slot2` and reaches
`replica|logical`, the first is cancelled, and the test asserts
`effective_wal_level` *stays* `logical` (the concurrent slot survives the abort).

Corpus impact: refresh
[[knowledge/files/src/backend/replication/logical/logicalctl.c.md]] — the
abort-callback contract, the `EnableLogicalDecoding`/`DisableLogicalDecoding`
lock ordering vs. the critical section, and the new `was_enabled` gating are all
load-bearing. Lock-ordering note for any `locking`/replication synthesis: the
`LogicalDecodingControlLock` now spans the full enable critical section rather
than nesting inside it. See `source/src/backend/replication/logical/logicalctl.c`
`abort_logical_decoding_activation`, `EnableLogicalDecoding`,
`DisableLogicalDecoding`.

### 2. `cd7b204b2df9` — Disallow direct use of the pgrepack logical decoding plugin (Álvaro Herrera)

`REPACK (CONCURRENTLY)` (the PG19 successor work to `pg_repack`/CLUSTER) captures
concurrent row changes during a non-blocking repack by spinning up an **internal
logical decoding output plugin named `pgrepack`** over a temporary slot. The
problem: that plugin was registered like any other output plugin, so any user
could do `pg_create_logical_replication_slot('s', 'pgrepack')` and drive it
through the SQL decoding interface — an unsupported path whose private state
(`RepackDecodingState`) the SQL callers never initialize correctly. `[from-diff]`

**The guard.** `repack_startup()` (the plugin's `LogicalOutputPluginInit`
startup callback, in `replication/pgrepack/pgrepack.c`) now rejects any caller
that is not the repack worker:

```c
if (!AmRepackWorker())
    ereport(ERROR,
            errcode(ERRCODE_FEATURE_NOT_SUPPORTED),
            errmsg("unsupported use of logical decoding plugin \"%s\"", "pgrepack"),
            errdetail("This plugin can only be used by %s.", "REPACK (CONCURRENTLY)"));
```

The `test_decoding` `repack` test now asserts a direct
`pg_create_logical_replication_slot('s_repack','pgrepack')` fails in the startup
callback ("in the startup callback") and that the slot is never created.

**State-ownership refactor (the rest of the diff).** Initialization of
`RepackDecodingState` moves *into* the plugin:

- `repack_startup()` now `palloc0_object(RepackDecodingState)` and creates the
  per-change `change_cxt` (`AllocSetContextCreate` in `ctx->context`), storing it
  in `ctx->output_writer_private`. `repack_setup_logical_decoding()`
  (`commands/repack_worker.c`) then fills in `relid`, `worker_cxt`,
  `worker_resowner` after `CreateInitDecodingContext()`. Previously the worker
  allocated `dstate` itself *after* context creation. `[from-diff]`
- `RepackDecodingState.relid` is now **unconditional** in
  `include/commands/repack_internal.h` (was `#ifdef USE_ASSERT_CHECKING`),
  because it's used to skip decoding of unrelated relations, not merely asserted.
- Cosmetic but worth noting for future grep: the slot-name prefix changes
  `repack_%d` → `pg_repack_%d`, and macro `REPL_PLUGIN_NAME` → `PGREPACK_PLUGIN`;
  the local slot-name buffer goes from `NameData` to `char[NAMEDATALEN]`. The
  setup is reordered so `DecodingContextFindStartpoint()` runs *after* the
  `dstate` fields are populated, then the page-read callback is swapped to the
  non-blocking `read_local_xlog_page_no_wait`.

Security/hardening angle (relevant to the Phase D thread in `STATE.md`): this
closes a path where an internal, assumption-laden decoding plugin was reachable
from unprivileged SQL — exactly the "internal machinery exposed via the generic
plugin registry" shape worth flagging.

Corpus impact: **no existing per-file docs** for `repack_worker.c`,
`pgrepack.c`, or `repack_internal.h` (REPACK is newer than the last deep-corpus
pass). Candidate backfill targets — see `progress/refresh-2026-06-09.md`. Note
for [[knowledge/subsystems/replication.md]]: output plugins that are
"internal-only" should gate their startup callback on a worker-identity check
(`AmRepackWorker()` here) rather than trusting the slot/plugin registry.

### 3. `9d8cdcbe0c8a` — Record dependencies on graph labels and properties (Peter Eisentraut)

A missing-dependency bug in the **PG19 SQL/PGQ property-graph (`GRAPH_TABLE`)**
feature. `find_expr_references_walker()` in `catalog/dependency.c` walks an
expression tree to emit `pg_depend` rows for every catalog object an expression
references. Two property-graph node types were not handled, so a view (or other
stored expression) that referenced a graph label or property recorded **no
dependency** on the underlying `pg_propgraph_label` / `pg_propgraph_property`
catalog row. Consequence: `ALTER PROPERTY GRAPH … DROP LABEL` /
`… ALTER LABEL … DROP PROPERTIES (…)` would succeed even while a view depended on
them, and the next `pg_get_viewdef()` blew up with a cache-lookup error.
`[from-diff]`

**The fix** adds two `IsA` arms to the walker:

```c
else if (IsA(node, GraphLabelRef))
    add_object_address(PropgraphLabelRelationId, glr->labelid, 0, context->addrs);
else if (IsA(node, GraphPropertyRef))
    add_object_address(PropgraphPropertyRelationId, gpr->propid, 0, context->addrs);
```

so dropping an in-use label/property is now blocked with the standard
"cannot drop … because other objects depend on it / view … depends on …" error
(asserted by the new `graph_table` regress cases). The catversion bump
`202606051 → 202606091` in `include/catalog/catversion.h` accompanies the change
(new dependency semantics for these objects ⇒ forced initdb).

Corpus impact: refresh
[[knowledge/files/src/backend/catalog/dependency.c.md]] —
`find_expr_references_walker` now knows about `GraphLabelRef` /
`GraphPropertyRef` and the `Propgraph{Label,Property}RelationId` catalogs; and
[[knowledge/files/src/include/catalog/catversion.h.md]] (constant value). See
`source/src/backend/catalog/dependency.c` `find_expr_references_walker`.

### 4. `bd4bd30ce6a7` — Use correct type for catalog_xmin (Fujii Masao)

One-line type-correctness fix in slot synchronization. In
`update_local_synced_slot()` (`replication/logical/slotsync.c`) the local that
snapshots the pre-advance value of `slot->data.catalog_xmin` was declared
`XLogRecPtr old_catalog_xmin` — but `catalog_xmin` is a `TransactionId`. The
declaration is corrected to `TransactionId old_catalog_xmin`. `[from-diff]`

`XLogRecPtr` is 64-bit and `TransactionId` is 32-bit, so the old code silently
widened the xid into an LSN-typed slot; functionally benign in the current use
but a latent trap (and wrong for anyone reading the type to reason about wraparound
semantics). Corpus impact: minor — note on
[[knowledge/files/src/backend/replication/logical/slotsync.c.md]] that
`old_catalog_xmin` in `update_local_synced_slot` is a `TransactionId`. See
`source/src/backend/replication/logical/slotsync.c` `update_local_synced_slot`.
