## Deep explainers

### ★ `a5918fddf10d` — logical replication conflicts logged to a table [catalog · replication · flagship]

The window's headline feature. Until now, apply-worker conflicts (insert_exists,
update_differ, delete_missing, …) were only emitted as plain text to the server
log via `conflict.c`. This commit adds a `conflict_log_destination` subscription
option (`log` | `table` | `all`) to `CREATE`/`ALTER SUBSCRIPTION`.

Key mechanics from the diff `[from-upstream-commit a5918fddf10d]`:

- **New system namespace `pg_conflict`**, OID `8988`, symbol
  `PG_CONFLICT_NAMESPACE` in `src/include/catalog/pg_namespace.dat`. When table
  logging is enabled, an internal relation `pg_conflict_log_<subid>` is created
  in this schema automatically. The namespace is *system-managed*: new
  predicates `IsConflictLogTableNamespace(Oid)` / `IsConflictLogTableClass(Form_pg_class)`
  in `src/backend/catalog/catalog.c` (+ decls in `catalog.h`) gate it out of
  direct DML/DDL, mirroring the `IsToastNamespace` pattern. Guards were threaded
  into `lockcmds.c`, `policy.c`, `statscmds.c`, `tablecmds.c`, `trigger.c`,
  `rewriteDefine.c`, `execMain.c`, and `aclchk.c`.
- **Two new `pg_subscription` columns**: `subconflictlogdest` (the enum) and
  `subconflictlogrelid` (OID of the per-sub log table, 0 when none). Read into
  `Subscription` in `pg_subscription.c` (`sub->conflictlogrelid = subform->subconflictlogrelid`);
  populated/updated in `subscriptioncmds.c`.
- **New option plumbing** in `commands/subscriptioncmds.c`: bitmask
  `SUBOPT_CONFLICT_LOG_DEST 0x00080000`, an `ConflictLogDest conflictlogdest`
  field on the local opts struct, default `CONFLICT_LOG_DEST_LOG`, parsed via
  `GetConflictLogDest(val)`. `drop_sub_conflict_log_table()` tears the table
  down; an internal dependency ties the table to the subscription so
  `DROP SUBSCRIPTION` removes it.
- **Conflict payload**: local + remote tuples stored as JSON columns (schema-
  agnostic), plus local/remote xid, LSN, commit timestamp, and conflict type —
  a full post-mortem record. Emission lives in `replication/logical/conflict.c`
  (+ `include/replication/conflict.h`).
- Tests: `src/test/regress/{sql,expected}/subscription.*` and TAP
  `src/test/subscription/t/035_conflicts.pl`. `initdb.c` creates the namespace.

Corpus impact: touches `catalog/catalog.c`, `catalog/pg_subscription.c`,
`catalog/pg_publication.c`, `catalog/aclchk.c`, `commands/subscriptioncmds.c`,
`commands/tablecmds.c`, `commands/trigger.c`, `commands/lockcmds.c`,
`commands/policy.c`, `commands/statscmds.c`, `executor/execMain.c`,
`rewrite/rewriteDefine.c`, `replication/logical/conflict.c`, `bin/initdb/initdb.c`
— all corpus-tracked (see refresh report). `[[knowledge/subsystems/replication]]`,
`[[knowledge/subsystems/catalog]]`. A follow-up pgindent commit `fdad19e1cfe4`
touched the same tree.

### ★ `51cd5d6f0523` — reuse `ssup_datum_*_cmp` for int2/oid/oid8 [sort · nbtree]

The hand-written `btint2fastcmp` / `btoidfastcmp` / `btoid8fastcmp` sortsupport
comparators in `access/nbtree/nbtcompare.c` are functionally identical to the
generic `ssup_datum_int32_cmp` (int2) and `ssup_datum_unsigned_cmp` (oid, oid8)
introduced by commit `697492434`. This commit deletes the bespoke comparators
and points `ssup->comparator` at the generic ones
`[from-upstream-commit 51cd5d6f0523]`. Bonus: these types now become eligible
for **radix sort**, which keys off the generic comparators.
`[[knowledge/subsystems/sort]]`, `[[knowledge/subsystems/access-nbtree]]`.

### ★ `7b12ae729e6a` — jsonpath `.decimal()` honors silent mode [jsonpath · bugfix, backpatch 17]

`jsonpath_exec.c`'s `.decimal(precision[, scale])` built its numeric typmod by
calling `numerictypmodin()` through `DirectFunctionCall1()`, which throws a hard
`ereport(ERROR)` on bad precision/scale — breaking jsonpath *silent mode*, which
must never fail. Fix refactors `numeric.c` to add a soft-error-capable
`make_numeric_typmod_safe()` taking an `ErrorSaveContext *`; `numerictypmodin()`
now wraps it with a NULL context (unchanged behavior), and jsonpath passes its
own escontext when in silent mode. Nice side effect: the helper takes scalar
precision/scale directly instead of the array `numerictypmodin` expected.
Oversight in `66ea94e8e606`. Backpatch-through 17.
`[[knowledge/idioms/soft-error-reporting]]`, `[[knowledge/subsystems/jsonpath]]`.

### ★ `3eca140531f1` — precision fix in `pg_stat_us_to_ms()` [pgstat · bugfix]

`pg_stat_us_to_ms()` multiplied a µs count by the constant `0.001` to get ms.
`0.001` has no exact binary float representation, so with default
`extra_float_digits` displayed values grew trailing-digit noise
(`0.009000000000000001` instead of `0.009`). Fix: divide by `1000.0` instead,
matching the idiom used elsewhere in the tree
`[from-upstream-commit 3eca140531f1]`. `[[knowledge/subsystems/pgstat]]`.

### ★ `53e6f51eef55` — pg_plan_advice: no FOREIGN_JOIN advice for single relation [contrib · optimizer]

A foreign scan can reach the `fs_relids` branch of `pgpa_build_scan()` while
targeting a *single* relation — e.g. when postgres_fdw pushes an aggregate down
over one foreign table. In that case emitting FOREIGN_JOIN advice is wrong, so
the branch now bails out for `bms_membership(fs_relids) != BMS_MULTIPLE`
`[from-upstream-commit 53e6f51eef55]`. Touches `contrib/pg_plan_advice/pgpa_scan.c`
(corpus-tracked). `[[knowledge/subsystems/optimizer]]`.
