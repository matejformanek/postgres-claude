## Deep dives

### 1. `b85f9c00fb88` — `StandbyAcquireAccessExclusiveLock()` OOM resilience (Michael Paquier)

**The data structure.** During Hot Standby, the startup process replays
`AccessExclusiveLock` acquisitions made on the primary. Each is tracked in two
shmem-free local structures: `RecoveryLockXidHash` (keyed by xid → an
`xidentry` carrying a singly-linked list `head` of lock entries) and
`RecoveryLockHash` (keyed by `(xid, dbOid, relOid)` → a `lockentry` whose
`next` threads the per-xid list). `StandbyReleaseXidEntryLocks()` walks
`xidentry->head` to release everything at commit/abort. [[corpus]]
`knowledge/files/src/backend/storage/ipc/standby.c.md` documents this
RM_STANDBY_ID replay machinery.

**The bug.** In the `!found` (brand-new lock) branch, the pre-fix order was:

1. link the new `lockentry` into `xidentry->head`;
2. `SET_LOCKTAG_RELATION(...)` + `(void) LockAcquire(&locktag, AccessExclusiveLock, true, false)`.

`LockAcquire()` can `ereport(ERROR)` on out-of-memory while growing the lock
table's hash. If it does, the `lockentry` is *already* threaded onto the XID's
list, but the heavyweight lock was never actually granted. A later
`StandbyReleaseXidEntryLocks()` would then walk the list and call
`LockRelease()` on a lock that was never held — an inconsistency between the
recovery-lock bookkeeping and the real lock-manager state.

**The fix.** Swap the order: acquire the lock first, link into the list only
after `LockAcquire()` returns. Post-image (`standby.c:1019-1031 @ b85f9c00`):

```c
if (!found)
{
    /* First, acquire the lock ... */
    SET_LOCKTAG_RELATION(locktag, dbOid, relOid);
    (void) LockAcquire(&locktag, AccessExclusiveLock, true, false);

    /* ... and then, as it is new, link it into the XID's list. */
    lockentry->next = xidentry->head;
    xidentry->head = lockentry;
}
```

Now an OOM inside `LockAcquire()` leaves the freshly-`HASH_ENTER`ed `lockentry`
present in `RecoveryLockHash` but unlinked from any XID list — harmless (no bogus
release), and a retry finds it via `HASH_ENTER`'s `found=true`. The locking
discipline at play is the heavyweight `LockAcquire`/`LockRelease` pairing on
`LOCKTAG_RELATION` — see the `locking` skill's heavyweight-lock layer. This is a
member of the same "make recovery/startup paths survive OOM mid-mutation" series
as `b85f9c00`'s sibling `73dab127` below and the 2026-06-18 `GetSnapshotData`
OOM-resilience commit `29fb598b`.

### 2. `73dab12719ee` — type cache init resilience on OOM re-entry (Michael Paquier)

**The structures.** `lookup_type_cache()` lazily first-initializes three
backend-lifetime objects: `TypeCacheHash` (the main oid→`TypeCacheEntry` table),
`RelIdToTypeIdCacheHash` (relid→composite-typid back-map), and `in_progress_list`
(a small array tracking entries being filled, used to break recursion).
[[corpus]] `knowledge/files/src/backend/utils/cache/typcache.c.md` documents the
"entries are immortal" invariant and the `lookup_type_cache` (`:389`) entry.

**The bug.** Pre-fix, the entire first-time init lived under one guard,
`if (TypeCacheHash == NULL)`, which (a) created `TypeCacheHash`, (b) created
`RelIdToTypeIdCacheHash`, (c) registered four SI-invalidation callbacks, then (d)
allocated `in_progress_list`. The guard variable, `TypeCacheHash`, is set in
step (a). If any of (b)/(c)/(d) threw OOM, a later call would see
`TypeCacheHash != NULL`, skip the whole block, and then hit the
`Assert(TypeCacheHash != NULL && RelIdToTypeIdCacheHash != NULL)` / NULL-deref
because `RelIdToTypeIdCacheHash` or `in_progress_list` was never created.

**The fix.** Post-image (`typcache.c:395-447 @ 73dab127`):

- The outer gate becomes `if (in_progress_list == NULL)` — and `in_progress_list`
  is deliberately the **last** allocation in the block ("This is the last
  allocation on purpose, done after the two others.").
- `TypeCacheHash` and `RelIdToTypeIdCacheHash` each get their own
  `if (... == NULL)` sub-guard, so a re-entry after partial init creates only
  what is still missing.
- The four `CacheRegister{Relcache,Syscache}Callback` calls move to the **end**,
  after every allocation, with the comment that they "can fail only with a FATAL
  error" — so they never leave a half-registered state that a non-FATAL retry
  could observe.

The net effect: a non-FATAL OOM during first init no longer poisons the cache for
the life of the backend; the next `lookup_type_cache()` call resumes and finishes
initialization. Same hardening theme as `b85f9c00`.

### 3. `d4a657b0a4db` — fix misreporting of publisher sequence permissions (Fujii Masao)

**Context.** The PG 18 sequence-sync worker copies each publisher sequence's
current value via a batched remote query, then classifies each row into
`COPYSEQ_{SUCCESS,MISMATCH,INSUFFICIENT_PERM,SKIPPED}`. [[corpus]]
`knowledge/files/src/backend/replication/logical/sequencesync.c.md` documents the
single-worker, batched (`MAX_SEQUENCES_SYNC_PER_BATCH`) design and notes
`REMOTE_SEQ_COL_COUNT = 10` — **now stale (→ 11)**.

**The bug.** `pg_get_sequence_data()` returns NULL sequence data in *two*
distinct cases: (a) the publisher role lacks SELECT on the sequence, and (b) the
sequence was dropped concurrently after it was seen in the catalog snapshot.
Pre-fix, `get_and_validate_seq_info()` saw only the NULL and returned
`COPYSEQ_SKIPPED` for both — so a genuine **publisher-side permission** problem
was misreported to the user as a sequence "missing from publisher" rather than a
privilege issue, with no actionable warning.

**The fix.** Post-image (`sequencesync.c @ d4a657b0`):

- `REMOTE_SEQ_COL_COUNT` 10 → 11 (`:75`); the remote probe SQL prepends
  `has_sequence_privilege(c.oid, 'SELECT')` as a new first data column
  (`copy_sequences`, `:498-501`), and the `seqRow` Oid template gains a leading
  `BOOLOID` (`:430-431`).
- `get_and_validate_seq_info()` reads that bool first
  (`remote_has_select_priv`, `:275-283`); on NULL seq-data it now returns
  `COPYSEQ_PUBLISHER_INSUFFICIENT_PERM` when the publisher lacked SELECT, else
  `COPYSEQ_SKIPPED` (truly dropped).
- The `CopySeqResult` enum splits `COPYSEQ_INSUFFICIENT_PERM` into
  `COPYSEQ_SUBSCRIBER_INSUFFICIENT_PERM` (the existing local-apply path,
  `copy_sequence()` `:377`) and `COPYSEQ_PUBLISHER_INSUFFICIENT_PERM`.
- `report_sequence_errors()` gains a fourth list and emits a distinct
  `errmsg_plural("insufficient privileges on publisher sequence(s) (%s)")`
  WARNING under `ERRCODE_OBJECT_NOT_IN_PREREQUISITE_STATE` (`:198-220`),
  separate from the subscriber-side warning.
- `036_sequences.pl` is updated: the publisher-revoke case now asserts the
  WARNING text `insufficient privileges on publisher sequence ("public.regress_s2")`,
  and a new sub-case drops the sequence on the publisher to confirm a *truly*
  removed sequence is still reported as missing (the two cases are now
  distinguishable in the log).

The `errcode`/`errmsg_plural` choices here are textbook for the `error-handling`
skill (SQLSTATE selection + plural message forms).
