## Deep explainers

### `03bcb6a7` — Fix REASSIGN OWNED for subscriptions in other databases (Jeff Davis)

Subscriptions live in a **shared** catalog (`pg_subscription`) but semantically
belong to a single database (`subdbid`). `REASSIGN OWNED` scans the shared
catalog and previously tried to reassign *every* subscription regardless of which
database it belonged to — wrong for cross-database subscriptions, and unsafe
because much subscription code assumes `MyDatabaseId` context.

- `subscriptioncmds.c`: `AlterSubscriptionOwner_oid()` gains a
  database-membership check that **skips** subscriptions whose `subdbid`
  differs from `MyDatabaseId`; `AlterSubscriptionOwner_internal()` gains an
  assertion of the same invariant. `[from-commit-diff]` (~lines 2868-3011)
- `pg_subscription.c`: `GetSubscription()`, `DisableSubscription()`, and
  `UpdateDeadTupleRetentionStatus()` each gain
  `Assert(subform->subdbid == MyDatabaseId)` — hardening the "operate only on
  local subscriptions" invariant. `[from-commit-diff]` (~lines 114, 278, 723)
- Docs (`reassign_owned.sgml`, `user-manag.sgml`) now state subscriptions are
  reassigned only when the command runs in the owning database.
- Test coverage added under the subscription TAP suite (the fetch reported a
  `t/010_subscription.pl`-style file; exact path `[unverified]` — the header
  attribution looked garbled, treat as "a subscription TAP test").

Corpus tie-in: `[[knowledge/files/src/backend/commands/subscriptioncmds.c.md]]`,
`[[knowledge/files/src/backend/catalog/pg_subscription.c.md]]`,
`[[knowledge/subsystems/logical-replication]]` (via the `logical-replication`
skill). Both per-file docs should note the new `subdbid == MyDatabaseId`
guard/assert as an invariant.

### `1df9e8d9` — Fix edge case in remove_useless_result_rtes() with outer joins (Tom Lane)

The RTE_RESULT-elimination pass in `prepjointree.c` could remove a result RTE
that was the **only base relation** at which a `PlaceHolderVar` could be
evaluated, when the PHV's `phrels` set also mentioned outer-join relids (which
are not base relations). Removing it left the PHV with nowhere to compute →
planner error / wrong plan.

- `remove_useless_result_rtes()` now precomputes
  `baserels = get_relids_in_jointree(jointree, false, false)`. `[from-commit-diff]` (~line 3889)
- `remove_useless_results_recurse()`, `find_dependent_phvs()`, and
  `find_dependent_phvs_in_jointree()` all gain a `Relids baserels` parameter,
  threaded through the recursion. `[from-commit-diff]` (~lines 3973, 4370, 4393)
- `find_dependent_phvs_walker()` now intersects the PHV's `phrels` with
  `baserels` before the `bms_equal` match test, so stale outer-join relids no
  longer make the PHV look movable. `[from-commit-diff]` (~lines 4334-4346)
- Regression coverage in `join.sql` / `join.out`.

Corpus tie-in: `[[knowledge/files/src/backend/optimizer/prep/prepjointree.c.md]]`.
This is a subtle PHV/outer-join invariant — worth an idiom cross-ref if the
corpus has a placeholder-var note. Load `executor-and-planner` before citing.

### `03165931` — Restore prefix `->` and `\|` operators (Tom Lane)

A regression (likely from the recent operator-precedence / grammar work) had
dropped the ability to use `->` and `|` as **unary prefix** operators. This
re-adds the productions in both expression grammars:

- `gram.y` `a_expr`: `\| RIGHT_ARROW a_expr` and `\| '|' a_expr` →
  `makeSimpleA_Expr(AEXPR_OP, "->"/"|", NULL, $2, @1)`. `[from-commit-diff]` (~lines 15973-15980)
- `gram.y` `b_expr`: identical prefix productions for the restricted
  expression grammar. `[from-commit-diff]` (~lines 16457-16464)

Grammar change ⇒ per R13 this class of change would require
`regress + ecpg + contrib-SQL-parsing` suites if we were implementing it.
Corpus tie-in: `[[knowledge/files/src/backend/parser/gram.y.md]]` — note that
`->`/`|` have both binary and unary-prefix forms again.

### `d451ca69` — Remove redundant null-treatment check in window dedup (Tatsuo Ishii)

Window-function duplicate detection in `ExecInitWindowAgg()` compared an
`ignore_nulls` flag between candidate perfuncs. That field was redundant (the
information is already captured elsewhere in the WindowFunc), so it is removed:

- Deletes `uint8 ignore_nulls;` from `WindowStatePerFuncData`. `[from-commit-diff]` (~line 113)
- Simplifies the dedup conditional from
  `if (i <= wfuncno && wfunc->ignore_nulls == perfunc[i].ignore_nulls)` to
  `if (i <= wfuncno)`. `[from-commit-diff]` (~lines 2739-2746)

Corpus tie-in: `[[knowledge/files/src/backend/executor/nodeWindowAgg.c.md]]` and
`[[knowledge/files/src/include/executor/nodeWindowAgg.h.md]]` — the struct field
list in the header doc must drop `ignore_nulls`.

### `f87375bf` — Fix underscore parsing in pg_plan_advice occurrence numbers (Daniel Gustafsson)

`pg_plan_advice`'s scanner accepted digit-group underscores loosely; it now
matches core PG numeric-literal rules and errors on malformed input:

- `pgpa_scanner.l`: swaps `#include "common/string.h"` for
  `"utils/builtins.h"`; integer conversion moves from `strtoint()`+errno to
  `pg_strtoint32_safe(yytext, (Node *) &escontext)` using an `ErrorSaveContext`
  (soft-error idiom). `[from-commit-diff]` (~lines 11-12, 139-148)
- Adds an `integer_junk {decinteger}{identifier}` pattern → "trailing junk
  after numeric literal". `[from-commit-diff]` (~lines 85, 150-155)
- New syntax tests: `x#1_0` (underscores stripped → `10`), `x#1_0_`
  (trailing-junk error), `x#99999999999_99` (integer-out-of-range error).

No per-file corpus doc for `pgpa_scanner.l` yet (pg_plan_advice is a newer
contrib). Soft-error usage cross-refs the `error-handling` skill's
`escontext`/`pg_strtoint32_safe` pattern.
