## Deep explainers

### `b574fec00f` — collation-lookup failure on `"char"` columns
`[from-diff]`

The single-row planner-estimation path `scalarineqsel()` →
`convert_to_scalar()` → `convert_string_datum()` converts a string Datum to a
sort-key form using the column's collation. For the `"char"` type (the
internal single-byte `char`, OID 18 — *not* `character(1)`), `attcollation` is
`InvalidOid`: the type is not collatable. The old code passed that straight to
`pg_newlocale_from_collation()`, which rejects `InvalidOid` and threw, so any
`WHERE charcol < 'Q'` selectivity estimate against an ANALYZEd `"char"` column
errored out at plan time.

The fix is a four-line early-out in
`convert_string_datum()` — `source/src/backend/utils/adt/selfuncs.c` ~5298
`[from-diff]`:

```c
/* If we don't have a collation, act as though it's "C". */
if (!OidIsValid(collid))
    return val;
```

`return val` short-circuits to the already-detoasted string, treating it as
byte-sortable — exactly the `"C"` semantics. The comment hedges ("perhaps there
are other cases"), so the guard is deliberately general, not `"char"`-specific.
New regress coverage in `planner_est.sql` builds a temp `"char"` table,
ANALYZEs it, and EXPLAINs `c < 'Q'`. Corpus: re-read
`[[knowledge/files/src/backend/utils/adt/selfuncs.c]]` — the
`convert_string_datum` notes should record the new `InvalidOid → "C"` contract.

### `0cd17fdd3c` — inherited CHECK constraints can't be weakened
`[from-diff]`

This closes a soundness hole in the NOT ENFORCED constraint feature: a child
could be set `NOT ENFORCED` while its parent's matching CHECK stayed
`ENFORCED`, silently weakening the inheritance-merged constraint. The change
runs through `ATExecAlterCheckConstrEnforceability()` —
`source/src/backend/commands/tablecmds.c` ~12653 `[from-diff]` — and adds:

- A new static helper `ATCheckCheckConstrHasEnforcedParent()` (forward-declared
  at `tablecmds.c` ~12466) that scans for a matching parent constraint still
  enforced and not itself part of this ALTER, returning the offending parent's
  OID. `[from-diff]`
- A `List *changing_conids` thread-through on
  `ATExecAlterCheckConstrEnforceability()` and
  `AlterCheckConstrEnforceabilityRecurse()` — the set of equivalent CHECK
  conids this one command will touch, so the parent-scan can exclude
  constraints the same statement is already changing. `[from-diff]`
- Two outcomes when a still-enforced parent is found: a **direct** ALTER
  (`!recursing`) raises `ERRCODE_INVALID_OBJECT_DEFINITION` ("cannot mark
  inherited constraint \"%s\" as NOT ENFORCED"); a **recursive** descent
  instead keeps the child `ENFORCED` (`target_enforced = true`) so the
  merged enforceability still reflects the remaining enforced parent.
  `[from-diff]`
- A "merged enforceability" write: instead of blindly applying
  `cmdcon->is_enforced`, it computes `target_enforced` and only updates the
  catalog tuple when `currcon->conenforced != target_enforced`. `[from-diff]`

Partitions are exempt from the recursive parent-check (a partition has exactly
one direct parent, so the simpler `!recursing` path covers them). The companion
docs `f03ecd26` (`alter_table.sgml`) and `d16be860` (`ddl.sgml`) spell out the
new user-visible rule. Corpus: this is a meaty addition to the ALTER TABLE
constraint machinery — flag `[[knowledge/files/src/backend/commands/tablecmds.c]]`
for an audit pass covering the new helper + the `changing_conids` recursion
contract.

### `effb923d9d` + `02f699c141` — COPY TO JSON column order
`[from-diff]`

`BeginCopyTo()` builds a projected `TupleDesc` so `composite_to_json()` emits
the selected columns with the right names/types. The old guard fired only when
the column list was *shorter* than the table —
`list_length(cstate->attnumlist) < tupDesc->natts` — so a same-length but
**reordered** list (`COPY t (c,b,a)`) fell through to the full descriptor and
JSON came out in physical column order. The fix widens the guard at
`source/src/backend/commands/copyto.c` ~1054 `[from-diff]`:

```c
if (rel && (attnamelist != NIL ||
            list_length(cstate->attnumlist) < tupDesc->natts))
```

i.e. project whenever an explicit list was given at all. Regress adds a
`copytest_order (a,b,c)` table and checks `(c,b,a)` output in text, CSV, and
JSON — only JSON was actually wrong before. The follow-up `02f699c141`
re-indents the new block (the original commit tripped the buildfarm
`indent-check` stage; see buildfarm-lessons 2026-06-28). Corpus link:
`[[knowledge/files/src/backend/commands/copyto.c]]`.

### `d6ed87d198` — named boolean params for `pg_get_*_ddl()`
`[from-diff]`

Ergonomics + catalog change. The `pg_get_*_ddl()` reporting functions in
`ddlutils.c` took positional boolean option flags; this reworks them to named
boolean parameters (`pretty => true` style), letting callers pass only the
flags they care about. The diff is net −195 lines (95 add / 290 del) — most of
the deletion is collapsed option-decoding boilerplate. Because builtin function
argument *names* live in `pg_proc.dat`, the change edits
`source/src/include/catalog/pg_proc.dat` and bumps
`source/src/include/catalog/catversion.h` `[from-diff]`. Docs updated in
`func-info.sgml`; the `test_misc/t/012_ddlutils.pl` TAP test exercises the new
call style. Corpus: `pg_proc.dat` has no per-file doc, but
`[[knowledge/files/src/backend/utils/adt/ddlutils.c]]` and
`[[knowledge/files/src/include/catalog/catversion.h]]` should note the bump.
