## Deep dive — `9bcb8a694b3b` — PL/Python on funccache.c + bug #19480 (Tom Lane)

### What `funccache.c` buys PL/Python

`funccache.c` is the generic function-cache layer factored out of plpgsql: it
keys a `CachedFunction` by `(fn_oid, trigger relation/type, input collation,
...)`, recompiles on catalog invalidation, and reference-counts entries so an
in-flight execution keeps its compiled form alive even if the function is
`CREATE OR REPLACE`-d underneath it. [[corpus]]
`knowledge/files/src/backend/utils/cache/funccache.c.md` documents the
`cached_function_compile()` / `CachedFunctionHashKey` / use-count contract.

Before this commit PL/Python rolled its own hash (`init_procedure_caches()`,
removed from `_PG_init` — see `plpy_main.c` diff dropping the call) and looked
procedures up by bare OID via `PLy_procedure_get(funcoid, relid, trigtype)`.
After it, the procedure is a `CachedFunction` subtype: the `PLyProcedure` struct
now puts `CachedFunction` **first** so funccache can manage it
(`src/pl/plpython/plpy_procedure.h:42-46` @ `9bcb8a69`), and lookup goes through
`cached_function_compile()` (`src/pl/plpython/plpy_procedure.c:113` @ `9bcb8a69`).

### The new per-call-site object: `PLyProcedureCache`

The key structural addition is `PLyProcedureCache`, stored in the FmgrInfo's
`fn_extra` (`src/pl/plpython/plpy_procedure.h:80-89` @ `9bcb8a69`):

```c
typedef struct PLyProcedureCache
{
    PLyProcedure *proc;        /* the long-lived, funccache-managed compiled proc */
    MemoryContextCallback mcb; /* fires RemovePLyProcedureCache on fn_mcxt reset */
    MemoryContext fcontext;    /* = fn_mcxt, context holding this struct */
    PLySRFState *srfstate;     /* SRF iterator state, NULL when not mid-SRF */
    bool        shutdown_reg;  /* true once the ExprContext shutdown cb is registered */
} PLyProcedureCache;
```

`PLy_procedure_get()` now takes the whole `fcinfo` (not a bare OID) and returns
this cache object, allocating it in `fn_mcxt` and registering a memory-context
reset callback `RemovePLyProcedureCache` (`plpy_procedure.c:62`, `:77-81`,
`:536` @ `9bcb8a69`). The handler `plpython3_call_handler` was rewritten to call
`PLy_procedure_get(fcinfo, false)` once at the top of its `PG_TRY` and pass the
cache through to all three execution paths (`plpy_main.c`, the
`plpython3_call_handler` hunk). The validator `plpython3_validator` had to
synthesise a **fake** `fcinfo`/`flinfo` (+ `TriggerData`/`EventTriggerData`
context) because `PLy_procedure_get` now derives call context from the fcinfo
rather than an explicit `trigtype` argument, then drops the use-count it took
(`plpy_main.c` validator hunk).

### The SRF rework — off `SRF_*`, onto `rsi->isDone`

PL/Python returns sets in **ValuePerCall** mode: the handler is re-entered once
per row. Previously this leaned on the executor's `FuncCallContext` /
`SRF_FIRSTCALL_INIT` / `SRF_PERCALL_SETUP` / `SRF_RETURN_NEXT` machinery, parking
the iterator in `funcctx->multi_call_memory_ctx`. Now the iterator lives in the
per-call-site `pcache->srfstate`, and "first call of a new iteration" is detected
by `pcache->srfstate == NULL` (`src/pl/plpython/plpy_exec.c:70` @ `9bcb8a69`).
The handler signals continuation/termination by writing `rsi->isDone` directly:

- `rsi->isDone = ExprMultipleResult; return rv;` for each row
  (`plpy_exec.c:300` @ `9bcb8a69`);
- on iterator exhaustion, unregister the shutdown callback, clean up, set
  `rsi->isDone = ExprEndResult`, `fcinfo->isnull = true`, `return (Datum) 0`
  (`plpy_exec.c:294` @ `9bcb8a69`).

`PLy_exec_function`'s signature changed accordingly to take `PLyProcedureCache *`
(`plpy_exec.c:41`; header `plpy_exec.h`).

### The four SRF termination paths (the leak-safety core)

Python iterator + saved-args hold **Python refcounts**, which — unlike SQL
resources — are *not* released by transaction abort. The commit centralises all
release in `PLy_function_cleanup_srfstate()` (`plpy_exec.c:357` @ `9bcb8a69`),
whose header comment enumerates the only four ways a PL/Python SRF can end:

1. **Normal completion** — called from the handler's normal exit.
2. **Error inside the SRF** — called from the handler's `PG_CATCH`.
3. **Early query termination** (e.g. `LIMIT`, or a rescan of the owning plan
   node) — called via the `ExprContext` shutdown callback `ShutdownPLyFunction`
   (`plpy_exec.c:319` @ `9bcb8a69`), registered on `rsi->econtext`.
4. **Error elsewhere in the query** — called during (sub)xact abort via the
   `fn_mcxt` reset callback `RemovePLyProcedureCache`.

The function is idempotent (no-op if `srfstate == NULL`) because several paths
can both fire. This replaces the old single `plpython_srf_cleanup_callback`
hung off `multi_call_memory_ctx` (deleted in the diff). [[corpus]]
`knowledge/files/src/pl/plpython/plpy_exec.md` and
`knowledge/idioms/` (the SRF ValuePerCall idiom) should note the
`RegisterExprContextCallback`/`UnregisterExprContextCallback` pairing as the
PL/Python SRF cleanup mechanism.

### Bug #19480 — invalidate-mid-iteration must finish on the *original* proc

The regression test added in `expected/plpython_setof.out` is the spec: a SRF
that `CREATE OR REPLACE`-s *itself* on its second yield must still finish the
current call returning rows from the **original** body, and only the *next*
top-level call sees the replacement. This now holds because the iteration's
compiled `PLyProcedure` is pinned (use-count held) in `pcache->proc` for the
whole ValuePerCall sequence rather than being re-looked-up per row — the same
guarantee funccache already gave plpgsql SET-returning functions. The test also
adds `LIMIT`-truncated SRF cases, exercising termination path 3.

### Risk surface

- **No catalog / WAL / on-disk / ABI change.** Pure PL/Python-internal refactor
  plus a behaviour fix; `typedefs.list` gains `PLyProcedureCache`.
- **Extension ABI:** `PLy_exec_function`'s signature changed, but it is internal
  to the plpython module (not exported for third-party use), so no external break.
- The validator's fake-fcinfo construction is the most error-prone new code; it
  must keep matching whatever context fields `PLy_procedure_get` inspects.
