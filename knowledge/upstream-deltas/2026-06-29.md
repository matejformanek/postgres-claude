### `6f4bac854fb7` тАФ Hardwire RI fast-path end-of-xact cleanup into xact.c (Amit Langote)

The recently-added referential-integrity *fast-path batching* mechanism
(`ri_triggers.c`) previously registered a `RegisterXactCallback(
ri_FastPathXactCallback, тАж)` lazily, the first time a fast-path hash entry
was created. This commit removes that callback entirely and instead wires a
new `AtEOXact_RI(bool isCommit)` directly into the three end-of-xact paths in
`xact.c`:

- `CommitTransaction()` (`xact.c:2517`, after `AtEOXact_HashTables`)
- `PrepareTransaction()` (`xact.c:2813`)
- `AbortTransaction()` (`xact.c:3044`)

`AtEOXact_RI()` (now exported via `include/commands/trigger.h:313`) only
resets backend-local static pointers (`ri_fastpath_cache`,
`ri_fastpath_callback_registered`); the cache memory itself lives in
`TopTransactionContext` and is freed by the normal context reset. On a clean
commit/prepare the cache must already be empty (flushed by
`ri_FastPathEndBatch()`, an `AfterTriggerBatchCallback`); a survivor means a
trigger batch went unflushed тЖТ FK checks may have been skipped, so it now
`elog(WARNING, "RI fast-path cache not flushed at end of transaction")` plus
an `Assert`. The header comment notes the reset has **no ordering dependency**
on surrounding `AtEOXact_*` steps because it touches no relations/locks/
buffers/catalogs.

Why it matters: replacing an opportunistic XactCallback with a hardwired
end-of-xact step is the standard PG pattern (cf. `AtEOXact_Files`,
`AtEOXact_HashTables`, `AtEOXact_ComboCid`) тАФ more predictable ordering, no
registration race. **[corpus]** Per-file docs exist for all three touched
files; see the crossref report.

### `b574fec00f27` тАФ Avoid collation lookup failure when considering a "char" column (Tom Lane)

### `8612f0b7ce09` тАФ plpython NULL-deref hardening (Richard Guo)

A broken Python sequence or mapping object (one whose `__len__`, `items()`,
or `__getitem__` lies or returns `NULL` with an exception set) could crash
the backend in PL/Python's type-IO and SPI result paths. The fix adds
defensive checks across `plpy_typeio.c`, `plpy_spi.c`, and
`plpy_cursorobject.c`: validate `PySequence_Size`/`PyMapping_Size < 0`,
`items() == NULL`, each `PySequence_GetItem() == NULL`, and that mapping
`items()` entries are 2-tuples тАФ converting each into a clean
`ereport`/`PLy_elog(ERROR, тАж)` instead of dereferencing NULL. Mirrored in
the `hstore_plpython` / `jsonb_plpython` contrib transforms with matching
expected-output updates. Not in the C-backend corpus (PL + contrib), so it
does not gate an anchor bump, but it's a good defensive-coding exemplar.
