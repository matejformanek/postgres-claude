## Deep explainers

### #11 `323530e00d27` ‚Äî heap_get_root_tuples: sanity check tested the wrong variable (Peter Geoghegan)

**The bug.** Commit `c7aeb775` rewrote the HOT-chain offset sanity checks in
three places. In `heap_get_root_tuples()` the rewrite accidentally tested
`offnum` ‚Äî the *outer* loop variable, already bounded by the loop condition ‚Äî
instead of `nextoffnum`, the offset actually passed to `PageGetItemId()`.

```c
/* src/backend/access/heap/pruneheap.c ~2374, heap_get_root_tuples() */
-  if (offnum < FirstOffsetNumber)      break;   /* wrong: offnum is already bounded */
+  if (nextoffnum < FirstOffsetNumber)  break;
   ...
-  if (offnum > maxoff)                 break;   /* wrong var again */
+  if (nextoffnum > maxoff)             break;
   lp = PageGetItemId(page, nextoffnum);          /* the value we should be guarding */
```

**Why it matters.** With the check testing the wrong variable it was
**ineffective**: a stale `t_ctid` chaining to an out-of-range `nextoffnum` could
make `PageGetItemId()` read *past the end of the line-pointer array*. This is
exactly the corruption-detection case the check exists to catch ‚Äî the guard was
present in source but silently doing nothing since `c7aeb775`. The pre-`c7aeb775`
code tested `nextoffnum`; this restores that. Reported by Konstantin Knizhnik.
Backpatch-through 15 (so the regression window spans every branch that took
`c7aeb775`).

`[[knowledge/files/src/backend/access/heap/pruneheap.c.md]]` ‚Äî the
`heap_get_root_tuples` notes should flag that the HOT-chain walk guards
`nextoffnum`, not the loop variable. `[[knowledge/subsystems/access-heap.md]]`
HOT-chain traversal section is impacted.

### #1 `4dbc54506c32` ‚Äî visibilitymap_clear: restore elog(ERROR) for wrong-buffer (Melanie Plageman)

A direct follow-up to `ed62d26caca` (the VM-clear WAL-logging corruption fix
tracked in the 07-16/07-14 windows). That fix made an *incidental* change:
it downgraded the wrong-buffer guard in `visibilitymap_clear()` from an
`elog(ERROR)` to an `Assert()`, and moved `mapBlock` under
`#ifdef USE_ASSERT_CHECKING`.

```c
/* src/backend/access/heap/visibilitymap.c ~154-172 */
-#ifdef USE_ASSERT_CHECKING
   BlockNumber mapBlock = HEAPBLK_TO_MAPBLOCK(heapBlk);
-#endif
   ...
-  Assert(BufferIsValid(vmbuf) && BufferGetBlockNumber(vmbuf) == mapBlock);
+  if (!BufferIsValid(vmbuf) || BufferGetBlockNumber(vmbuf) != mapBlock)
+      elog(ERROR, "wrong buffer passed to visibilitymap_clear");
```

**Rationale.** `visibilitymap_set()` still reports the same wrong-buffer
condition with `elog(ERROR)`, so `clear()` should match for consistency ‚Äî a new
caller passing the wrong buffer must fail loudly even in a non-assert
production build, not just under `cassert`. The change was unrelated to the
WAL-logging fix and belonged in its own commit. Reported by Fujii Masao;
Backpatch-through 19.

`[[knowledge/files/src/backend/access/heap/visibilitymap.c.md]]` ‚Äî the
`visibilitymap_clear` notes should record that the wrong-buffer guard is an
`elog(ERROR)` (not an Assert) as of this commit, symmetric with `_set`.

### #3 `c9ea37231a2e` ‚Äî GiST index-only scan: deform xs_hitup with the AM's own descriptor (Peter Geoghegan)

**The bug.** An index-only scan (IoS) filled its result slot from the
`HeapTuple` an index AM returns in `scan->xs_hitup` by deforming it with the
*virtual slot's own* tuple descriptor. But GiST / SP-GiST form that heap tuple
using **their own** descriptor `scan->xs_hitupdesc`, which can disagree with the
IoS virtual slot's descriptor about a column's precise alignment. Result:
"can't happen" errors, and ‚Äî much less likely ‚Äî hard crashes, when reading a
misaligned datum.

**The fix + refactor.** `StoreIndexTuple()` is re-signatured to take the
`IndexScanDesc` and to deform with the descriptor the tuple was *formed with*
(`xs_hitupdesc` for the heap-tuple path, `xs_itupdesc` for the index-tuple
path), unifying the two paths in `IndexOnlyNext()`:

```c
/* src/backend/executor/nodeIndexonlyscan.c ~194 */
-  if (scandesc->xs_hitup) { ... ExecForceStoreHeapTuple(...); }
-  else if (scandesc->xs_itup) StoreIndexTuple(node, slot, scandesc->xs_itup, scandesc->xs_itupdesc);
-  else elog(ERROR, "no data returned for index-only scan");
+  StoreIndexTuple(node, slot, scandesc);   /* now handles hitup + itup uniformly */
```

**Blast radius.** In practice extremely unlikely to hit: the only affected core
opclass is GiST `range_ops`; the datum had to be the second-or-later column of a
multicolumn GiST index *and* an earlier column had to leave the range datum at
an offset where the AM/slot alignments actually disagree (e.g. an odd-length
varlena). It was found while testing a patch that changes the table-AM API used
during index scans, not by a field report. Reviewed by Tomas Vondra.

`[[knowledge/files/src/backend/executor/nodeIndexonlyscan.c.md]]` ‚Äî the
`StoreIndexTuple` / `IndexOnlyNext` notes need the new signature and the
"deform with the descriptor the AM formed the tuple with" invariant.
`[[knowledge/subsystems/executor-and-planner.md]]` IoS section impacted.
`[[knowledge/skills/access-method-apis]]` ‚Äî `xs_hitup` / `xs_hitupdesc` vs
`xs_itup` / `xs_itupdesc` contract is exactly this skill's territory.

### #13 `cea74f48d36b` ‚Äî hash AM: use a fake LSN in log_split_page for unlogged rels (Peter Geoghegan)

An oversight in `e5836f7b`, which added fake-LSN support to the hash index AM
but missed `log_split_page()`. For an unlogged relation there is no WAL record,
so the old code left the page LSN unset; the fix hoists `PageSetLSN` out of the
`RelationNeedsWAL` branch and feeds it `XLogGetFakeLSN(rel)` in the unlogged
case:

```c
/* src/backend/access/hash/hashpage.c ~1477, log_split_page() */
   if (RelationNeedsWAL(rel)) { ...; recptr = XLogInsert(RM_HASH_ID, XLOG_HASH_SPLIT_PAGE); }
+  else
+      recptr = XLogGetFakeLSN(rel);
+  PageSetLSN(BufferGetPage(buf), recptr);
```

Defensive consistency: all hash-AM atomic actions on an unlogged relation now
carry a monotonic fake LSN, matching the rest of the AM. Backpatch-through 19.

`[[knowledge/files/src/backend/access/hash/hashpage.c.md]]` ‚Äî `log_split_page`
notes should record the fake-LSN branch. `[[knowledge/skills/wal-and-xlog]]`
fake-LSN / unlogged-relation section is the relevant idiom.

### #2 `a32733d8f10f` ‚Äî Revert "Add GROUP BY ALL" (Tom Lane)

wordsmithing on it). **Why:** a postcommit review found GROUP BY ALL missed
PostgreSQL's special handling of entries that *also* appear in an `ORDER BY` ‚Äî
so a query returned **wrong results** when the ORDER BY specified non-default
equality semantics. Fixable with refactoring, but "too much code churn for late
beta"; the plan is to try again in v20.

(‚àí63 lines of transform), `parser/analyze.c`, `utils/adt/ruleutils.c`
(deparse), `nodes/parsenodes.h` (**`struct Query` field removed ‚Üí catversion
bump**), `parser/parse_clause.h`, plus docs and `aggregates.{sql,out}` tests.
Some comment/doc cleanup from the original was *retained* as unrelated to the
feature.

`[[knowledge/files/src/backend/parser/parse_clause.c.md]]`,

### #8 `1f414035135c` ‚Äî base64 (non-url) must still require padding (Daniel Gustafsson)

Commit `e1d917182` added base64url (URL/filename-safe variant, no `=` padding).
It unintentionally relaxed the padding requirement for **plain base64** too, so
a truncated base64 string like `'AQ'` was silently accepted instead of raising
`invalid base64 end sequence`. The one-line fix gates the "accept short final
group" branches on the `url` flag:

```c
/* src/backend/utils/adt/encode.c ~583, pg_base64_decode_internal() */
-  if (pos == 2)      { ... }   /* accepted for base64 AND base64url ‚Äî wrong */
+  if (url && pos == 2)      { ... }
-  else if (pos == 3) { ... }
+  else if (url && pos == 3) { ... }
```

Regress addition in `strings.sql`: `SELECT decode('AQ', 'base64')` must now
error. Backpatch-through 19 (where base64url was introduced).

`[[knowledge/files/src/backend/utils/adt/encode.c.md]]` ‚Äî record the
`url`-gated padding rule: base64url tolerates missing padding, plain base64 does
not.

### #9 `32e4508db27d` ‚Äî pg_stat_io: prune impossible rows for data-checksum procs (Fujii Masao)

The data-checksums launcher and workers (`B_DATACHECKSUMSWORKER_LAUNCHER` /
`_WORKER`, the online enable-checksums feature) were exposed in `pg_stat_io`
with the *full* general-bgworker matrix of object/context combos ‚Äî including
combinations they can never accumulate (`bulkwrite`, relation `init`, temp
relation, launcher `vacuum`). The commit teaches
`pgstat_tracks_io_object()` and `pgstat_tracks_io_op()` the actual I/O shape of
these processes:

```c
/* src/backend/utils/activity/pgstat_io.c ~464, pgstat_tracks_io_object() */
+  if (bktype == B_DATACHECKSUMSWORKER_LAUNCHER) {
+      if (io_object == IOOBJECT_WAL ||
+          (io_object == IOOBJECT_RELATION &&
+           (io_context == IOCONTEXT_BULKREAD || io_context == IOCONTEXT_NORMAL)))
+          return true;
+      return false;
+  }
+  if (bktype == B_DATACHECKSUMSWORKER_WORKER) { /* + IOCONTEXT_VACUUM */ ... }
```

Kept: catalog scans (incl. bulkread), worker relation processing with a vacuum
access strategy, WAL writes/init. Excluded: WAL reads (added to the existing
WAL-read exclusion list in `pgstat_tracks_io_op`) and the impossible
object/context combos. Pure accounting-fidelity change ‚Äî the `stats.out`
expected file loses the now-impossible rows.

`[[knowledge/files/src/backend/utils/activity/pgstat_io.c.md]]` ‚Äî update the
`pgstat_tracks_io_object` / `pgstat_tracks_io_op` per-BackendType matrix to add
the two `B_DATACHECKSUMSWORKER_*` special cases. `[[knowledge/skills/pgstat-framework]]`
I/O-tracking dispatch is the relevant idiom.
