## Deep dives

### `4cb2a9863d` — Fix LATERAL references in GRAPH_TABLE multi-label patterns

A `GRAPH_TABLE (... MATCH ...)` whose element pattern carries **multiple labels**
(`-[IS a | b]->`) is rewritten into a `UNION` of one path query per label
combination. The bug: when each path query is wrapped as a subquery of that
UNION, any `Var` that already referenced **outside** the GRAPH_TABLE (a LATERAL
reference to the enclosing FROM, e.g. `WHERE c.customer_id = x1.a`) was left at
its old `varlevelsup`, so it pointed one query level too shallow after the extra
UNION-subquery nesting was introduced — wrong results or an assertion failure.

Fix is one call in `generate_setop_from_pathqueries()`
(`src/backend/rewrite/rewriteGraphTable.c:720`):

```c
/* Each path query becomes a UNION subquery -> bump outer Vars one level. */
IncrementVarSublevelsUp((Node *) lquery, 1, 1);
```

Classic outer-Var-rebasing idiom (same `IncrementVarSublevelsUp` used when the
rewriter/planner pushes a query down a level). Single-label patterns were never
affected because they skip the UNION wrapping. Regression coverage added in
`graph_table.{sql,out}` (multi-label `customer_orders | customer_wishlists`).
Touches `[[knowledge/files/src/backend/rewrite/rewriteGraphTable.c.md]]`.

### `72498a8698` — objectaddress support for element-label / label-property

SQL/PGQ property graphs grew two more catalogs:
`pg_propgraph_element_label` and `pg_propgraph_label_property`. This wires their
relation OIDs through the object-address machinery so DROP/COMMENT/`pg_describe`
and `pg_identify_object*` work on them. Three sites in
`src/backend/catalog/objectaddress.c`:

- `object_type_map[]` ~`:898` / `:906` — two new `"property graph element label"`
  and `"property graph label property"` rows (mapped `-1`, i.e. not directly
  addressable by `ALTER`).
- `getObjectTypeDescription()` ~`:4906` — `PropgraphElementLabelRelationId` /
  `PropgraphLabelPropertyRelationId` cases returning the human strings.
- `getObjectIdentityParts()` ~`:6229` — two new blocks that `table_open` the
  catalog, fetch by OID via `get_catalog_object_by_oid`, and build a qualified
  identity (`<label> of <element>` / `<prop> of <label>`), composing the parent
  element/label identity recursively.

Pure plumbing — no behavior change for non-PGQ users — but it's a good template
for "what it takes to make a new catalog object first-class to the dependency
system." Touches `[[knowledge/files/src/backend/catalog/objectaddress.c.md]]`.

### `1a5b19e447` — pg_subscription column-privilege fix for subwalrcvtimeout

When `subwalrcvtimeout` was added to `pg_subscription`, the public
column-level `GRANT SELECT` list in `system_views.sql` was not updated, so the
new column was silently *not* readable by non-superusers (the catalog's design
is: every column except `subconninfo` is world-readable; `subconninfo` is
withheld because it holds the connection string / password). Fix adds
`subwalrcvtimeout` to the GRANT list
(`src/backend/catalog/system_views.sql:1530`) and bumps
`CATALOG_VERSION_NO` to `202606051`. New regress assertion confirms the
invariant directly: *only* `subconninfo` lacks public `SELECT`
(`subscription.sql`, via `has_column_privilege` over `pg_attribute`). A
catversion bump means an initdb is required to pick this up in `dev/`.

### `f2ff15e4c3` — Hangul NFC off-by-one at TBASE (U+11A7)

`recompose_code()` composes an LV Hangul syllable with a trailing consonant (T)
into an LVT syllable. The trailing-jamo guard used `code >= TBASE`, but `TBASE`
(U+11A7) is the *filler* slot that is **not** a real trailing consonant — index
0 must not combine. Result: `normalize(U&'\AC00\11A7', NFC)` wrongly produced a
combined syllable instead of leaving the two code points separate.

One-character fix in `src/common/unicode_norm.c:239`:

```c
-           code >= TBASE && code < (TBASE + TCOUNT))
+           code > TBASE  && code < (TBASE + TCOUNT))
```

Extensive new Hangul L+V / LV+T / L+V+T composition + NFD decomposition
coverage in `unicode.{sql,out}`, including the U+11A7 "do not combine" case.
This is shared `src/common` code → affects backend `normalize()`, frontend, and
any tool linking libpgcommon. Touches
`[[knowledge/files/src/common/unicode_norm.c.md]]`.

### `1fbe2066dc` — refint: remove the home-grown SPI plan cache

`contrib/spi/refint.c` (the legacy `check_primary_key` / `check_foreign_key`
example triggers) carried its own plan cache: a static `EPlan` array keyed by
`"<trigger>$<reloid>"`, with `find_plan()` lookup and `SPI_keepplan` into
`TopMemoryContext`. This commit deletes all of it — the `EPlan` struct, the
`FPlans`/`PPlans` statics, `find_plan()`, the `ident[]` keys, and the
`memutils.h` include — and just calls `SPI_prepare` each invocation (SPI's own
plan cache now suffices). Net `-235/+115`. No new tests (example module;
existing `contrib/spi` coverage exercises the paths). Not corpus-tracked
(`contrib/spi` has no per-file doc), but a clean illustration of "stop
hand-rolling what SPI already caches."
