## Deep explainers

### #1 + #2 — Logical-decoding status: two follow-ups to the PG19 dynamic-activation feature (Sawada)

Both fixes plug gaps in `67c20979ce7`, the new machinery that dynamically
**activates and deactivates logical decoding** based on whether any logical
replication slot currently exists (rather than requiring `wal_level=logical` to
be set permanently at postmaster start). The live status lives in shared memory
and is snapshotted into checkpoint records so it can be restored at startup.

**#1 `8108765f04b8` — `pg_controldata` was not printing the status.** One-line
oversight: the checkpoint record already carries
`checkPointCopy.logicalDecodingEnabled`, but `pg_controldata` never emitted it.
The fix adds a single `printf` next to the existing `full_page_writes` line
(`src/bin/pg_controldata/pg_controldata.c:266-267`), so operators can now read
"Latest checkpoint's logical decoding: on/off" when diagnosing dynamic
activation/deactivation issues. Backpatch-through: 19 (the feature's release).
`[[knowledge/files/src/bin/pg_controldata/pg_controldata.c.md]]` — the impacted
per-file doc; the printed-fields list there needs the new row appended.

**#2 `c209931bd40c` — recovery-time assertion failure with `wal_level=minimal`.**
The real bug. `UpdateLogicalDecodingStatusEndOfRecovery()` had a fast-path that,
under `wal_level=minimal`, asserted logical decoding is *never* active and
returned early
(`src/backend/replication/logical/logicalctl.c:561`, old lines 564-572). But
crash recovery under `minimal` can legitimately **replay an
`XLOG_LOGICAL_DECODING_STATUS_CHANGE` record that turns decoding on** — if the
server previously ran at a higher `wal_level`, dropped its last logical slot, and
crashed *after* that but *before* the checkpointer deactivated decoding, then
restarted with `wal_level=minimal`. Replaying the record is correct (it reflects
the status when written), but it violated the assertion; in production builds
decoding would wrongly stay active under `minimal`. The fix **deletes the
special case** and lets the normal recomputation run under the
`LogicalDecodingControlLock`: since `RestoreSlotFromDisk()` rejects logical slots
under `minimal`, `CheckLogicalSlotExists()` returns false and the recompute
always lands on `new_status = false`, writing the corresponding status-change
record to disable decoding. Net: fewer special cases, correct behavior. This is
a classic "the assertion encoded an invariant that recovery can transiently
break" bug. `[[knowledge/files/src/backend/replication/logical/logicalctl.c.md]]`
— impacted; the doc's description of `UpdateLogicalDecodingStatusEndOfRecovery`
should drop the minimal-WAL early-return branch and note the crash-window
scenario. Backpatch-through: 19.

### #3 `7575215e0b9c` — uuidv7() interval shift: reject infinite / out-of-range (Sawada)

`uuidv7(shift interval)` shifts the embedded timestamp by an interval. Two
holes are closed in `src/backend/utils/adt/uuid.c`
(`uuidv7_interval`, around old line 690):

1. **Infinite intervals** (`'infinity'::interval`) flowed into
   `timestamptz_pl_interval` arithmetic before any check. Now an
   `INTERVAL_NOT_FINITE(shift)` guard `ereport`s
   `ERRCODE_DATETIME_VALUE_OUT_OF_RANGE` ("interval out of range for UUID version
   7") *before* any arithmetic.
2. **Out-of-range result timestamps.** UUIDv7 encodes time as a **48-bit
   unsigned millisecond field relative to the Unix epoch**, so the representable
   window is `[1970-01-01, ~year 10889]`. The commit adds two named bounds,
   `UUIDV7_MIN_TIMESTAMP` / `UUIDV7_MAX_TIMESTAMP`, expressed in *PostgreSQL-epoch
   microseconds* (via a new `PG_UNIX_EPOCH_OFFSET_US` macro), and range-checks the
   shifted `TimestampTz` **before** converting to Unix-epoch microseconds — the
   ordering matters because doing the check in PG-epoch units guarantees the
   later `us = ts + PG_UNIX_EPOCH_OFFSET_US` conversion cannot itself overflow.
   Out-of-range shifts now raise a clear error instead of silently producing a
   garbage UUID.

Nice secondary cleanup: the two open-coded epoch-offset expressions collapse into
the single `PG_UNIX_EPOCH_OFFSET_US` macro. `[[knowledge/files/src/backend/utils/adt/uuid.c.md]]`
— impacted; add the two new range macros + the finite/range guards to the
`uuidv7_interval` section.

### #4 `637aa273e633` — transformIndexStmt: transform expressions in INCLUDING too (Tom Lane)

`transformIndexStmt` already ran `transformExpr` +
`assign_expr_collations` over expression elements in the *key* column list
(`stmt->indexParams`). It did **not** do so for `stmt->indexIncludingParams`
(the `INCLUDING (...)` payload columns). Today expression elements in INCLUDING
are rejected downstream, so this was latent — but the parse tree was
inconsistent, and the moment INCLUDING-expressions become supported it would
have produced un-transformed nodes (uncollated, unresolved) reaching the
executor. The fix adds a symmetric `foreach` over `indexIncludingParams` that
transforms any `ielem->expr` and fixes its collations, right after the key-column
loop (`src/backend/parser/parse_utilcmd.c:3125-3142`). Tom's comment is explicit
that this is defensive future-proofing ("those will be rejected later on, but
probably someday we'll wish to support them"). Pure correctness/consistency, no
behavior change for currently-accepted syntax.
`[[knowledge/files/src/backend/parser/parse_utilcmd.c.md]]` — impacted; note the
INCLUDING expression-transform loop.

### #5 `55f518684895` — sequencesync: tolerate concurrent sequence drops (Fujii Masao)

Part of the PG19 **logical replication of sequences** work. During sequence
synchronization the subscriber reads a batch of remote sequence rows via a
function that reports, per sequence, whether the publisher role has SELECT
privilege and the sequence's current data.
`get_and_validate_seq_info()` in
`src/backend/replication/logical/sequencesync.c` (around line 289) previously
did `remote_has_select_priv = DatumGetBool(slot_getattr(...)); Assert(!isnull);`
— it assumed the privilege column is never NULL. But
`has_sequence_privilege()` returns **NULL, not false**, when the sequence was
**dropped concurrently** on the publisher after it was captured in the catalog
snapshot (see `has_sequence_privilege_id()`). The old `Assert(!isnull)` would
fire in cassert builds; in production the NULL Datum would be misread as a bool.
The fix reads the column, and on `isnull` returns `COPYSEQ_SKIPPED` (treat as a
concurrently-dropped sequence — skip it) rather than asserting. Symmetric with
the existing NULL-handling on the sequence-state column just below. TAP coverage
added in `src/test/subscription/t/036_sequences.pl`.
`[[knowledge/files/src/backend/replication/logical/sequencesync.c.md]]` —
impacted; the `get_and_validate_seq_info` skip-code table gains the
privilege-column-NULL → COPYSEQ_SKIPPED path.

### #13 `3cf5264557be` — libpqwalreceiver: validate CREATE_REPLICATION_SLOT response shape (Fujii Masao)

Defensive protocol hardening in
`src/backend/replication/libpqwalreceiver/libpqwalreceiver.c`
(`libpqrcv_create_slot`, around line 1000). After sending
`CREATE_REPLICATION_SLOT` and confirming `PGRES_TUPLES_OK`, the code
immediately indexed `PQgetvalue(res, 0, 1)` etc. — assuming the primary always
returns exactly **1 row × 4 columns**. A malformed / unexpected response (buggy
or hostile primary, protocol drift) would read out of bounds. The fix adds an
explicit `PQnfields(res) != 4 || PQntuples(res) != 1` guard that raises
`ERRCODE_PROTOCOL_VIOLATION` ("invalid response from primary server") with a
detail line naming the actual vs expected row/field counts, before any
`PQgetvalue`. Small, purely defensive; a good example of the "validate wire
response shape before indexing" idiom.
`[[knowledge/files/src/backend/replication/libpqwalreceiver/libpqwalreceiver.c.md]]`
— impacted; add the response-shape guard to the `libpqrcv_create_slot` notes.
