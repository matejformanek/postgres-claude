## Deep dives

### 1. `4cc02b80774e` — Nail pg_parameter_acl in relcache (Jeff Davis)

A GUC named in the startup packet of a **physical replication** connection can
trigger an ACL check (`pg_parameter_aclcheck`) very early in backend startup —
before `RelationCacheInitializePhase3()` has finished building the non-critical
relcache. If `pg_parameter_acl` (and its lookup indexes) aren't yet available,
the lookup errors out. The fix promotes `pg_parameter_acl` to a *nailed*
critical shared relation, mirroring how `pg_authid` / `pg_auth_members` are
nailed for client-auth lookups `[from-diff]`.

Three coordinated edits:

- `utils/cache/catcache.c` `IndexScanOK()` (~`:1338`) adds `PARAMETERACLNAME`
  and `PARAMETERACLOID` to the set of caches allowed to do index scans before
  the relcache is fully initialized `[from-diff]`.
- `utils/cache/relcache.c`: new `#include "catalog/pg_parameter_acl.h"`
  (~`:59`); a `Desc_pg_parameter_acl` formrdesc descriptor (~`:124`);
  `formrdesc("pg_parameter_acl", …)` in `RelationCacheInitializePhase2()`
  with `NUM_CRITICAL_SHARED_RELS` raised 5 → 6 (~`:4089`); and two
  `load_critical_index()` calls (`ParameterAclParnameIndexId`,
  `ParameterAclOidIndexId`) in `RelationCacheInitializePhase3()` with
  `NUM_CRITICAL_SHARED_INDEXES` raised 6 → 8 (~`:4232`) `[from-diff]`.
- `include/catalog/pg_parameter_acl.h` gains
  `BKI_ROWTYPE_OID(2173,ParameterAclRelation_Rowtype_Id) BKI_SCHEMA_MACRO`
  on the `CATALOG(...)` line (~`:32`) so the nailed descriptor has a stable
  composite rowtype OID `[from-diff]`.

Regression introduced by `a0ffa885e4`; no back-patchable fix was found, so it
lands master-only with a catversion bump `[from-commit-msg]`. Touches
`[[knowledge/files/src/backend/utils/cache/relcache.c.md]]`,
`[[knowledge/files/src/backend/utils/cache/catcache.c.md]]`, and
`[[knowledge/files/src/include/catalog/pg_parameter_acl.h.md]]` — the relcache
doc's "nailed / critical shared relations" inventory and the catcache doc's
`IndexScanOK` pre-relcache list both need the new entries; the
`NUM_CRITICAL_SHARED_{RELS,INDEXES}` counts (now 6 / 8) are exactly the kind of
magic constant a corpus doc tends to quote and go stale on.

### 2. `2af70e937478` — Fix variadic pg_get_*_ddl() declarations (Tom Lane / Chao Li)

An ordinary VARIADIC function's last parameter must be an array type. The
recently-added `pg_get_role_ddl()`, `pg_get_tablespace_ddl()` (both oid and
name variants), and `pg_get_database_ddl()` declared their final `VARIADIC
options` arg as `text` rather than `text[]`. In `pg_proc.dat` this shows as
`proargtypes => '… text'` / `proallargtypes => '{…,text}'` corrected to
`_text` / `{…,_text}` (the `_text` = `text[]` array type) `[from-diff]`.

The reason it slipped through: the `opr_sanity` guard query that should flag a
VARIADIC function whose final arg isn't the array of `provariadic` used a bare
`!= provariadic` comparison, which yields NULL (not TRUE) when the CASE arm
returns NULL — so offending rows never appeared. Fixed to
`IS DISTINCT FROM provariadic` in both `opr_sanity.sql` and the `.out`
(~`:360` / `:487`) `[from-diff]`. catversion bumped for the pg_proc change.
The catalog path `src/include/catalog/pg_proc.dat` has no per-file corpus doc,
but `[[knowledge/idioms/fmgr-and-spi.md]]` (VARIADIC declaration rules) and
any `catalog-conventions` material on `provariadic` should note the
array-type requirement and the now-fixed sanity check.

### 3. `049b742daad0` — psql: tighten BEGIN/END heuristic in CREATE SCHEMA (Tom Lane / Chao Li)

Since `d51697484`, psql's lexer treats `CREATE SCHEMA` as a statement that may
contain SQL-standard routine bodies (`BEGIN ATOMIC … END`), counting BEGIN/END
so embedded semicolons don't terminate the command early. But it counted
BEGIN/END *anywhere* in the CREATE SCHEMA, so legal input like

```sql
CREATE SCHEMA s CREATE VIEW begin AS SELECT 1;
```

was mis-tracked (the identifier `begin` bumped `begin_depth`) `[from-commit-msg]`.

The rewrite moves the heuristic out of the `{identifier}` flex rule into new
static helpers in `fe_utils/psqlscan.l` (~`:958`): `psqlscan_record_initial_keyword()`,
`psqlscan_is_create_routine()`, and `psqlscan_track_identifier()` `[from-diff]`.
It now keeps two keyword-prefix buffers — `init_idents` for the whole statement
and `sub_idents` reset at each top-level `CREATE` keyword inside a
`CREATE SCHEMA` — and only counts BEGIN/CASE/END when the active (sub)clause
matches `CREATE [OR REPLACE] {FUNCTION|PROCEDURE}` `[from-diff]`. The
`PsqlScanStateData` struct in `include/fe_utils/psqlscan_int.h` (~`:117`)
renames `identifier_count`/`identifiers[4]` to `init_idents_count`/
`init_idents[4]` and adds `sub_idents_count`/`sub_idents[4]` `[from-diff]`.
Secondary fix: `\;` now resets BEGIN/END tracking only at the outer level
(paren_depth == 0 && begin_depth == 0), matching plain `;` `[from-diff]`.
Touches `[[knowledge/files/src/include/fe_utils/psqlscan_int.h.md]]` — the
struct-field inventory there is now stale (renamed + two new fields).

### 4. `2a7e95b659df` — Readable identity strings for property graph objects (Peter Eisentraut / Ashutosh Bapat)

`pg_identify_object()`'s identity column for SQL/PGQ property-graph objects
produced ambiguous chains like `a of l of e of g` where the intermediate
object types were invisible. `getObjectIdentityParts()` in
`catalog/objectaddress.c` now interpolates the object type into the `of`
connectors at five sites (~`:6175`–`:6292`): property-graph element / label /
property get `%s of property graph `; element-label gets `%s of element `;
label-property gets `%s of label ` `[from-diff]`. So the identity reads e.g.
`a of label v1 of element v1 of property graph create_property_graph_tests.gt`
`[from-diff]`. Pure presentation change (and the matching `.out`); follows the
existing precedent set by user-mapping / role-membership identity strings
`[from-commit-msg]`. Touches
`[[knowledge/files/src/backend/catalog/objectaddress.c.md]]` — if that doc
enumerates the `getObjectIdentityParts` per-class format strings, the five
property-graph arms changed.

### 5. `ef01ca6dbca5` + `80bb0ebcc11f` — the #19527 double-free pair (Tom Lane / Yuelin Wang)

These two land together and are best read as one fix at two layers.

**`80bb0ebcc11f` (the local bug).** `contrib/pgcrypto/openssl.c`'s
`ResOwnerReleaseOSSLCipher()` (~`:834`) freed the `OSSLCipher` without first
clearing `cipher->owner` `[from-diff]`. Every other ReleaseResource callback
(including the sibling `OSSLDigest` one) nulls the back-pointer first; omitting
it meant `free_openssl_cipher()` re-entered `ResourceOwnerForget()` on an owner
that was mid-release → a recursive forget and double-free. Oversight in
`cd694f60d`. Fix: `cipher->owner = NULL;` before the free `[from-diff]`.

**`ef01ca6dbca5` (the structural backstop).** Even with the pgcrypto fix, the
crash showed that `ResourceOwnerReleaseAll()` in `utils/resowner/resowner.c`
(~`:347`) had an unsafe ordering: it called `kind->ReleaseResource(value)`
*before* decrementing the stored item count, so if `ReleaseResource` threw,
abort cleanup re-entered and tried to release the same item again `[from-diff]`.
The fix tracks `using_arr` and updates `owner->narr` / `owner->nhash` to forget
the item *before* invoking `ReleaseResource` (~`:398`), matching the ordering of
`ResourceOwnerReleaseAllOfKind` and restoring the pre-`b8bff07da` behavior —
"prefer to leak the item than crash" `[from-diff]`. Backpatch to 17.

This pair is a good reentrancy case study for the resowner machinery. Touches
`[[knowledge/files/src/backend/utils/resowner/resowner.c.md]]` — the release-loop
ordering invariant (forget-before-release) is exactly the kind of subtle
contract that belongs in the doc's "gotchas / invariants" section; the
`openssl.c` callback has no per-file doc but is a textbook instance of the
"every ResOwnerRelease* must null the back-pointer before freeing" idiom.
