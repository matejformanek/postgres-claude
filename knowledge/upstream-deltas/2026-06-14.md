### `0131e8f` — postgres_fdw: fix the `RemoteAttributeMapping` qsort comparator

**Etsuro Fujita, 2026-06-14T07:00:00Z.** Touches a corpus-tracked file
(`contrib/postgres_fdw/postgres_fdw.c`) → **anchor NOT bumped** (recipe
step 9).

One-line fix in `remattrmap_cmp()`, the comparator passed to the qsort over
the `RemoteAttributeMapping` array:

```c
-	return strncmp(r1->remote_attname, r2->remote_attname, NAMEDATALEN);
+	return strcmp(r1->remote_attname, r2->remote_attname);
```

What changed and why it matters:

- This is a follow-up to **`aa1f93a33`** ("postgres_fdw: Replace buffers in
  `RemoteAttributeMapping` with pointers", 2026-05-16). Before that commit,
  `remote_attname` was an *inline fixed `NAMEDATALEN` buffer*, so bounding
  the compare with `strncmp(..., NAMEDATALEN)` was a defensible (if
  redundant) way to compare two embedded name fields. After `aa1f93a33`
  the field became a **pointer to a NUL-terminated string** of arbitrary
  length, so `strncmp(..., NAMEDATALEN)` is now an actual bug: it would
  treat two remote attribute names that share their first `NAMEDATALEN`
  (64) bytes but differ later as *equal*, corrupting the sort order (and
  therefore any subsequent `bsearch`/ordering assumption over the mapping).
- The fix is the textbook idiom: once a name is a real C string, compare it
  with `strcmp`, not a length-bounded `strncmp`. Identifiers in PG can be
  truncated to `NAMEDATALEN-1` on the *local* side, but a remote column
  name arriving over the wire is not bound by the local `NAMEDATALEN`, so
  the unbounded compare is the correct contract here.
- Severity: low blast radius (a comparator returning 0 for non-equal keys
  only matters when two remote attnames collide in their first 64 bytes,
  which is rare in practice), but it is a correctness bug, not cosmetic —
  hence the prompt same-day follow-up.

**Corpus impact:** the `RemoteAttributeMapping` struct and `remattrmap_cmp`
are *newer than* our `postgres_fdw.c.md` per-file doc — neither
`RemoteAttributeMapping`, `remattrmap_cmp`, nor `NAMEDATALEN` appears in
`[[knowledge/files/contrib/postgres_fdw/postgres_fdw.c.md]]` or
`[[knowledge/subsystems/contrib-postgres_fdw.md]]`. So there is no stale
claim to fix, but there *is* a coverage gap: the post-`aa1f93a33` remote
attribute-mapping machinery (introduced 2026-05-16) is undocumented in the
corpus. Flagged in the refresh report for a backfill pass.

Idiom angle (`[[knowledge/idioms/...]]`): this is a clean instance of the
"string field type changed from inline buffer to pointer ⇒ audit every
`strncmp(..., SIZEOF_BUFFER)` on it" refactor hazard. Worth a one-liner in
a C-idioms note on name comparison if/when one exists.
