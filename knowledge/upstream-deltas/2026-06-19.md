## Deep dives

### 1. `dc5116780846` — Update JIT tuple deforming code for virtual generated columns (David Rowley)

**The bug.** `llvmjit_deform.c` JIT-compiles a per-`TupleDesc` heap-tuple
deformer. Two optimizations in that codegen assume a column declared `NOT NULL`
is physically present and non-NULL in every tuple:

1. **`guaranteed_column_number`** (the rightmost attno certain to exist in every
   tuple) lets the emitted function skip the `natts >= attnum+1` bounds branch up
   to that index. Pre-fix it was set whenever
   `attnullability == ATTNULLABLE_VALID && !atthasmissing && !attisdropped`
   ([[corpus]] `knowledge/files/src/backend/jit/llvm/llvmjit_deform.c.md`
   documents this at `:82, 110-129`).
2. The per-column **null-check elision** (`if (att->attnullability !=
   ATTNULLABLE_VALID)` → only then emit a null-bitmap test) and the **alignment
   fast-path** ("after a NOT NULL fixed-width column whose length is a multiple
   of its alignment, the next column is aligned") both lean on the same
   NOT-NULL-implies-present-and-non-NULL premise.

VIRTUAL generated columns (introduced for PG18 by `83ea6c540`, named in the
commit message as the commit that "missed updating this code") break that
premise: they are **stored as NULL in the tuple** yet may be declared `NOT
NULL`. So a NOT-NULL virtual column could (a) make the JIT think more columns are
guaranteed present than really are, and (b) make it skip the null check for a slot
that is in fact NULL, and (c) miscompute alignment for the following column.

**The fix** (3 guard sites, all adding the virtual-generated test alongside the
existing `CompactAttribute` checks; the commit now fetches the full
`Form_pg_attribute attr = TupleDescAttr(desc, attnum)` so `attgenerated` is in
scope, since `CompactAttribute` doesn't carry it):

- `guaranteed_column_number` assignment — now also requires
  `attr->attgenerated != ATTRIBUTE_GENERATED_VIRTUAL`. The comment is reworded to
  "columns that have to exist **in all tuples**" and explicitly likens virtual
  generated columns to `atthasmissing` columns (could be absent, or present-as-
  NULL).
- Null-check emission — now `if (att->attnullability != ATTNULLABLE_VALID ||
  attr->attgenerated == ATTRIBUTE_GENERATED_VIRTUAL)`, i.e. **always** null-check
  a virtual generated column.
- Alignment fast-path — the "next column is aligned" inference is now suppressed
  for virtual generated columns
  (`... && attr->attgenerated != ATTRIBUTE_GENERATED_VIRTUAL && (att->attlen %
  alignto) == 0`).

**Why it matters to the corpus.** This is the exact `guaranteed_column_number`
machinery our per-file doc walks through, and the doc's description of the three
fast-paths is now one anchor behind. The deform JIT mirrors the interpreted
`slot_deform_heap_tuple()` in `heaptuple.c`; the interpreter already handled
virtual columns correctly (they go through the normal null-bitmap path), so this
is purely the JIT side catching up. The `[[corpus]]` cross-ref and required
refresh are filed in `progress/refresh-2026-06-19.md`.

### 2. `f25a07b2d94c` — Make pg_mkdir_p() tolerant of a concurrent directory creation (Andrew Dunstan, co-author Tom Lane)

**The bug.** `pg_mkdir_p()` (the `mkdir -p` engine behind `initdb`, tablespace
creation, etc.) walked each path component as **`stat()` first, then `mkdir()`
only if `stat()` reported the component absent** ([[corpus]]
`knowledge/files/src/port/pgmkdirp.c.md` documents this exact order in its
"Component walk" landmark, `:109-142`). That open window is a TOCTOU race: if
another process creates the component between this process's `stat()` and
`mkdir()`, the `mkdir()` returns `EEXIST` and the old code treated it as a hard
error — unlike POSIX `mkdir -p`, which is meant to be idempotent. The reported
trigger is **parallel `initdb` runs whose data directories share a common
temporary ancestor**: one process wins the race to create the shared ancestor,
the others die with `could not create directory "...": File exists`.

**The fix.** Invert the order to **`mkdir()` first, and only `stat()` on
`EEXIST`** to confirm the existing thing is a directory:

- Non-WIN32: on `mkdir()` failure, save `errno`; the component is acceptable only
  if `save_errno == EEXIST && stat(path, &sb) == 0 && S_ISDIR(sb.st_mode)`.
  Otherwise restore `save_errno` (the comment is explicit: *"Don't let stat
  replace mkdir's errno"*) and fail.
- WIN32: `stat()` opens a handle and participates in share-mode locking, so it
  can **transiently fail on a directory another process is concurrently
  creating**. The fix probes with `GetFileAttributes()` instead — it requests
  only `FILE_READ_ATTRIBUTES`, is exempt from share-mode denial, and reliably
  sees a concurrently-created dir. Acceptable iff `errno == EEXIST && attr !=
  INVALID_FILE_ATTRIBUTES && (attr & FILE_ATTRIBUTE_DIRECTORY)`.
- The top-of-function `struct stat sb;` is removed; `sb` is now declared locally
  inside the non-WIN32 EEXIST branch. The umask dance and WIN32 prefix-skip are
  untouched.

The commit message also notes Tom back-patched `039f7ee0f`'s effects on this
function so `pgmkdirp.c` stays byte-identical across all live branches.
Backpatch-through 14.

**Why it matters to the corpus.** Our per-file doc's "Component walk" landmark
and its "Invariants & gotchas" both describe the now-removed stat-first ordering
and the `EEXIST`/`ENOTDIR` errno assignment that the new code restructures. The
doc is two behaviors stale (and still anchored at the very old
`4b0bf0788b0`). Refresh filed in `progress/refresh-2026-06-19.md`.

Relevant skill: `coding-style` (the WIN32 `#ifdef` split + errno-preservation
idiom). Confidence: `[from-diff]`.
