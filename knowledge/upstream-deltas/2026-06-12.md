## Deep dives

### 1. `a8ee70bd5e00` ‚Äî Fix second race with timeline selection during promotion (Michael Paquier)

A WAL-read timeline-selection race in `read_local_xlog_page_guts()`
(`access/transam/xlogutils.c`), the page-read callback used by in-process WAL
readers (logical decoding, `pg_walinspect`, etc.). On the recovery branch the
callback computed the read horizon with `read_upto = GetXLogReplayRecPtr(&currTLI)`
and then read on `currTLI`. During promotion there is a window where recovery
has effectively ended and a new **insertion** timeline has been established, but
`GetXLogReplayRecPtr`'s reported replay TLI still reflects the *old* timeline ‚Äî
so the reader selects WAL on the wrong timeline. `[from-diff]`

**The fix** adds, inside the recovery branch, a check that mirrors the existing
logic in `logical_read_xlog_page()`:

```c
TimeLineID  insertTLI;
read_upto = GetXLogReplayRecPtr(&currTLI);
/* If the insertion timeline has already been set, use it. */
insertTLI = GetWALInsertionTimeLineIfSet();
if (insertTLI != 0)
    currTLI = insertTLI;
```

`GetWALInsertionTimeLineIfSet()` returns 0 until the insertion TLI is durably
established at end-of-recovery, so the override only kicks in once promotion has
fixed the new timeline ‚Äî exactly the window the old code mis-handled. The commit
subject calls this the "second" race, i.e. a sibling of an earlier promotion
timeline-selection fix in the same area. `[from-diff]`

The TAP coverage in `recovery/t/035_standby_logical_decoding.pl` is widened to
exercise the **SQL** decoding interface across the switch: it opens a
`background_psql` session on the cascading standby *before* promotion, creates a
second slot `race_slot_sql`, and after the injection-point-driven promotion
asserts `pg_logical_slot_peek_changes('race_slot_sql', ‚Ä¶)` returns the expected
rows ‚Äî previously only the `pg_recvlogical` (walsender) path was checked.

Corpus impact: refresh
[[knowledge/files/src/backend/access/transam/xlogutils.c.md]] ‚Äî the
`read_local_xlog_page_guts` recovery branch now consults
`GetWALInsertionTimeLineIfSet()` and may override `currTLI`. Worth a cross-link
from [[knowledge/subsystems/replication.md]] (logical decoding on standbys across
promotion) and any `wal-and-xlog` synthesis noting that in-process WAL readers
must prefer the insertion TLI once set. See
`source/src/backend/access/transam/xlogutils.c` `read_local_xlog_page_guts`,
and `logical_read_xlog_page` for the pattern it copies. `[from-diff]`

### 2. `0e47bb5fbeec` ‚Äî Fix out-of-bounds write in RI fast-path batch on re-entry (Amit Langote)

A memory-safety fix in the **PG19 referential-integrity (RI) fast-path
batching** machinery (`utils/adt/ri_triggers.c`). The feature buffers FK-check
rows in a fixed array `RI_FastPathEntry.batch[RI_FASTPATH_BATCH_SIZE]`
(`RI_FASTPATH_BATCH_SIZE == 64`, per the regress test's `generate_series(1, 64)`)
and probes them in one shot when the batch fills. `[from-diff]`

**The bug.** `ri_FastPathBatchFlush()` ran user-defined cast and equality
functions while the batch was still full (`batch_count` was reset to 0 only
*after* the flush returned). If one of those user functions performed DML on the
**same** FK table, the after-trigger fired re-entrantly and `ri_FastPathBatchAdd`
wrote `batch[batch_count++]` with `batch_count` already at
`RI_FASTPATH_BATCH_SIZE` ‚Äî a write one past the end of the array. `[from-diff]`

**The fix** has three coupled parts:

- A new per-entry `bool flushing` flag on `RI_FastPathEntry`. `ri_FastPathBatchAdd`
  now short-circuits when it is set:
  `if (unlikely(fpentry->flushing)) { ri_FastPathCheck(...); return; }` ‚Äî the
  re-entrant row takes the per-row path instead of touching the mid-flush array.
- A defensive bounds check around the array write
  (`if (fpentry->batch_count < RI_FASTPATH_BATCH_SIZE) ‚Ä¶ else elog(ERROR, "RI
  fast-path batch unexpectedly full")`), and the full-batch flush is triggered on
  `== RI_FASTPATH_BATCH_SIZE`.
- `ri_FastPathBatchFlush` wraps the probe in `PG_TRY`/`PG_FINALLY` that sets
  `flushing = true` before and resets `flushing = false; batch_count = 0` after ‚Äî
  **even on error**, so an entry whose flush threw a (savepoint-caught) violation
  is left empty and reusable rather than retaining stale rows. The old
  reset-after-flush (`batch_count = 0` at function end) is removed.
- `ri_FastPathGetEntry` initialises `entry->flushing = false; entry->batch_count
  = 0` on entry creation.

Two regress scenarios are added: a cast function that re-enters with DML during a
full-batch flush (must not corrupt the array), and a savepoint-caught violation
followed by reuse of the same FK in the same transaction (the entry must be empty
‚Äî no stale `999`, no overflow).

Corpus impact: refresh
[[knowledge/files/src/backend/utils/adt/ri_triggers.c.md]] ‚Äî the
`RI_FastPathEntry` struct gains `flushing`; `ri_FastPathBatchAdd`/`Flush`/
`GetEntry` change shape. This is the **first** of a coordinated two-commit pair
the same night (companion `4113873a` below); document them together. See
`source/src/backend/utils/adt/ri_triggers.c` `ri_FastPathBatchAdd`,
`ri_FastPathBatchFlush`, `ri_FastPathGetEntry`. `[from-diff]`

### 3. `4113873a5ab0` ‚Äî Confine RI fast-path batching to the top transaction level (Amit Langote)

The companion correctness fix to #2, same file, landed ~30 min later. Where #2
stopped the re-entrant **OOB write**, this one stops a re-entrant / subxact path
from **dropping FK checks entirely**. `[from-diff]`

**The subxact problem.** The batch cache lived for the whole transaction and was
torn down on subxact abort by `ri_FastPathSubXactCallback` (which NULLed the
static cache pointer). But rows buffered *earlier in the same statement* and not
yet flushed were silently discarded when an after-trigger aborted a
subtransaction ‚Äî so a genuine FK violation among those buffered rows went
**undetected**. The batch cannot be cleanly unwound on subxact abort, so the fix
is to **not batch inside a subtransaction at all**:

```c
if (AfterTriggerIsActive() &&
    GetCurrentTransactionNestLevel() == 1 &&
    !ri_fastpath_flushing)
    ri_FastPathBatchAdd(riinfo, fk_rel, newslot);   /* batched path */
else
    ri_FastPathCheck(riinfo, fk_rel, newslot);      /* per-row path */
```

The per-row path now covers three cases (documented in a new comment): ALTER
TABLE validation, any FK check inside a subtransaction, and a re-entrant check
from user cast/operator code during a flush. `[from-diff]`

**Supporting changes:**

- A new file-static `bool ri_fastpath_flushing`. `ri_FastPathEndBatch` sets it
  for the duration of its `hash_seq_search` scan, wrapped in `PG_TRY`/`PG_FINALLY`
  (reset in finally), so a flush that re-enters with an FK check cannot add a new
  entry into the hash table being iterated (which could land in an
  already-scanned bucket and be torn down unflushed).
- `ri_FastPathSubXactCallback` and its `RegisterSubXactCallback` are **deleted**
  entirely ‚Äî with batching confined to the top level there is no subxact state to
  unwind, so the abort path is handled solely by the ResourceOwner and the
  XactCallback. `ri_FastPathXactCallback` additionally clears
  `ri_fastpath_flushing` defensively at transaction end.

Regress: an after-trigger that aborts a subxact (`1/0` caught by an inner
`BEGIN‚Ä¶EXCEPTION`) on a statement inserting a violating row (`999`) now correctly
raises the FK violation rather than swallowing it.

Corpus impact: refresh
[[knowledge/files/src/backend/utils/adt/ri_triggers.c.md]] together with #2 ‚Äî
the batching applicability predicate (`RI_FKey_check`), the removed
`SubXactCallback`, and the `ri_fastpath_flushing` re-entrancy guard. Lock/
callback note for any RI synthesis: the fast-path now registers **only** an Xact
callback (no SubXact), and batching is a **top-transaction-level-only**
optimisation. See `source/src/backend/utils/adt/ri_triggers.c` `RI_FKey_check`,
`ri_FastPathEndBatch`, `ri_FastPathXactCallback`. `[from-diff]`

### 4. `8bf257aebac1` ‚Äî Fix handling of namespace nodes in xpath() (Michael Paquier)

A robustness fix in `xml_xmlnodetoxmltype()` (`utils/adt/xml.c`), which converts
a libxml node returned by an XPath evaluation into a PostgreSQL `xmltype`. An
XPath expression can yield **namespace nodes** (e.g.
`xpath('//namespace::foo', '<root xmlns:foo="http://127.0.0.1"/>')`). libxml
represents those as `xmlNsPtr`, a struct with a *different layout* from
`xmlNodePtr`, but the XPath nodeset hands them back aliased as `xmlNodePtr`.
The old condition only special-cased `XML_ATTRIBUTE_NODE` and `XML_TEXT_NODE`;
a namespace node fell through to the general `xmlNodeDump`/buffer path, which
interprets the aliased `xmlNsPtr` as a full node ‚Äî undefined behaviour. `[from-diff]`

**The fix** is one added disjunct:

```c
if (cur->type != XML_ATTRIBUTE_NODE &&
    cur->type != XML_TEXT_NODE &&
    cur->type != XML_NAMESPACE_DECL)
{ /* general node-serialisation path */ }
```

so namespace-decl nodes take the simple text path and serialise to their URI
string (the test expects `{http://127.0.0.1}`). `[from-diff]`

The `xml_2.out` expected-output variant catch-up landed separately as
`0b74df66f0` (this window), and the buildfarm caught the intervening gap ‚Äî see
`knowledge/buildfarm-lessons/2026-06-12.md` (massasauga `xml` regression at
`a8ee70b`).

Corpus impact: refresh
[[knowledge/files/src/backend/utils/adt/xml.c.md]] ‚Äî `xml_xmlnodetoxmltype` now
filters `XML_NAMESPACE_DECL` alongside attribute/text nodes. See
`source/src/backend/utils/adt/xml.c` `xml_xmlnodetoxmltype`. `[from-diff]`

### 5. `389bd4c5b934` ‚Äî amcheck: Fix missing allequalimage corruption report (Fujii Masao)

A false-negative in `bt_index_check_callback()`
(`contrib/amcheck/verify_nbtree.c`). When a btree metapage claims
`allequalimage` (deduplication is safe) but the index actually uses an opclass
for which dedup is *not* safe, amcheck must report corruption. The reporting
`ereport(ERROR, ‚Ä¶)` was placed **inside** the loop scanning
`indrel->rd_opfamily[i]` for `INTERVAL_BTREE_FAM_OID`, so it only fired on the
specific interval-opclass case ‚Äî any *other* incorrectly-`allequalimage` index
finished the loop and was reported **clean**. `[from-diff]`

**The fix** moves the report out of the loop: the loop now just sets
`has_interval_ops = true` and `break`s; the `ereport(ERROR, ‚Ä¶)` runs
unconditionally after the loop, keeping the interval-specific `errhint` ("known
of \"interval\" indexes last built on a version predating 2023-11.") gated on
`has_interval_ops`:

```c
for (...) if (... == INTERVAL_BTREE_FAM_OID) { has_interval_ops = true; break; }
ereport(ERROR,
        (errcode(ERRCODE_INDEX_CORRUPTED),
         errmsg("index \"%s\" metapage incorrectly indicates that deduplication is safe", ...),
         has_interval_ops ? errhint(...) : 0));
```

Net effect: amcheck now flags **all** bogus-`allequalimage` indexes, not only the
interval-opclass subset. `[from-diff]`

Corpus impact: refresh [[knowledge/files/contrib/amcheck/verify_nbtree.md]] ‚Äî
the corruption report in `bt_index_check_callback` is now unconditional after the
opfamily scan. See `source/contrib/amcheck/verify_nbtree.c`
`bt_index_check_callback`. `[from-diff]`

### 6. `44196fd4f378` ‚Äî Fix md5_password_warnings for role and database settings (Fujii Masao)

A GUC-timing fix spanning `libpq/crypt.c`, `utils/init/postinit.c`, and the
`miscadmin.h` header. The `md5_password_warnings` GUC gates a deprecation
WARNING emitted when a client authenticates with MD5. The warning text was
*decided* at authentication time in `md5_crypt_verify()` ‚Äî which runs **before**
per-role / per-database GUC settings (`ALTER ROLE ‚Ä¶ SET md5_password_warnings =
off`) are applied ‚Äî so a per-role override was ignored. `[from-diff]`

**The fix** turns the connection-warning store into a **deferred-filter**
mechanism so the GUC is read at *emit* time (after role/db settings land):

- `miscadmin.h` adds `typedef bool (*ConnectionWarningFilter) (void);` and
  `StoreConnectionWarning` gains a third `ConnectionWarningFilter filter`
  parameter (an **extension-visible signature change** ‚Äî note for ABI).
- `postinit.c` replaces the two parallel lists
  (`ConnectionWarningMessages`/`Details`) with a single `List *ConnectionWarnings`
  of a new `ConnectionWarning { char *message; char *detail;
  ConnectionWarningFilter filter; }` struct (added to `typedefs.list`).
  `EmitConnectionWarnings()` ‚Äî called *after* startup + role/db GUCs are applied
  ‚Äî now does `if (warning->filter == NULL || warning->filter()) ereport(WARNING,
  ‚Ä¶)` and `pfree`s each field (the store now takes ownership of the strings).
  The loop switches from `forboth` over two lists to `foreach_ptr`.
- `crypt.c`: `md5_crypt_verify` now *always* stores the MD5 warning (dropping its
  own `if (md5_password_warnings)` gate) and passes a new
  `md5_password_warning_enabled()` callback (`return md5_password_warnings;`) as
  the filter, so the decision is made at emit time against the final GUC value.
  The pre-existing `StoreConnectionWarning(warning, detail)` call for password
  expiry passes `NULL` (unconditional). `[from-diff]`

TAP coverage (`authentication/t/001_password.pl`) adds a role with
`md5_password_warnings = off` and asserts it connects with no warning while still
authenticating via md5.
