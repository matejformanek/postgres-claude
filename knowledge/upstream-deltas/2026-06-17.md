## Deep dives

### `4e5920e6` — parser: reject RESPECT/IGNORE NULLS on every non-window function

**File:** `src/backend/parser/parse_func.c`, `ParseFuncOrColumn()` ~L356
(@`4e5920e6`). Touches
[[knowledge/files/src/backend/parser/parse_func.c.md]].

`RESPECT NULLS` / `IGNORE NULLS` ("null treatment") is, per SQL spec, only legal
on window functions. The old code only rejected it inside the **aggregate**
branch (the `FUNCDETAIL_AGGREGATE` arm carried an `ignore_nulls !=
NO_NULLTREATMENT` → error). That left other `fdresult` cases — a plain function,
a coercion, a column projection — to silently *accept and ignore* a stray
`RESPECT/IGNORE NULLS`. The fix hoists the check to a single early gate right
after the `fdresult` is determined: if `fdresult != FUNCDETAIL_WINDOWFUNC &&
ignore_nulls != NO_NULLTREATMENT`, raise `ERRCODE_WRONG_OBJECT_TYPE` ("Only
window functions accept RESPECT/IGNORE NULLS"), and the now-redundant aggregate
-branch copy is deleted. The regress delta (`window.out`) shows the message text
changing from "aggregate functions do not accept …" to the unified "Only window
functions accept …". Net: a parser-completeness fix — the grammar allowed the
clause syntactically; the analysis pass now rejects it uniformly. (See
`ab3023ad` immediately after, which fixes a `TREATEMENT` typo in the new
comment.) Behavior change in error wording; no plan/exec impact.

### `68ace967` — RI fast path: look through domains to the base type before casting

**File:** `src/backend/utils/adt/ri_triggers.c`, `ri_HashCompareOp()` ~L4113
(@`68ace967`). Touches
[[knowledge/files/src/backend/utils/adt/ri_triggers.c.md]].

The referential-integrity fast path uses pre-computed equality operators
(`pf_eq_oprs`) that can be **cross-type** when the FK and PK columns differ —
e.g. `int48eq` for an `int4` PK and an `int8` FK. To decide whether the FK datum
needs a cast before being fed to the operator's right-hand input, the code
compared the FK column's `typeid` against the operator's expected `righttype`,
taking the no-cast path when equal. The bug: when the FK column is a **domain**
(`CREATE DOMAIN fp_int8dom AS int8`), `typeid` is the *domain* OID, which never
equals the operator's base-type `righttype`, so the code fell into the cast
branch and then failed with "no conversion function from … to …" — wrongly
rejecting **valid** rows. The one-line fix wraps the comparison in
`getBaseType()`: `if (getBaseType(typeid) == righttype)` takes the no-cast path,
because a domain is binary-coercible to its base type. Regress adds a
domain-over-`int8` FK referencing an `int4` PK and asserts both the satisfied and
the violated/NULL cases behave. Genuine correctness fix (valid data was being
rejected); back-patch candidate to wherever the RI fast path landed.

### `1f7dfe8c` — regress: tuple-deformation test for virtual generated columns (turned the buildfarm red)

**Files:** `src/test/regress/sql/generated_virtual.sql` +
`expected/generated_virtual.out` (and the `generated_stored` pair), test-only
(@`1f7dfe8c`). Touches the generated-columns test surface; no per-file corpus doc.

David Rowley added a regress case that builds tables mixing VIRTUAL generated
columns with regular NOT NULL columns and reads them back, specifically to
exercise the **tuple-deformation** path for virtual generated columns (PG 18+
feature). The commit itself is test-only and *passed for the committer*, but on
several Debian-Sid buildfarm animals (`urutu` clang16, `taipan` gcc13,
`bushmaster` clang17, `canebrake` gcc13) the new `generated_virtual` test
(test #159) went red with a **wrong value for a trailing real column**:

```
 CREATE TABLE gtest21c (a int NOT NULL, b int GENERATED ALWAYS AS (a * 2) VIRTUAL NOT NULL, c int NOT NULL);
 INSERT INTO gtest21c (a, c) VALUES (10, 42);
 SELECT a, b, c FROM gtest21c;
- 10 | 20 | 42      (expected)
+ 10 | 20 |  0      (actual on red animals)
```

i.e. column `c` deforms to `0` instead of `42` when a VIRTUAL generated column
`b` sits before it. This is a **real, nondeterministic correctness bug in
virtual-generated-column tuple deformation**, not a flaky test — the test simply
made an existing latent bug observable. See
`knowledge/buildfarm-lessons/2026-06-17.md` §1 for the full failure analysis.
Flagged as the highest-signal item of the day. [verified-by-code: buildfarm
regression diff on urutu/taipan/bushmaster]

### `3048e813` — WAL I/O stats: don't count failed/short writes and reads as completed I/O

**Files:** `src/backend/access/transam/xlog.c` (`XLogWrite` ~L2455,
`XLogFileInitInternal` ~L3331), `src/backend/access/transam/xlogreader.c`
(`WALRead` ~L1597), `src/backend/access/transam/xlogrecovery.c` (`XLogPageRead`
~L3390), `src/backend/replication/walreceiver.c` (`XLogWalRcvWrite` ~L954)
(@`3048e813`). Touches
[[knowledge/files/src/backend/access/transam/xlog.c.md]],
[[knowledge/files/src/backend/access/transam/xlogreader.c.md]],
[[knowledge/files/src/backend/access/transam/xlogrecovery.c.md]],
[[knowledge/files/src/backend/replication/walreceiver.c.md]].

The `pg_stat_io` WAL counters are fed by `pgstat_count_io_op_time(IOOBJECT_WAL,
…, start, 1, nbytes)`. In five call sites the count was emitted **immediately
after the `pg_pwrite`/`pg_pread`**, i.e. *before* the code checked whether the
syscall actually succeeded. On a failed write (`written <= 0`) or a short/zero
read the counter was nonetheless bumped — and worse, with the (possibly negative)
`written`/`readbytes` byte count — feeding garbage into the I/O timing/volume
stats before the subsequent `ereport(ERROR)` even fired. The fix uniformly
**moves the `pgstat_count_io_op_time()` call to after the success check** in
`XLogWrite`, `XLogFileInitInternal`, `WALRead`, and `XLogWalRcvWrite`; in
`XLogPageRead` it guards the count with `if (r > 0)` ("Count I/O stats only for
successful short reads"). No functional WAL behavior change — purely accurate
`pg_stat_io` accounting on the error/short-I/O paths. Back-patch candidate to
wherever WAL `pg_stat_io` counting landed.

### `d2cea630` — vacuumdb --missing-stats-only: detect partitioned-index expression stats

**File:** `src/bin/scripts/vacuuming.c`, `retrieve_objects()` catalog query
~L738 (@`d2cea630`); TAP `100_vacuumdb.pl`. No per-file corpus doc (`src/bin`).

`vacuumdb --analyze-only --missing-stats-only` builds a catalog query that finds
relations whose attributes lack pg_statistic rows, so it only analyzes what's
missing. For **partitioned indexes** with an expression key (e.g. `CREATE INDEX
ON parted ((a + 1))`) the query mis-handled the `stainherit` dimension: it
joined the not-exists subquery on `s.stainherit = p.inherited`, which let an
inherited-stats row mask a missing non-inherited row (or vice-versa), so a
partitioned index that genuinely lacked stats was skipped. The fix moves the
inheritance condition **out of the subquery** into the outer predicate as `AND
NOT p.inherited` and drops the `s.stainherit = p.inherited` correlation, so the
"missing" test keys only on `(starelid, staattnum)`. TAP adds the expression
index on a partitioned table and asserts vacuumdb now targets it. Client-tool
correctness fix; no backend change.

### `da82fbb8` — jsonb PL transforms: stack-depth guard + interrupt check on recursion

**Files:** `contrib/jsonb_plperl/jsonb_plperl.c` (`Jsonb_to_SV` ~L67,
`SV_to_JsonbValue` ~L183), `contrib/jsonb_plpython/jsonb_plpython.c`
(`PLyObject_FromJsonbContainer` ~L144, `PLyObject_ToJsonbValue` ~L414)
(@`da82fbb8`). No per-file corpus doc (contrib transform modules).

The `jsonb_plperl` / `jsonb_plpython` transform modules convert between `jsonb`
and the PL's native value type with **mutually-recursive** helpers (a jsonb
object recurses into its members; a Perl/Python container recurses into its
elements). None of the four recursion entry points called
`check_stack_depth()`, so a deeply-nested jsonb value — or a deeply-nested
Perl/Python structure — could drive the C stack to overflow (a crash, not a
clean error). Tom Lane adds `check_stack_depth()` at the top of all four
recursive converters (with `#include "miscadmin.h"`), so over-deep nesting raises
the normal "stack depth limit exceeded" `ERROR`. Additionally, the Perl
`SV_to_JsonbValue` dereferences `SvROK` references in a `while` loop;
**circular** Perl references would make that loop spin forever, so a
`CHECK_FOR_INTERRUPTS()` is added inside the deref loop to make it cancellable
(the comment judges true cycle-detection "more trouble than it's worth"). Robust
ness / DoS-hardening fix in maintained contrib transforms; back-patch candidate.
Relates to the [[knowledge/idioms/error-handling.md]] stack-depth idiom and the
`check_stack_depth` / `CHECK_FOR_INTERRUPTS` recursion-guard pattern.

### `4bd477dc` — libpq-oauth: log libcurl + SSL version under OAUTHDEBUG_UNSAFE_TRACE

**File:** `src/interfaces/libpq-oauth/oauth-curl.c` (`initialize_curl()` ~L2690,
`pg_start_oauthbearer()` ~L3079) (@`4bd477dc`). Touches
[[knowledge/files/src/interfaces/libpq-oauth/oauth-curl.c.md]].

A debuggability change to the client-side OAuth flow. `initialize_curl()`
already fetched `curl_version_info(CURLVERSION_NOW)` to verify thread-safety, but
only under `#if HAVE_THREADSAFE_CURL_GLOBAL_INIT`. The patch (1) **unconditionally
fetches** `info` (moving the `curl_version_info_data *info` declaration out of the
`#if`), and (2) threads a new `debug_flags` argument into `initialize_curl()` so
that, when `OAUTHDEBUG_UNSAFE_TRACE` is set, it prints `"[libpq] initialized
libcurl <version> (<ssl_version>)\n"` to stderr — with NULL fallbacks ("version
unknown" / "no SSL") rather than crashing on a degenerate install. The
`debug_flags` are now parsed once in `pg_start_oauthbearer()` via
`oauth_parse_debug_flags()` and passed both to `initialize_curl()` and into
`actx->debug_flags`. Pure diagnostics for network debugging; no behavior change
in the non-trace path. Part of the same-day OAuth-test cluster (`62222331`,
`fd5ea2e9`).
