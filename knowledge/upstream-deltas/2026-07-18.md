## Deep explainers

### #2 `1df9e8d969d6` тАФ remove_useless_result_rtes() outer-join PHV bug (Tom Lane)

A **wrong-results** planner bug, backpatched all the way to **16**. When the
planner considers deleting an `RTE_RESULT` join-tree node, `find_dependent_phvs()`
/ `find_dependent_phvs_in_jointree()` decide whether any `PlaceHolderVar` still
depends on that RTE by comparing the PHV's `phrels` against the singleton set
`{result_rt_index}` тАФ the reasoning being that if `phrels` names any *other* rel,
that other rel is a valid place to evaluate the PHV instead. But `phrels` was
redefined years ago to **also include outer-join relids**, so a stray OJ relid in
`phrels` could make the check believe there is somewhere else to evaluate the PHV
when there isn't, letting the planner drop an `RTE_RESULT` that leaves the PHV
with no valid evaluation site. No error is thrown; the plan silently fails to
substitute NULL for the PHV where an outer join requires it.

The fix threads a `Relids baserels` (the query's base-relation set) down through
`remove_useless_results_recurse` / `find_dependent_phvs[_in_jointree]` and
intersects `phrels` with `baserels` before the singleton comparison, so only
**base** relids count тАФ restoring the code to its original design intent
(`src/backend/optimizer/prep/prepjointree.c`, `find_dependent_phvs` and its
jointree walker, ~line 166 forward for the new signatures). Bug #19553, reported
by Viktor Leis; patch by Matheus Alcantara + Richard Guo.
`[[knowledge/files/src/backend/optimizer/prep/prepjointree.c.md]]` тАФ impacted;
the doc's description of the RESULT-RTE removal path and the
`find_dependent_phvs` helpers must note the new `baserels` parameter and the
base-relid-only semantics. `[verified-by-code]` from the diff.

### #3 `0316593146d5` тАФ restore `|` and `->` as prefix operators (Tom Lane)

Follow-up to `2f094e7ac`, which turned `|` and `->` from generic `Op` tokens into
**built-in operator names** with dedicated grammar productions. That commit added
infix productions and matched `Op`'s precedence, but **forgot the prefix (unary)
productions** тАФ so `SELECT | x` / `SELECT -> x` (used by at least one extension)
stopped parsing. The fix adds four productions in `src/backend/parser/gram.y` тАФ a
`RIGHT_ARROW a_expr` / `'|' a_expr` pair in `a_expr` and the same pair in
`b_expr` тАФ each building `makeSimpleA_Expr(AEXPR_OP, "->"/"|", NULL, $2, @1)`
(NULL left operand = prefix). Bug #19558, reported by Pierre Senellart, patch by
Pierre Forstmann. Backpatch-through **19**.
`[[knowledge/files/src/backend/parser/gram.y.md]]` тАФ impacted; if the doc
enumerates the `a_expr`/`b_expr` operator productions it should add the two prefix
rows. `[verified-by-code]`.

### #4 `03bcb6a7778e` тАФ REASSIGN OWNED for subscriptions in other databases (Jeff Davis / Dilip Kumar)

`pg_subscription` is a **shared** catalog (so the logical-rep launcher can scan
every database's subscriptions) but subscriptions are *conceptually database-
local*. Readers other than the launcher must therefore filter by `MyDatabaseId`.
Most already did; **REASSIGN OWNED did not**, so it could rewrite ownership of a
subscription living in a *different* database. The fix (a) adds explicit
`MyDatabaseId` guards in `src/backend/catalog/pg_subscription.c` (12 new lines) to
catch this class of bug going forward, (b) fixes the REASSIGN OWNED path in
`src/backend/commands/subscriptioncmds.c` (+14) to filter, and (c) clarifies the
`reassign_owned.sgml` / `user-manag.sgml` docs plus a new
`contrib/postgres_fdw/t/010_subscription.pl` cross-database regression case.
Reported by Noah Misch. Backpatch-through **19**.
`[[knowledge/files/src/backend/catalog/pg_subscription.c.md]]`,
`[[knowledge/files/src/backend/commands/subscriptioncmds.c.md]]` тАФ both impacted;
note the new "non-launcher readers must filter by MyDatabaseId" invariant.
`[verified-by-code]`.

### #6 `2cf212db5286` тАФ Generate unicode_limits.h (Jeff Davis)

Overflow-safety plumbing for **case mapping on 32-bit platforms**. Case mapping
(`upper`/`lower`/`initcap` under the builtin/ICU providers) can *expand* a UTF-8
string; to guarantee we never overflow a `size_t` while doing so we need the
maximum expansion factor. The commit computes that maximum during the
`update-unicode` table-generation step and emits a new generated header
`src/include/common/unicode_limits.h` (39 lines), then adds a `StaticAssertDecl`
that a `MaxAllocSize` text value expanded by that factor still has a length
fitting in `size_t` (`src/backend/utils/adt/pg_locale_builtin.c` +11, with
supporting edits in `src/common/unicode/generate-unicode_case_table.pl` +194/-71,
`src/common/unicode_case.c`, and the two `unicode_case_table.h` /
`unicode_limits.h` headers). Reported by Tom Lane; not backpatched (master-only
build-time change). `[[knowledge/files/src/backend/utils/adt/pg_locale_builtin.c.md]]`,

`[[knowledge/files/src/include/common/unicode_case_table.h.md]]` тАФ impacted; a new

### #10 `c9ea37231a2e` тАФ GiST index-only scan column alignment (Peter Geoghegan)

A **can't-happen-error / rare-crash** executor bug, backpatched to **14**. During
a GiST or SP-GiST index-only scan, the executor filled the result virtual slot by
deforming the `HeapTuple` returned in `scan->xs_hitup` using the *virtual slot's
own* tuple descriptor. But index AMs form that tuple with **their own** descriptor
`scan->xs_hitupdesc`, whose per-column alignment can disagree with the IoS slot's
descriptor. Reading the second-or-later column of a multicolumn GiST index could
then land at a mis-aligned offset тЖТ "can't happen" errors, and hard crashes were
possible though much less likely. The fix deforms `xs_hitup` with
`xs_hitupdesc` (the descriptor it was actually formed with), making the
`xs_hitup` path uniform with the neighbouring nbtree `xs_itup` handling
(`src/backend/executor/nodeIndexonlyscan.c`, ~113 lines reworked). The only
affected core opclass is GiST `range_ops`, and only when an earlier column left
the range datum at an odd offset (not the common unaligned-short-varlena case) тАФ
which is why it was found during table-AM-API testing, not in the field. Reviewed
by Tomas Vondra. `[[knowledge/files/src/backend/executor/nodeIndexonlyscan.c.md]]`
тАФ impacted; the `StoreIndexTuple`/`xs_hitup` handling note must say the tuple is
deformed with `xs_hitupdesc`. `[verified-by-code]`.

### #15 `1f414035135c` тАФ base64 vs base64url padding (Daniel Gustafsson)

`e1d917182` added **base64url** (URL/filename-safe base64, no `=` padding) but
unintentionally **relaxed the padding requirement for plain base64 too**: the
truncation logic in `pg_base64_decode_internal` accepted a 1- or 2-remainder
input (`pos == 2` / `pos == 3`) for *both* variants. The fix gates those two
branches on the `url` flag (`if (url && pos == 2)` / `else if (url && pos == 3)`)
so plain base64 again requires full `=` padding, restoring
`decode('AQ','base64')` to raise "invalid base64 end sequence"
(`src/backend/utils/adt/encode.c` ~line 583, plus a `strings.sql`/`.out`
regression row). Backpatch-through **19** (where base64url landed). Reviewed by
David E. Wheeler. `[[knowledge/files/src/backend/utils/adt/encode.c.md]]` тАФ
impacted; note the `url`-gated truncation rule. `[verified-by-code]`.

### #16 `32e4508db27d` тАФ pg_stat_io entries for data-checksum processes (Fujii Masao)

The data-checksums launcher/workers (`B_DATACHECKSUMSWORKER_LAUNCHER` /
`_WORKER`) were exposed in `pg_stat_io` with the same broad object├Чcontext matrix
as generic background workers, including combinations they can **never** perform
(bulkwrite, relation-init, temp relation, launcher vacuum). The fix teaches
`pgstat_tracks_io_object()` to whitelist only the real combinations тАФ WAL, plus
relation bulkread/normal for the launcher, plus relation vacuum for the worker
(it processes relations with a vacuum access strategy) тАФ and adds both backend
types to the `pgstat_tracks_io_op()` WAL-read exclusion list
(`src/backend/utils/activity/pgstat_io.c` +33, and a shrunk `stats.out` expected
set). Reviewed by Daniel Gustafsson. Backpatch-through **19**.
`[[knowledge/files/src/backend/utils/activity/pgstat_io.c.md]]` тАФ impacted; the
per-backend-type tracking table needs the two data-checksum rows.
`[verified-by-code]`.
