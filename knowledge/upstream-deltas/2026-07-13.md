## Deep explainers

### `7b87f08e8778` — Make blkreftable API use size_t/ssize_t consistently
[[knowledge/files/src/common/blkreftable.c.md]] ·
[[knowledge/files/src/include/common/blkreftable.h.md]] ·
[[knowledge/subsystems/backup-and-recovery]] (incremental backup)

The block-reference table is the on-disk structure that WAL summarization
writes and incremental backup reads to know which blocks changed since the
prior backup. Its serialization goes through a caller-supplied I/O callback
pair. Before this commit the read/write callback contract used `int length`
and returned `int`; the internal `BlockRefTableBuffer` tracked its fill with
`unsigned` `used`/`cursor` fields.

Changes (`source/src/common/blkreftable.c`):
- `struct BlockRefTableBuffer` (@173) — `used`/`cursor` (and the mirror in
  `struct BlockRefTableWriter` @223) move `unsigned` → `size_t`.
- `BlockRefTableRead()` (@1219, @1234, @1271) and `BlockRefTableFlush()`
  (@1206) — the `int length` / `int` return types become `ssize_t`.
- Header `src/include/common/blkreftable.h` (@43) — the
  `BlockRefTableReadCallback` / `WriteCallback` typedefs change to
  `ssize_t (*)(void *, void *, size_t)`.

The callers move in lockstep: `walsummary.c`'s `ReadWalSummary()` (@290) and
`WriteWalSummary()` (@306), the `walsummary.h` prototypes (@42), and
`pg_walsummary.c` (@41, @241). **Impact on corpus:** any per-file doc that
quotes these signatures (`blkreftable.c.md`, `blkreftable.h.md`,
`walsummary.c.md`, `walsummary.h.md`, `pg_walsummary.c.md`) now shows the old
`int` type — flag for a cite refresh, but no semantic doc change is required
(the algorithm is untouched; only the width/signedness of the length type
changed, which matters only on LLP64 Windows where `int` < `size_t`).

### `302222eddc39` — Clean up readlink() return type
[[knowledge/files/src/port/dirmod.c.md]] ·
[[knowledge/files/src/backend/backup/basebackup.c.md]] ·
[[knowledge/subsystems/backup-and-recovery]]

`readlink(2)` returns `ssize_t`, but PG historically stored the result in an
`int` in ~7 call sites that read a tablespace symlink target. This commit
threads `ssize_t` through the shim and its callers:
- `src/port/dirmod.c` `pgreadlink()` (@389) grows a proper `ssize_t` return
  path (+10 lines — the largest single hunk in the series).
- `src/include/port.h` (@319) and `win32_port.h` (@237) — the `pgreadlink`
  prototype return type.
- Backend callers: `xlog.c` `do_pg_backup_start()` (@9667),
  `basebackup.c` `sendDir()` (@1410), `pg_tablespace.c`
  `get_tablespace_location()` (@31). Frontend: `initdb/findtimezone.c`
  (@533), `pg_combinebackup.c` (@1229), `pg_rewind/file_ops.c` (@457),
  `win32stat.c` (@176/@234). No behavior change — on a symlink whose target
  fits `MAXPGPATH` the old `int` was always sufficient; this is
  standards-correctness and silences signedness warnings.

### `1ae87df63ff6` — Clean up copy_file_range() return type
[[knowledge/files/src/bin/pg_combinebackup/reconstruct.c.md]]

Single-line: in `write_reconstructed_file()` (`reconstruct.c` @695) the
`copy_file_range()` result was captured in `int wb;`, now `ssize_t wb;`.
`copy_file_range(2)` returns `ssize_t`; on a short/interrupted copy the value
is compared in a `do { } while` retry loop, so the type is the correct-form fix
even though `BLCKSZ - nwritten` never exceeds `INT_MAX` in practice.

### `8ef9896e61e7` — Move Windows ssize_t definition earlier
[[knowledge/files/src/include/port.h.md]] (Windows port surface)

Relocation only, no code added: the `#ifdef _MSC_VER` block defining
`ssize_t` (`long` on Win32, `__int64` on Win64), `mode_t`, and the
`F_OK`/`W_OK`/`R_OK` access-mode macros moves from after the
`AddUserToTokenDacl` declaration (@549) up to just after the `_pgftello64`
block (@210). The commit message states the reason plainly: "To be used by a
subsequent patch" — declarations earlier in `win32_port.h` will soon need
`ssize_t`, which is exactly what the `readlink`/`blkreftable` `ssize_t` sweep
above sets up. This is the load-bearing "infrastructure" commit of the series.
