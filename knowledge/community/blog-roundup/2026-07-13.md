## 1. Radim Marek — The Tests Passed. The Plan Didn't. (RegreSQL 2.0)
- **URL:** https://postgr.es/p/9pn — Boring SQL, 2026-07-12
- RegreSQL 2.0 adds **query-plan testing** alongside output validation:
  it flags queries that return *correct results via an inefficient plan*,
  catching perf regressions that result-only tests miss.
- Measures actual buffer usage and cardinality error rather than trusting
  planner cost estimates — i.e. asserts on `EXPLAIN (ANALYZE, BUFFERS)`
  shape, not just the result set.
- **Read if you care about:** regression-testing query plans, catching
  silent plan flips in CI, `EXPLAIN`-based assertions.
  `[[knowledge/skills/executor-and-planner]]`,
  `[[knowledge/skills/planner-cost-model]]`.

## 2. Christophe Pettus — "All Your GUCs in a Row": enable_seqscan
- **URL:** https://postgr.es/ (thebuild.com series) — ~2026-07-13
- Demystifies `enable_seqscan`: it does **not** forbid sequential scans, it
  adds a large cost penalty, so the planner still uses a seqscan when no
  alternative exists.
- Common misconception corrected: `enable_*` GUCs are cost-nudges, not hard
  switches — useful for diagnosis, dangerous as permanent settings.
- **Read if you care about:** planner GUC semantics, debugging why a scan
  choice won't change. `[[knowledge/skills/planner-cost-model]]`,
  `[[knowledge/skills/gucs-config]]`.

## 3. Christophe Pettus — "All Your GUCs in a Row": enable_sort
- **URL:** https://postgr.es/ (thebuild.com series) — ~2026-07-12
- Same treatment for `enable_sort`: disabling it penalizes explicit sort
  nodes, pushing the planner toward index-ordered or hash-based paths where
  available.
- Notes the interaction with `work_mem` and when a forced no-sort plan is
  actually *slower*.
- **Read if you care about:** sort vs hash-agg planning, ORDER BY / GROUP BY
  path selection. `[[knowledge/skills/planner-cost-model]]`,
  `[[knowledge/skills/executor-and-planner]]`.
