1. **How SQL/PGQ Rewrites to Joins on PostgreSQL 19**
   — https://postgr.es/p/9pC — 2026-07-14
   - Walks through how PostgreSQL's SQL/PGQ (property-graph query) support
     lowers a `GRAPH_TABLE (... MATCH ...)` pattern into ordinary relational
     joins that the existing optimizer already knows how to cost and execute —
     no separate graph engine, just a rewrite front-end over the join planner.
   - Shows the rewritten plan so you can see vertex/edge tables become join
     relations and pattern quantifiers become join conditions.
   - *Read if you care about:* how new SQL:2023 surface syntax reuses the
     existing planner rather than bolting on a new executor, and where the
     graph-pattern rewrite sits relative to the join-order search.
     `[[executor-and-planner]]` `[[parser-and-nodes]]`

2. **All Your GUCs in a Row: escape_string_warning**
   — https://postgr.es/p/9pB — 2026-07-14
   - Christophe Pettus's GUC series covers `escape_string_warning`: a
     20-year-old parameter that warns when a backslash is used as an escape
     in a regular (non-`E''`) string literal — a canary for SQL that assumed
     the pre-`standard_conforming_strings` world.
   - The practical angle: leaving it on surfaces application strings that
     silently changed meaning when `standard_conforming_strings` became the
     default, which is exactly the kind of latent bug that hides in old code.
   - *Read if you care about:* lexer string-literal handling, the
     `standard_conforming_strings` history, and GUC-as-diagnostic patterns.
     `[[gucs-config]]` `[[parser-and-nodes]]`

3. **All Your GUCs in a Row: enable_tidscan**
   — https://postgr.es/p/9po — 2026-07-13
   - Same GUC series, on `enable_tidscan`: TID scans only fire when the query
     has an explicit `ctid = ...` (or `ctid IN (...)`) reference, so for the
     vast majority of workloads this planner GUC is inert — flipping it off
     changes nothing because there's no TID-scan path to disable.
   - Clarifies the narrow niche where a TID scan actually appears (self-joins
     on `ctid`, `WHERE ctid = '(0,1)'` maintenance queries).
   - *Read if you care about:* the TID-scan executor path, when the planner
     even considers it, and the "enable_* GUCs are cost penalties or no-ops,
     not switches" theme. `[[executor-and-planner]]` `[[planner-cost-model]]`

4. **The tests passed. The plan didn't.** *(reach-back, 07-12)*
   — https://postgr.es/p/9pn — 2026-07-12
   - Introduces RegreSQL 2.0, a regression-testing harness that asserts on the
     **query plan and measured performance**, not just the output rows — so a
     refactor that returns the right answer via a suddenly-worse plan (seq scan
     where an index scan used to fire) fails the test.
   - Directly relevant to the "output-equivalent but plan-regressed" failure
     class that bites during planner/executor changes.
   - *Read if you care about:* plan-stability testing, catching silent
     planner regressions in CI, and why row-equality tests miss performance
     cliffs. `[[testing]]` `[[executor-and-planner]]`

5. **All Your GUCs in a Row: enable_sort** *(reach-back, 07-12)*
   — https://postgr.es/p/9pm — 2026-07-12
   - GUC series on `enable_sort`: turning it off doesn't make sorts free or
     fix a slow query — it just tells the planner to prefer non-sort paths
     (index order, hash agg) at a large cost penalty, and the sort often
     reappears anyway when it's genuinely required.
   - The real fix for sort-bound queries is a matching index or more
     `work_mem`, not the GUC.
   - *Read if you care about:* sort-node costing, `work_mem` interaction with
     external merge sorts, and planner cost-penalty mechanics.
     `[[executor-and-planner]]` `[[planner-cost-model]]`
