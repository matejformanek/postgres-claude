## 1. All Your GUCs in a Row: enable_sort
- Christophe Pettus · 2026-07-12 · https://postgr.es/p/9pm
- TL;DR:
  - `enable_sort=off` doesn't make sorting fast — it raises the planner's cost
    estimate for a Sort node so alternatives (index scan, hash agg) are
    preferred; the sort still runs if it's the only option.
  - The real levers are `work_mem` (to avoid external merge sorts) and a
    matching index that supplies rows pre-ordered.
- Read if you care about: planner cost knobs, Sort-node avoidance.

## 3. All Your GUCs in a Row: enable_seqscan
- Christophe Pettus · 2026-07-11 · https://postgr.es/p/9pi
- TL;DR:
  - `enable_seqscan=off` can't truly forbid sequential scans — with no usable
    index the planner falls back to a seq scan regardless, just with an inflated
    cost that distorts the rest of the plan.
  - Framed as a diagnostic tool (does an index even help?), not a production
    setting.
- Read if you care about: planner GUCs, scan-path selection.

## 4. All Your GUCs in a Row: enable_self_join_elimination
- Christophe Pettus · 2026-07-10 · https://postgr.es/p/9pg
- TL;DR:
  - Walks the **PG18 self-join elimination** optimization: when a table is
    joined to itself on its primary key, the planner proves the join is
    redundant and drops it.
  - On by default; the GUC exists mainly to disable it for debugging.
- Read if you care about: PG18 planner internals, join removal
  (cf. [[executor-and-planner]] — `reduce_unique_selfjoins`).
