## 1. Hubert 'depesz' Lubaczewski — Waiting for PostgreSQL 20: Add min() and max() aggregate support for uuid

- **URL:** <https://postgr.es/p/9pb>
- **Published:** 2026-07-09 12:41

TL;DR:
- depesz's "Waiting for PostgreSQL 20" series covers a committed patch adding
  `min()` and `max()` aggregates for the `uuid` type — the type already has a full
  set of comparison operators and a B-tree operator class, so the aggregates are a
  thin wiring job on top of existing infrastructure rather than new comparison
  semantics.
- Fills a small but real ergonomic gap: `MAX(uuid_col)` previously errored, forcing
  a cast or a `ORDER BY … LIMIT 1` workaround.

**Read if you care about:** the aggregate/opclass wiring pattern (how a type with a
btree opclass gets min/max nearly for free), PG20 feature tracking. Pairs with
`[[knowledge/skills/catalog-conventions]]` (pg_aggregate.dat entries) and
`[[knowledge/skills/fmgr-and-spi]]`.

## 2. Haki Benita — How to Achieve Pruning When Querying by Non-Partitioned Columns in PostgreSQL

- **URL:** <https://postgr.es/p/9pa>
- **Published:** 2026-07-08 21:00

TL;DR:
- A deep how-to on getting partition elimination when you filter on a *non-*
  partition-key column, by exploiting correlation: add per-partition `CHECK`
  constraints describing the range of the correlated column in each partition, and
  the planner can then prune on that column via constraint exclusion.
- Handles the messy parts: outliers that would widen a partition's range are
  isolated using multiple ranges (the same idea as multi-minmax BRIN), and the
  article solves the gaps-and-islands problem to compute the tightest constraint
  ranges from the actual data.

**Read if you care about:** partition pruning mechanics, constraint exclusion vs
partition-key pruning, BRIN multi-minmax as a modeling idea, correlation-driven
query optimization. Strongly internals-adjacent — pairs with
`[[knowledge/skills/executor-and-planner]]` (partition pruning /
constraint exclusion) and the `enable_partition_pruning` post from the 07-06
roundup.

## 3. Ruohang Feng — Happy 30th Birthday, PostgreSQL

- **URL:** <https://postgr.es/p/9p9> (redirects to
  vonng.com/en/pg/happy-30-birthday/)
- **Published:** 2026-07-08 00:00

TL;DR:
- A community-history piece marking PostgreSQL's 30th anniversary (dated to Marc
  Fournier's first CVS commit in July 1996), arguing that *extensibility* is the
  through-line that turned PG from one database into a platform — PostGIS,
  TimescaleDB, pgvector cited as the payoff.
- Forward-looking thesis: PG's open, community-controlled governance is what
  positions it for an era of AI agents and data-sovereignty concerns, where a
  vendor-neutral, self-hostable core matters more.

**Read if you care about:** PG history and governance, the extensibility argument,
the community's own framing of why it wins. Non-internals, but a well-sourced take
on the platform's trajectory.
