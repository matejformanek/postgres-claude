## 1. All Your GUCs in a Row: enable_hashjoin

- Author: Christophe Pettus. 2026-06-28. <https://postgr.es/p/9nv>
- TL;DR: Per-GUC explainer on `enable_hashjoin` — using `enable_hashjoin = off`
  as a diagnostic to flush out hash joins that spill to disk and hurt query
  performance.
- Frames the GUC as a troubleshooting lever (force the planner off hash joins
  to see whether a spill is the culprit), not a permanent setting.
- Read if you care about: planner/executor join selection, diagnosing hash
  spills. [[knowledge/idioms/parallel-hash-join.md]] /
  [[knowledge/subsystems/optimizer.md]]

## 2. All Your GUCs in a Row: enable_hashagg

- Author: Christophe Pettus. 2026-06-27. <https://postgr.es/p/9nt>
- TL;DR: Covers `enable_hashagg` and PG13's memory-safe (spill-capable) hash
  aggregation, including the regression some users hit after upgrading when
  hashagg started respecting `work_mem` and spilling instead of overrunning
  memory.
- Useful historical context: the PG13 hashagg spill change altered real-world
  plan performance and is a common "why did my query get slower after upgrade"
  answer.
- Read if you care about: hash aggregation, work_mem spill behavior, upgrade
  regressions. [[knowledge/idioms/aggregate-hash-vs-sort.md]]

## 3. Stop Punishing Your Postgres for a Crash That Won't Happen

- Author: Lætitia AVROT. 2026-06-27. <https://postgr.es/p/9nu>
- TL;DR: Challenges the reflex of keeping checkpoint timeouts short "for fast
  recovery," arguing that longer `checkpoint_timeout` does not necessarily mean
  longer crash recovery — especially when replication already covers
  availability.
- Reframes the checkpoint-tuning tradeoff: short timeouts buy little recovery
  benefit while raising steady-state write amplification (more full-page
  images, more I/O).
- Read if you care about: checkpoint tuning, WAL/full-page-image cost, recovery
  RTO. [[knowledge/idioms/checkpoint-coordination.md]] /
  [[knowledge/idioms/wal-page-write-flush.md]]
