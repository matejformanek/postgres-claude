## 1. Christophe Pettus — All Your GUCs in a Row: enable_partitionwise_join

- **URL:** <https://postgr.es/p/9p1>
- **Published:** 2026-07-08 01:00

TL;DR:
- Next entry in the per-GUC series, on `enable_partitionwise_join`: when a join
  between two identically-partitioned tables can be decomposed into a set of
  smaller per-partition-pair joins instead of one join over the whole tables,
  and why the planner needs matching partition bounds on both sides to do it.
- Covers the trade-off the GUC defaults *off* for: partitionwise join can slash
  work on well-aligned partitioned tables but costs planning time and memory
  (more paths to consider), so it's opt-in on workloads that actually benefit.

**Read if you care about:** partitioned-table query planning, the
`enable_partitionwise_*` family, why a join over partitioned tables isn't already
per-partition. Pairs with `[[knowledge/skills/executor-and-planner]]` and the
07-07 roundup's `enable_partitionwise_aggregate` sibling post.

## 2. Mayur B. — My Dishonest Benchmark

- **URL:** <https://postgr.es/p/9p0>
- **Published:** 2026-07-07 07:21

TL;DR:
- A methodology piece: walks through how a benchmark can be *technically
  accurate yet misleading* — cherry-picked workloads, warm-vs-cold cache framing,
  ignoring variance, comparing tuned-vs-default, and choosing the metric that
  flatters your conclusion.
- Practical value is the checklist it implies for reading (and writing) PG
  performance claims: state the hardware, the cache state, the config deltas, and
  the distribution, not just the median.

**Read if you care about:** evaluating performance claims about PG features,
writing honest benchmarks, not being fooled by a vendor graph. General-audience,
not internals — but useful hygiene for anyone reading CF performance threads.

## 3. Shridhar Khanal — PostgreSQL Disaster Recovery with pgBackRest TLS Transport

- **URL:** <https://postgr.es/p/9oD>
- **Published:** 2026-07-07 15:41

TL;DR:
- Operational walkthrough of configuring pgBackRest to move backups over a TLS
  server/client-certificate channel instead of SSH, so a repository host can be
  reached without distributing SSH keys — better for scale-out and for
  credential rotation.
- Covers the cert setup, the pgBackRest TLS server daemon, and why cert-based
  transport is easier to lock down and audit than shared SSH keys in a
  multi-node DR topology.

**Read if you care about:** backup/DR operations, pgBackRest deployment, moving
off SSH for backup transport. Ops-focused; complements the backup-internals
material in `[[knowledge/skills/backup-and-recovery]]` (the "what the server
does" side of the same story).

## 4. Gabriele Bartolini — CNPG Recipe 25: Declarative Roles and Passwordless TLS in CloudNativePG 1.30

- **URL:** <https://postgr.es/p/9op>
- **Published:** 2026-07-06 08:56

TL;DR:
- Introduces CloudNativePG 1.30's `DatabaseRole` CRD (declarative role
  management reconciled by the operator) and built-in TLS client certificates
  for passwordless auth inside Kubernetes PG clusters.
- The declarative-roles angle is the notable one: roles become a
  desired-state object the operator converges, rather than something you `psql`
  in imperatively — closing a long-standing gap between K8s-native config and
  PG's own role model.

**Read if you care about:** running PG on Kubernetes, CloudNativePG, declarative
role/auth management, certificate-based `cert` authentication (the server side is
`[[knowledge/skills/wire-protocol]]` / pg_hba `cert` method).
