## 1. pg_stat_statements: everything it tells you — Radim Marek (Jun 2)

URL: https://postgr.es/p/9l3

- Frames `pg_stat_statements` entries as "running counters per shape" in
  fixed shared memory, keyed by a `queryid` hash sensitive to object OIDs,
  aliases, and structure — so ORMs fragment one logical query across hundreds
  of rows; notes PG18 collapses identical `IN`-lists.
- The extension freezes the *first-seen* query text (keeps static comment tags,
  loses dynamic per-request values); averages hide latency distribution — no
  percentiles or history are stored internally.

**Read if you care about:** how query fingerprinting/`queryid` works and why
your statement stats fragment. (cf. [[knowledge/architecture/query-lifecycle.md]])

## 2. pg_stat_statements: everything it can't — Radim Marek (Jun 3)

URL: https://postgr.es/p/9l6

- Second installment on the limits: fixed-size hash table evicts
  least-executed entries (watch the `dealloc` counter and `stats_reset`
  timestamp), external query-text file can fail, and there are no plans,
  actual row counts, parameter values, or failed-query rows.
- Monitors one node only — invisible to replicas; answers "what is slow?" but
  not "why" or "where."

**Read if you care about:** the boundary of what pg_stat_statements can
observe; pairs with the executor instrumentation story. (cf.
[[knowledge/subsystems/executor.md]])

## 3. The Night Our Tables Wouldn't Stop Growing — semab tariq (Jun 2)

URL: https://postgr.es/p/9l0

- Logical-replication incident: destination tables bloated past 400 GB while
  source stayed small. Root cause: a 1-minute `statement_timeout` on the
  *publisher* repeatedly interrupted the initial `COPY`, each interruption
  leaving dead tuples autovacuum couldn't keep pace with.
- Fix: disable `statement_timeout` for the replication role via `ALTER ROLE`.

**Read if you care about:** logical-replication initial-sync mechanics and how
interrupted COPY interacts with dead-tuple accumulation. (cf.
[[knowledge/architecture/replication.md]], [[knowledge/architecture/mvcc.md]])

## 4. Handling graphs with SQL/PGQ in PostgreSQL — Hans-Juergen Schoenig (Jun 3)

URL: https://postgr.es/p/9l7

- Walks the PG19 SQL/PGQ (ISO/IEC 9075-16) feature: define property graphs over
  vertex/edge tables, then query with `GRAPH_TABLE` + pattern matching
  ("friends of friends").
- Key internals point: graph queries **compile to standard SQL joins** under
  the hood — no separate graph engine.

**Read if you care about:** the parser/executor reach of property-graph
queries and how `GRAPH_TABLE` lowers to relational joins. (cf.
[[knowledge/architecture/query-lifecycle.md]], [[knowledge/architecture/planner.md]])
