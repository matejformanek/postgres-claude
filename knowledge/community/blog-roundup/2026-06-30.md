## 1. Waiting for SQL:202y: Stockholm (BMA) meeting report

- Author: Peter Eisentraut
- URL: <https://postgr.es/p/9nP> (2026-06-30)
- TL;DR:
  - Reports out of the ISO SQL standards committee meeting: new syntax features
    adopted including `QUALIFY`, `INSERT BY NAME`, `SELECT EXCLUDE`, and
    `JOIN TO ONE`.
  - These are the standard-track features PostgreSQL implementers will weigh
    for future cycles — a forward look at what the grammar may eventually grow.
- **Read if you care about:** the SQL standard roadmap, parser/grammar
  direction, and which new query syntax might land in a future PG.
  [[knowledge/subsystems/parser-and-rewrite.md]]

## 2. Too many tables are bad for you

- Author: Laurenz Albe (CYBERTEC)
- URL: <https://postgr.es/p/9nN> (2026-06-30)
- TL;DR:
  - Explains how a very large number of tables/partitions in one database
    exhausts backend memory and slows metadata queries via unbounded catalog
    (relcache/syscache) growth.
  - Offers mitigation strategies for systems already in this state.
- **Read if you care about:** catalog cache behavior, relcache memory pressure,
  and the practical ceiling on partition/table counts.
  [[knowledge/subsystems/relcache.md]], [[knowledge/idioms/syscache-relcache.md]]

- URL: <https://postgr.es/p/9nL> (2026-06-30)

## 5. Same rows, different SUM

- Author: Radim Marek
- URL: <https://postgr.es/p/9nx> (2026-06-29)
- TL;DR:
  - Demonstrates that parallel aggregates can return slightly different
    floating-point sums run-to-run because of order-dependent rounding — the
    parallelism reorders the additions, it isn't "wrong."
  - Clarifies a frequent source of confusion in reproducibility complaints.
- **Read if you care about:** parallel aggregation semantics, float
  non-associativity, and why `SUM` results wobble.
  [[knowledge/subsystems/executor.md]], [[knowledge/idioms/parallel-query.md]]
