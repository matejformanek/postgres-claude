# Planet PostgreSQL blog roundup — 2026-06-09

## 1. PGConf.dev 2026 in Vancouver
- **Author:** Christoph Berg — **URL:** https://postgr.es/p/9lB (Jun 9)
- TL;DR:
  - First-hand developer-conference recap touching packaging, contributor
    recognition, and the hallway-track of in-flight internals work —
    **TDE (transparent data encryption)** implementation, **key joins**, and
    **temporal queries** among the named threads. `[from-website]`
  - Useful as a signal of which large features have committer mindshare heading
    into the PG20 cycle. `[from-website]`
- **Read if you care about:** where core-dev attention is going — TDE, temporal,
  and the social machinery of getting a big patch landed.

## 2. My Three Top PostgreSQL 19 Features
- **Author:** Stefanie Janine — **URL:** https://postgr.es/p/9lz (Jun 8)
- TL;DR:
  - Highlights three PG19-beta features: **`INSERT ... ON CONFLICT` taking a
    `SELECT` source** (richer conflict handling), a **native `REPACK` command
    with `CONCURRENTLY`** (in-core online table rewrite, the pg_repack idea
    upstreamed), and **SQL/PGQ property-graph queries**. `[from-website]`
  - `REPACK CONCURRENTLY` is the most internals-heavy of the three — an in-core
    bloat-reclaim that must coordinate with MVCC/vacuum and the buffer manager. `[inferred]`
- **Read if you care about:** PG19's user-visible headline features, especially
  in-core REPACK vs. the contrib/extension lineage. cf.
  [[knowledge/architecture/mvcc.md]].

## 3. How the Other Half Counts
- **Author:** Christophe Pettus — **URL:** https://postgr.es/p/9lx (Jun 8)
- TL;DR:
  - Compares how PostgreSQL gathers **planner statistics** (the ANALYZE-driven
    `pg_statistic` MCV/histogram model) against Oracle, MySQL, SQLite, DuckDB,
    and Snowflake — block-sampling vs. full scans, and when each refreshes. `[from-website]`
  - Good context for the same-week pgsql-hackers extended-statistics and MCV
    threads ([[knowledge/community/threads/2026-06-09.md]]). `[inferred]`
- **Read if you care about:** the optimizer's statistics model and how PG's
  sampling design differs from other engines. cf.
  [[knowledge/architecture/planner.md]], [[knowledge/subsystems/optimizer.md]].

## 4. All Your GUCs in a Row: DateStyle
- **Author:** Christophe Pettus — **URL:** https://postgr.es/p/9ly (Jun 9)
- TL;DR:
  - Walks through the `DateStyle` GUC and its well-known footgun: it **controls
    two unrelated things at once** — output formatting *and* ambiguous-input
    interpretation (MDY vs. DMY) — a frequent source of confusion. `[from-website]`
  - Part of Pettus's ongoing per-GUC series; a clean reference for the
    DateStyle parsing/formatting split. `[from-website]`
- **Read if you care about:** GUC semantics and date/time input parsing. cf.
  [[knowledge/idioms/guc-variables.md]].

## 5. pg_background 2.0: Run SQL in the Background
- **Author:** Vibhor Kumar — **URL:** https://postgr.es/p/9lw (Jun 8)
- TL;DR:
  - Update to the `pg_background` extension (autonomous-style background SQL via
    a dynamic background worker): cleaner API, **structured error reporting**,
    **stricter security defaults**, and PG19 compatibility. `[from-website]`
  - The bgworker + dynamic-execution pattern is exactly the surface the corpus
    tracks for trust-boundary and SQL-injection-sink concerns. `[inferred]`
- **Read if you care about:** background-worker extension design and the
  security posture of run-SQL-in-the-background tooling. cf.
  [[knowledge/idioms/bgworker-and-parallel.md]], [[knowledge/idioms/spi.md]].
