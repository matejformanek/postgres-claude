## 1. pg_rewind does not rewind diverging timelines

- **Author:** Andrey Borodin (Jun 8, 10:48) `[from-website]`
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606080000
- Bullets:
  - `pg_rewind`'s job is to make an old primary re-followable after a failover
    by copying back only the blocks that diverged since the two clusters'
    histories split. The report is that when the **timelines themselves
    diverge** (each side advanced onto a *different* new timeline), the rewind
    does not correctly identify or roll back the divergent segment. `[inferred]`
  - The hazard is a silently-incomplete rewind: the rewound node appears to
    follow the new primary but retains blocks from its own abandoned timeline,
    a classic split-brain residue. `[inferred]`
  - Diverging-timeline handling lives in the WAL history-file comparison that
    finds the last common checkpoint; the bug is most likely in that
    common-ancestor search rather than the file-copy step. `[inferred]`
- **Why it matters:** `pg_rewind` correctness is load-bearing for HA failover;
  an incomplete rewind is undetectable corruption on a node operators believe
  is a clean replica. Same "name/history-based trust without full validation"
  family flagged in the corpus's pg_rewind notes. cf.
  [[knowledge/architecture/wal.md]], [[knowledge/architecture/replication.md]],
  [[knowledge/subsystems/access-transam.md]].

## 2. Fix minRecoveryPoint not advanced past checkpoint in CreateRestartPoint

- **Author:** Nitin Jadhav (Jun 9, 12:38) `[from-website]`
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606080000
- Bullets:
  - On a standby, `CreateRestartPoint()` is the recovery-side analogue of a
    checkpoint: it flushes buffers and lets WAL before the restartpoint be
    recycled. The report is that `minRecoveryPoint` (the LSN below which the
    cluster may not be brought consistent) is **not advanced past the
    checkpoint record** the restartpoint just processed. `[inferred]`
  - If `minRecoveryPoint` lags, the control file under-reports how far recovery
    has safely progressed; a crash-and-restart can redo more than necessary, or
    in the worst reading allow WAL recycling that the consistency bound should
    have pinned. `[inferred]`
  - This is the recovery-side bookkeeping pair to the primary's checkpoint LSN
    accounting; the fix is a one-LSN advancement in the restartpoint path. `[inferred]`
- **Why it matters:** `minRecoveryPoint` is the invariant that guarantees a
  standby never opens for queries before it is crash-consistent; an under-set
  value is a correctness bound being relaxed silently. cf.
  [[knowledge/architecture/wal.md]], [[knowledge/subsystems/access-transam.md]].

## 3. Proposal: Conflict log history table for Logical Replication

- **Author:** shveta malik (Jun 8, 06:12) `[from-website]`
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606080000
- Bullets:
  - Builds on the recent logical-replication **conflict detection** work
    (insert_exists / update_differ / delete_missing etc. now logged to the
    server log): the proposal is a durable **history table** that records each
    apply-worker conflict as a queryable row instead of only a log line. `[inferred]`
  - Motivation is operability — a subscriber's conflict history is currently
    scattered across server logs; a table lets monitoring/automation diff and
    alert, and supports future automatic conflict-resolution policies. `[inferred]`
  - Design questions surfaced by the subject: where the table lives (per-DB
    catalog vs. user table), retention/pruning, and how the apply worker writes
    it without re-entering conflict handling. `[inferred]`
- **Why it matters:** Logical-replication conflict observability is a gating
  feature for active-active and large fan-out topologies; turning ephemeral log
  lines into queryable state is the same "monitoring-as-first-class-state" arc
  the corpus tracks in replication. cf. [[knowledge/subsystems/replication.md]],
  [[knowledge/issues/include-replication.md]].

## 4. [PATCH] Avoid orphaning buffers when a relation's file is missing

- **Author:** Adam Lee (Jun 9, 09:55) `[from-website]`
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606080000
- Bullets:
  - When a relation's underlying file is missing (dropped/truncated underfoot,
    or smgr open failure), buffers already pinned/registered for that relation
    in shared buffers can be **orphaned** — left in the buffer pool with no
    valid backing file, so later scans of `BufferDesc` state hit a buffer that
    can be neither flushed nor cleanly evicted. `[inferred]`
  - The patch makes the missing-file path discard or properly invalidate the
    affected buffers so the buffer mapping stays consistent with smgr reality. `[inferred]`
  - Touches the `smgr` ↔ `bufmgr` boundary — the same surface where the corpus
    flags `RBM_ZERO_ON_ERROR` / `READ_BUFFERS_IGNORE_CHECKSUM_FAILURES` as
    silent-tolerance read paths (`bufmgr.h`). `[inferred]`
- **Why it matters:** An orphaned buffer is a small but real shared-state leak
  that can wedge checkpoint/eviction and mask the underlying missing-file
  condition; buffer-pool ↔ smgr coherence is a core storage invariant. cf.
  [[knowledge/subsystems/storage-buffer.md]], [[knowledge/issues/include-storage.md]].

## 5. Report oldest xmin source when autovacuum cannot remove tuples

- **Author:** Sami Imseih (Jun 8, 16:35) `[from-website]`
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606080000
- Bullets:
  - When autovacuum runs but **cannot remove dead tuples** because some xmin
    horizon is held back, today the operator gets little visibility into *which*
    holder is to blame — a long-running transaction, an abandoned replication
    slot, `hot_standby_feedback`, or a prepared transaction. The proposal makes
    vacuum **report the source** of the oldest xmin pinning the horizon. `[inferred]`
  - This is diagnostics, not a behavior change: surface the limiting xmin's
    origin (backend PID / slot / prepared-xact) in the vacuum log or stats so
    bloat investigations stop being guesswork. `[inferred]`
  - Directly complementary to the same-week `catalog_xmin`-separation work
    ([[knowledge/community/threads/2026-06-08.md]] §2): both are about making
    horizon accounting legible and less over-conservative. `[inferred]`
- **Why it matters:** "Vacuum runs but bloat grows" is one of the most common
  production MVCC pathologies; naming the xmin holder turns a multi-hour
  forensic exercise into a single log line. cf.
  [[knowledge/architecture/mvcc.md]], [[knowledge/subsystems/access-heap.md]].
