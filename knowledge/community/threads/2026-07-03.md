## 1. BUG: ReadStream look-ahead exhausts local buffers when effective_io_concurrency ≥ 64

- **Author(s):** Xuneng Zhou (23:50, 01:24, 03:54) — multi-message active thread
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202607020000
- Summary:
  - A ReadStream consumer scanning a temp/local relation over-issues look-ahead
    reads: at `effective_io_concurrency >= 64` the stream's distance target
    exceeds the number of local buffers the backend can pin, and the pin pool is
    exhausted before the stream can retire earlier buffers.
  - Local buffers (temp tables) are not backed by the shared buffer pool and have
    a much smaller, per-backend pin budget; the shared-buffer look-ahead heuristic
    doesn't clamp against that smaller pool.
  - Fix direction under discussion: cap the ReadStream look-ahead distance by the
    available local-buffer headroom when the target relation uses local buffers.
- **Why it matters:** the read-stream prefetch machinery is the PG 17+ unified
  I/O path; a distance heuristic that ignores the local-buffer ceiling turns a
  performance GUC (`effective_io_concurrency`) into a correctness/OOM footgun on
  temp-table scans. `[[knowledge/idioms/read-stream-prefetch.md]]`,
  `[[knowledge/subsystems/storage-buffer.md]]`

## 2. GetBufferDescriptor() being called for local buffers from MarkBufferDirtyHint()

- **Author(s):** Michael Paquier (04:46)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202607020000
- Summary:
  - `MarkBufferDirtyHint()` reaches for `GetBufferDescriptor()` on a buffer that
    can be local (temp relation), but that accessor is defined for shared buffers;
    a local buffer needs `GetLocalBufferDescriptor()`.
  - The two descriptor arrays are disjoint; indexing the shared array with a
    local buffer number reads the wrong slot (silent, not a crash on most builds).
  - Proposed guard: branch on `BufferIsLocal()` before choosing the descriptor
    accessor, matching the pattern used elsewhere in bufmgr.
- **Why it matters:** a shared-vs-local descriptor mixup is exactly the class of
  buffer-manager correctness bug that produces rare, hard-to-reproduce corruption;
  pairs with thread 1 as a "local buffers keep getting treated like shared" motif.
  `[[knowledge/subsystems/storage-buffer.md]]`,
  `[[knowledge/idioms/hint-bits-setbufferdirty.md]]`

## 3. Prevent crash when calling pgstat functions with unregistered stats kind

- **Author(s):** Bertrand Drouvot (multi-message: 03:27, 04:06, 04:23, 04:43, 05:00, 05:02)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202607020000
- Summary:
  - Calling a `pg_stat_*` SQL function whose backing stats kind was never
    registered (e.g. an extension's custom kind after the extension is dropped, or
    a bad kind id) dereferences a NULL entry in the pgstat kind table → crash.
  - The very active message cadence (6 messages in <2h) is iterating on where to
    put the guard: at the SQL-function boundary vs inside `pgstat_get_kind_info()`.
  - Ties into thread 4 (register-time validation) — same maintainers hardening the
    custom-stats-kind API surface introduced for extensions.
- **Why it matters:** custom cumulative-stats kinds are a PG 18 extension-facing
  API; an unregistered-kind crash is a stability bug on the exact surface
  extensions are being pointed at. `[[knowledge/idioms/pgstat-flush-timing.md]]`

## 4. add validations for required callbacks during pgstat_register_kind()

- **Author(s):** Sami Imseih (21:48)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202607020000
- Summary:
  - `pgstat_register_kind()` currently accepts a `PgStat_KindInfo` without
    checking that the mandatory callbacks (e.g. the fixed/variable accessors) are
    non-NULL; a partially-filled struct fails later at flush/snapshot time.
  - Proposal: validate the required callback set at registration so the failure is
    an immediate, clear error at `_PG_init` rather than a deferred NULL-deref.
  - Companion to thread 3: register-time validation (this) + call-time guard
    (thread 3) close both ends of the custom-kind lifecycle.
- **Why it matters:** fail-fast at registration is the right ergonomics for an
  extension API; this is the "make the framework teach you how to use it" half of
  the pgstat hardening cluster. `[[knowledge/idioms/pgstat-flush-timing.md]]`

## 6. WIP: replacing join_collapse_limit with "join hardness" estimate

- **Author(s):** John Naylor (14:34)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202607020000
- Summary:
  - Long-standing `join_collapse_limit` / `from_collapse_limit` are blunt integer
    cutoffs that flatten the join search space regardless of how expensive the
    actual enumeration would be for a given query shape.
  - The WIP proposes a "join hardness" estimate — a measure of the combinatorial
    cost of the join search for *this* query — to decide when to collapse, instead
    of a fixed GUC threshold.
  - Still exploratory (WIP tag); the hard part is a cheap, robust hardness metric
    that doesn't itself blow up on the queries it's meant to protect.
- **Why it matters:** the collapse limits are a well-known planner sharp edge
  (queries silently get worse plans past the cutoff); replacing them with an
  adaptive estimate is a structural planner improvement. `[[knowledge/subsystems/optimizer.md]]`

## 7. Batching in executor

- **Author(s):** Amit Langote (05:19, 06:05)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202607020000
- Summary:
  - Continued design work on pushing tuple *batches* (rather than one-tuple-at-a-
    time `ExecProcNode` calls) through the executor to cut per-tuple dispatch
    overhead and open the door to vectorized inner loops.
  - Touches the `PlanState` pull-model contract: batching changes the unit that
    flows between nodes, so every node type's `ExecProcNode` shape is affected.
  - Active multi-message iteration; the architectural question is how much of the
    node protocol has to change vs a batch-aware opt-in per node.
- **Why it matters:** the one-tuple-at-a-time executor loop is PG's dominant
  CPU-overhead source on analytic queries; batching is the highest-leverage
  executor-architecture thread in flight. `[[knowledge/subsystems/executor.md]]`,
  `[[knowledge/idioms/expression-evaluator-flow.md]]`

## 8. Bypassing cursors in postgres_fdw to enable parallel plans

- **Author(s):** Robert Haas (17:57)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202607020000
- Summary:
  - `postgres_fdw` currently drives remote scans through a server-side cursor,
    which forces the remote query into a serial, non-parallel execution mode.
  - Proposal: for eligible remote scans, bypass the cursor so the remote planner
    can choose a parallel plan, materializing or streaming results without the
    cursor's single-portal constraint.
  - Trade-off under discussion: cursors give incremental fetch + abort semantics;
    bypassing them needs a different story for large result sets and cancellation.
- **Why it matters:** removes a structural cap on remote-scan performance in the
  standard FDW — most sharding/federation setups run through `postgres_fdw`.
  `[[knowledge/subsystems/foreign.md]]`, `[[knowledge/subsystems/contrib-postgres_fdw.md]]`

## 9. Deadlock detector fails to activate on a hot standby replica

- **Author(s):** Xuneng Zhou (09:31)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202607020000
- Summary:
  - On a hot standby, a lock-wait situation that would trigger the deadlock
    detector on a primary does not arm the detector path, so a genuine wait can
    hang past `deadlock_timeout` without the usual detect-and-abort.
  - Standby lock waits (recovery-conflict interactions with `max_standby_*_delay`)
    take a different code path than primary heavyweight-lock waits; the detector
    activation is gated out there.
  - Under investigation whether this is a missing arm or an intentional (but
    surprising) design gap on the replica.
- **Why it matters:** a lock-wait on a standby that never gets a deadlock verdict
  is a silent-hang class bug for read-replica workloads — the same "detector
  doesn't fire on the replica" motif seen in prior weeks. `[[knowledge/idioms/deadlock-detection.md]]`,
  `[[knowledge/subsystems/storage-lmgr.md]]`

## 10. Introduce XID age based replication slot invalidation

- **Author(s):** Masahiko Sawada (23:17)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202607020000
- Summary:
  - Adds a mechanism to invalidate a replication slot once the transactions it
    pins exceed an XID-age threshold, bounding how far a stuck/abandoned slot can
    hold back the cluster's xmin horizon.
  - Complements the existing size-based (`max_slot_wal_keep_size`) invalidation:
    this one targets the *xmin*/bloat dimension rather than WAL retention.
  - Design questions: which age GUC governs it, and how invalidation interacts
    with in-progress logical decoding on the slot.
- **Why it matters:** abandoned slots pinning xmin is a top operational hazard
  (unbounded bloat, vacuum stall); an age-based backstop is a real reliability
  lever. `[[knowledge/idioms/replication-slot-advance.md]]`,
  `[[knowledge/idioms/xmin-horizon-management.md]]`

## 11. Fix XLogFileReadAnyTLI silently applying divergent WAL from wrong timeline

- **Author(s):** surya poondla (17:59)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202607020000
- Summary:
  - `XLogFileReadAnyTLI()` can pick up a WAL segment from a timeline that has
    diverged from the one the recovery is following, applying records that don't
    belong to the intended history without flagging the mismatch.
  - The any-TLI search is meant to be permissive (find the segment on any listed
    timeline), but permissiveness plus a divergent history = silently applying the
    wrong WAL.
  - Fix direction: tighten the timeline-history check so a divergent segment is
    rejected rather than replayed.
- **Why it matters:** silently replaying WAL from the wrong timeline is a
  data-integrity / recovery-correctness bug at the heart of PITR and standby
  promotion. `[[knowledge/idioms/xlog-region-replay.md]]`,
  `[[knowledge/subsystems/access-transam.md]]`
