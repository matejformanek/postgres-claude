## 1. Error processing invalidation message during ATTACH PARTITION leaves invalid relcache entry

- **Author:** Alexander Lakhin (Jun 7, 10:00)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606070000
- Bullets:
  - During `ALTER TABLE ... ATTACH PARTITION`, an error while *processing a
    relcache invalidation message* can leave a **stale/invalid relcache entry**
    behind — the entry is neither rebuilt nor discarded, so later lookups see a
    half-updated `RelationData`.
  - The hazard is the invalidation-replay path running inside a failing
    transaction: if `RelationCacheInvalidateEntry` / rebuild throws partway, the
    cache is left inconsistent with the catalog the txn rolled back to.
  - Reported by Lakhin (the project's most prolific edge-case fuzzer), so expect
    a minimal reproducer and a backpatch discussion.
- **Why it matters:** Relcache coherence is a load-bearing invariant — a stale
  entry surfaces as wrong tuple descriptors or missing partition routing long
  after the failing DDL. Cache-invalidation correctness underpins every catalog
  read. cf. [[knowledge/idioms/catalog-conventions.md]],
  [[knowledge/subsystems/access-heap.md]].

## 2. catalog_xmin in walsender hot-standby feedback (two threads)

- **Authors:** Amit Kapila — *Separate catalog_xmin from xmin in walsender hot
  standby feedback* (Jun 8, 11:29); Fujii Masao — *Use correct type for
  catalog_xmin* (Jun 8, 05:34, reply Amit Kapila 06:07)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606070000
- Bullets:
  - **Separation thread:** today the standby's hot-standby feedback folds
    `catalog_xmin` into the reported `xmin`, so the primary holds back its
    *data* horizon more than necessary when only *catalog* tuples need pinning
    (the logical-decoding case). Splitting the two lets the primary advance the
    data-row horizon (more aggressive vacuuming) while still protecting catalog
    rows the standby's slots need.
  - **Type thread:** a companion correctness fix — `catalog_xmin` is being
    carried in a too-narrow / wrong C type along the feedback path; Fujii Masao
    flags the mismatch and Kapila concurs same-day.
  - Both touch the `GlobalVisState` / `ComputeXidHorizons` horizon accounting at
    the walsender↔procarray boundary.
- **Why it matters:** Horizon over-holding is a direct cause of bloat on
  primaries feeding logical-decoding standbys; separating the two xmins is a
  real vacuum-aggressiveness win. The type fix is the kind of silent-truncation
  bug that bites at xid wraparound. cf. [[knowledge/architecture/mvcc.md]],
  [[knowledge/subsystems/replication.md]].

## 3. synchronized_standby_slots inconsistent with quorum-based synchronous replication

- **Author:** Ashutosh Sharma (Jun 8, 06:08); follow-ups Ashutosh Sharma
  (08:53), Amit Kapila (09:51)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606070000
- Bullets:
  - `synchronized_standby_slots` (the GUC that makes logical slots wait for
    listed physical standbys before advancing, to prevent slot-ahead-of-standby
    data loss) assumes an **all-of** semantics, but quorum-based
    `synchronous_standby_names` (`ANY k (...)`) only guarantees *k of N*.
  - The mismatch means a logical subscriber can receive changes that a
    quorum-synchronous commit has *not* yet durably replicated to the specific
    standbys named in `synchronized_standby_slots` — breaking the no-data-loss
    promise on failover.
  - Discussion is whether the slot-sync wait should track the quorum set
    dynamically or whether the two GUCs are simply incompatible and should be
    documented/guarded as such.
- **Why it matters:** This is a failover data-integrity corner where two sync
  mechanisms interact; getting it wrong means a promoted standby is missing rows
  a logical subscriber already saw. cf. [[knowledge/subsystems/replication.md]],
  [[knowledge/architecture/replication.md]].

## 4. Subquery pull-up increases jointree search space

- **Author:** Tomas Vondra (Jun 7, 18:10); follow-ups Tom Lane (Jun 7, 18:57),
  Andrei Lepikhov (Jun 8, 07:45), Tomas Vondra (11:04), Andrei Lepikhov (13:32)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606070000
- Bullets:
  - Pulling a subquery up into the parent jointree (the `pull_up_subqueries`
    transform) can **explode the join-order search space** — more base rels in
    one jointree means combinatorially more join paths for the dynamic-programming
    join search to enumerate.
  - Vondra reports a planning-time regression on queries where the flattened form
    is not actually cheaper; Tom Lane weighs the classic tension between
    normalizing the query (good for plan quality) and bounding planner cost.
  - Discussion turns on `from_collapse_limit` / `join_collapse_limit`
    interactions and whether pull-up should be gated when it would push the
    jointree past the collapse threshold.
- **Why it matters:** Planner-time blowup is a real production hazard (one bad
  query stalls planning); the pull-up vs. search-space tradeoff is core
  optimizer-design territory. cf. [[knowledge/architecture/planner.md]],
  [[knowledge/subsystems/optimizer.md]].

## 5. Remove redundant DISTINCT when GROUP BY already guarantees uniqueness

- **Author:** Ilia Evdokimov (Jun 8, 11:12)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606070000
- Bullets:
  - When a query has both `GROUP BY` and a `DISTINCT` whose target list is a
    subset of (or functionally dependent on) the grouping keys, the `DISTINCT`
    is provably a no-op — the grouped output is already unique.
  - The patch detects this and drops the redundant `Unique`/hash-distinct step,
    saving a whole dedup pass at execution time.
  - Sits in the same family as the long-running "remove redundant self-join" and
    "redundant GROUP BY key" planner simplifications.
- **Why it matters:** Cheap, safe planner wins that eliminate an executor node
  compound across analytic workloads. Good cross-link material for the
  optimizer's uniqueness-proving machinery (`query_is_distinct_for`). cf.
  [[knowledge/architecture/planner.md]].
