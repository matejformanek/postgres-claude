## 1. read stream: Backward I/O combining — Nazir Bilal Yavuz

`/message-id/CAN55FZ1nB1p_mqnzVY1_CA-OXgQD1yTEJdHXX4E1hDZxVHQRQA%40mail.gmail.com`
(cont. Tomas Vondra)

- Extends the read_stream API to combine adjacent I/Os when a scan walks
  blocks in *descending* order — today's combining logic assumes forward
  progress and can't coalesce a backward run.
- Targets backward index scans and any consumer that emits a
  monotonically-decreasing block sequence, so the same
  `io_combine_limit`-sized reads apply to reverse traversal.
- Vondra pushing on whether the block-number bookkeeping stays correct at
  the stream-restart / lookahead boundary.

**Why it matters:** read-stream combining is the PG 17/18 mechanism that
turns per-block `ReadBuffer` calls into batched AIO; extending it backward
closes a gap where reverse scans silently fall back to one-block-at-a-time.
[[knowledge/skills/aio-readstream]] · [[knowledge/subsystems/buffer-manager]]

## 2. [PATCH] Batched clock sweep to reduce cross-socket atomic contention — Greg Burd

`/message-id/8ffd4113-1d35-4d11-9d6b-4b49eac9985c%40app.fastmail.com`

- Reworks `StrategyGetBuffer`'s clock-sweep so a backend claims a *batch* of
  victim buffer candidates per pass instead of one, cutting the number of
  atomic increments on the shared `nextVictimBuffer` counter.
- Motivation is NUMA: the single hot atomic on the sweep hand ping-pongs
  cache lines across sockets under high buffer-eviction pressure.
- Trade-off under discussion: batching risks handing out a stale victim that
  was re-pinned between claim and use.

**Why it matters:** the clock-sweep counter is a known multi-socket
scalability bottleneck; batching is the recurring proposed fix and this is a
concrete patch to evaluate. [[knowledge/idioms/clock-sweep-eviction]] ·
[[knowledge/skills/buffer-manager]]

## 3. Checkpointer write combining — Haibo Yan

`/message-id/CABXr29Fe5wmVP%2Bpvtty1j8G1AUHE0-87ubXP7wgEh6S%2BHPqSiQ%40mail.gmail.com`

- Proposes coalescing checkpointer buffer writes into larger sequential I/Os
  when dirty buffers happen to be physically adjacent on disk.
- Companion in spirit to the read-stream write side — the checkpointer today
  issues one `smgrwrite` per dirty buffer even when neighbors are contiguous.
- Early-stage: no benchmarks posted yet, discussion framing the API shape.

**Why it matters:** checkpoint write bursts dominate I/O stalls on
write-heavy workloads; write-combining here is the write-path analogue of
the read_stream work. [[knowledge/idioms/checkpoint-coordination]]

## 4. Batching in executor — Amit Langote

`/message-id/CA%2BHiwqGQPDwxK7QQNUWhqhAEwaH4Z%2BYQhHG5jNPYHGNzDMfsXw%40mail.gmail.com`

- Revives the long-running effort to make executor nodes pass *batches* of
  tuples (vectors of TupleTableSlots) between `ExecProcNode` calls rather
  than one tuple per call.
- Aim is to amortize per-tuple dispatch overhead and open the door to
  SIMD-friendly expression evaluation.
- Central design question: how batching interacts with the volatile
  per-tuple memory-context reset discipline.

**Why it matters:** tuple-at-a-time is PG's foundational executor contract;
any batching model reshapes `execProcnode.c` and every node's state machine.
[[knowledge/subsystems/executor]] · [[knowledge/skills/executor-and-planner]]

## 6. Tepid: selective index updates for heap relations — Greg Burd

`/message-id/84a886d1-37af-4da2-ad1b-15cea9c7e965%40app.fastmail.com`

- "Tepid" proposes updating only the *indexes whose keys actually changed* on
  a heap UPDATE, rather than the current all-or-HOT binary choice.
- Sits between HOT (update no indexes, same page) and a cold update (rewrite
  all index entries): a middle path when one indexed column changed but
  others didn't.
- Raises visibility/vacuum questions — a heap tuple reachable via some
  indexes but not others complicates dead-tuple reasoning.

**Why it matters:** index-write amplification on UPDATE is a major write-path
cost; selective updates could cut it substantially but touch the delicate
HOT invariants. [[knowledge/subsystems/access-heap]]

## 7. sequencesync worker race with REFRESH SEQUENCES — shveta malik

`/message-id/CAJpy0uDVb9iHC532U7bKrr%2B-63voPnxA1kPSu-GdNGZJfYMWgg%40mail.gmail.com`
(cont. Amit Kapila, vignesh C, Hayato Kuroda)

- Bug + fix for a race between the new logical-replication sequence-sync
  worker and a concurrent `ALTER SUBSCRIPTION ... REFRESH PUBLICATION
  SEQUENCES`.
- The worker and the refresh path can both touch the sequence-sync state and
  interleave badly, leaving a sequence half-synced.
- Multi-committer thread — this is active PG18/19 sequence-replication
  hardening.

**Why it matters:** sequence replication is new and its concurrency corners
are still being found; this is exactly the class of bug that silently
diverges a replica. [[knowledge/skills/logical-replication]]

## 8. Proposal: Conflict log history table for Logical Replication — Amit Kapila

`/message-id/CAA4eK1Jx2dHyPAry5%3DPGHUvFygpXErDzuNkNgtLWzpBC49-Bqw%40mail.gmail.com`

**Why it matters:** conflict *detection* landed recently; durable conflict

## 11. Why clearing the VM doesn't require registering vm buffer in wal record — Tom Lane / Melanie Plageman

`/message-id/1065814.1784514869%40sss.pgh.pa.us`

- Analysis thread: when a heap modification *clears* a visibility-map bit,
  the VM buffer need not be WAL-registered because replay re-derives the
  clear from the heap change.
- Distinguishes the set-VM path (must be logged) from the clear-VM path
  (implied), documenting an asymmetry that's easy to get wrong.

**Why it matters:** VM/WAL registration asymmetry is a classic source of
replica-only corruption bugs; this is the reasoning that keeps the
optimization safe. [[knowledge/skills/wal-and-xlog]] ·
