## 1. BM_IO_ERROR flag is lost in TerminateBufferIO due to order of operations in UnlockBufHdrExt — Melanie Plageman

URL: `/message-id/CAAKRu_YX9KsmZUyJQj%2BcQa9gs_aCwikVSXrzE_CkZuZC3k_ukQ%40mail.gmail.com`

- Reports that `TerminateBufferIO` can drop the `BM_IO_ERROR` state bit
  because of the order in which `UnlockBufHdrExt` clears/writes the buffer
  descriptor's atomic state word. `[from-mailing-list]`
- A lost `BM_IO_ERROR` means a buffer whose IO failed can be observed as if
  the IO succeeded — the error signal a waiter relies on is erased before it
  is seen. `[inferred]`
- Lives in the buffer-manager IO-completion path where the state word is
  CAS-updated and waiters are woken. `[inferred]`

**Why it matters:** correctness of the shared-buffer IO state machine — the
`BM_IO_IN_PROGRESS` → `BM_IO_ERROR`/`BM_VALID` handoff is what every backend
waiting on a buffer read trusts. See
[[knowledge/subsystems/storage-buffer.md]] and
[[knowledge/data-structures/bufferdesc-state.md]] (the PG18 atomic state-word
notes are directly relevant here).

## 2. hashjoins vs. Bloom filters (yet again) — Andrei Lepikhov, Tomas Vondra

URL: `/message-id/...` (multiple message-ids across 2026-06-01..02)

- Revives the recurring proposal to build a Bloom filter on the hash-join
  build side and push it down to cheaply reject probe-side rows before they
  are hashed. `[from-mailing-list]`
- "yet again" flags this as a periodically-reattempted optimization; the hard
  parts are costing the filter and deciding when the selectivity pays for the
  build. `[inferred]`
- Touches `nodeHash`/`nodeHashjoin` execution plus join costing in the
  optimizer. `[inferred]`

**Why it matters:** a runtime-filter pushdown changes both executor hot paths
and the planner's join cost model. See [[knowledge/subsystems/executor.md]],
[[knowledge/subsystems/optimizer.md]], and [[knowledge/architecture/planner.md]].

## 3. Fix race in ReplicationSlotRelease for ephemeral slots — Xuneng Zhou

URL: `/message-id/CABPTF7VyH1-W2xnDspECDEzFGQj%3DWTFpZBCqKfM11OAZa6gQHQ%40mail.gmail.com`

- Reports a race in `ReplicationSlotRelease` when releasing an *ephemeral*
  (RS_EPHEMERAL) slot, where the slot's lifetime/cleanup can interleave with
  concurrent access. `[from-mailing-list]`
- Ephemeral slots are the transient slots created during slot setup before
  they are persisted; their release/drop path is a known sharp edge. `[inferred]`
- Sits at the intersection of replication-slot state machine and the LWLock
  protecting `ReplicationSlotControlLock`/per-slot mutex. `[inferred]`

**Why it matters:** slot lifecycle races can corrupt the slot array or leak/
double-free a slot, affecting every logical and physical consumer. See
[[knowledge/architecture/replication.md]] and
[[knowledge/idioms/locking-overview.md]].

## 4. should we have a fast-path planning for OLTP starjoins? — Robert Haas, Tomas Vondra, Joel Jacobson

URL: `/message-id/...` (multiple message-ids on 2026-06-01)

- Asks whether the planner should detect simple OLTP star-join shapes and
  short-circuit the full join-order search with a cheaper fast path. `[from-mailing-list]`
- Motivation: `join_collapse_limit`/`from_collapse_limit` and the dynamic-
  programming join search cost real planning time on queries whose shape is
  obvious; a heuristic fast path could cut latency for OLTP. `[inferred]`
- Design tension is the usual one — a heuristic that wins on the common case
  without regressing plans where the full search mattered. `[inferred]`

**Why it matters:** join-order enumeration is the planner's combinatorial
core; any fast-path bypass changes which plans are even considered. See
[[knowledge/architecture/planner.md]] and [[knowledge/subsystems/optimizer.md]].

## 5. Wrong unsafe-flag test in check_output_expressions() — Richard Guo, David Rowley

URL: `/message-id/CAMbWs49Q_xnF_P2QSUyDzJ34MnrO7dh-cUAaK2HJPgSgh88NcA%40mail.gmail.com`

- Reports an incorrect "unsafe to push down" flag test in
  `check_output_expressions()` (subquery pull-up / pushdown safety checks in
  the optimizer). `[from-mailing-list]`
- The consequence of a wrong safety test is either an illegally-pushed
  expression (correctness bug) or a needlessly-blocked pushdown (lost
  optimization). `[inferred]`
- Lives in `allpaths.c`'s subquery-pushdown safety machinery. `[inferred]`

**Why it matters:** the pushdown-safety predicates are exactly the kind of
"confident but subtly wrong" logic the corpus warns about; a flipped test
silently changes query results. See [[knowledge/subsystems/optimizer.md]] and
[[knowledge/architecture/planner.md]].
