## 1. Missing FSM Update when Updating VM On-access

- **Author:** Andrey Borodin — 2026-07-04 18:41
- **URL:** <https://www.postgresql.org/message-id/2B4BE330-EE88-4C90-93CC-AA75DD127BE8%40yandex-team.ru>

Bullets:
- Reports a free-space-map staleness bug in the *on-access* visibility-map
  repair path: when a backend clears a VM bit lazily during a read (the
  "on-access" VM correction, distinct from the VACUUM-driven update), the
  matching FSM category for that heap page is **not** refreshed, so the FSM can
  advertise a page as having space it no longer has (or vice-versa) until the
  next VACUUM re-derives it.
- The consequence is not corruption but *placement inefficiency* — the inserter
  may repeatedly probe a page the FSM mis-describes, or skip a page that in fact
  has room, degrading insert locality on tables that see heavy on-access VM
  churn between vacuums.
- Proposes coupling the FSM update to the same code path that flips the VM bit,
  so the two page-level metadata structures stay consistent without waiting for
  a vacuum pass.

**Why it matters:** the FSM↔VM↔heap three-way consistency story is exactly the
recurring `gap:fsm` the user-question-harvester has flagged four windows
running. This thread is a concrete instance of *when* the FSM drifts, which is
the load-bearing question a future `freespace-map` subsystem doc has to answer.
`[[knowledge/subsystems/storage-freespace.md]]` (if/when written) should anchor
its "who updates the FSM, and when" section on this path.

## 2. GIN index fast list search may become un-interruptible for a long time

- **Author:** Kirill Reshke — 2026-07-04 17:38
- **URL:** <https://www.postgresql.org/message-id/CALdSSPisYC-zHiWJU36AP-zrPN-im-gEZrK-QD8TPn3Vb8xMvw%40mail.gmail.com>

Bullets:
- The GIN *pending-list* (fast-update) scan path can spin over a large pending
  list without a `CHECK_FOR_INTERRUPTS()`, so a query scanning a GIN index whose
  pending list has grown large (fast-update on, infrequent vacuum) becomes
  effectively **un-cancellable** — `Ctrl-C` / statement_timeout / query cancel
  don't take effect until the scan of the pending list completes.
- Root cause is the tight inner loop in `ginscan`/`scanPendingInsert` lacking an
  interrupt check; the fix is the standard `CHECK_FOR_INTERRUPTS()`-in-the-loop
  pattern, placed so it doesn't add measurable per-tuple overhead on the hot
  path.
- Ties into the broader design tension of GIN fast-update: the pending list
  trades insert speed for read-time work, and an unbounded pending list is
  already a known operational footgun; the interruptibility gap makes a slow
  scan also an *unkillable* one.

**Why it matters:** a textbook example of the PG "every long loop needs a
`CHECK_FOR_INTERRUPTS()`" idiom, and a good pin for
`[[knowledge/idioms/interrupt-checking.md]]`. Also a live GIN-fastupdate
behaviour note for any `access-gin` subsystem work.

## 3. support CREATE INDEX on virtual generated column

- **Author:** ZizhuanLiu X-MAN — 2026-07-05 03:04, 09:00
- **URL:** <https://www.postgresql.org/message-id/tencent_DAA3363C495B3FADD6C4CB778EA51AF55D06%40qq.com>

Bullets:
- Proposes lifting the current restriction that forbids building an index on a
  **virtual** (non-stored) generated column. Because a virtual generated
  column's value isn't materialised on the heap, an index on it must recompute
  the generation expression at index-build and insert time — effectively an
  expression index whose expression is pinned to the column's generation
  definition.
- Raises the semantic question of what happens when the generation expression is
  changed or the column redefined: the index would need to be invalidated /
  rebuilt, paralleling how expression indexes already track their underlying
  expressions.
- Touches `DefineIndex`, the index-AM `ambuild` inputs (the tuple slot must
  evaluate the generation expr), and the dependency machinery that ties the
  index to the column definition.

**Why it matters:** intersects the index-AM build path with the generated-column
executor plumbing — squarely in scope for
`[[knowledge/skills/access-method-apis]]` and the parser/nodes handling of
generated-column expressions. A clean case study of "an index over a computed,
never-stored value."

## 5. Fix race condition in pg_get_publication_tables with concurrent DROP TABLE

- **Author:** Bharath Rupireddy — 2026-07-04 05:26
- **URL:** <https://www.postgresql.org/message-id/CALj2ACUhhxChS0viUYP7hGqj%2BL5U38UPKXxAPX-bpHraAzO5mw%40mail.gmail.com>

Bullets:
- `pg_get_publication_tables()` enumerates a publication's member tables, but
  between resolving the membership list and opening each relation a concurrent
  `DROP TABLE` can remove one, producing a cache-lookup failure / spurious error
  in the middle of what should be a read-only introspection call.
- The fix takes the appropriate lock (or tolerates the missing relation) so the
  function skips a concurrently-dropped table rather than erroring — the standard
  "snapshot the OID list, then re-check existence under lock" pattern for
  catalog-enumerating SRFs.
- Relevant to logical-replication tooling and any monitoring query that walks
  `pg_publication_tables`, which can now fail transiently under DDL churn.

**Why it matters:** a clean instance of the "enumerate-then-open under
concurrent DDL" hazard that recurs across catalog-walking SRFs (cf. the same
class in `pg_get_*` functions). Reinforces the locking-order material in
`[[knowledge/skills/locking]]`.

## 6. explain plans for foreign servers

- **Author:** dinesh salve — 2026-07-04 17:07
- **URL:** <https://www.postgresql.org/message-id/CAP%2BB4TBATGojLy4ySehGRLWYHyt9WW-ggL%2B3T%3DGSn006kiuqqQ%40mail.gmail.com>

Bullets:
- Proposes surfacing the **remote** plan of a foreign-scan node in `EXPLAIN`, so
  a query against a `postgres_fdw` foreign table can show not just the local
  `Foreign Scan` node but the plan the remote server would use for the pushed-down
  query — closing a long-standing debuggability gap for distributed queries.
- Implementation touches the FDW `ExplainForeignScan` callback and requires the
  remote server to return its `EXPLAIN` for the deparsed remote query, then
  splicing that text into the local plan output under the foreign-scan node.
- Raises questions about version-skew (remote plan format differs by PG version),
  cost of an extra remote round-trip during `EXPLAIN`, and whether it should be
  gated behind an option like `EXPLAIN (REMOTE)`.

**Why it matters:** an executor/EXPLAIN-plumbing feature at the FDW boundary —
relevant to `[[knowledge/skills/executor-and-planner]]`'s EXPLAIN-wiring section
and to anyone reasoning about push-down debuggability. Also a nice example of the
`ExplainForeignScan` extension point.

## 7. SQL/JSON: json_table PLAN clause

- **Author:** Alexander Korotkov — 2026-07-04 21:02
- **URL:** <https://www.postgresql.org/message-id/CAPpHfdu-1oDzhKHXRF0vVSqab5Q3V5Sx1iQ5%2BtOmL_%2BnqY5_tA%40mail.gmail.com>

Bullets:
- Advances the `JSON_TABLE(... PLAN ...)` clause — the SQL/JSON standard's
  mechanism for explicitly specifying how nested path columns join (inner/outer,
  cross/union) rather than relying on the default sibling/child plan derivation.
- The PLAN clause lets a query author control the row-multiplication semantics
  when a `JSON_TABLE` has multiple nested column paths, which matters for
  correctness of the flattened output, not just performance.
- Builds on the already-committed core `JSON_TABLE` machinery; this is the
  remaining standard-conformance piece that governs multi-path join shape.

**Why it matters:** SQL/JSON is one of the larger standard-conformance surfaces
landing across recent cycles; the PLAN clause is the executor/parse-analysis
piece that decides how nested paths compose. Feeds a future SQL/JSON note and
touches the parser/nodes representation of `JsonTable*`.

## 8. Don't use the deprecated and insecure PQcancel in our frontend tools anymore

- **Author:** Heikki Linnakangas, Zsolt Parragi — 2026-07-04 19:11, 21:29, 22:57
- **URL:** <https://www.postgresql.org/message-id/aea19cfb-0518-4485-b3d8-c7647d450186%40iki.fi>

Bullets:
- Moves the in-tree frontend tools (psql, pg_dump, pg_amcheck, etc.) off the
  legacy blocking `PQcancel()` / `PQgetCancel()` API and onto the newer
  `PQcancelCreate()` + `PQcancelBlocking()`/`PQcancelStart()` family introduced
  to support the encrypted (TLS/GSS) cancel-request protocol.
- The old `PQcancel` path sends the cancel key over a fresh **unencrypted**
  connection, which leaks the cancel key and can't traverse setups that require
  encryption — the motivation for the modern API and for finally retiring the
  old one from our own tools.
- Part of a multi-message cleanup; also intersects the parallel effort to remove
  `getpgusername()` and other deprecated libpq/portability crumbs (same author,
  same day).

**Why it matters:** the query-cancel protocol is a subtle corner of the FE/BE
wire protocol (cancel keys, the separate cancel connection). This is a good
anchor for `[[knowledge/idioms/query-cancellation.md]]` and for libpq-facing
security reasoning.

  so it now does materially more catalog work / worse join order, making `\d` on

  to `describe.c`'s SQL made the common case pay for the rare case" regression.

**Why it matters:** a compact case study in how a catalog-query "cleanup" can
regress interactive tooling, and how it's diagnosed (attribute the regression to
a specific SHA, then reshape). Illustrative for anyone touching `describe.c`.

## 12. Yet another bug in nondeterministic LIKE

- **Author:** Tom Lane — 2026-07-04 17:59
- **URL:** <https://www.postgresql.org/message-id/391592.1783187986%40sss.pgh.pa.us>

Bullets:
- Another correctness bug in `LIKE`/pattern matching against a column with a
  **nondeterministic** collation (where multiple byte sequences compare equal),
  in which the matcher's backtracking makes an assumption that only holds for
  deterministic collations, yielding wrong match/no-match results.
- Nondeterministic-collation pattern matching has been a repeated source of edge
  cases because `LIKE`'s character-at-a-time matching model doesn't map cleanly
  onto "these two substrings are equal but not byte-identical."
- Fix constrains the matcher's shortcutting to the cases where the collation's
  equality actually permits it.

**Why it matters:** nondeterministic collations are a subtle correctness surface
that keeps producing bugs; a running note of *which* string operations have
tripped on them (LIKE, now repeatedly) is useful for anyone touching
`like_match.c` / `varlena.c` collation handling.
