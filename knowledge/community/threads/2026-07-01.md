## 1. Deadlock detector fails to activate on a hot standby replica

- Author: Xuneng Zhou (06-30 03:14, continuing).
  Archive: <https://www.postgresql.org/search/?m=1&q=Deadlock+detector+fails+to+activate+on+a+hot+standby+replica>
- Still the open corner from the 06-30 window: on a hot standby the deadlock
  detector can stay dormant for a recovery-conflict buffer-pin wait — the
  startup process replaying a cleanup record and a backend holding a pin wedge
  without the deadlock timeout ever escalating to a check.
- Discussion this window narrows the suspect to the deadlock-timer arming around
  `ResolveRecoveryConflictWithBufferPin`: the standby path uses a different
  wait/wakeup shape than a normal backend's `ProcSleep`, and the
  `enable_timeout_after(DEADLOCK_TIMEOUT, …)` arming can be skipped or lost.
- Pairs directly with the 06-28 "signal storm" report as the two failure modes
  of one fragile timer — no signal here, too many signals there.
- **Why it matters:** a dormant deadlock detector on a standby turns a transient
  recovery conflict into an indefinite hang that never logs `deadlock detected`
  — the exact silent-LWLock-style hang the locking skill's lldb-attach recipe
  exists to diagnose. `[from-website]` / [inferred]
  [[knowledge/subsystems/storage-lmgr.md]], [[knowledge/idioms/deadlock-detection.md]]

## 2. 048_vacuum_horizon_floor.pl hangs due to wakeup lost inside LockBufferForCleanup

- Author: Fujii Masao, Xuneng Zhou (06-30 01:40 / 02:25).
  Archive: <https://www.postgresql.org/search/?m=1&q=048_vacuum_horizon_floor.pl+hangs+wakeup+lost+LockBufferForCleanup>
- A TAP recovery test intermittently hangs because a wakeup is lost inside
  `LockBufferForCleanup`: the super-exclusive (cleanup) buffer lock waiter parks
  on its latch/semaphore and a concurrent unpin doesn't reliably re-signal it,
  so the cleanup lock is never granted.
- The window's discussion ties this to the same standby recovery-conflict
  machinery as thread 1 — `LockBufferForCleanup` is what the startup process
  calls when replaying VACUUM/HOT-prune cleanup records, and the missed wakeup
  is the "no signal" half of the standby-lock fragility pair.
- Fix direction under review is tightening the pin-count-reaches-zero wakeup so
  the cleanup waiter is signalled deterministically rather than relying on a
  racy re-check.
- **Why it matters:** `LockBufferForCleanup` sits on the VACUUM ↔ replay hot
  path; a lost wakeup there is both a test flake *and* a latent production hang
  on any standby replaying cleanup traffic under concurrent pins. `[from-website]`
  [[knowledge/subsystems/storage-buffer.md]], [[knowledge/data-structures/bufferdesc-state.md]],
  [[knowledge/idioms/vacuum-hot-prune.md]]

## 3. GetBufferDescriptor() being called for local buffers from MarkBufferDirtyHint()

- Author: Ayush Tiwari, Ashutosh Bapat (06-30 19:17 → 07-01 05:33).
  Archive: <https://www.postgresql.org/search/?m=1&q=GetBufferDescriptor+local+buffers+MarkBufferDirtyHint>
- `MarkBufferDirtyHint()` on a **local** (temp-table) buffer reaches through
  `GetBufferDescriptor()`, which indexes the *shared* `BufferDescriptors` array —
  the wrong descriptor array for a negative/local buffer number. It happens to
  not crash today only because the hint-dirty path is lenient, but it's reading
  the wrong `BufferDesc`.
- The fix threads a `BufferIsLocal()` branch into the hint-dirty path so local
  buffers go through `LocalBufferDescriptors` / the local `dirty` bookkeeping
  instead of the shared array and its `buf_state` atomics.
- Reviewers flag the broader audit: how many other "set a hint bit, mark dirty"
  callers assume shared-buffer geometry and silently mis-index for local
  buffers.
- **Why it matters:** local vs shared buffer descriptor confusion is a classic
  temp-table corruption footgun — a hint bit written against the wrong
  `buf_state` word can flip dirty/valid flags on an unrelated shared buffer.
  Direct continuation of the 06-30 local-buffer correctness thread. `[from-website]`
  [[knowledge/subsystems/storage-buffer.md]], [[knowledge/data-structures/bufferdesc-state.md]],
  [[knowledge/idioms/hint-bits-setbufferdirty.md]]

## 4. autovacuum launcher crash: assert in pgstat_count_io_op (IOOP_EXTEND on pg_database's VM)

- Author: Ewan Young (06-30 05:41).
  Archive: <https://www.postgresql.org/search/?m=1&q=autovacuum+launcher+crash+assert+pgstat_count_io_op+IOOP_EXTEND>
- An assertion fires in `pgstat_count_io_op` when the autovacuum launcher extends
  the **visibility map** fork of `pg_database` (a shared/nailed catalog): the
  IO-op stat path asserts on an object/context combination
  (`IOOBJECT_*` × `IOOP_EXTEND`) that the launcher wasn't expected to hit for a
  shared relation's VM fork.
- Root cause under discussion is that the launcher touches shared-catalog VM
  pages in a code path whose `pgstat` IO-context classification doesn't allow
  `EXTEND`, so the cumulative-stats accounting trips its own invariant.
- Fix candidates: either teach the IO-stat classifier that this
  object/context/op triple is legal, or stop the launcher from extending that
  fork in the first place.
- **Why it matters:** it's an assert-build crash of the *launcher* — the process
  that schedules all autovacuum work — so on a cassert cluster one bad VM extend
  on a shared catalog stops autovacuum fleet-wide. Sits at the VM ↔ pgstat-IO
  seam this week's cluster keeps re-surfacing. `[from-website]`
  [[knowledge/idioms/autovacuum-launcher.md]], [[knowledge/idioms/visibility-map-update.md]],
  [[knowledge/data-structures/pgstat-counter.md]]

## 5. Add per-backend lock statistics

- Author: Bertrand Drouvot, Michael Paquier (06-30 05:22 / 08:07).
  Archive: <https://www.postgresql.org/search/?m=1&q=Add+per-backend+lock+statistics>
- Proposes per-backend lock statistics (acquisitions, waits, wait-time) exposed
  through the cumulative-stats framework, layered on top of the `pg_stat_lock`
  double-precision `wait_time` work that landed/matured earlier in the week.
- Design debate this window is granularity and cost: per-backend rows vs a
  fixed-size shared array indexed by backend, and whether the counters live in
  `PgBackendStatus` / the pgstat shared hash or a dedicated stats kind.
- Ties into the parallel `Add pg_stat_kind_info system view` and "unregistered
  stats kind" crash threads — the pgstat "kind" registry is getting exercised
  hard as several new stat kinds land at once.
- **Why it matters:** lock observability has historically been coarse
  (`pg_locks` is a snapshot, not a counter); per-backend accumulated lock stats
  would make contention attributable to a specific backend/query without
  sampling. The design choice sets how much shared memory every backend pays.
  `[from-website]` / [inferred]

## 6. Fix HAVING-to-WHERE pushdown with mismatched operator families

- Author: Richard Guo, Tender Wang (06-30 09:22 → 07-01 05:58).
  Archive: <https://www.postgresql.org/search/?m=1&q=Fix+HAVING-to-WHERE+pushdown+with+mismatched+operator+families>
- Continuation of the 06-29 wrong-results report: the optimizer pushes a `HAVING`
  clause down into `WHERE` when it looks equivalent, but if the clause's operator
  belongs to a *different* operator family than the grouping/equality assumed by
  the transform, the pushdown changes result semantics.
- The fix gates the push-down on operator-family compatibility so a clause is
  only relocated when its operator genuinely matches the family the transform
  relies on; several v-N patches this window tighten the exact predicate.
- Reviewers cross-check against the collation/opfamily edge cases that produced
  earlier planner wrong-results this cycle — the recurring theme is transforms
  that assume "same operator name ⇒ same semantics."
- **Why it matters:** a `HAVING`→`WHERE` push that silently drops or admits rows
  is a wrong-results bug with no error and no crash — the highest-severity class
  of planner defect because the output simply lies. `[from-website]`
  [[knowledge/subsystems/optimizer.md]], [[knowledge/idioms/view-pushdown-via-rewriter.md]]

## 7. Handle concurrent drop when doing whole database vacuum

- Author: Michael Paquier, Nathan Bossart, surya poondla, cca5507
  (06-30 04:47 → 07-01 07:21).
  Archive: <https://www.postgresql.org/search/?m=1&q=Handle+concurrent+drop+when+doing+whole+database+vacuum>
- A database-wide `VACUUM` builds its relation list up front, then a concurrent
  `DROP TABLE` removes one of those relations before vacuum reaches it — vacuum
  must skip the vanished OID gracefully rather than erroring on
  `try_relation_open` / a missing relcache entry.
- The window's patches harden the per-relation open in the whole-db loop to treat
  "relation disappeared since the list was built" as a skip, not a failure, and
  debate whether to re-check under lock or tolerate the open failure.
- Overlaps with the temp-table "accessing tables of other sessions" and
  `pg_get_publication_tables` concurrent-DROP threads — a cluster of "list built,
  then object dropped, now revalidate" races surfacing at once.
- **Why it matters:** whole-DB vacuum is routine maintenance; an unhandled
  concurrent drop turns a cron `VACUUM` into a hard error, and the fix pattern
  (tolerate disappearance between enumeration and open) recurs across every
  catalog-list-then-act path. `[from-website]`
  [[knowledge/idioms/vacuum-two-pass-heap.md]], [[knowledge/subsystems/access-heap.md]]

## 8. Proposal: Conflict log history table for Logical Replication

- Author: Peter Smith, shveta malik, Amit Kapila, Dilip Kumar
  (06-30 01:55 → 07-01 07:46, very active).
  Archive: <https://www.postgresql.org/search/?m=1&q=Proposal+Conflict+log+history+table+for+Logical+Replication>
- Proposes persisting apply-side conflicts (insert_exists, update_missing,
  delete_missing, update_origin_differs, …) into a queryable history table
  rather than only emitting them to the server log, so operators can audit and
  reconcile divergence after the fact.
- The long thread debates schema and retention: a system catalog vs a regular
  table in a known schema, per-subscription vs global, what columns
  (conflict type, relation, key, origin, LSN, timestamp) and who prunes it.
- Builds directly on the conflict-detection/`conflict resolution` machinery added
  for logical replication — this is the "record what the resolver did" layer on
  top of the resolution policy.
- **Why it matters:** logical-replication conflicts are today fire-and-forget in
  the log; a structured history table is what makes multi-master / bidirectional
  setups operable, because you can prove which side won and when. `[from-website]`
  [[knowledge/subsystems/replication.md]], [[knowledge/idioms/apply-conflict-resolution.md]],

## 9. Fix race condition in pg_get_publication_tables with concurrent DROP TABLE

- Author: Masahiko Sawada (06-30 17:37).
  Archive: <https://www.postgresql.org/search/?m=1&q=Fix+race+condition+in+pg_get_publication_tables+with+concurrent+DROP+TABLE>
- `pg_get_publication_tables()` expands a publication's membership (including
  `FOR TABLES IN SCHEMA` / `FOR ALL TABLES` and row-filter/column-list handling),
  and a concurrent `DROP TABLE` between enumeration and per-table processing
  leaves a dangling OID that the function then tries to open.
- The fix revalidates each table under the proper lock (or skips vanished OIDs),
  mirroring the whole-DB-vacuum concurrent-drop fix in thread 7 — same
  "enumerate then act" race, different catalog surface.
- Directly relevant to the several `Support EXCEPT for … publications` threads
  active this window, which all extend publication membership computation and
  inherit the same race surface.
- **Why it matters:** publication membership is consulted on every logical-
  decoding change; a race that errors (or worse, mis-includes a relation) can
  break a subscription's initial sync or stall the walsender. `[from-website]`
  [[knowledge/subsystems/replication.md]], [[knowledge/idioms/output-plugin-callbacks.md]]

## 10. Add SPLIT PARTITION / MERGE PARTITIONS commands

- Author: Alexander Korotkov, Pavel Borisov (06-30 11:06 / 11:25 / 19:38).
  Archive: <https://www.postgresql.org/search/?m=1&q=Add+SPLIT+PARTITION+MERGE+PARTITIONS+commands>
- Long-running feature to add `ALTER TABLE … SPLIT PARTITION` and
  `MERGE PARTITIONS` DDL, letting a range/list partition be divided or coalesced
  in place instead of manual detach + create + data-move.
- This window's discussion is lock strength and data movement: what lock level

  and transactional/atomicity guarantees when the split boundary rewrites rows.

- **Why it matters:** in-place SPLIT/MERGE is a long-requested partitioning
  ergonomics gap; getting the lock level and atomicity right is the hard part,
  since a too-weak lock risks concurrent inserts landing in the wrong child mid-
  split. `[from-website]` / [inferred]

## 11. Tepid: selective index updates for heap relations

- Author: Greg Burd (06-30 17:21, new proposal).
  Archive: <https://www.postgresql.org/search/?m=1&q=Tepid+selective+index+updates+for+heap+relations>
- New proposal ("Tepid") to make heap updates selectively skip index maintenance
  for indexes whose keys didn't change — a generalization/relative of the HOT
  optimization that would extend "don't touch the index" beyond the strict
  same-page HOT conditions.
- Early design discussion: how to decide per-index whether an update is a no-op
  for that index, how it interacts with HOT chains and index-only-scan
  visibility, and what new bookkeeping the heap tuple/line-pointer needs.
- Positioned against the existing HOT machinery — reviewers will want to know
  where Tepid wins that HOT can't (cross-page updates, subset-of-indexes
  unchanged) and what it costs in vacuum/prune complexity.
- **Why it matters:** index write amplification on update is a dominant cost for
  wide-index tables; any scheme that safely skips unchanged-index maintenance is
  high-value, but the correctness bar (visibility, amcheck, vacuum) is severe.
  `[from-website]` / [inferred]
  [[knowledge/subsystems/access-heap.md]], [[knowledge/idioms/vacuum-hot-prune.md]],

## 12. Batching in executor

- Author: Amit Langote (07-01 09:18, continuing).
  Archive: <https://www.postgresql.org/search/?m=1&q=Batching+in+executor>
- Ongoing architectural thread on batch-at-a-time execution: pushing multiple
  tuples through executor nodes per `ExecProcNode` call instead of the strict
  one-tuple-per-call volcano model, to cut per-tuple dispatch overhead and open
  the door to vectorized expression evaluation.
- The discussion weighs how batching threads through the `PlanState` /
  `TupleTableSlot` contract, which nodes can batch transparently vs need new
  APIs, and how it coexists with the existing single-tuple path during a long
  transition.
- Related to the day's `Improve pg_stat_statements scalability` and
  atomics-cleanup threads in spirit — all are executor/infra throughput work
  targeting the next cycle rather than a bug fix.
- **Why it matters:** the volcano one-tuple-per-call model is PostgreSQL's
  oldest executor bottleneck; a credible batching contract is the foundation any
  future vectorization would build on, so the API shape chosen here is
  load-bearing for years. `[from-website]` / [inferred]
  [[knowledge/subsystems/executor.md]], [[knowledge/data-structures/planstate.md]],
  [[knowledge/data-structures/tupletableslot.md]]
