## 1. pg_xmin_horizon: a system view of everything pinning the xmin horizon
- **Author / time:** Scott Ray, 2026-07-12 00:22
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Proposes a system view that enumerates every source currently holding
  back the global xmin horizon — long-running backends, replication
  slots, prepared xacts, `hot_standby_feedback` standbys — in one place.
- Today diagnosing "why won't VACUUM remove dead tuples" means manually
  joining `pg_stat_activity`, `pg_replication_slots`, `pg_prepared_xacts`
  and `pg_stat_replication`; the view collapses that into one query.
- Pairs with thread #2 (same author, autovacuum-side reporting).
- **Why it matters:** the xmin horizon is the single most common cause of
  bloat that DBAs can't explain; a first-class view is a recurring wishlist
  item. `[[knowledge/skills/snapshot-management]]`,
  `[[knowledge/skills/vacuum-autovacuum]]`.

## 2. Report oldest xmin source when autovacuum cannot remove tuples
- **Author / time:** Scott Ray, 2026-07-12 00:52
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Complements #1 from the autovacuum side: when a vacuum leaves tuples
  un-removable because they're newer than the horizon, log *which* source
  pinned the horizon (slot name / backend PID / prepared-xact gid).
- Turns the silent "N dead tuples could not be removed yet" line into an
  actionable "…because slot `sub_orders` holds xmin 12345".
- **Why it matters:** closes the observability gap between "vacuum ran but
  bloat persists" and the root cause, without a manual view lookup.
  `[[knowledge/skills/vacuum-autovacuum]]`.

## 3. Introduce XID age based replication slot invalidation
- **Author / time:** Bharath Rupireddy, 2026-07-12 20:26
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Adds an age-based (rather than size-based `max_slot_wal_keep_size`)
  invalidation policy so a slot that pins `catalog_xmin`/`xmin` past a
  configurable XID age gets invalidated before it threatens wraparound.
- Directly addresses the failure mode #1/#2 diagnose: an abandoned slot
  silently blocking the horizon until the cluster is near wraparound.
- **Why it matters:** wraparound-by-forgotten-slot is a real outage class;
  an age GUC gives operators a wraparound-safety valve independent of WAL
  volume. `[[knowledge/skills/logical-replication]]`,
  `[[knowledge/skills/vacuum-autovacuum]]`.

## 4. sequencesync worker race with REFRESH SEQUENCES
- **Author / time:** Amit Kapila, 2026-07-13 03:23 (multiple follow-ups)
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Reports a race between the new logical-replication **sequence sync**
  worker and a concurrent `ALTER SUBSCRIPTION ... REFRESH SEQUENCES`: the
  worker can be mid-sync when the refresh rewrites the sequence-state
  catalog, leaving a sequence half-synced.
- Part of the PG18/19 sequence-replication push (sequences over logical
  replication were long a headline gap).
- **Why it matters:** sequence replication is a marquee new feature; a
  sync-vs-refresh race is exactly the kind of correctness bug that must be
  nailed before it ships stable. `[[knowledge/skills/logical-replication]]`.

## 5. Use streaming read I/O when enabling data checksums online
- **Author / time:** Enrique Sanchez, 2026-07-12 14:49 (multiple follow-ups)
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Converts the online `pg_enable_data_checksums` background worker's
  full-relation scan from `ReadBuffer` loops to the `read_stream_*` API so
  the checksum-enable sweep issues prefetch and overlaps I/O.
- Another consumer migrating onto the PG17/18 read-stream infrastructure;
  the checksum sweep is I/O-bound and an obvious prefetch beneficiary.
- **Why it matters:** enabling checksums on a live large cluster is slow
  and I/O-heavy; read-stream prefetch is the standard remedy and a clean
  incremental win. `[[knowledge/skills/aio-readstream]]`,
  `[[knowledge/skills/buffer-manager]]`.

## 7. Batching in executor
- **Author / time:** Denis Smirnov, 2026-07-12 17:08
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Revives the perennial "process tuples in batches, not one-at-a-time
  through `ExecProcNode`" direction — vectorized/batched execution to
  amortize per-tuple dispatch overhead across the node tree.
- Touches the core `ExecProcNode` volcano model; any batching design has
  to reconcile with the existing pull-based `TupleTableSlot` protocol.
- **Why it matters:** executor batching is the single biggest structural
  lever for analytical throughput in PG and a recurring long-horizon RFC.
  `[[knowledge/skills/executor-and-planner]]`.
