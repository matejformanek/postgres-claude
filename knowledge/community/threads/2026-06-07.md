## 1. GetBufferDescriptor() called for local buffers from MarkBufferDirtyHint()

- **Author:** Ashutosh Bapat (08:07)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606060000
- Bullets:
  - `MarkBufferDirtyHint()` is reaching for `GetBufferDescriptor()` even on
    **local** (temp-table) buffers, where the shared-buffer descriptor array
    is the wrong structure — local buffers have their own `LocalBufferDescriptors`.
  - Hint-bit dirtying on local buffers should never touch the shared
    descriptor table or the shared usage-count machinery; this is a
    correctness/abstraction-leak bug in the buffer-access path.
  - Likely fix is an `BufferIsLocal()` guard routing to the local-buffer
    descriptor accessor before any shared-buffer bookkeeping.
- **Why it matters:** `MarkBufferDirtyHint` is the hint-bit / FPI-suppressing
  dirty path; getting the local-vs-shared dispatch wrong risks reading the
  wrong descriptor for temp relations. Touches the core invariants in
  [[knowledge/subsystems/storage-buffer.md]].

## 2. index prefetching

- **Author:** Peter Geoghegan (00:52)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606060000
- Bullets:
  - Continuation of the long-running index-prefetch effort — driving the
    read-stream / AIO machinery from index scans so heap fetches for upcoming
    TIDs are issued ahead of need.
  - Sits on top of the read-stream API; the design question is how far ahead
    an index scan may look without breaking visibility/locking ordering.
  - Recurs alongside the AIO/io_uring cluster the user-question-harvester has
    flagged repeatedly — a strong driver for a `storage-aio` corpus doc.
- **Why it matters:** Index prefetch is one of the highest-leverage PG20-era
  performance threads; it crosses the executor index-scan node and the buffer
  manager. cf. [[knowledge/subsystems/access-nbtree.md]],
  [[knowledge/architecture/access-methods.md]].

## 3. Fix race in ReplicationSlotRelease for ephemeral slots

- **Author:** Zhijie Hou (Fujitsu) (09:35); reply Xuneng Zhou (13:25)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606060000
- Bullets:
  - A race in `ReplicationSlotRelease()` when the slot is **ephemeral**
    (RS_EPHEMERAL): the slot can be concurrently dropped/cleaned while a
    backend is releasing it.
  - Ephemeral slots are the transient slots created e.g. during base backup /
    slot creation that get promoted to persistent or dropped on release.
  - Fix narrows the window where the slot's in-memory state is mutated without
    the slot's mutex / proper ordering.
- **Why it matters:** Replication-slot lifecycle races surface as crashes or
  orphaned slots on standbys. cf. [[knowledge/subsystems/replication.md]],
  [[knowledge/architecture/replication.md]].

## 4. PG19 FK fast path: OOB write and missed FK checks during batched apply

- **Authors:** Nikolay Samokhvalov (08:30), Amit Langote (09:13)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606060000
- Bullets:
  - The PG19 foreign-key "fast path" (batched FK checking) has an
    **out-of-bounds write** and can **miss FK checks** during batched apply —
    a correctness bug in newly-added code, not legacy.
  - Reported with a reproducer; Amit Langote (author of much of the FK/
    partition machinery) engaged same-day.
  - Both a memory-safety issue (OOB write) and a semantic issue (FK
    constraint silently not enforced for some rows in the batch).
- **Why it matters:** A missed FK check is silent data-integrity corruption,
  and an OOB write is a security-adjacent memory bug — same dual class as the
  A7/A11 hardening findings. cf. [[knowledge/subsystems/access-heap.md]],
  [[knowledge/issues/utils.md]].

## 5. Report oldest xmin source when autovacuum cannot remove tuples

- **Author:** Scott Ray (18:40)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606060000
- Bullets:
  - Observability patch: when autovacuum can't remove dead tuples because the
    horizon is held back, log **which** source pins the oldest xmin (a
    long-running xact, a replication slot, a prepared xact, `hot_standby_feedback`).
  - Today the DBA sees "tuples not removable" without the *why*; this attaches
    the blocking source to the message.
  - Surfaces the `ComputeXidHorizons` / `GlobalVisState` accounting at the
    user-visible layer.
- **Why it matters:** The "why won't my table vacuum" question is one of the
  most common MVCC operational pains; naming the xmin holder closes the loop.
  cf. [[knowledge/architecture/mvcc.md]].

## 6. (watch) Don't call ereport(ERROR) from recovery target GUC assign hooks

- **Author:** Scott Ray (20:10); reply JoongHyuk Shin (10:30)
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606060000
- Bullets:
  - GUC `assign_*` hooks must not `ereport(ERROR)` — they run at a point where
    throwing is unsafe; validation belongs in the `check_*` hook.
  - The recovery-target GUCs (`recovery_target_*`) violate this; patch moves
    the rejection into the check phase.
  - Cross-links the GUC machinery rules in the `gucs-bgworker-parallel` skill
    (check vs assign hook contract).
- **Why it matters:** A reliable, recurring GUC-hook footgun; good corpus
  cross-link material for the GUC idiom notes. `[from-website]`
