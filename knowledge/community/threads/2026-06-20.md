## 1. Checkpointer write combining — Melanie Plageman

- Continuation of the effort to coalesce adjacent dirty-buffer writes the
  checkpointer issues, reducing syscall count and letting the kernel/AIO path
  see larger contiguous I/Os.
- Active reply 06-19 22:13; sits at the intersection of the buffer manager's
  writeback path and the checkpointer's sorted-scan flushing.
- Pairs with the broader AIO/io_method work landing this cycle (io_uring fd-leak
  fix #6617 in the CF shortlist).

**Why it matters:** fewer, larger checkpoint writes is one of the highest-leverage
durability-path wins; touches the BufferDesc writeback ordering this corpus tracks.
[[knowledge/idioms/checkpoint-coordination.md]] [[knowledge/subsystems/storage-buffer.md]]

## 2. Add a hook for handling logical decoding messages on subscribers — Masahiko Sawada

- New thread (06-19 22:33). Proposes an extension hook on the *apply* side so
  subscribers can react to logical-decoding messages (`pg_logical_emit_message`)
  rather than only the publisher/output-plugin side.
- Opens a customization point for conflict-handling / auditing extensions in the
  apply worker.
- Lands near the existing "Conflict log history table for Logical Replication"
  discussion (also active this window) — both push apply-side observability.

**Why it matters:** a sanctioned apply-worker hook would let extensions
participate in logical replication without forking the apply loop.
[[knowledge/idioms/logical-decoding-snapshot.md]] [[knowledge/architecture/replication.md]]

## 3. Fix tuple deformation with virtual generated NOT NULL columns — David Rowley / Tom Lane

- Active 06-19 (Rowley 03:30, Lane 03:34). This is the fix for the
  virtual-generated-column tuple-deformation bug that had been red on multiple
  Debian-Sid JIT buildfarm animals (`c=0 vs 42`) for several days.
- Upstream-watcher reported the JIT-side fix (`dc511678`) closed the animals;
  this thread is the deform-path correctness discussion behind it.

**Why it matters:** virtual generated columns interact with the JIT'd
tuple-deform fast path; a NOT NULL on a never-stored column is exactly the edge
the deformer must not trust. [[knowledge/subsystems/access-heap.md]]
[[knowledge/subsystems/jit.md]] [[knowledge/subsystems/executor.md]]

## 4. check_stack_depth() vs. tail-recursion optimization — Tom Lane

- New thread (06-19 17:49). Compilers performing tail-call optimization can
  collapse recursive C frames, so `check_stack_depth()`'s stack-base arithmetic
  may under-count actual depth and miss runaway recursion before a real overflow.
- Raises whether the guard needs hardening (or a barrier) against aggressive
  `-O2` tail-recursion on recursive walkers.

**Why it matters:** `check_stack_depth()` is the backend's only line of defense
against C-stack overflow in deeply recursive parse/plan/exec walkers; a TCO hole
turns a clean ERROR into a SIGSEGV. [[knowledge/subsystems/tcop.md]]

## 5. Fix race in ReplicationSlotRelease for ephemeral slots — Amit Kapila / Xuneng Zhou

- Active 06-19 (Kapila 12:08) and 06-20 (Zhou 06:41, Kapila 09:42). A race in
  releasing RS_EPHEMERAL slots where the slot can be cleaned up / reset
  concurrently with release.
- Ephemeral slots are the short-lived slots used during slot creation and base
  backup; the window is narrow but reachable under concurrent drop.

**Why it matters:** slot lifecycle races corrupt the in-memory ReplicationSlot
array and can wedge a walsender or leak a slot; the release path's locking is
the invariant under scrutiny. [[knowledge/idioms/replication-slot-advance.md]]
[[knowledge/architecture/replication.md]]

## 6. Improve UNION's output rowcount estimate — Richard Guo

- New thread (06-20 02:21). Refines the selectivity/rowcount the planner assigns
  to a `UNION` (dedup) result, which currently over- or under-estimates the
  post-dedup cardinality feeding parent nodes.
- Better top-of-UNION estimates ripple into join-order and memory (hash/sort)
  sizing above the set-op.

**Why it matters:** set-op rowcount feeds every cost decision above it; a sharper
estimate is a cheap, broad planner win. [[knowledge/subsystems/optimizer.md]]

## 7. Adding REPACK [concurrently] — Antonin Houska

- Active 06-19 06:48. Proposes a `REPACK` command for online table rewrite /
  compaction (a CLUSTER/VACUUM FULL successor that holds weaker locks), with a
  CONCURRENTLY mode to avoid the AccessExclusiveLock.
- Touches heap rewrite, snapshot management during the rewrite, and
  index rebuild coordination.

**Why it matters:** online compaction without an exclusive lock is a long-standing
operational gap; the rewrite + concurrent-DML reconciliation is the hard part.
[[knowledge/subsystems/access-heap.md]]

## 8. Compile warnings under -DMEMORY_CONTEXT_CHECKING and no assertions — Michael Paquier

- New thread (06-20 07:40). The `MEMORY_CONTEXT_CHECKING` build knob, when
  enabled *without* `--enable-cassert`, surfaces unused-variable / set-but-unused
  warnings in the mcxt code paths.
- Cleanup so the checking build is warning-clean independent of assertions.

**Why it matters:** keeps the memory-context debugging build (used to chase
overruns/leaks) compilable standalone; touches the AllocSet/mcxt guard code this
corpus documents. [[knowledge/idioms/memory-context-allocset-internals.md]]
[[knowledge/subsystems/utils-mmgr.md]]

## 9. Unexpected behavior after OOM errors — Matthias van de Meent / Michael Paquier / Alexander Lakhin

- Very active both days (van de Meent 06-19 07:18/07:22; Paquier 06-19 11:12,
  06-20 06:03/07:45; Lakhin 06-20 08:00). Investigates backend state left
  inconsistent after an out-of-memory ereport — contexts/flags not fully reset
  on the longjmp out.
- Intersects error-recovery (PG_TRY/sigsetjmp) and memory-context teardown.

**Why it matters:** OOM must be a clean transaction-abort, not a wedged backend;
any state surviving the error longjmp is a latent corruption source.
[[knowledge/subsystems/utils-mmgr.md]] [[knowledge/idioms/memory-contexts.md]]

## 10. Include sequences in publications created by pg_createsubscriber — vignesh C

- New thread (06-19 13:59). Makes `pg_createsubscriber` add sequences to the
  publications it generates, so the new PG19 sequence-replication machinery is
  actually exercised by the convert-standby-to-subscriber tool.
- Directly downstream of the sequence-in-logical-replication feature (see today's
  Shaun Thomas blog roundup item).

**Why it matters:** without this, sequences silently don't replicate through the
pg_createsubscriber path even though the underlying feature now supports it.
[[knowledge/architecture/replication.md]]

## 11. Multi-Entry Indexing for GiST & SP-GiST — Matthias van de Meent

- Active 06-19 08:26. Lets a single GiST/SP-GiST index entry expand to multiple
  index tuples per heap tuple (multi-key / multi-value indexing) — generalizing
  what GIN does to the tree AMs.
- Touches the AM insert path and the scan-side de-duplication of heap TIDs.

**Why it matters:** multi-entry support would let GiST/SP-GiST index array- and
range-decomposed keys natively. [[knowledge/idioms/spgist-insert-and-picksplit.md]]
[[knowledge/subsystems/access-nbtree.md]]

## 12. PROXY protocol support — Julien Riou

- Active 06-20 (05:46, 09:18). Adds HAProxy PROXY-protocol parsing to the backend
  listener so the real client address survives an L4 proxy, instead of the
  backend seeing the proxy's IP.
- Affects connection setup, `pg_hba.conf` matching, and log_line_prefix `%h`.

**Why it matters:** correct client-IP attribution behind a load balancer is a
real-world auth/audit gap; touches the libpq backend accept path.
[[knowledge/subsystems/libpq-backend.md]]
