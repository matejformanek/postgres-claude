## 1. GetBufferDescriptor() being called for local buffers from MarkBufferDirtyHint()

- **Author:** thread w/ Michael Paquier, Ashutosh Bapat, Andres Freund
  (Jun 10, 03:40 → 14:36) `[from-website]`
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606090000
- Bullets:
  - `MarkBufferDirtyHint()` is the hint-bit / non-WAL-logged dirty path (used for
    setting `PD_ALL_VISIBLE`, killed-index-tuple bits, etc.). The report is that
    it reaches `GetBufferDescriptor()` — the **shared**-buffer descriptor
    accessor — even when handed a **local** buffer (temp-table / local-buffer
    pool), where the correct accessor is `GetLocalBufferDescriptor()`. `[inferred]`
  - A shared accessor on a local buffer number indexes the wrong descriptor
    array; on most builds it reads a bogus `BufferDesc` rather than crashing,
    which is why it survived undetected — exactly the class of bug an assert or
    `BufferIsLocal()` guard at the top of the path is meant to catch. `[inferred]`
  - Andres Freund joining (14:36) signals this touches the AIO/bufmgr rework
    surface he owns; the fix is likely a `BufferIsLocal()` branch (mirroring the
    split already present in `MarkBufferDirty`) rather than a behavioral change. `[inferred]`
- **Why it matters:** the shared-vs-local buffer split is a load-bearing
  invariant of the buffer manager; calling a shared accessor on a local buffer
  is the same descriptor-array-confusion family the corpus tracks around
  `BufferDesc`/`LocalBufferDescriptors`. A hint-bit path is low-visibility but
  runs constantly. cf. [[knowledge/subsystems/storage-buffer.md]],
  [[knowledge/issues/include-storage.md]].

## 2. LLVM JIT: any JIT-compiled query crashes (SIGILL) on a libLLVM 19 + ASAN build

- **Author:** Henson Choi (Jun 10, 02:09) `[from-website]`
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606090000
- Bullets:
  - On a build linking **libLLVM 19** with **AddressSanitizer**, any query that
    actually JIT-compiles (expression eval / tuple deform past the cost
    threshold) dies with **SIGILL** — an illegal-instruction trap, i.e. the
    emitted machine code or its relocation is malformed for the host. `[inferred]`
  - SIGILL (not SIGSEGV) points at code generation rather than a data fault:
    likely an ABI / relocation-model mismatch between PG's ORC-JIT layer
    (`llvmjit.c`, `llvmjit_expr.c`) and LLVM 19's codegen under ASAN's
    instrumentation, or an unsupported instruction-set assumption. `[inferred]`
  - The ASAN qualifier matters: it suggests the sanitizer's shadow-memory or
    interceptor layout perturbs what the JIT emits/links, so this may be a
    test-infrastructure blocker (sanitizer CI) as much as a user-facing one. `[inferred]`
- **Why it matters:** the LLVM-version treadmill is a perennial JIT maintenance
  tax, and an ASAN-build crash blocks the very sanitizer runs that catch other
  memory bugs. Relevant to anyone running the corpus's ASan profile
  (`/setup-pg-asan`). cf. [[knowledge/subsystems/jit.md]].

## 3. Commit Sequence Numbers and Visibility

- **Author:** Matthias van de Meent (Jun 10, 12:19) `[from-website]`
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606090000
- Bullets:
  - Revives the long-running **CSN** (Commit Sequence Number) line of work:
    replacing/augmenting the snapshot model's running-xid array with a monotonic
    commit-order number so visibility checks become an LSN/CSN comparison rather
    than an O(n) "is this xid in my snapshot's in-progress set" test. `[inferred]`
  - The hard part is unchanged from prior CSN attempts: making
    `XidInMVCCSnapshot` / `HeapTupleSatisfiesMVCC` consult a CSN map cheaply,
    handling the still-in-progress→committed transition atomically, and not
    regressing the common small-snapshot case. `[inferred]`
  - Matthias driving it ties to the same-window orioledb/64-bit-xid interest in
    the corpus's ideology notes — CSN is the in-core analogue of the
    commit-order tricks external MVCC engines lean on. `[inferred]`
- **Why it matters:** snapshot visibility is the hottest read-path predicate in
  the backend; a CSN model is one of the few changes that could move MVCC's
  asymptotic cost, which is why it keeps resurfacing despite the catalog/WAL
  ripple it implies. cf. [[knowledge/architecture/mvcc.md]],
  [[knowledge/subsystems/access-heap.md]], [[knowledge/subsystems/access-transam.md]].

## 4. PG19 FK fast path: OOB write and missed FK checks during batched apply

- **Author:** thread w/ Amit Langote, Nikolay Samokhvalov, Ayush Tiwari
  (Jun 9, 13:31 → Jun 10, 12:48) `[from-website]`
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606090000
- Bullets:
  - Concerns the PG19 **foreign-key "fast path"** (the RI-trigger batching that
    checks many referencing rows in one pass). The report names two distinct
    failures: an **out-of-bounds write**, and **missed FK checks** during the
    batched path — i.e. some referencing rows skip RI validation entirely. `[inferred]`
  - An OOB write is a memory-safety defect (the batch buffer is sized/indexed
    wrong), while a *missed* FK check is a correctness/data-integrity defect —
    the more dangerous of the two because it silently admits rows that violate
    the constraint. Both in the same path suggests a shared off-by-one in the
    batch cursor. `[inferred]`
  - High committer engagement across two days (Langote, plus reproduction notes
    from Samokhvalov) marks this as a release-blocker-class regression in
    newly-added PG19 code rather than a long-standing bug. `[inferred]`
- **Why it matters:** referential integrity is a correctness contract users rely
  on absolutely; an FK fast path that both corrupts memory and skips checks is a
  must-fix before the feature ships. cf. [[knowledge/subsystems/executor.md]],
  [[knowledge/architecture/executor.md]].

## 5. Deadlock detector fails to activate on a hot standby replica

- **Author:** thread w/ Vitaly Davydov, Fujii Masao (Jun 9, 16:18 → Jun 10, 08:42)
  `[from-website]`
- **URL:** https://www.postgresql.org/list/pgsql-hackers/since/202606090000
- Bullets:
  - On a **hot standby**, recovery-conflict and lock interactions can still form
    a wait cycle, but the report is that the **deadlock detector does not fire**
    there — the `deadlock_timeout`-driven `CheckDeadLock()` path is not being
    armed (or not reached) for the recovery/startup-process locking that a
    standby uses. `[inferred]`
  - If true, a deadlock on a standby would hang until something else breaks the
    cycle (a query cancel, a recovery-conflict timeout) instead of being
    detected and one waiter aborted — degraded liveness exactly when an operator
    least expects a "stuck" standby. `[inferred]`
  - Fujii Masao engaging points at the hot-standby / recovery-conflict
    machinery; the fix likely ensures the startup process and standby backends
    participate in the lock-wait graph the detector walks. `[inferred]`
- **Why it matters:** the deadlock detector is the safety net that guarantees no
  permanent lock cycle; a standby where it silently doesn't run is a liveness
  hole in HA read-replica deployments. cf. [[knowledge/subsystems/storage-lmgr.md]],
  [[knowledge/idioms/locking-overview.md]], [[knowledge/architecture/replication.md]].
