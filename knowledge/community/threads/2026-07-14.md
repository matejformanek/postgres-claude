## 1. LockHasWaiters() crashes on fast-path locks
- Author: Robert Haas · 2026-07-14 15:02
- URL: https://www.postgresql.org/message-id/flat/CA%2BTgmoLockHasWaitersFastPath%40mail.gmail.com
- Bullets:
  - `LockHasWaiters()` walks the heavyweight lock's wait queue, but a lock
    acquired via the **fast-path** mechanism has no entry in the shared lock
    table until it's promoted — so the routine dereferences a `PROCLOCK`
    that was never materialized and crashes.
  - Fast-path locking is the optimization that lets a backend take a weak
    relation lock (`AccessShareLock` etc.) by flipping a bit in its own
    `PGPROC` rather than touching the shared `LockMethodLockHash`, avoiding
    lock-manager partition contention on the hot read path.
  - The fix has to teach `LockHasWaiters()` to recognize the fast-path case
    and return "no waiters" without chasing a null proclock — or force
    promotion before the check.
- Why it matters: any code path asking "does anyone want this lock?" (used
  by cancel-friendly waits and some extension code) can be reached with a
  fast-path lock held; a reproducible crash here is a robustness bug in the
  core lock manager.
- Corpus: [[locking]] (fast-path locks, PROCLOCK, partition contention),
  [[process-lifecycle]] (PGPROC).

## 2. Change checkpoint-record-missing PANIC to FATAL
- Author: Michael Paquier · 2026-07-13 00:56
- URL: https://www.postgresql.org/message-id/flat/checkpoint-record-missing-panic%40paquier.xyz
- Bullets:
  - When recovery cannot read the checkpoint record named by `pg_control`,
    the startup process currently **PANIC**s — which on a primary triggers a
    crash-restart loop rather than a clean stop, and muddies the operator's
    view of what actually failed.
  - The proposal downgrades this specific unreadable-checkpoint case to
    **FATAL**: the startup process exits with a clear error, the postmaster
    doesn't cycle, and the DBA can intervene (restore, `pg_resetwal`, etc.).
  - Review is drawing the line carefully — PANIC vs FATAL matters because
    PANIC forces shared-memory reinit and a crash-recovery pass, which is the
    wrong response when the on-disk WAL is simply gone.
- Why it matters: recovery-path elevel choices are load-bearing operational
  behavior; a PANIC that should be a FATAL turns a diagnosable stop into a
  restart storm.
- Corpus: [[wal-and-xlog]] (checkpoint record, pg_control), [[error-handling]]
  (PANIC vs FATAL elevel ladder), [[process-lifecycle]] (startup process).

## 3. Missing FSM Update when Updating VM On-access
- Author: Melanie Plageman · 2026-07-13 15:40
- URL: https://www.postgresql.org/message-id/flat/CAAKRu_missing-fsm-vm-onaccess%40mail.gmail.com
- Bullets:
  - The recently landed "set the visibility-map bit lazily on access" path
    updates the **VM** when a reader finds an all-visible page, but forgets
    to also refresh the corresponding **FSM** free-space category — so the
    page's advertised free space can drift out of sync with reality.
  - This is the coherence sibling of the on-access VM work: VM and FSM are
    two separate fork-level maps that must stay mutually consistent, and the
    on-access shortcut touched one without the other.
  - Fix threads the FSM update into the same on-access code so both maps move
    together; review is checking the locking order between the VM buffer and
    the FSM update to avoid a new pin-order hazard.
- Why it matters: an FSM that overstates free space sends `INSERT`/`COPY`
  to pages that can't actually take the tuple, wasting a buffer read per
  retry; the recurring VM/FSM-coherence theme keeps surfacing edge cases.
- Corpus: [[free-space-map]] (category coherence), [[vacuum-autovacuum]]
  (VM bits), [[buffer-manager]] (pin order).

## 4. walsummarizer can get stuck when switching timelines
- Author: Robert Haas · 2026-07-13 20:39
- URL: https://www.postgresql.org/message-id/flat/CA%2BTgmoWalsummarizerTimeline%40mail.gmail.com
- Bullets:
  - The **WAL summarizer** (the PG 17+ auxiliary process that produces the
    block-reference files incremental backup consumes) can wedge when the
    server crosses a **timeline switch** — it waits for WAL on the old
    timeline that will never arrive because recovery jumped to a new one.
  - The stuck summarizer stops advancing `pg_wal/summaries/`, which in turn
    blocks `summarize_wal`-dependent features and can pin WAL removal.
  - Fix teaches the summarizer loop to notice a timeline change and follow it,
    rather than blocking indefinitely on the superseded timeline's LSN.
- Why it matters: incremental backup correctness rides on the summarizer
  keeping pace; a timeline-switch wedge silently breaks the backup chain on
  exactly the servers (promoted standbys) most likely to switch timelines.
- Corpus: [[backup-and-recovery]] (WAL summarization, incremental backup),
  [[wal-and-xlog]] (timelines), [[process-lifecycle]] (aux process loop).

## 5. blkreftable.c needs more sanity checks
- Author: Robert Haas · 2026-07-13 22:17
- URL: https://www.postgresql.org/message-id/flat/CA%2BTgmoBlkreftableSanity%40mail.gmail.com
- Bullets:
  - `blkreftable.c` builds and parses the **block-reference table** that maps
    which blocks changed since a prior backup — the on-disk format underneath
    incremental backup. The thread adds defensive validation so a corrupt or
    truncated summary file is rejected loudly instead of producing a silently
    wrong incremental.
  - Proposed checks cover block-number monotonicity, chunk bounds, and
    limit-block sanity when deserializing a summary read from disk.
  - Pairs naturally with thread #4 (same subsystem, same author): #4 keeps
    the summarizer producing correct files, #5 hardens the reader against bad
    ones.
- Why it matters: a bad incremental that restores without error is the
  worst backup failure mode; sanity checks convert a silent data-loss risk
  into a detectable error at backup or reconstruct time.
- Corpus: [[backup-and-recovery]] (block-reference table, incremental backup),
  [[error-handling]] (defensive validation / ereport).

## 6. hashjoins vs. Bloom filters (yet again)
- Author: Matheus Alcantara · 2026-07-13 19:26 [multiple follow-ups]
- URL: https://www.postgresql.org/message-id/flat/hashjoins-bloom-filters-yet-again%40mail.gmail.com
- Bullets:
  - Revives the long-standing idea of building a **Bloom filter** on the hash
    join's build side and pushing it down to the probe side, so outer tuples
    that can't possibly match are discarded before the (expensive) hash probe
    and before any earlier scan/filter work when the filter can be pushed to
    the scan.
  - The "yet again" framing acknowledges prior attempts stalled on *when* the
    filter pays off — it's a win for highly selective joins over large outer
    inputs and a pure cost for low-selectivity joins, so the planner needs a
    reliable break-even estimate.
  - Follow-ups this thread are working the cost-model gating and where in the
    executor the filter is applied (probe-time vs pushed into the outer scan).
- Why it matters: selective-join Bloom pushdown is one of the larger
  outstanding executor wins; the recurring blocker is estimation, exactly
  the planner's weakest spot on join selectivity.
- Corpus: [[executor-and-planner]] (hash join, nodeHash), [[planner-cost-model]]
  (join selectivity, break-even estimation).

## 7. Bug: GiST index-only scans can use tuple descriptor with incorrect alignment when deforming
- Author: Peter Geoghegan · 2026-07-13 22:20
- URL: https://www.postgresql.org/message-id/flat/CAH2-Wz-gist-ios-tupdesc-alignment%40mail.gmail.com
- Bullets:
  - A **GiST index-only scan** reconstructs the indexed values from the index
    tuple using a tuple descriptor, but for some opclasses the descriptor's
    alignment metadata doesn't match how the values were physically stored —
    so `heap_deform_tuple`-style deformation reads at the wrong offsets.
  - The hazard is opclass-specific: it shows up when the index-column type's
    storage alignment differs from what the IOS reconstruction path assumes,
    yielding garbage (or a crash) rather than the stored value.
  - Fix aligns the reconstruction tuple descriptor with the actual index
    storage; review is enumerating which GiST opclasses are affected to bound
    the blast radius.
- Why it matters: index-only scans are supposed to return exactly what a heap
  fetch would; a silent alignment mismatch returns wrong query answers — the
  most severe class of bug — for affected GiST opclasses.
- Corpus: [[access-method-apis]] (GiST, index-only scan, amcanreturn),
  [[node-infrastructure]] (tuple descriptor / deform alignment).

## 8. sequencesync worker race with REFRESH SEQUENCES
- Author: Amit Kapila · 2026-07-13 03:23 [multiple follow-ups]
- URL: https://www.postgresql.org/message-id/flat/CAA4eK1sequencesync-refresh-race%40mail.gmail.com
- Bullets:
  - The new logical-replication **sequence synchronization** worker races
    against a concurrent `ALTER SUBSCRIPTION ... REFRESH PUBLICATION SEQUENCES`:
    both can try to (re)sync the same sequence's state, and the interleaving
    can leave a sequence's synced value inconsistent or the sync worker
    operating on a stale membership set.
  - Continues the multi-day sweep on sequence-replication correctness; today's
    follow-ups worked the locking/coordination between the sync worker's
    catalog snapshot and the REFRESH command's membership update.
  - The fix pattern mirrors the older tablesync-vs-REFRESH coordination — take
    the right lock / recheck membership so exactly one actor owns a given
    sequence's sync at a time.
- Why it matters: sequence replication is a headline PG20 logical-replication
  feature; a sync/refresh race that corrupts sequence values undermines the
  correctness guarantee the feature is sold on.
- Corpus: [[logical-replication]] (sequencesync, REFRESH PUBLICATION),
  [[locking]] (coordination / membership recheck).

## 9. Wait event timing and tracing instrumentation
- Author: Michael Paquier · 2026-07-13 01:09
- URL: https://www.postgresql.org/message-id/flat/wait-event-timing-tracing%40paquier.xyz
- Bullets:
  - Proposes attaching **timing** (and optional tracepoints) to wait events so
    tools can measure *how long* a backend spent in each wait, not just sample
    *which* wait it was in at snapshot time — turning `pg_stat_activity`'s
    point-in-time wait into a cumulative-duration signal.
  - The design tension is overhead: reading a clock on every wait
    enter/exit is measurable on hot LWLock waits, so the feature has to be
    gate-able (GUC or build option) and cheap when off.
  - Ties into the broader pgstat instrumentation work; review is weighing a
    per-wait timestamp vs a sampled/statistical approach.
- Why it matters: "which wait and for how long" is the single most-requested
  observability gap in PG performance analysis; getting the overhead model
  right is what has blocked prior attempts.
- Corpus: [[pgstat-framework]] (wait events, cumulative stats), [[locking]]
  (LWLock wait hot path), [[gucs-config]] (gating GUC).

## 10. Report oldest xmin source when autovacuum cannot remove tuples
- Author: Jacob Champion · 2026-07-13 15:06
- URL: https://www.postgresql.org/message-id/flat/report-oldest-xmin-source-autovacuum%40mail.gmail.com
- Bullets:
  - When VACUUM can't remove dead tuples because some `xmin` horizon is
    holding them, the log today says *that* it happened but not *who* — long
    transaction, replication slot, prepared xact, or standby feedback. This
    patch reports the **source** of the limiting horizon.
  - Implementation surfaces the `ComputeXidHorizons` result's binding
    constraint (which of the several candidate horizons actually won) into the
    autovacuum log line / a reportable field.
  - Directly attacks the #1 "why is my table bloating despite autovacuum"
    diagnosis loop, which today requires manually cross-checking
    `pg_stat_activity`, `pg_replication_slots`, and `pg_prepared_xacts`.
- Why it matters: xmin-horizon starvation is the most common real-world
  bloat cause; naming the culprit in the log collapses a multi-view manual
  investigation into a single line.
- Corpus: [[vacuum-autovacuum]] (dead-tuple removal, xmin horizon),
  [[snapshot-management]] (ComputeXidHorizons), [[mvcc]].

## 11. Refactor SLRU to always use long file names
- Author: Aleksander Alekseev · 2026-07-13 11:47
- URL: https://www.postgresql.org/message-id/flat/slru-long-file-names%40mail.gmail.com
- Bullets:
  - **SLRU** segment files historically used short (4-hex-digit) names, which
    wrap around and constrain how large an SLRU (clog, multixact, subtrans,
    …) can grow before filenames collide. PG 17 introduced longer names for
    some SLRUs; this refactor makes **all** SLRUs use the long-name scheme
    uniformly.
  - Uniform long names remove a class of wraparound edge cases and simplify
    the segment-management code by deleting the short-vs-long branching.
  - Review is checking `pg_upgrade` / on-disk compatibility — existing
    clusters have short-named segments that must still be found after upgrade.
- Why it matters: consistent SLRU naming is groundwork for larger multixact /
  clog address spaces and removes a subtle wraparound foot-gun; the pg_upgrade
  compatibility angle is the load-bearing risk.
- Corpus: [[slru-infrastructure]] (segment files, naming), [[multixact]],
  [[pg-upgrade-internals]] (on-disk compatibility).

## 12. SELECT over partitioned table with LIMIT 1 performance regression
- Author: abrahim abrahao · 2026-07-13 18:38
- URL: https://www.postgresql.org/message-id/flat/partitioned-limit1-regression%40mail.gmail.com
- Bullets:
  - A `SELECT ... FROM parted ORDER BY k LIMIT 1` that used to terminate early
    after reading one partition now scans more partitions than necessary — a
    regression traced to how the **MergeAppend / partition-ordered** path
    interacts with the LIMIT's early-stop.
  - The report includes a plan showing the executor pulling tuples from
    partitions it should have been able to skip once the first ordered tuple
    was found, inflating buffer reads on wide partition sets.
  - Discussion is bisecting which planner/executor change altered the
    early-termination behavior and whether the fix belongs in path costing or
    in the MergeAppend runtime.
- Why it matters: `ORDER BY ... LIMIT 1` over partitioned tables is an
  extremely common access pattern; an early-stop regression turns an
  O(1-partition) query into an O(all-partitions) one.
- Corpus: [[executor-and-planner]] (MergeAppend, LIMIT early stop),
  [[planner-cost-model]] (partition path costing).
