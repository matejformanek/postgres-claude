## 1. Adding basic NUMA awareness

- Author(s): Jakub Wartak, Tomas Vondra · latest 2026-06-16 12:39
- URL: https://www.postgresql.org/message-id/86e7e218-9013-4cdb-9ed2-dfda49640b4d%40vondra.me

- Long-running effort to make shared memory / buffer-pool allocation
  NUMA-node aware so a backend touches buffers on its local memory node
  rather than paying cross-socket latency.
- This window continues tuning how the buffer descriptors and the buffer
  blocks themselves are partitioned/interleaved across nodes, and what the
  observable win is on large multi-socket boxes.
- Still a "basic" framing: the goal is the cheap correctness/locality wins
  first, not a full NUMA scheduler.

**Why it matters:** Touches the shared-buffer allocation path and PGShmem
layout — the most invasive shmem change since the dynamic-shmem work, with
on-startup partitioning implications. [[knowledge/subsystems/storage-buffer.md]]

## 2. Randomize B-Tree page split location to avoid oscillating patterns

- Author: Dmitry Dolgov · 2026-06-16 10:13
- URL: https://www.postgresql.org/message-id/jotfal6422zpqvux4add7xcyftaujjiksscfcc5uuibaldw4sv%40pzriktpx4gyl

- Proposes randomizing the chosen split point in `nbtsplitloc.c` so that
  specific insertion orders stop producing a pathological oscillating
  fill pattern that wastes space.
- The tension is against the existing "fillfactor-aware / rightmost-leaf"
  heuristics that deliberately bias splits for append-heavy workloads.
- Discussion centers on whether randomness defeats the suffix-truncation
  and "many duplicates" split logic that the current code is tuned for.

**Why it matters:** `nbtsplitloc.c` split-point selection is a hot, carefully
tuned path; a change here moves on-disk index density for everyone.
[[knowledge/subsystems/access-nbtree.md]]

## 3. Fix race in ReplicationSlotRelease for ephemeral slots

- Author(s): Xuneng Zhou, Zhijie Hou, Amit Kapila, Fujii Masao · latest 2026-06-16 12:45
- URL: https://www.postgresql.org/message-id/CABPTF7VdFwiROsch4T7VbOCqQYpRbh%3D%3DgAZPM6tJeff5Ou80Qw%40mail.gmail.com

- An ephemeral (RS_EPHEMERAL) slot can be concurrently dropped while
  `ReplicationSlotRelease()` is mid-cleanup, leading to a window where the
  slot's in-memory state is inconsistent with `ReplicationSlotControlLock`
  invariants.
- The patch reorders the release/cleanup so the slot is marked and the
  effective-xmin recompute happens under the right lock coverage.
- Reviewers are checking the interaction with `ReplicationSlotCleanup()`
  and on-error paths (the SQL-callable slot functions release the slot
  on `ERROR`).

**Why it matters:** Slot-state races corrupt the xmin/catalog_xmin horizon
that protects against premature vacuum of still-needed rows — a data-loss
class bug. [[knowledge/idioms/logical-decoding-snapshot.md]]

## 4. Direction for test frameworks: Perl TAP vs. Python/pytest

- Author(s): Andrew Dunstan, Tom Lane, Heikki Linnakangas, Jacob Champion,
  Melanie Plageman, Michael Paquier, Daniel Gustafsson · 30+ messages 06-16
- URL: https://www.postgresql.org/message-id/cdaaf722-4529-435b-9340-cedf1a3a277f%40dunslane.net

- The most active thread of the window: whether the project should adopt
  Python/pytest alongside (or instead of) the Perl-based TAP harness for
  new integration tests.
- Arguments span contributor onboarding (Perl expertise is thinning) vs.
  the cost of a second test toolchain in the buildfarm and packaging.
- Tom Lane and others push back on fragmenting the test infrastructure;
  others note specific TAP ergonomics (async, structured assertions) that
  pytest would improve.

**Why it matters:** A project-direction decision that would touch every
future TAP test under `src/test/` and the buildfarm client. No code yet,
but the outcome shapes how internals patches ship their tests.
[[knowledge/idioms/pgstat-flush-timing.md]]

## 5. Proposal: Conflict log history table for Logical Replication

- Author(s): vignesh C, Dilip Kumar, Amit Kapila, shveta malik · 15+ msgs 06-16/17
- URL: https://www.postgresql.org/message-id/CALDaNm2Dd2Pi2yMVWZcb_QKycD_5%2BDktOXukXt%2BsDr-OKL%2B4Ww%40mail.gmail.com

- Proposes a persistent catalog/table recording logical-replication apply
  conflicts (insert_exists, update_origin_differs, delete_missing, etc.)
  rather than only emitting them to the server log.
- Open design questions: retention/pruning of the history table, whether
  it is a real heap relation or a stats view, and the cost on the apply
  hot path of writing a row per conflict.
- Ties into the existing `pg_stat_subscription_stats` conflict counters
  shipped recently.

**Why it matters:** Adds durable state to the logical-apply path; design
choice (catalog vs. stats) sets precedent for observability features that
must not slow the apply worker. [[knowledge/architecture/replication.md]]

## 6. Multi-Entry Indexing for GiST & SP-GiST

- Author: Alexander Nestorov · 2026-06-16 21:41
- URL: https://www.postgresql.org/message-id/a8dc2a89-8cfd-4b36-bf0f-a3d1c76bf5fe%40Spark

- Proposes letting a single GiST / SP-GiST opclass emit multiple index
  entries per heap tuple (as GIN already does), to index array/composite
  values without a separate AM.
- Requires changes to the `amInsert` contract and the AM's notion of
  one-TID-per-call, plus opclass support functions to decompose a value.
- Early-stage: discussion of whether this belongs in the AM layer or is
  better served by existing GIN/`intarray`-style approaches.

**Why it matters:** Changes the index-AM tuple-count contract for two AMs;
relevant to anyone implementing a custom index AM's insert path.
[[knowledge/idioms/tableam-index-fetch.md]]

## 7. REPACK enhancements + "Do not lock tables in get_tables_to_repack"

- Author(s): Antonin Houska (REPACK enhancements, 06-16 12:53);
  cca5507 (lock fix, 06-16 07:34)
- URL: https://www.postgresql.org/message-id/109367.1781614382%40localhost
- URL: https://www.postgresql.org/message-id/tencent_9F290B256A3F52B66542F1140E32ECC64309%40qq.com

- REPACK (the in-core successor to pg_repack-style rewrite) is getting
  follow-up: cca5507 reports `get_tables_to_repack()` taking locks on every
  candidate relation up front, which serializes against concurrent DDL.
- Houska's "enhancements" thread collects the post-commit follow-ups
  (generated columns, the lock-acquisition strategy, progress reporting).
- A separate report this window — "REPACK CONCURRENTLY fails on tables with
  generated columns" (Ewan Young) — feeds the same cleanup.

**Why it matters:** REPACK rewrites a relation's storage while concurrent
DML runs; lock-acquisition order and generated-column handling are exactly
where such rewrites historically break. [[knowledge/subsystems/access-heap.md]]

## 8. Unexpected behavior after OOM errors

- Author: Alexander Lakhin · 2026-06-17 06:00
- URL: https://www.postgresql.org/message-id/e77acaac-a1b3-40b3-99ee-5769b4e453e4%40gmail.com

- Reports that after a backend hits an out-of-memory `ereport(ERROR)`,
  some subsystem state is left inconsistent rather than fully reset by the
  transaction-abort cleanup.
- The interesting cases are where cleanup itself wants to allocate (and
  can't), or where a `PG_TRY/PG_CATCH` block assumes a context still
  exists after the OOM longjmp.
- Lakhin (a prolific fuzzer of these paths) provides a reproducer.

**Why it matters:** Error-path correctness under OOM is the classic
longjmp-safety trap; cleanup that itself palloc's is the anti-pattern.
[[knowledge/idioms/memory-contexts.md]]

## 10. mxid_score can become Infinity in pg_stat_autovacuum_scores

- Author(s): Nathan Bossart, Sami Imseih · 06-16 14:38 / 16:21
- URL: https://www.postgresql.org/message-id/ajFf4IAHnk2VWpIF%40nathan

- The new autovacuum-scoring view can compute an `Infinity` multixact
  score when the denominator (a freeze-distance / age ratio) is zero,
  surfacing a bogus value to operators.
- Fix clamps or special-cases the divide so the score stays a finite,
  rankable number.

**Why it matters:** Autovacuum scheduling decisions key off these scores;

  Zsolt Parragi, Jacob Champion (require-SSL thread) · 06-16/17
- URL: https://www.postgresql.org/message-id/CAOYmi%2BmbAYXDkqta5LXvGCtGSK-L-i%3DdOPcJO0D3yB4Z25iS%2BA%40mail.gmail.com

## 12. Fix tuple deformation with virtual generated NOT NULL columns

- Author(s): David Rowley, Tom Lane · 06-17 05:04 / 06:11
- URL: https://www.postgresql.org/message-id/CAApHDvrJBXEhet4%3DEs_wHBKdv5PCV5OGCaJOSmJexeaFqfmUHA%40mail.gmail.com

- Virtual generated columns (no on-disk storage) combined with a NOT NULL
  marking confuse `slot_deform_heap_tuple` / attribute-count logic, which
  expects the column to occupy a tuple slot it never does.
- Fix adjusts the deform path / `TupleDesc` natts handling so virtual
  generated columns are skipped without throwing off the physical attribute
  walk.
- Tom Lane reviews whether the fix is in the deform fast path or the
  expression-evaluation that materializes the virtual value.

**Why it matters:** Tuple deformation is in the executor's hottest inner
loop; a miscount here is a wrong-results-or-crash bug on a feature (virtual
generated columns) that's still stabilizing. [[knowledge/subsystems/access-heap.md]]
