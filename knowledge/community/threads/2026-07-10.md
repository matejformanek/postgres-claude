## 1. Batched clock sweep to reduce cross-socket atomic contention

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Greg Burd,
  2026-07-10 14:00)
- Summary:
  - New patch from Greg Burd reworking the buffer-manager clock sweep so a
    backend claims a *batch* of victim buffer candidates per pass instead of
    advancing `StrategyControl->nextVictimBuffer` one slot at a time under the
    shared free-list/strategy atomics.
  - The motivation is NUMA scalability: on many-socket machines the single
    global sweep counter is a cross-socket atomic-increment hot spot, so every
    backend hunting a victim serialises on the same cache line. Batching amortises
    the atomic over N candidate buffers claimed at once.
  - Ties into the broader "restructured shared-buffer hash table" and buffer-eviction
    scalability work that has been recurring on -hackers for several weeks; the
    hard part is keeping the clock-sweep usage-count semantics correct when a
    backend holds a run of pre-claimed candidates that others might also want.
- **Why it matters:** the clock sweep is on the hot path of every buffer miss;
  its scalability directly bounds read-heavy throughput on large boxes. This is a
  textbook "the algorithm is fine, the atomic is the bottleneck" scaling patch.
  See `[[knowledge/skills/buffer-manager]]` (clock-sweep replacement + BufferAccessStrategy)
  and `[[knowledge/skills/locking]]` (atomics vs spinlock trade-offs on shared counters).

## 2. Missing FSM Update when Updating VM On-access

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Andres Freund,
  2026-07-10 14:49)
- Summary:
  - Andres Freund flags a correctness/efficiency gap: when a page's visibility-map
    bit is set lazily on access (rather than by VACUUM), the corresponding
    free-space-map entry is *not* refreshed, so the FSM can keep advertising free
    space on a page that has just been marked all-visible / all-frozen.
  - Consequence: an inserter directed to that page by the FSM can dirty a page
    that was cleanly all-visible, silently clearing the VM bit again and defeating
    index-only-scan eligibility and freeze durability — churn that the on-access
    VM set was supposed to lock in.
  - Discussion centers on where the FSM update belongs relative to the VM update
    so the two maps don't disagree, without adding a WAL record or lock to what is
    meant to be a cheap hint-setting path.
- **Why it matters:** this is the recurring FSM/VM-coherence class the corpus has
  tracked under `gap:fsm` — the two per-relation auxiliary maps have no shared
  lock and drift under concurrency. Wrong FSM data doesn't corrupt, but it quietly
  erodes the all-visible/all-frozen state that VACUUM works hard to establish.
  See `[[knowledge/skills/free-space-map]]` and `[[knowledge/skills/vacuum-autovacuum]]`
  (VM bit lifecycle + freeze).

## 3. Hash index bucket split bug

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Amit Kapila
  10:45, Peter Geoghegan 15:24 / 16:31)
- Summary:
  - Carried from 07-09 with substantive new posts. The report is a
    correctness bug in hash-index bucket splits: under concurrency a split can
    leave a scan able to miss or double-return tuples, tied to how the split
    relocates tuples between the old and new bucket while a scan holds a bucket
    lock/pin.
  - Peter Geoghegan pushes on isolating a reproducible interleaving and contrasts
    it with nbtree's more hardened page-split-vs-scan model; Amit Kapila (hash AM
    maintainer) works the cleanup-lock discipline on the old bucket.
  - The emerging consensus is that the fix must tighten the split's lock/pin
    protocol rather than paper over it in the scan, since the scan side already
    assumes a stable bucket while pinned.
- **Why it matters:** hash indexes have been WAL-logged and crash-safe since PG 10
  but are lightly used, so split-vs-scan races surface slowly and return wrong
  answers rather than crashing. See `[[knowledge/skills/access-method-apis]]`
  (index AM callbacks) and `[[knowledge/skills/locking]]` (buffer cleanup-lock
  discipline).

## 4. PG19 FK fast path: OOB write and missed FK checks during batched apply

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Amit Langote
  2026-07-09 11:37)
- Summary:
  - Amit Langote's report (carried, still active) of an out-of-bounds write plus
    missed foreign-key checks in the PG19 batched FK-check fast path: the RI
    trigger machinery batches referenced-key lookups, and the batch index can run
    past the allocated array while some required checks are skipped entirely.
  - Both failure modes are serious — the OOB write is a memory-safety defect, and
    the missed checks mean referential integrity can be silently violated on a
    batched INSERT/UPDATE path that was added for PG19.
  - Fix direction is to correct the batch bookkeeping in `ri_triggers.c` so the
    per-tuple check count and the array bound stay in lockstep.
- **Why it matters:** a regression in a *new* PG19 performance path that both
  corrupts memory and breaks the integrity guarantee FKs exist to provide — the
  worst pairing. This is exactly the kind of new-fast-path bug R14-style adversarial
  batch tests are meant to catch. See `[[knowledge/skills/executor-and-planner]]`
  (trigger/RI execution) and `[[knowledge/idioms/ri-triggers]]` where present.

## 5. [WIP] Pipelined Recovery

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Imran Zaheer,
  2026-07-10 09:01)
- Summary:
  - New WIP proposal to pipeline WAL recovery: overlap the decode/read of the next
    WAL records with the apply of the current ones, so the startup process is not
    strictly serialising "read record → apply record → read next" on a single
    thread.
  - The design leans on the existing WAL prefetch (read-stream) infrastructure to
    stage record reads ahead of apply, aiming to close the well-known gap where
    crash/standby recovery is single-threaded and I/O-bound while the CPU sits idle
    between reads.
  - Early-stage: the thread is scoping how far apply can be decoupled from read
    without breaking the strict LSN-ordering and buffer-consistency invariants that
    redo relies on.
- **Why it matters:** recovery throughput bounds failover time and standby lag;
  single-threaded redo has long been a scalability ceiling. Pipelining is the
  conservative first step toward parallel replay. See `[[knowledge/skills/wal-and-xlog]]`
  (redo loop + WAL prefetch) and `[[knowledge/skills/aio-readstream]]` (the
  read-stream API the patch reuses).

## 6. Add per-backend AIO statistics

  2026-07-09 04:19 / 2026-07-10 04:56)
- Summary:

    counters for the async-I/O subsystem so `pg_stat_get_backend_io`-style views
    can attribute AIO submissions/completions/waits to individual backends rather
    than only aggregating cluster-wide.
  - New revisions this window refine where the counters are incremented in the AIO
    completion callbacks so io_method = worker vs io_uring vs sync all report

    AIO + pgstat surface observable per-backend.
- **Why it matters:** AIO landed in PG17/18 but its observability is still coarse;
  per-backend attribution is what lets an operator find *which* backend is
  saturating the I/O method. See `[[knowledge/skills/aio-readstream]]` (AIO methods
  + completion callbacks) and `[[knowledge/skills/pgstat-framework]]` (fixed-size

## 7. Fix data checksum processing for temp relations and dropped databases

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Fujii Masao,
  multiple 07-09/07-10; Kyotaro Horiguchi, Daniel Gustafsson)
- Summary:
  - Carried, very active (spawned sibling threads "Restrict data checksums entries
    in pg_stat_io" and "Fix pg_stat_progress_data_checksums counter
    initialization"). The core fix: the online-checksum-enable machinery mishandles
    temporary relations and databases dropped mid-operation, either double-counting
    or touching relations it shouldn't.
  - Fujii Masao is driving several coordinated fixes: skip temp relations correctly
    during the checksum worker's scan, handle a database dropped while the worker
    is processing it, and fix the `pg_stat_progress_data_checksums` counter
    initialization so progress reporting doesn't start from stale values.
  - The `pg_stat_io` sibling thread proposes restricting which checksum-related
    I/O gets attributed, so the checksum-enable pass doesn't pollute the regular
    I/O stats.
- **Why it matters:** online checksum enabling (PG18) is a long-running background
  operation over the whole cluster; temp-relation and dropped-database edge cases
  are exactly where such sweepers break. Progress-counter correctness matters for
  operators watching a multi-hour enable. See `[[knowledge/skills/wal-and-xlog]]`
  (checksums) and `[[knowledge/skills/pgstat-framework]]` (progress views).

## 8. Proposal: Conflict log history table for Logical Replication

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Dilip Kumar 09:15,
  Kedar Anavardekar, shveta malik, Robert Haas 14:09, Amit Kapila 2026-07-10 06:47)
- Summary:
  - Carried with heavyweight new participation. The proposal adds a persistent
    history table recording logical-replication apply conflicts (the PG18+ conflict
    types: insert_exists, update_differ, update_missing, delete_missing, etc.) so
    conflicts are queryable after the fact rather than only emitted to the log.
  - Robert Haas and Amit Kapila weigh in on the design's cost model: how to write
    conflict rows without making the apply worker's hot path pay for a table
    insert per conflict, retention/pruning of the history, and whether it belongs
    in a system catalog vs a user-visible table.
  - Open question is whether the history is opt-in per subscription and how it
    interacts with the existing conflict-resolution (last-update-wins) settings.
- **Why it matters:** conflict *tracking* landed in PG18 but conflict *forensics*
  is still log-scraping; a queryable history is what makes multi-master / active-active
  logical replication operable. See `[[knowledge/skills/logical-replication]]`
  (apply worker + conflict tracking).

## 9. Introduce XID age based replication slot invalidation

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Masahiko Sawada,
  2026-07-09 21:52)
- Summary:
  - Carried. The patch adds a mechanism to invalidate a replication slot once the
    XID it pins falls too far behind (by configurable age), preventing an
    abandoned or slow slot from holding back `catalog_xmin` / `xmin` indefinitely
    and bloating the cluster or blocking VACUUM.
  - Parallels the existing `max_slot_wal_keep_size` (WAL-size-based invalidation)
    but keys on transaction-ID age instead of WAL volume — the failure mode where
    a slot holds an old xmin without necessarily retaining much WAL.
  - Masahiko Sawada (a logical-replication maintainer) is refining the invalidation
    cause reporting so `pg_replication_slots.invalidation_reason` distinguishes
    XID-age from the WAL-size and horizon causes.
- **Why it matters:** an abandoned slot pinning an old xmin is a classic production
  bloat/vacuum-stall footgun; age-based invalidation gives operators a safety valve
  independent of WAL retention. See `[[knowledge/skills/logical-replication]]`
  (slot.c invalidation) and `[[knowledge/skills/vacuum-autovacuum]]` (xmin horizon).

## 10. Tepid: selective index updates for heap relations

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Alexander Korotkov
  2026-07-09 11:26, Greg Burd 15:41)
- Summary:
  - Carried. "Tepid" proposes selectively skipping index updates for heap tuples
    under specific conditions — a middle ground between HOT (no index update when
    no indexed column changed and the new tuple fits on the same page) and a full
    index update, extending the set of cases where an update avoids touching some
    indexes.
  - Alexander Korotkov frames the mechanism and the invariants it must preserve so
    index scans still find the right tuple version; Greg Burd probes the corner
    cases where a "tepid" update interacts with index-only scans and the
    visibility map.
  - The design's risk surface is the same as HOT's: any case where an index is
    incorrectly skipped silently loses index entries and returns wrong answers.
- **Why it matters:** write amplification from index maintenance is a dominant cost
  in update-heavy workloads; broadening HOT-like avoidance is high-value but
  correctness-critical. See `[[knowledge/skills/vacuum-autovacuum]]` (HOT + heap
  pruning) and `[[knowledge/skills/access-method-apis]]` (index AM insert path).

## 11. libpq: Process buffered SSL read bytes to support records >8kB on async API

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Heikki Linnakangas
  12:55 / 15:52, Daniel Gustafsson 15:06 / 17:03)
- Summary:
  - Carried, converging. The bug: on the async (non-blocking) libpq API, when a TLS
    record larger than the 8 kB read buffer arrives, libpq can leave decrypted
    bytes buffered inside the SSL layer while telling the caller there is nothing
    to read — the socket shows no readable data, so an epoll/select-driven client
    stalls.
  - Heikki Linnakangas's fix drains buffered SSL bytes before returning control so
    the async consumer sees all available data; Daniel Gustafsson reviews the
    interaction with the read-ready signalling and OpenSSL/LibreSSL differences.
  - Related to the same window's "LibreSSL and OpenSSL separation in libpq"
    thread (Daniel Gustafsson) preparing for OpenSSL 1.1.1 deprecation.
- **Why it matters:** a stall that only manifests with large TLS records on
  non-blocking connections is nasty to diagnose in production drivers built on
  libpq's async API. See `[[knowledge/skills/wire-protocol]]` (libpq frontend
  I/O) where present.

## 12. Do not lock tables in get_tables_to_repack

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Álvaro Herrera,
  2026-07-10 14:22)
- Summary:
  - Álvaro Herrera's patch removes table locks taken while *enumerating* the set
    of tables to repack (the REPACK / REPACK CONCURRENTLY planning phase), so
    building the work list doesn't itself block concurrent DDL/DML on tables that
    may not even end up repacked.
  - The concern is lock-footprint minimization: REPACK CONCURRENTLY is meant to be
    low-impact, but locking every candidate table just to decide the plan
    undermines that promise. The fix defers locking to the point where a table is
    actually processed.
  - Requires care that a table dropped/altered between enumeration and processing
    is handled gracefully rather than assumed still valid under a lock.
- **Why it matters:** REPACK (the pg_repack-style in-core table rewrite, new in the
  PG19/20 line) lives or dies on its lock discipline; taking fewer locks earlier is
  what keeps it "concurrent." See `[[knowledge/skills/locking]]` (heavyweight lock
  scope) and `[[knowledge/skills/vacuum-autovacuum]]` (table-rewrite maintenance).
