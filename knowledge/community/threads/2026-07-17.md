## 1. Why clearing the VM doesn't require registering vm buffer in wal record

- **Author(s):** Fujii Masao (00:09 root) → Melanie Plageman (22:35)
- **URL:** `/message-id/CAHGQGwF_PzOv5y7ucFh7Fqqqa8ar83zYwvWugqarRD6%2B7GCtEQ%40mail.gmail.com`
- Summary:
  - Direct follow-on to the master commit `ed62d26` VM-clear WAL-logging
    corruption fix (the headline of the 07-15 upstream-watcher batch). The
    question: after that fix, does clearing a visibility-map bit on a heap
    modification need the VM page itself registered in the WAL record, or is
    re-deriving the clear from the heap redo sufficient?
  - Melanie's reply argues the VM clear is reconstructable from the heap
    change during redo, so registering the VM buffer would be redundant WAL
    volume — but only if the redo path is guaranteed to re-clear on every
    code path that clears in the primary. The thread is auditing exactly that
    guarantee.
  - Open point: torn-page / crash-window interaction — if the VM page is
    written without full-page-image protection, a bit could be left stale
    after a crash even though redo "would have" cleared it.
- **Why it matters:** This is the theoretical backstop for the corruption
  class `ed62d26` fixed. Whether the VM buffer needs registering decides if
  there's a residual staleness window after the commit landed.
  `[[idioms/visibility-map-update]]`, `[[idioms/index-only-scan-vm-check]]`,
  `[[subsystems/contrib-pg_visibility]]`

## 2. Reduce cleanup lock contention on standby replay

- **Author:** Matthias van de Meent (07-17 14:28)
- **URL:** `/message-id/CAEze2WgUnbCy3hkWoNQqAs5CMoKVw83QrJn%3Dq9W8nEjXmKVQzw%40mail.gmail.com`
- Summary:
  - Standby replay takes cleanup locks (buffer-content lock in exclusive +
    pin-count-1 wait) on pages during redo of certain heap/index ops, which
    serialize against read-only queries pinning those buffers. On busy hot
    standbys this stalls replay behind long-running readers.
  - Proposal reworks when replay actually needs the *cleanup* lock (waits for
    sole pinner) vs. just an exclusive content lock, narrowing the cases that
    block on concurrent pins.
  - Correctness hinge: any redo that physically removes line pointers a
    concurrent scan could be mid-traversal on still needs the full cleanup
    lock; the patch has to prove the relaxed cases don't.
- **Why it matters:** Replay lag on hot standbys is a recurring production
  pain; cleanup-lock contention is one of its least-visible causes.
  `[[subsystems/storage-lmgr]]`, `[[idioms/lwlock-rank-discipline]]`,
  `[[subsystems/storage-buffer]]`

## 3. Improve pg_stat_statements scalability

- **Author:** Kyotaro Horiguchi (07-16 04:22)
- **URL:** `/message-id/20260716.132240.1034537909765069601.horikyota.ntt%40gmail.com`
- Summary:
  - `pg_stat_statements` uses a shared hash table guarded by a single LWLock
    for entry lookup/insert plus a per-entry spinlock for counter bumps. At
    high query-diversity + high concurrency the shared lock and the dealloc
    (garbage-collection) pass become the bottleneck.
  - Patch reduces contention on the hot path — candidate directions include
    partitioned locking of the entry table and a lighter counter-update
    protocol so the common "entry already exists, just add counters" case
    doesn't touch the exclusive lock.
  - Tension with the `pgss_store` dealloc heuristic: making inserts cheaper
    can raise churn, which pushes more work onto the GC pass.
- **Why it matters:** `pg_stat_statements` is on in most production clusters;
  its lock is a real observability-tax on OLTP workloads.
  `[[subsystems/contrib-pg_stat_statements]]`, `[[idioms/pgstat-flush-timing]]`

## 5. Hash index AM fake LSN oversight

- **Author:** Peter Geoghegan (07-16 22:11)
- **URL:** `/message-id/CAH2-WznfA%3D%3DhUFdN8YEYkhFiTL0X_roQ1MDeCiRygfqZLNDw%2BA%40mail.gmail.com`
- Summary:
  - Hash index pages, like other index AMs, use a "fake LSN" scheme for
    unlogged relations so buffer eviction ordering stays sane without real
    WAL. Geoghegan flags a spot where the hash AM doesn't set/advance the
    fake LSN where the analogous nbtree/gist paths do.
  - The consequence is subtle: fake-LSN gaps can let the buffer manager make
    wrong eviction/flush-order decisions for unlogged hash indexes, or upset
    checksum/page-verification assumptions.
  - Fix is localized (set the fake LSN on the missing modify path) but needs
    an audit that every hash write path is covered.
- **Why it matters:** Fake-LSN discipline is one of those invariants that's
  invisible until an unlogged-relation edge case corrupts eviction order.
  `[[idioms/hash-page-layout]]`, `[[idioms/hash-bucket-split]]`,
  `[[idioms/wal-page-format]]`

- Summary:
  - The WAL summarizer (incremental-backup infrastructure) can wedge when the
    server switches timelines — it keeps trying to read summary-covered WAL on

  `[[subsystems/replication]]`, `[[subsystems/access-transam]]`

## 7. Batching in executor

- **Author:** Denis Smirnov (07-17 04:17)
- **URL:** `/message-id/80692DF4-04F0-4292-8AF0-E13D5E13CE74%40gmail.com`
- Summary:
  - Proposes moving PG's executor toward batched (vectorized-ish) tuple flow
    — passing groups of tuples between nodes instead of the one-tuple-per-
    `ExecProcNode` volcano model — to cut per-tuple dispatch overhead.
  - Early design-discussion stage: where the batch boundary lives (slot array
    vs. columnar batch), how it interacts with the existing
    `TupleTableSlot` abstraction, and which nodes benefit first (seqscan →
    filter → agg).
  - Big-lift thread; expect a long design debate before any code that could
    land, given how much of the executor the volcano contract touches.
- **Why it matters:** Batched execution is the perennial "how do we close the
  gap to columnar engines" question; even a partial win on the scan→filter
  path would be significant. `[[subsystems/executor]]`

## 9. [PATCH] Use Boyer-Moore-Horspool for simple LIKE contains patterns

- **Author:** Atsushi Ogawa (07-17 10:23)
- **URL:** `/message-id/CAEah3%3DOZHXUDj%3D4vXDi62vTOw2Attn1mKsq6LJDPwOBGQpqc8Q%40mail.gmail.com`
- Summary:
  - `LIKE '%needle%'` currently scans the haystack with a naive
    character-by-character matcher in `MatchText`. Patch swaps in
    Boyer-Moore-Horspool for the pure-contains case (no wildcards inside the
    needle), which skips ahead on mismatch.
  - Wins scale with needle length and haystack size; degenerate cases (very
    short needles, multibyte-encoding boundaries) need to fall back so BMH's
    skip table stays correct across encodings.
  - Encoding correctness is the review crux — the skip table indexes by byte,
    so multibyte/`pg_mbstrlen` handling has to be watertight.
- **Why it matters:** `%contains%` is an extremely common query shape with no
  index help; a matcher speedup is a broad, if unglamorous, win.
  `[[subsystems/executor]]`

## 11. [PATCH] Don't call ereport(ERROR) from recovery target GUC assign hooks

- **Author:** Fujii Masao (07-17 09:30)
- **URL:** `/message-id/CAHGQGwEpfE0CDUODjBt7GO9U4ZF11hqga_Ci3wP8%3DO49oFKVw%40mail.gmail.com`
- Summary:
  - `recovery_target_*` GUCs currently throw `ereport(ERROR)` from their
    assign hooks on bad combinations. Assign hooks run at a point where a
    longjmp out is unsafe/undesirable (GUC machinery mid-commit of a value),
    so the patch moves the validation to check hooks (which are allowed to
    reject) and keeps assign hooks side-effect-only.
  - Aligns recovery-target GUCs with the documented check/assign contract:
    check hooks validate + can reject, assign hooks must not fail.
  - Small but correctness-relevant for config-reload paths where an assign-time
    ERROR could leave GUC state half-applied.
- **Why it matters:** A clean illustration of the check-vs-assign-hook
  discipline; also removes a foot-gun on `recovery_target` reconfiguration.
  `[[subsystems/access-transam]]`, `[[idioms/wal-record-construction]]`

## 12. Init connection time grows quadratically

- **Author:** Matthias van de Meent (07-17 08:54)
- **URL:** `/message-id/CAEze2Wgyq-ArsJ_MbJj%2BhGxB%3DhZYXA3j51N_362EwN%2BhWPEEtw%40mail.gmail.com`
- Summary:
  - Reports that per-connection initialization cost grows quadratically with
    some catalog/relcache dimension (e.g. number of relations, or entries
    loaded during `InitPostgres`), so backend startup latency degrades badly
    on large schemas.
  - Investigation points at an O(n²) build/scan during relcache or catalog
    warm-up in the fresh-fork backend path.
  - Connection-storm sensitivity: the fork model means every new `psql`
    connect pays this, so a quadratic term is a scaling wall for
    high-connection-churn deployments.
- **Why it matters:** Connection setup latency is a top complaint for
  connection-pooler-less and serverless-style workloads; a quadratic term on
  large schemas is a concrete scaling bug. `[[subsystems/utils-cache]]`,
  `[[subsystems/tcop]]`
