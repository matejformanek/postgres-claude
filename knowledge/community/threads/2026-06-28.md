## 1. Possible Visibility Map corruption in supported branches?

- Author: Jehan-Guillaume de Rorthais (06-26 10:19, active same day 12:55).
  Archive: <https://www.postgresql.org/search/?m=1&q=Possible+Visibility+Map+corruption+in+supported+branches>
- A report that the visibility map can end up corrupt (a VM bit set while the
  heap page is not actually all-visible / all-frozen) on currently-supported
  branches, with the thread working to pin down the reproduction and whether a
  back-patch is warranted.
- The danger of a stale-set VM bit is silent wrong answers: an index-only scan
  trusts the VM and skips the heap fetch, and VACUUM can skip pages it should
  process — so a single bad bit becomes either incorrect query results or a
  freeze that never happens.
- Why it matters: VM corruption is a correctness-and-durability bug that
  evades normal detection because nothing errors — it just returns wrong rows
  or lets xmin advance past tuples that should have blocked it.
  [[knowledge/idioms/visibility-map-update.md]] /
  [[knowledge/idioms/index-only-scan-vm-check.md]] /
  [[knowledge/subsystems/contrib-pg_visibility.md]]

## 2. Why clearing the VM doesn't require registering vm buffer in wal record

- Active: Robert Haas (06-26 20:05), Andrey Borodin (06-27 10:41).
  Archive: <https://www.postgresql.org/search/?m=1&q=Why+clearing+the+VM+doesn%27t+require+registering+vm+buffer+in+wal+record>
- A design-rationale thread: why heap WAL records that clear a VM bit do not
  need to register the VM buffer in the WAL record itself. The answer turns on
  the asymmetry between *setting* a VM bit (must be WAL-logged and crash-safe)
  and *clearing* it (conservatively re-derivable, so a lost clear is
  self-healing on the next modification).
- Pairs directly with thread #1: the clearing path's looseness is exactly what
  makes a stale-set bit possible, so the two threads bracket the VM
  correctness model from both ends.
- Why it matters: this is load-bearing folklore about the heap/VM crash-safety
  contract that rarely gets written down; getting it wrong in new code (e.g. a
  new AM or a VM-touching patch) reintroduces corruption of the #1 shape.
  [[knowledge/idioms/visibility-map-update.md]] /
  [[knowledge/idioms/wal-record-construction.md]]

## 3. Proposal: Conflict log history table for Logical Replication

- Very active multi-party thread across the whole window: Peter Smith, shveta
  malik, vignesh C, Amit Kapila, Dilip Kumar (06-26 03:43 → 06-27 02:37,
  ~15 messages).
  Archive: <https://www.postgresql.org/search/?m=1&q=Conflict+log+history+table+for+Logical+Replication>
- Proposal to persist logical-replication apply conflicts into a queryable
  history table rather than only emitting them to the server log. Debate covers
  the table's schema, retention/pruning, whether it lives in a catalog vs a
  regular table, and how it interacts with the existing conflict-resolution
  machinery added in recent cycles.
- The hard questions are operational: who prunes the table, how to avoid it
  becoming an unbounded write amplifier on a busy subscriber, and how to keep
  conflict logging from itself conflicting.
- Why it matters: conflict *observability* is the missing half of the conflict
  *resolution* work — without a durable record, operators can't audit what the
  apply worker decided. [[knowledge/idioms/apply-conflict-resolution.md]] /
  [[knowledge/idioms/apply-handlers-insert-update-delete.md]] /
  [[knowledge/subsystems/replication.md]]

## 4. Deadlock detector fails to activate on a hot standby replica

- Active: Vitaly Davydov (06-26 09:31), Xuneng Zhou (06-27 06:11).
  Archive: <https://www.postgresql.org/search/?m=1&q=Deadlock+detector+fails+to+activate+on+a+hot+standby+replica>
- A bug report that the deadlock detector does not fire on a hot standby in a
  scenario involving recovery-conflict / buffer-pin waits, so a wait that
  should be broken by the detector instead hangs until an external timeout.
- The standby's wait graph differs from the primary's (recovery conflicts,
  startup-process buffer-pin waits) and the detector's activation path doesn't
  cover all of it — so the usual "deadlock detected" safety net has a hole on
  replicas.
- Why it matters: a hung-not-detected wait on a replica is a silent
  availability failure; deadlock detection is supposed to be the universal
  backstop and a replica-only gap undermines that guarantee.
  [[knowledge/idioms/deadlock-detection.md]] /
  [[knowledge/subsystems/storage-lmgr.md]]

## 6. 048_vacuum_horizon_floor.pl hangs due to wakeup lost inside LockBufferForCleanup

- Active: Fujii Masao (06-26 13:40), Xuneng Zhou (06-27 01:56).
  Archive: <https://www.postgresql.org/search/?m=1&q=048_vacuum_horizon_floor+hangs+LockBufferForCleanup>
- A TAP-test hang traced to a lost wakeup inside `LockBufferForCleanup` — the
  cleanup-lock acquisition (which needs the buffer pin count to drop to one)
  misses a wakeup and the waiter never re-checks, so the test stalls.
- This is a genuine cleanup-lock signalling bug surfaced by a flaky recovery
  test, not just test flakiness — the fix has to plug the lost-wakeup window in
  the pin-wait/cleanup-lock handshake.
- Why it matters: `LockBufferForCleanup` is on VACUUM's critical path; a
  lost-wakeup there can hang a real vacuum, not only a test. Buffer-pin cleanup
  waits are a notorious source of subtle missed-wakeup bugs.
  [[knowledge/subsystems/storage-buffer.md]] /
  [[knowledge/data-structures/bufferdesc-state.md]]

## 9. Add per-backend lock statistics (pg_stat_lock cluster)

- Author: Tatsuya Kawata (06-27 10:55), plus sibling threads the same day:
  "doc: clarify pg_stat_lock.fastpath_exceeded scope" (14:14) and "Change
  wait_time column of pg_stat_lock to double precision" (17:50).
  Archive: <https://www.postgresql.org/search/?m=1&q=Add+per-backend+lock+statistics>
- A cluster of patches building out per-backend lock instrumentation: a new
  `pg_stat_lock` view exposing per-backend lock-wait counters, refining the
  `fastpath_exceeded` semantics, and widening `wait_time` to double precision
  for sub-millisecond resolution.
- This is observability infrastructure layered on the heavyweight lock manager
  and the fast-path lock arrays — it has to account for both fast-path and
  full-table-lock acquisitions without adding hot-path overhead.
- Why it matters: lock-wait visibility per backend is a long-standing operator
  ask; doing it cheaply requires care in the lmgr counters, which is exactly
  where this thread lives. [[knowledge/idioms/fastpath-locks.md]] /
  [[knowledge/subsystems/storage-lmgr.md]] /
  [[knowledge/idioms/locking-overview.md]]

## 11. [PATCH] Adjust autovacuum thresholds using relallvisible

- Author: wenhui qiu (06-26 03:43).
  Archive: <https://www.postgresql.org/search/?m=1&q=Adjust+autovacuum+thresholds+using+relallvisible>
- Proposes folding `relallvisible` (the count of all-visible pages from
  `pg_class`) into the autovacuum threshold computation, so a table that is
  mostly all-visible is treated differently from one that is churning — tuning
  when autovacuum decides a relation is worth processing.
- Ties the VM state (threads #1/#2) into the autovacuum scheduling heuristic:
  the more of a table is provably all-visible, the less urgent a scan.
- Why it matters: autovacuum threshold tuning is perennially contentious and
  has broad performance impact; using VM coverage as a signal is a plausible
  refinement but risks under-vacuuming if `relallvisible` is stale.
