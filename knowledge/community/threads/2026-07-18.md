## 1. Why clearing the VM doesn't require registering vm buffer in wal record

- **Author:** Melanie Plageman · Jul 17 20:51 ·
  [msg](https://www.postgresql.org/message-id/CAAKRu_Y9DZ7%3DAXpUfX%2BESgavoHFbS8if_2gmp-B6tP6vJxzXWA%40mail.gmail.com)
- The core question is the design rationale that has surfaced repeatedly in
  this month's VM-clear WAL work: when a heap modification clears a
  visibility-map bit, the VM buffer is **not** separately registered in the
  WAL record — only the heap page is. Why is that correct?
- The answer distilled in the thread: VM clearing is **idempotent and
  monotonic**. Re-clearing an already-clear bit during replay is a no-op, and
  an over-cleared VM only costs an extra visibility check — never a
  correctness failure. The VM is an optimization layer, not a correctness
  invariant, so it does not need byte-exact reconstruction the way a heap
  tuple does.
- Consequence: heap-modification WAL records carry enough to re-derive the
  needed VM state on replay; a crash that leaves VM bits set-but-stale is the
  case the *separate* VM-clear logging (the `ed62d26` series flagged in our
  07-16/07-17 corpus) exists to prevent — clearing that *loses* the bit is
  always safe, clearing that *fails to lose* it is the dangerous direction.
- **Why it matters:** this is the conceptual companion to the VM-clear
  WAL-logging corruption-fix series our `pg-upstream-watcher` and
  `pg-quality-auditor` runs have been tracking all week (07-16 #617 headline
  `ed62d26`, 07-18 #637 re-anchored the visibilitymap.h / heapam_xlog.h
  cluster). Understanding *which direction of VM staleness is safe* is the
  invariant that makes the whole series make sense.
  [[knowledge/subsystems — visibility map / heap-vacuum]] · [[free-space-map]]
  (VM sibling).

## 3. Restructured Shared Buffer Hash Table

- **Author:** Dhruv Aron (Databricks) · Jul 17 22:31 ·
  [msg](https://www.postgresql.org/message-id/CAAStW3Kq5_55X_G74P%3DaovoDBuUEeE_pionGVQ6F_dS7BThLqg%40mail.gmail.com)
- Proposes replacing the general-purpose `dynahash` used for the
  buffer-lookup table with a **specialized flat hash table** tuned for the
  one hottest lookup in the backend. Keeps dynahash's linear-hashing trick
  (grow/shrink one bucket at a time, no global rehash) and reuses the
  existing buffer-partition locks — no new synchronization primitive.
- Maintains a **1:1:1 mapping** between shared buffers, table entries, and
  buckets, and stores only the low mask + bucket count as atomics (not the
  high mask) so a concurrent resize can't produce a torn read that sends a
  backend to the wrong bucket.
- Benchmarks are honest about the tradeoff: ~50ns extra spinlock hold at one
  entry/bucket; with pathological chains (length 64) it's 248.6ns vs
  dynahash's 102.0ns. The win is cache locality (no directory indirection)
  and native support for **dynamic buffer-pool resize** (`buf_resize.c`) —
  the table grows/shrinks with `shared_buffers`.
- **Why it matters:** the buffer-mapping table is on the hot path of every
  buffer access; this is a concrete follow-on to the online
  `shared_buffers`-resize work. Directly touches the structures the
  buffer-manager skill documents. [[buffer-manager]] · [[locking]]
  (BufferMapping partition ranks).

## 4. index prefetching

- **Author:** Peter Geoghegan · Jul 17 20:58 ·
  [msg](https://www.postgresql.org/message-id/CAH2-WzmpQV7wqbwE_bW%2B1Ja32chnh87i%3Dwx2Y6NkhFiuEVsBzw%40mail.gmail.com)
- The ~3-year, 31-version epic reaches a commit boundary: Geoghegan plans to
  land **0001** (the slot-based table-AM index-scan interface) imminently,
  with 0002 (bounding `get_actual_variable_range` leaf reads) split into its
  own CF entry.
- Architecturally it's now fully wired into **read_stream / AIO**: a new
  batch index-AM API (`amgetbatch`) returns many TIDs at once (adopted in
  nbtree/hash/GiST/SP-GiST, patches 0003-0006); heapam then stages heap-block
  prefetches via read_stream before visiting them (0008), with a pin-transfer
  optimization (0007) and EXPLAIN IO visibility (0009).
- AIO-layer refinements are the remaining churn: `read_stream_reset()` gaining
  a don't-wait-for-completion mode (0010) and `pgaio_io_wait` race fixes for
  staged IOs (0011-0012) — refinement, not a fundamental blocker.
- **Why it matters:** this is the marquee read-stream *consumer* for index
  scans — the payoff of the whole PG17/18 AIO buildout on the OLTP-critical
  index-scan path. The `amgetbatch` interface is a new index-AM contract worth
  tracking for anyone implementing a custom AM.
  [[aio-readstream]] · [[access-method-apis]].

## 5. Reduce cleanup lock contention on standby replay

- **Author:** Matthias van de Meent · Jul 17 14:28 ·
  [msg](https://www.postgresql.org/message-id/CAEze2WgUnbCy3hkWoNQqAs5CMoKVw83QrJn%3Dq9W8nEjXmKVQzw%40mail.gmail.com)
- Targets the replay side: the startup process taking buffer **cleanup
  locks** (super-exclusive pins) during WAL replay can stall read-only
  queries on a hot standby. Proposes reducing how often / how broadly the
  cleanup lock is demanded during replay.
- Touches the intersection of buffer-pin discipline and recovery-conflict
  handling — cleanup locks on the standby are a known source of query
  cancellation / replay lag.
- **Why it matters:** standby query latency under replay pressure is a
  recurring HA pain point; anything that narrows the cleanup-lock window on
  replay is directly felt by read-scaling deployments.
  [[buffer-manager]] (pin/cleanup lock) · [[physical-replication]].

## 6. Parallel heap vacuum

- **Author:** Bharath Rupireddy · Jul 17 17:03 ·
  [msg](https://www.postgresql.org/message-id/CALj2ACUMbUrcYzzbdSoPgsv%2B3kFdkGovDZe6aP3e6XQCzFqy9g%40mail.gmail.com)
- Extends parallelism from the index-vacuum phase (already parallel) into the
  **heap scan / prune phase** of VACUUM, so large tables can spread the heap
  pass across workers rather than serializing it behind parallel index vacuum.
- Interacts with the parallel-vacuum leader/worker plumbing and the dead-TID

  parallel heap workers).
- **Why it matters:** heap scan is often the dominant VACUUM cost on
  wide/bloated tables; parallelizing it is the natural next step after the
  parallel-index-vacuum and the radix-tree TID store landed.
  [[vacuum-autovacuum]] · [[parallel-query]].

## 7. Batching in executor

- **Author:** Denis Smirnov · Jul 17 04:17 ·
  [msg](https://www.postgresql.org/message-id/80692DF4-04F0-4292-8AF0-E13D5E13CE74%40gmail.com)
- A design thread on moving PostgreSQL's executor from strict
  one-tuple-at-a-time `ExecProcNode` pull toward **batched/vectorized** tuple
  movement between nodes, to cut per-tuple dispatch overhead and improve cache
  behavior on analytic workloads.
- This is architecturally load-bearing: batching changes the `PlanState`
  execution contract that every node implements, and interacts with the TAM
  slot interface (see #4's slot-based batch scan).
- **Why it matters:** executor batching is one of the highest-leverage,
  highest-risk directions for PG's analytic performance; it's the kind of
  cross-cutting change that would ripple through every `nodeXxx.c`.
  [[executor-and-planner]] · [[node-infrastructure]].

## 8. hashjoins vs. Bloom filters (yet again)

- **Author:** Tomas Vondra · Jul 17 16:22 ·
  [msg](https://www.postgresql.org/message-id/7df7a692-f910-4c73-bd04-031d910af52c%40vondra.me)
- Revives the perennial proposal to build a **Bloom filter on the hash-join
  build side** and push it down to the probe side (or a scan below it) to
  discard non-matching outer rows early — "yet again" signals this is a
  re-attempt with fresh benchmarks / a narrower scope.
- Cost-model question at its heart: when does the filter's build+probe cost
  pay for itself vs. the rows it eliminates, and where in the plan tree the
  filter should be applied.
- **Why it matters:** runtime filters are standard in analytic engines;
  landing a well-costed version in PG's hash join is a concrete planner+executor
  win. [[executor-and-planner]] · [[planner-cost-model]].

## 11. Init connection time grows quadratically

- **Author:** Matthias van de Meent · Jul 17 08:54 ·
  [msg](https://www.postgresql.org/message-id/CAEze2Wgyq-ArsJ_MbJj%2BhGxB%3DhZYXA3j51N_362EwN%2BhWPEEtw%40mail.gmail.com)
- Performance report: backend connection-establishment cost scales
  **quadratically** with something (likely catalog/relcache or invalidation
  volume) — a scaling cliff for connection-churny workloads that don't pool.
- Backend startup runs `InitPostgres`, relcache/catcache priming, and
  invalidation replay; an O(n²) term there hits exactly the deployments (many
  short connections) that can least absorb it.
- **Why it matters:** connection setup latency is a top-line metric for
  serverless / high-churn PG usage; a quadratic term is the kind of thing that
  looks fine in dev and bites at scale. [[process-lifecycle]] · [[type-cache]].

## 12. Tepid: selective index updates for heap relations

- **Author:** Bharath Rupireddy · Jul 17 20:12 ·
  [msg](https://www.postgresql.org/message-id/CALj2ACV3YaHh4zxaOFkNqKeQzkX0-uW1ghRd4LwdZsOjRB1T4Q%40mail.gmail.com)
- "Tepid" proposes **selective index maintenance** — updating only the
  indexes whose keys actually changed on an UPDATE, rather than the current
  all-or-nothing (HOT skips *all* index updates, non-HOT does *all* of them).
  A middle tier between HOT and full re-index-all.
- Touches the heap-update path's decision logic and every index AM's insert
  contract; correctness hinges on never leaving an index missing an entry it
  should have.
- **Why it matters:** on wide tables with many indexes where an UPDATE touches
  one indexed column, skipping the unchanged indexes is a large write-amp
  reduction — but the correctness surface is exactly where index-corruption
  bugs live. [[access-method-apis]] · [[vacuum-autovacuum]] (HOT/prune).
