## 1. Why clearing the VM doesn't require registering vm buffer in wal record
- **Author / time:** Robert Haas, 2026-07-15 18:27 (+ Melanie Plageman 19:34, 21:58; Haas 20:10)
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Digs into the asymmetry: *setting* a visibility-map bit is WAL-logged and
  the VM buffer is registered in the record, but *clearing* the VM bit on a
  heap modification is not separately logged — the heap-update redo path
  re-clears it implicitly.
- The discussion pins down why this is crash-safe: a stale set VM bit is the
  dangerous direction (it can suppress a needed heap scan / index-only-scan
  recheck), a stale *cleared* bit only costs an extra visit, so under-clearing
  is never a correctness hazard while under-setting would be.
- Clarifying the invariant matters for anyone touching `visibilitymap.c` or
  the heap redo routines — it's exactly the kind of "obvious once stated"
  rule that a new AM or a VM-touching patch gets wrong.
- **Why it matters:** the set-is-logged / clear-is-implicit rule is a
  load-bearing MVCC+recovery invariant; getting it documented in-thread
  prevents the next index-only-scan visibility bug.
  `[[knowledge/skills/wal-and-xlog]]`, `[[knowledge/idioms/hint-bits-setbufferdirty]]`,
  `[[knowledge/skills/snapshot-management]]`.

## 2. walsummarizer can get stuck when switching timelines
- **Author / time:** Amit Kapila, 2026-07-15 06:28 (+ Robert Haas, Andrey Borodin, Thom Brown across the day)
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Reports the WAL summarizer (the PG 17+ process that feeds incremental
  backup) can wedge when a timeline switch happens mid-summarization,
  leaving summary files that never advance past the switch point.
- Thread converges on making the summarizer notice the timeline boundary and
  resume on the new TLI rather than spinning on the old one.
- Directly affects incremental-backup correctness: a stuck summarizer means
  `pg_basebackup --incremental` silently loses its reference point.
- **Why it matters:** incremental backup is only as trustworthy as the WAL
  summary chain; a timeline-switch hang is a real data-protection gap on
  standbys promoted mid-backup-window.
  `[[knowledge/skills/backup-and-recovery]]`, `[[knowledge/skills/wal-and-xlog]]`,
  `[[knowledge/skills/physical-replication]]`.

## 4. Fix FSM range endpoint after relation extension
- **Author / time:** Michael Paquier, 2026-07-15 01:10 (+ Melanie Plageman 14:07, 20:00)
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Fixes an off-by-one / stale-endpoint issue in the Free Space Map after a
  relation is extended: the FSM's notion of the highest addressable block
  can lag the true relation size, so newly extended pages aren't offered to
  inserters until a later FSM vacuum catches up.
- Melanie's review ties it to the `heap_multi_insert` / extension fast-path
  interaction where bulk loads extend many blocks at once.
- Manifests as inserters unnecessarily extending the relation further instead
  of reusing just-added free space — a bloat-adjacent inefficiency.
- **Why it matters:** the FSM range endpoint is exactly the kind of quiet
  accounting bug that shows up as unexplained relation growth under COPY /
  bulk INSERT. `[[knowledge/skills/free-space-map]]`,
  `[[knowledge/skills/buffer-manager]]`, `[[knowledge/skills/copy-family]]`.

## 5. Batching in executor
- **Author / time:** Denis Smirnov, 2026-07-15 07:12
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Revives the perennial proposal to move the executor from tuple-at-a-time
  `ExecProcNode` toward batch/vectorized processing of multiple tuples per
  call, to amortize per-tuple dispatch overhead.
- The hard part, as always, is the volcano-model contract: every node's
  `ExecProcNode` returns one slot, so batching requires either a parallel
  batched protocol or selective opt-in nodes.
- Early-stage design discussion, not a committable patch — but a recurring
  direction worth tracking for anyone in `execProcnode.c` / `nodeXxx.c`.
- **Why it matters:** batched execution is the single biggest structural
  lever on OLAP throughput PG hasn't pulled; even a scoped opt-in would
  reshape the executor's hot loop. `[[knowledge/skills/executor-and-planner]]`,
  `[[knowledge/skills/parallel-query]]`.

## 6. sequencesync worker race with REFRESH SEQUENCES
- **Author / time:** Hayato Kuroda, 2026-07-15 01:31 (+ Noah Misch, vignesh C, Amit Kapila, Tom Lane across the day)
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Reports a race between the sequence-sync worker (the logical-replication
  machinery that copies sequence values to a subscriber) and a concurrent
  `ALTER SUBSCRIPTION ... REFRESH PUBLICATION SEQUENCES`, where the refresh
  can observe or leave partially-synced sequence state.
- Tom Lane and Noah Misch weigh in on the correct locking / catalog-visibility
  boundary so the refresh either sees a fully-synced set or none.
- Sequence replication is newer machinery (subscriber-side sequence sync);
  its concurrency corners are still being mapped.
- **Why it matters:** logical replication of sequences is a headline feature
  whose correctness hinges on these worker/DDL races being closed before
  users lean on it in production. `[[knowledge/skills/logical-replication]]`.

## 7. Parallel Apply
- **Author / time:** Hayato Kuroda, 2026-07-15 05:02
- **Archive:** https://www.postgresql.org/list/pgsql-hackers/
- Continued work on applying logical-replication changes in parallel on the
  subscriber, spreading transaction apply across multiple workers instead of
  the single serial apply worker.
- The correctness core is preserving commit-order / dependency constraints
  between transactions that touch overlapping rows while still parallelizing
  the independent ones.
- One of the larger in-flight logical-replication throughput efforts this
  cycle.
- **Why it matters:** apply-side single-threading is the standard logical
  replication bottleneck under write-heavy publishers; parallel apply is the
  direct remedy. `[[knowledge/skills/logical-replication]]`,
  `[[knowledge/skills/parallel-query]]`.
