## 1. [PATCH] Prevent repeated deadlock-check signals in standby buffer pin waits

- Author: Xuneng Zhou (06-28 02:10; active same day — JoongHyuk Shin 10:05,
  Xuneng Zhou 12:52).
  Archive: <https://www.postgresql.org/search/?m=1&q=Prevent+repeated+deadlock-check+signals+in+standby+buffer+pin+waits>
- On a hot standby, a recovery-conflict buffer-pin wait arms the deadlock-check
  timeout; the patch reports the timer being re-armed repeatedly so the startup
  process keeps signalling pinning backends more often than the
  `deadlock_timeout` cadence intends.
- Fix narrows the re-signal path so a single buffer-pin wait doesn't generate a
  storm of `PROCSIG_RECOVERY_CONFLICT_BUFFERPIN` signals — one check per
  interval, not one per wakeup.
- Touches the startup-process side of recovery conflict handling, adjacent to
  `ResolveRecoveryConflictWithBufferPin` and the standby deadlock timer.
- **Why it matters:** signal storms to pinning backends are wasted work and can
  perturb wait-event accounting on busy standbys; this is a low-risk latency /
  noise fix in a corner that's hard to test. [[knowledge/subsystems/storage-lmgr.md]],
  [[knowledge/idioms/deadlock-detection.md]]

## 2. Report bytes and transactions actually sent downstream

- Author: Amit Kapila (06-28 06:56; continues into 06-29).
  Archive: <https://www.postgresql.org/search/?m=1&q=Report+bytes+and+transactions+actually+sent+downstream>
- Proposes that logical-replication WAL-sender stats count the bytes and
  transactions *actually shipped to the subscriber* rather than what was read
  from WAL — filtered/empty transactions currently inflate the read-side view.
- Aims to make `pg_stat_replication` / replication-slot statistics reflect true
  downstream traffic, which matters for monitoring filtered publications and
  row-filter / column-list publications.
- Sits on the WAL-sender + ReorderBuffer decode path where empty transactions
  are skipped before send.
- **Why it matters:** observability for logical replication is only as good as
  what the sender counts; conflating read vs sent bytes misleads capacity
  planning on heavily-filtered publications. [[knowledge/subsystems/replication.md]]

## 4. [PATCH] Avoid collation lookup for "char" statistics

- Author: Feng Wu (06-28 13:59; active — Tom Lane 15:39, 16:41; Feng Wu 16:53).
  Archive: <https://www.postgresql.org/search/?m=1&q=Avoid+collation+lookup+for+char+statistics>
- ANALYZE / selectivity code for the internal `"char"` type (single-byte, not
  `char(n)`) was doing a collation lookup it doesn't need, since `"char"` has no
  meaningful collation.
- Patch short-circuits the collation resolution for `"char"` columns in the
  stats / selfuncs path, avoiding a needless `lookup_collation` and the
  associated catalog touch.
- Tom Lane's replies steer the exact predicate (type OID check vs typcollation)
  so the skip is correct for all callers.
- **Why it matters:** a small but real per-column ANALYZE cost on catalogs and
  system columns of type `"char"`; also tightens the contract that `"char"` is
  collation-free. [[knowledge/subsystems/optimizer.md]],
  [[knowledge/idioms/analyze-mcv-histogram-correlation.md]]

## 5. Handle concurrent drop when doing whole database vacuum

- Author: Michael Paquier (06-29 00:13; active 4+ messages).
  Archive: <https://www.postgresql.org/search/?m=1&q=Handle+concurrent+drop+when+doing+whole+database+vacuum>
- A database-wide `VACUUM` builds a relation list up front; if another session
  drops one of those relations before the vacuum reaches it, the worker can hit
  a missing-relation error instead of silently skipping.
- Fix makes the per-relation open tolerant of concurrent drop (open-and-skip
  rather than error) so a whole-DB vacuum doesn't abort on a benign race.
- Touches `vacuum_rel` / the relation-list iteration in `vacuum()`, intersecting
  the invalidation path for dropped relations.
- **Why it matters:** whole-database VACUUM and autovacuum stragglers routinely
  race DDL; an abort here loses the entire pass, so open-and-skip is the correct
  robustness posture. [[knowledge/subsystems/access-heap.md]],
  [[knowledge/idioms/autovacuum-launcher.md]]

## 6. [PATCH] Change wait_time column of pg_stat_lock to double precision

- Author: Michael Paquier (06-29 02:11; very active — Bertrand Drouvot 04:27,
  04:56, 07:06; Michael Paquier 04:52).
  Archive: <https://www.postgresql.org/search/?m=1&q=Change+wait_time+column+of+pg_stat_lock+to+double+precision>
- The new `pg_stat_lock` view exposed lock wait time as an integer (ms); the
  thread switches it to `double precision` so sub-millisecond waits aren't
  truncated to zero.
- Catalog-visible type change on a system view definition → requires a
  catversion bump and `pg_proc` / view-definition adjustment.
- Discussion converges on double-ms (vs bigint-µs) to match the precision style
  of sibling stats columns.
- **Why it matters:** lock-wait observability is useless if every fast wait
  rounds to 0 ms; getting the column type right before the view ships in v20
  avoids a later incompatible change. [[knowledge/subsystems/storage-lmgr.md]],
  [[knowledge/idioms/catalog-conventions.md]]

## 7. Fix HAVING-to-WHERE pushdown with mismatched operator families

- Author: Tender Wang (06-29 02:31; continues 08:32, 10:59).
  Archive: <https://www.postgresql.org/search/?m=1&q=Fix+HAVING-to-WHERE+pushdown+with+mismatched+operator+families>
- The planner can push a `HAVING` clause down into `WHERE` when the grouping
  doesn't affect it, but the optimization mis-fired when the clause's operator
  belonged to a different operator family than the grouping equality, producing
  wrong results.
- Patch tightens the precondition so the pushdown only fires when the operator
  families are compatible, preserving grouping semantics.
- Lives in `subquery_planner` / the `hasHavingQual` lowering in `grouping_planner`.
- **Why it matters:** a wrong-results planner bug (not just a perf miss) — the
  most serious class. The op-family mismatch is exactly the kind of subtle
  equality-vs-ordering confusion that pushdown transforms are prone to.
  [[knowledge/subsystems/optimizer.md]]

## 8. [PATCH] Don't call ereport(ERROR) from recovery target GUC assign hooks

- Author: JoongHyuk Shin (06-29 04:45; active — Michael Paquier 05:47,
  JoongHyuk Shin 07:59).
  Archive: <https://www.postgresql.org/search/?m=1&q=Don%27t+call+ereport+ERROR+from+recovery+target+GUC+assign+hooks>
- `recovery_target*` GUC *assign* hooks could throw `ERROR`, but assign hooks
  run at a point (commit of a GUC change) where throwing is unsafe — validation
  belongs in the *check* hook, not the assign hook.
- Patch moves the rejection logic into the check-hook phase so an invalid
  recovery target is refused cleanly instead of erroring during assignment.
- Canonical GUC discipline: check-hook validates and may reject; assign-hook
  must be side-effect-only and non-throwing.
- **Why it matters:** an ERROR from an assign hook can leave GUC state
  inconsistent or destabilize recovery startup; this is a correctness +
  robustness fix in the recovery-target machinery. [[knowledge/subsystems/access-transam.md]],
  [[knowledge/idioms/crash-recovery-startup.md]]

## 9. GetBufferDescriptor() called for local buffers from MarkBufferDirtyHint()

- Author: Ayush Tiwari (06-29 07:59).
  Archive: <https://www.postgresql.org/search/?m=1&q=GetBufferDescriptor+local+buffers+MarkBufferDirtyHint>
- Reports that `MarkBufferDirtyHint()` reaches for `GetBufferDescriptor()`
  (shared-buffer descriptor array) on a buffer that may be a *local* (temp-table)
  buffer, where `GetLocalBufferDescriptor()` is the correct accessor.
- A mismatched descriptor lookup on a local buffer indexes the wrong array — a
  latent bug for hint-bit dirtying on temporary relations.
- Sits in `bufmgr.c`'s shared-vs-local buffer dispatch, one of the easiest
  places in PG to conflate the two buffer pools.
- **Why it matters:** local/shared buffer confusion is a classic source of
  hard-to-reproduce corruption; catching it in the hint-dirty path before it
  manifests is exactly the kind of audit win that matters.
  [[knowledge/subsystems/storage-buffer.md]]

## 10. coerce_type discards unnecessary CollateExprs

- Author: jian he (06-29 11:25; active — Tom Lane 13:38, jian he 13:54).
  Archive: <https://www.postgresql.org/search/?m=1&q=coerce_type+discard+unnecessary+CollateExprs>
- During type coercion the parser could wrap a node in a `CollateExpr` that is
  immediately redundant (the target type carries no collation or the same
  collation); patch strips these to keep the parse tree minimal.
- Cleaner trees mean fewer no-op `CollateExpr` nodes flowing into the planner
  and `ruleutils` deparse, reducing `EXPLAIN` / view-definition clutter.
- Tom Lane's reply scopes which coercion contexts may safely drop the collate
  wrapper without changing semantics.
- **Why it matters:** redundant collate nodes are a perennial source of
  deparse-roundtrip and plan-noise bugs; trimming at coercion time is the right
  layer. [[knowledge/subsystems/parser-and-rewrite.md]]

## 11. Per-thread leak in ECPG's memory.c

- Author: Thomas Munro (06-29 13:19; Thom Brown 14:30).
  Archive: <https://www.postgresql.org/search/?m=1&q=Per-thread+leak+in+ECPG+memory.c>
- ECPG's thread-local memory bookkeeping (`memory.c`) leaks an allocation per
  thread on teardown — surfaced by a sanitizer / valgrind run on the ECPG test
  suite.
- Fix frees the per-thread structure at thread exit so long-lived multi-threaded
  ECPG clients don't accumulate leaked blocks.
- Client-library (libecpg) territory rather than backend, but on the same
  memory-discipline contract.
- **Why it matters:** ECPG is the embedded-SQL preprocessor runtime; a
  per-thread leak compounds in connection-pooled / threaded apps and is the kind
  of slow bleed only a sanitizer catches. [[knowledge/subsystems/libpq-backend.md]]

## 12. Fix handling of copy_file_range() return value

- Author: Peter Eisentraut (06-29 11:12).
  Archive: <https://www.postgresql.org/search/?m=1&q=Fix+handling+of+copy_file_range+return+value>
- A `copy_file_range()` call site mishandled the syscall's partial-copy
  contract: like `read`/`write`, it can copy fewer bytes than requested and must
  be looped until completion, but the caller treated one short return as done.
- Patch wraps the call in the required retry loop so file cloning (used by
  `pg_upgrade` / `COPY`-style fast file copies) doesn't silently truncate.
- Portability-sensitive: `copy_file_range` semantics and availability vary
  across kernels/filesystems.
- **Why it matters:** a short-copy treated as complete is silent data loss in a
  file-copy fast path; this is a correctness fix in I/O plumbing that's easy to
  get subtly wrong. [[knowledge/subsystems/port.md]]
