## WAL / recovery

### 1. Fix "unexpected logical decoding status change" error from concurrent logical decoding activation
`/message-id/CAD21AoC_%3DBxTo4Xpr91_bn-JWGHb97tyxY4Vr0mfA83YutrAvw%40mail.gmail.com`

- The `pg_control` logical-decoding status field (the one whose catversion
  bump `99e949f846e9` this corpus tracked on 07-17) can flip under two
  backends racing to activate decoding, tripping an unexpected-status
  `elog`.
- Fix serializes the status transition so a concurrent activator observes a
  consistent value rather than an intermediate one.
- Ties directly to the checkpoint-record status the anchor-refresh flagged;
  a correctness follow-up to that same PG20 logical-decoding-status work.
- **Why it matters:** it's the concurrency hardening of the very
  `pg_control_checkpoint` status field the corpus just re-anchored — the
  gap between "field added" and "field safe under races." `[[knowledge/subsystems/wal-and-xlog.md]]`

### 2. Change checkpoint-record-missing PANIC to FATAL
`/message-id/CAMm1aWZpkS1UAsibG2W7zEcka%3DR20zk78-TgoBCfgj24pyJ4pw%40mail.gmail.com`

- Proposes demoting the "could not locate a valid checkpoint record" case
  from `PANIC` to `FATAL`, so a startup that can't find its checkpoint fails
  the backend cleanly instead of crashing the postmaster.
- The distinction matters operationally: PANIC implies shared-memory
  corruption and forces a full crash-recovery cycle; a missing checkpoint at
  startup is a recoverable-config situation (wrong `restore_command`, stale
  `backup_label`).
- Debate centers on whether any caller relies on PANIC's shmem-reset side
  effect.
- **Why it matters:** elevel discipline at the recovery boundary — exactly
  the PANIC-vs-FATAL judgment the `error-handling` skill governs.
  `[[knowledge/idioms/... error-handling]]`

### 3. walsummarizer can get stuck when switching timelines
`/message-id/CA%2BTgmobiOWKwec1koA6js27srZ%3DCuXy9OKuQ3BHgDM_GGtGfFw%40mail.gmail.com`

- The WAL summarizer (incremental-backup infrastructure, PG17+) can wedge
  when a timeline switch happens mid-summarization, leaving it spinning on a
  segment it will never see complete.
- Reported by the incremental-backup author line; fix teaches the summarizer
  to notice the TLI change and re-derive its read position.
- Relevant to anyone running `pg_basebackup --incremental` across a failover.
- **Why it matters:** incremental backup correctness across failover is a
  young code path (PG17); timeline-switch edge cases are still being flushed
  out. `[[knowledge/subsystems/backup-and-recovery]]`

### 4. Add a pg_wal_preallocate() SQL function to eagerly create future WAL segments
`/message-id/CAJTYsWXAPabSWBeT%3DbqJD%2B6w0D4-ozvoVoH7KiFUncM88bsQOg%40mail.gmail.com`

- New SQL function to pre-create N future WAL segment files up front, so a
  latency-sensitive workload never pays the segment-creation cost inline on
  the commit path.
- Overlaps conceptually with `file_extend_method` / `wal_init_zero`; the
  discussion probes whether a manual trigger beats the existing background
  preallocation.
- Interacts with the same file-extension machinery Christophe Pettus's
  `file_extend_method` GUC blog (07-22) describes.
- **Why it matters:** a user-visible knob for WAL write-stall avoidance;
  the cross-link to the planet blog is a nice corpus/community convergence.

## Buffer manager

### 5. [PATCH] Batched clock sweep to reduce cross-socket atomic contention
`/message-id/CANwKhkNSfSts%3DpsKPBYk3CL%2B0QYkqzFjUpmTv-Hyq%2Be9RQTY7w%40mail.gmail.com`

- Reworks the clock-sweep victim-buffer search to claim a *batch* of
  candidate buffers per pass rather than one at a time, cutting the number
  of cross-socket atomic operations on `nextVictimBuffer`.
- Targets the well-known NUMA scalability cliff where the single global
  sweep counter becomes a coherence hotspot at high core counts.
- Measurements on the thread show contention reduction on many-socket boxes;
  the risk is fairness/accuracy of the approximation-LRU under batching.
- **Why it matters:** direct scalability work on `bufmgr.c`'s replacement
  algorithm — the exact clock-sweep loop the buffer-manager skill documents.
  `[[knowledge/subsystems/buffer-manager.md]]`

## Executor / planner

### 6. Batching in executor
`/message-id/CA%2BHiwqGQPDwxK7QQNUWhqhAEwaH4Z%2BYQhHG5jNPYHGNzDMfsXw%40mail.gmail.com`

- Long-running design thread on making the executor pass batches of tuples
  between nodes (vectorized-style) instead of the one-tuple-at-a-time
  `ExecProcNode` volcano model.
- Touches slot representation, how `TupleTableSlot` would carry N tuples,
  and which nodes benefit first (agg, hashjoin, seqscan).
- A foundational-but-invasive change; the thread is scoping what a minimal
  first landing looks like.
- **Why it matters:** if it lands, it reshapes the `ExecProcNode` dispatch
  contract the executor-and-planner skill is built around.
  `[[knowledge/subsystems/executor-and-planner]]`

### 7. Tracking per-RelOptInfo uniqueness during planning
`/message-id/CAMbWs4-iLcqBr_n_F5gNrzQbBMrKgkpGwqTu7boWeoYepf%3D%2B8g%40mail.gmail.com`

- Proposes caching a "this relation is unique on these keys" fact on each
  `RelOptInfo`, so downstream join/agg planning can skip redundant
  distinct-ness proofs.
- Enables more aggressive join-removal and DISTINCT elimination without
  re-deriving uniqueness at every join level.
- Overlaps with the existing `remove_useless_joins` machinery (see thread 12
  below) — both mine uniqueness for plan simplification.
- **Why it matters:** uniqueness tracking is load-bearing for join removal;
  a per-RelOptInfo cache changes the planner's book-keeping surface.

### 8. hashjoins vs. Bloom filters (yet again)
`/message-id/b3288b86-de97-49e5-999a-bb773c2773c5%40vondra.me`

- Revives the perennial proposal to push a Bloom filter built from the hash
  side down to the probe (outer) side, so the scan can discard
  non-matching rows before they reach the join.
- "Yet again" signals prior attempts stalled on cost-model integration:
  when does the filter pay for itself vs. hurt?
- Vondra's framing focuses on selectivity thresholds and building the filter
  cheaply during the hash build.
- **Why it matters:** a classic executor/planner optimization that keeps
  almost-landing; the blocker is always the cost model, not the mechanics.

## Vacuum

- Adds visibility into which tables autovacuum/autoanalyze *skipped* (locked,

## Logical replication

### 10. sequencesync worker race with REFRESH SEQUENCES
`/message-id/CAJpy0uCP2zYJr%3DAa%3DFW-Kg6TsNmPj0yhgBb29RWybtbnaBpyvg%40mail.gmail.com`

- The new sequence-synchronization worker (PG18 sequence logical-rep work)
  can race a concurrent `ALTER SUBSCRIPTION ... REFRESH PUBLICATION SEQUENCES`,
  double-syncing or skipping a sequence.
- This is the same sequence-sync correctness vein `pg-upstream-watcher`
  flagged on 07-20/21 (subscriptionCheck ↔ sequence-sync buildfarm
  correlation); the hackers thread is the upstream discussion behind those
  commits.
- Fix adds interlock so the worker and the REFRESH command can't both drive
  a sequence's sync state simultaneously.
- **Why it matters:** sequence logical replication is brand-new surface; the
  corpus has been watching its buildfarm flakes — this is the root-cause
  thread. `[[knowledge/subsystems/logical-replication]]`

## Role concurrency

### 11. Fix races conditions in DropRole() and GrantRole()
`/message-id/al8/4SWljR9D%2B/%2Bf%40bdtpg`

- Concurrent `DROP ROLE` / `GRANT` can interleave such that a role is
  dropped while a grant referencing it is committing, leaving a dangling
  `pg_auth_members` row or tripping a cache-lookup failure.
- Fix tightens the locking around role catalog mutation so membership
  changes serialize against role removal.
- **Why it matters:** `gap:role-management` has been a *standing* harvester
  signal (flagged 07-19/21/22, queued for promotion to a missing-skill
  item) — this thread is live upstream evidence that role-mutation
  concurrency is under-documented in the corpus. Worth a knowledge leaf.
  `[[progress/_queues/... gap:role-management]]`
