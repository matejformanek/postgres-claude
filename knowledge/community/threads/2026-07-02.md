## 1. Add pg_stat_kind_info system view

- Authors: Michael Paquier, Bertrand Drouvot, Sami Imseih (07-01 01:35 →
  07-02 00:37, very active — 9 messages across the window).
  Archive: <https://www.postgresql.org/search/?m=1&q=Add+pg_stat_kind_info+system+view>
- Proposes a SQL-visible catalog view exposing the registered pgstat "kinds" —
  the entries in the fixed + variable `PgStat_KindInfo` registry that today are
  only reachable from C. Each row would surface a kind's name, id, whether it is
  fixed-numbered or variable-keyed, and its shared/snapshot sizing.
- Debate centres on (a) whether custom (extension-registered) stats kinds should
  appear, since their ids are assigned at load time and are not stable across
  restarts, and (b) how much of the internal `PgStat_KindInfo` struct is safe to
  expose without freezing it as an ABI-visible contract.
- Sami pushes for including the custom-kind id range so extension authors can
  introspect their own registration; Michael is cautious about pinning the
  layout. Consensus is forming around a minimal first cut (name + id +
  fixed/variable flag) with sizing columns deferred.
- **Why it matters:** the pgstat "kind" registry is the extensibility seam of the
  whole cumulative-statistics framework; a system view makes the otherwise-opaque
  registration observable and is a natural companion to custom stats kinds.
  `[[knowledge/idioms/pgstat-flush-timing]]`,
  `[[knowledge/data-structures/pgstat-counter]]`.

## 2. Prevent crash when calling pgstat functions with unregistered stats kind

- Authors: Bertrand Drouvot, Ewan Young, Michael Paquier (07-01 07:19 →
  07-02 05:02, very active — 10 messages, driven to a patch this window).
  Archive: <https://www.postgresql.org/search/?m=1&q=Prevent+crash+when+calling+pgstat+functions+with+unregistered+stats+kind>
- A SQL-callable pgstat function handed a stats-kind id that was never registered
  (e.g. a stale id from a dropped extension, or an out-of-range value) indexes
  past the `PgStat_KindInfo` registry and dereferences garbage → segfault rather
  than a clean error.
- The fix adds a bounds + registration check on the kind id at the entry points
  that accept a kind from SQL, raising `ERROR` for an unknown kind instead of
  reaching into the registry array. Discussion refines *which* helper is the
  right choke point (`pgstat_get_kind_info()` vs the per-function guards) so the
  check lives in exactly one place.
- Michael and Bertrand converge on centralising the validation in the kind-lookup
  helper so every caller inherits it; Ewan supplies a repro that crashes an
  assert-enabled build. A test exercising the unregistered-kind path is being
  added alongside.
- **Why it matters:** a pure robustness/security fix — user-supplied kind ids must
  never index the registry unchecked. Directly reinforces thread 1's
  introspection surface: exposing kinds via SQL makes bad-kind inputs more
  reachable. `[[knowledge/data-structures/pgstat-counter]]`,
  `[[knowledge/idioms/pgstat-flush-timing]]`.

## 3. Improve pg_stat_statements scalability

- Authors: Bertrand Drouvot, Kyotaro Horiguchi (07-01 05:17 → 07-02 06:24).
  Archive: <https://www.postgresql.org/search/?m=1&q=Improve+pg_stat_statements+scalability>
- Targets the lock contention on the `pg_stat_statements` shared hash under
  high query-id cardinality: the single LWLock guarding the entry table
  serialises inserts/evictions on busy multi-tenant workloads, and the dealloc
  (garbage-collection) pass takes the lock exclusively.
- Proposal explores finer-grained locking (partitioned entry table) and reducing
  the exclusive-lock window of the eviction sweep. Horiguchi raises the
  interaction with the query-text file compaction, which also serialises under
  the same lock.
- **Why it matters:** `pg_stat_statements` is the single most-deployed observability
  extension; its shared-hash lock is a known scaling ceiling on high-core boxes.
  `[[knowledge/subsystems/contrib-pg_stat_statements]]`,
  `[[knowledge/idioms/lwlock-rank-discipline]]`.

## 4. GetBufferDescriptor() called for local buffers from MarkBufferDirtyHint()

- Authors: Ashutosh Bapat, Ayush Tiwari, Michael Paquier (07-01 05:33 →
  07-02 04:59, continuing from 06-30).
  Archive: <https://www.postgresql.org/search/?m=1&q=GetBufferDescriptor+being+called+for+local+buffers+from+MarkBufferDirtyHint>
- A dirty-hint set on a *local* buffer (temp-table / local-buffer-manager page)
  reaches `MarkBufferDirtyHint()`, which unconditionally calls the *shared*
  `GetBufferDescriptor()` — a shared-pool lookup keyed by a buffer id that, for a
  local buffer, actually indexes the local buffer array. The negative buffer-id
  convention that distinguishes local from shared buffers is not honoured on this
  path.
- The window's debate settles the fix: branch on `BufferIsLocal()` and route to
  `GetLocalBufferDescriptor()` for local buffers, mirroring the split every other
  buffer-touching routine already makes. Ayush confirms the mislookup is benign
  today only because the arrays happen to overlap in the tested configs — it is a
  latent corruption bug, not cosmetic.
- **Why it matters:** the shared/local buffer-descriptor split is a load-bearing
  invariant; any path that forgets it can dirty or pin the wrong descriptor.
  `[[knowledge/subsystems/storage-buffer]]`,
  `[[knowledge/data-structures/bufferdesc-state]]`,
  `[[knowledge/idioms/hint-bits-setbufferdirty]]`.

## 5. Fix HAVING-to-WHERE pushdown with mismatched operator families

- Authors: Tender Wang, Richard Guo (07-01 01:49 → 05:58).
  Archive: <https://www.postgresql.org/search/?m=1&q=Fix+HAVING-to-WHERE+pushdown+with+mismatched+operator+families>
- The planner optimisation that pushes a HAVING clause down into WHERE (valid
  when the clause references no aggregates) can fire even when the equality
  operator in the clause belongs to a *different* operator family than the one
  the grouping/pushdown reasoning assumes — yielding wrong results for types with
  multiple incompatible equality semantics.
- Fix tightens the pushdown eligibility test to require operator-family
  compatibility before relocating the qual. Richard reviews the family-matching
  predicate against the existing `equality_ops_are_compatible` machinery.
- **Why it matters:** a wrong-results correctness fix in a hot planner
  transformation — silent wrong answers are the worst failure class.
  `[[knowledge/subsystems/optimizer]]`,
  `[[knowledge/idioms/view-pushdown-via-rewriter]]`.

## 6. Handle concurrent drop when doing whole database vacuum

- Authors: cca5507, Kyotaro Horiguchi (07-01 07:21 → 07-02 06:03, continuing).
  Archive: <https://www.postgresql.org/search/?m=1&q=Handle+concurrent+drop+when+doing+whole+database+vacuum>
- A database-wide `VACUUM` snapshots the relation list up front, then iterates;
  if another session drops one of those relations mid-pass, the vacuum errors out
  on a now-missing relation instead of skipping it. The fix makes the per-relation
  step tolerate a concurrently-dropped OID (re-check existence under lock, skip
  cleanly) rather than aborting the whole command.
- Horiguchi refines the lock/recheck ordering so the skip is race-free: acquire
  the relation lock, then re-test the catalog tuple's liveness before proceeding.
- **Why it matters:** whole-database vacuum is routine autovacuum-adjacent work;
  a concurrent DROP should degrade to a skip, not a failed maintenance run.
  `[[knowledge/idioms/xmin-horizon-management]]`,
  `[[knowledge/subsystems/access-heap]]`.

## 7. Fix race in ReplicationSlotRelease for ephemeral slots

- Authors: Amit Kapila, Xuneng Zhou (07-02 04:26 → 04:45, new).
  Archive: <https://www.postgresql.org/search/?m=1&q=Fix+race+in+ReplicationSlotRelease+for+ephemeral+slots>
- An ephemeral (RS_EPHEMERAL) replication slot — one created but not yet
  persisted — can be released concurrently with its own cleanup, so
  `ReplicationSlotRelease()` races the drop path that removes an ephemeral slot on
  error. The window opens the thread with a proposed ordering fix that takes the
  slot's mutex around the active-PID clear and the ephemeral-drop decision.
- **Why it matters:** ephemeral slots back the create-slot handshake for logical
  decoding and base backups; a release/drop race can leak a slot or double-free
  its shared state. `[[knowledge/idioms/replication-slot-advance]]`,
  `[[knowledge/subsystems/replication]]`.

## 8. satisfies_hash_partition crash

- Authors: Robert Haas, Tender Wang, Ewan Young (07-01 19:11 → 07-02 05:43,
  active).
  Archive: <https://www.postgresql.org/search/?m=1&q=satisfies_hash_partition+crash>
- The `satisfies_hash_partition()` support function (used by hash-partition
  constraint checking) crashes on a particular argument shape — the window's
  repro passes a modulus/remainder combination or datum set that trips an
  unchecked assumption about the partition's bound collection.
- Robert opens with the crash; Tender and Ewan bisect the offending input and
  propose a guard on the modulus/remainder validity before the hash computation.
- **Why it matters:** hash-partition constraint evaluation runs on every tuple
  routed to a hash-partitioned table; a crash here is a availability bug reachable
  from ordinary DML. `[[knowledge/subsystems/partitioning]]`,
  `[[knowledge/idioms/partition-bound-comparison]]`.

## 9. Batching in executor

- Author: Amit Langote (07-01 09:18, new proposal).
  Archive: <https://www.postgresql.org/search/?m=1&q=Batching+in+executor>
- Design thread proposing that plan nodes exchange *batches* of tuples rather
  than one `TupleTableSlot` per `ExecProcNode()` call — amortising the per-tuple
  dispatch overhead of the Volcano iterator model over a vector of tuples, the
  precondition for later vectorised execution.
- Langote sketches the slot-vector interface, the opt-in per-node negotiation
  (a node advertises batch support; parents fall back to single-tuple for
  non-batching children), and the EXPLAIN/instrumentation implications. Early
  replies weigh it against the existing per-tuple memory-context reset discipline.
- **Why it matters:** executor batching is the long-discussed foundation for
  vectorised execution in PostgreSQL; even a partial adoption changes the
  `ExecProcNode` contract every node implements.
  `[[knowledge/subsystems/executor]]`,
  `[[knowledge/idioms/expression-evaluator-flow]]`.

## 10. WIP: replacing join_collapse_limit with "join hardness" estimate

- Author: Tomas Vondra (07-01 19:17, new WIP).
  Archive: <https://www.postgresql.org/search/?m=1&q=replacing+join_collapse_limit+with+join+hardness+estimate>
- Proposes retiring the blunt `join_collapse_limit` / `from_collapse_limit`
  cutoffs — which cap join-search size by a fixed count of relations — in favour
  of an estimated "hardness" of the join problem, so the planner spends its
  enumeration budget on genuinely expensive join searches and collapses trivially
  cheap ones without the arbitrary count ceiling.
- WIP posts the estimator sketch: weight the search cost by relation count *and*
  the connectivity/selectivity of the join graph, then gate GEQO / exhaustive
  search on the estimate rather than the raw count. Vondra flags the GUC-migration
  and plan-stability questions as open.
- **Why it matters:** the collapse limits are a decades-old planner heuristic with
  well-known pathologies (a single fixed count mis-serves both star and chain
  schemas); a hardness estimate is a structural improvement to join-search
  budgeting. `[[knowledge/subsystems/optimizer]]`,
  `[[knowledge/data-structures/plannerinfo]]`, `[[knowledge/idioms/cost-join-paths]]`.

## 11. Avoid recalculating pgprocno in ProcArrayAdd()

- Author: solai v (07-01 11:03).
  Archive: <https://www.postgresql.org/search/?m=1&q=Avoid+recalculating+pgprocno+in+ProcArrayAdd>
- Micro-optimisation in `ProcArrayAdd()`: the routine recomputes a proc's
  `pgprocno` (its index into the global `PGPROC` array) when it is already known
  to the caller, doing a redundant pointer-subtraction under the ProcArrayLock.
  The patch threads the known value through instead.
- **Why it matters:** `ProcArrayAdd()` runs under the exclusive `ProcArrayLock` on
  every backend startup; trimming work inside that critical section is a small but
  real contention win, and this is one of the RfC watch-list entries (CF #6426)
  carried since the 06-25 CommitFest snapshot.
  `[[knowledge/data-structures/pgproc-fields]]`,
  `[[knowledge/subsystems/storage-ipc]]`.

## 12. Changing the state of data checksums in a running cluster

- Author: Palak Chaturvedi (07-02 05:06, new).
  Archive: <https://www.postgresql.org/search/?m=1&q=Changing+the+state+of+data+checksums+in+a+running+cluster>
- Revives online data-checksum enabling/disabling: flip the cluster's checksum
  state without an offline `pg_checksums` pass, by driving a background scan that
  checksums (or clears) every page and coordinating the in-progress state across
  backends via the control file + a shared flag so concurrent writers agree on
  whether to compute checksums.
- The window's post lays out the state machine (off → in-progress → on) and the
  crash-safety requirement: the in-progress state must survive restart and resume,
  and a torn transition must never leave pages half-checksummed-yet-marked-on.
- **Why it matters:** enabling checksums today requires downtime; an online
  transition is a long-requested operational feature whose hard part is exactly
  the crash-safe state coordination across all backends.
  `[[knowledge/subsystems/access-transam]]`,
  `[[knowledge/idioms/checkpoint-coordination]]`.
