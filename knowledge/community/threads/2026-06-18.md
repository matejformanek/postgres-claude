## 1. TID recycling race during nbtree index-only scans on a standby
- **Author:** Peter Geoghegan · 22:16
- **URL:** https://www.postgresql.org/message-id/CAH2-Wzkro678xEDOrsxrq-YDVP4w7nXybhVJV2MxrzTj-5OU4w%40mail.gmail.com
- Index-only scan on a hot standby can observe a heap TID that the primary
  has already recycled (LP_DEAD → reuse), because the standby's visibility
  horizon and the primary's page-level kill bits can disagree across replay.
- Touches the interaction between `_bt_killitems` LP_DEAD setting and the
  recovery conflict / `ResolveRecoveryConflictWithSnapshot` machinery.
- Proposed fix direction: gate index-only-scan TID trust on standby by the
  recovery snapshot horizon rather than the page kill bits alone.
- **Why it matters:** a wrong-answer (not crash) hazard on read replicas — the
  worst class of MVCC bug because it's silent. [[knowledge/subsystems/access-nbtree.md]]

## 2. Deadlock detector fails to activate on a hot standby replica
- **Author:** Fujii Masao · 02:48 (Jun 18)
- **URL:** https://www.postgresql.org/message-id/CAHGQGwF01sOHnh6_9yDWiAdBMNnGWsTpM89%2B8XeztMjV%3Dxk%2Bcw%40mail.gmail.com
- On a standby the deadlock-timeout path doesn't arm correctly for backends
  blocked on recovery-conflict / buffer-pin waits, so a genuine wait cycle
  can hang past `deadlock_timeout` without the detector running.
- Report distinguishes the startup (recovery) process's wait handling from a
  normal backend's `CheckDeadLock` arming.
- **Why it matters:** the silent-LWLock-style hang failure mode, but on the
  replica side; exactly the class the locking corpus warns about.
  [[knowledge/idioms/deadlock-detection.md]]

## 3. 048_vacuum_horizon_floor.pl hangs due to lost wakeup in LockBufferForCleanup
- **Author:** Alexander Lakhin · 20:00
- **URL:** https://www.postgresql.org/message-id/7685519a-0bf9-4e17-93ca-7e3aa10fa29c%40gmail.com
- A TAP test intermittently hangs: a backend waiting in `LockBufferForCleanup`
  misses the wakeup when the last pin is dropped, so the cleanup lock is never
  granted.
- Points at the buffer-pin wait-queue / `UnpinBuffer` signalling path racing
  with `ProcWaitForSignal`.
- **Why it matters:** cleanup-lock acquisition is on the VACUUM hot path; a
  lost-wakeup here stalls vacuum indefinitely. [[knowledge/subsystems/storage-buffer.md]]

## 4. Commit Sequence Numbers and Visibility

## 5. Fix race in ReplicationSlotRelease for ephemeral slots
- **Author:** Xuneng Zhou · 07:08
- **URL:** https://www.postgresql.org/message-id/CABPTF7Vyt13sex0VCQaVSw-4Hiru5%3DnL9S-9GkfRDQAMwUFe8A%40mail.gmail.com
- An ephemeral (RS_EPHEMERAL) slot released concurrently with its drop can
  hit a window where `ReplicationSlotRelease` touches a slot another backend
  is freeing.
- Patch tightens the slot's `active_pid` / in-use handshake under
  `ReplicationSlotControlLock`.
- **Why it matters:** ephemeral slots back base-backup and short-lived logical
  setup; a release race risks slot-state corruption.
  [[knowledge/idioms/logical-decoding-snapshot.md]]

## 6. [Bug Report + Patch] File descriptor leak when io_method=io_uring
- **Author:** Lucas DRAESCHER · 09:09
- **URL:** https://www.postgresql.org/message-id/PFlQhqaAtrdF4rJvytflIJ7SVtFmeEhF14OaL3bpGotMDnsJE6ht32yyMEYJTYrVp5iFXwOVvQMbgMG9zqkmocmNRAIKgm4NCrJ0gJxWoEg%3D%40draescher.fr
- Each backend opening an io_uring instance leaks the ring fd on certain
  error/exit paths, exhausting the per-process fd limit under many short
  connections.
- Maps to **CommitFest #6617** (same author, carried on the RfC/NR watch list).
- **Why it matters:** io_uring is the newest AIO backend; an fd leak gates its
  production readiness. [[knowledge/subsystems/storage-buffer.md]]

## 7. faulty error handling around pgstat_count_io_op_time()
- **Author:** Michael Paquier · 00:26
- **URL:** https://www.postgresql.org/message-id/ajHprjZERUqMP2Kj%40paquier.xyz
- An ereport thrown between the `INSTR_TIME` start capture and
  `pgstat_count_io_op_time()` leaves the IO timing accounting half-updated.

## 8. Fix tuple deformation with virtual generated NOT NULL columns
- **Author:** David Rowley · 05:04 (very active thread, 7+ replies)
- **URL:** https://www.postgresql.org/message-id/CAApHDvrJBXEhet4%3DEs_wHBKdv5PCV5OGCaJOSmJexeaFqfmUHA%40mail.gmail.com
- `slot_deform_heap_tuple` mishandles a virtual generated column marked NOT
