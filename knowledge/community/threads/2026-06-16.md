## 1. BM_IO_ERROR flag is lost in TerminateBufferIO due to order of operations in UnlockBufHdrExt
**Author:** Andres Freund · 2026-06-15 16:18

- Reports that `TerminateBufferIO` can drop the `BM_IO_ERROR` flag: the
  ordering of operations in `UnlockBufHdrExt` clears/overwrites buffer-header
  state such that an I/O that failed is no longer marked as failed.
- The consequence is a buffer whose read errored can be treated as valid by a
  subsequent pinner — a correctness hazard, not just a stats glitch.
- Freund (who owns much of the recent bufmgr / AIO rework) filed it, so it sits
  right on the BufferDesc state-machine that the async-I/O path leans on.

**Why it matters:** the BufferDesc flag word is the single source of truth for
"is this buffer's content trustworthy"; losing `BM_IO_ERROR` is exactly the
class of bug that silently serves corrupt/zero pages. [inferred]
[[knowledge/subsystems/storage-buffer.md]] · [[knowledge/data-structures/bufferdesc.md]]

## 2. [PATCH] Fix PITR pause bypass when initial XLOG_RUNNING_XACTS has subxid overflow
**Author:** Jan Nidzwetzki · 2026-06-15 08:58

- Point-in-time recovery can blow past its configured pause target when the
  first `XLOG_RUNNING_XACTS` record seen during replay has the subxid-overflow
  bit set, so the snapshot-consistency bookkeeping mis-sequences the pause check.
- The fix tightens when recovery decides it has reached a consistent/pausable
  state so `recovery_target` + `pause` is honored even on the overflow path.
- Touches the startup-process replay loop, where ordering between consistency,
  pause, and target-reached decisions is notoriously subtle.

**Why it matters:** a PITR pause that doesn't pause defeats the whole point of
recovery_target — operators rely on it to stop exactly before a bad
transaction. [inferred] [[knowledge/subsystems/access-transam.md]] ·
[[knowledge/architecture/wal.md]]

## 3. Randomize B-Tree page split location to avoid oscillating patterns
**Author:** Dmitry Dolgov · 2026-06-15 10:13

- Proposes injecting randomness into where a btree page split picks its split
  point, to break pathological oscillating-split patterns that some insertion
  orders fall into (repeated near-50/50 splits churning the same pages).
- Trades a touch of fill-factor determinism for avoiding the degenerate case
  where index bloat/IO amplifies under adversarial key arrival order.
- Sits on `_bt_findsplitloc` heuristics — a much-tuned area where any change has
  to defend against regressing the common monotonic-insert fast path.

**Why it matters:** btree split-location heuristics decide index density and
write amplification for every insert-heavy workload; an oscillation fix is a
real-world bloat win if it doesn't dent the append-only case. [inferred]
[[knowledge/subsystems/access-nbtree.md]] · [[knowledge/data-structures/btree-page-layout.md]]

## 4. [PATCH] REPLICA IDENTITY USING INDEX accepts column with invalid NOT NULL
**Author:** Álvaro Herrera · 2026-06-15 16:50

- `REPLICA IDENTITY USING INDEX` is supposed to require the index's columns be
  NOT NULL, but the check accepts a column whose NOT NULL constraint is itself
  marked invalid (not yet validated), which violates the replica-identity
  contract.
- A nullable replica-identity key breaks the assumption logical decoding and
  apply make when building the old-tuple key for UPDATE/DELETE.
- Herrera (working the NOT NULL-as-catalogued-constraint refactor) ties it to
  the broader invalid-constraint bookkeeping.

**Why it matters:** replica identity is the join key logical replication uses to
locate the target row downstream; a NULL leaking in produces silent mis-apply on
subscribers. [inferred] [[knowledge/architecture/replication.md]] ·
[[knowledge/subsystems/access-heap.md]]

## 7. [oauth] Increased CPU usage during device flow with libcurl 8.20.0
**Author:** Jacob Champion (with Tom Lane) · 2026-06-15 19:28–23:20

- The built-in OAuth device-flow client spins CPU during the authorization-wait
  poll when built against libcurl 8.20.0 — a behavior change in curl's
  multi-handle/timeout semantics the PG side busy-loops against.
- Champion (the OAuth author) and Tom Lane traded diagnosis the same evening on
  whether the fix is a curl-version guard or a corrected wait/timeout in PG's
  `curl_multi` integration.
- Concrete portability bug: PG's libcurl usage tracks a moving external API and
  this is a regression triggered by a specific curl release.

**Why it matters:** the OAuth flow is new client-auth surface; a busy-loop tied
to a common curl version burns cycles on every device-flow login and erodes
trust in the feature's polish. [inferred] [[knowledge/subsystems/libpq-backend.md]]

## 8. synchronized_standby_slots behavior inconsistent with quorum-based synchronous replication
**Author:** Ashutosh Sharma · 2026-06-15 10:39

- `synchronized_standby_slots` (the GUC that holds back logical slot advancement
  until named physical standbys confirm) behaves inconsistently when synchronous
  replication is configured in quorum (`ANY k (...)`) mode rather than
  first-/priority-based.
- The mismatch: the slot-sync wait can require *all* listed standbys while the
  sync-rep quorum only needs *k*, so logical decoding stalls behind standbys the
  quorum already considers optional.
- Raises the question of whether slot synchronization should track the same
  quorum predicate as `synchronous_standby_names`.

**Why it matters:** this is the failover-safety knob for logical slots on
standbys; an over-strict wait silently throttles logical replication throughput
in quorum-sync deployments. [inferred] [[knowledge/architecture/replication.md]]

## 9. [PATCH] Fix sequence value may rollback after CREATE DATABASE TEMPLATE with WAL_LOG strategy
**Author:** Daisuke Higuchi · 2026-06-15 04:01

- With `CREATE DATABASE ... STRATEGY WAL_LOG`, a sequence's on-disk value in the
  newly-created database can effectively roll back after crash recovery — the
  WAL_LOG copy path doesn't durably capture the sequence's advanced state.
- Sequences are a special case: their "current value" lives in a relation page
  that the WAL_LOG createdb strategy must copy with the right LSN/flush ordering.
- The fix is in the createdb WAL_LOG block-copy path's handling of the sequence
  relation.

**Why it matters:** WAL_LOG is the modern default-ish createdb strategy; a
sequence that rewinds after recovery means duplicate generated keys in the new
database — a data-integrity bug, not a cosmetic one. [inferred]
[[knowledge/architecture/wal.md]] · [[knowledge/subsystems/access-heap.md]]

## 10. Fast-path FK checks reject valid inserts for domain-typed FK columns
**Author:** Amit Langote · 2026-06-15 11:19

- The RI fast-path that short-circuits foreign-key checks mis-handles FK columns
  whose type is a domain over the referenced type, rejecting inserts that are in
  fact valid.
- Root cause is the fast-path's type/operator lookup not unwrapping the domain to
  the base type when matching the PK side, so the equality check is built wrong.
- Langote frames it as a correctness regression in the referential-integrity
  trigger machinery, with a targeted fix in the RI query construction.

**Why it matters:** a valid INSERT being rejected by an FK check is a hard
user-facing correctness break; domain-over-type FK columns are common in
schemas that lean on domains for validation. [inferred]
[[knowledge/subsystems/executor.md]] · [[knowledge/idioms/fmgr.md]]

## 11. mxid_score can become Infinity in pg_stat_autovacuum_scores
**Author:** Bharath Rupireddy · 2026-06-15 18:20

- The newly-added `pg_stat_autovacuum_scores` view can report `Infinity` for the
  multixact-based score component when a divisor (members/offset budget) is zero
  or the multixact horizon math degenerates.
- An `Infinity` score corrupts the prioritization autovacuum uses to pick the
  next table, so a legitimately urgent table can be starved or mis-ranked.
- Fix is to clamp/guard the score computation against the degenerate-divisor
  case.

**Why it matters:** autovacuum scoring is the new mechanism for steering vacuum
toward multixact/wraparound pressure; a poisoned score defeats the very
prioritization it exists to provide. [inferred]
[[knowledge/idioms/autovacuum-launcher.md]] · [[knowledge/subsystems/access-heap.md]]

## 12. Adding a stored generated column without long-lived locks
**Author:** Laurenz Albe · 2026-06-15 18:38

- Proposes letting `ALTER TABLE ... ADD COLUMN ... GENERATED ALWAYS AS (...)
  STORED` avoid the full-table-rewrite-under-AccessExclusive that the operation
  forces today, so adding a derived column doesn't lock out the table.
- The hard part is materializing the stored values for existing rows without
  holding a strong lock for the whole rewrite — likely a phased/online approach
  akin to other concurrent DDL.
- Discussion is at the design stage: which lock level is actually safe and how
  to backfill generated values consistently with concurrent DML.

**Why it matters:** long-lived AccessExclusive locks on ALTER TABLE are a top
operational pain; making stored-generated-column adds online removes a real
maintenance-window blocker. [inferred]
[[knowledge/subsystems/storage-lmgr.md]] · [[knowledge/idioms/locking-overview.md]]
