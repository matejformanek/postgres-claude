### 1. Fix race condition in SSI when reading PredXact->SxactGlobalXmin
- **Author:** Josh Curtis · new thread 2026-06-12 01:04. `[from-website]`
- `SxactGlobalXmin` is the serializable-transaction global xmin tracked under
  `SerializableXactHashLock`; reading it without the right lock (or with a
  torn read on a non-atomic field) can let `SummarizeOldestCommittedSxact` or
  the SLRU-trimming path act on a stale bound. `[inferred]`
- This is the **second SSI-xmin race report in the 59 cycle** (cf. CF #6037
  enumerated in the 06-10/06-11 CommitFest snapshots), suggesting the
  predicate-lock xmin bookkeeping is getting renewed scrutiny. `[inferred]`
- **Why it matters:** SSI correctness bugs are silent — a too-aggressive xmin
  advance drops `SIRead` locks early and can let a serialization anomaly slip
  through with no error. cf. [[knowledge/subsystems/storage-lmgr.md]],
  [[knowledge/data-structures/snapshot-lifecycle.md]].

### 2. Fix race in ReplicationSlotRelease for ephemeral slots
- **Author:** Xuneng Zhou · 2026-06-12 02:52, reply from Amit Kapila 10:54.
  `[from-website]`
- `RS_EPHEMERAL` slots are dropped automatically when their owning backend
  releases them; the race is between `ReplicationSlotRelease` clearing
  `active_pid` and the cleanup path that frees/drops the ephemeral slot, where
  a concurrent acquirer could observe an inconsistent slot. `[inferred]`
- Amit Kapila's same-day engagement (a logical-replication committer) puts
  this on a fast review track. `[inferred]`
- **Why it matters:** ephemeral slots back `pg_basebackup`,
  `CREATE SUBSCRIPTION`'s initial sync, and temporary decoding; a release race
  can leak a slot or wedge `max_replication_slots`. cf.
  [[knowledge/subsystems/replication.md]].

### 3. Deadlock detector fails to activate on a hot standby replica
- **Author:** Vitaly Davydov · 2026-06-12 06:00, reply from Fujii Masao 09:23.
  `[from-website]`
- On a hot standby, recovery-conflict resolution and the
  `deadlock_timeout`-driven `CheckDeadLock` path interact differently than on a
  primary; the report is that a genuine lock cycle among read-only backends (or
  between a backend and the startup process) isn't being detected/broken.
  `[inferred]`
- Fujii Masao (a recovery-area committer) replying same-day signals this is
  treated as a real recovery-side lock-manager gap, not a config issue.
  `[inferred]`
- **Why it matters:** a non-firing deadlock detector means a hung standby
  backend never gets the `ERROR: deadlock detected` escape hatch and sits until
  a statement/idle timeout — or forever. cf.
  [[knowledge/subsystems/storage-lmgr.md]],
  [[knowledge/idioms/locking-overview.md]].

### 4. Commit Sequence Numbers and Visibility
- **Author:** Matthias van de Meent · 2026-06-12 10:03, reply from Jeff Davis
  2026-06-13 00:25. `[from-website]`
- A CSN-based visibility design recurs roughly once per release cycle: replace
  (or augment) the snapshot's xip array + `XidInMVCCSnapshot` linear scan with
  a monotonic commit-sequence-number stamped at commit, so visibility becomes
  an O(1) CSN compare rather than an O(snapshot-size) membership test.
  `[inferred]`
- Two senior contributors (van de Meent on storage/access, Davis on the
  executor/snapshot side) engaging suggests a serious design re-float, not a
  drive-by. `[inferred]`
- **Why it matters:** CSN visibility is the single highest-leverage change to
  the MVCC core — it touches `SnapshotData`, `HeapTupleSatisfies*`,
  `GetSnapshotData` contention, and hot-standby feedback all at once. Any
  movement here is worth tracking closely. cf.
  [[knowledge/data-structures/snapshot-lifecycle.md]],
  [[knowledge/subsystems/access-heap.md]].

### 5. Remove redundant DISTINCT when GROUP BY already guarantees uniqueness
- **Author:** David Rowley · 2026-06-12 03:19, reply from Andres Taylor 07:47.
  `[from-website]`
- When a query has both `GROUP BY` and a `DISTINCT` whose distinct-clause is a
  subset of (or functionally determined by) the grouping columns, the
  `DISTINCT` is provably a no-op and the upper unique/agg node can be elided in
  the planner. `[inferred]`
- A Rowley-authored planner micro-optimization with same-day review traffic —
  the usual fast path for self-contained `createplan.c`/`planner.c` tidy-ups.
  `[inferred]`
- **Why it matters:** removing a redundant uniquification node saves a sort or
  hash-agg on the query's widest intermediate result — a real win on
  reporting/OLAP queries that pile `DISTINCT` on top of `GROUP BY`. cf.
  [[knowledge/subsystems/optimizer.md]].

### 6. Subquery pull-up increases jointree search space
- **Author:** Andrei Lepikhov · 2026-06-13 10:07, replies from Tomas Vondra
  (13:04, 13:54). `[from-website]`
- Pulling a subquery up into the parent jointree (`pull_up_subqueries`) is
  usually a win, but it can explode the number of relations the join-search
  (GEQO threshold / `join_collapse_limit`) must consider, turning a cheap
  subplan into an expensive planning problem. `[inferred]`
- Vondra's two same-day replies indicate an active design discussion about
  whether/when pull-up should be gated by the resulting join-search size.
  `[inferred]`
- **Why it matters:** this is the classic planner tension between a better plan
  shape and planning *time*; getting the heuristic wrong regresses either plan
  quality or planning latency on large queries. cf.
  [[knowledge/subsystems/optimizer.md]],
  [[knowledge/subsystems/parser-and-rewrite.md]].

### 7. PG19 FK fast path: OOB write and missed FK checks during batched inserts
- **Author:** Amit Langote · 2026-06-12 02:46; related thread *Fast-path FK
  checks reject valid inserts for domain-typed FK columns* (Ewan Young 03:36,
  Amit Langote 04:02). `[from-website]`
- The PG19 RI fast-path (the in-executor FK check that skips the SPI query for
  simple cases) has two defects: an out-of-bounds write during batched inserts,
  and over-eager rejection when the FK column is a domain over the referenced
  type. `[inferred]`
- This is a **new-feature-in-PG19 correctness regression** — an OOB write is a
  memory-safety bug, the highest-severity class, and the same fast path also
  produces wrong answers (both false rejects and missed checks). `[inferred]`
- **Why it matters:** a missed FK check corrupts referential integrity
  silently; the OOB write is a crash/exploit surface. Both gate the PG19
  fast-path feature. cf. [[knowledge/subsystems/executor.md]],
  [[knowledge/idioms/catalog-conventions.md]].

### 8. amcheck: fix bug of missing corruption in allequalimage validation
- **Author:** Fujii Masao · 2026-06-12 00:42, reply from Chao Li 00:51.
  `[from-website]`
- `amcheck`'s B-tree verification has an `allequalimage` (deduplication-safe
  opclass) code path that, due to the bug, fails to flag a class of corrupted
  index pages — so the verifier returns clean on a genuinely broken index.
  `[inferred]`
- **Why it matters:** a false-negative in the corruption checker is worse than
  no checker — operators trust an `amcheck` pass and skip a reindex. The fix
  restores detection coverage for dedup-capable opclasses. cf.
  [[knowledge/subsystems/access-nbtree.md]].

### 9. Better shared data structure management and resizable shared data structures
- **Author:** Heikki Linnakangas · 2026-06-12 15:37, reply from Haoyu Huang
  15:51. `[from-website]`
- Today most shmem structures are sized once at postmaster start from GUCs and
  never grow; the proposal is infrastructure for shared data structures that
  can be resized at runtime (e.g. grow the buffer mapping table or a lock
  table without a restart). `[inferred]`
- A Heikki-authored shmem-infrastructure thread is architecturally significant
  — resizable shmem is a long-standing wishlist item that unblocks online
  `shared_buffers` resize and elastic lock tables. `[inferred]`
- **Why it matters:** the "everything sized at startup" model is the root cause
  of why `shared_buffers`/`max_locks_per_transaction` need a restart; generic
  resizable-shmem plumbing is foundational. cf.
  [[knowledge/subsystems/storage-ipc.md]],
  [[knowledge/subsystems/storage-buffer.md]].

### 10. LLVM JIT: any JIT-compiled query crashes (SIGILL) on a libLLVM 19 + ASAN build
- **Author:** Henson Choi · 2026-06-12 01:48. `[from-website]`
- On a build linking libLLVM 19 with AddressSanitizer, every JIT-compiled
  expression/tuple-deform routine traps with SIGILL — likely an ASAN-inserted
  instruction or a code-model/relocation mismatch in the ORC-JIT-emitted code
  under the newer LLVM ABI. `[inferred]`
- **Why it matters:** this blocks the sanitizer build that the project (and
  this meta-repo's ASan profile, `/setup-pg-asan`) relies on for memory-bug
  hunting whenever `jit=on`; the practical workaround is `jit=off` under ASAN
  until fixed. cf. [[knowledge/subsystems/jit.md]].

### 11. injection_points: switch wait/wakeup to use atomics rather than latches
- **Author:** Andrey Borodin · 2026-06-12 07:02. `[from-website]`
- The `injection_points` test infrastructure currently parks a waiting backend
  on a latch and wakes it via `SetLatch`; the proposal moves the
  wait/wakeup handshake to a shared atomic (spin/poll on a
  `pg_atomic_uint32`), removing latch-signal ordering subtleties from test
  synchronization. `[inferred]`
- **Why it matters:** injection points are how concurrency/race tests
  deterministically pause a backend at a code site; flaky wakeups there
  manifest as flaky isolation/TAP tests across the buildfarm. cf.
  [[knowledge/idioms/locking-overview.md]].
