### 1. Disable startup progress timeout during standby WAL replay
- **Author:** Fujii Masao · new thread 2026-06-11 00:13 (replies same day from
  Kyotaro Horiguchi, Fujii Masao). `[from-website]`
- The `log_startup_progress_interval` timer drives periodic "redo in progress"
  progress logging during crash/standby recovery; on a long-lived hot standby
  it keeps firing for the life of the standby, not just an initial catch-up.
  `[inferred]`
- Proposal appears to be to **stop arming the startup-progress timeout once a
  standby has reached its streaming/steady-replay state**, so the timer is a
  startup-only instrument rather than a perpetual one. `[inferred]`
- Discussion (Horiguchi) likely turns on *where* the "no longer starting up"
  boundary is — reaching consistency vs. entering the main redo loop vs.
  switching to streaming. `[inferred]`
- **Why it matters:** touches the startup/redo control flow and the
  timeout-arming discipline in the recovery path; a wrong boundary either
  silences legitimate catch-up logging or leaks a periodic timer. cf.
  [[knowledge/architecture/wal.md]], [[knowledge/architecture/replication.md]].

### 2. Fix race in ReplicationSlotRelease for ephemeral slots
- **Author:** thread active 2026-06-11 with Amit Kapila and Fujii Masao
  reviewing. `[from-website]`
- Ephemeral (`RS_EPHEMERAL`) slots are dropped automatically when their owning
  session releases them; the race is presumably between `ReplicationSlotRelease`
  marking/cleaning the slot and a concurrent observer (checkpointer scanning
  slots, another backend, or slot-array bookkeeping). `[inferred]`
- Likely fix tightens the **order of clearing `in_use` / `active_pid` vs.
  signalling** under `ReplicationSlotControlLock` / the slot's mutex so no
  concurrent scanner sees a half-released ephemeral slot. `[inferred]`
- **Why it matters:** replication-slot release/drop is a recurring
  concurrency-correctness hot spot (compare the 06-10 RfC #6451
  slot-minimum-LSN race); ephemeral slots are the ones created during
  `CREATE SUBSCRIPTION`/base-backup handshakes, so a release race can surface as
  a transient assertion or slot leak. cf. [[knowledge/subsystems/replication.md]],
  [[knowledge/idioms/locking-overview.md]].

### 3. [PATCH] fix AddRelsyncInvalidationMessage
- **Author:** Stolpovskikh Danil · new thread 2026-06-11 11:44. `[from-website]`
- `AddRelsyncInvalidationMessage` queues relation-synchronization-cache
  invalidations (the relsync cache that logical-replication *apply* workers use
  to track per-relation decoding/apply state). `[inferred]`
- The fix likely addresses a **missing or mis-targeted relsync invalidation** —
  an entry not invalidated on the right DDL/relcache event, leaving an apply
  worker acting on stale relation metadata. `[inferred]`
- **Why it matters:** invalidation-message correctness is the classic
  silent-corruption class — a dropped invalidation doesn't error, it just makes
  a cache lie. On the logical-replication apply side that means applying changes
  against a stale tuple-descriptor/column mapping. cf.
  [[knowledge/subsystems/utils-cache.md]], [[knowledge/subsystems/replication.md]].

### 4. Show estimated number of groups for IncrementalSort in EXPLAIN
- **Author:** Ilia Evdokimov · new thread 2026-06-10 15:50. `[from-website]`
- Incremental Sort sorts within *presorted groups*; the planner already
  estimates group count for costing, but EXPLAIN doesn't currently surface that
  estimate the way it shows estimated rows. `[inferred]`
- Proposal adds the **estimated group count to the Incremental Sort EXPLAIN
  output**, giving a way to see why the planner chose incremental over full sort
  and to spot group-count mis-estimates. `[inferred]`
- **Why it matters:** small executor/EXPLAIN instrumentation change, but it
  exposes a costing input that's otherwise invisible — useful for diagnosing
  incremental-sort plan regressions. cf. [[knowledge/architecture/planner.md]],
  [[knowledge/subsystems/executor.md]].

### 5. Discard ORDER BY/DISTINCT when an ANY/IN sublink is pulled up to a join
- **Author:** Ewan Young · new thread 2026-06-10 12:16 (replies from Zsolt
  Parragi, Ewan Young on 06-11). `[from-website]`
- When an `ANY`/`IN` sublink is pulled up into a semijoin, an inner
  `ORDER BY`/`DISTINCT` on the subquery is **semantically irrelevant** — the
  semijoin only cares about existence of a match, not order or duplicate
  removal. `[inferred]`
- The patch drops the redundant sort/distinct during sublink pull-up, saving a
  needless sort node (and its `work_mem`) in the resulting plan. `[inferred]`
- **Why it matters:** a targeted planner simplification in the sublink-pullup
  path (`pull_up_sublinks` / `convert_ANY_sublink_to_join` territory); the
  correctness argument is "the semijoin makes these no-ops", which reviewers
  will want pinned to the exact cases (e.g. does a `LIMIT` inside change it?).
  cf. [[knowledge/architecture/planner.md]], [[knowledge/subsystems/optimizer.md]].
