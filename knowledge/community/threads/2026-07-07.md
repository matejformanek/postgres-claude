## 1. Introduce XID age based replication slot invalidation

- **Author:** Bharath Rupireddy — 2026-07-07 06:34
- **URL:** <https://www.postgresql.org/message-id/CALj2ACWmrz0%2B9YUv6gfxim4f%2BWbuR-K820e8OWY4G%3D4Mt7wr3g%40mail.gmail.com>

Bullets:
- Proposes a new mechanism to invalidate replication slots once the XID they pin
  (via `catalog_xmin` / `xmin`) falls further behind the cluster's current XID
  than a configurable age threshold — complementing the existing
  `max_slot_wal_keep_size` (WAL-size-based) invalidation with an
  **XID-age-based** guard against wraparound danger.
- The core risk a stuck slot creates is not only unbounded WAL retention but
  **held-back `datfrozenxid` / vacuum horizon**: a forgotten logical slot pins
  `catalog_xmin`, blocking catalog vacuum and pushing the cluster toward
  transaction-ID wraparound. A size cap does not bound XID age; this closes that
  gap.
- Discussion turns on the GUC surface (`max_slot_xid_age`?), which invalidation
  cause code to report in `pg_replication_slots.invalidation_reason`, and how the
  age is computed against `ShmemVariableCache->nextXid`.

**Why it matters:** slot invalidation is the load-bearing safety valve that keeps
a mismanaged replica or decoding consumer from wrapping the primary. Pairs with
`[[knowledge/skills/logical-replication]]` (slot.c invalidation) and the vacuum
horizon material in `[[knowledge/skills/vacuum-autovacuum]]`. A natural `[[link]]`
target for a "slot invalidation causes" note.

## 2. Per-table resync for logical replication subscriptions

- **Author:** Cagri Biroglu — 2026-07-07 09:41
- **URL:** <https://www.postgresql.org/message-id/CAA36mso9zgvMVr4UNHMMAQyU-dQy33EqxjZ%3DyEDG8SAB6yYDJw%40mail.gmail.com>

Bullets:
- Adds an `ALTER SUBSCRIPTION ... RESYNC` path (per-table granularity) so a single
  out-of-sync table can be re-copied from the publisher without dropping and
  recreating the whole subscription — today's only recovery from a corrupted or
  diverged table state is `REFRESH PUBLICATION` semantics that don't force a fresh
  initial copy of an already-synced table.
- Mechanically it drives the affected table's `pg_subscription_rel` row back to
  the `INIT`/tablesync state, so the tablesync worker re-runs the initial `COPY`
  for that relation only (reusing the existing tablesync machinery — see
  `[[knowledge/idioms/tablesync-initial-copy.md]]`).
- Open questions: interaction with in-flight apply for the same table, whether the
  resync should be transactional, and locking against concurrent
  `REFRESH PUBLICATION`.

**Why it matters:** targeted resync is a long-standing operational gap; the patch
is a clean extension of the tablesync state machine rather than new infrastructure.
Direct corpus material for `[[knowledge/skills/logical-replication]]` (subscriber
tablesync) and the `copy-family` skill (tablesync's initial copy IS COPY).

## 3. BUG: ReadStream look-ahead exhausts local buffers when effective_io_concurrency>=64

- **Author:** Xuneng Zhou — 2026-07-07 08:27
- **URL:** <https://www.postgresql.org/message-id/CABPTF7Uy%2BysA%2ByB%2Bs_OhKyceOsxdPOycSpEMyHS6u14h7-K5dw%40mail.gmail.com>

Bullets:
- A concrete bug report (now with a reproducer) for the recurring local-buffer
  exhaustion: when a scan over a **temporary table** uses the read-stream API with
  `effective_io_concurrency >= 64`, the stream's look-ahead distance pins more
  local buffers than the fixed local-buffer pool can supply, and the backend
  errors out ("no unpinned local buffers available") rather than throttling.
- Root cause is that the read-stream distance calculation is driven by the global
  `io_combine_limit` / `eic` without accounting for the **much smaller local
  buffer pool** used by temp relations — shared-buffer scans have thousands of
  buffers to lend, local scans have `num_temp_buffers` (default 1024) and every
  pinned look-ahead block is unavailable to the actual read.
- Proposed fix clamps the read-stream's max pin count for local-buffer scans to a
  safe fraction of `num_temp_buffers`.

**Why it matters:** this is the AIO/read-stream subsystem's first user-visible
correctness regression on temp tables — exactly the class
`[[knowledge/skills/aio-readstream]]` exists to reason about (per-buffer pin
accounting, look-ahead distance). Pairs with `[[knowledge/skills/buffer-manager]]`
(local buffers). Fourth-plus night in the ReadStream/local-buffer cluster, now
escalated from proposal to filed bug.

## 4. Propagate stadistinct through GROUP BY / DISTINCT in subqueries and CTEs

- **Author:** Richard Guo — 2026-07-07 05:31
- **URL:** <https://www.postgresql.org/message-id/CAMbWs49G9U%2BQWMDhU9sSLQQyq1i%3DgrQXReXLk5Zg5WTPmqACRw%40mail.gmail.com>

Bullets:
- Teaches the planner to carry a **stadistinct** (n-distinct) estimate up through a
  `GROUP BY` / `DISTINCT` subquery or CTE: today, once a column passes through a
  grouping step, the outer query loses the base-table ndistinct and falls back to
  a crude default, mis-estimating joins and aggregates above the subquery.
- The grouping output's distinctness is knowable — a `GROUP BY col` produces
  exactly `ndistinct(col)` rows — so the patch synthesizes a per-column stat for
  the subquery's target list from the child's `pg_statistic` (via
  `examine_variable` / `get_variable_numdistinct`), letting selectivity estimation
  above the grouping node use a real number.
- Discussion covers where to attach the synthesized stat (RelOptInfo vs.
  PathTarget) and multi-column grouping interactions.

**Why it matters:** cardinality estimation across grouping boundaries is a
long-standing planner blind spot; correct ndistinct propagation improves join
order for a large class of analytic queries. Corpus fit:
`[[knowledge/skills/planner-cost-model]]` (selectivity, `get_variable_numdistinct`)
and `[[knowledge/skills/extended-statistics]]` (ndistinct).

## 5. Add enable_groupagg GUC parameter to control GroupAggregate usage

- **Author:** Richard Guo — 2026-07-07 06:47
- **URL:** <https://www.postgresql.org/message-id/CAMbWs49RCumHErWYOLm4jDcvLSGqhSm1Asnwsv9Tdn2Yjn0r2A%40mail.gmail.com>

Bullets:
- Proposes a planner GUC `enable_groupagg` to disable (cost-penalize) the
  sort-based **GroupAggregate** path, symmetric to the existing `enable_hashagg` —
  giving users and the regression suite a knob to force hash-based aggregation and
  vice versa when diagnosing plan choices.
- Motivation is both diagnostic (isolating why the planner picked a sort+group
  over a hash agg) and test-stability: several regression tests want to pin a
  specific aggregation strategy without contorting the data or `work_mem`.
- The thread debates whether "GroupAgg" is the right user-facing name and whether
  disabling it should be a hard veto or a `disable_cost` bump (the standard
  `enable_*` semantics — a large additive cost, not a true prohibition).

**Why it matters:** the `enable_*` GUC family is the canonical example of
`disable_cost`-style soft vetoes in the cost model; a new member is a clean
teaching case. Fits `[[knowledge/skills/planner-cost-model]]` and
`[[knowledge/skills/gucs-config]]` (PGC_USERSET planner flags).

## 6. Batching in executor

- **Author:** Denis Smirnov — 2026-07-06 09:27
- **URL:** <https://www.postgresql.org/message-id/3D8D1BA5-B8A6-400A-B605-42EEE37890B4%40gmail.com>

Bullets:
- Opens a design discussion on moving PG's executor from strict **tuple-at-a-time**
  (Volcano) iteration toward **batched** (vectorized) execution — passing arrays
  of tuples between `ExecProcNode` calls to amortize per-tuple dispatch overhead
  and open the door to SIMD-friendly inner loops.
- Frames the incremental path: rather than a full engine rewrite, start with
  batch-friendly nodes (scans, aggregates) that can emit/consume a `TupleBatch`
  while falling back to single-tuple mode at the boundary with unmodified nodes —
  the compatibility shim being the hard part.
- Early replies weigh the memory-context and slot-lifetime implications (a batch
  of `TupleTableSlot`s vs. the current single reused slot) and whether the win
  justifies the churn given JIT already attacks per-tuple overhead.

**Why it matters:** batched/vectorized execution is one of the largest latent
architectural questions in the executor; even an exploratory thread maps the
design space. Core `[[knowledge/skills/executor-and-planner]]` material
(`ExecProcNode` dispatch, slot lifetime) and touches the custom-scan/table-AM
boundary where columnar engines already batch.

## 7. hashjoins vs. Bloom filters (yet again)

- **Author:** Tomas Vondra — 2026-07-06 13:34
- **URL:** <https://www.postgresql.org/message-id/d43186bc-dffa-473c-84c0-0b7232ff6ef9%40vondra.me>

Bullets:
- Revives the recurring proposal to build a **Bloom filter** over the hash-join
  build side and push it down to the probe (outer) side, so tuples that provably
  cannot match are discarded before they are hashed or emitted — cutting probe
  work and, when pushed into a scan, I/O.
- The "yet again" framing acknowledges prior attempts stalled on **when the filter
  pays off**: it's a clear win for highly-selective joins with a large outer input
  but pure overhead when most outer tuples match. The new angle is better
  cost-model integration and adaptive disabling when observed selectivity is low.
- Discussion covers filter sizing (false-positive rate vs. build-side ndistinct),
  where to install the probe (join node vs. pushed into the outer scan), and
  parallel-hash interaction (per-worker vs. shared filter).

**Why it matters:** semi-join / anti-join style pruning via Bloom filters is a
standard analytic-DB technique PG has repeatedly circled; the cost-model framing
is the crux. Fits `[[knowledge/skills/executor-and-planner]]` (nodeHashjoin) and
`[[knowledge/skills/planner-cost-model]]`.

## 8. pg_stat_io_histogram

- **Author:** Jakub Wartak — 2026-07-06 11:41
- **URL:** <https://www.postgresql.org/message-id/CAKZiRmxU%2BbnJD7pN1WihX_ooap38B1sy5QiECm6vkH-jQ9LbYg%40mail.gmail.com>

Bullets:
- Proposes a companion to `pg_stat_io` that exposes **latency histograms** for I/O
  operations (bucketed read/write wait times) rather than only the existing
  aggregate counts and total times — so operators can distinguish "steady 1ms
  reads" from "mostly-fast with a tail of 100ms stalls" that the mean hides.
- Built on the cumulative-stats (pgstat) framework: a new fixed-shape stats kind
  accumulating per-bucket counters, flushed via the standard pgstat pending →
  shared path (`[[knowledge/skills/pgstat-framework]]`).
- Debate over bucket boundaries (fixed log-scale vs. configurable), the shared-mem
  cost of N buckets × M io-object rows, and whether AIO's in-flight/completion
  timing (`[[knowledge/skills/aio-readstream]]`) is the right measurement point.

**Why it matters:** histogram-shaped I/O observability is a frequently-requested
gap; the patch is a textbook new-`PgStat_Kind` exercise. Direct corpus fit for
`[[knowledge/skills/pgstat-framework]]` (the cumulative-stats gap flagged by the
user-question-harvester as the #1 skill demand for weeks).

## 10. VACUUM FULL / CREATE INDEX fails: "missing chunk number 0 for toast value"

- **Author:** Matthias van de Meent — 2026-07-06 15:44
- **URL:** <https://www.postgresql.org/message-id/CAEze2WgRUfoR%3DtSQQm8zexVNeBbEYi9ZSPztrK63ObyfPxKpBw%40mail.gmail.com>

Bullets:
- Reports `VACUUM FULL` (and `CREATE INDEX`) intermittently failing with
  **"missing chunk number 0 for toast value NNN"** — the classic symptom of a
  main-table tuple referencing a TOAST pointer whose out-of-line chunks are no
  longer present in the TOAST relation, surfaced when the rewrite/scan detoasts.
- The thread investigates whether the root cause is a concurrent-vacuum /
  snapshot-horizon race that reclaims TOAST chunks still referenced by a tuple the
  rewrite is about to copy, versus pre-existing on-disk corruption — i.e. is this
  a live bug in the rewrite's snapshot handling or a corruption detector firing.
- Reproducer discussion centers on the interaction between the cluster/rewrite
  path's `SnapshotNow`-like visibility and the TOAST fetch's own snapshot.

**Why it matters:** TOAST chunk-visibility during table rewrite is a subtle
correctness surface; a genuine race here would be a data-integrity bug. Fits
`[[knowledge/skills/toast-storage]]` (detoast / chunk fetch) and
`[[knowledge/skills/snapshot-management]]` (rewrite visibility).

## 11. Refactoring postmaster's code to cleanup after child exit

- **Author:** Michael Banck — 2026-07-07 08:39
- **URL:** <https://www.postgresql.org/message-id/20260707083956.GB15075%40p46.dedyn.io>

Bullets:
- Refactors the postmaster's child-reaping path — the `reaper()` /
  `CleanupBackend()` / `HandleChildCrash` logic that runs when a backend or
  auxiliary process exits — consolidating the scattered per-child-type cleanup and
  crash-vs-clean-exit branching into a clearer structure.
- Motivation is maintainability: the postmaster's exit handling has accreted
  special cases (walsender, bgworker, autovac launcher, archiver) each with subtly
  different restart/crash-triggers-reset semantics, making it hard to reason about
  which child deaths force a full crash-recovery cycle.
- Ties into the same author's investigation of a replication deadlock across the
  child-lifecycle boundary (see his Planet post this window).

**Why it matters:** the postmaster's child-lifecycle state machine is the root of
PG's process model and crash-recovery triggering; a cleanup here lowers the risk
surface for the whole cluster. Core `[[knowledge/skills/process-lifecycle]]`
material (`reaper`, `CleanupBackend`, crash-recovery triggering).

## 12. satisfies_hash_partition crash

- **Author:** Robert Haas — 2026-07-06 17:04
- **URL:** <https://www.postgresql.org/message-id/CA%2BTgmoZRkbkGPE8%2B-rY9Nm%2BBbk1rLWg_-NADjvzmNKebVgmZ3Q%40mail.gmail.com>

Bullets:
- Follows up a recently-committed backpatched fix (the `satisfies_hash_partition`
  VARIADIC-NULL crash) with further hardening: the partition-constraint support
  function `satisfies_hash_partition()` — used to validate that rows land in the
  correct hash partition — could still be driven to crash or misbehave on certain
  argument shapes reachable from a hand-written `CHECK` invocation.
- The function is callable directly (it's an ordinary catalog function), so its
  argument validation must be defensive against arbitrary user input, not just the
  well-formed calls the partitioning machinery generates.
- Discussion covers tightening the argument-count / type / NULL checks and whether
  the function should be hidden from direct SQL use.

**Why it matters:** a SQL-reachable crash in a partition-support function is a
robustness (and mild DoS) concern; the hardening pattern — "internal support
function must validate as if hostile" — recurs across the catalog. Fits
`[[knowledge/skills/catalog-conventions]]` and the partitioning executor paths.
