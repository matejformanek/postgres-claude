## 1. Replace spin-wait with futex-mutex in LWLockWaitListLock() on Linux aarch64

New/active thread.

- Proposes swapping the spin-wait loop guarding the LWLock wait-list lock for a
  futex-backed mutex on Linux aarch64, where the existing spin behavior is said
  to scale poorly under contention.
- Touches the core `LWLockWaitListLock` / `LWLockWaitListUnlock` fast path in
  `lwlock.c` — the bit-spinlock embedded in `LWLock->state` that serializes
  wait-queue manipulation.
- Platform-specific: the change is gated to aarch64, so x86 spin behavior is
  untouched; the debate is whether a futex syscall path belongs in this
  hottest-of-hot inner lock at all.
- **Why it matters:** the wait-list lock is taken on every LWLock
  acquire/release that has to touch the queue; a regression or win here moves
  every contended shared-buffer / buffer-mapping / lock-manager partition.
  cf. [[knowledge/idioms/locking-overview.md]],
  [[knowledge/subsystems/storage-lmgr.md]],
  [[knowledge/data-structures/bufferdesc-state.md]]. `[inferred]`

## 2. GetBufferDescriptor() being called for local buffers from MarkBufferDirtyHint()

New/active thread.

- Bug report: `MarkBufferDirtyHint()` reaches for `GetBufferDescriptor()` (the
  *shared*-buffer descriptor accessor) on a buffer that may be a **local**
  (temp-table) buffer, which has its own `GetLocalBufferDescriptor()` array.
- For a local buffer the shared-descriptor index is wrong, so the hint-dirty
  path either touches the wrong descriptor or mishandles the local case — a
  latent correctness bug in the hint-bit / VM-bit dirtying path.
- Fix direction is to branch on `BufferIsLocal()` before selecting the
  descriptor, mirroring how the rest of `bufmgr.c` splits local vs shared.
- **Why it matters:** `MarkBufferDirtyHint` is the no-WAL "it's only a hint"
  dirtying entry point (hint bits, VM bits); getting the local/shared split
  wrong here corrupts dirty-tracking for temp relations.
  cf. [[knowledge/subsystems/storage-buffer.md]],
  [[knowledge/data-structures/bufferdesc-state.md]],
  [[knowledge/files/src/backend/storage/buffer/bufmgr.c.md]]. `[inferred]`

## 3. [Bug Report + Patch] File descriptor leak when io_method=io_uring

Active thread (Ilmar Yunusov, Lætitia Avrot) — carried from the 06-05 window,
now with a patch attached.

- Bug: running with `io_method=io_uring` (the PG18+ AIO backend) leaks file
  descriptors — the io_uring ring/registered-fd lifecycle doesn't release every
  descriptor on the relevant teardown path.
- Patch closes the leak on the io_uring submission/completion path in the AIO
  method layer (`src/backend/storage/aio/`).
- An fd leak in a long-lived backend eventually trips the per-process fd limit
  (the exact failure mode the warda-bibi Planet post this week describes).
- **Why it matters:** io_uring is the newest AIO method and the most likely to
  harbor resource-lifecycle bugs; this is the **fourth flag for the AIO /
  read-stream / io_uring cluster** the corpus has logged in days — reinforces
  the standing `knowledge/subsystems/storage-aio.md` candidacy noted in
  STATE.md. cf. [[knowledge/subsystems/storage-buffer.md]]. `[inferred]`

## 4. Deadlock detector fails to activate on a hot standby replica

Active thread — carried from the 06-05 window (still iterating).

- Bug report: on a hot-standby replica the deadlock detector does not fire as
  expected, so a deadlock involving recovery-side lock waits can hang past
  `deadlock_timeout` without resolution.
- Implicates the standby lock-wait path and whether the deadlock-check timer is
  armed for backends blocked behind the startup/recovery process.
- Discussion this window narrows whether the gap is in timer arming vs the
  detector's view of recovery-held locks.
- **Why it matters:** the deadlock detector is the safety net that guarantees
  lock waits terminate; a standby-side gap means a class of waits never gets the
  periodic deadlock check. cf. [[knowledge/subsystems/storage-lmgr.md]],
  [[knowledge/idioms/locking-overview.md]]. `[inferred]`

## 5. Fix race during concurrent logical decoding activation

New/active thread.

- Bug/fix: a race when logical decoding is *activated concurrently* — two paths
  bringing a logical slot online (or a slot create racing with decoding start)
  can interleave on the slot's state, around the `catalog_xmin` /
  effective-decoding-start handshake.
- Sits in the replication-slot + logical-decoding bootstrap
  (`logical.c` / `slot.c`), the same machinery as the 06-05 "Fix race in
  ReplicationSlotRelease for ephemeral slots" thread — a recurring
  slot-lifecycle-race theme this week.
- **Why it matters:** slot activation sets the `catalog_xmin` floor that holds
  back catalog vacuum; a race that mis-sequences it risks either an
  over-aggressive vacuum (decoding sees missing catalog rows) or a stuck
  xmin floor. cf. [[knowledge/architecture/replication.md]],
  [[knowledge/subsystems/replication.md]]. `[inferred]`
