## 1. Replace pg_atomic_flag with pg_atomic_bool

- **Author:** Heikki Linnakangas — 2026-07-05 22:27 (Ashutosh Bapat, 07-06 15:05)
- **URL:** <https://www.postgresql.org/message-id/bb0ba423-816c-4e21-a40f-b1be13b54c5f%40iki.fi>

Bullets:
- Proposes retiring the `pg_atomic_flag` type — PG's test-and-set primitive whose
  only operations are `pg_atomic_test_set_flag` / `pg_atomic_clear_flag` /
  `pg_atomic_unlocked_test_flag` — in favour of a new `pg_atomic_bool` that
  carries a genuine readable boolean value with the full compare-exchange /
  load / store surface, rather than the flag's write-only "someone has set this"
  semantics.
- Motivation is that `pg_atomic_flag` is a leaky abstraction: on platforms with
  real atomics it *is* a `bool`, but its API deliberately hides the value so it
  can degrade to a spinlock-backed fallback; in practice every fallback platform
  PG still supports has native atomics, so the hidden-value restriction buys
  nothing and forces awkward code where callers actually want to read the state.
- Touches `src/include/port/atomics.h` and the handful of `pg_atomic_flag`
  call-sites; the discussion turns on whether to keep the flag type as a thin
  wrapper for source compatibility or excise it outright.

**Why it matters:** the atomics layer is the foundation of PG's lock-free code
paths, and the `flag` vs `bool` distinction is exactly the kind of subtle
primitive-choice question `[[knowledge/skills/locking]]` (atomics tier) exists to
answer. A clean anchor for a `knowledge/idioms/atomics.md` note on "when a flag,
when a bool, when a spinlock."

## 2. pg_threads.h take II

- **Author:** Thomas Munro — 2026-07-06 12:09 (Peter Eisentraut / Munro follow-ups same day)
- **URL:** <https://www.postgresql.org/message-id/CA%2BhUKG%2BK%2BLmYtz54%2BSv1j5Zu1Qo6szzAMbMyuKSeQmb7De4Ewg%40mail.gmail.com>

Bullets:
- A second attempt at a portable in-tree threading abstraction header
  (`pg_threads.h`), wrapping the C11 `<threads.h>` API (or a pthreads/Win32
  fallback) so PG's growing set of *frontend* thread users — parallel `pg_dump`,
  parallel `pg_restore` — can share one abstraction instead of each tool
  open-coding platform `#ifdef`s.
- Directly enables the parallel effort to remove `TerminateThread()` from
  `pg_dump` (the Windows-only forced-thread-kill that corrupts locks) and the
  companion *"pg_dump: use threads for parallel workers on all platforms"* thread
  (Heikki, 07-05 23:21) — replacing the fork-on-Unix / thread-on-Windows split
  with threads everywhere behind one header.
- Design debate is over how thin the wrapper should be (mirror C11 names exactly
  vs. PG-flavoured names) and whether the backend (which is emphatically
  *not* thread-safe) should be able to include it at all, or whether it stays
  frontend-only.

**Why it matters:** PG is famously a process-per-connection, thread-averse
codebase; a sanctioned threading header for *frontend* tools is a notable
architectural line-drawing exercise. Relevant to anyone reasoning about the
fork-vs-thread model and to `[[knowledge/architecture/process-model.md]]`'s
"why processes, not threads" section — this thread marks precisely where the
project *does* allow threads.

## 3. [PROPOSAL] Natively expose primary key and NOT NULL flags in pgoutput Relation messages

- **Author:** 이민재 (Minjae Lee) — 2026-07-06 12:04
- **URL:** <https://www.postgresql.org/message-id/CAL_7KJnnaMdCqD1mayhH7cv8GPwWmtrJ6e82CTxN9homy41Y6g%40mail.gmail.com>

Bullets:
- Proposes extending the logical-replication `pgoutput` wire protocol's
  *Relation* message so each column advertises whether it participates in the
  primary key and whether it is `NOT NULL`, information a downstream subscriber
  (or a non-PG decoding client) currently cannot recover from the replication
  stream without an out-of-band catalog query.
- Motivation is heterogeneous / non-PostgreSQL consumers of the logical stream
  (CDC pipelines, custom subscribers) that need key/nullability metadata to build
  correct upserts and schema mappings; today they must reach back to the
  publisher's catalog, which defeats the point of a self-describing stream.
- Raises protocol-version-bump questions: the Relation message format is part of
  the versioned logical-replication protocol, so adding fields means a new
  protocol version and careful negotiation so old subscribers keep working.

**Why it matters:** the pgoutput protocol is the contract between logical decoding
and every downstream consumer; each field added is a durable ABI commitment.
Relevant to `[[knowledge/skills/replication-overview]]` and to any
`knowledge/subsystems/logical-decoding.md` — a concrete case of "what belongs in
the self-describing stream vs. what stays a catalog lookup."

## 4. Scrollable cursors at the protocol level

- **Author:** Dave Cramer — 2026-07-06 11:26
- **URL:** <https://www.postgresql.org/message-id/CADK3HHLTzTA983XA739n4vB9Ogi0ROZxpP6w-KymOQ-RF6_cGg%40mail.gmail.com>

Bullets:
- Proposes exposing **scrollable cursor** semantics (fetch backward, absolute /
  relative positioning) directly in the FE/BE extended-query protocol, rather
  than only via SQL-level `DECLARE ... SCROLL CURSOR` + `FETCH`, so client
  drivers (the thread comes from the JDBC world) can scroll a portal without
  round-tripping SQL text.
- Today the extended-query protocol's `Execute` against a portal only fetches
  *forward*; backward/absolute fetch requires SQL `FETCH` statements, which
  drivers must construct and parse, losing the efficiency of the binary portal
  interface.
- Touches the portal/`PortalRun` machinery and the wire-protocol message set —
  a portal already tracks position for `SCROLL` cursors, so the question is
  whether to surface `FETCH`-direction as a protocol-level parameter on the
  existing `Execute`-style message or add a new message type.

**Why it matters:** the FE/BE protocol is a rarely-extended, highly-stable
surface; a proposal to add cursor directionality at that layer is a good study
of portal execution semantics. Anchors `[[knowledge/architecture/query-lifecycle.md]]`'s
portal/Execute section and libpq/driver-facing reasoning.

  (the standby-side "block until this LSN has been replayed" primitive, in active

class" discipline, parallel to the "every long loop needs `CHECK_FOR_INTERRUPTS`"
idiom. Good pin for a `knowledge/idioms/wait-events.md` note and for
`[[knowledge/skills/replication-overview]]`'s standby-replay section.

## 6. ri_Fast* crash w/ nullable UNIQUE constraint (+ PG19 FK fast-path OOB write)

- **Author:** Noah Misch — 2026-07-05 21:05 (FK fast-path OOB thread: Noah Misch 07-05 22:21, Amit Langote 07-06 14:29)
- **URL:** <https://www.postgresql.org/message-id/20260705210533.ee.noahmisch%40microsoft.com>

Bullets:
- Reports a crash in the referential-integrity *fast-path* trigger machinery
  (`ri_Fast*`) when the referenced key is a **nullable** `UNIQUE` constraint: the
  fast-path RI check makes an assumption that holds for a true primary key
  (non-null) but not for a nullable unique key, and dereferences accordingly.
- Sits alongside the companion *"PG19 FK fast path: OOB write and missed FK checks
  during batched"* thread — the PG19 batched-FK-check fast path both writes
  out-of-bounds **and** can silently skip FK checks under batching, a correctness
  hole (missed checks) on top of a memory-safety hole (OOB write).
- Together these are a cluster of bugs in the newly-added RI/FK fast-path
  optimisation: the fast path was added to cut per-row trigger overhead but its
  edge cases (nullable unique, batched insert) were under-tested; fixes constrain
  when the fast path is eligible.

**Why it matters:** a live memory-safety + correctness regression in a *new*
performance optimisation — the canonical "the fast path forgot an edge case"
failure mode. Directly relevant to anyone touching `ri_triggers.c`; a cautionary
data point for `[[knowledge/skills/executor-and-planner]]` on RI trigger
fast-paths and for the R14 "adversarial edge-case suite" discipline in general.

## 7. Proposal: Conflict log history table for Logical Replication

- **Author:** (multi — Amit Kapila, Dilip Kumar, shveta malik, Peter Smith active) — 2026-07-05/06
- **URL:** <https://www.postgresql.org/message-id/CAA4eK1LrUSYKOHX%2BTtQT8kF5QqGh0HTBDtgFAfYxhRc4Qj_Uuw%40mail.gmail.com>

Bullets:
- Proposes a persistent **conflict-history table** for logical replication:
  instead of (or in addition to) logging apply-time conflicts — insert/update/
  delete conflicts, update-missing, delete-missing, unique-key violations — to the
  server log, record them into a queryable catalog/system table so operators can
  audit and reconcile conflicts after the fact.
- The heavily-discussed design points (this was one of the busiest threads of the
  window, 8+ messages) are: where the table lives (system catalog vs. a
  user-space table an extension manages), retention / bloat control, whether
  writing it participates in the apply transaction (and thus can itself fail /
  roll back), and how it interacts with the existing conflict-resolution GUCs.
- Builds directly on the conflict-*detection* and conflict-*resolution*
  infrastructure landed across recent cycles — this is the "now make conflicts
  durably observable" follow-on.

**Why it matters:** conflict handling is the load-bearing operational surface of
active-active / bidirectional logical replication; a durable conflict log is what
turns "conflicts happened somewhere in the log" into "here are the rows."
Strongly relevant to `[[knowledge/skills/replication-overview]]` and a future
`knowledge/subsystems/logical-replication-apply.md`.

## 8. Split fd.c into multiple source files

- **Author:** Heikki Linnakangas — 2026-07-05 17:33
- **URL:** <https://www.postgresql.org/message-id/a5c1dc64-16e9-484a-9659-58793c6ef7ea%40iki.fi>

Bullets:
- Proposes breaking up `src/backend/storage/file/fd.c` — PG's ~4k-line virtual
  file descriptor manager — into multiple source files along its natural seams:
  the VFD pool / `FileRead`/`FileWrite` core, the temporary-file machinery, the
  `OpenTransientFile`/`AllocateFile` transient layer, and the data-directory-sync
  (`SyncDataDirectory`, `fsync`-at-recovery) code.
- `fd.c` has accreted many loosely-related responsibilities behind one very large
  file; the split is pure code-organisation (no behaviour change) aimed at making
  each concern independently readable and reducing the merge-conflict surface on
  a hot file.
- The debate is the usual refactor tension: reviewers weigh the readability win
  against the `git blame` / back-patch friction of moving large blocks between
  files, plus bikeshedding the file boundaries.

**Why it matters:** `fd.c` is one of the corpus's most-cited storage files; a
split changes many `file:line` anchors at once. A pure-refactor case study
(R11-exempt, existing tests cover it) and a heads-up that any
`knowledge/files/src/backend/storage/file/fd.c.md` cites will need re-anchoring
if this lands. Good example of "when a file is too big to reason about."

## 9. hang during shutdown

- **Author:** Jeremy Schneider — 2026-07-05 21:54 (Srinath Reddy, Anthonin Bonnefoy, Jakub Wartak follow-ups 07-06)
- **URL:** <https://www.postgresql.org/message-id/CA%2BfnDAa8Kbbx3KkqpfxutTx41RchJsT3Lykqwcedm0CVr5Tw%3DA%40mail.gmail.com>

Bullets:
- Reports a reproducible **hang during server shutdown**: the postmaster (or a
  backend) fails to make progress through the shutdown sequence, leaving the
  cluster stuck instead of cleanly stopping — the kind of bug that turns a
  `pg_ctl stop` into a `SIGKILL`-and-hope situation.
- Diagnosis in-thread points at the shutdown-signal / process-exit ordering:
  which processes must acknowledge the shutdown signal, in what order, and a
  window where a process is waiting on a latch/condition that the shutdown path
  never satisfies — a classic "who signals whom, and who's still asleep" ordering
  gap in the postmaster state machine.
- Multiple committers picked it up quickly on the first working day, indicating
  it's treated as a real shutdown-correctness regression rather than a
  configuration artifact.

**Why it matters:** postmaster shutdown ordering is a subtle, under-documented
state machine, and a hang there is high-visibility (it breaks orchestration /
failover tooling). Relevant to `[[knowledge/architecture/process-model.md]]`'s
shutdown-sequence section and to anyone reasoning about signal handling and
`WaitLatch(WL_EXIT_ON_PM_DEATH)` teardown.

## 10. Fix HAVING-to-WHERE pushdown with mismatched operator families

- **Author:** Richard Guo — 2026-07-06 07:32
- **URL:** <https://www.postgresql.org/message-id/CAMbWs4_vnF4mZoCK9NUSc8x%2B-aQr4UFaa3xEL2nLT%2Bt_C%3DQWyQ%40mail.gmail.com>

Bullets:
- Fixes a planner bug where a `HAVING` clause is pushed down into `WHERE` (a
  standard optimisation when the clause references no aggregates) even though the
  qual's operator belongs to a **different operator family** than the one the
  transform's correctness assumption relies on — yielding a wrong plan / wrong
  results in the mismatched-opfamily case.
- The HAVING→WHERE transform is only sound when the pushed-down predicate has the
  same semantics evaluated before grouping as after; a mismatched operator family
  (e.g. a cross-type operator whose family differs) breaks that equivalence, so
  the fix gates the pushdown on operator-family compatibility.
- A precise, small correctness fix in `subquery_planner`/`preprocess` HAVING
  handling — the kind of opfamily-equivalence subtlety that recurs whenever the
  planner moves a qual across a grouping boundary.

**Why it matters:** operator-family equivalence is one of the trickiest planner
correctness invariants (it also underlies index matching and equivalence
classes); a HAVING-pushdown opfamily bug is a clean teaching case. Directly
relevant to `[[knowledge/skills/executor-and-planner]]` and any
`knowledge/subsystems/optimizer.md` qual-pushdown section.

## 11. Tepid: selective index updates for heap relations

- **Author:** Greg Burd — 2026-07-06 13:16
- **URL:** <https://www.postgresql.org/message-id/a35235bb-a3b4-45cc-8e4a-7e57556a9250%40app.fastmail.com>

Bullets:
- "Tepid" proposes **selective index updates** — a middle ground between a full
  index update on every heap tuple change and the existing HOT (heap-only tuple)
  optimisation that skips index updates entirely when no indexed column changed.
  The idea is to update only the *subset* of indexes whose columns actually
  changed, rather than all-or-nothing.
- HOT already avoids index churn when *no* indexed column changes, but as soon as
  *any* indexed column changes, every index gets a new entry; "tepid" would let an
  update that touches one indexed column of many avoid inserting into the
  unaffected indexes, cutting write amplification on wide multi-index tables.
- The hard part (discussed in-thread) is visibility and cleanup: index entries
  that point at a tuple whose chain some indexes follow and others don't
  complicate VACUUM and index-only-scan visibility reasoning — the same
  invariants HOT had to get exactly right.

**Why it matters:** write amplification from index maintenance is a first-order
performance ceiling on update-heavy workloads, and this sits right at the
heap-AM / index-AM boundary that HOT defined. Excellent raw material for
`[[knowledge/subsystems/access-heap.md]]`'s HOT-chain section and
`[[knowledge/skills/access-method-apis]]`.

## 12. DROP INVALID INDEXES command

- **Author:** Álvaro Herrera — 2026-07-06 08:54
- **URL:** <https://www.postgresql.org/message-id/aktp2HKdYM2KnSeP%40alvherre.pgsql>

Bullets:
- Proposes a dedicated command to drop **invalid indexes** — the leftover
  `indisvalid = false` index entries that `CREATE INDEX CONCURRENTLY` /
  `REINDEX CONCURRENTLY` leave behind when they fail partway — without the
  operator having to hand-write catalog queries to find them and `DROP INDEX`
  each one.
- Failed concurrent index builds are a well-known operational footgun: they leave
  a half-built, unusable index that still consumes space and, worse, still incurs
  write overhead on every insert while being useless for reads; there is currently
  no ergonomic bulk-cleanup path.
- Design questions in-thread: is it a new top-level `DROP` variant, a function, or
  a `REINDEX`-adjacent option; scoping (per-table vs. per-schema vs. database);
  and whether it should also cover invalid constraints backed by such indexes.

**Why it matters:** a DDL / catalog-maintenance ergonomics feature closing a real
operational gap around `CONCURRENTLY` failure recovery. Relevant to
`[[knowledge/skills/catalog-conventions]]` (the `pg_index.indisvalid` lifecycle)
and to `[[knowledge/skills/access-method-apis]]` index-build failure semantics.
