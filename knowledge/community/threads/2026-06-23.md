## 1. Fix unsafe coding in ResourceOwnerReleaseAll() — Tom Lane

URL: `/message-id/646741.1782157515%40sss.pgh.pa.us`

- New thread: Tom Lane flags unsafe coding in `ResourceOwnerReleaseAll()` —
  the resource-owner teardown loop manipulates the owner's array while
  releasing entries, a pattern that can misbehave if a release callback
  re-enters or reallocates.
- Touches the `resowner.c` release machinery that backs every
  buffer-pin / relcache-ref / lock / tupdesc cleanup at transaction and
  subtransaction abort.

Why it matters: ResourceOwner is the backbone of leak-safe cleanup on the
error path; an unsafe release loop risks use-after-free during abort — the
worst time to have one. [[knowledge/data-structures/resourceowner.md]]

## 2. The PostgreSQL C Dialect — Christophe Pettus

URL: `/message-id/933DA583-B0F4-4FFB-BB0B-64901D4D94AD%40thebuild.com`
(Tom Lane, Bruce Momjian, John Naylor, Bertrand Drouvot, Roberto Mello all
replying — one of the highest-traffic threads of the window.)

- Proposal to write down "the PostgreSQL C dialect" — the de-facto subset of
  C plus house conventions (memory contexts, ereport, the no-`//`-comments
  rule, palloc-not-malloc, the typedef/field naming) that the codebase
  actually uses, as an explicit document for newcomers.
- Discussion ranges over what's prescriptive vs descriptive, C99-subset
  boundaries, and whether to formalise the `pg_attribute_*` / `StaticAssert`
  idioms.

Why it matters: codifying the dialect lowers the contributor on-ramp and
gives reviewers a citable standard — directly parallels this repo's
`knowledge/idioms/` corpus. [[knowledge/idioms/memory-contexts.md]]

## 3. index prefetching — Peter Geoghegan

URL: `/message-id/CAH2-WzkZTkDuyVFszLwPJesF9pS5E8m0UA%2B344bx-B-zfA5kaw%40mail.gmail.com`

- The long-running index-prefetching effort advances: Peter Geoghegan posts
  on the read-stream-driven prefetch of heap pages during index scans, so the
  table-fetch side of an index scan can issue I/O ahead of the consumer.
- Intersects the new `read_stream` AIO abstraction, the nbtree
  `amgettuple` callback, and the index-scan → heap-fetch boundary.

Why it matters: index-scan heap fetches are a classic latency sink; AIO-backed
prefetch is one of the marquee PG20-cycle performance threads.
[[knowledge/subsystems/access-nbtree.md]]

## 6. Deadlock detector fails to activate on a hot standby replica — Vitaly Davydov / Fujii Masao

URL: `/message-id/16cdffd4-f06c-4957-9132-1849a719f3d4%40postgrespro.ru`
(Fujii Masao posts a patch same window.)

- Reports that the deadlock detector does not arm correctly on a hot standby:
  a recovery-conflict / buffer-pin standoff involving the startup process can
  hang without the deadlock timeout firing the detector.
- Fujii Masao proposing a fix to ensure the deadlock-check timer is armed on
  the standby's waiting backends.

Why it matters: a non-firing deadlock detector on a standby turns a
detectable deadlock into an indefinite hang. [[knowledge/idioms/deadlock-detection.md]]

## 7. [WIP] Pipelined Recovery — Imran Zaheer

URL: `/message-id/CA%2BUBfamzeXcdEbmhdOdjWn5X_YVt9n8xUpH5ZwmA7S8VWvaoXw%40mail.gmail.com`

- WIP patch to pipeline WAL recovery: overlap WAL-record decoding / page
  reads with replay so the startup process is not stalled waiting on each
  page fetch serially.

  standby catch-up.

Why it matters: replay is single-process and I/O-bound; pipelining it is one

## 11. File locks for data directory lockfile in the context of Linux namespaces — Dmitry Dolgov

URL: `/message-id/xhzl7ll7fwmlecg4htsgvr3fcwcec4cvugqebphofvcz4srkkn%40vpwa2zgswuwy`

- Revisits how `postmaster.pid` data-directory locking behaves under Linux
