## 1. Subquery pull-up increases jointree search space
- **Author:** Andrei Lepikhov ↔ Tomas Vondra (5 messages, 06-13) · most
  active thread of the window.
- Pulling a subquery up into the parent jointree can multiply the number
  of join-order permutations the planner must cost, so a "simplifying"
  transform can blow up planning time on many-relation queries.
- Discussion centers on when pull-up is a net win vs. when it should be
  gated (e.g. a join-count threshold or a cost guard before flattening).
- Vondra and Lepikhov trading reproducers/measurements — the shape of a
  regression-vs-improvement tradeoff, not yet a settled patch.
- **Why it matters:** join-search explosion is a classic planner
  scalability cliff; any heuristic change here moves planning time for a
  whole class of queries. [[knowledge/subsystems/optimizer.md]] ·
  [[knowledge/architecture/planner.md]]

## 2. Commit Sequence Numbers and Visibility
- **Author:** Jeff Davis (06-13 00:25, continuing thread).
- CSN-based visibility replaces "is this XID in a snapshot's in-progress
  array?" scans with a monotonic commit-sequence-number comparison —
  long-standing proposal to cut snapshot cost on high-core machines.
- The visibility-rule rework touches the heart of MVCC: how a tuple's
  xmin/xmax are tested against a snapshot.
- Davis's note is part of the ongoing design refinement (interaction with
  subtransactions, hot standby, and the existing clog/xact machinery).
- **Why it matters:** one of the highest-leverage MVCC reworks on the
  list; if it lands it changes the cost model of every visibility check.
  [[knowledge/architecture/mvcc.md]] · [[knowledge/subsystems/access-heap.md]]
  · [[knowledge/idioms/snapshot-acquisition.md]]

## 3. Return value of XLogInsertRecord() for XLOG_SWITCH record
- **Author:** ZizhuanLiu X-MAN (06-13 13:33, +06-14 14:37 follow-up).
- `XLOG_SWITCH` forces a WAL segment boundary; the question is what LSN
  `XLogInsertRecord()` should return for it (end-of-record vs.
  start-of-next-segment) and whether callers depend on the distinction.
- Edge case around the padding to the segment boundary and how the
  returned LSN feeds flush/wait logic.
- Likely a correctness/clarity fix rather than a feature.
- **Why it matters:** WAL insert return values feed durability waits and
  replication progress; a subtle off-by-a-segment here is the kind of bug
  that only bites at boundaries. [[knowledge/architecture/wal.md]] ·
  [[knowledge/idioms/wal-record-construction.md]]

## 4. Proposal: Conflict log history table for Logical Replication
- **Author:** Dilip Kumar ↔ Amit Kapila (06-13, two messages).
- Proposal to persist apply-side conflicts (insert_exists,
  update_missing, delete_missing, etc.) into a history table rather than
  only emitting them to the server log.
- Design questions: catalog vs. regular table, retention/pruning, and how
  much detail (old/new tuple, origin) to record per conflict.
- Builds on the existing conflict-detection/resolution work in the apply
  worker.
- **Why it matters:** observability for logical-replication conflicts is a
  recurring operator pain point; a queryable history changes how
  divergence is diagnosed. [[knowledge/architecture/replication.md]] ·
  [[knowledge/idioms/apply-conflict-resolution.md]] ·
  [[knowledge/idioms/apply-worker-loop.md]]

## 5. Handle concurrent drop when doing whole database vacuum
- **Author:** cca5507 (06-14 07:12, new thread).
- A `VACUUM` (database-wide) builds a relation list, then vacuums each;
  if a relation is dropped concurrently between enumeration and
  processing, the worker must tolerate the missing relation rather than
  error.
- Fix is about racing the catalog snapshot of "tables to vacuum" against
  concurrent DROP.
- **Why it matters:** whole-DB vacuum and autovacuum both walk a
  materialized relation list; concurrent-drop races are a known
  sharp-edge class. [[knowledge/subsystems/access-heap.md]] ·
  [[knowledge/idioms/vacuum-skip-pages.md]]

## 6. Bug? pg_rewind produces unusable but starting database with standby recovery
- **Author:** Zsolt Parragi (06-13 19:30).
- Report that `pg_rewind` against a standby-recovered source can yield a
  data dir that *starts* but is subtly unusable — i.e. the failure is
  silent rather than a clean refusal.
- Interaction between the rewind point selection and the standby's
  recovery state / control file.
- **Why it matters:** pg_rewind is the standard re-attach-after-failover
  tool; a "starts but corrupt" outcome is worse than an outright error
  because it defeats the operator's smoke test.
  [[knowledge/architecture/replication.md]] ·
  [[knowledge/idioms/crash-recovery-startup.md]]

## 7. REPACK CONCURRENTLY fails on tables with generated columns
- **Author:** Srinath Reddy Sadipiralla (06-14 03:56).
- `REPACK` (the in-tree table-rewrite/cluster successor work) fails its
  CONCURRENTLY path when the table has generated columns — the rewrite
  path likely mishandles the computed-column tuple descriptor.
- Bug report against an in-progress feature; intersects with the
  virtual/stored generated-column fixes also active this window (#3 in the
  CF-adjacent traffic).
- **Why it matters:** generated columns keep surfacing rewrite-path bugs
  (see also the virtual-column SET EXPRESSION thread below); REPACK is a
  headline maintenance feature whose correctness on every column kind
  gates its credibility. [[knowledge/subsystems/access-heap.md]]

## 9. injection_points: switch wait/wakeup to use atomics rather than latches
- **Author:** Andrey Borodin (06-14 09:23).
- The `injection_points` test framework's wait/wakeup primitive moves from
  latch-based signaling to an atomics-based handshake — reduces test
  flakiness and the cost of the wait point.
- Touches the shared-memory state the injection-point machinery uses to
  park and release a backend at a named code site.
- **Why it matters:** injection points underpin a growing share of PG's
  race-condition regression tests; making the primitive cheaper/sturdier
  improves the whole test suite's determinism.
  [[knowledge/idioms/spinlock-discipline.md]] ·
  [[knowledge/subsystems/storage-ipc.md]]
