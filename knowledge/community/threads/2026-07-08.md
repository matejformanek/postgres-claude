## 1. Restructured Shared Buffer Hash Table

- **Author:** Heikki Linnakangas — 2026-07-07 19:13
- **URL:** <https://www.postgresql.org/message-id/dbbd1998-19ff-4ac2-b4b1-a39f4ec1b0f5%40iki.fi>

Bullets:
- Proposes replacing the current `dynahash`-backed buffer-mapping table (the
  `SharedBufHash` that maps `BufferTag → buffer id`) with a purpose-built hash
  structure, aiming to cut the per-lookup cost and the `BufMappingLock`
  partition contention that dominates high-concurrency read-mostly workloads.
- Motivation is the well-worn hot path: every `ReadBuffer` / `BufTableLookup`
  takes a partition lock and chases dynahash bucket chains; a flatter,
  cache-friendlier layout (open-addressing / inline entries) reduces pointer
  chasing and false sharing on the partition locks.
- Thread discusses how to keep the pin/eviction invariants intact under the new
  layout and whether the win survives once `BufMappingLock` is already
  partitioned 128-way.

**Why it matters:** the buffer-mapping table sits under literally every heap and
index page access; a structural win here is a whole-server throughput lever.
Pairs directly with `[[knowledge/skills/buffer-manager]]` (BufTable / clock-sweep
/ BufMappingLock partitions) and the partition-rank rules in
`[[knowledge/skills/locking]]`.

## 2. index prefetching

- **Author:** Peter Geoghegan — 2026-07-07 23:00
- **URL:** <https://www.postgresql.org/message-id/CAH2-WzmsebafEMoBp-sWq%2BDp75SKV6j%2BRWerKSJNfJkWV592tg%40mail.gmail.com>

Bullets:
- Long-running effort to teach index scans to issue heap prefetches ahead of the
  current TID, layering the `read_stream` API under `nbtree` index-scan heap
  fetches so the executor can overlap I/O the way sequential and bitmap-heap
  scans already do.
- Central design tension is preserving nbtree's kill-tuple / index-only-scan
  semantics and correct visibility ordering while reading TIDs out of order for
  prefetch — the scan must not report a heap tuple before it is provably
  visible-safe to do so.
- Ties into the same AIO machinery the ReadStream local-buffer exhaustion BUG
  (07-07 roundup) stress-tested; getting prefetch depth right without blowing
  local/pinned buffer budgets is the shared hard part.

**Why it matters:** index-scan heap access is the last major scan type without
async prefetch; landing it closes the AIO coverage gap for OLTP point-and-range
lookups. Pairs with `[[knowledge/skills/aio-readstream]]` (read_stream consumers)
and `[[knowledge/skills/access-method-apis]]` (amgettuple / TID semantics).

## 3. TID recycling race during nbtree index-only scans that run on a standby

- **Author:** Peter Geoghegan — 2026-07-07 18:32
- **URL:** <https://www.postgresql.org/message-id/CAH2-Wzm--Yd-wuCURZnunVf_N9awb60pTLrv1La006Yf5zaAhw%40mail.gmail.com>

Bullets:
- Reports a correctness race: on a hot standby, an index-only scan can consult
  the visibility map and return a TID whose heap slot was concurrently recycled
  by replay of a heap-cleanup/`btree` VACUUM record, so the scan can see a tuple
  that no longer logically exists at that TID.
- The primary side is protected by the usual `LockBufferForCleanup` +
  index-vacuum interlock, but on the standby the interlock is between WAL replay
  and read-only backends, which take snapshots differently (recovery conflicts,
  `ResolveRecoveryConflictWithSnapshot`).
- Discussion weighs whether the fix belongs in how the standby applies the VACUUM
  cleanup WAL vs. how index-only scans validate VM bits against the recovery
  snapshot horizon.

**Why it matters:** a visibility race on a standby is a silent wrong-results
class, the worst kind. Pairs with `[[knowledge/skills/snapshot-management]]`
(recovery snapshots / conflict resolution) and
`[[knowledge/skills/physical-replication]]` (hot-standby feedback, recovery
conflicts).

## 4. Bug: mdunlinkfiletag unlinks mainfork seg.0 instead of indicated fork+segment

- **Author:** Heikki Linnakangas — 2026-07-07 17:08
- **URL:** <https://www.postgresql.org/message-id/c5b85d9c-5a95-45a7-83c7-5dca0d22fddf%40iki.fi>

Bullets:
- Reports that `mdunlinkfiletag()` (the `md.c` sync-request handler that unlinks a
  relation-file segment on behalf of the checkpointer) ignores the fork number
  and segment number in the passed `FileTag` and always unlinks main-fork
  segment 0 — a mismatch between the request tag and the file actually removed.
- In practice the path is reached during specific drop/truncate flows routed
  through the pending-ops table (`RememberSyncRequest` → `SYNC_UNLINK_REQUEST`),
  so the wrong-file unlink is a latent data-loss / orphaned-segment hazard rather
  than an everyday crash.
- Fix is to honor `tag->forknum` / segment when constructing the path to unlink;
  thread discusses adding a regression/assertion so a wrong-fork tag is caught
  at request time.

**Why it matters:** silent wrong-file unlink in the storage manager is
corruption-adjacent; even a rarely-hit path warrants a precise fix + guard.
Relates to the `smgr` / `md.c` layer under `[[knowledge/skills/buffer-manager]]`
(the storage side of page eviction) and the checkpointer sync-request machinery.

## 5. libpq: Process buffered SSL read bytes to support records >8kB on async API

- **Author:** Heikki Linnakangas — 2026-07-07 15:58
- **URL:** <https://www.postgresql.org/message-id/be2f229a-25e8-4d19-a307-e4ce3312f212%40iki.fi>

Bullets:
- Fixes an async-API stall: when a single TLS record exceeds libpq's 8 kB read
  buffer, bytes already decrypted and buffered inside OpenSSL are not drained
  before libpq waits on the socket, so `PQconsumeInput`-driven event loops can
  hang waiting for a read-ready that never comes (the data is already in the SSL
  layer, not the kernel socket).
- The correct behavior is to check `SSL_pending()` / continue draining the SSL
  buffer before returning control to the caller's poll loop, so a large record
  fully surfaces without another socket wakeup.
- Matters specifically for non-blocking clients (PgBouncer, pipeline-mode apps)
  under TLS with large rows/results.

**Why it matters:** a protocol-layer liveness bug that only bites async TLS
clients is easy to miss and hard to reproduce. Pairs with
`[[knowledge/skills/wire-protocol]]` (libpq buffering / PQconsumeInput
lifecycle).

## 6. [POC] Implement async DNS with getaddrinfo_a

- **Author:** Jacob Champion — 2026-07-07 21:18
- **URL:** <https://www.postgresql.org/message-id/CAOYmi%2BmBW0h9F5ZKkaUPuK%3D6T080Qs49%2BXCsv6Cee%3Dg_zmitbw%40mail.gmail.com>

Bullets:
- Proof-of-concept to make libpq's hostname resolution non-blocking via glibc's
  `getaddrinfo_a()`, so a connection attempt with `connect_timeout` and multiple
  hosts (`host=a,b,c`) does not block the whole event loop on a slow/failing DNS
  lookup for one host.
- The pain point today: `getaddrinfo()` is synchronous, so async clients and
  multi-host failover strings can stall for the OS resolver timeout even though
  the socket layer is otherwise non-blocking.
- POC caveats: `getaddrinfo_a` is glibc-specific (portability/fallback needed),
  and it spawns helper threads — interaction with libpq's threading assumptions
  is the open design question.

**Why it matters:** connection-establishment latency and failover correctness are
first-order for pooolers and HA client drivers. Pairs with
`[[knowledge/skills/wire-protocol]]` (connection setup) and the multi-host
failover semantics in libpq.

## 7. Implement waiting for wal lsn replay: reloaded

- **Author:** Xuneng Zhou — 2026-07-08 12:08
- **URL:** <https://www.postgresql.org/message-id/CABPTF7XdpM8LCvrkQdXrj3qt88auofiVw_STbUnXg0GEuzbDNw%40mail.gmail.com>

Bullets:
- Revives the `pg_wal_replay_wait()` / `WAIT FOR LSN` line of work: a
  server-side primitive letting a backend on a standby block until replay has
  reached a specified LSN, so read-your-writes-after-failover and
  causal-consistency patterns don't need client-side polling loops.
- "Reloaded" because the prior iteration was reverted; the redesign centers on
  where the wait is safe (not inside a transaction snapshot that would pin the

- Interacts with recovery-conflict handling and the standby's replay-progress
  reporting; the hard part is making the wait interruptible and
  snapshot-neutral.

**Why it matters:** synchronous read consistency across a primary→standby split
is a recurring app-level ask; a correct server primitive removes a whole class of
client hacks. Pairs with `[[knowledge/skills/physical-replication]]` (replay
progress, recovery conflicts) and `[[knowledge/skills/wal-and-xlog]]`.

## 8. pg19b1: TRAP failed Assert("pgstat_bktype_io_stats_valid(...)")

- **Author:** Justin Pryzby — 2026-07-08 14:57
- **URL:** <https://www.postgresql.org/message-id/ak5lccE4qiQpOBHn%40pryzbyj2023>

Bullets:
- Beta-1 assertion failure: a backend trips
  `Assert(pgstat_bktype_io_stats_valid(bktype_shstats, MyBackendType))` in the
  cumulative-stats I/O path, meaning a backend recorded I/O against a
  `(BackendType, IOObject, IOContext, IOOp)` combination the validity matrix
  marks impossible for that backend type.
- These asserts encode "which backend types are allowed to do which I/O"; a trip
  usually means either a new/changed code path issues I/O from a backend type the
  matrix didn't expect, or the matrix in `pgstat_io.c` is stale after a recent
  refactor.
- Being a cassert-only trap on a fresh beta, it's a "tighten the matrix or the
  call site" bug rather than a production hazard, but it flags a real
  accounting-correctness gap.

**Why it matters:** the pgstat I/O validity matrix is the invariant that keeps
`pg_stat_io` honest; a beta trip is exactly the pre-release signal to fix before
GA. Pairs with `[[knowledge/skills/pgstat-framework]]` (IOObject/IOContext/IOOp
accounting, pending-vs-shared flush).

## 10. Do not lock tables in get_tables_to_repack

- **Author:** Álvaro Herrera — 2026-07-07 16:53
- **URL:** <https://www.postgresql.org/message-id/ak0ujOl6WeakTEqx%40alvherre.pgsql>

Bullets:
- Part of the in-flight `REPACK` / `REPACK CONCURRENTLY` work: the enumeration
  step that builds the candidate table list (`get_tables_to_repack`) currently
  takes locks on every candidate while merely deciding what to process, which
  serializes and can block DDL far more than necessary.
- Fix defers locking to the point where a table is actually repacked, so listing
  is lock-light and the concurrent variant lives up to its name.
- Discussion covers the usual enumerate-then-act TOCTOU: a table can be dropped
  between listing and processing, so the later per-table path must tolerate the
  relation vanishing.

**Why it matters:** lock footprint during maintenance-command planning is a
concurrency-availability lever on busy systems. Pairs with
`[[knowledge/skills/locking]]` (heavyweight lock acquisition ordering) and the
VACUUM/maintenance orchestration in `[[knowledge/skills/vacuum-autovacuum]]`.

## 12. Reject ill-formed range bounds histograms in pg_restore_attribute_stats()

- **Author:** Ewan Young — 2026-07-08 04:16
- **URL:** <https://www.postgresql.org/message-id/CAON2xHNGEB%3DTaga0P-osTcsbWTz6jha9yc74bkx9cASZTueA-g%40mail.gmail.com>

Bullets:
- Hardens the statistics-injection API `pg_restore_attribute_stats()` (the
  `pg_dump --statistics` restore path that writes planner stats directly) to
  reject a range-bounds histogram whose bound array is ill-formed rather than
  storing it and risking a later planner crash when the selectivity estimator
  reads it.
- The API is a documented foot-gun: it lets a superuser stuff arbitrary stat
  arrays into `pg_statistic`, and a malformed range histogram can violate
  invariants the range-type selectivity code assumes (ordering, element count).
- Fix validates the shape at restore time and errors cleanly, keeping bad stats
  out of the catalog instead of surfacing as a downstream `elog`/segfault.

**Why it matters:** the dump/restore stats path is new surface (PG 18+) that
feeds untrusted-shape data straight into the planner; validating at the boundary
is the right defense. Pairs with `[[knowledge/skills/extended-statistics]]` and
the single-column stats format in `pg_statistic`.
