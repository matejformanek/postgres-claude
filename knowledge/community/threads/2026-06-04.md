## 1. BM_IO_ERROR flag is lost in TerminateBufferIO due to order of operations in UnlockBufHdrExt

- Author activity: Andres Freund (2026-06-03 03:21), follow-up on a thread
  opened earlier in the window by Melanie Plageman (2026-06-02 16:15).
- Archive: search `pgsql-hackers` for the subject; window
  `https://www.postgresql.org/list/pgsql-hackers/since/202606030300/`.
- Summary `[inferred]`:
  - `BM_IO_ERROR` is the buffer-descriptor flag set when an async/shared-buffer
    I/O fails, so a waiter that completes the I/O hand-off sees the error
    instead of treating the page as valid.
  - The report is an **ordering bug** in the buffer-header unlock path
    (`UnlockBufHdrExt` / `TerminateBufferIO`): the error bit can be dropped as
    the header state word is updated, so the failure is silently swallowed.
  - A lost `BM_IO_ERROR` means a backend can proceed as if a failed read/write
    succeeded — a correctness hazard, not just a missing log line.
- Why it matters: this is the exact class of buffer-state-machine bug the
  corpus's buffer invariants are meant to pin down — the flag transition on the
  `BufferDesc` state word under the header spinlock.
  cf. [[knowledge/subsystems/storage-buffer.md]].

## 2. Race conditions in logical decoding / Fix race during concurrent logical decoding activation

- Author activity: Mihail Nikalayeu (2026-06-03 16:37), continuing Masahiko
  Sawada / Chao Li exchanges from 2026-06-01..02.
- Archive: `https://www.postgresql.org/list/pgsql-hackers/since/202606031600/`.
- Summary `[inferred]`:
  - Concurrent activation of logical decoding (slot creation / first
    `START_REPLICATION`) races against the snapshot-builder reaching a
    consistent point, so two paths can disagree on slot/snapshot state.
  - Adjacent in-window thread "Fix race in ReplicationSlotRelease for ephemeral
    slots" (Fujii Masao, 06-03 12:03) touches the same slot-lifecycle surface.
  - Fixes here are about the ordering of slot-state publication vs. the
    `xl_running_xacts` / `SnapBuild` handshake.
- Why it matters: replication-slot + snapshot-builder ordering is a documented
  hot spot; a missed barrier here corrupts a subscriber's initial data sync.
  cf. [[knowledge/subsystems/replication.md]], [[knowledge/architecture/mvcc.md]].

## 3. hashjoins vs. Bloom filters (yet again)

- Author activity: Tomas Vondra (2026-06-03 10:07), Oleg Bartunov (09:20).
- Archive: `https://www.postgresql.org/list/pgsql-hackers/since/202606030900/`.
- Summary `[inferred]`:
  - Long-running proposal to push a **Bloom filter** built from the hash-join
    inner side down so the outer scan can skip tuples that cannot match before
    they reach the hash probe.
  - The "yet again" signals the recurring tension: real win on low-selectivity
    joins vs. build/probe CPU + memory cost and the planner's ability to *cost*
    the filter (false-positive rate, when to bother building it).
- Why it matters: this is an executor + costing change at the
  `nodeHashjoin` / `cost_hashjoin` boundary — the planner must price a runtime
  filter it cannot fully predict.
  cf. [[knowledge/subsystems/executor.md]], [[knowledge/subsystems/optimizer.md]].

## 4. Optimize CPU usage of dropping buffers during recovery

- Author activity: Jingtang Zhang (2026-06-04 07:19).
- Archive: `https://www.postgresql.org/list/pgsql-hackers/since/202606040700/`.
- Summary `[inferred]`:
  - During redo of relation truncation/drop, the startup process scans shared
    buffers to evict pages belonging to the dropped relation; on large
    `shared_buffers` this linear sweep is CPU-bound.
  - Proposal targets the `DropRelationBuffers` / recovery eviction path to cut
    the per-drop scan cost (e.g. better lookup than a full buffer-pool walk).
- Why it matters: recovery throughput is gated by exactly these O(NBuffers)
  sweeps; the same pattern recurs in `DropRelFileLocatorBuffers`.
  cf. [[knowledge/subsystems/storage-buffer.md]], [[knowledge/architecture/wal.md]].

## 5. generic plans and "initial" pruning

- Author activity: Amit Langote (2026-06-04 00:25).
- Archive: `https://www.postgresql.org/list/pgsql-hackers/since/202606040000/`.
- Summary `[inferred]`:
  - Concerns the interaction between cached **generic plans** (no per-execution
    parameter values) and **initial partition pruning**, which normally uses
    runtime parameter values to prune partitions before execution begins.
  - The hard part: a generic plan must stay reusable across executions while
    still doing parameter-dependent pruning safely, including the locking of
    surviving partitions.
- Why it matters: initial-pruning + plancache + executor-startup locking is a
  known correctness corner (which partitions get locked, and when).
  cf. [[knowledge/subsystems/executor.md]], [[knowledge/architecture/planner.md]].
