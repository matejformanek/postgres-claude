## 1. GetBufferDescriptor() called for local buffers from MarkBufferDirtyHint()

- Subject: **Re: GetBufferDescriptor() being called for local buffers from
  MarkBufferDirtyHint()**
- Participants: Ayush Tiwari, Ashutosh Bapat, Michael Paquier
- Archive: <https://www.postgresql.org/message-id/akcnadjOV4pv7ESL%40paquier.xyz>
  (Paquier, 07-03 03:07) and the Bapat/Tiwari replies at 00:52 / 02:47.
- Summary:
  - `MarkBufferDirtyHint()` is the hint-bit dirtying path (setting a page dirty
    *without* WAL, used by visibility-map hint bits, HOT-prune hint bits, and
    similar cheap best-effort marks). The reported defect: for a **local**
    buffer (a temp-table / temp-relation buffer, negative buffer number) the
    code reaches for `GetBufferDescriptor()` — the *shared* buffer-pool
    descriptor array — instead of `GetLocalBufferDescriptor()`, so it indexes
    the wrong descriptor table.
  - Discussion converges on the guard: `BufferIsLocal(buffer)` must gate the
    descriptor lookup so local buffers take the `LocalBufHdrGetBlock` /
    local-descriptor branch, matching how `MarkBufferDirty()` already splits.
  - Paquier's two back-to-back replies push toward a minimal, backpatchable fix
    plus a regression-style check that a temp-relation hint-dirty doesn't touch
    the shared array.
- Why it matters: this is a shared-vs-local buffer confusion in a hot, WAL-less
  dirtying path — exactly the class of bug that is silent until a temp relation
  happens to alias a valid shared-descriptor index, then corrupts an unrelated
  page's state. Same failure family as the 06-30/07-01 VM/FSM/buffer-cleanup
  cluster. `[[knowledge/subsystems/storage-buffer.md]]`,
  `[[knowledge/data-structures/bufferdesc-state.md]]`,
  `[[knowledge/idioms/hint-bits-setbufferdirty.md]]`.

## 2. ReadStream look-ahead exhausts local buffers when effective_io_concurrency>=64

- Subject: **Re: BUG: ReadStream look-ahead exhausts local buffers when
  effective_io_concurrency>=64**
- Participants: Xuneng Zhou (07-03 01:24, 03:54)
- Archive: <https://www.postgresql.org/message-id/CABPTF7VV0H5DRZBsC3T-SOenVE5MkdwpsWSy9pEJ_sPOuCC5uQ%40mail.gmail.com>
- Summary:
  - When a ReadStream scans a **temp relation** (local-buffer pool) and
    `effective_io_concurrency` is set high (≥64), the stream's look-ahead
    distance tries to pin more local buffers than the fixed-size local pool
    (`num_temp_buffers`) can supply — the prefetch window outruns the buffer
    budget and the backend hits "no unpinned local buffers available".
  - The fix direction: clamp the ReadStream look-ahead distance for local-buffer
    scans to what the local pool can actually pin, rather than the global
    `effective_io_concurrency`-derived distance used for shared-buffer scans.
  - Continuation of the thread first surfaced in the 07-03 community-pulse (#456)
    — still in active back-and-forth on the exact clamp formula.
- Why it matters: ReadStream is the new unified prefetch/read abstraction that an
  increasing number of scan paths route through; a local-buffer-pool footgun that
  only triggers above a GUC threshold is easy to ship and hard to catch in CI
  (default `effective_io_concurrency` is well under 64). Pairs with thread 1 as
  the ongoing local-buffer-correctness front. `[[knowledge/idioms/read-stream-prefetch.md]]`,
  `[[knowledge/subsystems/storage-buffer.md]]`.

## 3. Path Traversal Vulnerability in pg_dump Directory Format

- Subject: **Path Traversal Vulnerability in pg_dump Directory Format** (new)
- Participants: Dilip Kumar (opener, 07-03 06:51), Jonathan Gonzalez V.,
  Imran Zaheer, Dilip Kumar (07-04 13:33)
- Archive: <https://www.postgresql.org/message-id/CAFiTN-v8aH1ZMNuN5xQr_TRFM0Dbcg7A6BU2s5Y5%3DXNPey2BRQ%40mail.gmail.com>
- Summary:
  - The directory-format archive (`pg_dump -Fd` / `pg_restore` reading a
    directory archive) records per-object file names in the TOC. If a crafted or
    tampered directory archive contains a TOC entry whose file name includes
    `../` path components, `pg_restore` can be induced to read/write **outside**
    the archive directory — a classic path-traversal via an untrusted archive.
  - Discussion covers where to validate: reject file names containing path
    separators / `..` at TOC-read time in `pg_backup_directory.c`, so a
    malicious archive is refused before any file open.
  - Threat model debated: a directory archive is normally author-trusted, but
    "restore an archive someone handed me" is a real workflow, so the tightening
    is warranted even if severity is moderate.
- Why it matters: security-relevant hardening of a core client tool, and a clean
  example of the "validate untrusted external file names before opening" idiom.
  No corpus subsystem doc covers `pg_dump`/`pg_restore` internals yet — candidate
  gap. Related to the `pg-docs-miner` pg_dump/restore family.

## 4. Can we get rid of TerminateThread() in pg_dump?

- Subject: **Can we get rid of TerminateThread() in pg_dump?** (new)
- Participants: Thomas Munro (opener, 07-03 03:32; follow-ups 07-04 00:38,
  00:51), Bryan Green, Heikki Linnakangas, Jelte Fennema-Nio
- Archive: <https://www.postgresql.org/message-id/CA%2BhUKGJgO%3Do-vLFahGdR2WesuX3h1-0j%3Da8z72fChc-MG1Hveg%40mail.gmail.com>
- Summary:
  - On Windows, parallel `pg_dump`/`pg_restore` uses threads (not `fork()` as on
    Unix), and the shutdown path calls the Win32 `TerminateThread()` to kill
    worker threads. `TerminateThread()` is documented-dangerous: it does not run
    unwinders, can leave locks held and the CRT heap corrupted.
  - Munro proposes replacing forced termination with cooperative cancellation
    (signal the workers to unwind at a safe point / let them observe a cancel
    flag), aligning the Windows path with the Unix signal-based worker shutdown.
  - Thread digs into whether libpq's Windows socket wait can be interrupted
    cleanly and how the archive-worker cancel loop should poll.
- Why it matters: removes a long-standing "we know this is unsafe" Win32 call
  from a parallel client tool and unifies the cross-platform worker-cancel model.
  Touches the parallel-worker coordination idioms on the client side.
  `[[knowledge/idioms/parallel-worker-coordination.md]]`.

## 5. add validations for required callbacks during pgstat_register_kind()

- Subject: **Re: add validations for required callbacks during
  pgstat_register_kind()**
- Participants: Sami Imseih (07-03 16:04)
- Archive: <https://www.postgresql.org/message-id/CAA5RZ0tr9yLrx2RSXJ%3DbKB48CF2yV4nOGD-j3FEk1U%2B-2M5aTw%40mail.gmail.com>
- Summary:
  - `pgstat_register_kind()` is the entry point a custom cumulative-statistics
    "kind" (fixed or variable-numbered) registers through. The patch adds
    register-time assertions that the callback table is coherent: e.g. a
    variable-numbered kind must supply the flush/serialize callbacks; a
    fixed-numbered kind must supply its shared-memory init — so a mis-declared
    kind fails loudly at registration rather than crashing later inside the
    stats flush.
  - Direct continuation of the multi-night pgstat-framework hardening cluster
    (unregistered-kind crash guard from #456, plus the `pg_stat_kind_info` view).
- Why it matters: the custom-pgstat-kind API is becoming an extension surface,
  and register-time validation is the cheapest place to turn a latent NULL-callback
  crash into a clear error. Fifth consecutive night pgstat-framework is the top
  corpus gap; strongest standing candidate for a `pgstat-and-progress`
  subsystem/skill doc. `[[knowledge/idioms/pgstat-flush-timing.md]]`,
  `[[knowledge/data-structures/pgstat-counter.md]]`.

## 6. [PATCH v1 0/7] Wait event timing and tracing instrumentation

- Subject: **[PATCH v1 0/7] Wait event timing and tracing instrumentation** (new)
- Participants: Dmitry Fomin (opener + follow-ups, 07-03 20:22–21:34)
- Archive: <https://www.postgresql.org/message-id/CAPHG-0mAOn05ae6Kqx1wHXxzOk4E5W7ajjd%3DQBhgkR7a0uyQmw%40mail.gmail.com>
- Summary:
  - A 7-patch series adding *timing* to wait events: today PG reports which wait
    event a backend is *currently* in (`pg_stat_activity.wait_event`), but not
    how long it has spent there cumulatively. The series instruments the
    `WaitEventSet` / `pgstat_report_wait_start`/`_end` boundary to accumulate
    per-event durations and expose a tracing hook.
  - Fomin's own follow-ups refine the sampling-vs-always-on cost tradeoff (timing
    every wait is not free on high-frequency LWLock waits) and the shape of the
    trace output.
- Why it matters: wait-event *duration* is one of the most-requested observability
  gaps — it turns "backend is waiting on X right now" into "backend spent N ms on
  X". The cost-of-timing debate is the crux (same tension that kept
  `track_wait_timing`-style GUCs contentious). Ties the wait-event machinery to
  the latch/WaitEventSet layer. `[[knowledge/data-structures/latch-waiteventset.md]]`,
  `[[knowledge/idioms/pgstat-flush-timing.md]]`.

## 7. Bug in asynchronous Append / Asynchronous MergeAppend

- Subject: **Bug in asynchronous Append** (new) + **Re: Asynchronous MergeAppend**
- Participants: Alexander Korotkov (openers, 07-03 22:00 / 23:05), Etsuro Fujita
  (07-04 09:04)
- Archive: <https://www.postgresql.org/message-id/CAPpHfduMOTnV5Zj2KGJ7zanL_10QvccZHtPUaDfJvBhsh9axnQ%40mail.gmail.com>
- Summary:
  - Async `Append` drives multiple async-capable subplans (typically
    `postgres_fdw` foreign scans) concurrently, interleaving their I/O via the
    executor's async event loop. Korotkov reports a case where the async-request
    bookkeeping (which subplans are still pending vs. ready) desynchronizes,
    producing wrong results or a hang under a specific interleaving.
  - The companion MergeAppend thread extends the concern to the ordered variant,
    where the async completion order must be reconciled with the merge key
    ordering.
  - Fujita (the async-append author) engages on the exact state-machine
    transition at fault.
- Why it matters: async Append is the executor machinery that makes sharded
  `postgres_fdw` fan-out fast; a correctness bug in the pending/ready bookkeeping
  is high-severity for foreign-data-wrapper-heavy deployments. `[[knowledge/subsystems/executor.md]]`,
  `[[knowledge/idioms/fdw-iterate-scan.md]]`.

## 8. Fix race conditions in DropRole() and GrantRole()

- Subject: **Fix races conditions in DropRole() and GrantRole()** (new)
- Participants: Bertrand Drouvot (opener, 07-04 07:47)
- Archive: <https://www.postgresql.org/message-id/aki6fMNLUx6%2BBR8K%40bdtpg>
- Summary:
  - `DropRole()` and `GrantRole()` mutate `pg_authid` / `pg_auth_members` under
    catalog locks, but the report identifies windows where concurrent
    role-DDL sessions can observe an inconsistent membership state — e.g. a role
    dropped between the membership check and the catalog update, or two
    `GRANT`/`REVOKE`s racing on the same membership row.
  - Fix direction: take the right object lock (or `LockSharedObject` on the role
    OID) early enough to serialize the check-then-modify sequence, closing the
    TOCTOU window.
- Why it matters: role/membership races are security-adjacent (a mis-serialized
  GRANT can leave a privilege that a concurrent DROP should have removed).
  Textbook heavyweight-lock-ordering / TOCTOU fix in catalog DDL.
  `[[knowledge/subsystems/storage-lmgr.md]]`, `[[knowledge/idioms/fastpath-locks.md]]`.

## 9. Re-read subscription state after lock in AlterSubscription

- Subject: **Re: Re-read subscription state after lock in AlterSubscription**
- Participants: Hayato Kuroda, Bertrand Drouvot, Dilip Kumar, Zhijie Hou,
  Amit Kapila (07-03 throughout; 07-04 08:00, 08:19)
- Archive: <https://www.postgresql.org/message-id/CAA4eK1%2BOU57xnjQXC-z6qMT%2BbdPsHMzQVXLPMdBmhhq%2B%2Bzwu%2Bg%40mail.gmail.com>
- Summary:
  - `ALTER SUBSCRIPTION` reads the `pg_subscription` row, then takes a lock to
    modify it; between the read and the lock another session (or the apply
    worker) can change subscription state, so the modification proceeds on a
    stale snapshot of the row. The fix re-reads the subscription tuple *after*
    acquiring the lock (lock-then-read, the standard pattern) so the mutation is
    based on the committed state.
  - Long, careful thread — Kapila and Drouvot debate exactly which fields must be
    re-read and whether the apply worker's concurrent state changes need
    additional interlocking.
- Why it matters: canonical "read-under-lock, not before" catalog-concurrency fix
  in the logical-replication control path; a stale-read here can mis-apply a
  subscription option or race the apply worker's own state machine.
  `[[knowledge/idioms/apply-worker-loop.md]]`, `[[knowledge/subsystems/replication.md]]`.

## 11. hashjoins vs. Bloom filters (yet again)

- Subject: **Re: hashjoins vs. Bloom filters (yet again)**
- Participants: Tomas Vondra (07-03 10:10), Matheus Alcantara (07-03 12:50)
- Archive: <https://www.postgresql.org/message-id/ff89ad50-4bad-43d2-9c06-4a433551f35d%40vondra.me>
- Summary:
  - Revives the idea of building a Bloom filter on the hash-join *build* side and
    pushing it down to the *probe* side (and ideally further, to the scan feeding
    the probe) so non-matching outer rows are discarded before the hash lookup /
    before they're even read.
  - Vondra revisits the cost-model question: when does the filter pay for its
    build + probe overhead, and how to estimate selectivity well enough for the
    planner to decide to build one. Alcantara engages on the executor plumbing.
- Why it matters: a long-standing "obvious win that's hard to cost correctly"
  planner/executor optimization; the filter helps most exactly when the join is
  selective, which is also when mis-estimation hurts most. `[[knowledge/idioms/parallel-hash-join.md]]`,
  `[[knowledge/idioms/cost-join-paths.md]]`, `[[knowledge/subsystems/optimizer.md]]`.

## 12. Batching in executor

- Subject: **Re: Batching in executor**
- Participants: Amit Langote (07-03 05:19, 06:05)
- Archive: <https://www.postgresql.org/message-id/CA%2BHiwqHv1CggSbpn%3Dn%2BGU%3DCp1AqDD25%3Dhzx_PTYyqjMyZO2%2BmA%40mail.gmail.com>
- Summary:
  - Explores moving the executor from strictly tuple-at-a-time (`ExecProcNode`
    returns one slot per call) toward passing *batches* of tuples between nodes,
    to amortize per-tuple dispatch overhead and open the door to vectorized
    expression evaluation.
  - Langote's messages work through where batching interacts with the existing
    `TupleTableSlot` contract, EPQ recheck, and nodes that can't naturally batch
    (e.g. LIMIT, correlated subplans).
- Why it matters: batched execution is the structural prerequisite for
  vectorization in PG's executor; the hard part is retrofitting it under the
  established slot/`ExecProcNode` protocol without breaking the volcano-model
  nodes. `[[knowledge/subsystems/executor.md]]`,
  `[[knowledge/idioms/expression-evaluator-flow.md]]`,
  `[[knowledge/data-structures/tupletableslot.md]]`.
