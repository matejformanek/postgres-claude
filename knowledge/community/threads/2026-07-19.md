## 2. Checkpointer write combining
- **Archive:** https://www.postgresql.org/message-id/D0E7C430-76A9-4030-B8AB-109E7D7ED8C1%40yandex-team.ru
- Andrey Borodin revives the idea of coalescing adjacent dirty-buffer writes in
  the checkpointer into larger sequential I/Os instead of per-8kB `write()`s.
- Sits directly on the AIO / read-stream-era I/O-batching trend: the checkpointer
  is one of the last major writers not yet write-combining `[inferred]`.
- **Why it matters:** checkpoint write bandwidth is a known throughput ceiling on
  write-heavy workloads; combining writes reduces syscall and WAL-flush pressure
  at checkpoint. `[[knowledge/idioms/checkpoint-coordination]]`

## 3. walsummarizer can get stuck when switching timelines
- **Archive:** https://www.postgresql.org/message-id/A44B14F1-DE37-4E45-BEB4-FAC362A1E2EC%40yandex-team.ru
- Andrey Borodin continues a bug thread on the WAL summarizer (the PG17
  incremental-backup engine) wedging when a timeline switch occurs mid-summary.
- The failure mode is the summarizer looping/blocking rather than advancing past
  the timeline boundary — a liveness bug, not corruption `[inferred]`.
- **Why it matters:** WAL summarization underpins incremental `pg_basebackup`; a
  stuck summarizer silently stalls the whole incremental-backup chain. Flagged in
  prior pulses (07-14 cluster). `[[knowledge/subsystems/replication]]`

## 4. pg_xmin_horizon: a system view of everything pinning the xmin horizon
- **Archive:** https://www.postgresql.org/message-id/_9LLi6qXTdunP1OYGi4li3De6B96wFahhKvCsUSWH_iiap4W-vC40Vcp3ib6HPnzW8QAW58Es8RfOcZEttTbqfUcEJ_lj_4GBoss-a5bHzk%3D%40scottray.io
- Scott Ray proposes a new system view enumerating every source holding back the

  cannot fix; a canonical view is a real DBA-diagnostics win.
  `[[knowledge/idioms/xmin-horizon-management]]`

## 5. Bug in asynchronous Append
- **Archive:** https://www.postgresql.org/message-id/CAPmGK161UGXxjOs4zPqUdqT%2BjpupDZ4oU7p4XfZEunMz5%3DUuWg%40mail.gmail.com

- **Why it matters:** async Append is the mechanism that lets partitioned foreign

## 6. sequencesync worker race with REFRESH SEQUENCES
- **Archive:** https://www.postgresql.org/message-id/CALDaNm2RP5QAF0GTcyyEsCx8SZHrRtEB4ET-bp6JfFJUqOG-jw%40mail.gmail.com
- Amit Kapila and vignesh C dig into a race between the logical-replication
  sequence-sync worker and a concurrent `ALTER SUBSCRIPTION ... REFRESH
  PUBLICATION SEQUENCES`.
- Sequence replication is a newer logical-rep surface; the sync worker and the
  refresh path can collide over which sequences are in-scope `[inferred]`.
- **Why it matters:** sequence sync is part of making logical replication a
  complete failover story; races here corrupt sequence state on the subscriber.
  `[[knowledge/idioms/tablesync-initial-copy]]`

## 7. ri_Fast* crash w/ nullable UNIQUE constraint
- **Archive:** https://www.postgresql.org/message-id/20260718032532.f2.noahmisch%40microsoft.com
- Noah Misch reports a backend crash in the referential-integrity fast-path
  checks when a UNIQUE constraint permits NULLs on the referenced side.
- The RI fast-path assumes a total unique key; a nullable column violates that
  assumption and trips a bad code path `[inferred]`.
- **Why it matters:** RI triggers run on every FK-touching DML; a crash here is a
  correctness-and-availability bug in a very hot path.
  `[[knowledge/idioms/ri-fkey-check]]`

## 8. amcheck: allow interrupting the child-level rightlink walk
- **Archive:** https://www.postgresql.org/message-id/CAH_PY8t%2BJ%3DVP%3D%3DSrMVVhSiu-AfsO19C6pFQ7kxaar%2B-D7G%2BAqw%40mail.gmail.com
- 김선동 (Sun-dong Kim) proposes adding `CHECK_FOR_INTERRUPTS()` into amcheck's
  child-level rightlink traversal so a long verification of a large index can be
  cancelled.
- Today the rightlink walk can run uninterruptibly for a long time on a big
  btree, making `bt_index_check` effectively un-cancellable.
- **Why it matters:** amcheck is the primary btree-corruption detector; making it
  responsive to cancel is a real operability fix for large indexes.
  `[[knowledge/subsystems/contrib-amcheck]]`

## 9. Avoid unnecessary StringInfo allocation in tablesync COPY buffer
- **Archive:** https://www.postgresql.org/message-id/1A4D7102-522B-4865-B34D-149AA64F7C1B%40gmail.com
- Denis Smirnov targets a redundant `StringInfo` allocation on the tablesync
  initial-COPY hot path in logical replication.
- Micro-optimization, but on the per-row COPY buffer path it removes allocator
  churn during the initial data sync of a subscription `[inferred]`.
- **Why it matters:** tablesync initial COPY dominates the cost of adding a table
  to a subscription; allocation savings compound over millions of rows.
  `[[knowledge/idioms/tablesync-initial-copy]]`

## 10. [PATCH] Don't call ereport(ERROR) from recovery target GUC assign hooks
- **Archive:** https://www.postgresql.org/message-id/CACSdjfMge3o-6FbSFTL-3ghdzY348ddQJxshv92F%3DPhapvQQng%40mail.gmail.com
- JoongHyuk Shin fixes assign-hooks for `recovery_target_*` GUCs that call
  `ereport(ERROR)` — forbidden from a GUC assign hook, which must not longjmp.
- Assign hooks run at a point where throwing can leave GUC state half-applied;
  validation belongs in the *check* hook, not the assign hook.
- **Why it matters:** a textbook GUC-hook-discipline bug on the recovery-config
  path; wrong-phase erroring can wedge startup.
  `[[knowledge/idioms/guc-variables]]` · `[[knowledge/idioms/error-handling]]`

## 11. Restore vacuum_delay_point() in GIN posting-tree leaf vacuum
- **Archive:** https://www.postgresql.org/message-id/178447127453.110.12276981925360691905%40mail.gmail.com
- Paul Kim restores a `vacuum_delay_point()` call that was dropped from GIN
  posting-tree leaf vacuuming, so cost-based vacuum delay applies there again.
- Without it, GIN posting-tree cleanup ignores `vacuum_cost_delay` and can hog
  I/O during autovacuum of GIN-indexed tables.
- **Why it matters:** cost-delay honoring is what keeps autovacuum from
  saturating I/O; a gap in GIN vacuum breaks that throttle for a whole index
  type. `[[knowledge/idioms/gin-tree-structure]]`

## 12. Raise default track_activity_query_size to 16kB (from 1kB)
- **Archive:** https://www.postgresql.org/message-id/CAA5RZ0u8mQY1tBcMwV6iMPQy%2BBgZ-Oe2zOLuGW%2BQWO1SDBOang%40mail.gmail.com
- Sami Imseih argues the 1kB default (unchanged since 8.x) truncates modern
  queries in `pg_stat_activity.query`, and proposes 16kB.
- Trade-off is per-backend shared-memory footprint (`track_activity_query_size ×
  MaxBackends`) vs. losing the tail of large queries in monitoring.
- **Why it matters:** query truncation in `pg_stat_activity` silently defeats
  monitoring/observability tooling on ORM- and generated-SQL-heavy workloads.
  `[[knowledge/idioms/pgstat-flush-timing]]`
