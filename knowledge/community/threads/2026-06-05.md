## 1. Reduce LEFT/FULL JOIN to ANTI JOIN in more cases — Richard Guo

New thread (2026-06-05 02:55), active same-day (Isaac Morland, wenhui qiu,
Richard Guo follow-ups).

- Proposes extending the planner's join-strength reduction so more LEFT/FULL
  joins collapse to ANTI JOIN when the nullable side is provably unreferenced
  and the join qual + a NOT-matched condition allow it.
- Touches `reduce_outer_joins` / join-removal logic in the optimizer prep
  phase; ANTI JOIN lets the executor stop at the first match, a real win on
  anti-semi shaped queries.
- Same-day pushback/refinement on the exact provability conditions (when a
  FULL join is safe to weaken).
- **Why it matters:** outer-join strength reduction is one of the cheapest
  planner wins — it shrinks the join search space before path generation.
  cf. [[knowledge/subsystems/optimizer.md]],
  [[knowledge/architecture/planner.md]]. `[inferred]`

## 2. Archive-fed logical decoding: pausing recovery on slot conflict — Andrey Borodin

New thread (2026-06-05 07:06).

- Addresses logical decoding driven from archived WAL (not a live primary):
  when recovery would remove catalog rows a logical slot still needs, the slot
  conflicts and is normally invalidated.
- Proposal is to *pause* recovery on such a slot conflict instead of
  invalidating the slot — giving the logical consumer a chance to catch up
  before the conflicting catalog_xmin is overrun.
- Interacts with `InvalidateObsoleteReplicationSlots` / hot-standby feedback
  and the recovery-conflict machinery.
- **Why it matters:** logical decoding from archive (vs. streaming) is the
  basis of several failover/cascading designs; slot-conflict handling decides
  whether a consumer survives a catalog-vacuum on the source.
  cf. [[knowledge/architecture/replication.md]]. `[inferred]`

## 3. Deadlock detector fails to activate on a hot standby replica — (Fujii Masao reply)

Active thread (reply 2026-06-05 11:58).

- Bug report: on a hot-standby replica, the deadlock detector does not fire as
  expected, so a deadlock involving recovery-side lock waits can hang past
  `deadlock_timeout` without resolution.
- Points at the standby's lock-wait path and how the deadlock-check timer is
  (not) armed for backends blocked behind the startup/recovery process.
- **Why it matters:** the deadlock detector is the safety net that guarantees
  lock waits terminate; a standby-side gap means a class of waits never gets
  the periodic deadlock check. cf. [[knowledge/subsystems/storage-lmgr.md]],
  [[knowledge/idioms/locking-overview.md]]. `[inferred]`

## 4. Optimize CPU usage of dropping buffers during recovery — Jingtang Zhang

Active thread (reply 2026-06-04 07:19).

- During recovery, dropping a relation's buffers (`DropRelationBuffers` /
  `DropRelationsAllBuffers`) scans the whole buffer pool; under large
  `shared_buffers` this is a hot CPU path when many relations are dropped/
  truncated in the WAL stream.
- Discussion is about reducing that O(NBuffers) scan cost on the recovery side
  (batching, or a smarter lookup) without regressing the steady-state path.
- **Why it matters:** recovery throughput is gated by exactly these full-pool
  scans; the same `DropRelation*Buffers` cost shows up in standby replay lag.
  cf. [[knowledge/subsystems/storage-buffer.md]]. `[inferred]`

## 5. Adding basic NUMA awareness — Tomas Vondra

Active thread (reply 2026-06-05 12:52).

- Long-running effort to make PostgreSQL NUMA-aware: interleaving / pinning
  shared-memory regions (buffer pool, PGPROC, etc.) across NUMA nodes so a
  backend on one socket isn't paying cross-node latency for every buffer touch.
- This window's traffic is on the shape of the shared-memory placement API and
  which structures benefit most.
- **Why it matters:** on large multi-socket boxes, uncontrolled NUMA placement
  silently caps shared_buffers scalability; this is foundational
  shared-memory-layout work. cf. [[knowledge/subsystems/storage-buffer.md]],
  [[knowledge/idioms/locking-overview.md]]. `[inferred]`
