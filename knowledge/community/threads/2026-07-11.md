## 1. Reduce cleanup lock contention on standby replay `[NEW]`

- **Author:** SATYANARAYANA NARLAPURAM (reply from Peter Geoghegan)
- **URL:** https://www.postgresql.org/message-id/CAHg%2BQDf3NcB3vOAqQ5EFcV5DYYUZ9snCSJAU-x1JDbtcjK3eBQ%40mail.gmail.com
- Three-patch series attacking the recovery-conflict-on-buffer-pin stall: WAL
  replay of `XLOG_HEAP2_PRUNE` needs a **cleanup lock** (exclusive + zero pins)
  on the target buffer, so any standby backend holding a pin — even a trivial
  PK SELECT — blocks replay until `max_standby_streaming_delay` fires, forcing
  query cancels and replication lag. `[from-thread]`
- **Patch 1** shrinks buffer-pin lifetime for unique-index point lookups: two
  new `ScanOption` flags (`SO_MATERIALIZE_ON_FETCH` for IndexScan,
  `SO_RELEASE_PIN_ON_FETCH` for IndexOnlyScan) drop the `xs_cbuf` heap pin
  right after the fetch, exploiting that a unique index visits ≤1 heap page so
  there is no same-page pin-reuse to preserve. **Patch 2** skips the cleanup
  lock for freeze-only prune records by threading `do_prune` (not an
  unconditional `true`) into `log_heap_prune_and_freeze()`, so freeze-only
  records omit `XLHP_CLEANUP_LOCK` and replay takes only a content lock.
  **Patch 3** adds the buffer tag (rel/blk) to the recovery-conflict log line
  so DBAs can see which page is hot. `[from-thread]`
- **Why it matters:** the pin-held-during-executor pattern is the single most
  common cause of standby query cancellation under a read-heavy replica; Patch
  2 in particular is a clean WAL-format-level win that removes conflicts for
  the freeze-only case entirely. Touches `[[executor-and-planner]]`,
  `[[buffer-manager]]`, `[[wal-and-xlog]]`, `[[physical-replication]]`,
  `[[vacuum-autovacuum]]`.

## 2. Missing FSM Update when Updating VM On-access

- **Author:** Melanie Plageman / Andres Freund (active back-and-forth)
- **URL:** https://www.postgresql.org/message-id/gmuwqrfyjdzmm7fu2uugvgmt32fz5itclkv7vftmbottkaj3i6%4025u2clp2wwod
- The bug: when `heap_page_prune_opt()` newly marks a page all-visible during
  on-access pruning, the FSM entry for that page is left stale — and because a
  later VACUUM *skips* all-visible pages, the FSM entry may never be corrected,
  so the map over-reports free space on a page nobody will re-measure.
  `[from-thread]`
- Melanie's fix records real free space at the transition: when
  `presult.newly_all_visible`, set `record_free_space = true` and
  `freespace = PageGetHeapFreeSpace(page)` so `RecordPageWithFreeSpace()` runs
  immediately. Andres pushes back on the deleted "avoid reuse by unrelated
  UPDATEs/INSERTs" comment — argues the related/unrelated distinction is
  meaningless since the backend can't know future tuple provenance, and favors
  the simpler unconditional update given the release-cycle constraints.
  `[from-thread]`
- **Why it matters:** a stale FSM over-estimate sends `INSERT`/`COPY` to a page
  with no room, wasting a lock+read+retry per hit; the coherence gap between
  on-access pruning and VACUUM's FSM maintenance is exactly the kind of
  invisible drift that only shows as bloat/latency. Andres also floats caching
  a pinned FSM page in the scan descriptor (mirroring the VM-page cache) to
  kill repeated pin/unpin churn. Touches `[[free-space-map]]`,
  `[[vacuum-autovacuum]]`, `[[buffer-manager]]`.

## 3. BM_IO_ERROR flag is lost in TerminateBufferIO (order of operations in UnlockBufHdrExt) `[COMMITTED]`

- **Author:** Andres Freund (fix pushed)
- **URL:** https://www.postgresql.org/message-id/hm4wqcu3zvl76go6zvyvr77tzgbiehs5v4lvaa53uwlj5wqar7%40nfohqqm3r3i3
- The `BM_IO_ERROR` flag could be lost inside `TerminateBufferIO()` because
  `UnlockBufHdrExt()` added flag bits and then subtracted bits in an order that
  let `BM_IO_ERROR` be clobbered between the two operations. `[from-thread]`
- The fix reverses the order — **subtract bits first, then add** — so the two
  masks can't overlap and clobber, and callers don't have to hand-roll bit math
  to protect a flag. Andres pushed it and marked the open item resolved; full
  exercise deferred to PG20 pending the AIO write-path conversion that will lean
  on this flag. `[from-thread]`
- **Why it matters:** a lost `BM_IO_ERROR` means a buffer whose I/O failed can
  be treated as clean/valid — a silent-corruption-shaped hazard — and this is
  load-bearing for the upcoming AIO write path. Touches `[[buffer-manager]]`,
  `[[aio-readstream]]`, `[[locking]]`.

## 4. sequencesync worker race with REFRESH SEQUENCES `[NEW]`

- **Author:** Noah Misch (replies from vignesh C, Hayato Kuroda, shveta malik, Amit Kapila)
- **URL:** https://www.postgresql.org/message-id/20260710045217.f0.noahmisch%40microsoft.com
- Reproducible race in logical replication of sequences: the sequencesync
  worker can mark a subscribed sequence **READY** even when the subscriber's
  value is older than what the last `REFRESH SEQUENCES` synced — violating the
  consistency guarantee that READY implies caught-up. Ships a repro patch
  (`test-refresh-race-sequencesync-v0.patch`) confirming it as a genuine defect.
  `[from-thread]`
- Comes with a companion report (`sequence-logrep-defects.md`) enumerating **17
  findings** in the sequence-logrep path; Misch prioritizes finding-2
  (`default_transaction_read_only=on` behavior), finding-7 (an Assert failure),
  and finding-17 (undocumented PG19+ publisher requirement) as the ones worth
  fixing now. `[from-thread]`
- **Why it matters:** sequence logical replication is a young PG19 feature and
  this is the first systematic defect sweep against it; the READY-with-stale-data
  race is a correctness bug that a failover would surface as duplicate/rewound
  sequence values. Very active thread (7+ replies in-window). Touches
  `[[logical-replication]]`.

## 6. Batched clock sweep to reduce cross-socket atomic contention

- **Author:** Greg Burd
- **URL:** https://www.postgresql.org/message-id/5bcaa70b-89c7-44b6-b197-638d10997cae%40app.fastmail.com
- Batches the clock-sweep victim search so the shared `nextVictimBuffer` atomic
  is advanced in strides rather than one-at-a-time, cutting cross-socket
  cache-line ping-pong on the `StrategyControl` counter. `[inferred]`
- Recurring thread; the clock sweep's single hot atomic is a known NUMA
  bottleneck at high buffer-eviction rates. `[inferred]`
- **Why it matters:** on many-socket boxes the clock-sweep atomic can dominate
  eviction cost; batching is a low-risk way to amortize it. Touches
  `[[buffer-manager]]`, `[[locking]]`.
