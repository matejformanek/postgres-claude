## 1. Hash index bucket split bug

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Peter Geoghegan
  20:39, Amit Kapila 10:45)
- Summary:
  - Report of a correctness bug in hash-index bucket splits: under concurrency a
    split can leave the index in a state where a scan misses or double-returns
    tuples, tied to how the split moves tuples between the old and new bucket
    while a scan holds a bucket lock.
  - Amit Kapila (the hash-index AM maintainer) engages on the split algorithm's
    lock/pin discipline — the split acquires a cleanup lock on the old bucket and
    the interaction with in-flight scans is where the invariant breaks.
  - Peter Geoghegan frames it against nbtree's more battle-tested page-split
    concurrency model as the reference for "how split-vs-scan should behave."
- **Why it matters:** hash indexes are WAL-logged and crash-safe since PG 10 but
  see far less production use than nbtree, so concurrency bugs surface slowly. A
  split-vs-scan race is exactly the class of AM bug that silently returns wrong
  answers. See `[[knowledge/skills/access-method-apis]]` (index AM callbacks) and
  `[[knowledge/skills/locking]]` (buffer cleanup-lock discipline).

## 2. PG19 FK fast path: OOB write and missed FK checks during batched apply

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Amit Langote 11:37)
- Summary:
  - Amit Langote reports an out-of-bounds write plus missed foreign-key checks in
    the PG19 "FK fast path" batching optimization — the code that batches RI
    trigger checks to avoid one probe per row.
  - The OOB write is a memory-safety defect (writing past an array sized for the
    non-batched case); the missed-check path means some FK violations would go
    undetected, a data-integrity hole.
  - This lands against a feature that is new in the PG19 cycle, so it is a
    must-fix before the branch stabilizes rather than a back-patch.
- **Why it matters:** a memory-safety bug *and* a silent integrity gap in the same
  new code path — the worst pairing. Referential integrity that sometimes doesn't
  fire is a data-corruption vector. Relevant to `[[knowledge/skills/executor-and-planner]]`
  (trigger/RI execution) and the RI trigger machinery in `ri_triggers.c`.

## 3. Add per-backend AIO statistics

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Bertrand Drouvot,
  Michael Paquier, Andres Freund — active thread)
- Summary:
  - Proposal to expose asynchronous-I/O counters per backend in the cumulative
    statistics system, so the new AIO subsystem (sync/worker/io_uring methods) is
    observable the way per-backend I/O timing already is.
  - Andres Freund (AIO author) and Bertrand Drouvot debate which counters are
    stable enough to expose and how to fit them into the fixed-size per-backend
    pgstat entry without bloating shared memory.
  - Michael Paquier weighs in on the pgstat-framework mechanics (kind
    registration, flush timing) for the new counter set.
- **Why it matters:** AIO shipped as plumbing in PG 17/18 but was largely opaque
  to operators; per-backend counters are what make io_method tuning empirical
  rather than guesswork. See `[[knowledge/skills/aio-readstream]]` and
  `[[knowledge/skills/pgstat-framework]]`.

## 4. Fix data checksum processing for temp relations and dropped databases

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Fujii Masao 02:24)
- Summary:
  - Fujii Masao fixes checksum handling that mishandles temporary relations and
    databases dropped mid-operation — the checksum path either processes pages it
    shouldn't (temp rels use local buffers, not shared) or trips over a database
    directory that vanished underneath it.
  - Timely against the PG19 "checksums on by default" work: more code now runs the
    checksum path, so its edge cases matter more than when checksums were opt-in.
- **Why it matters:** PG19 makes data checksums the default (see the Shaun Thomas
  "Checksums For All" blog in the 07-05..07 roundups), which promotes every latent
  checksum-path bug from rare to universal. `[[knowledge/skills/buffer-manager]]`
  (local vs shared buffers) is the relevant boundary.

## 5. Proposal: Conflict log history table for logical replication

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Dilip Kumar 09:15,
  Kedar Anavardekar 11:41, Robert Haas 14:09)
- Summary:
  - Proposal to persist logical-replication apply conflicts into a history table
    so conflicts (insert_exists, update_missing, delete_missing, etc., tracked
    since PG18) are queryable after the fact rather than only visible in the log.
  - Robert Haas pushes on the design questions: retention/pruning of the table,
    whether it should be a real catalog or a regular table, and how writing to it
    interacts with the apply worker's own transaction.
  - Kedar and Dilip iterate on the schema and what columns a conflict row needs to
    be actionable (LSN, origin, relation, conflict type, remote/local tuple).
- **Why it matters:** conflict *detection* landed in PG18; conflict *observability
  and resolution* is the PG19+ frontier. A durable conflict table is a building
  block for automated resolution. See `[[knowledge/skills/logical-replication]]`.

## 6. Support EXCEPT for TABLES IN SCHEMA / ALL SEQUENCES publications

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Peter Smith,
  Amit Kapila, shveta malik, Nisha Moond, Shlok Kyal — two linked threads)
- Summary:
  - Two related publication-syntax extensions: `EXCEPT` for `TABLES IN SCHEMA`
    publications (publish a whole schema minus named tables) and `EXCEPT` for the
    new `ALL SEQUENCES` publication form (sequence replication is PG19 work).
  - The reviewers converge on grammar shape and on the catalog representation for
    an exclusion list, reusing the pattern established by column-list and
    row-filter publications.
- **Why it matters:** "everything except X" is the ergonomic gap in publication
  DDL — today you enumerate inclusions. Combined with sequence replication, this
  rounds out the publication surface for PG19. `[[knowledge/skills/logical-replication]]`.

## 7. Preserve replication origin OIDs in pg_upgrade

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (shveta malik,
  Hayato Kuroda, Amit Kapila, Ajin Cherian)
- Summary:
  - pg_upgrade currently does not carry replication-origin OIDs across the upgrade;
    the patch preserves them so a subscriber's origin-tracking (the "how far have I
    replayed from this source" bookmark) survives a major-version upgrade.
  - Without this, an upgraded subscriber can re-apply already-applied changes or
    lose its progress marker, breaking bidirectional/multi-origin setups.
- **Why it matters:** logical-replication topologies increasingly outlive a single
  major version; origin continuity across upgrade is what makes in-place upgrade
  safe for them. See `[[knowledge/skills/pg-upgrade-internals]]` and
  `[[knowledge/skills/logical-replication]]` (replication origins).

## 8. Propagate stadistinct through GROUP BY/DISTINCT in subqueries and CTEs

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Richard Guo 00:57)
- Summary:
  - Richard Guo proposes propagating the `stadistinct` (n-distinct) statistic up
    through `GROUP BY` / `DISTINCT` in subqueries and CTEs, so the planner has a
    real distinctness estimate for the output of a grouped subquery instead of a
    default guess.
  - Better distinctness estimates on grouped subquery outputs improve join-size and
    grouping estimates in the enclosing query — a common source of bad plans on
    layered/CTE-heavy queries.
- **Why it matters:** grouped-subquery outputs are an estimation blind spot; this
  is exactly the kind of statistic-propagation that turns a mis-estimated nested
  plan into a good one. See `[[knowledge/skills/planner-cost-model]]` and
  `[[knowledge/skills/extended-statistics]]`.

## 10. Fix race conditions in DropRole() and GrantRole()

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (surya poondla 04:45,
  Bertrand Drouvot 07:05)
- Summary:
  - Report and fix for races in `DropRole()` and `GrantRole()`: concurrent role
    drops/grants can operate on stale catalog state because the locking around the
    `pg_authid` / `pg_auth_members` updates is insufficient, allowing a window
    where one session's view of role membership is inconsistent.
  - Also in-flight nearby: "REVOKE's CASCADE protection doesn't work with INHERITed
    table owners" (Jacob Champion, Ayush Tiwari) — a related privilege-machinery
    correctness thread.
- **Why it matters:** role/privilege catalog updates that race can produce
  security-relevant inconsistencies (a membership that half-applies). Locking
  discipline on shared catalogs is the fix class — see `[[knowledge/skills/locking]]`.

## 11. injection_points: switch wait/wakeup to use atomics rather than latches

- **Archive:** <https://www.postgresql.org/list/pgsql-hackers/> (Michael Paquier
  06:29, Heikki Linnakangas 07:17, Andrey Borodin 19:17)
- Summary:
  - Michael Paquier reworks the injection-points test framework's wait/wakeup
    primitive to use atomic variables instead of latches, making the
    deterministic-pause mechanism cheaper and less entangled with a backend's real
    latch (which carries other signals).
  - Heikki and Andrey review the atomics-vs-latch trade-off for the test harness's
    synchronization needs.
- **Why it matters:** injection points are how PG now writes deterministic tests for
  race conditions (the standby TID-recycle and hash-split bugs above are exactly
  what they test). A cleaner wait primitive makes those tests less flaky. See
  `[[knowledge/skills/testing]]` and `[[knowledge/skills/locking]]` (atomics).
