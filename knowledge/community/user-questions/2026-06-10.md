## 1. Why is the LSN reported for pg_logical_emit_message() different

- URL: https://www.postgresql.org/message-id/d99c688994ab3a998afe26e61fe4f69f%40oss.nttdata.com
- List: pgsql-hackers
- Summary: A user observes that the LSN returned by `pg_logical_emit_message()`
  does not match the LSN that later shows up when the message is decoded /
  read back. The question is whether the returned value is the record's start
  LSN or its end LSN, and which one downstream consumers should rely on.
- **Skill tag:** `wal-and-xlog` (+ `replication-overview`)
- Ideal answer: explain `XLogInsert` returns the *end* LSN (byte just past the
  record) while logical decoding reports the record *start*; cite
  `knowledge/idioms`/`wal-and-xlog` on the XLogInsert return contract and
  `knowledge/subsystems/replication.md` for the decode-side LSN semantics.
  Tie to `XactLastRecEnd` vs record ReadRecPtr.

## 4. [PATCH] Avoid orphaning buffers when a relation's file is missing

- URL: https://www.postgresql.org/message-id/aifizBXakKoiaAiA%40MAC-CVW1VHW5R6
- List: pgsql-hackers
- Summary: Adam Lee reports that if a relation's underlying file is missing
  (e.g. after an external deletion or a half-finished drop), buffers for that
  relation can be left pinned/dirty in shared buffers with no way to evict or
  drop them cleanly. The patch makes the smgr/bufmgr path discard such buffers
  instead of orphaning them.
- **Skill tag:** `gap:storage-buffer-eviction` (closest existing:
  `knowledge/subsystems/storage-buffer.md`; no dedicated skill)
- Ideal answer: trace `DropRelationBuffers` / `smgrdounlink` and the
  `InvalidateBuffer` path in `knowledge/subsystems/storage-buffer.md`; explain
  the pin/dirty invariants (INV-* tags) that an orphaned buffer violates and
  why a missing file must not leave `BM_DIRTY` buffers un-droppable.
