### 1. Transaction Isolation level for ERP software
- URL: `https://www.postgresql.org/message-id/CAHyXU0w%2ByHUCGdBKPGY%2B4jOBdoSeQh0YbOTpjFkLfz_snRNuZA%40mail.gmail.com`
  (thread start 2026-07-15, Mihir Kandoi; active reply Merlin Moncure 07-17)
- Source: pgsql-general
- Summary: An ERPNext developer adding PostgreSQL support asks the community to
  confirm that **READ COMMITTED** is the better default than **REPEATABLE
  READ** for a highly-concurrent ERP workload. They already added row-level
  locking and advisory locks throughout the app; colleagues pushed
  retry-with-jitter instead. Real question underneath: what serialization
  anomalies does READ COMMITTED expose that RR/SSI would catch, and when do
  advisory locks vs. `SELECT … FOR UPDATE` vs. SSI apply?
- Skill tag: `locking` (isolation-level lock semantics, advisory locks,
  `FOR UPDATE`/`FOR KEY SHARE`) + `snapshot-management` (per-statement vs.
  per-transaction snapshots; how READ COMMITTED re-reads a fresh snapshot each
  statement while RR pins one).
- Ideal answer outline: `snapshot-management` — READ COMMITTED takes a new
  snapshot per statement (see `GetTransactionSnapshot`/`GetLatestSnapshot`
  in `source/src/backend/utils/time/snapmgr.c`), so it never raises
  serialization failures but is exposed to lost-update/write-skew; RR pins one
  snapshot for the xact and raises `40001` on write conflicts; SERIALIZABLE
  adds SSI predicate locks. Cross to `locking` for advisory-lock scope
  (`pg_advisory_xact_lock`) vs. tuple locks, and the multixact interaction
  when many sessions take `FOR KEY SHARE` on the same row.

### 2. Checkpointer write combining
- URL: `https://www.postgresql.org/message-id/D0E7C430-76A9-4030-B8AB-109E7D7ED8C1%40yandex-team.ru`
  (Andrey Borodin, 2026-07-19)
- Source: pgsql-hackers
- Summary: Proposal to reduce checkpointer I/O syscall overhead by combining
  writes of physically adjacent dirty buffers into larger batched writes.
  Refactors victim-buffer selection + batch construction so the strategy path
  looks forward-only while client-backend/bgwriter paths look bidirectionally
  around the target block. Borodin suggests validating with `pg_stat_io`
  eviction/reuse ratios under concurrent bulk-write workloads so ring-buffer
  behavior doesn't regress.
- Skill tag: `buffer-manager` (BufferAccessStrategy rings, clock-sweep victim
  selection, `FlushBuffer`) + `aio-readstream` (the write-combining shape
  mirrors the read-stream batching the AIO layer already does).
- Ideal answer outline: `buffer-manager` — `BufferSync`/`SyncOneBuffer` drive
  checkpoint flushes today one buffer at a time
  (`source/src/backend/storage/buffer/bufmgr.c`); combining requires sorting/
  grouping by `RelFileLocator`+block and issuing a vectored write, which is
  exactly the write-side analogue of `read_stream`'s per-block-lookup
  callback. Watch the `BufferAccessStrategy` ring interaction the author flags.

### 3. Restore vacuum_delay_point() in GIN posting-tree leaf vacuum
- URL: `https://www.postgresql.org/message-id/178447127453.110.12276981925360691905%40mail.gmail.com`
  (Paul Kim, 2026-07-19)
- Source: pgsql-hackers
- Summary: A refactor (commit fd83c83d094) inadvertently dropped the
  `vacuum_delay_point()` call from `ginVacuumPostingTreeLeaves()`, so the GIN
  posting-tree leaf sweep ignores `vacuum_cost_delay` and — worse — becomes
  non-interruptible (immune to cancel, statement_timeout, autovacuum lock
  conflicts). Large posting trees can hang, and a corrupted rightlink cycle
  loops forever. Sibling loops in `ginbulkdelete()`/`ginvacuumcleanup()` kept
  their delay points. Patch restores it after buffer release (no content lock
  held during the delay).
- Skill tag: `vacuum-autovacuum` (vacuum cost delay + interruptibility) +
  `access-method-apis` (GIN `ambulkdelete` internals).
- Ideal answer outline: `vacuum-autovacuum` — `vacuum_delay_point()` is the
  cooperative yield point that both throttles (cost-based delay) and runs
  `CHECK_FOR_INTERRUPTS`; dropping it from an unbounded leaf loop is a
  correctness bug, not just a throttling regression. Cross to
  `access-method-apis` for where AM bulk-delete loops must place delay points
  (`source/src/backend/access/gin/ginvacuum.c`).
