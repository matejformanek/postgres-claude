### 6. Why clearing the VM doesn't require registering vm buffer in wal record
- URL: `/message-id/CAAKRu_Y9DZ7%3DAXpUfX%2BESgavoHFbS8if_2gmp-B6tP6vJxzXWA%40mail.gmail.com`
- Source: pgsql-hackers
- Summary: A "why does PG do it this way" question: heap modifications clear
  the corresponding VM bit during replay without the VM page being a registered
  block in the WAL record — the asker wants the invariant that makes this safe.
- **Skill tag:** `wal-and-xlog`, `buffer-manager`
- Ideal answer outline: `wal-and-xlog` — redo of the heap record re-clears the
  VM bit via `visibilitymap_clear` because an all-visible VM bit is only ever a
  *hint that can be conservatively wrong* (set requires the heap bit; clear can
  lag); losing a clear is corrected on next heap touch, so it needn't be
  full-page-logged. A model teaching case for the hint-bit safety argument.
