## 2. COPY: validate option presence rather than option values
- URL: https://www.postgresql.org/message-id/CAD21AoCsbjvgk5tOarepYF28dW7P6NG8SLVDQ72A7AJ3yEjriw%40mail.gmail.com
- Source: pgsql-hackers
- Summary: Reports that COPY validates the *parsed value* of an option instead
  of whether the option was *specified*: `COPY … (format csv, force_array false)`
  wrongly succeeds even though `FORCE_ARRAY` is json-only, and `COPY … (format
  json, escape)` complains about ESCAPE syntax rather than "ESCAPE not allowed
  for json". Proposes checking option presence during `ProcessCopyOptions`.
- Skill tag: `error-handling` (+ `gap:copy-command`)
- Ideal answer: cite `knowledge/idioms/error-handling.md` on choosing the right
  SQLSTATE + `errmsg`/`errhint` shape for option-conflict errors; the fix lives
  in `ProcessCopyOptions()` in `src/backend/commands/copy.c`. Part of a
  `gap:copy-command` cluster this run (see entries 3, 4) — the COPY command
  family has no dedicated skill.

## 3. COPY FROM with RLS
- URL: https://www.postgresql.org/message-id/5a0cf3f1.83bb.19f12de3c8b.Coremail.jiezhilove%40126.com
- Source: pgsql-hackers
- Summary: Explores implementing row-level-security enforcement for `COPY FROM`
  (currently RLS-enabled tables reject COPY). Argues COPY FROM is simpler than
  INSERT (no RETURNING / ON CONFLICT / MERGE, no SELECT-privilege interplay),
  so it can reuse `get_row_security_policies` → build WithCheckOptions →
  `ExecWithCheckOptions`; the hard part is policies containing sublinks/subqueries,
  which the author proposes to error out on initially.
- Skill tag: `executor-and-planner` (+ `gap:copy-command`, `gap:rls`)
- Ideal answer: cite `knowledge/idioms/row-security-policy-application.md` and
  `security-barrier-views.md` for the WCO machinery, and `executor-and-planner`
  for wiring `ri_WithCheckOptions` / `ri_WithCheckOptionExprs` on the COPY
  ResultRelInfo; flag the subquery-in-policy case as the real design risk.

## 7. Do not scan index in right table if left-join condition is false from left-table columns
- URL: https://www.postgresql.org/message-id/e419bafa-cfed-4324-89ad-9b2978fff512%40gmail.com
- Source: pgsql-hackers
- Summary: A user asks why, for `parent LEFT JOIN child ON p.id=c.id AND
  p.dtype='B' WHERE p.id=1` where the sole parent row has `dtype='A'`, the plan
  still does an Index Only Scan on `child` (loops=1) even though the join
  condition is unsatisfiable from left-table columns alone — it shows up as a
  Join Filter removing the row afterward rather than skipping the inner scan.
- Skill tag: `executor-and-planner`
- Ideal answer: cite `knowledge/subsystems/optimizer.md` + `executor.md` and
  `knowledge/idioms/cost-join-paths.md`; explain nestloop param evaluation and
  why a non-outer-referencing join clause becomes a Join Filter rather than a
  pushed-down inner scan-key, and when the planner could prove the inner scan
  unnecessary.

## 10. Refactor function pg_get_multixact_stats
- URL: https://www.postgresql.org/message-id/CAEudQAonQh7be%3DwOR-CJFW%3DbgMBz5wW_bv4t0OFxbgn-794JCQ%40mail.gmail.com
- Source: pgsql-hackers
- Summary: `pg_get_multixact_stats()` calls `GetMultiXactInfo()` and computes
  the derived `members`/`membersBytes` values unconditionally, *before* checking
  `pg_read_all_stats` membership — so unprivileged callers (which get an
  all-NULL row) waste the work every time. Patch moves the computation inside the
  privileged branch and narrows the locals' scope.
- Skill tag: `fmgr-and-spi` (+ `gap:pgstat` adjacent)
- Ideal answer: cite `fmgr-and-spi` for the SQL-function body + privilege-gate
  pattern and `knowledge/data-structures/multixactid.md` /
  `knowledge/idioms/multixact-slru.md` for what `GetMultiXactInfo()` reads; note
  the `has_privs_of_role(pg_read_all_stats)` guard idiom.

## 11. Tepid: selective index updates for heap relations
- URL: https://www.postgresql.org/message-id/af02b486-6adc-494c-a357-1dd72f655dcf%40app.fastmail.com
- Source: pgsql-hackers
- Summary: A large patch series extending the HOT model: "Selective Index
  Updates" (SIU / "HOT-indexed", nicknamed "Tepid") lets an UPDATE that touches
  an indexed column stay a heap-only tuple on the same page and insert a fresh
  entry only into the *directly affected* indexes, instead of forcing a non-HOT
  update that writes every index. Goal: cut index write-amplification, bloat, and
  VACUUM overhead.
- Skill tag: `access-method-apis` (+ `wal-and-xlog`)
- Ideal answer: cite `knowledge/subsystems/access-heap.md` and
  `knowledge/idioms/vacuum-hot-prune.md` + `heaptuple-update-chain.md` for the
  HOT-chain invariants this changes; `access-method-apis` for the per-index
  `aminsert` implications and `wal-and-xlog` for the new update-record shape.
  This is the kind of on-disk/HOT-invariant change `pg-patch-review`'s
  breaking-change critic exists for.

## 13. [SP-]GiST index-only-scan visibility bug
- URL: https://www.postgresql.org/message-id/CAF0whuce3Y8ubVcS5z%2Bd%2BHrOH4VkxSxrAAOsaqq_%3D%3D6O5%2BMpYw%40mail.gmail.com
- Source: pgsql-hackers
- Summary: GiST and SP-GiST support index-only scans but don't interlock with
  index vacuum strongly enough: an IOS can return a TID that vacuum has already
  cleaned and whose heap page VACUUM then marked ALL_VISIBLE, so a dead tuple can
  be revived. Analogous to the btree fix `459e7bf8`; the thread splits this out
  as its own bug with a targeted interlock rather than the heavier full fix.
- Skill tag: `access-method-apis` (+ `gap:visibility-map`)
- Ideal answer: cite `knowledge/idioms/index-only-scan-vm-check.md` and
  `visibility-map-update.md` for the "TID must be known-valid before the VM
  check" invariant, and `access-method-apis` for `amgettuple` / pin-interlock
  obligations across GiST and SP-GiST; cross-reference the btree `459e7bf8`
  precedent.
