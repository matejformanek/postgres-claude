## 22. Guidance on calculating max_locks_per_transaction for a highly partitioned environment

- **Source:** pgsql-general · [CAK+uD7g=EE4ypAsuF8L9ZQ3FSuRD-YJ8nOkkyDnSooA_jidycQ@mail.gmail.com](https://www.postgresql.org/message-id/CAK%2BuD7g%3DEE4ypAsuF8L9ZQ3FSuRD-YJ8nOkkyDnSooA_jidycQ%40mail.gmail.com) · Vivek Gadge / Ron Johnson (2026-06-29 … 06-30)
- **Q:** A user with thousands of partitions asks how to size
  `max_locks_per_transaction` — how many lock-table slots a query touching N
  partitions actually consumes, and how the shared lock table is dimensioned.
  (The one genuine end-user internals question in the general list this window.)
- **Skill tag:** `locking` (heavyweight lock table sizing) + `gucs-config`
  (`max_locks_per_transaction` is PGC_POSTMASTER shared-memory sizing).
- **Ideal answer outline:** `locking` §heavyweight — the lock table is sized
  `max_locks_per_transaction × (max_connections + max_prepared_transactions)`;
  each partition + its indexes take a lock per statement, so a partition-wise
  join across N partitions needs ~N×(1+indexes) slots. `gucs-config`
  §PGC_POSTMASTER for why it needs a restart.
