## 1. Separate catalog_xmin from xmin in walsender hot standby feedback

- URL: https://www.postgresql.org/message-id/273DD115-F20F-4521-880C-FC08A7DE99D3%40gmail.com
- List: pgsql-hackers
- Summary: The thread proposes that a standby's hot-standby-feedback message
  send `catalog_xmin` separately from the data `xmin`, so the primary only
  holds back catalog tuples for logical decoding without pinning the data
  horizon. Today a single conflated value over-retains dead tuples on the
  primary.
- **Skill tag:** `replication-overview` (+ `wal-and-xlog`)
- Ideal answer: trace the feedback path (`XLogWalRcvSendHSFeedback`,
  `ProcessStandbyHSFeedbackMessage`) and the `GetOldestXmin` / `replication
  slot xmin` interaction; cite `knowledge/architecture/replication.md` and
  `knowledge/subsystems/replication.md` for slot `xmin` vs `catalog_xmin`
  semantics and why splitting them lets vacuum advance the data horizon.

## 2. Deadlock detector fails to activate on a hot standby replica

  `knowledge/idioms/locking-overview.md` and

  recovery-conflict interrupt mechanism that bypasses it.
