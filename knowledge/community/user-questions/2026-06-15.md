## 1. injection_points: Switch wait/wakeup to use atomics rather than latches

- URL: https://www.postgresql.org/message-id/ah5Y-1t_Xfw-CU_T%40paquier.xyz
- List: pgsql-hackers
- Summary: Michael Paquier proposes reworking the `injection_points` test
  module's wait/wakeup primitive to use shared-memory atomics instead of the
  current latch-based mechanism, so a test can reliably block a backend at a
  named injection point and wake it without depending on wait-event polling.
  Robert Haas, Heikki Linnakangas, and Andrey Borodin debate how to guarantee
  the waiting backend has actually reached the wait position before the test
  proceeds.
- **Skill tag:** `locking` (+ `testing`; closest corpus
  `knowledge/idioms/locking-overview.md`)
- Ideal answer: contrast the latch/`WaitLatch` idiom against a busy/`pg_atomic`
  spin on a shmem flag, and why injection-point determinism wants the
  wait-position observably reached; cite `knowledge/idioms/locking-overview.md`
  for the atomics-vs-latch primitive ladder and `knowledge/conventions/testing.md`
  for how injection points are wired into regress/iso/TAP. The wait/wakeup
  ordering question is the same happens-before concern the locking skill frames.

## 2. Report oldest xmin source when autovacuum cannot remove tuples

- List: pgsql-hackers
- Summary: Kyotaro Horiguchi argues that information about *what currently holds
  back the xid horizon* belongs in a SQL-visible view rather than reconstructed

  inputs (running xacts, replication slots, prepared xacts, hot-standby
  feedback) that bound removability, and how a view would iterate those sources;
  cite `knowledge/architecture/mvcc.md` for the xid-horizon definition and
  `knowledge/subsystems/access-heap.md` for where VACUUM consults the cutoff.
  `gap:vacuum-horizon` — the corpus has no focused doc on what pins OldestXmin.

  and `knowledge/architecture/executor.md` for where the dest-receiver sits at
  the top of the executor. `gap:wire-protocol` — DataRow/RowDescription encoding
  has no focused corpus doc.

## 5. autovacuum launcher crash: assert in pgstat_count_io_op (IOOP_EXTEND on pg_database's VM)

- URL: https://www.postgresql.org/message-id/CAON2xHPxvBO2xwPjEpc0s%2Bnp3nWypyOuTThb1iYPfVnR1W7vFA%40mail.gmail.com
- List: pgsql-hackers
- Summary: Ewan Young reports an autovacuum-launcher crash under heavy DDL/DML —
  `Assert(pgstat_tracks_io_op(MyBackendType, io_object, io_context, io_op))` in
  `pgstat_io.c:74`. Root cause is two commits interacting: one added an
  unconditional `visibilitymap_pin()` on the on-access prune path, another made
  INSERT set `pd_prune_xid`, so on-access pruning fires on insert-heavy catalog
  tables like `pg_database`; the launcher's read-only catalog scan then EXTENDs
  the VM fork, an `IOOP_EXTEND` the pgstat I/O matrix never expected from the
  launcher backend type.
- **Skill tag:** `debugging` (+ `gap:pgstat-io`; storage/visibility-map angle via
  `knowledge/subsystems/storage-buffer.md`)
- Ideal answer: explain the `pgstat_tracks_io_op` backend-type × io-object ×
  io-context × io-op matrix and why an autovacuum *launcher* (not worker) doing
  a VM EXTEND violates it, then point at the on-access-prune `visibilitymap_pin`
  and `pd_prune_xid`-on-INSERT change as the trigger; cite
  `knowledge/subsystems/access-heap.md` for the on-access prune path and
  `knowledge/subsystems/storage-buffer.md` for VM-fork extension. `gap:pgstat-io`
  — the I/O-stats tracking matrix has no corpus doc (recurs with the 06-13
  `gap:pgstat` and `gap:wait-events` observability cluster).

## 6. generic plans and "initial" pruning

- URL: https://www.postgresql.org/message-id/178042284979.1017.17926732655528064040.pgcf%40coridan.postgresql.org
- List: pgsql-hackers
- Summary: A CommitFest patch series implementing **initial partition pruning for
  generic (parameterized) prepared-statement plans**, so a cached generic plan
  can prune partitions at execution time from the bound parameter values instead
  of forcing a custom replan. A reviewer-visible win is reduced child-partition
  locking on reused generic plans (only the surviving partition is locked); the
  open edge is that ModifyTable/UPDATE still locks all result partitions.
- **Skill tag:** `executor-and-planner` (+ `locking` for the partition-lock
  reduction)
- Ideal answer: trace `ExecInitPartitionPruning` / `PartitionPruneInfo` and how
  initial (vs exec-time) pruning runs before lock acquisition on the cached
  plan, plus why `firstResultRels` is the lever for extending it to DML; cite
  `knowledge/subsystems/optimizer.md` for partition-pruning plan construction
  and `knowledge/idioms/locking-overview.md` for why pruning-before-lock changes
  which relation locks a generic plan takes.

## 8. WAL compatibility between Primary and Standby Server, chained clients

- URL: https://www.postgresql.org/message-id/DBAPR03MB6358A12B207F441E8B1BF984F11C2%40DBAPR03MB6358.eurprd03.prod.outlook.com
- List: pgsql-general
- Summary: Klaus Darilion asks whether WAL files are byte-identical between
  primary and standby under streaming replication, whether the standby rotates
  WAL segments in lock-step with the primary or independently, and whether a
  cascaded downstream client (e.g. Barman) must re-initialize after the standby
  re-syncs. A real operator question about WAL-segment identity and the
  streaming-replication protocol's segment boundaries.
- **Skill tag:** `replication-overview` (+ `wal-and-xlog`)
- Ideal answer: explain that streamed WAL is replayed and re-written into the
  standby's own pg_wal with identical *content* per LSN but that segment files
  are managed locally (walreceiver writes, the startup process replays), so LSNs
  match while file lifecycle is independent; cite
  `knowledge/subsystems/replication.md` for the walsender→walreceiver byte
  stream and `knowledge/architecture/wal.md` for the LSN/segment model that makes
  same-LSN content identical across the pair.

## 9. [PATCH] Fix libxml leaks in contrib/xml2 XPath functions

- URL: https://www.postgresql.org/message-id/20260602034314.7eadc4d7%40andrnote
- List: pgsql-hackers
- Summary: Andrey Chernyy reports a memory leak in `contrib/xml2`'s
  `pgxml_xpath()`: when an XPath syntax error is thrown before the function
  finishes assigning its `xpath_workspace`, the caller's error handler can't
  free the partially-built libxml state, so each failed XPath leaks ~83MB. The
  fix wraps the body in PG_TRY/PG_CATCH to free the `xmlXPathContext` and
  workspace on the error path before re-throwing.
- **Skill tag:** `error-handling` (+ `memory-contexts`)
- Ideal answer: explain why libxml allocations live outside palloc/MemoryContext
  bookkeeping so an ereport longjmp won't reclaim them, making PG_TRY/PG_CATCH
  with explicit `xmlXPathFreeContext` mandatory; cite
  `knowledge/idioms/error-handling.md` for the PG_TRY/PG_CATCH longjmp-safe
  cleanup contract and `knowledge/idioms/memory-contexts.md` for why
  non-palloc'd external resources need manual release on the error path.
