## Access methods / heap / index maintenance

### 1. Tepid: selective index updates for heap relations
- msgid: https://www.postgresql.org/message-id/CAPpHfdvOLrULXQn%2Bs%2BbEG4QiVrps%2B_3mg2EjUN39OYUKoBE20w%40mail.gmail.com
- source: pgsql-hackers
- Proposes a third heap-update mode (`HEAP_SELECTIVE_INDEX_UPDATE`) between
  full `UPDATE_ALL_INDEXES` and pure HOT: keep the new tuple heap-only with
  an inline changed-column bitmap and insert fresh index entries only into
  the indexes whose columns actually changed, pointing at the new HOT tuple
  rather than the chain root. Alexander Korotkov flags the fatal case —
  `BitmapAnd` across two indexes could drop a tuple present in one index's
  posting but not the other's, and bitmap scans tolerate false positives
  but not false negatives. [body-read]
- **Skill tag:** `access-method-apis` (+ `executor-and-planner` for the
  bitmap-scan interaction, `vacuum-autovacuum` for chain pruning)
- Ideal answer: contrast with HOT invariant in
  `knowledge/subsystems/access-heap.md`; `heap_update` index-update
  decision in `heapam.c`; bitmap-AND correctness constraint in
  `nodeBitmapAnd.c` / `nodeBitmapHeapscan.c`; why recheck can't rescue a
  false negative.

### 4. Rendezvous variables: safe to pass an access token between two extensions?
- msgid: https://www.postgresql.org/message-id/CAM_eQjy8rG7yExX5JMhA3UKycWfvtw6w-x-SBV%2BsPmKWAmXvMA%40mail.gmail.com

  that a third (malicious) extension could register the same name-keyed

  `_PG_init` order, not a global name lookup; note the trust boundary is
  "anything in `shared_preload_libraries` is already trusted".

### 30. Fix duplicate detection for null-treatment window functions
- msgid: https://www.postgresql.org/message-id/B1F56C92-9B12-41A0-A6C1-F4BDA3949237%40gmail.com
- source: pgsql-hackers
- The planner's duplicate-window-function elimination treats two window
  calls that differ only in RESPECT/IGNORE NULLS as identical, so it can
  collapse `lag(x) RESPECT NULLS` and `lag(x) IGNORE NULLS` into one and
  return wrong results. Fix includes null-treatment in the equality key. [from-subject]
- **Skill tag:** `executor-and-planner` (+ `parser-and-nodes` for the
  WindowFunc node equality)
- Ideal answer: window-function dedup in `planner`/`parse_func`; the
  `WindowFunc` node fields (`winfnoid`, args, and the missing
  null-treatment flag) that feed `equal()`; why a too-loose equality
  under-counts distinct windows.
