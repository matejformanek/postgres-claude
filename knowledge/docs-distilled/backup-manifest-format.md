# Backup Manifest Format (internals ch. 70)

The JSON sidecar `pg_basebackup` writes (`backup_manifest`) and `pg_verifybackup`
consumes. Relevant to the D data-leak / integrity threads: it's the integrity
contract over a physical backup, and the checksum-coverage gaps below are the
interesting part.

## Non-obvious claims

- **Whole manifest is a single UTF-8 JSON object.** Root is always a JSON object;
  generated by `pg_basebackup`, verified by `pg_verifybackup`; third-party tools
  may parse it. [from-docs: parent page]
- **Per-file entries live in a `Files` array.** [from-docs]
- **`Path` vs `Encoded-Path` are mutually exclusive.** `Path` (relative to backup
  root) is used when the path is valid UTF-8; `Encoded-Path` (a hex string, two hex
  digits per octet) is used when the path is **not** legal UTF-8, *or* when the user
  asks for all paths to be encoded. Tablespace files carry `pg_tblspc` + the
  tablespace OID as their first two path components. [from-docs: files leaf]
- **`Size`** — integer, the expected file size; always present. [from-docs]
- **🔑 `Last-Modified` is informational and is NOT used by `pg_verifybackup`.** It's
  the server's reported mtime at backup time; verification ignores it. So a
  manifest can "pass" with wildly wrong mtimes. [from-docs]
- **Checksum fields are conditional** — present only if the backup was taken with
  file checksums enabled:
  - `Checksum-Algorithm` — one of `CRC32C`, `SHA224`, `SHA256`, `SHA384`,
    `SHA512`. Currently uniform across a manifest but the format permits it to vary
    per file in future. [from-docs]
  - `Checksum` — hex string, two hex chars per checksum byte. [from-docs]
- **Implication:** with checksums disabled, integrity verification reduces to
  presence + `Size`; only `Size` mismatch is caught, not content corruption. The
  `Checksum-Algorithm` choice (CRC32C vs SHA-family) is the speed/strength dial.
  [inferred from the conditional-field wording]
