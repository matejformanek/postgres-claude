# Docs distilled — §68.2: BKI File Format

The lexical layer of `postgres.bki` — the bootstrap script `initdb` feeds to a
backend in bootstrap mode to lay down the initial catalogs. This page is the
*tokenizer* rules; the **command** semantics are §68.3
([bki-commands.md](./bki-commands.md)) and the file-level ordering constraints
are §68.4 ([bki-structure.md](./bki-structure.md)). Almost nobody hand-writes
BKI — `postgres.bki` is generated by `genbki.pl` from the `.dat`/`.h` catalog
sources — but knowing the lexical rules helps when debugging genbki output.

## What BKI input is, lexically

- A BKI file is a sequence of **commands**, each a sequence of **tokens**.
  `[from-docs]`
- Valid token classes: **keywords**, **special characters** (parens, commas,
  …), **identifiers**, **numbers**, and **single-quoted strings**. `[from-docs]`

## Token delimitation (the non-obvious part)

- The primary separator is **whitespace** — spaces, tabs, or newlines. `[from-docs]`
- But tokens **need not be whitespace-separated when there's no ambiguity**:
  the tokenizer ends a token as soon as the next character "syntactically
  cannot belong" to the current token. `[from-docs]`
- Command boundaries work the same way: **the next token that syntactically
  cannot belong to the preceding command starts a new command** — there is no
  explicit command terminator (no semicolon, no required newline). `[from-docs]`
- By **convention** each command is written on its own line for readability,
  but that's cosmetic, not required by the grammar. `[from-docs]`

## Quoting & literals

- Strings are enclosed in **single quotes**: `'value'`. `[from-docs]`
- The format is **case-sensitive** throughout. `[from-docs]`

## Comments

- A line whose first token starts with **`#`** is a comment and is ignored
  entirely. `[from-docs]` (Whole-line only — there is no inline/trailing comment
  form described here.)

## Why this matters for hackers

- This explains why `genbki.pl` output looks the way it does: the per-row
  `insert ( … )` lines, the bareword OIDs, the single-quoted text values. The
  tokenizer's "no ambiguity ⇒ no whitespace needed" rule is why generated BKI
  can be terse. `[inferred]`
- The lexical layer is intentionally minimal because BKI is a *machine*
  interchange format between `genbki.pl` and the bootstrap backend, not a
  human-authoring language. The real authoring surface is the catalog `.dat`
  files + `pg_*.h` headers. `[inferred]` (See
  [system-catalog-declarations.md](./system-catalog-declarations.md) and
  [system-catalog-initial-data.md](./system-catalog-initial-data.md) for that
  authoring layer.)
