# source/contrib/fuzzystrmatch/daitch_mokotoff.c

**Source pin:** master @ 4b0bf07. 571 LOC. Added 2023 by Dag Lem,
sponsored by Finance Norway / Trafikkforsikringsforeningen.

## Role

Daitch-Mokotoff Soundex implementation. Returns an ARRAY of 6-digit
soundex codes (multiple because the algorithm can produce several
alternate phonetic encodings per input word). Aims at correctness
vs the official rules + high performance via O(1) table lookup +
incremental tree-based code generation.

## Public API (SQL-callable)

- `daitch_mokotoff(text)` → text[] — array of 6-digit codes
  [source/contrib/fuzzystrmatch/daitch_mokotoff.c:122]. Returns
  NULL if input has no encodable characters.

## Invariants

- INV: each output code is exactly `DM_CODE_DIGITS = 6` characters
  [verified-by-code source/contrib/fuzzystrmatch/daitch_mokotoff.c:61,
  used at line 67]. Strings shorter than 6 are zero-padded
  ("000000" start template at line 92).
- INV: all per-call allocations live in a temp
  `AllocSetContextCreate("daitch_mokotoff temporary context",
  ...)` and the context is deleted before returning
  [verified-by-code dm.c:133-136, 148-156]. Tight cleanup model.
- INV: input is normalized to UTF-8 via `pg_server_to_any`
  [verified-by-code dm.c:139-140]. Non-encodable input
  (extension of UTF-8 conversion) → conversion ERROR upstream.
- INV: `read_char` converts UTF-8 → uppercase ASCII via a static
  ISO-8859-1 table for code points < 0x100, and a switch over
  specific Unicode code points (e.g., Ą, Ę, Ț) for the
  D-M-specific accent letters
  [verified-by-code dm.c:395-446]. Other non-ASCII chars become
  `'\x1a'` (SUB), which is filtered out by `read_valid_char`
  [dm.c:455-462].
- INV: the soundex code tree node template `start_node` is
  zero-initialized [dm.c:90-102].
- INV: `find_or_create_child_node` rejects further growth once
  `node->soundex_length == DM_CODE_DIGITS` (returns NULL)
  [dm.c:226]. Each completed leaf is `accumArrayResult`'d
  immediately [dm.c:246-255].
- INV: the main loop `while (codes && first_node[ix_node])`
  [dm.c:544] terminates when either the input is exhausted OR
  ALL pending soundex codes have reached 6 digits.

## Notable internals

- Two-tier static tables generated by `daitch_mokotoff_header.pl`
  (included as `daitch_mokotoff.h`): `letter_[26]` indexed by
  ASCII letter, each carrying sub-letter sequences for
  multi-letter phonemes.
- Per call, two alternating linked-list-of-leaves
  (`first_node[2]`) track the "current frontier" of partial
  soundex codes. `ix_node` alternates between 0 and 1 per letter.
- `update_node` is recursive — recurses once per digit in the
  multi-digit phoneme code [dm.c:316-320]. Recursion depth is
  bounded by digit count per phoneme (at most ~3 in the static
  rule tables).
- `dm_node` has `children[10]` for digits 0-9, plus the
  `next[2]` alternating linked list pointer.
- Worst-case node creation is bounded because each node carries
  a fixed soundex length, capped at 6 digits — total tree size
  is bounded by ~10^6 nodes in the absolute worst case, but
  realistically tiny.

## Trust-boundary / Phase-D surface

1. **No CHECK_FOR_INTERRUPTS** anywhere in this file
   [verified-by-code: grep returns nothing]. The main loop runs
   `update_leaves` per input letter, which iterates over the
   current frontier of partial codes (bounded by tree branching
   ≤ 10^6 but realistically tiny). For NORMAL inputs this is
   imperceptibly fast. For adversarial inputs that maximize
   alternate-code branching at every letter, the tree could grow
   per letter. **Worst-case analysis: each letter can fan out
   into at most 2 alternates × 2 next-codes × 2 prev-codes (the
   nested loops at dm.c:353-383), so the frontier doubles at
   most ~3× per letter — but is capped by leaf-completion at 6
   digits.** So the frontier saturates after ~6 letters and the
   per-letter cost stays bounded. This file is well-engineered
   on this axis.
2. **`palloc_object(dm_node)` per child node creation** at
   dm.c:230, 537 — each call allocates ~256 bytes (dm_node has
   10 child pointers + 2 next pointers + several ints). For a
   1MB input that creates K nodes, that's K × 256 bytes in the
   temp context. Bounded but unbounded-at-SQL-entry.
3. **Input length not capped at SQL entry** — text varlena bound
   only (1GB). For benign inputs, work scales linearly. For
   adversarial inputs, see #1 — the tree growth saturates so
   work is O(input_length × const).
4. **`pg_server_to_any(..., PG_UTF8)`** [dm.c:139-140] — encodes
   server-encoding text to UTF-8. If the server encoding is
   already UTF-8, this is essentially a copy. If not (SQL_ASCII
   or other), the conversion can fail or substitute. Non-issue
   for security.
5. **ISO-8859-1 table** at dm.c:113-114 is a 256-byte literal
   string. Lookup `iso8859_1_to_ascii_upper[c - 0x60]` for
   c in [0x60, 0xFF]. Bounds-safe by construction.
6. **The recursion in `update_node`** [dm.c:316-320] is bounded
   by the phoneme-code-digit count, which is at most 3 in the
   generated tables. Not a stack-DoS risk.
7. **Side-channel same as soundex/metaphone** — SELECT-grantable
   phonetic-coding oracle. Adversary with column SELECT can
   `daitch_mokotoff(col) @> ARRAY['<guessed code>']`.
8. **No bounds check on tree depth at runtime** — the
   `soundex_length < DM_CODE_DIGITS` test at dm.c:226 only
   limits per-leaf depth, not tree breadth. Trust in the static
   table's design that update fan-out is small.
