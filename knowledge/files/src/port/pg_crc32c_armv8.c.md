# src/port/pg_crc32c_armv8.c

## Purpose

ARMv8 hardware CRC32C implementations. Two arms: scalar
`pg_comp_crc32c_armv8` using ACLE `__crc32c{b,h,w,d}` intrinsics (single
instruction per chunk, byte/halfword/word/doubleword); and
`pg_comp_crc32c_pmull` using PMULL (Polynomial Multiplication Long, an ARM
crypto extension) for carry-less-multiply folding over 64-byte chunks. The
PMULL arm is the ARM analog of the AVX-512 VPCLMULQDQ arm in
`pg_crc32c_sse42.c` and is generated code from corsix/fast-crc32.
`[verified-by-code]` `[from-comment]`

## Internal landmarks

- Scalar arm (`:30-83`) — alignment preamble (`:40-57`) chews 1/2/4 bytes
  to reach uint64 alignment, then main loop (`:60-64`) does 8 bytes per
  `__crc32cd`, then trailing 4/2/1 byte residues. Identical structure to
  `pg_crc32c_loongarch.c`. `[from-comment]`
- PMULL arm (`:125-215`) — generated by corsix/fast-crc32 with PG-specific
  modifications listed in the comment block at `:88-95` (function decl,
  whitespace, pgindent-friendliness, early exit). Carries
  `pg_attribute_target("+crypto")`.
- Early-exit guard (`:136`) — if `len < 5 * sizeof(uint64x2_t)` (= 80 bytes)
  falls back to the scalar arm before the alignment preamble even runs.
  This avoids the vector setup overhead on small CRCs (the common case for
  per-WAL-record checksums). `[from-comment]`
- `clmul_lo_e`/`clmul_hi_e` helpers (`:103-121`) — inline asm wrapping
  `pmull`/`pmull2` followed by an EOR; written as asm because intrinsics
  forced extra moves. `[verified-by-code]`
- Main fold loop (`:184-191`) — four parallel 128-bit accumulators
  (`x0..x3`) each folding 16 bytes per iteration = 64 bytes total per
  loop. Reduction chain (`:193-211`) collapses them down to a single
  32-bit CRC via two more `__crc32cd` calls.

## Invariants & gotchas

- **PMULL is a `+crypto` extension, not part of base ARMv8.** The runtime
  check in `pg_crc32c_armv8_choose.c` verifies `HWCAP_PMULL`; building
  with `-march=armv8-a` does not guarantee PMULL availability. AWS
  Graviton2 has PMULL; older A53 cores don't. `[from-comment]`
- The early-exit (`:136`) means the PMULL arm only "kicks in" for CRCs over
  ~80 bytes. Most WAL records are larger so this is a perf-tuning bound,
  not a correctness one.
- **Generated code with manual PG-style edits.** The comment block at
  `:88-95` lists the four modifications applied to the corsix/fast-crc32
  output. Re-generating the file requires re-applying these edits. `[from-comment]`
- `pg_attribute_target("+crypto")` decorates the helpers and the entry; on
  GCC/Clang this enables the PMULL intrinsics for those functions only,
  not the rest of the TU.
