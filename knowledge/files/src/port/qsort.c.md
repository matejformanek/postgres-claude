# src/port/qsort.c

## Purpose

In-tree `qsort()` replacement. Always compiled and exposed as
`pg_qsort()`, regardless of platform — PostgreSQL does **not** rely on
libc's `qsort`, because historical libc implementations were quadratic on
adversarial inputs (and some were also unstable across versions).
`pg_qsort` is the canonical sort callers use across the backend; libc's
`qsort` is essentially unused. `[verified-by-code]`

The actual algorithm body is generated from `lib/sort_template.h` (a
header-based "template" mechanism that other sort variants reuse —
`qsort_arg`, tuple-sortsupport, etc.). `[verified-by-code]`

## Internal landmarks

- Template knobs set before `#include "lib/sort_template.h"`:
  - `ST_SORT pg_qsort` — generated function name (`qsort.c:7`).
  - `ST_ELEMENT_TYPE_VOID` — opaque `void *` elements (`qsort.c:8`).
  - `ST_COMPARE_RUNTIME_POINTER` — comparator passed at call time, not
    fixed at compile time (`qsort.c:9`).
  - `ST_SCOPE` empty → external linkage (`qsort.c:10`).
  - `ST_DECLARE` + `ST_DEFINE` → emit both prototype and body
    (`qsort.c:11-12`).

## Invariants & gotchas

- `pg_qsort` is **not stable** — equal keys can be reordered. Callers
  that need stability use a tiebreaker in the comparator (the common
  PG idiom is to append the original index as a secondary key).
- The standalone `pg_qsort_strcmp` comparator dereferences `char *
  const *`; misuse with a `char[]` array (rather than an array of
  pointers) is the classic footgun.
- Compiles into `libpgport` and is linked everywhere — both backend
  and frontend tools share the same sort routine.
