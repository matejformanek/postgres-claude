# snowball/ — Snowball stemmer integration for tsearch

## Purpose

Snowball is PostgreSQL's word-stemmer backend for the built-in
full-text-search (`tsearch`) dictionary stack. ~52 languages each get
their own `stem_<enc>_<lang>.c` file (autogenerated upstream from a
`.sbl` definition by the Snowball-to-C compiler at
<https://snowballstem.org>). PG ships the *derived* C files in tree —
**not** the `.sbl` sources — because most installations will not have
the Snowball compiler available [from-README: source/src/backend/snowball/README:1-30].

`dict_snowball.c` is the PG-authored glue: it implements the two
SQL-callable entry points (`dsnowball_init`, `dsnowball_lexize`) of
the `pg_ts_template` API and dispatches through a static
`stemmer_modules[]` table to the language-specific `stem(SN_env *)`
function pointer. The Snowball C runtime (`api.c`, `utilities.c`,
`snowball_runtime.h`) is also imported verbatim from upstream and
lives under `libstemmer/`.

## How it integrates into PG

1. The bootstrap install runs `snowball_create.pl` at build time. It
   emits a `pg_ts_template` row pointing at the C functions
   `dsnowball_init` / `dsnowball_lexize`, plus one `pg_ts_dict` row
   per language with the appropriate `language=...` and (where
   available) `stopwords=<lang>` option.
2. At `CREATE TEXT SEARCH DICTIONARY` (or session start), tsearch's
   `lookup_ts_dictionary_cache` calls `dsnowball_init(dictoptions)`.
   That function parses the `Language` / `StopWords` options and
   resolves the language to one of the entries in
   `stemmer_modules[]`, instantiating an `SN_env *` via the chosen
   stemmer's `_create_env` callback
   [verified-by-code: source/src/backend/snowball/dict_snowball.c:234-282].
3. At lex-time (e.g. during `to_tsvector`), tsearch calls
   `dsnowball_lexize(state, text, len)`. That function lowercases the
   input, applies the stopword filter, optionally re-encodes to UTF-8,
   calls the Snowball `stem(SN_env *)` callback under a per-dict
   private memory context, then re-encodes back
   [verified-by-code: source/src/backend/snowball/dict_snowball.c:284-362].

## Invariants

- **Generated files are read-only relative to PG.** Never hand-edit
  `libstemmer/stem_*.c` or `libstemmer/api.c` / `utilities.c`. To
  pick up an upstream fix, refresh from the Snowball repo (procedure
  below). The PG side may add minor `#include` shuffling on import
  but does not modify stemming logic
  [from-README: source/src/backend/snowball/README:24-30].
- **Contract surface is PG-authored only:** `dict_snowball.c`,
  `snowball_create.pl`, `snowball.sql.in`, `snowball_func.sql.in`. An
  upstream Snowball bump may force adjustments to these — most often
  to the `stemmer_modules[]` table when a language is added/renamed
  upstream.
- **Per-stemmer encoding is hardcoded** in `stemmer_modules[]`. A
  stemmer compiled for ISO-8859-1 cannot be used at runtime against
  a UTF-8 database; `locate_stem_module` will fall back to the UTF-8
  variant of the same language if one exists, otherwise raise
  `ERRCODE_UNDEFINED_OBJECT`
  [verified-by-code: source/src/backend/snowball/dict_snowball.c:193-232].
- **Stem-buffer ownership lives in `SN_env`.** Snowball's runtime
  allocates and grows the workspace buffer itself between calls,
  which is why `dict_snowball.c` switches to a long-lived
  `MemoryContext` (`d->dictCtx`) around each `stem()` call
  [from-comment: source/src/backend/snowball/dict_snowball.c:184-189].
- **Long-input cutoff is 1000 bytes.** Strings longer than that
  skip the stemmer entirely (still lowercased) — both to avoid
  wasted cycles on base64-like junk and to dodge known stemmer
  pathologies (the Turkish stemmer has an indefinite recursion that
  can crash on long inputs)
  [from-comment: source/src/backend/snowball/dict_snowball.c:293-302].

## How to refresh from upstream

Per `src/backend/snowball/README` [from-README]:

1. `git clone https://github.com/snowballstem/snowball.git`
2. `cd snowball && make` — builds the `.sbl`-to-C compiler and runs
   it against all bundled languages.
3. Copy the generated `*.c` from Snowball's `runtime/` and
   `src_c/` into `src/backend/snowball/libstemmer/` and the
   corresponding `*.h` files into
   `src/include/snowball/libstemmer/`.
4. The only PG-side hand-edits to the imported files are minor
   `#include` adjustments.
5. If upstream added/removed/renamed a language, update the
   `STEMMER_MODULE(...)` entries in
   `src/backend/snowball/dict_snowball.c:108-174`.
