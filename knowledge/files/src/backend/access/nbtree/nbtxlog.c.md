## Purpose

WAL replay (`btree_redo`) and masking (`btree_mask`) for nbtree. One redo function per `XLOG_BTREE_*` info byte. Also defines the AM-private memory context `opCtx` used during replay. [from-comment, nbtxlog.c:1-13]

## Helper functions

- `_bt_restore_page` (36, static) — rebuild a freshly-init'd page from a packed tuple stream. **Critical assumption**: the tuples in `from..from+len` were originally added in item-number order, so the lowest-on-page tuple has the highest item number. Adds tuples back in reverse so item numbers come out right. Used by `btree_xlog_split` and `btree_xlog_newroot` to reconstruct the right page (which is logged whole via `XLogRegisterBufData(1, ...)` rather than incrementally). [from-comment, nbtxlog.c:28-34; verified-by-code]
- `_bt_restore_meta` (80, static) — rebuild a metapage from `xl_btree_metadata`.
- `_bt_clear_incomplete_split` (137, static) — common helper to clear `BTP_INCOMPLETE_SPLIT` on a page; called from insert/split/newroot redos.

## Recovery-correctness notes [HIGH-RISK SECTION]

### Page-split replay (`btree_xlog_split`)

Replays the three or four blocks of the split record. **Lock acquisition order at replay matches primary order: left → right → original right sibling → child (clear INCOMPLETE_SPLIT).** The comment at lines 813-819 of `btree_xlog_unlink_page` makes this rule explicit for the unlink case ("In normal operation, we would lock all the pages this WAL record touches before changing any of them. In WAL replay, we at least lock the pages in the same standard left-to-right order"). The same principle applies to split replay; concurrent readers on the standby cannot observe inconsistent same-level state. [from-comment, nbtxlog.c:813-819]

For posting-list splits during page splits, replay must reconstruct `nposting` using `_bt_swap_posting` from the *original* `orignewitem` (logged) and `postingoff` (logged in `xl_btree_split.postingoff`). This is why the WAL record may carry `orignewitem` even on `_R` splits: see `nbtxlog.h:116-139` for the conditions. [from-comment, nbtxlog.h:97-139]

### Half-dead and unlink-page replay

- `btree_xlog_mark_page_halfdead`: re-creates the leaf page from scratch (`REGBUF_WILL_INIT`) with `BTP_HALF_DEAD | BTP_LEAF`, sibling links, and a dummy hikey carrying the `topparent` link. Re-applies the downlink-replacement on the subtree parent (incremental). [verified-by-code, nbtxlog.c:705-787]
- `btree_xlog_unlink_page` (789): blocks 0 (target), 1 (left sib, optional), 2 (right sib), 3 (leaf if separate), 4 (meta if applicable). Target is **rewritten from scratch** to an empty deleted page, sibling links bypass it, and `BTPageSetDeleted(page, safexid)` stamps the same safexid that the primary chose. **This safexid is the recovery-conflict gate for any future recycle of the page**: the *next* recycle of the page will emit `XLOG_BTREE_REUSE_PAGE` carrying this safexid in `snapshotConflictHorizon`, and standby replays it via `ResolveRecoveryConflictWithSnapshotFullXid`. The unlink record itself does **not** generate a snapshot conflict — that is deferred to recycle. [from-comment, nbtxlog.c:966-991; verified-by-code, nbtxlog.c:837-873]

### Delete vs Vacuum conflict generation

- `btree_xlog_delete` carries a `TransactionId snapshotConflictHorizon` + `isCatalogRel` and calls `ResolveRecoveryConflictWithSnapshot` early in replay (before changing any buffer). [verified-by-code, nbtxlog.c:640-700; from-comment, nbtxlog.h:201-208]
- `btree_xlog_vacuum` does **not** generate a conflict, because VACUUM ran heap pruning beforehand and that already emitted a conflict at the corresponding heap-page-vacuum WAL record. [from-comment, nbtxlog.h:202-208]

### Reuse-page record (`XLOG_BTREE_REUSE_PAGE`)

The standalone explanatory comment at lines 966-991 is the canonical reference for why this record exists and how it differs from "deletion" in conflict semantics. Key claim: "the GlobalVisCheckRemovableFullXid() test in BTPageIsRecyclable() is used to determine if it's safe to recycle a page. This mirrors our own test: the PGPROC->xmin > limitXmin test inside GetConflictingVirtualXIDs(). Consequently, one XID value achieves the same exclusion effect on primary and standby." [from-comment, nbtxlog.c:986-991]

The reuse record does **not** register a buffer — see explanatory comment at `nbtxlog.h:182-184`: the record includes a `RelFileLocator` explicitly because the buffer isn't part of the record.
