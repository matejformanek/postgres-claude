## Purpose

The big one. JIT-compiles a fully-formed `ExprState` step program (built by `execExpr.c`) into native code, emitting one LLVM IR function per expression. The `llvm_compile_expr` dispatcher walks `state->steps[]` and emits IR opcode-by-opcode for **121 cases** of `ExprEvalOp` ([verified-by-code] `:1009-2949`). The result is the `evalfunc_private` of the original `ExprState`; on first invocation the indirection function `ExecRunCompiledExpr` triggers actual LLVM emission via `llvm_get_function` and then patches `state->evalfunc` to the native pointer for subsequent calls. The other PG provider callbacks — release + reset-after-error — live in sibling files. ([verified-by-code] `:79-2976`).

## Internal landmarks

- **Forward declarations + tiny shim type** (`:48-67`):
  - `CompiledExprState { LLVMJitContext *context; const char *funcname; }` — what `state->evalfunc_private` becomes after JIT.
  - `ExecRunCompiledExpr` — the trampoline that links late, by symbol lookup.
  - `BuildV1Call` / `build_EvalXFuncInt` — call-site helpers.
- **`build_EvalXFunc` macro** (`:70-73`) — variadic wrapper over `build_EvalXFuncInt`. Lets a case site call e.g. `build_EvalXFunc(b, mod, "ExecEvalParamExec", v_state, op, v_econtext)` instead of building the argument array by hand.
- **`llvm_compile_expr` prelude** (`:79-197`) — creates the JIT context (`llvm_create_context`), mutable module, builds the function signature using `ExecEvalSubroutineTemplate` (`StructExprState*, ExprEvalStep*, ExprContext*` → `Datum`), and resolves entry-block `v_state` / `v_econtext` / `v_parent` from the function arguments.
- **Cached pointer loads for slots / values / nulls** (`:200-300`) — `v_scanslot`, `v_innerslot`, `v_outerslot`, `v_oldslot`, `v_newslot`, `v_resultslot` plus their `tts_values` / `tts_isnull` arrays. Loaded once at the entry block so the per-step IR can index without re-derefencing every time.
- **`opblocks` array** — one `LLVMBasicBlock` per executor step. Pre-allocated so backward / forward branches can be wired straightforwardly.
- **The opcode switch (`:1009-2949`)** — 121 cases. Major groupings:
  - **Termination / dispatch:** `EEOP_DONE_RETURN`, `EEOP_DONE_NO_RETURN`.
  - **Tuple-deform stubs:** `EEOP_INNER_FETCHSOME`, `EEOP_OUTER_FETCHSOME`, `EEOP_SCAN_FETCHSOME`, `EEOP_OLD_FETCHSOME`, `EEOP_NEW_FETCHSOME` — each emits a call into a deform function generated by `slot_compile_deform` (from `llvmjit_deform.c`) and falls back to `slot_getsomeattrs_int` for slot kinds the deformer rejects.
  - **Var fetch:** `EEOP_*_VAR`, `EEOP_*_SYSVAR`, `EEOP_ASSIGN_*_VAR`, `EEOP_WHOLEROW` — direct loads from the cached values/nulls arrays of the appropriate slot.
  - **Constants / params:** `EEOP_CONST`, `EEOP_PARAM_EXEC`, `EEOP_PARAM_EXTERN`, `EEOP_PARAM_CALLBACK`.
  - **Function call:** `EEOP_FUNCEXPR`, `EEOP_FUNCEXPR_STRICT`, `EEOP_FUNCEXPR_FUSAGE`, `EEOP_FUNCEXPR_STRICT_FUSAGE` — wire `fcinfo` and call `BuildV1Call`. Strict variants short-circuit on a NULL arg by checking each arg's null flag inline and branching to a precomputed "is-null result" block.
  - **Boolean logic:** `EEOP_BOOL_AND_*`, `EEOP_BOOL_OR_*`, `EEOP_BOOL_NOT_*` — implement three-valued logic with short-circuit branches.
  - **CASE / NULL tests / coalesce:** `EEOP_CASE_TESTVAL`, `EEOP_*_TEST`, `EEOP_NULLIF`, `EEOP_DISTINCT`, etc.
  - **Aggregation steps:** `EEOP_AGG_*` — many specialised forms (`STRICT_INPUT_CHECK`, `PLAIN_PERGROUP`, `PLAIN_PERGROUP_NULLCHECK`, `ORDERED_TRANS_*`, `PRESORTED_DISTINCT_*`). These are the analytics hot path — most of the JIT wins come from this region.
  - **"Hard" opcodes that fall through to interpreter helpers via `build_EvalXFunc`:** e.g. `EEOP_ARRAYEXPR`, `EEOP_XMLEXPR`, `EEOP_JSON_*`, `EEOP_SCALARARRAYOP`, `EEOP_SUBPLAN`, `EEOP_FIELDSELECT`. The JIT win for these is that the per-op dispatch loop is gone; the body still calls the C helper.
- **`llvm_compile_expr` epilogue** (`:2950-2976`) — disposes the IRBuilder, allocates a `CompiledExprState` (which holds only the context + funcname), points `state->evalfunc = ExecRunCompiledExpr`, charges `generation_counter`. The actual native emission is deferred — the comment at `:2953-2958` explains this is to batch many functions per emission pass.
- **`ExecRunCompiledExpr`** (`:2987-3005`) — first-call trampoline. `CheckExprStillValid` re-checks plan freshness, then `llvm_get_function` triggers emission, then `state->evalfunc` is overwritten with the native pointer so we never come through this trampoline again for this `ExprState`.
- **`BuildV1Call`** (`:3007-3054`) — the C-level fmgr V1 call site:
  1. Resolve callable via `llvm_function_reference` (declares the function by name in the module, or embeds a function pointer for unknown OIDs).
  2. `LLVMBuildStore(0)` the `fcinfo->isnull` slot before call.
  3. Emit the call, read back `isnull` after.
  4. **Pre-LLVM-22 only** (`:3032-3051`): emit `llvm.lifetime.end` intrinsic calls for `fcinfo->args` and `fcinfo->isnull` so the inliner can drop dead stores. LLVM 22 inferred this on its own.
- **`build_EvalXFuncInt`** (`:3059-3087`) — call into a referenced C function from `referenced_functions[]` of `llvmjit_types.c`. Catches arity mismatches early via `LLVMCountParams` (`:3070-3072`) — useful because LLVM only asserts much later.
- **`create_LifetimeEnd`** (`:3090-3118`) — once-per-module helper to add the `llvm.lifetime.end.p0` intrinsic. Removed for LLVM ≥ 22.

## Invariants & gotchas

- **`llvm_compile_expr` is called inside `llvm_enter_fatal_on_oom`** — codegen is the LLVM-touching surface; OOM here ⇒ FATAL by design. Counterpart `llvm_leave_fatal_on_oom` at `:2970`. [verified-by-code, `llvmjit_expr.c:2970`]
- **`state->evalfunc_private` is a `CompiledExprState *`** after JIT and remains so until next `ExecReadyExpr`. Anyone reading `evalfunc_private` after a state has been JITed must not assume the interpreter's structure. Fortunately the only sites are `ExecRunCompiledExpr` and the interpreter's free-time path. [verified-by-code, `llvmjit_expr.c:2961-2967`]
- **Late binding via `llvm_get_function` happens on first call, not first emit** — `INSTR_TIME_ACCUM_DIFF(emission_counter)` in `llvm_get_function` is the correct counter for what shows up under EXPLAIN's "JIT > Emission" line. [verified-by-code, `llvmjit_expr.c:2982-3005`, `llvmjit.c:391-407`]
- **`ExecRunCompiledExpr` overwrites `state->evalfunc`** (`:3002`) — after first call, `ExecRunCompiledExpr` is unreachable for that state. This is the "self-modifying dispatch" pattern; don't add stateful per-call logic to `ExecRunCompiledExpr` and expect it to keep running.
- **Strict func call short-circuit pre-evaluates every arg's nullness** before calling — adding a new "strict" variant means adding to the case list AND building parallel null-check blocks; you don't get strictness "for free." See e.g. `EEOP_FUNCEXPR_STRICT` ~`:1366`.
- **121 `case` labels** must stay in sync with `ExprEvalOp` (`execExpr.h` ends at `EEOP_LAST` at `:296`). Missing a case in the switch typically `elog(ERROR)`s at the default arm. Adding a new EEOP without a JIT case is allowed (compiler skips JIT for that expression), but adding without considering JIT impact has historically caused regressions.
- **No emission happens before `llvm_get_function`** — bitcode can be inspected mid-build with `jit_dump_bitcode = on` (writes `.bc` files into the data directory from `llvm_compile_module`, in `llvmjit.c:733-761`).
