## Purpose

Backend-private mapping that lets a single 32-bit field in the tuple header (`t_cid`) carry both `cmin` and `cmax` when *the same transaction* inserted and deleted the tuple. Without combos this would need two fields, costing 4 bytes per tuple header. [from-comment, combocid.c:6-15]

## Top-of-file comment (verbatim)

> "Before version 8.3, HeapTupleHeaderData had separate fields for cmin and cmax. To reduce the header size, cmin and cmax are now overlaid in the same field in the header. That usually works because you rarely insert and delete a tuple in the same transaction, and we don't need either field to remain valid after the originating transaction exits. To make it work when the inserting transaction does delete the tuple, we create a 'combo' command ID and store that in the tuple header instead of cmin and cmax. The combo command ID can be mapped to the real cmin and cmax using a backend-private array, which is managed by this module. To allow reusing existing combo CIDs, we also keep a hash table that maps cmin,cmax pairs to combo CIDs. ... With a 32-bit combo command id we can represent 2^32 distinct cmin,cmax combinations. In the most perverse case where each command deletes a tuple generated by every previous command, the number of combo command ids required for N commands is N*(N+1)/2. That means that in the worst case, that's enough for 92682 commands. ... The array and hash table are kept in TopTransactionContext, and are destroyed at the end of each transaction." [from-comment, combocid.c:6-31]

## Public surface

- `HeapTupleHeaderGetCmin(tup)` (104) — returns the real cmin; asserts `xmin == current xact`. If `HEAP_COMBOCID` set, indirects through `comboCids[]`.
- `HeapTupleHeaderGetCmax(tup)` (118) — symmetric for cmax; asserts updater xid is current (relaxed inside critical sections because `GetUpdateXid` may palloc).
- `HeapTupleHeaderAdjustCmax(tup, cmax*, iscombo*)` (153) — pre-update helper: if we're about to delete a tuple that *we inserted*, allocate a combo CID for `(cmin, *cmax)` and write that into `*cmax`. Must be called *before* entering the buffer-modifying critical section because it can OOM. [from-comment, combocid.c:147-151]
- `AtEOXact_ComboCid()` (182) — wipes the static pointers at xact end; does *not* `pfree` (TopTransactionContext teardown handles it).
- `EstimateComboCIDStateSpace`, `SerializeComboCIDState`, `RestoreComboCIDState` (297, 316, 342) — parallel-worker propagation.

## Key invariants

- **Combo CID state is per-backend, per-transaction.** Wiped by `AtEOXact_ComboCid` (and TopTransactionContext reset). [verified-by-code, combocid.c:182-193]
- **Array must have at least one free slot before `hash_search HASH_ENTER`.** If the repalloc grew the array *after* the new hash entry was inserted and then failed, the hashtable would point at a non-existent combocid. The code repallocs first (lines 241-248). [verified-by-code]
- **`GetCmax` Assert is weakened inside critical sections** because `HeapTupleHeaderGetUpdateXid` can do MultiXact disk lookups that palloc, and palloc is forbidden in critical sections. [from-comment, combocid.c:124-129]
- **Parallel-worker restore must produce the *same* combocid sequence as the leader.** `RestoreComboCIDState` calls `GetComboCommandId` for each entry and asserts the returned id equals the loop index `i` (line 361-362). If the leader and worker disagree, the worker errors out. [verified-by-code]
- **`HEAP_COMBOCID` infomask is the discriminator.** Both `GetCmin` and `GetCmax` test it before doing the array indirection; without the bit, the raw t_cid IS cmin or cmax. [verified-by-code, combocid.c:111, 133]
- **`HeapTupleHeaderAdjustCmax` is only entered when `!HeapTupleHeaderXminCommitted(tup) && current_xact == xmin(tup)`.** Otherwise, a regular cmax is sufficient. [verified-by-code, combocid.c:163-164]

## Functions of note

- **`GetComboCommandId`** (204) — lazy init: on first call creates the array (palloc in TopTransactionContext) before the hash table so that "existence of hash table asserts array exists" (line 219). Doubles array on overflow. On hash-table HIT returns the existing combocid; on MISS records `(cmin,cmax)` at `combocids[usedComboCids]`, writes that index into the entry, and increments.
- **`SerializeComboCIDState`** (316) — writes `int count` then the `ComboCidKeyData[]` array. Errors with `elog(ERROR, "not enough space to serialize ComboCID state")` if maxsize too small.
- **`RestoreComboCIDState`** (342) — preserves the leader's combocid → (cmin,cmax) mapping by calling `GetComboCommandId` in array order; the equality assertion at line 361 catches any subtle protocol drift.

## Open questions

- The 92682-commands theoretical maximum (per top comment) assumes int wraparound — no run-time guard exists; the assertion `combocid < usedComboCids` in `GetRealCmin/GetRealCmax` is a passive check. Whether wrap is actually reachable in practice is left unstated. [from-comment]
- Hash table is created with `HASH_BLOBS` and an explicit "no struct padding" comment at 252; padding could differ on exotic ABIs. [unverified]
