# `euc2004_sjis2004.c` — EUC_JIS_2004 ↔ SHIFT_JIS_2004

## Purpose

Direct (no UTF-8 trip) conversion proc for the JIS X 0213:2004
**EUC_JIS_2004 ↔ SHIFT_JIS_2004** pair. Backs the
`euc_jis_2004_to_shift_jis_2004` and `shift_jis_2004_to_euc_jis_2004`
rows in `pg_conversion`. Unlike the UTF-8 siblings in this directory,
this file is **table-free**: the transcoding is done by bytewise
algorithm in two static helpers (`euc_jis_20042shift_jis_2004` and
`shift_jis_20042euc_jis_2004`), exploiting the deterministic
relationship between the two encodings of the same JIS X 0213
codepoints.

## Internal landmarks

- Standard 6-arg fmgr unpack in both Datum functions.
- `CHECK_ENCODING_CONVERSION_ARGS(PG_EUC_JIS_2004, PG_SHIFT_JIS_2004)`
  (line 51) and inverse on line 67.
- Datum entry points delegate immediately to the local worker
  (`euc_jis_20042shift_jis_2004` / `shift_jis_20042euc_jis_2004`) which
  does the bytewise scan and writes into the caller-supplied buffer.
- Workers use `pg_encoding_verifymbchar(...)` to validate each input
  character before transcoding, and `report_invalid_encoding` /
  `report_untranslatable_char` for the failure paths (same helpers used
  by the inline `utf8_and_iso8859_1.c`).
- Returns `PG_RETURN_INT32(converted)` — the byte count of source
  consumed before any failure point.

## Invariants & gotchas

- Loaded via fmgr from `pg_conversion`.
- **Not** table-driven and **not** UTF-8-routed — this is a direct
  bytewise transcoder. Adding a new JIS X 0213 character would
  require manual updates to the byte-level decode/encode logic
  rather than re-generating a `.map`.
- JIS X 0213:2004 *does* have characters with multi-codepoint Unicode
  canonical forms, but combined characters are irrelevant here:
  both EUC_JIS_2004 and SHIFT_JIS_2004 encode the same set of
  ji-codepoints — no two-codepoint Unicode normalization is needed.
- The UTF-8 path for either encoding uses different files:
  `utf8_and_euc2004/utf8_and_euc2004.c` and
  `utf8_and_sjis2004/utf8_and_sjis2004.c`. Those DO use combined-char
  maps because the target side is Unicode.
- Invalid input → `ereport(ERROR, ...CHARACTER_NOT_IN_REPERTOIRE)`
  unless `noError = true`.
