## Purpose

"Routines to support inter-object dependencies." Implements the **graph walk** over pg_depend rows that drives DROP ... CASCADE / RESTRICT, the transformation of parser expression trees into dependency rows (`recordDependencyOnExpr`, `recordDependencyOnSingleRelExpr`), and the `ObjectAddresses` accumulator data structure. pg_depend rows themselves are written by `pg_depend.c`; this file consumes them to figure out what to delete and in what order. [verified-by-code, dependency.c:3-13]

## Mental model

- **pg_depend row:** (classid, objid, objsubid) depends on (refclassid, refobjid, refobjsubid) with deptype ∈ {`n`ormal, `a`uto, `i`nternal, `e`xtension, `x` AUTO_EXTENSION, `p`rimary (partition), `S` secondary partition, …}. See `dependency.h` for the full DependencyType enum.
- **Recursive walk:** `findDependentObjects()` scans pg_depend for entries where the target object is the *referenced* side; this gives "what depends on me". Each found dependent gets recursively visited. The walk handles internal/extension owner-flips by switching the visit-start to the owner so the owner is deleted *after* its members.
- **DEPFLAG bitset** (dependency.c:107-115) tagged onto every visited address: `ORIGINAL`, `NORMAL`, `AUTO`, `INTERNAL`, `PARTITION`, `EXTENSION`, `REVERSE`, `IS_PART`, `SUBOBJECT`. RESTRICT semantics later inspect these flags to decide which dependencies are "soft" (auto-droppable) versus "hard" (must error out).

## RecordDependencyOn semantics (load-bearing)

The actual *write* of a pg_depend row is `recordDependencyOn(depender, referenced, deptype)` in **pg_depend.c**. Semantically:

- `DEPENDENCY_NORMAL` ('n'): if referenced is dropped, depender drop requires CASCADE; otherwise RESTRICT errors out.
- `DEPENDENCY_AUTO` ('a'): depender is silently dropped when referenced is. Used for column defaults, autogenerated array types, etc.
- `DEPENDENCY_INTERNAL` ('i'): depender is an implementation artifact of referenced. Dropping referenced silently drops depender; **dropping depender directly is an error** (will be redirected up to referenced). [verified-by-code, dependency.c:651-720]
- `DEPENDENCY_EXTENSION` ('e'): like INTERNAL but the owner is an extension. Drop is redirected to dropping the extension.
- `DEPENDENCY_AUTO_EXTENSION` ('x'): hybrid — depender lives independently but is dropped if extension is dropped.
- `DEPENDENCY_PARTITION_PRI` ('P') / `_SEC` ('S'): partition relationship. Primary is dropped through the partition root; secondary is just an additional dep.

## Atomicity / locking

- `performDeletion` opens pg_depend with `RowExclusiveLock` once and threads the `Relation` through the recursion (dependency.c:289). [verified-by-code]
- Deletion locks (`AcquireDeletionLock`) on relations are AccessExclusive and **persist until end of xact** — releasing them mid-flight would let another backend see the half-dropped object.
- Pinned objects (catalog OIDs ≤ FirstNormalObjectId, i.e. created during initdb's "pin everything" phase) error immediately with "cannot drop … required by the database system" (dependency.c:550-554). The pinned-set is materialised in pg_depend as rows with refclassid=0; see `IsPinnedObject` in catalog.c:370.
