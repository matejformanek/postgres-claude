# src/include/common/int128.h

## Purpose
Portable 128-bit integer wrapper. Uses native `int128`/`uint128` (GCC/Clang
`__int128`) when `HAVE_INT128` is set; otherwise a struct of `{hi, lo}` and
hand-rolled arithmetic. Designed so callers don't branch on the
implementation choice тАФ the inline functions hide it.

## Role in PG
- **`numeric.c`** тАФ multi-digit numeric multiplication accumulators use
  INT128 for partial sums (`int128_add_int64_mul_int64`).
- **`int8.c`** тАФ int8 product overflow detection on platforms without
  native int128.
- **`src/test/modules/test_int128`** тАФ dedicated test module verifying
  the portable path matches the native path. (Mentioned in the header
  comment at `int128.h:9`.)

## API
- `typedef int128 INT128;` (native) **OR** `typedef struct { uint64 lo;
  int64 hi; } INT128;` (portable, with `WORDS_BIGENDIAN` swap)
- `PG_INT128_HI_INT64(i)` / `PG_INT128_LO_UINT64(i)` тАФ portable extraction
- `make_int128(hi, lo)` (`int128.h:75`)
- `int128_add_uint64`, `int128_add_int64`, `int128_add_int128`
  (`int128.h:92, 117, 144`)
- `int128_sub_uint64`, `int128_sub_int64` (`int128.h:158, 179`)
- `int128_add_int64_mul_int64(*i128, x, y)` (`int128.h:203`) тАФ **the hot
  workhorse for numeric**: adds the full 128-bit product `x*y` into
  `*i128`. Native path uses `(int128)x * (int128)y`; portable path
  decomposes into four 32├Ч32 тЖТ 64 partial products carefully sign-extended.
- `int128_sub_int64_mul_int64` (`int128.h:266`).
- `int128_div_mod_int32(*i128, v, *remainder)` (`int128.h:308`) тАФ divide
  by a signed 32-bit divisor returning quotient (in `*i128`) and
  remainder (in `*remainder`).
- `int128_is_zero`, `int128_sign`, `int128_compare`, `int64_to_int128`,
  `int128_to_int64`.

## State / globals
None.

## Phase D notes
- **Native vs portable consistency.** The `test_int128` module is the
  primary correctness check that the portable path matches native
  semantics. Without it, a subtle sign-extension bug in
  `int128_add_int64_mul_int64`'s manual decomposition would silently
  corrupt numeric arithmetic. [from-comment: int128.h:9]
- **No overflow detection.** This wrapper does *not* report 128-bit
  overflow. `int128_div_mod_int32`'s docstring (`int128.h:303-305`)
  explicitly punts: "*This provides no protection against dividing by 0,
  or dividing INT128_MIN by -1, which overflows. It is the caller's
  responsibility.*" Callers in numeric.c know their inputs' bounds.
  [from-comment]
- **Byte layout.** `int128.h:42-44` and the `WORDS_BIGENDIAN` `#ifdef`
  at `int128.h:56-62` aim to match what a native int128 would look like
  in memory, so `union { INT128 portable; int128 native; }` works for
  test harnesses. **This is a load-bearing layout invariant** тАФ anyone
  refactoring the struct must preserve it.
- **No untrusted input.** Pure math on caller-controlled integers.

## Potential issues
- [ISSUE-undocumented-invariant: portable INT128 byte layout must match
  native int128 on the same platform for the test module to work. If a
  future change adds padding, test_int128 might silently pass but produce
  wrong memcpy-based tests. (low)]
- [ISSUE-correctness: int128_div_mod_int32 documented to be unsafe on
  div-by-zero / INT128_MIN/-1. Caller-responsibility model. A future
  caller from a less-bounded context could trip it. (maybe)]
- [ISSUE-dead-code: portable (non-HAVE_INT128) path is rarely exercised
  on modern CI; mostly there for 32-bit and pre-int128 compiler support.
  (low)]
