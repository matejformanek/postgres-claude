#### Verdict

**REJECT-A** (per `review-checklist` Phase 0 rubric — identified all
critical design problems + proposed correct alternative).

#### Concrete reasons against the design (5)

1. **Network I/O inside `money_out()` violates immutability + safety
   invariants.** `money_out` is the per-row output function for the
   `money` type. It is called from any `SELECT money_col FROM …` and is
   marked `provolatile => 'i'` in `pg_proc.dat` (immutable). Reaching
   over the network turns it into volatile-at-best, and the function is
   called inside the executor's tight per-tuple loop. This forecloses on
   a documented PG invariant: type-output functions don't do I/O. Cite:
   `source/src/include/catalog/pg_proc.dat` `cash_out` row.

2. **Network access in a backend `Datum` function is not safe period.**
   No timeout policy, no DNS sandboxing, no retry semantics, no
   security model. PG backend `.c` does not make outbound HTTP — this
   is well-established design. (`src/backend/libpq/` is only for the
   wire protocol to the client.)

3. **Server-wide locale-dependent rendering of stored values breaks
   logical replication + dumps.** A row with money value `$100` would
   render differently on the subscriber than the publisher, depending
   on FX-rate lookup at decode time. Logical decoding output (per
   `knowledge/subsystems/replication.md`) assumes deterministic
   `*_out()` functions.

4. **`money` type is already deprecated guidance.** PG docs already
   recommend `numeric` for currency; adding new behavior to `money`
   deepens a footgun the project has been trying to phase out.

5. **No upgrade path / no test surface.** A row's textual rendering
   becoming time-dependent means `pg_dump` is not reproducible. The
   proposal includes no test plan that would demonstrate determinism.

**Tom Lane.** Type-system invariants + dump-determinism + immutability
discipline are core reflexes. Secondary objector: Andres Freund (per-row
performance, executor hot path). See `.claude/personas/tom-lane.md` if

#### Alternative (offered to the proposer)

If the underlying goal is "render money values in a user-chosen target
currency", the right shape is an immutable user-callable function
`money_in_currency(money, text) returns text` that takes the exchange
rate as a parameter (not a side effect), keeping `money_out()` pure. FX
rates can come from a user table the function joins against.

### baseline

This proposal has obvious problems:

1. You can't make HTTP calls from a backend C function. PostgreSQL backends
   don't have an HTTP client and would block the executor on network I/O.

2. `money_out()` is called for every row in any SELECT that returns a money
   column. Calling out to an external API per row would be catastrophic for
   performance.

3. Output functions in PG are expected to be deterministic. The dump/restore
   cycle would produce different output than the live database.

I'd recommend the author drop the patch and instead provide an extension
that exposes a callable function `currency_convert(money, text)` returning
text, where the user explicitly invokes the conversion. That keeps
`money_out()` pure.

proposal anyway. Looks like one.
