# Adding a new top-level SQL statement (BLEND)

Touch four layers in order, build between each. The skill's checklist (`.claude/skills/parser-and-nodes/SKILL.md` ¬ß"new SQL statement") drives the sequence.

## 1. Keywords ‚Äî `src/include/parser/kwlist.h` and `src/backend/parser/gram.y`

- Add `BLEND` (and any sub-words like `INTO` if not already there) to `kwlist.h` in **alphabetical order**, with the appropriate category (UNRESERVED_KEYWORD / COL_NAME_KEYWORD / TYPE_FUNC_NAME_KEYWORD / RESERVED_KEYWORD). The file uses `PG_KEYWORD("blend", BLEND, UNRESERVED_KEYWORD, BARE_LABEL)` style entries.
- In `gram.y`, add `BLEND` to the matching `unreserved_keyword:` / `col_name_keyword:` / `reserved_keyword:` rule so the parser knows about the token (this is in addition to `kwlist.h`).

Note: a new *keyword* alone needs no `scan.l` change ‚Äî keywords flow through `kwlist.h`. Only touch `scan.l` if you're inventing a brand-new *token shape* (a new comment syntax, a new literal form, etc.). [SKILL.md ¬ßlexer]

## 2. Grammar rule ‚Äî `src/backend/parser/gram.y`

- Add a top-level rule `BlendStmt: BLEND target INTO source { ... }` that uses `makeNode(BlendStmt)` in the action and fills fields from `$2`, `$4`, with `@1` as the `location`.
- Wire it into the `stmt:` rule (search for `stmt:` in `gram.y` ‚Äî that's the top-level production listing every statement kind).
- Pick a template close in shape to what you want. The skill calls out `DropStmt` / `VariableSetStmt` as good simple-statement templates. For something MERGE-shaped, copy MERGE's rule (`MergeStmt:`). Do **not** invent helper functions in gram.y if `parser/parse_utilcmd.c` already has one.

## 3. Parse node ‚Äî `src/include/nodes/parsenodes.h`

(Utility statements live in parsenodes.h; expression nodes go in primnodes.h; plan nodes in plannodes.h. BLEND-as-statement is parsenodes.h.)

```c
typedef struct BlendStmt
{
    NodeTag     type;            /* must be first */
    RangeVar   *target;
    RangeVar   *source;
    List       *whenClauses;     /* if you have them */
    /* ... */
    ParseLoc    location;        /* last field for top-level stmts */
} BlendStmt;
```

- First field is `NodeTag type;` ‚Äî `makeNode(BlendStmt)` will set it to `T_BlendStmt` via `palloc0` (so every other field starts NULL/0).
- Last field by convention is `ParseLoc location;`.
- Per-field annotations like `query_jumble_ignore`, `equal_ignore` can be applied if the field shouldn't participate in jumble/equality. Struct-level `pg_node_attr(...)` for the whole node. Full list in `source/src/include/nodes/nodes.h:43-125`.

`parsenodes.h` is already in `gen_node_support.pl`'s `@all_input_files` list (see `gen_node_support.pl:53-77`), so the `T_BlendStmt` tag is generated automatically into `nodes/nodetags.h`. **Do not hand-edit `nodetags.h`.**

## 4. Generated copy/equal/out/read funcs ‚Äî automatic

`gen_node_support.pl` runs as part of the build. It produces:

- `copyfuncs.funcs.c` ‚Äî `_copyBlendStmt` (deep copy)
- `equalfuncs.funcs.c` ‚Äî `_equalBlendStmt`
- `outfuncs.funcs.c` ‚Äî `_outBlendStmt` (text serialization for `nodeToString`)
- `readfuncs.funcs.c` ‚Äî `_readBlendStmt` (inverse, for `stringToNode`)
- `queryjumblefuncs.funcs.c` ‚Äî for query-id

After your first build, inspect those generated chunks under `dev/build-debug/src/backend/nodes/` and verify each field is handled the way you want. If something's wrong (e.g. you wanted a field skipped from equality), add a `pg_node_attr(...)` annotation or per-field attr and rebuild.

If the autogenerated bodies don't fit (e.g. you want custom copy semantics like `Const` has), add `pg_node_attr(custom_copy_equal)` on the struct and write `_copyBlendStmt` / `_equalBlendStmt` by hand in `copyfuncs.c` / `equalfuncs.c` ‚Äî the generator will skip yours. (`copyfuncs.c.md` shows the pattern for `_copyConst`, `_copyA_Const`, `_copyExtensibleNode`.)

## 5. Analyze / execute

Is BLEND optimizable (like MERGE/INSERT/UPDATE ‚Äî runs through the planner) or utility (like CREATE/DROP)?

- **Optimizable**: write `transformBlendStmt(ParseState *pstate, BlendStmt *stmt)` in `src/backend/parser/analyze.c`. Wire it into the `switch (nodeTag(parseTree))` in `transformStmt` (`analyze.c:334-444`). It must build a `Query{commandType=CMD_BLEND}` (you'd need a new CmdType too) or, more likely, reuse `CMD_MERGE`-style semantics. **Crucial caution from `analyze.c:363-367`**: any new case in `transformStmt` must also be added to `stmt_requires_parse_analysis()` (`:468-505`) and `analyze_requires_snapshot()` (`:512-529`). Three sites change together.
- **Utility** (DDL-like): no `transformBlendStmt` needed ‚Äî the default branch wraps it untransformed in `Query{commandType=CMD_UTILITY, utilityStmt=stmt}`. Execution lives under `src/backend/commands/` and is dispatched by `ProcessUtility` / `standard_ProcessUtility` in `src/backend/tcop/utility.c`.

If utility, you'll also write a `parse_utilcmd.c` transform that runs at execution time, and a `commands/blend.c` that actually does the work.

## 6. Walker integration ‚Äî `src/backend/nodes/nodeFuncs.c`

If BlendStmt holds child expressions (a WHERE clause, target lists, etc.), make sure `expression_tree_walker_impl` / `expression_tree_mutator_impl` / `raw_expression_tree_walker_impl` descend into them. Grep for `T_MergeStmt` (or another sibling) in `nodeFuncs.c` to find every site that needs a new case. The README explicitly calls this out as "step 3 ‚Äî add cases to the functions in nodeFuncs.c". [from-README + nodeFuncs.c.md]

## 7. Catversion bump?

Yes if BlendStmt (or the resulting Query) can end up **stored** in a catalog parse tree ‚Äî typically: rule actions (`pg_rewrite.ev_action`), view definitions, anything serialized via `nodeToString` into a system catalog. Bump `CATALOG_VERSION_NO` in `src/include/catalog/catversion.h`. [SKILL.md ¬ß"Adding a node type renumbers existing tags"]

Adding any new node type also renumbers later `T_*` enum values, so a full rebuild (`--enable-depend` helps) is required ‚Äî but no initdb is needed because nodetags never go to disk; they're just a runtime dispatch table. (`knowledge/idioms/node-types-and-lists.md:28-31`.)

## 8. Test

- Add a regression test under `src/test/regress/sql/` + matching `expected/`.
- Pre-submit smoke from SKILL.md:
  ```
  cd dev/build-debug && ninja && meson test --suite regress -q
  ```
- For tree-shape changes, run a separate initdb with `PG_TEST_INITDB_EXTRA_OPTS="-c debug_copy_parse_plan_trees=on -c debug_write_read_parse_plan_trees=on -c debug_raw_expression_coverage_test=on"` ‚Äî these exercise the copy/serialize/walker paths on every test query and will scream if your new node has a bad round-trip.
