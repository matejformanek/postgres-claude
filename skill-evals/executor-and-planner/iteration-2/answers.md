## Eval 2 — Path → Plan, where does set_plan_references fit?

### with_skill

Two-currency model:
- **Path** = lightweight, cost-comparable candidate stored in
  `RelOptInfo.pathlist`. Has `pathtype`, costs, pathkeys, param info,
  parallel flags, rows.
- **Plan** = the actual instruction tree the executor will run.
  Created only for the chosen Path.

**Pipeline** (`planner.c:333` and downstream):
1. `planner` → `standard_planner` builds `PlannerGlobal`.
2. `subquery_planner` (`planner.c:775`) runs per `Query`. Calls
   `grouping_planner` (`planner.c:1775`).
3. `grouping_planner` → `query_planner` → `make_one_rel` → join search.
   Every `RelOptInfo` ends with a populated `pathlist` and
   `cheapest_total_path` / `cheapest_startup_path` set by `set_cheapest`.
4. Upper-rel paths added for grouping, windows, distinct, ordered, limit.
5. `create_plan(best_path)` (`createplan.c:339`) — Path → Plan handoff.
6. `create_plan_recurse` (`createplan.c:390`) — `switch
   (best_path->pathtype)` dispatching to `create_seqscan_plan`,
   `create_nestloop_plan`, etc. Walks the Path tree top-down,
   emitting a parallel Plan tree.
7. **`set_plan_references`** (`setrefs.c:291`) runs **after**
   `create_plan` returns.

**What `set_plan_references` actually does**: during planning, Vars in
quals and target lists are numbered per-subquery as `(varno, varattno)`
referencing each subquery's RTE table. The executor doesn't have that
view — at runtime it fetches Var values from `econtext->ecxt_scantuple`
/ `ecxt_innertuple` / `ecxt_outertuple`, which correspond to scan slot,
`INNER_VAR`, and `OUTER_VAR`. `set_plan_references` walks the finished
Plan tree and renumbers Vars from the planner's per-subquery scheme to
the executor's slot-based scheme. It also:

- Assembles `finalrtable` on `PlannerGlobal` by folding per-subquery
  RTEs together.
- Accumulates `relationOids` for plan-cache invalidation.
- Records `invalItems` so the plan cache can invalidate when relevant
  catalog rows change.

Internally the entry point at `setrefs.c:291` recurses into
`set_plan_refs` (`setrefs.c:642`) — a big switch on `nodeTag` that for
each plan node calls the appropriate walker:
- `fix_scan_expr` (`setrefs.c:160`) for scan-like nodes.
- `fix_join_expr` (`setrefs.c:186`) for join-like nodes.
- `fix_upper_expr` (`setrefs.c:196`) for nodes above joins
  (Agg/Window/Sort-like).

**For your own new node**: any new Plan node holding expressions must
be plugged into the appropriate walker in `set_plan_refs` —
otherwise Vars won't resolve at run time.

### baseline

The planner has two stages where data is shaped: first it generates
**Paths** (lightweight cost-comparable candidates), then once the
cheapest is chosen it converts that one Path into a **Plan** (the
actual instruction tree).

The Path → Plan happens in `create_plan`, which lives in
`createplan.c`. It dispatches per pathtype to per-type `create_*_plan`
helpers (`create_seqscan_plan`, `create_nestloop_plan`, etc).

After `create_plan` produces the Plan tree, `set_plan_references`
runs. Its main job is to renumber Vars: during planning, Vars are
in terms of the RTE table; at runtime, the executor wants to look
up values in inner/outer/scan tuple slots. So Vars get rewritten
from `(varno, varattno)` form to `INNER_VAR`/`OUTER_VAR`/scan-attno
form. It also collects a global rtable and the relation OIDs needed
for plan cache invalidation.

For exact file:line, I'd guess `createplan.c:~300-400` for
`create_plan` and `setrefs.c` for `set_plan_references` but I'd
need to grep. New plan nodes carrying expressions need to be added
to whatever walker `setrefs.c` uses — I think it's a family of
`fix_*_expr_walker` functions but I'd verify.

## Eval 3 — "unrecognized node type" after planner scaffolding

### with_skill

The skill names this exact scenario as the #1 cause of "unrecognized
node type": **tag asymmetry** between the dispatch switches.

There are **three (or four)** switches that must learn the new tag,
and they don't all switch on the same tag:

1. `ExecInitNode` (`execProcnode.c:142`) — switches on the **Plan**
   tag (input is a `Plan *`). Needs `case T_YourPath:` calling
   `ExecInitYourScan`.
2. `ExecEndNode` (`execProcnode.c:543`) — switches on the
   **PlanState** tag (input is a `PlanState *`). Needs
   `case T_YourScanState:`.
3. `ExecReScan` in **`execAmi.c`** (NOT `execProcnode.c`) — switches
   on the **PlanState** tag. Needs `case T_YourScanState:`.
4. `MultiExecProcNode` (`execProcnode.c:488`) — also PlanState tag.
   **Skip for a tuple-returning scan**; only needed for hashtable /
   bitmap producers.

Your statement "the dispatch in execProcnode.c looks like it should
work" is the giveaway: **`ExecReScan` lives in `execAmi.c`, not
`execProcnode.c`**. That's the most common miss. If you only audited
`execProcnode.c`, you missed it.

**Localize the bug** — look at where the error fires:
- During plan init? → `ExecInitNode` is missing its case (Plan tag).
- During cleanup at end of query? → `ExecEndNode` (PlanState tag).
- Only when something rescans (e.g., your node sits under a NestLoop
  inner)? → `ExecReScan` in `execAmi.c` (PlanState tag).

If the wrong tag was used (e.g., `T_YourPath` in `ExecEndNode`
instead of `T_YourScanState`), the switch will fall through to
default and emit "unrecognized node type".

**Also verify**:
- The header `nodeYourScan.h` is `#include`d in the alphabetical
  block at `execProcnode.c:77-119`.
- Both the Plan-side type and the PlanState-side type have NodeTags
  generated by `gen_node_support.pl`. The header must be reachable
  from `@all_input_files`.
- `meson.build` and `Makefile` include the .c file (otherwise the
  build wouldn't link your `ExecInit*` symbol — different failure
  mode, but worth confirming you're running the build you think
  you are).
- Your `create_plan_recurse` switch in `createplan.c:390` actually
  emits the right Plan tag from the right pathtype.

### baseline

The error means one of the executor dispatch switches doesn't have
a case for your new node tag. There are three switches I'd check:

1. `ExecInitNode` in `execProcnode.c` — needs a case calling your
   `ExecInit*`.
2. `ExecEndNode` in `execProcnode.c` — needs a case calling your
   `ExecEnd*`.
3. `ExecReScan` — I think this lives in `execProcnode.c` too, or
   maybe `execAmi.c`. I'd grep.

Plus possibly `MultiExecProcNode` if your node returns something
other than tuples — but for a scan, no.

Also things to check:
- Header included in `execProcnode.c`.
- NodeTags generated by `gen_node_support.pl` for both Plan-side
  and PlanState-side types — header must be in the input list.
- Path → Plan dispatch in `create_plan_recurse` actually emits
  your new node type.
- The PlanState struct field's first member is `PlanState ps`.

A subtle gotcha: the Init switch dispatches on the Plan-side tag
(it gets a `Plan *`), but End and ReScan get a `PlanState *`, so
they dispatch on the State-side tag. If you mixed those up, the
switch falls through.

To localize: the error message will tell you which switch fired —
init/end/rescan. That points at which case is missing.
