## Eval 2 — Path → Plan handoff and where set_plan_references fits

### Attempt A

The planner has a **two-currency model**: it explores alternatives as Paths
(cheap, comparable, structure-sharing) and only converts the chosen one
into a Plan (executable) at the very end.

**Pipeline** (from `planner.c:333`/`351` and onward):
1. `planner` → `standard_planner` constructs `PlannerGlobal`.
2. `subquery_planner` (`planner.c:775`) runs per `Query` — once at the top
   level plus once per surviving subquery/CTE/sublink. It calls
   `grouping_planner` (`planner.c:1775`).
3. `grouping_planner` calls `query_planner` → `make_one_rel` → eventually
   `make_rel_from_joinlist` → `standard_join_search` (or `geqo`). At the
   end of this, every considered `RelOptInfo` has a `pathlist` of Paths
   and `cheapest_total_path` / `cheapest_startup_path` set.
4. Back in `grouping_planner`, upper-rel paths are added for grouping,
   windows, distinct, ordered, limit.
5. The chosen final Path is handed to `create_plan` (`createplan.c:339`).
6. `create_plan_recurse` (`createplan.c:390`) is a `switch (best_path->pathtype)`
   that dispatches to `create_seqscan_plan`, `create_nestloop_plan`, etc.
   This walks the Path tree top-down and emits the parallel Plan tree.
7. **Then `set_plan_references`** runs (`setrefs.c`).

**What `set_plan_references` actually does**: during planning, Vars in
quals and target lists are numbered per-subquery / per-RTE. The executor
doesn't have that view — at runtime it knows about INNER tuple, OUTER
tuple, and the current scan tuple's columns. `set_plan_references` walks
the finished Plan tree and renumbers Vars from the planner's per-subquery
numbering scheme to the executor's slot-based scheme (INNER_VAR,
OUTER_VAR, scan attno). It also de-duplicates RTEs into the global
`finalrtable`, records relation OIDs for plan-cache invalidation, etc.

**Why this matters for new nodes**: if you add a new Plan node that
carries expressions (quals, tlists, anything with Vars), you have to add
a case to `setrefs.c`'s `fix_*_references_walker` family. The skill
flags this in "Common mistakes" — skipping it produces Vars that point
at the wrong slots at run time.

### Attempt B

Two-currency model:
- **Path**: lightweight, cost-comparable record. Stored in
  `RelOptInfo.pathlist`. Has `pathtype`, costs, pathkeys (sort order),
  param info, parallel flags, rows. Multiple paths per RelOptInfo
  compete via `add_path`'s dominance pruning.
- **Plan**: what the executor will run. Created only for the chosen Path.

**Where Path → Plan happens**: `create_plan(best_path)` at
`createplan.c:339`. Internally `create_plan_recurse` (`createplan.c:390`)
switches on `best_path->pathtype` and calls per-type builders like
`create_seqscan_plan` (`createplan.c:2755+`), `create_nestloop_plan`,
etc. The Path tree is walked top-down; for each Path it produces the
matching Plan node and recurses on subpaths.

A Path omits things the executor needs but the cost model doesn't (full
qual expression trees in evaluable form, complete tlist with
expressions). A Plan omits things the planner used only for cost reasoning
(pathkeys, parameterization). So `create_foo_plan` does the translation
work: copying qual lists, building the tlist, attaching any per-node
fields (sort keys, hash clauses) the executor will consume.

**`set_plan_references` (setrefs.c)** runs *after* `create_plan` returns.
Its job is the final renumbering pass:
- During planning, Vars are numbered as `(varno, varattno)` referring to
  the planner's RTE table at this subquery level.
- At execution, the executor needs to fetch Var values from
  `econtext->ecxt_scantuple`, `ecxt_innertuple`, or `ecxt_outertuple`.
  These correspond to `INNER_VAR`, `OUTER_VAR`, or a specific scan
  relation's slot.
- `set_plan_references` walks the finalized Plan and rewrites each Var
  reference to the slot-based form the executor expects. It also folds
  per-subquery RTEs into a global `finalrtable` on `PlannerGlobal`, and
  records `relationOids` and `invalItems` for plan-cache invalidation.

**For your own new node**: any new Plan node holding expressions must be
known to `setrefs.c`'s `fix_*_references_walker` cases. Skipping this is
listed under "Common mistakes" — you'll see Vars failing to resolve at
runtime.

Concrete sequence:
```
grouping_planner → set_cheapest paths chosen
  ↓
create_plan(best_path)        createplan.c:339
  ↓ create_plan_recurse switches on pathtype
                                createplan.c:390
  ↓ produces Plan tree
set_plan_references            setrefs.c
  ↓ Vars renumbered to INNER_VAR / OUTER_VAR / scan attno
  ↓ finalrtable assembled, relationOids recorded
return PlannedStmt
```
