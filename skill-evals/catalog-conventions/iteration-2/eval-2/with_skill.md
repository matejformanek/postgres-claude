# Eval 2 — with skill

**Prompt:** Add a new boolean column `relhasfrobnitz` to `pg_class`.

## Answer

### Edits in pg_class.h

`pg_class.h` (under `src/include/catalog/`) defines `FormData_pg_class`.
Add the new field at the **end of the fixed-length section**, before
any `CATALOG_VARLEN` block. Appending at the end of the fixed-length
section is the standard convention because it keeps the offsets of all
pre-existing `Form_pg_class->existing_field` references stable —
moving fields around mid-struct churns ABI for code that reads those
fields.

```c
    bool        relispopulated;
    char        relreplident;
    bool        relispartition;
    /* ... existing fields ... */
    bool        relhasfrobnitz BKI_DEFAULT(f);   /* NEW */

#ifdef CATALOG_VARLEN
    aclitem     relacl[1];
    /* ... */
#endif
```

The `BKI_DEFAULT(f)` annotation is what makes this tractable: any
bootstrap row that doesn't explicitly specify `relhasfrobnitz` gets
`false` automatically. That's the same machinery `pg_proc` defaults
use.

### Do I need to touch every existing row in pg_class.dat?

**No** — and in fact you can't, because `pg_class` doesn't ship a
`.dat` file. The bootstrap `pg_class` rows for system catalogs are
generated by `genbki.pl` from the `CATALOG(...)` declarations of each
catalog header. With `BKI_DEFAULT(f)` in place, those auto-generated
rows all get `false` without any further edits.

If this were `pg_proc` or another `.dat`-backed catalog, the same rule
holds: `BKI_DEFAULT` means you don't have to touch existing rows.

Varlena/nullable trailing columns must be inside `#ifdef CATALOG_VARLEN`
— this doesn't apply to a `bool`, but the rule is worth remembering
for future columns. `bool` is fixed-length and goes in the main
struct.

### "Without recompiling everyone" — can I avoid it?

You **cannot**. Adding a field to `FormData_pg_class` changes
`sizeof(FormData_pg_class)` and the offsets of every field that comes
after it. Any extension or out-of-tree code that does
`((Form_pg_class) GETSTRUCT(tup))->relsomething` for a field placed
after your new one needs to be recompiled against the new headers.
This is precisely why you append at the *end* of the fixed-length
section — fields placed *before* your new field keep their offsets, so
extensions that only touch those keep working.

PostgreSQL doesn't promise catalog ABI stability across minor versions
in the sense the question implies. After a `CATALOG_VERSION_NO` bump
all loadable modules effectively need a rebuild against the matching
headers.

### Catversion + rebuild + initdb

- Bump `CATALOG_VERSION_NO` in `src/include/catalog/catversion.h`
  (`YYYYMMDDN`). Required because you added a catalog column.
- Full rebuild — `ninja -C build`. `genbki.pl` regenerates
  `postgres.bki`, `pg_class_d.h`.
- `rm -rf data && initdb -D data`. Old clusters won't open.

### Regression test fallout

- `meson test -C build --suite regress` — expect to fix some
  `expected/*.out` files where queries enumerate `pg_class` columns
  (e.g. `psql_crosstab`, possibly some `rules`/`sanity_check` output).
  Read the failures, copy the new expected output if the only change
  is the extra column.
- `opr_sanity` / `type_sanity` won't complain about a bool column on
  `pg_class`, but watch them anyway.
