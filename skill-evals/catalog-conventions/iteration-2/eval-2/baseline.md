# Eval 2 — baseline (no skill)

**Prompt:** Add a new boolean column `relhasfrobnitz` to `pg_class`.

## Answer

### pg_class.h

`pg_class.h` is in `src/include/catalog/`. It defines
`FormData_pg_class` via a `CATALOG(...)` declaration. Add the new
field — typically you'd add it at the end of the fixed-length section
to avoid shifting offsets of preceding fields, which would otherwise
require recompiling lots of code that does `Form_pg_class->relfoo`:

```c
    bool relhasfrobnitz BKI_DEFAULT(f);
```

Use `BKI_DEFAULT(f)` so that the bootstrap rows (which `genbki.pl`
generates for system catalogs) and any explicit rows that omit your
column get `false` automatically.

For varlena or nullable trailing columns you'd need to wrap in
`#ifdef CATALOG_VARLEN`, but for a fixed-size `bool` that doesn't
apply.

### pg_class.dat

`pg_class` doesn't actually ship a `.dat` file — its rows are
generated by `genbki.pl` from the `CATALOG` declarations of each
catalog. So you don't have to touch every row; `BKI_DEFAULT` handles
it. For `.dat`-backed catalogs the same `BKI_DEFAULT` mechanism means
you don't need to update existing rows.

### Existing code reading Form_pg_class

You can't avoid recompiling code that reads `Form_pg_class`. Adding a
field to the struct changes `sizeof(FormData_pg_class)` and shifts the
offset of any field placed after it. Extensions and out-of-tree code
that touch fields placed AFTER your new field need a rebuild against
the new headers. That's exactly why you append at the end — fields
*before* your new field keep their offsets.

Postgres doesn't guarantee catalog ABI compatibility across major
versions, and after `CATALOG_VERSION_NO` bumps any loadable modules
effectively need to be rebuilt against the matching server headers.

### Catversion + rebuild + initdb

- Bump `CATALOG_VERSION_NO` in `catversion.h` (`YYYYMMDDN`).
- Full rebuild — `ninja -C build` or `make`. `genbki.pl` regenerates
  `postgres.bki` and `pg_class_d.h`.
- Re-initdb — old data dirs won't open with the new catversion.

### Regression fallout

- Some expected outputs that enumerate `pg_class` columns will need
  updating (`psql_crosstab`, maybe `rules`, `sanity_check`).
- `opr_sanity`/`type_sanity` likely won't complain about a `pg_class`
  bool column.

### Docs

Update `doc/src/sgml/catalogs.sgml` — there's a section that documents
each column of `pg_class`. Add a `<row>` for `relhasfrobnitz`.
